### Test Design

| Date | PR | Learning |
|------|----|----------|
| 2026-04-22 | #839 | Module-level constant computed from env var at import time is not overridable by monkeypatch.setenv — when `_DEFAULT_PATH = Path(os.environ.get("VAR", default))` is evaluated at module scope, the constant is frozen at import time; monkeypatching the env var in a test has no effect on the already-computed value. Tests that call the function with `path=None` (using the module constant) and set the env var via monkeypatch silently test the wrong path. Detection: when a module computes a configuration value from an env var at module scope, tests must either pass the override explicitly as a parameter, or reload the module inside the test. |
| 2026-04-22 | #839 | Assertion softening to "not-legacy-value" cannot distinguish wired-correctly from silently-failed-with-fallback — replacing `assert x == "specific"` with `assert x != "legacy"` passes for both the correct wired value and the fallback error value, so CI gives no signal about whether the feature it tests is actually working. When a test is loosened to allow multiple acceptable values, assert the absence of ALL non-acceptable values, or control the test environment tightly enough to assert a single specific value. |
| 2026-04-09 | #738 | Spec assertion softening (>= N instead of == N) masks cycle-count regressions — prefer exact assertions with a documented tolerance comment only when the tolerance is a deliberate design choice, not an implementation quirk |
| 2026-04-09 | #738 | Direct SQL field injection in tests signals an unsealed public API seam — when a field must be set via SQL because the upsert API doesn't accept it yet, document the seam explicitly so a future API addition is not a surprise |
| 2026-04-22 | #827 | A test that asserts now-prohibited behavior is a stale assertion: its failure confirms the fix is correct, but merging with it intact inherits a known CI failure. When a PR removes a behavior, delete or invert any test that asserts the removed behavior before merge |
| 2026-04-22 | #828 | Exception-narrowing fixes can silently remove positive assertions in the same edit. When a `try` block catches a broad exception and also contains a positive assertion (e.g., attribute check) in the happy path, narrowing the exception handling is a separate concern from the assertion. The minimal fix removes or narrows the `except` clause only; it does not remove happy-path assertions. Detection: when a PR modifies a `try/except` block in a test, verify that positive assertions in the `try` body are preserved unless there is an explicit stated reason to remove them. |
| 2026-04-20 | #800 | Locally-declared mirror of a production constant (e.g., `_MILLIS_THRESHOLD = 1e11` in test file mirroring `_MILLISECOND_TIMESTAMP_THRESHOLD = 1e11` in steward.py) creates a divergence risk: when the production constant changes, the test constant does not, causing the test to silently validate against a stale value. This is the same pattern named in PR #696 but recurring in a different form — the test uses the local constant only for branching in an assertion, so the test does not fail, but its branching logic silently uses the wrong threshold. Import production constants from the module under test rather than re-declaring them. |
| 2026-04-22 | #827 | A test that asserts behavior an oracle decision explicitly prohibits becomes a stale assertion after the fix lands — the test's failure is confirmation the fix is correct, not a regression. When a PR removes or gates a code path (e.g., age-only auto-advancement) that an existing test exercises, that test must be updated or deleted before merge. Merging with the stale test intact inherits a known CI failure that future PRs must suppress or explain. Detection: when a PR's stated purpose is to remove or constrain a behavior, grep the test suite for that behavior's name or description; any test whose name or assertion matches the prohibited behavior is a candidate stale test. Resolution: (a) delete the test if the prohibited behavior has no inverse-assertion value; (b) invert the assertion to verify the behavior is now blocked (this converts the stale test into a regression guard). |
| 2026-04-22 | #848 | Inline test copy of a production function (required because the module cannot be imported in CI due to a transitive dependency) is a synchronization debt with no tooling enforcement. The test copy verifies the function's behavior at merge time, but future changes to the production function require a deliberate two-file update that CI does not catch. Detection: when a test file contains a comment noting it is a "faithful copy" of a production function, flag it at oracle review and verify the copy matches production before APPROVED. The "interface migration without updating all callers" pattern (learnings.md PR #720) applies as a risk class — the test copy is effectively an unchecked caller of the production function's implicit contract. |
| 2026-04-22 | #848 | Dead code immediately before an unconditional override produces misleading intent signals. `heartbeat_ttl = max(300, min(timeout_seconds // 4, 300))` computed then overwritten by `heartbeat_ttl = 300` on the next line — the first assignment is unreachable. The comment preceding it describes per-UoW TTL tuning, amplifying the confusion. Detection: when a PR assigns a variable and the next line unconditionally reassigns it without conditional branching, the first assignment is dead. Remove it or convert it to a comment documenting the deferred design intent. |


### Timezone Handling

| Date | PR | Learning |
|------|----|----------|
| 2026-04-15 | #753 | Hardcoded UTC offset `timezone(timedelta(hours=-4))` drifts during EST (November through mid-March) — use `zoneinfo.ZoneInfo("America/New_York")` instead |
| 2026-04-20 | #798 | `datetime.fromtimestamp(ts, tz=timezone.utc)` interprets `ts` as Unix *seconds*; Node.js credential stores commonly write Unix *milliseconds*. A milliseconds value parsed as seconds produces a year ~57000 — no warning fires for an actually-expired token. Tests using `datetime.now().timestamp()` (seconds) do not catch this mismatch. When parsing an externally-sourced Unix timestamp, verify the epoch scale from the producing system's documentation before choosing `fromtimestamp`. |
| 2026-04-20 | #800 | `data.get("expires_in", 0)` default of 0 in an OAuth refresh response handler causes the written `expiresAt` to equal the current timestamp — immediately expired. On the next call, the expiry check returns EXPIRED again, triggering another refresh attempt. This creates a refresh loop that resolves only when the endpoint returns a valid `expires_in` or the refresh fails entirely. Use a non-zero default (e.g., the expected token lifetime) or validate `expires_in > 0` before writing. Detection: whenever an OAuth token response is merged into stored credentials, check whether the `expires_in` default produces an immediately-expired value. |

### CLI Query Semantics

| Date | PR | Learning |
|------|----|----------|
| 2026-04-20 | #804 | `gh issue list --label bug,hygiene` uses AND semantics — only issues with BOTH labels are returned. For OR semantics across labels, use two separate `--label` flags or a `--search` query with OR logic. When a metric's core signal depends on a gh CLI query with multi-label filtering, verify AND vs OR semantics by running the query against the live repo before finalizing the instruction. A query that silently undercounts will produce false ENTROPY ACCUMULATION escalations when resolution rate stays near zero. |
| 2026-04-20 | #804 | Bash inline `python3 -c "..."` in a scheduled task instruction document violates the `uv` convention (CLAUDE.md: "Always use uv instead of bare python3"). The failure mode is silent: if python3 is not in PATH (because uv manages the Python installation), the inline script produces empty output, a downstream variable is empty, and the conditional that depends on it is silently skipped. The `|| echo 0` fallback pattern is insufficient if it is only applied to one half of a two-variable comparison — verify both variables have fallbacks. |
| 2026-04-22 | #821 | Bare `python3` in bash scripts recurred — `emit_summary` in `usage-report.sh` uses `python3 - args <<'PYEOF'`. This is the second instance of the same pattern (first: PR #804). The pattern now spans shell scripts that embed inline Python, not just instruction documents. Detection: any new or modified `scripts/*.sh` file that embeds Python must use `uv run python -` or `uv run -` at the invocation site. Apply at oracle review whenever a bash script contains `python3` (bare) — do not limit the check to instruction documents. |
| 2026-04-22 | #830 | Bare `python3` in instruction-layer markdown recurred — nightly-consolidation.md step 1c embeds `python3 -c "..."` in a bash block. This is the third instance of the pattern (first: PR #804 instruction document; second: PR #821 shell script; third: PR #830 markdown instruction file). The failure mode is highest-consequence when the bare invocation is in the feedback-loop reader: if the script silently produces empty output, the downstream variable is empty and the proprioceptive bullet is never written — appearing to work while generating no signal. Detection scope now extends to ALL contexts where Python is invoked inline in any Lobster-managed text: instruction .md files, .sh scripts, and task files. |

### Document Review

| Date | Context | Learning |
|------|---------|----------|
| 2026-04-20 | PR #804 | Moving an instruction document from a runtime path to a versioned repo path requires: (a) a migration step removing or tombstoning the old runtime copy so it is not treated as authoritative, (b) a migration step updating any state files that the document's new behavior requires (e.g., adding new fields to a rotation-state.json), and (c) verifying that the new versioned copy does not perpetuate stale paths from the old runtime copy. Failure to migrate (a) leaves the old version authoritative for pre-upgrade instances. Failure to migrate (b) causes immediate false signals on first run (e.g., absent `cycle_start_timestamp` causes vision drift to fire unconditionally). Failure to check (c) ships the wrong path in the canonically correct file. |
| 2026-04-21 | PR #804 re-oracle | Migration path mismatch produces silent no-op: a migration added to fix Gap 3 (absent `cycle_start_timestamp`) targeted `${LOBSTER_WORKSPACE}/data/rotation-state.json` but the actual runtime file is at `${LOBSTER_WORKSPACE}/hygiene/rotation-state.json`. The `[ -f "$ROTATION_STATE" ]` guard evaluated false (file not found at the wrong path), the migration silently no-oped, `migrated` did not increment, and the original gap remained unclosed. Detection pattern: after writing a migration that targets a specific file path, verify the path resolves to an existing file on the canonical instance before committing. For state files, the canonical location is determined by the runtime code that reads them — trace the reader, not the writer's assumption about where the file should be. |
| 2026-04-22 | PR #804 post-merge | A versioned canonical template (`memory/canonical-templates/sweep-context.md`) can embed a stale runtime path from its origin document even after review and approval. | 2026-04-21 | PR #811 | Runtime instruction edit without updating canonical-templates counterpart creates installation-class divergence — when a runtime instruction file has a versioned canonical counterpart in `memory/canonical-templates/`, both must be updated in the same PR. Edit to runtime copy only means new installs seeded from canonical will not have the change. |
| 2026-04-22 | PR #804 post-merge re-oracle | An APPROVED merged PR with correct migrations does not self-apply: upgrade.sh must be run on the live instance for the migrations to take effect. | 2026-04-20 | PR #804 | When a new versioned copy of an instruction document is created from a runtime original, search the new copy for all path references and verify each against the current canonical paths — do not assume the runtime copy is current. The runtime copy may have accumulated legacy paths (e.g., `~/lobster-workspace/oracle/learnings.md`) that were corrected in the repo but not backported to the runtime copy. |
| 2026-04-20 | PR #796 | Path fix completeness: when fixing a path inconsistency, enumerate all path-reference forms (wrong prefix, bare-relative, partial) — not just the form observed in the failing case. A single grep for the observed wrong path misses other wrong forms in the same files. |
| 2026-04-20 | PR #799 | Sequential fix-sweep PR conflict: two PRs independently fixing overlapping subsets of the same path inconsistency from the same branch point produce a merge conflict when the first merges before the second is rebased. The second PR's diff base is stale; the locations it targets are already changed. Detection: before opening a follow-on sweep PR, verify whether a sibling PR has already merged to main and touched the same files. `gh pr view --json mergeable,mergeStateStatus` is the mechanical check. |
| 2026-04-09 | PR #720 R4 | Ad-hoc caller enumeration produces per-round discovery in typed interface migrations: four oracle rounds were needed because each round found a new missed file (R2: steward-heartbeat + steward_cycle_trace, R3: register_aware_diagnosis, R4: startup-sweep + register_mismatch, R5: execution_gate). The correct mitigation is a one-time repo-wide grep for the function name under migration before the first round, not after each rejection. Detection rule: when a PR changes a public function's return type, the oracle's first action before reading the diff is `grep -r 'function_name' tests/` to enumerate all files that mock or return values of the changed type. Any file returning a plain dict for a function now returning a typed dataclass is a missed caller. The engineer's stated test count ("87 passed across 3 files") is not evidence of completeness when the total test file count exceeds 3. |
| 2026-04-08 | PR #720 | Interface migration without updating all callers: `run_steward_cycle` return type changed from `dict[str, Any]` to `CycleResult` frozen dataclass, but `scheduled-tasks/steward-heartbeat.py` was not in the PR diff and still uses `result["prescribed"]` dict subscript access. The `CycleResult` dataclass provides an `as_dict()` backward-compat method, but the heartbeat does not call it. Result: every production steward cycle raises `TypeError: 'CycleResult' object is not subscriptable`. Detection rule: when a PR changes a public function's return type from dict to a typed class, enumerate all production callers (not just test files) and verify each uses attribute access, not subscript access. Test files may pass because they mock the function; the mock return value (still a dict) satisfies the test's assertions without exercising the real return type. | 2026-04-08 | PR #720 | Test file not updated when interface changes: `tests/unit/test_orchestration/test_steward_cycle_trace.py` was not in the PR diff. Its `_mock_github_client_open` still returns a plain dict. After the PR, `_process_uow` calls `issue_info.body` on that dict, raising `AttributeError`. This is the sister pattern to the production-caller defect: interface migrations in the module under test must enumerate all test helpers in all test files that mock or return values of the changed type — not just the test file in the PR diff. |
| 2026-04-08 | PR #719 | Wrong fallback constant in cleanup arc helper function: `_archive_uow_artifacts` uses `_DEFAULT_CYCLE_TRACE_DIR` as the None-guard fallback for `artifact_dir`, but the correct fallback should be an artifact directory constant. The production call chain always passes an explicit `artifact_dir`, so the wrong fallback never fires. Advisory pattern: when adding a fallback constant to a function that handles a specific directory type, verify the fallback is the same type — not a sibling constant from a related but distinct subsystem. The wrong fallback is invisible at code review if the production call path always provides a value. |
| 2026-04-08 | PR #719 | Best-effort arc composition for multi-step cleanup: when a cleanup arc has one mandatory terminal action (state transition + notification), each pre-terminal step should independently catch and log exceptions without blocking subsequent steps. This allows the arc to reach its terminal action even when intermediate steps fail. The pattern is structurally correct when partial cleanup is better than no cleanup — do not add inter-step dependencies unless a downstream step genuinely requires upstream output to be present. PR #719 (`_run_hard_cap_cleanup`) applies this correctly: archival failure does not block trace write; either failure does not block registry close_reason update; all failures do not block the notify_dan surface call. |

No TypeError; the bug is silent. Detection: when a public wrapper method's signature is extended, trace every new parameter through the delegation call to the private method. ### [2026-04-08] Pattern: `>=` on cumulative counter fires notification on every post-threshold cycle, not once at crossing

When a threshold predicate is changed from `==` to `>=` on a counter that increments every cycle (e.g., `lifetime_cycles + new_cycles`), the notification guarded by that predicate fires on every cycle from the threshold onward — not just at the crossing. Detection: when reviewing a threshold predicate change from `==` to `>=` on a monotonically increasing counter, count how many cycles will satisfy the new predicate before the next gate takes over, and verify that the notification hook can be called that many times without unintended duplication. ### [2026-04-01] Pattern: synthetic test coverage as reliability proxy — false confidence under unknown failure surface

When a reliability loop injects known-good synthetic workloads and measures pass rate as the primary reliability signal, the signal is only valid to the extent that the synthetic workloads exercise the actual failure surface. In the RALPH design, the proposed UoW types (simple-doc-write, multi-step-search, expected-idempotent) are chosen for testability and short completion time — not because they exercise the named failure modes in this codebase (timing races between sweep cadence and subprocess runtime, fire-and-forget dispatch with synchronous outcome claims, deduplication key volatility). The false confidence produced by this pattern is structurally worse than no test signal, because it substitutes a passing signal for the absence of observation. ### [2026-04-01] Pattern: deduplication key includes volatile data — issue spam under oscillation

When a deduplication check uses a string key that includes a measured value (e.g., actual line count), and the measured value changes between runs, the key will not match an existing open item even if the same underlying condition is still present. In file-size-monitor.py (PR #551), the GitHub issue title includes the actual line count: `"warn: {rel_path} exceeds {threshold}-line threshold ({actual} lines)"`. If a file is at 305 lines one week and 312 the next, a new issue is filed even if the prior one is still open — because the title differs. The correct deduplication key is the stable condition, not the current measurement. Applied specifically: the title used for dedup should be `"warn: {rel_path} exceeds {threshold}-line threshold"` (no count); the count belongs in the issue body only. This generalizes: any deduplication mechanism that constructs its key from a volatile measurement will produce duplicates under normal variation. The key should encode the condition type and threshold, not the snapshot value.

### [2026-04-01] Pattern: startup sweep with no minimum age threshold for active UoWs — timing race with long-running subprocesses

When a sweep component classifies `active` UoWs on every heartbeat invocation without a minimum age threshold, it will interrupt any executor whose subprocess runtime exceeds the heartbeat interval. The `_EXECUTOR_ORPHAN_THRESHOLD_SECONDS` pattern (apply a minimum age threshold before classifying a stuck UoW) was applied correctly to Population 2 (`ready-for-executor`) but was absent from Population 1 (`active` UoWs) in startup-sweep.py. Per-UoW accuracy: use `started_at + estimated_runtime` as the sweep threshold rather than a global constant. ### [2026-04-01] Pattern: subprocess argv as credential exposure vector when replacing authenticated sessions

When a Python script that previously used an authenticated library session (e.g., python-telegram-bot) is refactored to call a CLI tool (e.g., curl) via subprocess with credentials embedded in arguments, the token becomes visible in `/proc/<pid>/cmdline` and `ps aux` for the subprocess lifetime. The original library session keeps credentials in process memory, not in argv. This risk is non-obvious because `subprocess.run` with a list (not `shell=True`) correctly handles strings with spaces — the ergonomic correctness of the call masks the security regression. Watch for this whenever a direct API call via subprocess replaces a managed library session: the managed session contains credentials in the authenticating process's memory; the subprocess approach externalizes them to OS-visible process state. The canonical fix: use `urllib.request` or a requests-compatible library directly in Python rather than spawning a subprocess for HTTP calls.

### [2026-04-01] Pattern: outbox-mediated delivery vs. direct API call — structural buffering tradeoff

The outbox-mediated delivery pattern (write JSON to ~/messages/outbox/, bot picks up asynchronously) provides retry semantics, credential containment, error fallback, and a single credential-holding process. Direct API calls from report scripts are synchronous and give immediate caller feedback, but lose all of these properties. When evaluating a proposal to replace outbox delivery with direct API calls, the question is not "does the direct call work?" but "what does the outbox's buffering provide that the caller is giving up?" In this system the outbox handler has a retry counter (`_outbox_fail_counts`) across sweep cycles; the direct call has a single attempt. The ergonomic gain (synchronous confirmation) is real, but the structural properties lost are real too. The correct resolution is to use a Python HTTP client in-process rather than curl, which restores credential containment while preserving synchrony.

### [2026-03-30] Pattern: dispatcher-synchrony assumption as undocumented arc-breaking premise

The Steward's `_assess_completion` has a secondary correctness gate that checks output_ref content, which compensates — but the result.json outcome field is semantically misleading, and any audit query that reads outcome from result.json without also checking output_ref will produce false positives. Watch for monitoring or dashboard tooling that reads outcome from result.json at face value. The canonical form: executor.py `_run_execution` line 442 dispatches via inbox, line 450 writes `outcome=COMPLETE`. The Steward's secondary gate is `_assess_completion` in steward.py.

### [2026-03-27] Pattern: table-as-compaction-resistant encoding

When a behavioral rule must survive context compaction, encode it as a table row with three columns: trigger (one sentence), enforcement (structural or advisory), and verifiable difference. The Tier-1 Gate Register and Oracle Pattern Register use this format. ### [2026-03-27] Pattern: crash-safe two-step delivery

The correct delivery pattern for user-facing subagents is: (1) call \`send_reply(chat_id, text, task_id=...)\` first to deliver directly to the user, then (2) call \`write_result(sent_reply_to_user=True)\` to signal the dispatcher. This order is crash-safe: if the subagent crashes between the two calls, the user already has the reply. The reverse order (write_result first, then send_reply) is crash-unsafe — a crash after write_result but before send_reply leaves the user with no reply and the dispatcher believing one was sent. The \`task_id\` parameter in \`send_reply\` allows the server to auto-suppress duplicate delivery if \`write_result\` is later called with the same task_id. Each user-facing subagent rediscovers this pattern; the canonical name "crash-safe two-step delivery" allows agent definitions to reference it symbolically rather than re-explaining the rationale.

### [2026-03-27] Pattern: internal vs. user-facing subagent task dichotomy

Subagent tasks fall into two modes that require different delivery behavior. User-facing tasks: the subagent knows the user's chat_id, calls \`send_reply\` directly, then calls \`write_result(sent_reply_to_user=True)\`. Internal tasks: the subagent has no user-facing result to deliver (e.g., scheduled jobs, internal maintenance), calls only \`write_result\` with the job output. The dichotomy is structural: user-facing tasks require crash-safe two-step delivery; internal tasks require only \`write_result\`. Conflating the two modes (calling \`send_reply\` from an internal task, or omitting \`send_reply\` from a user-facing task) produces either double-delivery or silent drops. The signal for internal mode: \`chat_id\` was not provided in the task prompt, or the task is a scheduled job.

### [2026-03-27] Pattern: silent-drop sentinel via chat_id=0

\`chat_id=0\` is a structured no-op signal in the subagent result protocol. When \`write_result\` is called with \`chat_id=0\`, the dispatcher drops the result silently without attempting delivery. This is the correct pattern for internal subagent results (scheduled jobs, system maintenance, internal observers) that produce output for logging but have no user to notify. The pattern extends to system message types (\`agent_failed\`, \`cron_reminder\`) that arrive with \`chat_id: 0\` or \`source: "system"\` — the dispatcher reads and acts on these without calling \`send_reply\`. Any future protocol extension that needs a no-op sentinel should use \`chat_id=0\` rather than inventing a new signal. Future agents implementing internal-only behavior should document their use of this sentinel explicitly.

### [2026-03-27] Pattern: CLI-subprocess as integration path

The \`registry_cli.py\` pattern (and similar CLI-wrapper scripts) keeps the dispatcher's surface clean by routing structured operations through a subprocess rather than inline tool calls. Benefits: (1) the CLI is independently testable without a running dispatcher, (2) the dispatcher invocation is a single shell command rather than a sequence of tool calls, (3) the CLI can be versioned and replaced without changing dispatcher behavior. This pattern is the correct integration path when a capability needs: structured I/O, independent testability, and a clean boundary between the dispatcher's stateless routing role and the capability's stateful operations. Watch for cases where inline tool-call sequences in dispatcher prompts grow complex — these are candidates for extraction into a CLI subprocess.

### [2026-03-30] Pattern: deferred open decisions as Phase N+1 blocking dependencies

When a design document defers a foundational decision (e.g., "workflow artifact format: deterministic script vs. LLM prompt") to the next phase without specifying a resolution mechanism, that deferred decision becomes a blocking dependency that surfaces as a stall at the start of Phase N+1 rather than a decision point that can be prepared for. The correct encoding: mark deferred decisions as pre-phase gates with (1) a resolution owner, (2) a cheapest-test path, and (3) a fallback default if the gate is not cleared.

### [2026-03-30] Pattern: advisory self-detection of orientation distortion is structurally insufficient

A Steward-class agent that must detect its own orientation distortion as a condition for surfacing to a human cannot reliably satisfy this condition — orientation distortion is by definition not visible from inside the distorted orientation. ### [2026-03-30] Pattern: inter-component contract introduced without upstream documentation

When a component introduces a new file protocol or structured data requirement at the seam with an adjacent component that is not yet built, the contract exists in the implementation but not in the design document or the adjacent component's spec. ### [2026-03-30] Pattern: surface-to-Dan message body as the actionability bottleneck

A system that surfaces a UoW to Dan but omits the steward_log and steward_agenda from the message body creates an actionability bottleneck: Dan receives a notification that a UoW is stuck but has no context with which to act. ### [2026-03-30] Pattern: backward-compat field retention masking primary routing field absence

When a contract defines Field A as the primary routing signal and Field B as a "backward-compat convenience field" derived from Field A, implementations that read Field B and skip Field A satisfy the backward-compat requirement while violating the contract's stated routing intent. The executor-contract.md defines `outcome` as the primary routing field and `success: bool` as the backward-compat field. steward.py reads only `success` — which means all four outcome values (`complete`, `partial`, `failed`, `blocked`) are collapsed to a boolean, and the distinct handling paths required by the Steward Interpretation Table are never executed. The canonical detection test: when a contract says "X is the primary routing field; Y is backward-compat," check whether X is read anywhere in the implementation. If only Y is read, the implementation is backward-compat-compliant but contract-non-compliant. The failure mode: `blocked` outcomes (which the contract says "always route to Dan") are silently re-prescribed; `partial` outcomes (which should pass `steps_completed`/`steps_total` to re-diagnosis) receive generic re-prescription indistinguishable from `failed`.

### [2026-03-30] Pattern: contract uow_id cross-validation as required pre-read step

When a contract specifies "validate field X before reading any other field," that validation is a mandatory pre-condition, not a suggestion. The executor-contract.md specifies that the Steward must validate `uow_id` in the result file matches the UoW record's `id` before reading any other field. steward.py reads `success` without this check. A result file written by a crashed Executor for UoW-A but stored at UoW-B's output_ref path would be silently accepted. The correct pattern: any structured file protocol that carries an identity field (`uow_id`, `job_id`, `session_id`) must validate that field before parsing any content field. The validation is structural prevention against misrouted artifacts — it is not redundant even when the path convention is deterministic, because crashes can produce unexpected file states.

### [2026-03-30] Pattern: state machine and result contract inconsistency at the Executor exit transition

When a design doc's state machine defines `active` → `failed` as a valid Executor-triggered transition, but the Executor result contract routes all Executor exits (including failures) through `ready-for-steward` via a result file, the two artifacts describe different execution models at the same transition point. The wos-v2-design.md Transitions table lists `active` → `failed` (Actor: Executor) while executor-contract.md routes all `failed` outcomes to `ready-for-steward` (Executor writes result file, Steward reads and decides). These are not equivalent: the state machine model gives the Executor authority to declare failure and transition to a terminal state; the contract model gives the Executor signal responsibility only and keeps the transition decision with the Steward. When a design doc's state machine and its result contract diverge at the same transition, the contract is the authoritative model (it was written later, with more precision) and the state machine needs updating. Watch for design docs that have both a state machine and a result contract — keep them synchronized at every transition.

### [2026-03-30] Pattern: StrEnum redefinition across module boundary creates silent type divergence

When a component imports only part of an adjacent module's type vocabulary (e.g., imports `UoW` but not `UoWStatus` from registry.py) and then redefines the missing type locally with additional values, two parallel StrEnum types exist in the codebase. Watch for any module that imports a type from X and also defines a class with the same name — the redefinition is almost always wrong.

### [2026-03-30] Pattern: result.json not written on pre-execution failure path breaks contract

When an Executor's claim sequence commits the atomic transaction (status='active', output_ref written) and then fails during a post-commit validation step (e.g., NULL workflow_artifact, deserialization failure), the UoW transitions to `failed` via `_fail_uow` but no result.json is written at output_ref. The Steward's `_assess_completion` requires result.json to distinguish a planned stop from an orphan. Without it, the Steward loops on this UoW through the startup sweep path until the hard cap fires — treating a controlled failure as a series of crashes. The contract is unambiguous: "every intentional exit — complete, partial, failed, or blocked — must produce a result file." Pre-execution failures (workflow_artifact absent, malformed) are intentional exits, not crashes. The output_ref path is always computable from the UoW ID before the result file needs to be written. Detection: check every path that calls `_fail_uow` without a preceding `_write_result_json` call. Each such path is a contract violation.

### [2026-03-30] Pattern: outcome-based branching required but collapsed to binary in Steward routing

When a contract defines four distinct outcome values (`complete`, `partial`, `failed`, `blocked`) with distinct Steward handling for each, implementations that branch on `complete`/`blocked` correctly but collapse `partial` and `failed` into identical handling satisfy the binary (done/not-done) contract requirement while violating the four-outcome routing contract. The Steward Interpretation Table specifies: `partial` → re-diagnose with progress fraction; `failed` → re-diagnose with reason severity to choose retry vs. surface. Collapsing both to a generic "not complete, re-prescribe" path silently ignores the progress information carried by `partial` outcomes. Detection: when a result contract defines N outcome values (N > 2), each distinct outcome requires a distinct routing branch at the evaluation layer. Binary handling of a multi-outcome contract is a structural gap, not a conservative default.


### [2026-03-30] Pattern: autouse fixture pre-creation as silent mkdir collision source

When a `conftest.py` autouse fixture pre-creates directory trees (e.g., `tmp_path/workspace/`, `tmp_path/messages/`) for every test, older tests that predate the fixture and call `mkdir()` without `exist_ok=True` will collide silently. The pattern recurs whenever a shared autouse fixture adds directory pre-creation for a new feature: all existing tests that touch those paths inherit the collision risk. ### [2026-03-30] Pattern: same-table input/output as silent feedback loop precondition

When a pipeline component writes its outputs (e.g., `pattern_observation` events) into the same table that feeds its inputs, the read query must explicitly exclude the component's own output types — or the pipeline enters a self-reinforcing loop where outputs become inputs on the next cycle. The slow_reclassifier PR #66 is the canonical form: `write_pattern_event()` writes `type='pattern_observation'` rows into `events`; `read_recent_events()` previously fetched all `quick-v1`-tagged events without filtering by type; the quick_classifier tagged `pattern_observation` rows as `meta_reflection`; the slow_reclassifier found 2+ `meta_reflection` events and wrote more `pattern_observation` rows. The fix (WHERE `e.type != 'pattern_observation'`) is correct and minimal. The structural alternative — a separate output table — prevents the feedback loop by design rather than by filter. Watch for any pipeline where a component's write path and read path touch the same table: the WHERE-clause fix is always available and often correct at current scale, but it creates an implicit contract between the writer (must tag outputs with a distinguishable type) and the reader (must filter that type). If the writer changes its output type naming, the filter silently stops working. Document the contract at both ends.

### [2026-03-30] Pattern: importlib bootstrap skips create_module step — safe in CPython, not spec-guaranteed

When using `importlib.util.spec_from_file_location` + `module_from_spec` + `exec_module` to load a Python source file at runtime (e.g., for re-export bridging), the canonical three-step sequence per Python's importlib documentation is: (1) `spec = spec_from_file_location(...)`, (2) `mod = module_from_spec(spec)`, (3) `spec.loader.create_module(mod)` if `create_module` returns non-None, then `spec.loader.exec_module(mod)`. In practice, `create_module` returns `None` for file-based source loaders in CPython, making the skip safe. Detection: when reviewing importlib-based dynamic loading, check whether `create_module` is called before `exec_module`. ### [2026-03-30] Pattern: getattr as pre-use of an undeclared dataclass field

When a dataclass field does not exist yet but code pre-uses it via `getattr(obj, "field_name", None)` for forward-compatibility, the field is in use before it is declared in the canonical type definition. ### [2026-03-30] Pattern: pre-increment UoW object passed to post-prescription callable reads stale state

### [2026-03-30] Pattern: infrastructure-before-callers as deferred-value design

### [2026-03-31] Pattern: duplicate constant as implicit two-site maintenance contract

When the same path-derived constant is defined independently in two modules (e.g., `_GATE_CLEARED_FLAG` in both `steward.py` and `dispatcher_handlers.py`), each definition resolves via the same env-var logic and is functionally identical at time of writing. Detection: when reviewing a PR that introduces a path constant derived from `os.environ.get(...)`, grep the full codebase for the same env-var key. ### [2026-03-31] Pattern: one-way gate without re-lock path

A gate that has a programmatic clearing mechanism but no programmatic re-lock mechanism is a one-way ratchet. Once cleared, the only recovery path is manual state manipulation (deleting the flag file). For operational gates — those that an operator may need to reinstate during pipeline quality issues — a re-lock mechanism (e.g., `/wos relock` command) should ship alongside the unblock mechanism in the same PR. The asymmetry is invisible at PR review time because the unblock story is complete: the PR does what it claims to do. The re-lock gap becomes visible only when a post-unblock problem makes the gate's absence costly. Canonical instance: PR #391 ships `/wos unblock` (creates file flag) with no `/wos relock` (deletes file flag). If post-unblock processing reveals a pipeline quality problem, the gate cannot be reinstated without manual file deletion. ### [2026-03-31] Pattern: implicit rate-limiting via age gate removed without replacement

Lowering the gate to 3 days (and adding immediate-qualify labels) moved the intake rate from pipeline-speed-bound to issue-creation-speed-bound. When a codebase uses a migration runner (numbered `.sql` files, version tracking table), `schema.sql` becomes documentation, not DDL. ### [2026-03-31] Pattern: artifact-path vs. result-path conflation at the dispatch seam

In WOS, `output_ref` names the artifact path (`{uow_id}.json`) and `result.json` is a separate derived path (`{uow_id}.result.json` — same stem, different extension via `_result_json_path`). These are two different files with different semantics: `output_ref` is where the subagent writes execution output; `result.json` is where the execution outcome metadata (success, outcome, reason) is written. ### [2026-03-31] Pattern: optional identity field creates contract-enforcement gap at misrouting boundary

When a result file contract defines an identity field (`uow_id`) as required and the Steward validates it via `if field is not None and field != expected`, files written without the identity field (where `get("uow_id")` returns `None`) silently pass validation. ### [2026-03-31] Pattern: reminder-injection as advisory self-correction — weaker than audit tooling in drift recovery

When a skill addresses behavioral drift by re-injecting the canonical specification (the correct behavior), its effectiveness depends on whether the component receiving the injection can detect and correct its own deviation. ### [2026-04-08] Pattern: LIKE query on JSON column as audit-history probe — safe pattern with known constraints

When a query needs to determine whether a specific event occurred in the past, a `LIKE '%"key": "value"%'` query on a JSON audit_log note column is a valid lightweight probe under these conditions: (1) the JSON is produced by `json.dumps` with consistent default separators (`, ` and `: `), so the string format is deterministic; (2) the query is read-only and the result is used as a boolean gate (present/absent), not as a join key or aggregate; (3) the error-return default is safe (False = treat as fresh = allow dispatch). The correct structural fix in that case is a stored boolean flag or a normalized audit_log schema, not adjusting the LIKE pattern. Applied in PR #667 (`has_executor_orphan_history`): `json.dumps` with default separators produces `"classification": "executor_orphan"` which the pattern `%"classification": "executor_orphan"%` correctly matches. Detection: when reviewing a new LIKE query on JSON, verify the JSON serializer's separator convention at both the writer (production code) and the test helper. If they differ, the test will pass but production will silently misidentify.

### [2026-04-20] PR #798 — Unix timestamp epoch scale ambiguity in external credential stores

When parsing a Unix timestamp from an externally-produced credential store, `datetime.fromtimestamp(ts, tz=timezone.utc)` interprets `ts` as Unix *seconds*. ## WOS Failure Triage Protocol (2026-04-22)

When a WOS pipeline shows high failure rates, use this protocol before assuming the worst:

1. **Sample broadly** — `grep -rl '"outcome": "failed"' outputs/uow_*.result.json | shuf | head -20` avoids recency bias
2. **Extract reason fields** — read each result's `reason` or error text; categorize by distinct pattern
3. **Count distribution** — grep across all failures for each category to get true proportions
4. **Cross-reference traces** — `.trace.json` companions have full stack context for ambiguous cases
5. **Check environment** — `which claude`, env vars, cron context; failures are often environmental, not logic bugs
6. **Check logs** — steward.log, executor-heartbeat.log often show the same errors in aggregate

Applied: 295 "failed" UoWs decomposed into test contamination (154, env bug), cron auth missing CLAUDE_CODE_OAUTH_TOKEN (140, fixable), SIGTERM (1, normal). True production failure rate: ~21%, not 44%.

**Key insight**: High aggregate failure rates in distributed pipelines are often multi-cause. Sampling before fixing prevents chasing the wrong root cause.

### [2026-04-22] Pattern: instruction-accumulation as self-defeating behavioral rule remediation (premise-review pr-20260323190000)

**Context:** Across three separate instances (PR #8, PR #7, cycle-spec), the system's response to "a behavioral rule is not reliably followed" was consistently "add a stronger or parallel behavioral rule." The premise-review observation (pr-20260323190000) named compaction, context growth, and dispatcher pressure as candidate root causes but did not commit to one.

**Root cause diagnosed:** Context growth causes instruction absorption — earlier instructions are displaced when a document grows long enough that the model's attention distributes across too many competing instructions. The failure mode is self-reinforcing: when a rule is not followed, the remediation adds another rule, making the document longer, increasing absorption probability on the next cycle. CLAUDE.md and sys.dispatcher.bootup.md together present 1,400+ lines of instruction before any user message is encountered. The Tier-1 Gate Register competes for attention with startup procedures, session management, delegation patterns, and all message handlers.

**Why not compaction or dispatcher pressure:** Compaction is episodic; the pattern appears across normal operation cycles, not only post-compaction. Dispatcher pressure (7-second rule) is architectural and correct — the dispatcher is supposed to skip expensive inline work. Neither explains why rules fail under normal context-window conditions with no recent compaction. Context growth is a continuous, steady-state degradation that gets worse with each remediation round.

**Proposed intervention:** Externalize the Tier-1 Gate Register to a standalone file (e.g., `~/lobster/.claude/gate-register.md`) and load it via a per-message hook rather than embedding it in CLAUDE.md. The gate register is removed from the always-on accumulation pool; its instructions compete only against each other, not against 1,400+ lines of other material. The hook mechanism is already operational (on-fresh-start.py, on-message.py pattern); no new infrastructure is required. The intervention directly targets the structural cause: reduce the context-competition surface for the most critical behavioral rules.

**Generalized pattern:** When a behavioral rule is unreliable, ask whether the document hosting it is long enough that the rule's position in the document creates an absorption risk before adding a stronger version of the rule. Compression or externalization of the document is often the correct intervention. Additions at the bottom of a long document are the lowest-attention-probability location in the context window.
