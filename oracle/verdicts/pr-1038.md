### Gap 2 verification — multi-fire notifications (blocking)

**Fix:** `emit_inbox_notification()` now loads per-PR state from `wos-pr-sweeper-state.json`, filters PRs via `_should_notify()`, and atomically writes updated state after notification.

**`_should_notify()` logic:** Returns `True` if no prior entry exists or if `(now - last_notified) >= timedelta(hours=24)`. The comparison is `>=` on elapsed time — it fires at the crossing epoch and on subsequent runs if still stale AND the 24h window has fully elapsed. This is correct for a re-notification policy (not a one-time notification): a PR that remains stale for 48 hours correctly receives two notifications (at T=0 and T=24h). This is the intended behavior per the PR description.

**Atomic state write:** `_save_state()` uses tmp-then-rename (`tmp = path.with_suffix(".json.tmp"); tmp.write_text(...); tmp.replace(path)`). Correct pattern.

**Cooldown in dry-run:** Dry-run path prints the would-be state update count but does not write. Correct — dry-run should not modify state.

**Edge case — corrupt state file:** `_load_state()` catches all exceptions on parse and returns `{}`, which means all PRs will re-notify after a corrupt state file. This is correct (fail open for notification — better to re-notify than to silently drop). Gap 2 is closed.
