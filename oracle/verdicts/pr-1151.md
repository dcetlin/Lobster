## Stage 2: Quality Review

### What the PR changes

Three changes across two production files and one test file:

**1. `registry.py` — two bare import fixes**

`_upsert_typed` (line 663): `from orchestration.routing_classifier import classify_posture` → `from .routing_classifier import classify_posture`

`write_gate_fired` (line 1928): `from orchestration.gate_fired import _GATE_SEVERITY` → `from .gate_fired import _GATE_SEVERITY`

Both are lazy imports inside try blocks. Both convert bare absolute form to relative form. The fix is minimal and correct.

**2. `dispatcher_handlers.py` — HTML drilldown feature addition**

`handle_wos_done` gains a non-fatal HTML detail URL generation step. The function now:
- Extracts `uow_id` from the message
- If `uow_id` is present, imports `from .wos_uow_detail_gen import generate_and_upload` lazily and appends the returned URL to `text`
- Catches all exceptions and falls back to the original text unchanged

The relative import is correct. The non-fatal pattern (catch-all except with debug log) is appropriate for a supplementary UX feature. The function docstring is updated to reflect the new behavior.

**3. `test_wos_done_notifier.py` — three test changes**

a. `test_returns_text_from_message`: assertion softened from `== expected_text` to `startswith(expected_text)`. Comment added explaining the fallback behavior. The softening is intentional — `_make_msg()` now includes `uow_id=_SAMPLE_UOW_ID` by default, so the handler will attempt `generate_and_upload`, which will raise in a test env without a real registry. The exception is caught; text is unchanged. The `startswith` handles both the fallback case (text unchanged) and the hypothetical case where a test env has a registry (text + URL). The learnings.md "assertion softening" pattern applies: the `startswith` form cannot distinguish correct fallback from unexpected URL appendage. However, a dedicated test `test_no_url_appended_when_uow_id_missing` covers the no-URL case explicitly, and `test_appends_detail_url_when_generation_succeeds` covers the URL append path with a mock. The softening is acceptable given these two flanking tests.

b. `test_appends_detail_url_when_generation_succeeds`: new test. Patches `"orchestration.wos_uow_detail_gen.generate_and_upload"` with a fake URL. Asserts `result["text"] == f"{expected_text}\n{fake_url}"`. The mock path is correct: since the production code does `from .wos_uow_detail_gen import generate_and_upload` as a lazy import inside the function body, the name is fetched from `sys.modules['orchestration.wos_uow_detail_gen'].generate_and_upload` on each call. The patch on `"orchestration.wos_uow_detail_gen.generate_and_upload"` intercepts this correctly.

c. `test_no_url_appended_when_uow_id_missing`: new test. Deletes `uow_id` from the default message dict. Asserts `result["text"] == expected_text` (exact). Correct.
