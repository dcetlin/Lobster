### Stage 2: Quality Review

**Encoded Orientation check (constraint-3):** The unit files change durable system scheduling behavior, but the behavior is already encoded on the live instance. This PR encodes into version control what is already encoded at the OS level. No new behavioral defaults are introduced. Constraint-3 does not fire.

**Quality findings:**

- **Migration 87 m87_count double-increment (cosmetic bug).** The loop increments `m87_count` after `sudo cp` succeeds, and again after `systemctl enable --now` succeeds. On a fresh-install machine, each newly-installed timer contributes 2 to `m87_count`, inflating the reported "installed/enabled N unit(s)" count. The reported number will be up to 2x the actual timer count. This is cosmetic — the migration function correctly and the double-increment does not affect idempotency or correctness. Not blocking.

- **Content-only idempotency gap (latent maintenance debt).** The idempotency check is `[ ! -f "$dst/$timer_name" ]` — file presence only. If a unit file already exists at the destination but has stale content (e.g., a previous version with a different schedule), the migration will skip the copy. Re-running upgrade.sh on an existing instance will not update unit file content — only install unit files that are entirely absent. Future PRs that modify unit file schedules or ExecStart values will require either a new migration step or a manual systemctl reload. This is consistent with the behavior of other file-based migrations in upgrade.sh, but it is a latent maintenance pattern to be aware of. Not blocking for this PR since all files being installed are new and not yet present on any second instance.

- **Feature-bundling check** (learnings.md PR #712, #717): 22 files changed — 20 unit files in `services/`, 1 `upgrade.sh` modification, 1 new `docs/` file. All are named in the PR description. The CONVENTIONS.md update (runtime, outside the diff) is correctly noted as out-of-scope. No undocumented scope expansion.

- **Unit file correctness.** All service files use `Type=oneshot` and `User=lobster` with `ExecStart=/home/lobster/lobster/scheduled-tasks/dispatch-job.sh {job-name}`. All timer files use `Persistent=true` and `WantedBy=timers.target`. The `lobster-lobster-hygiene.timer` double-prefix matches the live system (engineer-confirmed). The `wos-hourly-observation` 7-fire daily schedule (06-12 UTC) matches the description. The `structural-hygiene-audit` March 31 annual schedule is valid; `Persistent=true` means a missed March 31 fires on next boot — acceptable for an annual job.
