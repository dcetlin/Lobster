## Round 1 — 2026-05-09

### Stage 1: Vision Alignment

**Prior entering review:** This implementation is solving the wrong problem, or solving the right problem in a direction that forecloses better paths.

**Theory of change under test:** The vision's fundamental intent is a system that knows what Dan is building toward and routes work accordingly. LOS Cycle 1 extracts first-person action commitments from conversation history and surfaces them as a queryable todo list with inline buttons. The theory is: if Dan's own commitments are captured and resurfaced, attention is freed for things requiring genuine judgment.

**What would have to be true for this to be the right path:** Dan's action commitments are currently lost because they are voiced in conversation and not persisted anywhere retrievable. The bottleneck for "what should I work on?" is capturing the action commitments Dan has already made, not orienting around the vision object. The cost of maintaining a separate action-items DB is lower than the attentional cost of not having it.

**What this forecloses:** The LOS module introduces a second personal task database alongside memory.db and the existing task system (create_task/list_tasks). Any future unification of task tracking will now need to merge or bridge three stores. The hourly extractor adds a recurring LLM cost ($) and a recurring scan of the processed/ directory. The inline button pattern is now established as the interaction model for task management, which has implications for future UI expansion.

**Opportunity cost signal:** vision.yaml principle-4 is "integration rate before new feature rate — wire what exists before building more." The existing task system (create_task/list_tasks via MCP) and memory.db already provide storage. Whether LOS is wiring what exists or adding a new thing is a judgment call the oracle cannot make from the diff alone, but the question should be explicit in the PR.

**Stage 1 finding:** The alignment question I cannot resolve from the diff: is the existing task MCP system (create_task/list_tasks) inadequate for this use case, and if so, why? The PR introduces a third tracking store without stating what the existing two stores cannot do. This is not a blocker — the use case (automated extraction, dedup, snooze) is genuinely new — but the justification for a third store should be explicit. Alignment verdict: **Questioned** (not Misaligned — the use case has clear vision grounding in "extending cognitive reach without screen dependency," but the store proliferation deserves an explicit note).

### Stage 2: Quality Review

**OODA / Encoded Orientation check:** The dispatcher bootup is modified to add a new callback routing branch and a new command handler. This is an Encoded Orientation decision (it changes behavioral defaults of the dispatcher). The change is traceable to the LOS feature itself (vision.yaml's cognitive extension framing) and the dispatcher bootup is the correct vehicle. The prior is implicit in the LOS design work. This passes constraint-3 at the substance level.

**Confirmed findings (engineer-flagged items):**

1. **Priority sort:** `ORDER BY priority ASC` correctly surfaces priority=1 (urgent) first. Confirmed correct.

2. **Dispatcher bootup callback numbering:** The numbering is broken in a structural sense. The LOS callback branch is inserted between step 6 and step 7 without a step number. The block starts with bare `elif` rather than `7. elif`. Steps 7, 8, 9 then follow. A dispatcher following the numbered list encounters an unnumbered branch between 6 and 7, which is syntactically valid pseudocode but violates the document's own numbering convention and would confuse a human reader. This is a cosmetic defect but in a document that is loaded as orientation context, document structure coherence matters.

3. **Migration bash heredoc quoting:** The async-deep-work.md append uses `<< 'ADWEOF'` (single-quoted delimiter). This prevents shell variable expansion inside the heredoc. The content contains no variables that require expansion. Confirmed safe.

4. **DB connection lifecycle in route_los_callback:** The `_own_conn` / `finally: conn.close()` pattern is correct. `mark_done`, `mark_dismissed`, `mark_snoozed` each call `conn.commit()` before returning, so writes persist before any exception in the return path. The finally block closes correctly on error. Confirmed correct.

**New findings:**

**Finding 1 (BLOCKER): Mirror-constant pattern in test_schema.py**

`tests/unit/los/test_schema.py` declares local constants `STATUS_OPEN = "open"`, `STATUS_DONE = "done"`, `STATUS_DISMISSED = "dismissed"`, `STATUS_SNOOZED = "snoozed"` and uses them in assertions. These mirror the raw string literals in `src/los/db.py`. Per the learnings.md pattern documented across PRs #696, #800, #967, #970, #973, #974, #976, #980: "locally-declared mirror of a production constant creates a divergence risk: when the production constant changes, the test constant does not." In this case, there is no StrEnum in db.py to import from — the status values are bare string literals in SQL. This is the root cause: the test mirrors match because the production module has no enum to import.

Per the golden-patterns.md StrEnum rule (2026-04-24): "The status field has a finite domain. DB write functions use raw string literals in SQL; queries filter on raw string literals. A status renamed in the future requires a grep across all SQL strings, not a single enum rename." The correct fix is to define a `ActionItemStatus` StrEnum in db.py and use it in all SQL and in tests. The tests' mirror constants are the symptom; the missing enum is the root cause. This is a NEEDS_CHANGES blocker per the pattern's documented severity.

**Finding 2 (BLOCKER): Dispatcher bootup step numbering breaks document structure**

The LOS callback branch is inserted between step 6 and step 7 without a step number. In the PR branch, line 836 reads `elif data.startswith("todo-"):` with no step prefix, while line 844 reads `7. elif data.startswith("decide_retry:")`. The missing step number is not a cosmetic issue in context: this file is read as orientation by the dispatcher. An unnumbered elif sitting between numbered branches creates a structural inconsistency that could cause a dispatcher following the "numbered steps" convention to misread the sequence.

The correct fix is `6.5.` or (better) renumber: the LOS branch becomes step 7, decide_retry becomes step 8, else becomes step 9, mark_processed becomes step 10.

**Finding 3 (advisory): Raw status strings in SQL — no StrEnum**

All SQL queries and write functions in `src/los/db.py` use raw string literals for status values: `'open'`, `'done'`, `'dismissed'`, `'snoozed'`. The codebase has an established pattern (golden-patterns.md, 2026-04-24) requiring StrEnum constants for all enum-backed DB columns. The domain is finite (4 values). This is the same issue as Finding 1 root cause. A single `ActionItemStatus` StrEnum fixes both.

**Finding 4 (advisory): /todos dispatch uses hardcoded path without LOBSTER_WORKSPACE env var**

In the dispatcher bootup LOS Commands section: `open(Path.home() / "lobster-workspace/scheduled-jobs/tasks/los-todos-query.md").read()`. This hardcodes the workspace path rather than using `Path(os.environ.get("LOBSTER_WORKSPACE", Path.home() / "lobster-workspace"))`. Minor: consistent with other patterns in the file, but the task file won't exist before migration 102 runs, which would raise a FileNotFoundError at dispatch time on a fresh install before the migration runs. A `Path.exists()` guard or the migration dependency should be explicit.

**Finding 5 (advisory): Dedup operates as check-then-act without DB-level uniqueness constraint**

`find_duplicate()` then `insert_action_item()` is a check-then-act sequence with no UNIQUE constraint on `dedup_key`. In a multi-process scenario this would be a race condition producing duplicate rows. In practice the hourly scanner is single-threaded and the callback handler is synchronous, so the practical risk is low. But the schema's stated design intent (dedup via dedup_key) is not enforced at the DB level. A `UNIQUE INDEX` on `(dedup_key, status)` filtered to `status IN ('open', 'snoozed')` would make the constraint structural. This is advisory for Cycle 1.

**What engineer flagged but is not an issue:**

The voice note indirect hook (hourly lag) is acknowledged and appropriate for Cycle 1. The `get_open_items` snoozed comparison (`snoozed_until < datetime('now')`) is correct: a date-only string like '2026-06-15' is lexicographically greater than today's datetime string during the day, and less than it after that day passes — this correctly produces date-granularity snooze behavior. No bug here.
