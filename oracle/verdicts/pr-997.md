### Findings

**Gap 1: "executing" bucket label is semantically incorrect at production scale (BLOCKING)**

`_classify_status` maps any status not in `_COMPLETE_STATUSES`, `_FAILED_STATUSES`, or `_ESCALATED_STATUSES` to the label `"executing"`. This means `"proposed"`, `"pending"`, `"blocked"`, and `"expired"` all report as `"executing"`. The `_EXECUTING_STATUSES` frozenset names `{"active", "executing", "ready-for-steward", "ready-for-executor", "diagnosing"}` but is not used in `_classify_status` — the function's else-branch is the catch-all regardless of what that frozenset contains. The `_EXECUTING_STATUSES` frozenset is defined but serves no behavioral purpose.

At current production scale the PR description shows 259 UoWs returned with 2 "done." This means approximately 257 UoWs appear in the report header as `executing: 257`. The pipeline at this scale has the majority of UoWs in `"proposed"` status — proposed UoWs are not executing; they have not yet been claimed by the Steward. A report header reading `executing: 257` gives the wrong operational picture for the most common production configuration. The report is designed for pipeline visibility; the primary summary number is misleading.

The implementer acknowledged this concern (#2 in known concerns). The test `test_proposed_maps_to_executing` explicitly tests and accepts this behavior. Acceptance in a test does not resolve the semantic question — the test documents the behavior; it does not justify the label.

**Required resolution (one of three acceptable paths):**

(a) Rename the catch-all bucket to `"in-pipeline"` or `"not-yet-started"` in `_classify_status`, the summary dict keys, and the report header line. Update `_EXECUTING_STATUSES` or remove it if the bucket is renamed. Update the test assertions.

(b) Add `"proposed"`, `"pending"`, `"blocked"`, `"expired"` to their own bucket(s) with accurate labels. The `_EXECUTING_STATUSES` frozenset already exists — make it the positive gate rather than the catch-all, and add a separate `"queued"` or `"proposed"` bucket for the rest.

(c) Dispute this gap with a specific stated reason why "executing" is the intended label for proposed UoWs at current scale, and add a comment to `_classify_status` documenting the intentional broadening.

Whichever path is chosen, the `_EXECUTING_STATUSES` frozenset must either be used in the logic or removed. A defined, exported frozenset that has no call site is dead code following the PR #906 anti-pattern ("A declared, exported, and tested threshold constant that is never referenced in the function it is meant to gate is a dead constant").

**Finding 2: fetch_in_window SQL excludes proposed UoWs with NULL started_at (advisory)**

The window filter is `WHERE started_at >= ? OR completed_at >= ?`. UoWs where both timestamps are NULL are excluded. A UoW in "proposed" status with no `started_at` set will not appear in any time window. The integration test `test_report_counts_proposed_uow_in_window` works around this by seeding `started_at` directly via SQL.

The PR description states a live test showed "259 UoWs returned" in a 48-hour window. This is consistent with the SQL behavior only if those 259 UoWs have `started_at` set. If the production DB has proposed UoWs with NULL `started_at`, they would be silently excluded with no signal to the operator. This is not necessarily wrong — a "pipeline activity" window may legitimately exclude UoWs that have never been touched — but it should be documented in the command's help text or docstring. Currently `cmd_report`'s docstring says "covers all UoWs whose started_at or completed_at falls within the requested window" without noting that UoWs with no timestamps are excluded.

This is advisory: the SQL behavior is correct for what the docstring says. Add a note that UoWs with no started_at and no completed_at are excluded from the window.

**Finding 3: _EXECUTING_STATUSES frozenset is dead code (referenced in Gap 1, non-blocking separately)**

`_EXECUTING_STATUSES = frozenset({"active", "executing", "ready-for-steward", "ready-for-executor", "diagnosing"})` is defined at module scope. It is not used in `_classify_status`, `_compute_summary`, `_format_report`, or `cmd_report`. It is not imported by the test file. This is the PR #906 dead-constant anti-pattern. If Gap 1 is resolved by path (a) or (c), remove this frozenset. If resolved by path (b), wire it into `_classify_status`.

**Finding 4: N+1 audit queries in _format_report (advisory, acknowledged)**

`_format_report` calls `registry.fetch_audit_entries(uow_id)` inside a per-row loop for rows with `status in ("failed", "needs-human-review", "ready-for-steward")`. This is an N+1 pattern against the audit log for those statuses. At current production scale and with the stated status distribution (mostly proposed), this fires rarely. The implementer named this as a known concern. For a human-invoked CLI command, this is acceptable. Note: the kill label fetch also fires on `"ready-for-steward"` — a status that is pre-terminal, not post-failure — which means UoWs waiting for Steward review will trigger an audit query even when no kill classification exists. The logic is functional but the status filter is broader than needed. Advisory, non-blocking.

---

**Finding 5: wall_clock_seconds derivation only for completed UoWs (correct, no action)**

The `CASE WHEN started_at IS NOT NULL AND completed_at IS NOT NULL THEN ... ELSE NULL END` in `fetch_in_window` is correct. NULL for non-completed UoWs is the right value. `_compute_summary` correctly filters to `_classify_status(...) == "complete"` for the median computation. This matches the implementer's intent.
