**Finding 1 (BLOCKER): Stale mock in test suite will fail after merge.**

Full-repo grep for `run_ttl_recovery` returns 3 hits in `tests/unit/test_orchestration/test_github_rate_limit_check.py`:

- Line 178: `patch.object(_hb_fresh, "run_ttl_recovery"),`
- Line 215: `patch.object(_hb_fresh, "run_ttl_recovery"),`
- Line 253: `patch.object(_hb_fresh, "run_ttl_recovery"),`

`patch.object` raises `AttributeError` when the named attribute does not exist on the target object. After this PR merges, `run_ttl_recovery` will not exist as a module-level name in `executor-heartbeat.py`. All three tests will fail with `AttributeError: <module ...> does not have attribute 'run_ttl_recovery'`.

This is the exact failure pattern named in the learnings.md "stale assertion" entry (2026-04-22, #827): "A test that asserts behavior an oracle decision explicitly prohibits becomes a stale assertion after the fix lands." Here the mock is not asserting the behavior exists — it is suppressing the call to prevent it from running during the test. But the mechanism (`patch.object`) requires the attribute to exist. The behavior is: after the fix lands, the patch fails, the test fails.

The PR description's test plan includes "No `run_ttl_recovery` references remain in the file" — but the check was scoped to the heartbeat file, not the test file. The full-repo grep requirement from the golden-patterns.md entry catches exactly this gap.

**Resolution:** Remove the three `patch.object(_hb_fresh, "run_ttl_recovery")` lines from `tests/unit/test_orchestration/test_github_rate_limit_check.py`. The mock was suppressing a function that no longer exists; removing the mock line is the correct fix. The tests should pass without it because the function is gone and cannot be called.
