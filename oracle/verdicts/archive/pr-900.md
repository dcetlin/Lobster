## Stage 1: Vision Alignment

The adversarial prior entering this review: the artifacts field may be the right schema addition but the wrong timing — if nothing reads `artifacts` at the time of merge, this is stored-but-unread data, and the extraction machinery is complexity in service of a column that produces no signal. This would be the "library with no wiring" pattern from learnings.md (PR #714): a new capability that requires a caller to be effective, where no caller is introduced.

The oracle vocabulary constrained this review in a specific way: the PR #714 learnings entry caused me to specifically look for whether any downstream consumer (steward query, retrospective job, wos_report) reads `uow_registry.artifacts`. None is introduced in this PR. This distinction matters — the Stage 1 question is not "will someone eventually read this?" but "is the absence of a current consumer a signal that we are building ahead of the governing structure?"

Against this concern: the `artifacts` field is stored in the canonical registry, directly queryable by any reader without structural wiring changes (unlike PR #714's routing module, which required a new import to become effective). The vision's principle-4 ("wire what exists before building more") is satisfied in the other direction here: the write path now exists, enabling a read path to be added without DB schema changes. The learnings.md PR #891 pattern ("Consumer-vs-producer distinction") is relevant: the gap in PR #714 was that the protocol layer was architecturally open — all producers were uniformly unconnected. Here, the producers (subagents) are connected; what's absent is a specific consumer. The absence of a consumer is advisory, not structural.

Vision alignment: this PR serves the WOS substrate goal (richer completion records for audit and routing) and the active_project phase intent (Registry knows what work exists and what was produced). Issue #880 closes correctly. The implementation is additive and does not foreclose any path.

## Stage 2: Quality Review

- **Migration safety**: `ALTER TABLE uow_registry ADD COLUMN artifacts TEXT NULL` is backward-compatible — existing rows get NULL default, no data loss. Migration is correctly scoped, documented, and forward-only. Reads by existing UoWs are unaffected (NULL artifacts treated as empty list by the deserializer).

- **Non-fatal failure path is correctly placed**: `maybe_complete_wos_uow` gates artifact extraction on `if result_text:` (avoiding the call when no result text is present), wraps the extraction and write in `try/except`, logs the warning, and does not block the UoW transition that already succeeded. This is the correct design posture for an advisory enrichment field. The PR #891 learnings pattern ("document the external gate that guarantees unconditional field values are correct") applies in reverse here: the external gate (`if result_text`) is the correct entry condition.

- **Naming ambiguity between `_extract_outcome_refs` and `_extract_artifact_refs`**: The two functions co-exist in the same file. They differ structurally (typed list of dicts vs. dict with three lists), serve different consumers (registry vs. result.json enrichment), and the PR adds a docstring note to `_extract_artifact_refs` explicitly distinguishing them. This is a documentation risk — a new reader will see two similarly-named extraction functions — but not a correctness defect. The note is present; the structural difference is observable in the return types.

- **SHA keyword-gate is document-level, not proximity-level**: The gate fires on `if re.search(r"\bcommit\b", text)`, then all SHA-looking strings in the document are candidates. If a result text mentions "commit" once and also contains an unrelated hex string elsewhere (e.g., a file hash in a different section), both would be extracted. For current WOS agent result texts this is unlikely to cause false positives, and the denylist handles the most common hex English words. This is a documented heuristic gap, acceptable for an advisory field.

- **Encoded Orientation check (constraint-3)**: No agent definitions, bootup files, CLAUDE.md, or behavioral defaults modified. The `artifacts` field stores data at UoW completion but does not alter routing decisions. Constraint-3 does not apply.

- **Feature bundling check**: 4 files changed; PR description accounts for all 4 (migration SQL, registry.py, wos_completion.py, test file). No undescribed additions.

- **Test coverage**: 24 new tests cover the pure function (_is_plausible_sha, _extract_outcome_refs), the DB write path (update_artifacts against real SQLite in tmp_path), and end-to-end via maybe_complete_wos_uow (artifacts populated from PR+issue mentions; empty result_text leaves column NULL; extraction failure does not block UoW transition). The non-fatal failure test correctly patches _extract_outcome_refs to raise RuntimeError and verifies UoW status reaches READY_FOR_STEWARD.

**Patterns introduced**: Registry enrichment at completion time as a non-fatal advisory field — the correct pattern for adding observability data that must not block the main transition arc.

**What this forecloses**: A more structured approach where subagents explicitly declare typed artifacts at `write_result` time (rather than having them extracted from prose). If that approach is pursued later, the regex extraction layer in `_extract_outcome_refs` becomes redundant. However, it can be removed without schema changes — the `artifacts` column is format-agnostic.

**Opportunity cost note**: No downstream consumer reads `artifacts` at merge time. The first reader (a steward query, wos_report, or retrospective job) will complete the wire. Until then, the field stores data that is queryable but not queried.
