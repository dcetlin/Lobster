## Stage 2: Quality Review

### Encoded Orientation check (OODA constraint-3)

The `.claude/sys.dispatcher.bootup.md` modification adds two new routing steps to the callback handler table. This is a change to dispatcher behavioral defaults — an Encoded Orientation decision under constraint-3. Prior exists: the handlers `handle_decide_retry` / `handle_decide_close` were introduced in an earlier PR with documented intent (issue #901 / PR #894 escalation arc). The prior is traceable to `vision.yaml principle-4` (wire what exists). The routing instruction is deterministic prefix-matching — no LLM judgment encoded. Constraint-3 is satisfied.

### Quality findings

- **Inline import chain in bootup doc is dense but correct.** Steps 7 and 8 encode a full `sys.path.insert + import + instantiate + call + reply` chain in a single numbered bullet. The golden-patterns.md "table-as-compaction-resistant encoding" pattern calls for trigger-in-one-sentence and verifiable-difference-in-one-sentence. This encoding is one sentence but long. The content is correct — the path `/home/lobster/lobster` is the canonical location, REGISTRY_DB from `src.orchestration.paths` is the correct constant, and `reply_to_message_id=msg.get("original_telegram_message_id")` uses the field added in `lobster_bot.py`. The dense encoding is a readability concern, not a correctness defect; the dispatcher will execute it correctly.

- **`original_telegram_message_id` field addition is correctly placed.** The `lobster_bot.py` change adds the field to the callback payload dict at line ~993, in the same dict as `callback_data` and `timestamp`. This is the correct producer location. The consumer (bootup doc step 7/8) reads `msg.get("original_telegram_message_id")`, which safely returns `None` if the field is absent (backward-compatible). The `reply_to_message_id=None` case produces a standalone reply rather than a threaded reply — acceptable degradation for pre-upgrade callbacks.

- **Test suite is sound and complete for its scope.** 14 new tests cover: status transitions (retry → ready-for-steward, close → failed), steward_cycles reset, confirmation text content, error handling for non-blocked UoWs, and callback_data parse round-trips. The non-blocked-UoW test assertions (`"not currently in"` or `"could not be"` in response) are checked against the actual function return text from `dispatcher_handlers.py` — both strings appear in the production return values. No assertion softening concern (learnings.md PR #839): the status assertions use exact string comparison against `DECIDE_RETRY_EXPECTED_STATUS` and `DECIDE_CLOSE_EXPECTED_STATUS` constants, not negative assertions. `test_callback_prefix_matches_steward_escalation_format` is a reflexive test (it asserts that the local constant equals itself in a `f"decide_retry:{uow_id}"` form) — this is low-value but not harmful. The `test_retry_resets_steward_cycles_to_zero` test correctly seeds via direct SQL and verifies via `registry.get()` — this is the correct test pattern for a field that the public API writes but does not expose as a constructor parameter.

- **Feature bundling check: clean.** Three files changed, PR description explicitly describes all three. No undescribed scope expansion.

- **Pre-existing test failure is correctly attributed.** `test_is_bootup_candidate_gate_active_reflects_flag` fails due to `src.ooda` import error on `main`. The PR author verified this by running against `main`. Not introduced by this branch.

### Failure modes

The `sys.path.insert(0, "/home/lobster/lobster")` in the bootup doc steps is a hardcoded path. If the lobster repo is relocated, these steps break silently — the dispatcher would fall through to `else: Unknown callback`. This is the same coupling already present in other bootup doc import patterns (not a regression introduced here). The risk is low given the path is canonical.

### What this makes easier

Future callback types following the `<prefix>:<uow_id>` pattern have a clear template: startswith check, slice the uow_id, import handlers, call, send_reply with threading. The pattern is encoded and readable.

### What this makes harder

Nothing is foreclosed. The pure handler functions remain independently callable. The bootup doc encoding could be refactored to a PreToolUse hook (advisory-to-hook promotion pattern, golden-patterns.md 2026-04-22) if the dispatcher callback table grows — but that promotion is not required now and is a one-way door in neither direction.
