### Stage 2: Quality Review

#### Does it do what it claims to do?

Yes. The three changes (UoW title column, category badge, canonical URL) are correctly implemented. The pure/near-pure function composition is well-applied: `render_html()` remains pure, side effects are isolated to `upload_html()` and `_enrich_uow_with_github_metadata()`, both called only from `main()`. Immutability is maintained via `{**uow_row, ...}` spreading rather than mutation. Graceful degradation is correct: every subprocess failure path returns None or a default, and the rendering code handles both.

#### The two-call design — NEEDS_CHANGES

The core design gap: `_enrich_uow_with_github_metadata` makes two sequential subprocess calls to `gh issue view` — one for `--json title` and one for `--json labels` — when a single call with `--json title,labels` would return both fields in one round-trip.

The PR comment acknowledges this:
> "A combined call (`--json title,labels`) would be slightly faster but the helpers are independently testable and the latency difference is negligible for a dashboard with typically <20 active UoWs."

This rationale does not hold. The testability argument is the specific problem: `_fetch_issue_title` and `_fetch_issue_labels` are always called together. They could be replaced by a single `_fetch_issue_metadata(url) -> dict` returning `{"title": ..., "labels": [...]}`, which is equally (and more cohesively) testable. The current design fragments what is a single logical operation — "get GitHub metadata for this issue" — into two functions, each making a separate network round-trip.

This creates two concrete defects:
1. **Doubled latency per UoW**: two sequential subprocess calls with 10-second timeouts each means up to 20 seconds per UoW at timeout boundary.
2. **Semantic inconsistency window**: if an issue is modified between the two calls (title and labels are fetched at different moments), the enriched row could reflect different versions of the issue's state. A single combined call eliminates this window.

The resolution is specific: replace `_fetch_issue_title` + `_fetch_issue_labels` with a single `_fetch_issue_metadata(url) -> dict | None` that calls `gh issue view <url> --json title,labels` once and returns `{"title": ..., "labels": [...]}`. `_derive_category_from_labels` remains unchanged. `_enrich_uow_with_github_metadata` calls `_fetch_issue_metadata` once.

#### Missing test for `_enrich_uow_with_github_metadata`

The module docstring's test coverage list does not include `_enrich_uow_with_github_metadata`. There is no test class for this function. Tests either go above (via `main()` — which mocks `_fetch_issue_title` and `_fetch_issue_labels` indirectly) or below (individual helper functions). The integration logic in `_enrich_uow_with_github_metadata` — which branches on `issue_url` presence and merges results — has no direct unit test.

This is addressable in the same pass as the two-call fix: a `TestEnrichUowWithGithubMetadata` class with cases for (a) no issue_url, (b) both title and labels fetch succeed, (c) title fetch fails, (d) labels fetch fails.

#### `python3 -m py_compile` in PR description

The PR description lists `python3 -m py_compile src/orchestration/wos_dashboard.py` as a manual check run. Learnings.md flags bare `python3` as a violation in scripts, instruction files, and agent definitions. A PR description is not an executable artifact — this is advisory, not a blocker.

#### Encoded Orientation check

This PR does not change behavioral defaults, system constraints, agent identity, or decision-making rules durably. No Encoded Orientation decision is present.

#### What this makes easier / harder

Easier: future callers of the dashboard can use a stable URL instead of discovering the output path. Easier: the pure-function composition pattern is now more complete — render is clearly separated from fetch.

Harder: adding new GitHub metadata fields now requires a code change to the enrichment layer. The two-call design makes this slightly worse because each new field means another call; the combined-call design makes it straightforward.
