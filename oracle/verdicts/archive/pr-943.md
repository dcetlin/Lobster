## Stage 2: Quality Review

### Prior design gaps

The design doc went through three oracle rounds. Round 3 was APPROVED. The two critical gaps from Rounds 1-2 were:

1. `route_wos_message()` is a pure function returning routing instructions — a daemon calling it must also call `_dispatch_via_claude_p` directly. **Status: addressed.** The implementation correctly calls `route_wos_message()` to get the decision, then calls `_dispatch_via_claude_p(instructions=prompt, uow_id=bare_uow_id)` directly.

2. Call signature: `_dispatch_via_claude_p(instructions: str, uow_id: str)` — two keyword args, and `task_id` has the `"wos-"` prefix that must be stripped to get `uow_id`. **Status: addressed.** The implementation does `bare_uow_id = task_id.removeprefix("wos-")` and calls `_dispatch_via_claude_p(instructions=prompt, uow_id=bare_uow_id)`.

### Does it do what it claims to do?

Yes. The daemon:
- Polls inbox every 30s, filtering for `type == "wos_execute"` client-side (no MCP dependency).
- Claims via rename (stamp then rename sequence — see below).
- Calls `route_wos_message()` → if `spawn_subagent`, calls `_dispatch_via_claude_p` with correct signature.
- Marks processed (rename to `processed/`) on success; failed on exception.
- Respects `wos-config.json execution_enabled` gate and `MAX_AGENTS_GATE`.
- Writes inbox alerts (not direct `send_reply`) for failure events — correct, since a daemon has no MCP session context.

### _claim_message race-safety

The reviewer is asked to pay special attention to this. The claim sequence is:
1. Check `src.exists()` — if missing, return False (already claimed).
2. Stamp `_processing_started_at` into the file (read → modify → write `.tmp` → replace).
3. `src.rename(dest)`.

**Finding:** There is a TOCTOU window between step 1 and step 3. Two processes can both pass the `src.exists()` check before either has executed the rename. The one that executes `src.rename(dest)` second will raise `FileNotFoundError` (on Linux, `rename()` for a missing source raises `FileNotFoundError`), which is caught by the broad `except OSError as exc` handler and returns `False`. The second claimer therefore correctly backs off. This is acceptable: Linux `rename()` is atomic when source and destination are on the same filesystem, so the race resolves correctly — one process claims, the other backs off at the `OSError` catch. The stamp step (read-modify-write) introduces a minor issue: if two processes both read the file and both write `.tmp`, the second `.tmp.replace(src)` overwrites the first's stamp but does not affect correctness (both stamps are valid timestamps). The subsequent `src.rename(dest)` is what enforces mutual exclusion. **This is race-safe in practice.**

One minor issue: the stamp step uses `src.with_suffix(".json.tmp")` to create a temp file in the same directory as `src` (inbox/). This means the tmp file is `inbox/message_id.json.tmp`. The `_read_inbox_messages` function globs `INBOX_DIR.glob("*.json")` — `.json.tmp` files are not `.json` files, so they are not picked up as messages. Not a bug, but worth noting that a crash between `tmp.write_text()` and `tmp.replace(src)` leaves a `.json.tmp` orphan in inbox/. These orphans will accumulate silently. They are harmless (not picked up as messages) but are invisible debris. This is a low-severity concern, not a blocking issue.

### _dispatch_via_claude_p call signature matching executor.py

Confirmed correct. The production signature at line 996 of `executor.py` is `def _dispatch_via_claude_p(instructions: str, uow_id: str) -> str`. The daemon calls it as `_dispatch_via_claude_p(instructions=prompt, uow_id=bare_uow_id)` with keyword arguments. The `task_id` prefix stripping (`bare_uow_id = task_id.removeprefix("wos-")`) is correct: `route_wos_message` returns `task_id="wos-{uow_id}"`, and `_dispatch_via_claude_p` expects the bare `uow_id`.

**One subtlety:** `_dispatch_via_claude_p` is a synchronous blocking subprocess call (see its docstring: "The executor blocks until the subprocess exits"). The daemon calls this in-process in `_route_single_message`, which is called in the poll cycle. This means the daemon's poll loop blocks for the duration of each `claude -p` invocation (potentially minutes). This is not the PR #914 violation (synchronous work inside `route_wos_message`, which is meant to be pure) — the daemon's poll loop is explicitly designed to do work, not just route. However, with `POLL_INTERVAL_SECONDS=30` and potentially multiple wos_execute messages per cycle, the daemon can be occupied for extended periods before processing subsequent messages. The `MAX_AGENTS_GATE` check happens once per cycle before processing all messages in the batch — it does not re-check between messages. This means N messages can each spawn a synchronous subprocess in sequence before the gate has a chance to re-evaluate active agent count. This is a known tradeoff in Phase 1; the implementation is functionally correct and the gate check is the correct design point.

### Migration 90 service-update logic

The migration handles three cases:
1. Source exists, destination does not: install + enable + start.
2. Both exist and differ: update (copy + daemon-reload + restart).
3. Source does not exist: skip with warning.

The idempotency guard for case 2 (diff-based check) is correct — it only restarts if the service file changed, consistent with the existing upgrade.sh pattern. The migration does not increment `migrated` in the diff-match-but-already-up-to-date path, which is correct (no work was done).

**One gap:** The migration's `warn` call in case 1 dispatch — `warn "Migration 90: could not copy $(_m90_svc_src) — manual install needed"` — has a typo: `$(_m90_svc_src)` uses command-substitution syntax `$(...)` instead of variable expansion `${_m90_svc_src}`. This would execute `_m90_svc_src` as a shell command and produce an error string rather than the file path in the warning message. The migration's functional behavior is unaffected (the `sudo cp` already failed; the `warn` message is advisory), but the warning message will be garbled. Low severity.

### Phase 1 fallback case

The dispatcher's existing `wos_execute` branch is untouched. The race resolution: the daemon polls every 30s and claims aggressively. The dispatcher's `wait_for_messages` loop receives the message but the daemon may have already claimed it (moved to `processing/`) by the time the dispatcher checks. The dispatcher's `mark_processing` on a file that no longer exists in `inbox/` will silently fail (or return an appropriate result from the MCP server). The dispatcher's fallback is correct: if the daemon claimed the message, the dispatcher's claim fails and the message is not processed twice.

**One edge:** If the daemon claims the message and then crashes before calling `_release_message_processed`, the message stays in `processing/` indefinitely. The dispatcher cannot recover it because the dispatcher only processes messages in `inbox/`. There is no TTL recovery mechanism in this PR for daemon-claimed messages stuck in `processing/`. This is a known operational gap in Phase 1; it should be flagged for Phase 2 (a heartbeat or TTL sweep for messages stuck in `processing/`). Not blocking for this PR.

### Test coverage

26 unit tests covering: gates (execution_enabled, MAX_AGENTS_GATE), happy path (spawn_subagent), send_reply alert path, dispatch failure, route_wos_message exception, claim race condition, and filter function. The test structure uses `importlib.util.spec_from_file_location` with mocked heavy dependencies, consistent with the importlib re-export bridge pattern (golden-patterns.md, 2026-03-30). Tests import via file path to avoid CI import conflicts with the orchestration package. Coverage is adequate for Phase 1.

The `test_dispatch_called_with_stripped_uow_id` test directly verifies the prefix-stripping logic — this is the most important correctness invariant and it is tested.

### Patterns introduced

- **Daemon-as-pure-file-I/O worker**: the daemon uses no MCP, no LLM, no sockets — only filesystem rename for claim/release and `write_inbox_message` for alerts. This is a clean, hermetic pattern for mechanical routing work.
- **Polling daemon with MAX_AGENTS_GATE check per cycle**: re-checking at the poll boundary (not per-message) is a reasonable Phase 1 approximation.
- **`write_inbox_message` as alert path for processes without MCP context**: consistent with learnings.md PR #856 (direct inbox file write instead of MCP HTTP endpoint).

### What this makes easier / harder

Easier: wos_execute throughput at high load; dispatcher context is freed from mechanical routing decisions; the daemon can be tuned independently of the dispatcher.

Harder: diagnosing issues when a message is stuck in `processing/` (claimed by daemon, daemon crashed). Phase 2 will need a recovery sweep. The dual-claimer design (daemon + dispatcher) creates a permanent need to understand which process claimed a given message.

### Summary of findings

No blocking issues. Two low-severity observations documented above (`.json.tmp` orphans in inbox/; `$(_m90_svc_src)` typo in warn message). Both are non-blocking. The critical design-doc gaps from Rounds 1-2 (route_wos_message call semantics, `_dispatch_via_claude_p` signature, prefix stripping) are correctly resolved in the implementation and verified by test. Constraint-3 is satisfied via the prior design verdict. Alignment is confirmed.
