### BLOCKING ISSUE: handle_steward_trigger runs synchronously in the dispatcher's main processing path

`route_wos_message` is called by the dispatcher directly in its main loop â€” claim, route, spawn/noop, mark_processed. For `wos_execute`, `route_wos_message` returns in under a second (it's a pure prompt-construction function). For `steward_trigger`, `route_wos_message` calls `handle_steward_trigger`, which blocks for 0â€“60 seconds on `subprocess.run(["uv", "run", steward_script], timeout=60)` before returning.

The dispatcher's 7-second rule exists precisely because blocking the main loop degrades system availability. The `handle_steward_trigger` subprocess can block for 60 seconds in the timeout case. In the normal case (steward completes in ~20-40 seconds based on observed heartbeat runtime), the dispatcher is blocked for that duration.

The consequence: any Telegram message, WOS execute message, or other inbox message arriving during a `steward_trigger` handling cycle waits up to 60 seconds before being processed. If completions cluster (multiple UoWs completing near-simultaneously), multiple `steward_trigger` messages queue, and each dispatcher handling cycle consumes 0-60 seconds. The fix that was meant to speed up pipeline drain could make the dispatcher less responsive under exactly the load conditions it's designed to handle.

**The correct architecture:** `handle_steward_trigger` should dispatch the steward cycle as a background subagent, not run it synchronously. The dispatcher already has the infrastructure:

```python
# Instead of subprocess.run(..., timeout=60), return:
return {
    "action": "spawn_subagent",
    "task_id": f"steward-trigger-{uow_id}",
    "prompt": f"---\ntask_id: steward-trigger-{uow_id}\nchat_id: 0\n---\n\nRun one steward heartbeat cycle. Execute: uv run ~/lobster/scheduled-tasks/steward-heartbeat.py. Call write_result when done.",
    "agent_type": "lobster-generalist",
}
```

This preserves the event-driven semantics (steward runs immediately after completion) while keeping the dispatcher non-blocking. The 3-minute cron fallback still applies if the subagent fails. The `chat_id: 0` convention (golden-patterns.md "silent-drop sentinel") ensures no user notification.

Alternatively, the `_write_steward_trigger` in `wos_completion.py` could write a message type that the dispatcher recognizes and immediately spawns as a background subagent via the normal Task tool, never calling a subprocess in the handler. The `action: "noop"` return with synchronous side effects is architecturally inconsistent with how the rest of `WOS_MESSAGE_TYPE_DISPATCH` works.

**The `action: "noop"` return value is structurally misleading.** `route_wos_message` was designed to return spawn instructions that the dispatcher acts on. Returning `noop` after completing real work (the subprocess call) breaks the expectation that `route_wos_message` is a pure routing function with no side effects. The subprocess is the side effect that the dispatcher's main loop should be performing (via Task), not `route_wos_message`.
