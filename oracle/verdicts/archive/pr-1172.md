## Stage 2: Quality Review

### Reviewer-flagged findings (from task prompt)

The task prompt includes two reviewer findings explicitly passed in:

**Finding 1: STATUS_ORDER includes phantom statuses ("completed", "paused") and is missing real statuses ("failed", "ready-for-executor", "diagnosing")**

This is confirmed by cross-referencing the task file against the production `UoWStatus` enum.

`UoWStatus` members: `proposed`, `pending`, `ready-for-steward`, `ready-for-executor`, `active`, `executing`, `diagnosing`, `blocked`, `done`, `failed`, `expired`, `cancelled`, `needs-human-review`.

`STATUS_ORDER` in the task file: `active`, `ready-for-steward`, `needs-human-review`, `blocked`, `pending`, `proposed`, `executing`, `paused`, `completed`, `done`, `expired`, `cancelled`.

Phantom entries in STATUS_ORDER:
- `"paused"` — not a `UoWStatus` member; no DB rows will ever carry this value
- `"completed"` — not a `UoWStatus` member; "done" is the terminal success state

Real statuses missing from STATUS_ORDER:
- `"failed"` — terminal failure state; a critical operational signal
- `"ready-for-executor"` — in-flight state between steward and executor
- `"diagnosing"` — active investigation state

The practical consequence: if there are failed UoWs in the DB, they will not appear in the ordered section at the top. The `extras` dict captures them (anything not in STATUS_ORDER falls through to sorted extras), so failed UoWs are not invisible — they appear at the bottom after all ordered statuses, sorted alphabetically. The ordering issue means that a failed state does not surface with the priority a pipeline operator would want. This is a display-quality defect, not a data-loss defect.

**Assessment:** This is a real defect but not a correctness blocker. Failed UoWs are not dropped — they appear in the `extras` fallback. The reviewer's flag is accurate. However, the impact is ordering/priority display rather than missing data. Given the PR's stated purpose (quick operational glance) and the fact that `extras` catches all real statuses, this is a non-blocking finding. The ordering fix requires updating `STATUS_ORDER` to remove phantom entries and add `failed`, `ready-for-executor`, and `diagnosing`. This is a one-line task-file edit.

### What this makes easier / harder

**Easier:** Dan can type `/wos` and get a focused pipeline snapshot without navigating to Bisque or running CLI commands. Future `/wos` enhancements (e.g., per-register breakdown, stale UoW count) have a clear extension point in the task file.

**Harder:** STATUS_ORDER is now a maintained list in a task file rather than being derived programmatically from the UoWStatus enum. If new statuses are added to the enum, STATUS_ORDER must be updated manually. This is a latent divergence risk — the same class as the mirror-constant pattern, but for status ordering rather than values. The `extras` fallback mitigates the functional impact (new statuses still appear) but not the ordering gap.

The two reviewer-flagged findings are confirmed as real. Neither is a correctness blocker:

1. STATUS_ORDER phantom entries and missing real statuses: Data is not lost (extras fallback catches everything), but ordering is wrong for the failure case. A one-line task-file fix would close this. At the scope of a read-only display command, this does not constitute a NEEDS_CHANGES gate.

2. No /wos unit test: Consistent with the established posture for /quota and /status (PR #1166 Round 2 APPROVED without this requirement). Not a blocker.

The implementation is structurally sound, crash-safe delivery is correctly implemented, migration numbering is correct, and the feature is aligned with vision.yaml principle-4 and constraint-4.
