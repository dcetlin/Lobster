## Stage 1: Vision Alignment

**Prior entering this review:** This implementation is solving the wrong problem, or solving the right problem in a direction that forecloses better paths.

**Theory of change:** The decisions ledger records choices surfaced via `send_reply` to a durable store. The question is whether migrating it from a flat markdown file to SQLite serves the active project phase (WOS Phase 1 + Vision Object substrate) or is infrastructure refactoring that serves coherence but not the vision's active constraint.

**What would have to be true for this work to be the right path:**
- The decisions ledger must serve an agent-queryable function, not only a human-browsable function — SQLite benefits queries, but a flat file is simpler for humans to audit directly.
- The flat file must be a reliability or query-velocity bottleneck that SQLite addresses meaningfully.
- The table must be in `memory.db` in a way that does not create naming or ownership ambiguity alongside the other data in that database.

**Assessment:** The decisions ledger, as introduced in PR #1043, explicitly routes `decision:` footer blocks to a durable store so that choices made in conversation are structurally preserved. The vision's success criterion "agents can answer 'is this aligned with Dan's intent?' by pointing to a specific field in vision.yaml" requires that prior decisions be queryable, not just scrollable. SQLite gives indexed access by date, category, and task_id — things a flat file cannot serve. The migration is not premature: it converts a write-only artifact into a queryable one, consistent with `principle-4` (wire what exists before building more). The concern about `memory.db` as the target — vs. a dedicated decisions database — is acknowledged below in Stage 2.

**What does this work foreclose:** The flat markdown file is still preserved as a historical artifact (not deleted), so the change is not a one-way door for existing data. Future additions of `category` and `task_id` fields to the insert path would require schema changes, but the columns exist already (nullable). The UNIQUE(date, summary) dedup constraint limits dedup to same-day identical decisions — two identical decisions on different days are both stored. This is a semantic choice, not a structural error.

**What is the opportunity cost:** Minimal. The decisions ledger migration is scoped and bounded. It does not pull attention away from the active WOS reliability work; it advances an existing artifact from write-only to queryable.

**Alignment verdict: Confirmed.** The migration is consistent with `principle-4` and serves the goal of structurally anchored intent. The pattern is not infrastructure refactoring for its own sake.

### DB schema and ownership

The `decisions` table lives in `memory.db` alongside events, sessions, and other data. The `ensure_table()` function uses `CREATE TABLE IF NOT EXISTS` — idempotent and correct. The columns are:

```
id         INTEGER PRIMARY KEY AUTOINCREMENT
date       TEXT NOT NULL
category   TEXT (nullable — not yet populated by the hook)
task_id    TEXT (nullable — not yet populated by the hook)
summary    TEXT NOT NULL
source     TEXT
created_at TEXT DEFAULT (datetime('now'))
```

The UNIQUE index on `(date, summary)` is the dedup key. The hook only populates `date`, `summary`, and `source='hook'`. The `category` and `task_id` columns are present but unused — they are schema-forward columns that reduce future migration cost. This is a reasonable seam-first design choice (golden-patterns.md "seam-first abstraction").

One concern: `memory.db` is a shared database containing events, sessions, and potentially other tables. Adding a `decisions` table here conflates two concerns (decisions-routing artifact vs. event/memory storage). The coupling is real but acceptable at current scale — the table is clearly named, the schema is self-contained, and the `source` column distinguishes origins. This is not a structural defect; it is a deferred concern that becomes relevant when the decisions query surface grows.

### `ensure_table` duplication

Both `decision-router.py` and `scripts/migrate-decisions-ledger.py` define an identical `ensure_table` function. This is deliberate: the migration script is a one-off tool, not a library; the hook is an always-on process. The duplication is the correct trade-off for two scripts with no import relationship. The risk is that if the schema changes in the hook, the migration script's copy becomes stale — but since the migration script runs once and is idempotent, the stale risk is bounded.

### `ENTRY_RE` in the backfill script

The pattern `r"\*\*(\d{4}-\d{2}-\d{2})\*\*\s+—\s+(.+)"` with `re.DOTALL` is correct for the flat-file format `**YYYY-MM-DD** — text`. The `re.DOTALL` flag means `.+` in group 2 matches across newlines — so multi-line decision text is captured. This is the correct behavior. The `strip()` applied to `m.group(2)` removes trailing newlines from the captured text.

One edge case: the regex matches the first `---` separator boundary but `content.split("\n---\n")` does the splitting before the regex runs. `parts[1:]` skips the header block. This is correct for the documented flat-file format.

### Encoded Orientation check (constraint-3)

This PR does not change behavioral defaults, agent identity, system constraints, or decision-making rules. It modifies the write path of an existing PostToolUse hook to use SQLite instead of a flat file. Not an Encoded Orientation decision. Constraint-3 does not apply.

### What would break without the key decisions made here?

- Without `CREATE TABLE IF NOT EXISTS`: the hook would fail on first invocation if the table didn't exist, requiring an explicit setup step.
- Without `INSERT OR IGNORE`: duplicate decision entries on the same day with the same text would raise a UNIQUE constraint error.
- Without the UNIQUE index: dedup would not work without the `OR IGNORE` clause having a constraint to trigger against.
- Without `conn.commit()` inside the context manager: SQLite's context manager does auto-commit on success, but the explicit `commit()` makes the behavior legible.

### What failure modes exist?

1. **`memory.db` absent:** `sqlite3.connect()` creates the file. No failure — the DB is created on first connection.
2. **Concurrent writes:** If two hook invocations fire simultaneously (unlikely given sequential tool calls, but possible in theory), SQLite's writer lock serializes them. No data corruption.
3. **Large summary text:** The `UNIQUE(date, summary)` index is unbounded — very long decision text is stored and indexed without truncation. Not a current concern but a future scaling note.
4. **Flat-file entries with multi-line decision text:** `ENTRY_RE` with `re.DOTALL` captures multi-line summaries. However, `content.split("\n---\n")` splits on the separator, so a multi-line decision text that accidentally contains `\n---\n` would be truncated. This is an edge case for the one-time migration script only, and the historical flat-file format is unlikely to contain embedded separator sequences.

### Pattern check: seam-first abstraction (golden-patterns.md)

The nullable `category` and `task_id` columns are schema-forward seams — they name the future query dimensions without requiring the hook to populate them today. This follows the seam-first pattern: minimal implementation now, but the columns mark where future change will happen. Correctly applied.

### What this makes easier/harder

**Easier:** Querying decisions by date range, category, or task_id. Future tooling (e.g., a `registry_cli decisions` command) can join against these columns without a schema migration.

**Harder:** Human-browsable audit of decisions is now a SQL query rather than a markdown file read. The flat-file is preserved but no longer authoritative — a maintainer who opens `decisions-ledger.md` will see stale data from before the migration. A tombstone comment in the flat file (or its deletion) would prevent future confusion, but this is a cosmetic concern, not a structural one.

### Verdict

The implementation is correct. No migration number conflicts exist. The upgrade path is sound. The `ensure_table` duplication is acceptable. The bare python3 convention follows existing hook registration patterns. The absence of automated tests is a pre-existing gap from PR #1043, not introduced here.
