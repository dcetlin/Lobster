### Stage 2: Quality Review

**Encoded Orientation check (constraint-3):** The subagent prompt update in `dispatcher_handlers.py` instructs executors to accumulate and report `token_usage`. This is advisory and opt-in ("Omit token_usage if you did not track it"). It does not change behavioral defaults, gate any automated action, or affect routing decisions. The field's absence produces NULL — not an escalation, not a different code path. This is not an Encoded Orientation decision under constraint-3. No prior logged decision required.

#### What it does and whether it does it correctly

**`complete_uow` now sets `completed_at`:** Previously, `complete_uow` updated `status` and `updated_at` but never set `completed_at`. This PR adds `completed_at = ?` to the UPDATE statement, using the same `now` timestamp as `updated_at`. This is correct. The fix is load-bearing for wall_clock_seconds.

**`COALESCE(?, token_usage)` semantics:** In SQLite, `COALESCE(?, token_usage)` where `?` is bound to Python `None` binds as SQL NULL. `COALESCE(NULL, existing_value)` returns `existing_value`. This is correct — a subsequent `complete_uow` call without `token_usage` will not overwrite an existing non-NULL value. The test `test_existing_token_usage_not_overwritten_by_null` exercises this directly with a pre-set value.

**`wall_clock_seconds` julianday expression:** The expression is `(julianday(completed_at) - julianday(started_at)) * 86400.0`. Parenthesization is correct — the subtraction happens before the multiplication. The PR description contains a typo (`julianday(completed_at) - julianday(started_at) * 86400` without outer parens), but the actual code is correctly parenthesized. Same-second timestamps yield 0, not NULL — the implementer noted this. Zero is a valid, interpretable value (sub-second execution) and not a semantic error. The `CASE WHEN` guard correctly returns NULL when either timestamp is NULL.

**`token_usage` validation:** `int(_token_usage_raw) > 0` rejects zero and negative values. The float-to-int cast (`int(float_value)`) is intentional — it handles cases where the Claude API returns a float count. This is safe since token counts are always whole numbers semantically.

**Propagation chain completeness:** The full chain `handle_write_result → _maybe_complete_wos_uow → maybe_complete_wos_uow → registry.complete_uow` is traced in the diff. Each link passes `token_usage=token_usage` as a keyword argument. No existing call sites are broken — all callers continue to work without the parameter.

**`executor_uow_view` updated:** `token_usage` is added to the view. `completed_at` and `started_at` were already in the view. Wall-clock computation at query time is correct — no stored derived column needed.

**Migration 0015:** Adds `token_usage INTEGER NULL`. The Registry `_run_migrations` auto-applies all numbered migrations. No upgrade.sh step is needed or expected for SQL-level migrations in this codebase. Migration is non-destructive (ALTER TABLE ADD COLUMN with NULL default).

#### Test coverage

The test file `test_outcome_telemetry.py` covers:
- `complete_uow` with token_usage stored (integer equality check against production DB)
- `complete_uow` without token_usage leaves NULL (correct semantic)
- COALESCE: existing non-NULL not overwritten (pre-set via direct SQL, then complete_uow with None)
- token_usage present in audit note when provided; absent (not null) when not provided
- `completed_at` set by `complete_uow`
- wall_clock_seconds computed correctly with ±2s tolerance
- wall_clock_seconds NULL when `started_at` is NULL
- `maybe_complete_wos_uow` forwards token_usage to registry (via patch of side effects)
- `aggregate_token_usage`: sum, all-null, partial, empty
- `compute_wall_clock_stats`: two-UoW avg, null exclusion, all-null, empty
- `format_digest`: token line present/absent, wall-clock line present/absent, both, zero-UoW window

**Named constants assessment (mirror-constant pattern, learnings.md recurring):** The test file declares `SMALL_TOKEN_COUNT = 1_234`, `LARGE_TOKEN_COUNT = 98_765`, `WALL_CLOCK_60S = 60` etc. These are fixture-data constants — representative values chosen for readability, not mirrors of production threshold constants. There are no named production constants for minimum token value (the production code uses the inline expression `int(x) > 0`). `TOKEN_USAGE_MIN_VALID = 1` is the test's name for this threshold, but there is no production constant to mirror. The mirror-constant pattern does not apply here because no production named constant exists for these values.

**`test_wos_metabolic_digest.py` updated:** The existing test fixture schema now includes `completed_at` and `token_usage INTEGER NULL` columns. This is correct.

#### What patterns this introduces

The PR introduces the pattern of computing derived telemetry fields at query time rather than storing them. This is correct for `wall_clock_seconds` — storing a derived value would create a synchronization obligation between the stored value and the source timestamps. The existing `steward_cycles` is already a counted field; `wall_clock_seconds` is a cleaner approach because it cannot diverge from its source data.

The audit note now conditionally includes `token_usage` using `{**({"token_usage": token_usage} if token_usage is not None else {})}` — presence/absence pattern rather than null value. This is a correct signal-encoding choice: the audit log should distinguish "not reported" from "reported as 0 tokens."

#### What this makes harder

Nothing is foreclosed. The metabolic digest gracefully omits telemetry lines when no data is available. Token accumulation remains the subagent's responsibility with no enforcement — this means the data set will be sparse initially, but sparse is better than absent.

#### Failure modes

The only notable failure mode is the one the implementer identified: if subagents never accumulate `token_usage`, the metabolic digest lines simply do not appear. This is not a defect — it is correct graceful degradation. The field will populate as subagents adopt the instruction.
