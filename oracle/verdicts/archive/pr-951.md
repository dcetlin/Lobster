### Stage 2: Quality Review

**Scope of actual delta from main:** 4 files — `cultivator.py`, `registry.py`, `test_file_scope_extraction.py`, `test_registry.py`. The `sys.dispatcher.bootup.md` diff shown by `gh pr diff` reflects commits already on main before this PR's branch point. The actual delta to bootup.md from main is zero.

**`_extract_file_scope()` in cultivator.py**

Pure function, correctly typed, correctly returns `list[str] | None`. Two-pass extraction: backtick-wrapped tokens filtered by prefix or extension. Design decisions are sound:

- Returns `None` (not empty list) for no-match — preserves null-scope exclusivity as safe default
- Prefix list (`src/`, `tests/`, `scripts/`, `scheduled-tasks/`, `oracle/`, `.claude/`) is conservative and matches the actual repo structure
- Extension fallback catches files outside known prefixes (e.g., `config/settings.yaml`)
- `sorted(set(paths))` produces deterministic, deduplicated output
- `None` body guard is correct

**Failure mode analysis:**
- False positives (wrong file extracted): produce extra serialization, not spurious parallelization. Safe by design — the null-scope exclusivity default means the cost of over-classification is a missed parallelization opportunity, not a race condition.
- Absolute paths (`/home/lobster/lobster/src/foo.py`): would be extracted via extension match. Harmless — absolute and relative representations of the same file don't string-match, so no conflict would be detected across UoWs. Creates noise in the field but doesn't cause incorrect behavior.
- URL-shaped tokens (`https://github.com/foo/bar`): no recognized extension, no recognized prefix, not extracted. Safe.

**registry.py — parameter threading**

Verified against the PR #717 pattern ("Public method accepts new parameter but does not forward it to the private implementation method"). The threading is complete:
- `upsert()` → `_upsert_typed()`: parameter forwarded
- `_upsert_typed()` → INSERT: serialized as `json.dumps(file_scope) if file_scope is not None else None`, then inserted as `file_scope_json`

Column/value tuple alignment verified: 19 columns, 16 `?` placeholders + 3 literal values, 16-element tuple. Correct.

JSON serialization is the correct choice for a `list[str]` field in SQLite. `shard_dispatch.py` deserializes this via `json.loads`; the format is stable.

**Tests**

`TestFileScopeUpsert`: three tests cover JSON storage, NULL default, and multi-path round-trip. Exercises the full upsert path through to DB read. No gap in the test of the UPDATE path — the UPDATE branch in `_upsert_typed` returns `UpsertAlreadyExists` without touching file_scope, which is the correct behavior (file_scope set at creation only).

`TestExtractFileScope`: 12 cases cover the key branches — prefix matching, extension matching, dedup, None body, empty body, no-slash tokens, unknown prefix with no extension. Test coverage is adequate for the function's branching structure.

Minor omission: no test for URL-shaped tokens or absolute paths. These are edge cases where the function's behavior is safe and predictable from reading the code, so the omission is acceptable.

**Encoded Orientation check (constraint-3):** This PR does not change behavioral defaults, system constraints, agent identity, or decision-making rules. The file_scope wiring is additive infrastructure. The dispatch gate in `shard_dispatch.py` is untouched. No Encoded Orientation decision is required.

**What this makes easier:** Future UoWs will have file_scope populated, allowing the shard dispatch gate to detect genuine file conflicts and permit parallelization of non-overlapping work. The PR #904 parallelism feature becomes live in production.
