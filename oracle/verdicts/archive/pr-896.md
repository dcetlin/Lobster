## Stage 2: Quality Review

### Prior gap tracking (Round 2 → Round 3)

**Gap 1: Encoded Orientation without prior logged decision** — **Addressed (Round 2).** Closed.

**Gap 2: Escalation multi-fires for patterns already above ESCALATION_THRESHOLD** — **Addressed in Round 3.**

The fix changes the escalation condition from `recurrence_count >= ESCALATION_THRESHOLD` to `recurrence_count == ESCALATION_THRESHOLD - 1`. With `ESCALATION_THRESHOLD = 2`, the condition is `recurrence_count == 1`.

**Is the first-crossing guard correct?**

Yes. `smell.recurrence_count` holds the pre-run value from `smell-patterns.yaml`. `write_back_recurrence_counts()` runs before the escalation block and increments the YAML on disk, but the in-memory `SmellDetection` frozen dataclass retains the pre-run count. The condition `recurrence_count == ESCALATION_THRESHOLD - 1` is therefore equivalent to "the in-memory (pre-run) count is exactly one below the threshold" — meaning this run's write-back will push the YAML count to threshold. Escalation fires exactly once at the crossing.

For the three existing above-threshold patterns (bare_python3=4, rolling_summary=3, health_probe=2): their in-memory counts are 4, 3, and 2 respectively. None satisfy `recurrence_count == 1`. No escalations fire. This correctly handles the patterns that triggered the Round 2 gap.

**New edge case: count reset between runs.**

If a smell is detected on run N (count 0→1, written back as 1), not detected on run N+1 (count reset to 0), then detected on run N+2 (count 0→1, crossing fires again) — escalation fires a second time. This is semantically correct: a smell that clears and recurs is a new event warranting a new crossing notification. The reset-between-runs path does not create unbounded escalation; it creates one notification per recurrence epoch. This is the correct behavior.
