## PR #1066 — fix(wos): off-by-one in execution_attempts retry cap

### Stage 1: Right problem?

Yes. This PR fixes two related issues in the WOS retry accounting:

1. **Off-by-one (the named fix):** `> MAX_RETRIES` → `>= MAX_RETRIES`. The old condition required `execution_attempts` to reach `MAX_RETRIES+1` (4) before escalating; the correct behavior is to escalate when confirmed attempts reach the cap (3). Straightforward fix.

2. **Deeper root cause (the broader fix):** The retry counter (`retry_count`) was treating infrastructure kill events (orphan recovery) identically to genuine execution failures. Evidence: 19 UoWs in the 2026-04-26 failure cohort had `lifetime_cycles = 0` — none completed an execution arc — but all were escalated to needs-human-review after 3 infrastructure kills. The PR adds `execution_attempts` — a separate counter that increments only when `return_reason` is not an orphan classification. `MAX_RETRIES` gates on `execution_attempts` instead of `retry_count`.

Both fixes together are the right repair. The first fix alone would be incomplete.
