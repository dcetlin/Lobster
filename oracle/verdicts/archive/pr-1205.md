## Stage 1: Vision alignment

**Prior entering review:** This implementation is solving the wrong problem, or solving the right problem in a direction that forecloses better paths.

**Theory of change:** The PR asserts that the root cause of oracle merge-gate failures is a write/read path divergence between where oracle agents write verdict files (`~/lobster-workspace/oracle/verdicts/`) and where `pr-merge-gate.py` reads them (`~/lobster/oracle/verdicts/`). The proposed fix is a symlink that makes the workspace path transparently route to the canonical repo location.

**What would have to be true for this to be the right path:**

1. The write path divergence is real and has caused actual merge failures. Confirmed: the PR states "every merge agent was tripping over this gap," and the workspace oracle directory already had a `.deprecated` file (dated 2026-04-25) indicating this was a known drift issue.

2. The symlink is architecturally preferable to fixing the write path in agents or fixing the read path in the gate. This holds because: fixing agent write paths requires auditing all oracle agent definitions (distributed fix surface); fixing the gate read path would make the canonical location secondary (wrong direction); the symlink collapses both paths to one (idiomatic Unix).

3. The workspace oracle directory carries no content that would be destroyed by `rm -rf`. Confirmed: the PR description notes the workspace oracle was already deprecated with unique verdicts already in `~/lobster/oracle/verdicts/archive/`. Live system check confirms the symlink is already in place.

**What this forecloses:** The symlink makes `~/lobster-workspace/oracle/` permanently equivalent to `~/lobster/oracle/`. Any future scenario where these should diverge (e.g., a multi-instance setup with different workspaces pointing to different repos) would require unwinding the symlink. This is an acceptable trade-off for this architecture's scale and is explicitly described in the PR.

**Opportunity cost:** The four changes in this PR (symlink migration, label limit, sweep cleanup, learnings entry) are independent. They could be four PRs, but they are small and cohesive enough that bundling them at the process-hygiene level is defensible. The PR description does not obscure this: each change is clearly described. This is not the feature-bundling anti-pattern from PR #1148 (which bundled unrelated workstreams with a placeholder description). The distinguishing factor: all four changes here address the same class of problem (sweep and oracle process reliability) and emerged from the same Night 11 sweep.

**Alignment verdict:** Confirmed. Vision.yaml `principle-1` (proactive resilience over reactive recovery) and `principle-4` (wire what exists before building more) both anchor this work directly. The PR closes gaps between existing systems rather than building new infrastructure.

**oracle vocabulary prior application:** The "canonical placement must be obvious from first encounter" golden pattern (2026-04-25) was used as a positive prior: the symlink is a structural choice that makes the write path unambiguous without requiring agents to know two canonical locations. It was weighed against the "path fix completeness" learning (2026-04-20, PR #796), which warns that enumerating only the observed failure form misses other forms. I checked whether agent oracle write paths are encoded in places beyond the workspace path — they appear as prose instructions to write to `~/lobster-workspace/oracle/` in agent definitions. The symlink approach handles these without requiring each to be hunted. This weighted the decision toward APPROVED rather than flagging incomplete fix scope.

## Stage 2: Quality review

### Migration 116 (upgrade.sh)

The three-branch if/elif/else is structurally correct:
- `[ -L "$_ws_oracle" ]` — symlink already exists: skip (idempotent).
- `[ -d "$_ws_oracle" ]` — regular directory: `rm -rf` then `ln -s`.
- else — missing: `ln -s`.

**Concern: `rm -rf` on a directory with verdict files.** The PR description states the workspace oracle was marked deprecated and unique content had already been moved to `~/lobster/oracle/verdicts/archive/`. The live system confirms the symlink is already in place (created manually before the PR). However, the migration code does not check whether `$_ws_oracle` contains files before `rm -rf`. On a fresh instance where the workspace oracle has active verdict files that have not been moved, `rm -rf` would silently delete them.

Severity assessment: **low**, not a blocker. The migration runs on a post-manual-verification system where the content was already confirmed safe to remove. The PR description explicitly calls this out. For future instances, verdict files are runtime artifacts not committed to git — their loss is recoverable by re-running oracle. The migration comment explains the intent clearly. This is a one-time migration that handles existing installs; the "already a symlink" branch makes it idempotent for all subsequent runs.

**Variable scope:** `_ws_oracle` and `_repo_oracle` are declared `local`, consistent with upgrade.sh conventions. `$WORKSPACE_DIR` and `$LOBSTER_DIR` are defined at lines 47-48. The variables resolve correctly.

**`migrated` counter increment:** Both the `rm -rf` + `ln -s` branch and the bare `ln -s` branch increment `migrated`. This is correct — both constitute a real migration action. The skip branch does not increment, consistent with the pattern throughout upgrade.sh.

**Dry-run gap:** Other migrations in upgrade.sh check `if [ "$DRY_RUN" = true ]` before destructive operations. Migration 116 does not. The `rm -rf` would execute in dry-run mode. This is a minor inconsistency with the upgrade.sh convention, but not a functional defect for the live system — it is a process hygiene issue.
