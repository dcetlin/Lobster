## Round 1 — 2026-05-21

### Stage 1: Vision Alignment

**Epistemic prior entering review:** This implementation is solving the wrong problem, or solving the right problem in a direction that forecloses better paths.

**Prior patterns consulted:**
- Learnings: None of the indexed failure modes (state-machine mismatch, boolean frozenset over-detection, recovery gate inverting principle-1) apply here. This is a threshold and retry-budget change, not a classification or interface decision.
- Golden patterns: No direct match. The `_strip_idempotent_add_columns` approach in migrate.py is related to the "structurally enforced correctness invariant" spirit of the claim_and_ack pattern — SQLite's lack of IF NOT EXISTS is handled at the Python layer rather than caught as an error after the fact.

**Vision alignment analysis:**

The work addresses a concrete operational failure: the WOS executor dispatches design/arch UoWs that legitimately run 10–30 minutes; the startup_sweep orphan threshold at 600s was detecting these as stalled before they had a chance to emit a heartbeat. The vision's `fundamental_intent` is a system that routes work according to intent without atrophying Dan's judgment. A system that reflexively kills its own long-running work contradicts this: it forces re-execution overhead and unreliability at exactly the layer that is supposed to operate without screen attention.

**What would have to be true for this to be wrong?** The 600s threshold would need to be correct — i.e., no legitimate UoW should take longer than 10 minutes to emit a first heartbeat. That is false for design UoWs per the existing evidence. Alternatively, there would need to be a cheaper solution (e.g., per-UoW estimated_runtime-based thresholds already existed for the `active` population). The PR author notes that `get_stale_heartbeat_uows` catches most real orphans within 5–8 minutes — the 600s threshold was redundant insurance for the pre-heartbeat window, and 1800s still provides that insurance against true crashes while stopping false kills.

**What does this foreclose?** True orphans that crash before writing any heartbeat will now go undetected for 30 minutes instead of 10. This is the correct tradeoff given that the heartbeat mechanism handles the 99% case, and the `claimed_until` visibility-timeout remains as the authoritative stall-detection path.

**Opportunity cost:** Low. No competing uses of this engineering time are more aligned with the current active focus.

**Alignment verdict:** Confirmed. This work directly serves the vision by making the infrastructure resilient to its own legitimate workload.

### Stage 2: Quality Review

**Does it do what it claims?**

Yes. The PR does exactly what it says:
1. `_EXECUTING_ORPHAN_THRESHOLD_SECONDS` raised from 600 to 1800 in `startup_sweep.py`.
2. `steward-heartbeat.py` reads `executing_orphan_threshold_seconds` from `wos-config.json` via `read_wos_config()` and passes it to `run_startup_sweep()`. Fallback is the constant default (1800s). The pattern mirrors the existing `execution_enabled` config-read in the same file.
3. `registry.py` adds `orphan_retry_count` field to `UoW` and `retry_orphan_kill()` method.
4. `steward.py` adds `ORPHAN_KILL_RETRY_BUDGET = 2` and retry gate in `_process_uow` for `orphan_kill_during_execution`.
5. `migrate.py` adds `_strip_idempotent_add_columns` to handle SQLite's lack of `ALTER TABLE ... ADD COLUMN IF NOT EXISTS`.
6. Migration `0026_orphan_retry_count.sql` adds the column.

**Correctness of the retry-budget gate:**

The guard at the `orphan_kill_during_execution` branch correctly excludes `orphan_kill_before_start` and legacy `executing_orphan` from the retry path — both fall through to permanent failure, which is correct: before-start has no partial work to retry, and `executing_orphan` is the pre-heartbeat classification that predates the kill taxonomy.

The `dry_run=True` path is correctly handled: the `Surfaced(condition="orphan_retry_scheduled")` is returned without the DB write, so dry-run accurately reports what would happen without side effects.

**`_strip_idempotent_add_columns` correctness:**

The regex `ALTER\s+TABLE\s+(\w+)\s+ADD\s+COLUMN\s+(\w+)` correctly captures table and column names for the simple single-column migration format used throughout this codebase. The split-on-semicolon approach is appropriate for migrations that are simple DDL statements. One edge case: if a migration SQL string contained a string literal with a semicolon, the split would misparse it — but all migrations in this codebase are pure DDL, not DML with embedded strings, so this is safe.

**What would break without the key decisions?**

Without the 1800s threshold change: design/arch UoWs would continue being killed before writing their first heartbeat, masking as orphans and re-executing from scratch every 10 minutes.

Without the `retry_orphan_kill` budget: a UoW killed by context compaction mid-execution would be permanently failed on first detection, discarding all partial progress.

Without `_strip_idempotent_add_columns`: re-running migrations on a database that already has the `orphan_retry_count` column would throw "duplicate column name" and leave the migration partially applied.

**Failure modes introduced:**

One narrow case: if `wos-config.json` contains `executing_orphan_threshold_seconds` as a float (e.g., `1800.0`), `int()` truncates correctly. If it contains a non-numeric string, `int()` raises `ValueError` and the sweep would crash at Phase 1 startup. A defensive try/except around the `int()` cast with fallback to the constant would make this more robust, but the config key is internal/tooling-managed and not user-facing, so the risk is low and does not warrant blocking approval.

**Patterns introduced:**

- `wos-config.json` as a tuning surface for steward behavior (extends existing `execution_enabled` pattern). This is a correct extension: the config file already carries operational state; adding a threshold is consistent.
- `orphan_retry_count` as a separate counter distinct from `retry_count` (which tracks steward-driven retries) and `execution_attempts`. The naming is clear and the separation is correct — orphan kills are environmental events, not execution quality signals.

**Encoded Orientation check (constraint-3):**

Does this PR constitute an Encoded Orientation decision? It changes behavioral defaults in the steward (how quickly orphans are killed, how many times they retry). The change is traceable to the operational problem (design UoWs getting false-killed), which is a concrete system behavior issue within the active development phase intent. No prior logged decision is required here — this is a calibration of an existing mechanism, not a new decision-making rule. The OODA register is Constrained (known situation class, known response pattern, threshold adjustment). constraint-3 is satisfied.

**Verdict:** APPROVED. The implementation is correct, the tradeoffs are sound, and the scope is tightly bounded to the stated problem.
