## Stage 2: Quality review

### What was changed

Three targeted changes to `footer_audit.py`:

1. **`SIDE_EFFECTS_NONE_RE` removed** — the module-level regex that matched `^side-effects:\s*none$` is deleted. Comment updated to say the bare null form is banned.

2. **`WRONG_LABEL_PATTERNS` extended** — two entries added:
   - Bare form: `re.compile(r"^side-effects:\s*none\s*$", re.MULTILINE | re.IGNORECASE)` — matches bare `side-effects: none` lines.
   - Block form: `re.compile(r"```side-effects:\s*\nnone\s*\n```", re.DOTALL | re.IGNORECASE)` — matches fenced block with `none` as the only content.

3. **`format_report()` hint updated** — the canonical-forms "No effects" line now reads `omit the footer entirely (side-effects: none is banned)` instead of the stale `side-effects: none (bare line)`.

### Regex correctness

Both new patterns are regex-identical to the enforcement patterns already in `signal-footer-check.py`:
- `signal-footer-check.py` line 38: `SIDE_EFFECTS_NONE_BARE_RE = re.compile(r"^side-effects:\s*none\s*$", re.MULTILINE | re.IGNORECASE)`
- `signal-footer-check.py` line 39: `SIDE_EFFECTS_NONE_BLOCK_RE = re.compile(r"```side-effects:\s*\nnone\s*\n```", re.DOTALL | re.IGNORECASE)`

The PR's claim of alignment is confirmed. The regexes match the same inputs with the same flags.

**Minor shared gap (not a PR defect):** A block body with leading whitespace (`  none  \n`) does not match either regex. This gap exists identically in both `signal-footer-check.py` and the new `footer_audit.py` entries — consistent behavior, not a divergence introduced by this PR. Not a blocker.

### Interaction analysis: `has_valid_footer` vs. `WRONG_LABEL_PATTERNS`

A subtle but correct interaction exists:

- The block form `\`\`\`side-effects:\nnone\n\`\`\`` IS matched by `SIDE_EFFECTS_BLOCK_RE` (which matches any fenced block with `side-effects:` label, regardless of content). So `has_valid_footer()` returns `True` for this form.
- However, `audit_messages()` checks `detect_wrong_labels()` first. If `wrong_labels` is non-empty, the `has_valid_footer` path for the missing-footer check is skipped entirely (line 103 guard: `if not wrong_labels`).
- Result: the block-none form is correctly reported as a `wrong_label` violation, not as a `missing_footer` violation. No double-reporting, no false negatives.

For the bare form (`side-effects: none`):
- `has_valid_footer()` now returns `False` (the `SIDE_EFFECTS_NONE_RE` branch was removed).
- `WRONG_LABEL_PATTERNS` catches it first.
- `audit_messages()` skips the missing-footer path due to the `if not wrong_labels` guard.
- Correctly reported as `wrong_label`.

The control flow is correct. All four cases the PR claims to handle (bare flagged, block flagged, `has_valid_footer` returns False for bare form, canonical still passes) are verified.

### What this makes easier / harder

**Easier:** Audit reports now surface `side-effects: none` violations that were silently passing before. The convention is enforced by two independent mechanisms that agree.

**Harder:** Nothing. The change is additive (two WRONG_LABEL_PATTERNS entries) and subtractive (one `has_valid_footer` branch removed). Both directions reduce the gap, not expand it.
