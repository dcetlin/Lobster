## Stage 2 — Quality Review

### What the document actually is

The spec is a design document for three changes: (1) agent-side heartbeat writes (fixing the sidecar-masking root cause), (2) a new /dispatch handler for dispatcher-direct UoW spawning (Posture 2, a new capability), and (3) claimed_until TTL disposition (cleanup). It is filed as DRAFT, dated 2026-05-19, and notes "status: DRAFT" in the document header. This satisfies the awareness-of-status requirement from the design-spec-blockquote golden pattern — the document does not present itself as describing current behavior.

**However:** the golden pattern requires a blockquote placed before the introductory prose that explicitly states (a) this is a design spec, not current behavior, (b) what the current production behavior IS with a pointer to the authoritative source, (c) which implementation issues close the gap, (d) a direct instruction for any agent using this as orientation. The "status: DRAFT" in the header metadata satisfies (a) partially but not (b), (c), or (d). An agent reading this document as orientation to the current WOS execution model would form an incorrect understanding — specifically for Posture 2 (dispatcher-direct), which does not yet exist. This is a mild finding, not a blocker, because the spec is not a bootup document and is unlikely to be consumed by agents as orientation. I note it but do not block on it.

### Encoded Orientation check (constraint-3)

Does this PR constitute an Encoded Orientation decision? It is a documentation PR that encodes a behavioral architecture — specifically, it names a new dispatcher command (/dispatch), specifies gate conditions for it, and prescribes how the dispatcher should handle it (soft gates, warns Dan, spawns background Agent). This is design documentation, not a change to CLAUDE.md or bootup files. The document itself does not change system defaults; it describes proposed changes for implementation in follow-on PRs. No code in dispatcher_handlers.py or bootup docs is modified. The Encoded Orientation check requires that changes to "behavioral defaults, system constraints, agent identity, or decision-making rules in a durable way" have a prior logged decision. This document does not durably change behavior — it specifies how follow-on PRs should change behavior. The check does not fire here.

### Root cause diagnosis — is it correct?

The spec's root cause (§1.1) states: UoW agents do not reliably write heartbeats to the registry while executing; the sidecar compensates by writing heartbeat_at for every active/executing UoW, masking dead agents. This is confirmed by the production incident (5 executing UoWs with no live agents, 22–24h blockage). The fix direction (make agent-originated heartbeats the primary signal, use uow_heartbeat_log to distinguish sidecar-only writes from agent writes) is structurally correct. The token_usage column in uow_heartbeat_log as the discriminator (sidecar writes have NULL token_usage; agent writes have non-NULL) is a clean distinguishing signal that avoids adding a new column.

**One gap in the root cause section:** The spec correctly identifies sidecar-masking as a failure mode but does not address the complementary question: why don't agents write heartbeats now? The current UoW agent prompt template has a heartbeat contract (§2.2 mentions it is "advisory"). If agents are failing to comply with an advisory contract, the fix is to make it structural (Change 1a). But if agents are being killed before they can write heartbeats (context compaction, session TTL), the prompt-template change alone is insufficient — the agent never reaches the heartbeat write. The spec acknowledges this indirectly ("When context compaction kills a dispatcher session mid-execution") but does not explicitly address whether the startup heartbeat (the first checkpoint, "before any other work") is reliably writable before compaction could kill the session. This is a design question, not a spec quality issue — the startup heartbeat-write-before-any-work instruction in §2.2 is the correct structural defense against pre-heartbeat kills.

### Migration path — does it hold up?

The five-step migration path is sound. Steps are independent, observable, and carry clear rollback semantics. The observability checkpoints in §5.3 are specific and verifiable. The ordering (prompt template first → observation loop second → sidecar demotion third → dispatch handler fourth → TTL shortening fifth) is correct — you cannot validate the observation loop's corpse detection until agents are writing heartbeats, and you cannot demote the sidecar until the observation loop is validated. The "at least 7 days with no false stalls" gate before sidecar demotion is appropriately conservative.

**Minor sequencing note:** §5.1 Step 4 says "Add to WOS_MESSAGE_TYPE_DISPATCH if appropriate, or handle as a text command." This is an undecided design question embedded in the migration step. The spec should take a position here, because the decision determines whether the handler lives in the dispatcher's message-type routing table or in its text-command handler. Both are implementable, but leaving it as "if appropriate" creates ambiguity for the implementing agent. This is a mild finding — the spec is a design doc, not an implementation ticket, and the decision can be made at implementation time.

### Internal consistency

The spec is internally consistent. The coexistence model (§1.3) correctly notes that Postures 1 and 2 converge on the same agent execution path and the same result file schema. The gate condition table (§3.3) correctly distinguishes hard gates (Posture 1) from soft gates with Dan-decides semantics (Posture 2). The result routing difference (chat_id=0 for Posture 1, admin chat_id for Posture 2) is consistent with the silent-drop sentinel golden pattern and the crash-safe two-step delivery golden pattern.

The §2.3 detection query (conceptual SQL) is correct in intent — it queries for UoWs in executing status whose uow_heartbeat_log contains no post-dispatch agent-originated rows. The 300-second grace window matches the dispatch-window-seconds learning (learnings.md PR #968). The status filter uses `IN ('active', 'executing')` — this is correct given the current status lifecycle, though the spec should note whether 'active' is a distinct status or an alias.

### Named failure patterns from learnings.md — active prior check

- **Mirror-constant pattern (PRs #696, #800, #967, #970, #973, #974, #1211):** Not applicable — this is documentation, no constants defined.
- **Advisory enforcement of structural invariant (PR #1148):** The spec explicitly addresses this — Change 1a upgrades the heartbeat contract from "advisory to structural." The spec is applying the correct remediation for the failure pattern the learnings.md entry named.
- **Feature-bundling (multiple PRs):** The spec bundles three changes (heartbeat fix, dispatcher-direct, claimed_until disposition) in a single document. This is intentional and appropriate for a design spec — the three changes are architecturally related and need to be understood together. The spec explicitly labels each change and makes their independence explicit. This is not the harmful bundling pattern from learnings.md (which applies to code PRs where unrelated workstreams merge review surfaces).
- **Design-spec-blockquote golden pattern:** Partially satisfied. DRAFT status is declared. The four-element blockquote (status, current source, implementation track, orienting instruction) is not present in the formal form, but the spec's §5 migration path serves the same function for items (c) and (d). The absence of a formal blockquote pointing to the authoritative current-behavior source is a mild gap.
- **Instruction-accumulation pattern (learnings.md 2026-04-22):** The spec explicitly identifies this — the prior fix to the executor slot threshold (5→20 via PR #1215) is named as "a guard rail, not a solution." The spec is applying the discriminator-improvement principle (vision.yaml principle-5) rather than adding more rules.

### What this makes easier / harder

**Easier:** The migration path gives implementing agents a clear, sequenced roadmap. The observability checkpoints prevent each step from being treated as complete without verification. The non-goals section (§6) is explicit and protective — it prevents scope creep during implementation, particularly the temptation to make the dispatcher an orchestrator-of-orchestrators.

**Harder:** Posture 2's existence in the spec creates pressure to implement it alongside the critical Change 1, even though Change 1 is the urgent operational fix and Posture 2 is a new capability. The spec explicitly says "independent and can be shipped separately" — but design documents have a tendency to be treated as a unit. This is a soft concern, not a spec quality issue.

### Verdict rationale

The spec is well-constructed. It identifies a real, documented root cause (not a hypothetical). Its migration path is sequential, observable, and defensively ordered. Its non-goals section actively limits scope. The document is self-aware about its DRAFT status. The three-change structure is internally consistent and the changes share the correct registry and execution path. No Encoded Orientation decision is encoded in a durable way by this document alone. The mild gaps (formal design-spec-blockquote blockquote absent, one undecided routing question in Step 4) do not rise to NEEDS_CHANGES for a design document at DRAFT status that will drive implementation PRs which will themselves be oracle-reviewed.

**VERDICT: APPROVED**
