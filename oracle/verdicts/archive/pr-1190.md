## Stage 1 — Vision Alignment

**Prior entering review:** This implementation is solving the wrong problem, or solving the right problem in a direction that forecloses better paths.

**Theory of change:** During hygiene run `uow_20260514_715dd1`, executors auto-closed PRs carrying `awaiting-sign-off` labels. No guard existed. The fix adds: (1) a `HUMAN_GATE_LABELS` frozenset, (2) an `is_pr_human_gated()` function, and (3) a mandatory PR-close guard section injected into every executor prompt.

**What would have to be true for this to be the right path:** (a) Executor PR-close is a legitimate automated action in normal flow — only gated-PR cases need protection. (b) Advisory prompt-layer enforcement is sufficient for a safety invariant. (c) The label check is genuinely interpretive enough to require LLM reasoning rather than deterministic code.

**Stage 1 finding — the adversarial prior is partially confirmed:**

The PR is solving the right problem (protecting human-gated PRs from automated close) but is doing so at the wrong enforcement layer. This is the same structural error named in PR #1148: adding if-then logic as prompt instructions rather than as code.

Vision.yaml `operating_principles[principle-3]`: "Determinism over judgment for conditionals. If-then logic and field checks are code, not LLM instructions. Use LLMs where genuine interpretation is required."

The PR-close guard is pure if-then logic: does `gh pr view` return any label in the gate set? If yes, skip. This is a deterministic label check with no interpretive component. The `golden-patterns.md` entry "advisory-to-hook promotion" (2026-04-22) directly applies: the gate condition is expressible as a string pattern on the tool input (`gh pr close` in a Bash command), and the label check is mechanically executable without LLM judgment. The pattern explicitly designates this category of gate — deterministic string-pattern conditions — as the correct candidates for PreToolUse hook promotion.

**Behavioral change from applying this pattern:** Because the advisory-to-hook promotion pattern directly names this case, I weighted the enforcement-layer question as the primary finding rather than a secondary quality note. I did not let the clean utility function implementation resolve the stage-1 concern. Without the pattern, I might have noted the docstring mismatch and approved — the Stage 1 finding changed the verdict.

**What this forecloses:** Shipping advisory enforcement establishes the pattern that safety guards belong in prompt text. Future executor safety issues will be addressed by adding more guard sections to the prompt template (the accumulation spiral named in learnings.md 2026-04-22). A PreToolUse hook covering `gh pr close` would protect all executors — including future ones dispatched via different prompt paths — without relying on prompt absorption.

**Opportunity cost assessment:** The utility function `is_pr_human_gated()` is well-designed and has the right fail-safe default (return non-None on failure, causing callers to skip). But it is not called from any production code path other than the prompt template (which merely instructs an LLM to perform the equivalent check). The function exists but is wired to nothing structural. The hook would make the function the operative enforcement mechanism rather than a library that is callable but not called.

**Alignment verdict: Questioned** — the problem is real and the response is directionally correct, but the enforcement layer is advisory where the codebase's own learnings (PR #1148) say it should be structural.

## Stage 2 — Quality Review

### Gap 1: Mirror-constant in prompt template (blocking)

The prompt template in `dispatcher_handlers.py` hardcodes the three gate label names as raw strings:

```python
f"2. If the output contains any of these labels — **`awaiting-sign-off`**, "
f"`needs-decision`, `blocked-on-dan` — **do not close the PR**. Instead:\n"
```

`dispatcher_handlers.py` does not import `HUMAN_GATE_LABELS` from `wos_issue_lifecycle.py`. The labels are duplicated rather than derived from the authoritative frozenset. If `HUMAN_GATE_LABELS` is updated to add or remove a label, the prompt template will not reflect the change. This is the mirror-constant pattern (learnings.md, six prior recurrences including escalation to production-module mirrors in PR #974). The prompt template should derive the label list from `HUMAN_GATE_LABELS`:

```python
from src.orchestration.wos_issue_lifecycle import HUMAN_GATE_LABELS
# ... in the f-string:
f"2. If the output contains any of these labels — "
f"{', '.join(f'**`{l}`**' for l in sorted(HUMAN_GATE_LABELS))} — **do not close the PR**."
```

This is the same failure class the learnings.md mirror-constant check names as a "first-pass oracle check" (2026-04-26 Meta entry). The detection precondition: before reading any PR diff, check whether a new constant's values are duplicated elsewhere. Here the duplication is in the prompt string.

### Gap 2: Docstring contract mismatch (blocking)

The docstring for `is_pr_human_gated()` says:

> "Returns None on gh CLI failure — callers must treat an error as 'unknown' and surface rather than silently close."

The implementation returns `"unknown-check-failed"` (a non-None string) on failure. The docstring is wrong about the return value. A caller reading the docstring and writing `if result is None: allow_close()` would behave correctly only by accident — the docstring describes the opposite return behavior from what the code produces. This is the comment/code mismatch defect class (learnings.md 2026-04-04, "state machine transition causes silent state divergence — state and comment must be synchronized"). Either the docstring must be corrected to say "Returns a non-None string on gh CLI failure (e.g., 'unknown-check-failed')" or the implementation must return None and require callers to handle error by other means.

The fail-safe design intent (fail closed, i.e., treat unknown as gated) is correct. The docstring is incorrect. Fix the docstring.

### Gap 3: No tests for `is_pr_human_gated()` (blocking)

The PR states "19 passed" for `test_wos_issue_lifecycle.py`. None of the 19 existing tests cover the new `is_pr_human_gated()` function — those 19 tests cover pre-existing functions. The new function has zero test coverage:

- No test for the happy path (PR has a gate label → returns the label)
- No test for the clean path (PR has no gate labels → returns None)
- No test for the error path (gh CLI fails → returns "unknown-check-failed")

The function is the only production-callable artifact that could be wired to structural enforcement in a follow-on PR. A function with no tests entering the codebase is a coverage gap the oracle must flag. The fix is adding at least three test cases to `test_wos_issue_lifecycle.py`.

### Gap 4: Advisory enforcement of a structural invariant violates principle-3 (blocking, structural)

The PR-close guard is injected as text into every executor prompt. This is advisory enforcement: an LLM executor is instructed to perform a label check, and the guard fires only if the executor absorbs and follows the instruction. The guard will not fire if:
- Context pressure causes the executor to skip the guard section
- A future executor is dispatched via a different prompt path that does not include the guard
- The instruction is absorbed but the executor misinterprets the label check output

The correct enforcement for a deterministic label check is a PreToolUse hook. The hook would intercept any Bash tool call containing `gh pr close`, extract the PR number, call the gh CLI label check, and abort the tool call if any gate label is present. This fires regardless of context state, agent definition, or instruction absorption.

The learnings.md entry from PR #1148 (2026-05-14) names this exact pattern: "Advisory enforcement of structural invariant (idempotency check as markdown instruction) violates vision.yaml principle-3." The `golden-patterns.md` "advisory-to-hook promotion" entry states: if the enforcement condition is a string pattern on `tool_input["command"]`, it should be a hook. `gh pr close` is exactly such a string pattern.

This gap is structural — not a code quality issue. The PR can ship with the prompt guard as a short-term mitigation for the acute incident, but the verdict is NEEDS_CHANGES until either: (a) a PreToolUse hook is implemented, or (b) a logged decision explicitly defers hook implementation with stated rationale and timeline. Option (b) would satisfy the oracle — a logged decision acknowledging the advisory-vs-structural gap and deferring the hook is preferable to shipping with no acknowledgment of the gap.

### Gap 5: `is_pr_human_gated()` is defined but not called from any structural enforcement path

The function exists in `wos_issue_lifecycle.py` but is not imported by `dispatcher_handlers.py` and is not called from any production code path. The only "enforcement" is the prompt text instructing an LLM to perform an equivalent check manually. The function is a library with no wiring — the same pattern named in learnings.md PR #714: "a new module with no imports in any existing file is a library with no wiring." Until `is_pr_human_gated()` is called from a structural enforcement path (a hook or a pre-close code path in an orchestration function), the function's existence does not improve system safety.

This is not a separate blocker from Gap 4 — it is the same structural gap viewed from the library side. Addressing Gap 4 (adding the hook) would resolve Gap 5 by creating a call site.

### Encoded Orientation check (constraint-3)

This PR injects a mandatory guard section into every executor prompt going forward — a durable behavioral default change. This is Encoded Orientation. No prior logged decision exists in `oracle/verdicts/` for a PR-close human-gate behavioral policy. No `oracle/verdicts/decision-*.md` file addresses this decision class.

However, the spirit of constraint-3 targets autonomous action decisions — where the system acts without Dan. This PR adds a restriction on autonomous action (preventing auto-close of gated PRs). The constraint-3 concern is lower-severity here than for a PR that adds autonomous capability. The gap is noted, not blocked on: if a decision file is added (e.g., `decision-pr-close-human-gate.md` citing vision constraint-1 or principle-1), constraint-3 is fully satisfied.

### Minor: `import json as _json` inside the try block

`json` is a stdlib module available everywhere. The inline import should be at the module level consistent with the rest of `wos_issue_lifecycle.py`'s import style. This is not a defect but is a style inconsistency.
