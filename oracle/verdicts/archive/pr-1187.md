## Stage 2: Quality Review

**Encoded Orientation check (constraint-3):** This PR refines an existing detection heuristic. It does not change behavioral defaults, agent identity, or decision-making rules. The smell-patterns.yaml `status: open` remains open; no escalation logic changes. Not an Encoded Orientation decision — no prior required.

**Does it do what it claims to do?**

Yes. The SQL query is correctly extended to fetch `artifacts` and `outcome_category`. The `done_unverifiable` count increments only when `status == 'done'` AND both fields are absent. The detection function uses the new count correctly. YAML metadata is updated consistently.

**Failure modes and quality observations:**

1. **Dead variable `total_done` with confusing condition (not blocking).** The PR introduces:
   ```python
   total_done = sum(
       uow_counts.get(k, 0)
       for k in (OUTCOME_PEARL, OUTCOME_SEED, OUTCOME_HEAT, OUTCOME_SHIT)
       if k == OUTCOME_SHIT or True  # all classified outcomes
   )
   ```
   The `if k == OUTCOME_SHIT or True` condition is always True, making the filter meaningless. More importantly, `total_done` is never referenced in the function body — the evidence string uses `total` (total UoW count). This is dead code per the PR #848 learnings.md pattern ("dead code immediately before an unconditional override produces misleading intent signals"). The variable name implies a numerator for a back-propagation rate that the evidence string does not compute. This is a code quality note, not a correctness defect.

2. **Inline string literal `"done_unverifiable"` rather than named constant (not blocking).** The established pattern in this file uses module-scope constants (`OUTCOME_PEARL`, `OUTCOME_SEED`, `OUTCOME_HEAT`, `OUTCOME_SHIT`). The new key is introduced as an inline string in two places within the same file. This is a mild inconsistency — lower severity than cross-file mirror cases documented in learnings.md, since the key does not cross module boundaries.

3. **Heuristic coverage adequacy (by design).** A done UoW with `outcome_category` populated but `artifacts` absent is treated as verifiable — the PR documents this as intentional. This is the correct choice: either field being present is sufficient evidence that wos_completion.py ran.

4. **YAML field `status: open` retained.** Correct — the smell pattern definition remains valid even with current recurrence_count of 0. The issue_ref reset to null and recurrence_count reset to 0 are both accurate.

**What this introduces:**

The pattern of using nullable DB fields (`artifacts`, `outcome_category`) as a back-propagation evidence signal reuses existing infrastructure from PR #891 rather than inventing new tracking. Aligned with vision.yaml principle-4 (integration rate before new feature rate).

**What this forecloses:**

Nothing is foreclosed. The dead `total_done` variable could confuse future rate-based evidence string attempts, but is easily removable.
