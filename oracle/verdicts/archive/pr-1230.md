### Stage 2: Quality Review

**Does it do what it claims to do?**

Yes. The PR correctly:
- Opens a per-execution log file at `_SUBPROCESS_LOG_DIR / f"{run_id}.log"` before `Popen`
- Passes `log_fh` as both `stdout` and `stderr` to the subprocess
- Wraps the Popen call in a `with open(log_path, "w") as log_fh:` block

The `with` block context manager deserves scrutiny: `Popen` is non-blocking and returns before the child process has written any output. When the `with` block's `__exit__` closes the parent's file descriptor, the child process has already inherited its own open file descriptor to the same inode. On Linux, closing the parent's fd does not close the child's fd — the child continues writing correctly. This is standard Unix fd inheritance behavior and the implementation is correct.

**run_id collision resistance:** The `run_id` is `f"{uow_id}-{datetime.now(timezone.utc).strftime('%Y%m%dT%H%M%SZ')}"`. The `uow_id` is a stable registry identifier; the timestamp component is second-granularity UTC. If two dispatches for the same `uow_id` occurred within the same second, they would share a log file. This is not a realistic scenario — a UoW is claimed before dispatch and the registry enforces one-at-a-time execution per UoW. For distinct `uow_id`s dispatched in the same second, the `uow_id` prefix ensures uniqueness. Collision resistance is adequate for the dispatch rate this system operates at.

**Log accumulation / unbounded growth:** This is a legitimate concern. The PR has no cleanup or rotation mechanism. Log files accumulate indefinitely. On the positive side: the failure-traces directory (a precedent in the same `orchestration/` tree) is capped at 500 files / 30-day age by steward-heartbeat cleanup. Log files for executor subagents could grow large for long-running agents, and for a continuously-running executor, this directory will accumulate files without bound.

This gap is real but is not a NEEDS_CHANGES blocker given:
1. The failure-traces cleanup precedent shows the system already has a pattern for periodic cleanup of `orchestration/` subdirectories
2. The immediate operational need (forensic evidence for live crashes) outweighs the operational cost of eventually-full disk in the short term
3. A follow-on PR adding cleanup (analogous to the failure-traces cap) is the correct path — adding cleanup to this PR would increase scope and delay a needed fix

The gap should be noted for tracking without blocking this PR.

**Test assertions — are they actually verifying the contract?**

The updated test `test_stdin_devnull_stdout_stderr_log_file` verifies:
- `stdin == subprocess.DEVNULL`
- `stdout != subprocess.DEVNULL`
- `stderr != subprocess.DEVNULL`
- `hasattr(stdout, "write")` — writable file object
- `hasattr(stderr, "write")` — writable file object
- `stdout is stderr` — same file handle

These assertions are substantive. The `stdout is stderr` assertion is load-bearing: it verifies that both streams share the same log file, preventing a scenario where they diverge to different files. The `hasattr(stdout, "write")` check verifies a file-like object rather than a sentinel — it would catch `subprocess.PIPE` as well as a real file handle (both have `.write`). This is intentionally broad: the oracle interest is that DEVNULL is not used, not that a specific file type is used.

**Encoded Orientation check (OODA constraint-3):** This PR changes the subprocess invocation in `_dispatch_via_popen`. This is a behavioral default change (where output goes), but it is not an Encoded Orientation decision in the constraint-3 sense — it does not change agent identity, decision-making rules, or behavioral routing. It changes observability plumbing. The OODA constraint-3 gate does not fire.

**Path correctness:** `Path.home() / "lobster-workspace" / "orchestration" / "logs"` maps to `~/lobster-workspace/orchestration/logs/`. This directory exists on the live instance (confirmed). The path is consistent with CONVENTIONS.md which designates `orchestration/` as the canonical home for orchestration agent runtime files.

**`mkdir -p` on every dispatch:** The `_SUBPROCESS_LOG_DIR.mkdir(parents=True, exist_ok=True)` call happens on every dispatch, not once at startup. This is slightly wasteful for the common case (directory already exists) but `exist_ok=True` makes it a no-op after the first call. The overhead is negligible compared to the subprocess spawn latency.

**Feature bundling check:** Single concern, two files (production + test). No bundling.

**Patterns this introduces:** The pattern of per-execution log files named by `run_id` is now load-bearing for post-mortem analysis. Future cleanup must preserve this naming convention to remain useful. This is a low-risk propagation since `run_id` is already the registry's audit correlation key.

**What this makes easier:** Post-mortem analysis of subagent crashes, OOM kills, and Claude binary exits. Log content will show the subagent's last output before death, enabling diagnosis that is currently impossible.

**What this makes harder:** Nothing. The log directory can be ignored by operators who don't need it; it doesn't change the dispatch API, the registry schema, or any existing tooling.

### Named gap for follow-on (not a NEEDS_CHANGES blocker)

**Gap: No log rotation or accumulation cap.** The `orchestration/logs/` directory will accumulate one log file per dispatch with no automated cleanup. The precedent for this problem (`failure-traces/` capped at 500 files / 30 days by steward-heartbeat) should be applied here. A follow-on issue should add steward-heartbeat cleanup for `orchestration/logs/` with equivalent caps. This is not blocking this PR because the immediate observability need is urgent and the cleanup mechanism is a known pattern that can be applied without risk.
