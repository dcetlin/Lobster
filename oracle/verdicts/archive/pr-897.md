## Stage 2: Quality Review

### Scope

The diff is exactly 5 lines in one file (`src/orchestration/registry.py`). It adds a `UoWStatus(new_status)` coercion at the top of `set_status_direct()` before any DB write. The PR body is accurate with respect to the diff.

### Correctness

`UoWStatus` is a `StrEnum`. Calling `UoWStatus(value)` for a valid member (e.g. `"done"`, `"proposed"`) is a no-op pass-through. For any value not in the enum, it raises `ValueError` with a clear message (`'complete' is not a valid UoWStatus`). This is the idiomatic Python way to validate enum membership at a boundary — pure, composable, and zero additional logic needed at call sites.

### Caller audit

The PR correctly identifies the only external caller (`garden_caretaker.py:564`) and confirms it passes `UoWStatus.PROPOSED` (an enum member). All SQL string literals in registry.py that set status use values that are valid enum members. No caller is broken by this change.

### Steward risk

The steward's optimistic lock pattern (`WHERE status = 'ready-for-steward'`) means the two anomalous rows (`complete`, `completed`) were never processed. The DB fix eliminates the risk; the code change prevents recurrence. The assessment in the PR body is accurate.

### DB fix

The fix was applied via Python `sqlite3` directly (not through the Registry class, which would have required valid enum values — a chicken-and-egg problem). Two rows corrected. Confirmed with SELECT. The approach is correct.

### Test coverage

The smoke test directly exercises the four cases: one valid status accepted, three invalid strings rejected. The WOS unit tests (102 passed) cover the surrounding infrastructure. The pre-existing `test_dispatch_job_sh` failure is documented and confirmed present on `main` before this branch.

### No concerns

The change is minimal, correct, and well-bounded. Nothing to flag.
