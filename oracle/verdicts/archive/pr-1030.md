## Stage 2 — Quality review

### Mirror-constant pattern check (first-pass, from learnings.md 2026-04-26)

The five `KILL_TYPE_*` constants introduced at module level in `registry_cli.py` hold string values (`"orphan_kill_before_start"`, `"orphan_kill_during_execution"`, `"hard_cap"`, `"user_closed"`, `"execution_failed"`, `"unknown"`) that partially overlap with raw string literals already embedded in `steward.py`, `dispatcher_handlers.py`, and `registry.py`. Specifically, `"orphan_kill_before_start"` and `"orphan_kill_during_execution"` appear as raw strings in all three of those files — no canonical module-level definition exists anywhere.

The severity is moderate. The `KILL_TYPE_*` constants are at least exported so callers can import them rather than re-declare locally, and the test file correctly imports them (no local redeclaration). But the constants are not the canonical source for the string values — they are a fourth parallel representation. If `registry.py`'s `kill_classification` default parameter or `steward.py`'s orphan routing ever changes a value, `registry_cli.py`'s constants diverge silently.

The test assertions import the constants rather than re-declaring them locally — this is correct per the mirror-constant learning (PRs #696, #800, #967, #970, #974). The test practice is sound; the production-layer fragmentation is the residual concern.

**Decision:** This is a pre-existing structural gap in the codebase (no canonical kill_type enum exists anywhere) that this PR slightly worsens by adding a fourth representation. It is not a blocker — the PR does not introduce test-file mirror constants, which is the failure mode the learnings.md pattern is most concerned about. The right fix (define a `KillType` StrEnum in `registry.py` and import it everywhere) is a follow-on item, not a gate condition for this PR.

### StrEnum golden pattern check (golden-patterns.md 2026-04-24)

`_fetch_failed_uows_since` uses `WHERE status = 'failed'` — a raw string literal in SQL for an enum-backed column. `UoWStatus.FAILED` is the correct reference per the StrEnum pattern. However, `UoWStatus` is not imported anywhere in `registry_cli.py` — the entire file uses string literals for status comparisons (e.g., `_FAILED_STATUSES = frozenset({"failed", "cancelled"})`). This PR is consistent with the file's existing pattern, not introducing a regression. The underlying StrEnum gap in `registry_cli.py` is pre-existing.

**Effect of the golden pattern on this analysis:** The StrEnum pattern caused me to look at the SQL query and verify the surrounding file context before flagging. Because the violation is pre-existing and the PR is consistent with the surrounding code, it is not a blocker. Without the pattern check, I would have passed this without noting the pre-existing gap.

### Classification priority ordering (implementer concern 1)

Priority 1 (orphan_kill_classified) over Priority 2 (hard_cap_cleanup close_reason) is correct. The `close_reason` records *why* the steward decided to close the UoW; the `orphan_kill_classified` audit event records *how* the executor was actually killed. An infrastructure kill is an infrastructure kill regardless of what triggered the closure decision. Giving orphan priority means infrastructure kills are always visible as infrastructure kills in the aggregate view, which is the diagnostic goal. The priority ordering is sound.

### Direct DB access via `registry._connect()` (implementer concern 2)

`_fetch_failed_uows_since` accesses `registry._connect()` directly rather than going through a Registry method. The Registry already exposes `fetch_audit_entries()` (which uses `_connect()` internally) and `list()` (filtered queries). Adding `list(status=UoWStatus.FAILED)` with an optional `since` parameter as a proper Registry method would be cleaner. The current approach works, and the connection is properly closed in a `finally` block. The seam-first golden pattern (2026-03-30) would argue for a named Registry method here as the seam where "CLI read logic" meets "DB access." This is an advisory finding, not a blocker — the pattern applies but the stakes are low for a CLI-only query.

### Classification result type

The classifier returns a bare string. The classification-result-as-typed-dataclass golden pattern (2026-04-04) applies to classifiers that drive automated decisions where misclassification has consequences. Here the output is human-readable display only — no routing or automated action depends on the classification. The bare string is acceptable for this context.

## Summary

The PR is clean. It closes a named observability gap with a minimal, pure-function implementation and a complete test suite that correctly imports production constants. Two structural observations are noted for follow-on work:

1. **Kill_type string canonicalization:** A `KillType` StrEnum in `registry.py` would unify the four parallel representations (`registry_cli.py`, `steward.py`, `dispatcher_handlers.py`, `registry.py` docstrings). This is a pre-existing gap, not introduced by this PR.
2. **`_fetch_failed_uows_since` DB access pattern:** Wrapping this as a named `Registry` method would be cleaner for the seam. Low stakes for a CLI-only command.

Neither observation blocks approval.
