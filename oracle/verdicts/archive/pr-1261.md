## Stage 1: Vision Alignment (formed before seeing implementation)

**Adversarial prior:** This implementation is solving the wrong problem, or solving the right problem in a direction that forecloses better paths.

**Theory of change:** The WOS steward generates prescriptions that executors consume. A structured, machine-readable v2 format (#573 spec) should make prescriptions more deterministic, auditable, and composable. #574 implements the generation side.

**Scenario where this is wasted effort:**
- The executor never migrates to reading v2 (migration tracked in #576, actively executing per labels)
- The 7-section format proves too rigid for actual UoW diversity
- The dual-write overhead (v2 audit trail entry per every steward cycle) creates noise without signal

**Key questions before seeing implementation:**
1. Is the transition period properly bounded? (executor still reads v1 WorkflowArtifact)
2. Does the dual-write have a clear migration path?
3. Does this foreclose simpler paths?

**Vision anchor check:** Current focus is "WOS executor now live" and "WOS bidirectional interface (#1118)." The v2 prescription work is infrastructure supporting the executor loop's quality — traceable to `active_project.phase_intent` ("UoWs carry auditable context") and principle-3 ("determinism over judgment for conditionals"). Not the primary focus, but not orthogonal either.

**Alignment verdict:** Confirmed — this is a planned implementation track (#228 → #573 → #574 → #576) with prior design decisions logged.

## Stage 2: Quality Review

**Does it do what it claims?**

Yes. The implementation adds:
- `PrescriptionV2` dataclass (frozen, slots, `to_json()`)
- `_build_v2_prescription_deterministic()` — schema-valid fallback without LLM
- `_llm_prescribe_v2()` — Claude subprocess call with JSON extraction and structured fallback chain
- `generate_v2_prescription()` — orchestrates the above, reads vision route, dan register, writes audit entry
- Hook in `_capturing_prescriber()` — dual-writes v2 to audit trail (guarded by try/except so v1 flow is never disrupted)

All 8 tests pass. 18 related prescription confidence tests pass.

**Implementation quality:**

The fallback chain is well-engineered: LLM call → JSON parse failure → missing required keys → deterministic fallback. Each failure mode is caught separately with a specific log message. The `try/except` wrapper in `_capturing_prescriber` ensures v1 flow is never disrupted by v2 failures.

The `dan_context` section in the deterministic fallback returns empty strings for orientation, priority_signal, and load_bearing_assumption. This is correct for fallback — the LLM path fills these from the dan register. Empty strings are schema-valid.

**Schema version mismatch:** The spec doc (`docs/prescription-format-spec.md`) is at version 0.1.0 and is explicitly marked "Design Specification — Not Yet Implemented." The implementation uses schema_version "1.0.0". This is a forward versioning decision — the implementation is shipping the first production version of the format, not the pre-implementation spec draft. This is intentional and not a defect.

**Encoded Orientation check (constraint-3):** This PR does NOT change behavioral defaults, agent routing, or decision-making rules in a durable way. The v2 prescription is written to the audit trail; the executor still reads v1 WorkflowArtifact. No Encoded Orientation decision. Constraint-3 does not apply.

**Failure modes:**
- LLM call produces malformed JSON → caught, falls back to deterministic
- LLM call times out → handled via `run_subprocess_with_error_capture`
- `generate_v2_prescription` throws any exception → caught in `_capturing_prescriber` with warning log, v1 flow continues
- `resolve_vision_route` raises → caught with generic Exception, `vision_orientation=""`, deterministic fallback uses empty vision context

**Patterns introduced:**
- Dual-write pattern (v2 alongside v1) with explicit transition scope comment and issue reference (#576)
- LLM-with-deterministic-fallback prescription generation (extends existing pattern from `_llm_prescribe`)

**What this makes easier:** Future executor migration to v2 (read from audit log), prescription auditability, pattern analysis across cycles.

**What this makes harder:** Nothing structurally foreclosed. The dual-write adds one audit entry per steward cycle — modest overhead.

**Prior oracle patterns applied:**
- `learnings.md`: No matching failure patterns detected. The fallback chain correctly avoids the "contract gap tolerating open executor contract" antipattern — the v2 flow fails gracefully and never corrupts the v1 flow.
- `golden-patterns.md`: The deterministic fallback follows the "determinism over judgment" principle (principle-3). The try/except wrapper in `_capturing_prescriber` is a correctly-applied proactive resilience pattern (principle-1).

**Verdict rationale:** The work is correctly scoped, well-tested, gracefully isolated (no v1 disruption), and explicitly tracked to the migration follow-up (#576). All tests pass. No gaps requiring changes.
