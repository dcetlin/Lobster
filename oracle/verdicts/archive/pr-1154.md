## Stage 1 — Vision Alignment (formed before seeing implementation)

**Adversarial prior:** This implementation is solving the wrong problem, or solving the right problem in a direction that forecloses better paths.

The dead-end gate suppresses re-dispatch of UoWs that have failed or been blocked enough times that further retries are likely futile. If infrastructure kills (session crashes, TTL evictions, orphan recoveries) are counted as genuine execution failures, the gate fires for work that was never actually attempted. This directly undermines the Registry's role as a reliable source of truth about work status — a WOS Phase 1 success criterion.

The background data (136 UoWs permanently suppressed, all 272 failure entries infrastructure kills) confirms this is an active production failure, not a hypothetical one.

**Stage 1 adversarial question:** Should this gate on `execution_attempts` (already a first-class column) rather than filtering at the audit-entry counting level?

The learnings.md PR #965 pattern is directly applicable: "Retry counter gated at the re-prescription site rather than at confirmed execution outcome conflates dispatch-level events with execution-level outcomes." The oracle/patterns.md `infrastructure-vs-execution discriminator` section explicitly describes the fix as using a separate `execution_attempts` counter. This prior exists and must be checked before the PR approach can be accepted.

**Resolution of Stage 1 question:** The `execution_attempts` column approach cannot replace the audit-entry-counting approach here because the dead-end gate fires on "failed OR blocked ≥2 times" (oracle/patterns.md §dead-end), and `execution_attempts` only tracks confirmed execution outcomes — not Steward-level blocks (`to_status='blocked'`), which are legitimate dead-end signals. A UoW blocked by the Steward without the executor ever running would have `execution_attempts=0` but should still count toward the dead-end gate if the block reflects a genuine stuck condition. Using `execution_attempts >= N` alone would miss this case.

The PR's approach (filter infra-kill entries from the failure count, preserve genuine failure and block entries) is the correct discriminant for this gate's semantics. The alignment verdict is Confirmed.

**Vision alignment:** This work directly serves WOS Phase 1 success criterion sc-1 ("Registry query gives a reliable answer to 'what should I work on?'") and operating principle-1 ("proactive resilience over reactive recovery"). The dead-end gate misfiring on infrastructure kills is a structural reliability failure — the Registry was silently discarding 136 UoWs that had never actually been tried. No cheaper test exists: the 136-UoW production event is itself the cheapest test, and the fix is a targeted, reversible change to one counting function.

## Stage 2 — Quality Review

### Does it do what it claims to do?

Yes. `_count_failed_or_blocked_transitions` now delegates infra-kill detection to `_is_infra_kill_audit_entry` before counting. The four codes (`executing_orphan`, `orphan_kill_before_start`, `orphan_kill_during_execution`, `ttl_exceeded`) match the actual strings passed to `Registry.fail_uow` in production:

- `executing_orphan`: `steward.py:4479` — `f"{reentry_posture}: subagent exited without calling write_result"` where `reentry_posture == "executing_orphan"`
- `orphan_kill_before_start` / `orphan_kill_during_execution`: same pattern, same line
- `ttl_exceeded`: `executor.py:1619` — `f"ttl_exceeded: UoW was in active state for more than {TTL_EXCEEDED_HOURS}h"`

All four producers write in `"<code>: <detail>"` format. The `startswith` discriminant is correct.

### The `startswith` risk flagged by the author

The author correctly flagged this. The risk is real but bounded: a future infra-kill code whose name is a prefix of an existing code (e.g., a new code `"executing"` would match `"executing_orphan: ..."`) would cause false positives. No such prefix collision exists in the current codebase. The risk is also offset by the fact that all infra-kill codes are controlled infrastructure-generated strings — not user-provided text.

### Frozenset coverage: absent codes

`executor_orphan` and `diagnosing_orphan` are absent from `DEAD_END_INFRA_KILL_REASONS`. This is correct:
- `executor_orphan` means "executor never claimed the UoW" — these UoWs get re-prescribed, not failed via `fail_uow`. They generate no `to_status='failed'` audit entries.
- `diagnosing_orphan` means "Steward crashed mid-diagnosis" — same: re-prescribing path, not `fail_uow`.

Neither code would create entries counted by `_count_failed_or_blocked_transitions`. The frozenset covers exactly the four codes that produce `fail_uow`-written audit entries with `to_status='failed'` and infra-kill reasons.

### StrEnum concern (minor, non-blocking)

`DEAD_END_INFRA_KILL_REASONS` uses raw string `"executing_orphan"` where `ReentryPosture.EXECUTING_ORPHAN` exists. The three other codes (`orphan_kill_before_start`, `orphan_kill_during_execution`, `ttl_exceeded`) have no StrEnum backing. Since StrEnum values compare equal to their string values, this is functionally equivalent. The risk is minor: if `ReentryPosture.EXECUTING_ORPHAN` were renamed, the frozenset would silently diverge. However, that same rename would break dozens of other usages throughout steward.py that use the raw string — this is not a new divergence risk introduced by this PR.

### Encoded Orientation check (constraint-3)

The dead-end gate is a Reactive/Constrained register decision. The prior logged decision is `oracle/patterns.md §infrastructure-vs-execution discriminator`, which explicitly prescribes this fix. The vision.yaml anchor is constraint-3 (OODA) and principle-1 (proactive resilience). The Encoded Orientation check passes.

### Test quality

- Tests import `DEAD_END_INFRA_KILL_REASONS` from production rather than re-declaring it — correct per the mirror-constant pattern (learnings.md PR #696 pattern).
- `_is_infra_kill_audit_entry` has 8 unit tests covering: each code, CalledProcessError (genuine failure), no note, non-JSON note, no reason field.
- `TestDeadEndGateInfraKillDiscrimination` has 5 integration tests testing `_check_dispatch_eligibility` with various infra/genuine mixes.
- `test_dead_end_infra_kill_reasons_constant_covers_all_known_codes` uses `issubset` — appropriate for asserting the four known codes are present without over-constraining future additions.
- The test helper `_infra_kill_entry` builds notes that match the production `fail_uow` format exactly.

### What would break without the key decisions here

The `DEAD_END_INFRA_KILL_REASONS` frozenset being module-scope (not function-scope) is correct per learnings.md PR #965 and #967 patterns — it prevents per-call reallocation and ensures enum-change safety.

The `_is_infra_kill_audit_entry` pure-function factoring is the right structural choice: it can be tested independently, the discriminant is explicit and observable, and the `_count_failed_or_blocked_transitions` function remains a pure transformation.

### What this makes easier/harder

Easier: adding a new infra kill type requires adding one string to `DEAD_END_INFRA_KILL_REASONS` — a single-file, single-constant change. Harder: nothing foreclosed. The approach is additive and reversible.

### Patterns introduced

The `_is_infra_kill_audit_entry` function is a named discriminant for an important category distinction. This pattern — named pure-function discriminants for infrastructure-vs-execution events at the gate level — is applicable to any future gate that must distinguish platform kills from genuine failures.
