### Stage 1: Vision alignment (formed before reading implementation)

The work under review is a resilience feature: surface unhandled exceptions from scheduled heartbeat tasks to the user via Telegram rather than swallowing them silently. The implicit theory is that the system's failure visibility is a prerequisite for "proactive resilience over reactive recovery" (principle-1). Before seeing the diff, the adversarial prior asks: is the assumption that crash detection belongs in the heartbeat scripts themselves correct? Could this be solved at the cron/monitoring layer instead, or via a watchdog that reads cron exit codes without touching application code?

The answer is: the cron layer does capture exit codes, but it provides no Telegram notification path. A watchdog would need its own crash surface. The heartbeat scripts have the richest failure context (exception type, traceback, job_name) and are the earliest point where an unhandled exception can be intercepted with that context intact. Application-layer interception is the right level. The feature is not solving the wrong problem.

Vision alignment: **Confirmed**. The work directly serves principle-1 ("structural prevention over reactive recovery") and principle-4 ("wire what exists before building more" — the inbox and dispatcher notification path already exist). The missing arrows between the heartbeat scripts and the dispatcher's notification channel are exactly what this PR closes.

### Stage 2: Quality review

**Does it do what it claims?**

Yes. Both heartbeat scripts now wrap `main()` around `_main_inner()`. Any unhandled exception reaching the outer handler: (1) logs `ERROR` with full traceback, (2) calls `_write_crash_alert()` which atomically writes a `scheduled_task_crash` message to the inbox, (3) returns exit code 1 so cron records the failure. The inbox message shape (`source`, `type`, `chat_id`, `job_name`, `timestamp`, `text`) matches what the dispatcher handler in bootup.md expects to relay. The dispatcher handler is correctly positioned between `agent_failed` and `cron_reminder`, follows the established three-step pattern, and requires no subagent.

**Key implementation decisions that hold up:**

- `_write_crash_alert` is best-effort: write failures are caught and logged without masking the original exception. This is the correct posture — a crash notification mechanism that can itself crash and mask the root cause would be strictly worse than nothing. The PR's test plan verifies this explicitly.
- Traceback truncated at 2000 chars from the tail. Keeping the tail (most recent frames) is more diagnostic than keeping the head (outer frames), which is correct.
- `_ADMIN_CHAT_ID` uses `os.environ.get("LOBSTER_ADMIN_CHAT_ID", "8075091586")` — env-var-injectable form. This satisfies the learnings.md 2026-05-10 (#1122) pattern requiring env-var form for `ADMIN_CHAT_ID` in scheduled tasks.
- Atomic write via tmp-then-rename is present, matching the canonical `src/utils/inbox_write.py` pattern.
- `LOBSTER_MESSAGES` env override respected for the inbox path, not hardcoded.

**The ImportError tightening in `run_ttl_recovery()`:**

`log.warning` → `log.error` plus `_write_crash_alert()` with a synthetic `ImportError` is a deliberate design choice: the function cannot be wrapped by the outer main() crash handler because the exception is already caught here. Constructing a synthetic exception and passing it to `_write_crash_alert()` is the correct workaround. The extra string naming the structural gap (#1216 partial implementation) is load-bearing — it tells Dan what the structural cause is, not just that a crash occurred. The primary reset_expired_claims() path being unaffected is called out explicitly.

**Encoded Orientation check (constraint-3):**

This constitutes an Encoded Orientation decision. The anchor is vision.yaml principle-1 ("proactive resilience over reactive recovery") and principle-4 ("wire what exists before building more"). Both are valid citable field paths. The prior oracle review is the logged decision. Constraint-3 is satisfied.

**What could break:**

- If the dispatcher's message loop encounters `type: "scheduled_task_crash"` before the bootup file is loaded with the new section, it will fall into an unhandled-type path. This is a deployment-sequencing concern, not a code defect — the bootup file update and message_types.py registration are in the same PR and will be live simultaneously after merge.
- The copy-paste of `_write_crash_alert` is a known future risk (issue #1227). If one copy is modified (e.g., to call the shared utility after extraction) but the other is not, the two scripts diverge silently. This is acceptable at the current scale with two scripts; it becomes urgent if more heartbeat scripts adopt the pattern.
- The outer except catches `Exception` but not `BaseException` — `SystemExit` and `KeyboardInterrupt` are not caught, which is correct. A `SystemExit(0)` from an intended early exit would not generate a spurious crash alert.

**Patterns introduced:**

The outer-crash-handler pattern (wrap `main()` around `_main_inner()`, catch-log-alert-exit) is a new structural pattern for scheduled tasks. It is correct and worth naming. This is a candidate for golden-patterns.md if it propagates to other heartbeat scripts.

**What this makes easier:**

Any future heartbeat script can adopt the same three-line wrapper. The dispatcher handler is now in place to relay messages from any script that writes `type: "scheduled_task_crash"`. The pattern is self-extending.

**What this makes harder:**

Nothing of note is foreclosed.
