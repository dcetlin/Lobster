## Stage 1: Vision Alignment

**Pre-implementation finding (locked before reading diff):**

The PR claims to close a mirror-constant advisory from PR #1116 round 2, removing locally-declared pricing constants from a test file and importing them from the production module instead.

The adversarial prior: is this solving the wrong problem, or solving the right problem in a direction that forecloses better paths?

Disconfirmed. The mirror-constant pattern is one of the most densely-documented failure modes in this codebase's learnings register — it has recurred in at least six PRs (#696, #800, #967, #970, #973, #974) and the learnings.md record explicitly elevated it to "first-pass oracle check, not per-PR discovery" as of 2026-04-26. The pattern is: a test locally re-declares a production constant, creating a divergence risk when the production constant changes. Pricing constants are volatile (vendor rate changes), making divergence risk non-theoretical. Importing from the production module is the named fix in every prior instance.

**What would have to be true for this to be wasted effort?** Either the constants will never change (false for pricing), or importing them creates a worse coupling (false — module-level constants are the stable public surface). Neither holds.

**What does this foreclose?** Nothing. It is a hygiene closure with no directional commitment.

**Oracle vocabulary applied:** The mirror-constant pattern (learnings.md, multiple entries) is the directly triggering pattern class. Reading it before the diff changed my analysis in one specific way: I weighted the primary fix as clearly correct before examining the secondary change, which prevented me from letting the secondary issue color my reading of the primary purpose. Without the mirror-constant pattern active as a named class, I might have opened with "the change is small and whether it matters depends on..." — the pattern resolves that ambiguity structurally.

The feature-bundling check (learnings.md 2026-05-14, #1148; also 2026-04-23, #882) is the second active prior. Reading it before the diff caused me to count changed files before reading content, which surfaced the executor-heartbeat.py change immediately as undescribed. Without this check as an active prior, I would have encountered the second file mid-review and likely weighted it as incidental rather than flagging it as a named pattern.

## Stage 2: Quality Review

The three constants `SONNET_4_6_INPUT_PER_MTK`, `SONNET_4_6_OUTPUT_PER_MTK`, `SONNET_4_6_CACHE_READ_PER_MTK` exist at module scope in `src/orchestration/wos_uow_detail_gen.py` lines 48-50. Result: 41 tests passed. No assertions, fixtures, or test logic changed — only the constant source.

The fix is correct, minimal, and closes the named mirror-constant gap. Future changes to production pricing constants will now cause test failures (loud) rather than silent stale assertions (the prior failure mode).

**Secondary change: scheduled-tasks/executor-heartbeat.py**

`log.info` → `log.debug` at the `uow_count == 0` branch of `_warn_if_legacy_registry_exists()`. Full context:
- 0-UoW branch (empty schema artifact): downgraded from INFO to DEBUG
- Non-zero branch (actual data remaining in legacy DB): still fires at WARNING, rate-limited by a /tmp/ sentinel
- Subsequent-boot suppressed invocations: already at DEBUG

This change is defensible: a zero-UoW legacy DB is not an operator-alert condition — it is a schema artifact that Migration 86 should have cleaned up. The WARNING path (non-empty data) is preserved. The change prevents ~480 INFO-level log lines/day (per the function's docstring) for installations that have the zero-UoW remnant but have no actual migration work remaining.

**Feature-bundling assessment:** This change is not mentioned in the PR description. The feature-bundling pattern (learnings.md 2026-05-14, #1148) applies. However, this instance is lower-severity than the named examples:
- Both changes are maintenance hygiene (test-constant fix + log-noise reduction)
- The secondary change is 1 line, non-behavioral (does not alter any warning or alert path that matters)
- No UoW references distinguish it as a separate workstream
- The diff is fully visible for inspection

Requiring a separate PR for a 1-line log-level cosmetic would impose process overhead disproportionate to the risk. The bundling is real but is not a NEEDS_CHANGES blocker. It is a learnings note.

**Encoded Orientation check (OODA constraint-3):** Neither change encodes a new behavioral default, agent identity change, system constraint, or decision-making rule. The log level change reduces noise but does not alter the WARNING path that handles the actionable case. No logged prior or vision.yaml anchor required.

**Failure modes examined:**
- Import fails if production module removes or renames the constants: loud ImportError at test collection time. This is strictly better than the prior failure mode (stale silent divergence).
- The `log.debug` change: if a developer needs to verify 0-UoW detection is firing, they need DEBUG logging active. Acceptable trade-off — the 0-UoW case is diagnostic, not operational.

**What this makes easier:** Pricing constant changes in production will surface as test failures immediately. The mirror-constant gap is closed for this test file.

**What this makes harder:** Nothing.

**Patterns introduced that will propagate:** The import-from-production-module pattern for constants-under-test. This is the correct and already-named golden pattern. The bundling-of-small-unrelated-changes pattern continues but at low severity.
