### Stage 2: Round 2 Quality Review

#### Gap 1: --skip-lock invariant analysis

**The fix is structurally valid.**

The approach taken: vault-processor.py receives a `--skip-lock` CLI flag. When present, `main()` calls `run_processor()` directly without attempting `acquire_lock_or_skip()`. `_invoke_processor()` in vault-watcher.py always constructs the subprocess command with `--skip-lock`. The docstring on `_invoke_processor()` explicitly documents why: `subprocess.run()` defaults to `close_fds=True`, the lock fd is not inherited, and mutual exclusion is already guaranteed by the parent's held lock.

**Is the invariant argument valid?** Yes — with a noted limitation. The invariant "vault-watcher holds the lock → no concurrent processor can run → --skip-lock is safe" is mechanically correct for the normal execution path. vault-watcher.py acquires the lock in `main()` before calling `_invoke_processor()`, `_invoke_processor()` always passes `--skip-lock`, and vault-processor.py's `--skip-lock` path skips its own lock acquisition entirely. The mutual exclusion guarantee from the parent process holds for the full synchronous subprocess run.

**Can --skip-lock be passed by a non-watcher caller?** Yes — it is a CLI flag accessible to any human or script that invokes vault-processor.py. The argparse `help` text contains an explicit warning: "Use ONLY when invoked by vault-watcher.py, which holds the lock on behalf of this process. Passing this flag when the caller does not hold the lock removes the mutual-exclusion guarantee." This is advisory, not structural enforcement. A human running `uv run vault-processor.py --skip-lock` when the watcher is not active would bypass the lock with no error. This is an accepted design tradeoff — the manual invocation path is debug/maintenance, and the warning is in the argparse help. The risk is human error in manual invocation, not systematic failure. Acceptable.

The module docstring for vault-processor.py also states: "When invoked directly (manual/debug), this script acquires the lock itself via acquire_lock_or_skip()." The `--skip-lock` flag is not mentioned in this summary sentence, which creates a minor documentation inconsistency — manual invocers reading the module docstring would need to scroll to the argparse section to learn about the flag. This is minor and does not block approval.

#### Gap 2: Do the 3 tests test the cross-process boundary?

**The tests do not test the cross-process boundary directly, but this is acceptable.**

The three tests are:

1. `test_vault_processor_skip_lock_flag_allows_run_while_lock_held`: Holds the lock in the test process, then calls `vault_processor_main()` in the same process with `sys.argv = ["vault-processor.py", "--config", ..., "--skip-lock"]` and `patch.object(_vault_processor, "run_processor", ...)`. Verifies `run_processor` is called once. This is a single-process simulation of the cross-process scenario.

2. `test_vault_processor_without_skip_lock_is_blocked_when_lock_held`: Same setup without `--skip-lock`. Verifies `run_processor` is NOT called. Documents the baseline blocking behavior.

3. `test_invoke_processor_passes_skip_lock_flag`: Patches `subprocess.run` at the vault-watcher level and verifies `--skip-lock` appears in the constructed command. This tests the watcher-side invariant (command construction), not the processor-side behavior.

The actual cross-process scenario — vault-watcher holding an `fcntl.flock` lock in OS process A, then spawning OS process B which would see a `BlockingIOError` on the same lock file — is not tested. A true cross-process test would require `subprocess.Popen` or `subprocess.run` in the test itself with the parent holding the lock via `fcntl.flock`.

However, the tests are sufficient for these reasons: (1) Test 1 verifies that `--skip-lock` causes `main()` to call `run_processor()` instead of the acquire-lock-or-skip path. (2) Test 3 verifies that `_invoke_processor()` always passes `--skip-lock` to the subprocess command. (3) The fix's correctness at the cross-process level is mechanically implied: if `--skip-lock` bypasses `acquire_lock_or_skip()` in `main()` (verified by test 1), and the command is always built with `--skip-lock` (verified by test 3), then the actual subprocess will skip lock acquisition regardless of what the parent holds. The cross-process `BlockingIOError` described in the Round 1 verdict is the motivation for the fix; the test suite correctly tests the mechanism of the fix at the unit level. An integration test would add confidence but is not required for approval.

### Named gaps requiring resolution

**Gap 1: Lock inheritance failure makes watcher-invoked processor inoperable (critical)**

`vault-watcher.py` acquires `fcntl.flock` on `/tmp/vault-processor.lock`, then invokes `vault-processor.py` as a subprocess. The subprocess is a separate process. `subprocess.run()` defaults to `close_fds=True` — the lock fd is not inherited. The processor calls `acquire_lock_or_skip(LOCK_PATH)`, finds the lock held by the parent process, and exits with "skipping: processor already running". No vault processing occurs on any watcher-triggered invocation.

Required fix: one of (a) release the lock in vault-watcher.py before invoking the processor, then re-acquire after; (b) add a `--already-locked` / `--skip-lock` flag to vault-processor.py for watcher-invoked calls; (c) restructure so vault-watcher.py calls `run_processor()` directly in-process rather than as a subprocess (most straightforward — the subprocess boundary was intended for isolation but creates the lock problem). The tests do not catch this because they test `run_processor()` directly, not via `_invoke_processor()`.

**Gap 2: render_active_todos() behavior change in todo_obsidian_sync.py is undocumented**

The new shared `render_active_todos()` always inserts `"- [ ] 🔒 DISABLE PROCESSING"` in its output. The original `render_active_todos()` in `todo_obsidian_sync.py` did not. This means `todo_obsidian_sync.py` now writes the guard line on every run — a behavior change from the "refactor only, behavior unchanged" claim. This may be the correct and necessary migration behavior (it sets up the invariant vault-processor.py requires). If intentional, it must be documented explicitly (PR description, commit message, or code comment) rather than hidden under "behavior unchanged." If unintentional, the guard line should be conditioned on `last_synced is not None` (vault-processor.py path) to preserve true zero-change behavior for `todo_obsidian_sync.py`.

Resolution options:
(a) Document as intentional migration behavior — add a code comment in `render_active_todos()` explaining that the guard line is always rendered because vault-processor.py requires it, and that `todo_obsidian_sync.py` uses this to set up the invariant on first run.
(b) Make the guard line conditional: render it only when `last_synced is not None` (vault-processor.py path). Add a separate migration step that inserts the guard on first deploy.
(c) Author states why the gap does not apply.
