#### Gap 1: Mirror-constant dead code (NEEDS_CHANGES)

The test file declares three module-scope constants:

```python
CC_QUOTA_STALE_THRESHOLD_MINUTES = 60
CC_QUOTA_COLOR_GREEN_MAX = 70
CC_QUOTA_COLOR_YELLOW_MAX = 90
```

These mirror production constants from `wos_dashboard.py`: `CC_QUOTA_STALE_THRESHOLD_MINUTES`, `CC_QUOTA_COLOR_GREEN_MAX`, and `CC_QUOTA_COLOR_RED_MIN`. Critically:

1. None of these test constants is referenced in any assertion in the diff. They are dead test constants — declared, not used.
2. `CC_QUOTA_COLOR_YELLOW_MAX = 90` is a renamed mirror of `CC_QUOTA_COLOR_RED_MIN = 90`. The rename introduces semantic confusion: a future test author consulting this constant would call it a "yellow max" threshold, while production calls it a "red min" threshold. The names describe different intents (ceiling of yellow vs. floor of red), which happen to equal the same value.

**How this pattern constrained my analysis:** The mirror-constant pattern from learnings.md is now a "first-pass oracle check, not a per-PR discovery" (learnings.md 2026-04-26 entry). Six recurrences (PRs #696, #800, #967, #970, #973, #974) and the learnings.md PR #1116 entry ("when authoring a new module with named constants, the companion test file should import those constants from the production module rather than re-declaring them — even if the values are correct at authoring time") made this a mandatory pre-read check. Without this pattern in my vocabulary, I might have noted the constants exist and moved on. Because the pattern is named, I checked: are these test constants imported from production? No. Are they used in assertions? No. Both failures present simultaneously.

**Resolution:** Remove `CC_QUOTA_STALE_THRESHOLD_MINUTES`, `CC_QUOTA_COLOR_GREEN_MAX`, and `CC_QUOTA_COLOR_YELLOW_MAX` from the test file. If any test needs to assert against threshold boundaries, import the constants from `src.orchestration.wos_dashboard`.

#### Gap 2: Assertion softening in color test (NEEDS_CHANGES)

`test_green_color_below_threshold` asserts:
```python
assert "green" in html.lower() or "#" in html  # color indicator present
```

The `or "#" in html` branch passes for any HTML containing any `#` character — including `#c0392b` (red) and `#a06000` (orange). The assertion never fails for any color output. It does not verify that green coloring appears. The actual green CSS value is `#1a7a3c` (defined in `_cc_quota_color`).

**How this pattern constrained my analysis:** The learnings.md PR #839 entry on assertion softening ("assertion softening to 'not-legacy-value' cannot distinguish wired-correctly from silently-failed-with-fallback") primed me to check whether color assertions are falsifiable. The `or "#" in html` addition is the exact form of softening — it makes the assertion pass regardless of which color is present. Without the named pattern, I might have read "color indicator present" and accepted the comment at face value.

**Resolution:** Replace the softened assertion with a specific check:
```python
assert "#1a7a3c" in html  # green CSS color
```
Alternatively, import and call `_cc_quota_color(40.0)` and assert the returned color appears in the HTML.

The same pattern in `test_red_color_at_or_above_90_percent` (`assert "red" in html.lower() or "#c0392b" in html or "#f87171" in html`) is looser than ideal but the `#c0392b` branch matches the exact production value — the `"red" in html.lower()` branch is redundant but not falsity-enabling. Advisory only, not a NEEDS_CHANGES blocker.

### Advisory findings (not NEEDS_CHANGES blockers)

**`fetched_at` variable scope:** `fetched_at` is assigned inside a `try` block and used after it. Python scope allows this, and the control flow guarantees `fetched_at` is set whenever the `age_seconds` line is reached (because `state` would have been set to `None` otherwise). Correct but brittle. A future refactor adding a new early-return condition could introduce a `NameError`. Suggest assigning `fetched_at = None` before the try block and adding a safety check, or restructuring to keep `fetched_at` clearly in scope. Advisory only.

**Timezone enforcement for `now` parameter:** `_format_cc_quota_widget(state, now)` performs `now - fetched_at` where `fetched_at` is always timezone-aware. If a caller passes a naive `datetime.now()`, this raises `TypeError`. The only production call site (`build_dashboard_data`) passes `datetime.now(timezone.utc)`, and all tests pass timezone-aware datetimes. The signature does not enforce this. A docstring note or type annotation `now: datetime` → `now: datetime` with a note about UTC requirement would close the gap. Advisory only.
