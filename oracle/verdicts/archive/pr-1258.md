### Stage 2: Quality Review

**Does it do what it claims to do?**

Partially. The design is correct. The sidecar filter (`_claim_expired`) is correctly implemented. The steward `claim_expired` handler is correctly implemented. The routing change in `reset_expired_claims()` is correct in intent. However, there is a critical correctness bug in the SQL query that makes `reset_expired_claims()` a silent no-op in production.

**Critical bug: SQLite datetime string comparison mismatch**

`reset_expired_claims()` uses this SQL to find expired claims:

```sql
SELECT id FROM uow_registry
WHERE status IN ('active', 'executing')
  AND claimed_until IS NOT NULL
  AND claimed_until < datetime('now')
```

The executor writes `claimed_until` using Python's `datetime.now(timezone.utc).isoformat()`, which produces:

```
'2026-05-22T02:04:08.265126+00:00'
```

SQLite's `datetime('now')` returns:

```
'2026-05-22 02:04:08'
```

SQLite string comparisons are lexicographic. The ISO 8601 `T` separator (`0x54`) is ASCII value 84, greater than the space character (`0x20`, value 32) used in SQLite's datetime format. As a result, `'2026-05-22T...' < '2026-05-22 ...'` is always **false** — the condition never fires.

This was confirmed by running the test suite:

```
FAILED tests/unit/test_orchestration/test_registry.py::TestResetExpiredClaims::test_reset_expired_claims_transitions_to_ready_for_steward
AssertionError: Expected uow_id 'uow_20260522_c7cd83' to be in reset_ids; got []
```

The fix is to wrap `claimed_until` in SQLite's `datetime()` function:

```sql
WHERE ... AND datetime(claimed_until) < datetime('now')
```

The same fix is needed in the SELECT query and the UPDATE `WHERE` clause:

```python
# In reset_expired_claims():
rows = conn.execute(
    """
    SELECT id FROM uow_registry
    WHERE status IN ('active', 'executing')
      AND claimed_until IS NOT NULL
      AND datetime(claimed_until) < datetime('now')
    """
).fetchall()
# ...
cursor = conn2.execute(
    """
    UPDATE uow_registry
    SET status = 'ready-for-steward', claimed_until = NULL, updated_at = ?
    WHERE id = ? AND status IN ('active', 'executing') AND datetime(claimed_until) < datetime('now')
    """,
    (now, uow_id),
)
```

**Note:** This bug likely predates this PR — the original `reset_expired_claims()` on `main` has the same string comparison in its SQL. The PR introduced new tests that would have caught this, but the tests were written against the buggy implementation, not a corrected one. The test failure is new information.

**What would break without the key decisions:**

- Without the sidecar filter: stall detection in the steward's observation loop is permanently masked for dead agents. The steward can never see `now - heartbeat_at > heartbeat_ttl + buffer` for a UoW whose agent crashed.
- Without the routing change to `ready-for-steward`: orphan retry budgets (`orphan_retry_count`, `ORPHAN_KILL_RETRY_BUDGET`, `MAX_RETRIES`) are bypassed. UoWs cycle indefinitely.
- With the SQL bug as written: `reset_expired_claims()` returns an empty list every call. Both fixes described above (sidecar filter + steward routing) are correct in the code path that executes after a UoW is returned by the SELECT — but no UoW is ever returned. The PR's primary fix is unreachable in production.

**Failure modes introduced by this PR:**

- None new beyond the pre-existing SQL comparison bug (which this PR exposes via tests).

**Patterns introduced:**

- `_claim_expired(uow, now)` as a pure function: well-structured. `getattr` with fallback correctly handles UoWs that predate the `claimed_until` field.
- The `elif event == "claim_expired"` branch in `_most_recent_return_reason` is dead code — the generic `if "return_reason" in note_data: return note_data["return_reason"]` check that precedes all event-type checks will always catch `executing_orphan` from the audit note before the `claim_expired` branch is reached. The PR description notes this ("the explicit branch makes the intent self-documenting and provides defense-in-depth if the generic check changes") — the rationale is valid but the comment in the code should note this explicitly to prevent confusion for future reviewers.

**What this makes easier/harder:**

- Easier: steward now sees expired-claim UoWs and can apply orphan retry budgets correctly. This closes the indefinite cycle.
- Harder: nothing.
