The theory of change: closing test coverage gaps in the LOS todo pipeline with integration tests that exercise real I/O (real SQLite DB, real filesystem, no mocking of storage layer) is the correct tool for a pipeline whose failure surface is at the seams between components. PR 4 of 5 is specifically the integration/E2E layer.

Adversarial prior held entering Stage 1: "this implementation is solving the wrong problem, or solving the right problem in a direction that forecloses better paths." Finding: no traction. The gap document (`docs/los/test-coverage-gaps.md`) was authored before implementation and correctly diagnoses the unverified territory (DB ↔ file ↔ subprocess seam). The tests are not over-engineered — they are placed precisely at the seam identified by the gap analysis, consistent with the seam-first abstraction golden pattern. The work is tests-only, introduces no Encoded Orientation decisions, and serves vision principle-4 (wire what exists before building more).

**Alignment verdict: Confirmed**

### Stage 2: Quality Review

**Test execution:** All 5 tests pass with `uv run pytest tests/los/test_e2e_todo_pipeline.py -v`: 5 passed, 0.80s.

**Mirror-constant check:** `tests/test_no_mirror_constants.py` fails with 7 violations. All 7 violations are pre-existing in other files (test_vault_watcher.py, test_wos_uow_detail_gen.py, test_cc_usage_poller.py, test_executor_heartbeat.py). Verified by running the test against both the parent commit (a41ac958, pre-PR) and the PR head — identical violation set in both cases. `test_e2e_todo_pipeline.py` appears in zero violations. The pre-existing failures are not introduced or worsened by this PR.

**Real I/O verification:** All 5 tests use `_make_db(tmp_path)` (real SQLite via `src.los.db.connect()`) and `_make_vault(tmp_path)` (real filesystem path under pytest's tmp_path). No mocking of DB or filesystem operations. The git operations in `_make_vault` initialize a real local git repo with no remote, causing `git_pull` to complete trivially (no remote configured) and `git_commit_and_push` to skip the push step — a clean approach to neutralizing network I/O without mocking.

**Test isolation:** Each test function uses a distinct `tmp_path` (pytest provides a unique directory per test). DB connections are closed at end of each test via `db.close()`. No shared state between tests. The global `/tmp/vault-processor.lock` used by `main()` in test 5 is released via the `finally: release_lock(lock_fd)` block — no lock leakage between tests in sequential execution.

**Distinct behaviors across 5 tests:**
1. `apply_status_delta` in the DB→file direction (done in DB + unchecked in file → flipped to [x])
2. `sync_obsidian_to_db` in the file→DB direction (checked in file + open in DB → marked done in DB)
3. `apply_status_delta` additions-block path (DB-only item → appended in lobster-additions block)
4. Idempotency of double run (same input produces same output on second invocation)
5. `main()` orchestration via dry-run path (full entry point, no write, no crash)

These cover different functions, different code paths within the same function, and different pipeline entry points. They are not trivially equivalent.

**DISABLE PROCESSING guard:** Tests 1-4 do not use the DISABLE PROCESSING guard in their vault files — this is correct, since `apply_status_delta` and `sync_obsidian_to_db` skip the guard line during processing, so its presence would not change the observable assertions in those tests. Test 5 includes the guard line in the initial_content with the comment "required by the guard check," but `main()` routes to `sync_obsidian_to_db` (not `apply_status_delta`) which does not filter the DISABLE PROCESSING line — meaning the guard line is parsed as a regular open item and inserted into the DB. The test asserts `count >= 1` which passes regardless of whether 1 or 2 items were inserted. The guard line is a red herring in test 5's context; it is not tested by this test, and the module docstring comment about it being "required" is inaccurate for the `sync_obsidian_to_db` code path.

**Advisory findings (non-blocking):**

1. **Softened assertion in test 5:** `assert count >= 1` rather than `assert count == 1` (or == 2 if the DISABLE PROCESSING line is also inserted). This matches the learnings.md pattern from PR #738 (assertion softening masks count regressions). The test's primary purpose (no crash + file unchanged) is sound; the count assertion is secondary verification that would benefit from being exact.

2. **Documentation inconsistency in test 2:** The separator comment at line 166 says "DB item stays open (DB is authoritative)" but the test body implements and asserts the opposite: the DB item is marked done. The docstring and assertion are correct; only the separator comment is wrong.

3. **Placement deviation from gap document:** The gap document specified `tests/integration/los/test_pipeline_e2e.py` (new directory). The actual placement is `tests/los/test_e2e_todo_pipeline.py`. This follows the established convention from PR 1137 (test_todo_obsidian_sync.py also landed in tests/los/), not a new deviation introduced here.

**Encoded Orientation check:** Not applicable. No change to behavioral defaults, system constraints, agent identity, or decision-making rules.

**Patterns introduced:** The test establishes a canonical pattern for LOS E2E tests using real SQLite + real filesystem + no-remote git repo initialization. The `_make_vault` + `_make_db` helper pattern is reusable for future LOS integration tests and follows the PR #805 positive pattern (env-var-at-call-time, real fixtures, no module-level constant divergence).

**What this makes harder:** Nothing structurally. The only forward risk is that test 5's loose `>= 1` count assertion will not catch future changes that alter how many items `sync_obsidian_to_db` inserts from a given vault content.
