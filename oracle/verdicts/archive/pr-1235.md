### Stage 1 — Vision Alignment (formed before reading implementation)

**Prior entering review:** This implementation is solving the wrong problem, or solving the right problem in a direction that forecloses better paths.

**Theory of change:** Bare `utils.X` imports are a latent runtime fragility. `conftest.py` masks them in tests by inserting `src/` onto sys.path; runtime invocations from the repo root (via MCP server entrypoints, cron, systemd services) have no such insertion. The venv `.pth` file adds the repo root only, making `from src.utils.X` the correct form. Fixing the imports and codifying a CONVENTIONS.md section prevents a category of `ImportError` that would surface only at runtime, not in CI.

**Alignment verdict: Confirmed.** The work is directly traceable to vision.yaml principle-1 ("proactive resilience over reactive recovery") — structural prevention of a runtime failure class before it manifests. The WOS executor is now live (current_focus: "WOS executor now live, enabled 2026-05-18"), which means MCP server and daemon invocations are on the critical path. A bare-import `ImportError` in `inbox_server.py` or `reliability.py` would break message delivery. Fixing this during active operation is correctly prioritized.

**What would have to be false for this to be wrong:** The bare imports would have to be harmless — i.e., the `.pth` file also adds `src/` to sys.path, or runtime invocations always set up the path correctly. Verified: the `.pth` file at `.venv/lib/python3.12/site-packages/_editable_impl_lobster.pth` contains only `/home/lobster/lobster` (repo root). `from utils.X` fails when `src/` is not separately on sys.path. The import is a genuine runtime risk in the listed files.

**Pattern citation — adversarial prior seeding (golden-patterns.md, 2026-03-27):** Applying this pattern, I held the prior through Stage 1 and looked specifically for the scenario where the import fix is solving the wrong problem. The scenario would be: if the files self-insert `src/` onto sys.path before the import, fixing the import is aesthetic, not functional. I checked each file in Stage 2 and found that some files with remaining bare imports do self-insert (making them safe), while the 7 fixed files did not have that protection in all cases (e.g., `outbox.py`, `circadian.py`, `reliability.py`, `message_store.py` lack the sys.path surgery). The prior was disconfirmed by evidence, not dismissed.

**What does CONVENTIONS.md foreclose:** The `from src.utils.X` convention encodes the directory hierarchy as a stable contract. It does not foreclose `utils` being restructured, but it does mean any restructuring of `src/utils/` would require updating call sites that use `from src.utils.X` — these are now grep-discoverable. This is the right tradeoff: explicit imports are the canonical over implicit path manipulation.

**Opportunity cost:** Minimal. This is a maintenance correctness fix on a directly active code path. The time-equivalent alternatives (documentation only, sys.path standardization, or doing nothing) all carry higher operational risk given the executor is live.

**CONVENTIONS.md Encoded Orientation check (constraint-3):** CONVENTIONS.md adds a new behavioral default for import style. This is an Encoded Orientation decision. The traceable vision.yaml anchor is principle-1 (proactive resilience). The prior logged decision basis is the oracle verdict for PR #1173, which flagged the bare import problem and is cited by ref in CONVENTIONS.md itself. Both elements are present. Constraint-3 is satisfied.
