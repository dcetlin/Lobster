The adversarial prior: is bloat the actual problem, or is it a symptom of a different architectural issue — agents using rolling-summary.md for a purpose that doesn't fit a snapshot model, or the file carrying load that more authoritative sources (session files, WOS registry, handoff doc) should be carrying?

## Stage 2 — Quality Review

**Encoded Orientation check (constraint-3):** This PR changes behavioral defaults in two agent definition files (.claude/agents/). The overwrite-vs-append choice is a durable behavioral default change that affects how memory is written every cycle. This is an Encoded Orientation decision per constraint-3. A prior logged decision in oracle/verdicts/ does not exist for this specific policy change. However, the vision.yaml constraint-4 anchor is traceable (unbounded accumulation degrades ergonomics), and the change is remedial rather than expansive — it fixes a concrete operational problem rather than introducing a new behavioral capability. The missing prior is a process gap, not a substantive alignment gap. Noted but not blocking.

**Does the change do what it claims to do?**
- compact-catchup.md Phase 3: old steps 14-16 (read, merge into sections, size-check) are replaced with a full-synthesis overwrite. The Stable Context section is carried forward explicitly. The size cap drops from 100 → 75 lines with a defined compression cascade (drop oldest Recent Decisions first, then merge Open Threads, then abbreviate Stable Context). The atomic write pattern (temp file + rename) is preserved. This is correct.
- nightly-consolidation.md step 3: old "Prepend a new dated entry" instruction is replaced with the same full-file overwrite synthesis using an identical structure and section rules. The proprioceptive gate-miss bullet is preserved and correctly placed in Open Threads. The atomic write pattern is added (it was absent before — this is an improvement). The "What NOT to do" rule is updated consistently.

**What would break without the key decisions made here:**
- Carrying forward only the Stable Context section (not all sections) is the correct discriminator. If all sections were carried forward, the overwrite would be no different from a merge. If nothing were carried forward, infrastructure/contact context would be silently lost on each run. The Stable Context carry-forward is the load-bearing design choice.
- The 75-line cap with a defined compression cascade prevents the cap itself from becoming a failure mode (the agent has explicit instructions for what to drop in what order).

**Failure modes:**
- The "Recent Decisions: last 7 days, max 5/3 bullets" rules differ between the two agents: compact-catchup says "max 5 bullets, drop older ones" while nightly-consolidation says "5-10 bullets max, prune any decisions older than 7 days." The compression cascade in compact-catchup says "drop oldest Recent Decisions beyond 3." These are inconsistent: at compression time compact-catchup drops to 3, but nightly-consolidation allows 5-10 with no compression-cascade minimum stated. An agent that runs nightly-consolidation after compact-catchup may expand Recent Decisions back to 10 bullets, defeating the compression the catchup agent applied. This is a minor inconsistency — not a breaking failure mode, but it means the 75-line cap enforcement is asymmetric across the two agents.
- Open Threads handling differs slightly: nightly-consolidation adds a required Proprioceptive bullet that compact-catchup does not include. This means the file structure varies depending on which agent last wrote it. Agents reading the file (including Phase 4 of compact-catchup, which reads back the rolling summary) will encounter a structurally variable file. This is low-risk but worth noting.
- The "bare python3" recurrence pattern from learnings.md (PRs #804, #821, #830) does not apply here — no inline Python invocations are present in these instruction files.

**Patterns introduced:**
- Overwrite-as-snapshot is now the canonical rolling-summary.md write pattern. This is a clean pattern that prevents accumulation by design. It aligns with the "structural prevention over reactive recovery" principle (vision.yaml principle-1).
- The Stable Context carry-forward convention is a named design choice: one section of a snapshot file survives full rewrites because its content is slow-moving. This is a reusable pattern for any memory file that mixes high-frequency and low-frequency content.

**What this makes harder:**
- Retrospective review of what was decided in rolling-summary.md across multiple days. The historical record now lives entirely in daily-digest.md and session files. If those sources are incomplete or inaccessible, the rolling-summary.md history is gone.
- Diagnosing why an agent made a decision 3 days ago based on context from rolling-summary.md at that time — the file no longer preserves that state.
