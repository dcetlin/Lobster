## Stage 1 — Vision Alignment (before seeing implementation)

**Prior entering review:** This implementation is solving the wrong problem, or solving the right problem in a direction that forecloses better paths.

**What would have to be true for this work to be the right path?** The WOS pipeline is maturing toward a phase where type safety at the data boundary has material consequences — specifically, where an invalid or unknown `type`, `posture`, or `register` value propagating silently would cause a misrouting or incorrect dispatch. The golden-patterns.md StrEnum pattern (2026-04-24) establishes this as the correct design direction for exactly this class of problem: "Raw string literals for enum-backed DB columns are forbidden." A PR that extends StrEnum boundary enforcement to three remaining untyped fields on the UoW dataclass is applying that golden pattern, not inventing a new one.

**What does this work foreclose?** Nothing consequential. StrEnum values compare equal to their string counterparts, so the migration is explicitly backward-compatible. No dispatch logic, routing logic, or state machine is touched.

**Is there a cheaper test of the underlying assumption?** The assumption is that unknown DB values for these three fields are a realistic risk worth enforcing. Given existing precedent (`UoWStatus` has had this enforcement since day one), this is not a hypothesis requiring a cheaper test — it is consistent completion of an established pattern.

**Alignment verdict: Confirmed.** The StrEnum golden pattern (2026-04-24) directly anchors this work. The pattern mandates: "Any function that returns one of N fixed string choices should define a StrEnum." The `type`, `posture`, and `register` fields are exactly this class. The work does not introduce a new design direction — it closes a gap the golden pattern named as unfilled. This constrained my Stage 1 assessment: the adversarial prior entered but found no foothold. The question "is this solving the wrong problem" is definitively answered by the pattern register: the wrong problem would be leaving these three fields as raw strings when the codebase already has a named golden pattern for exactly this situation.

- `UoW` dataclass fields `type`, `posture`, `register` are typed to those enums with correct defaults (`UoWType.EXECUTABLE`, `UoWPosture.SOLO`, `UoWRegister.OPERATIONAL`).
- `_row_to_uow` in registry.py correctly coerces all three at the read boundary (lines 476, 477, 496 on the PR branch) using the same `StrEnum(value or default)` pattern established by `UoWStatus`.
- `executor.py`: four occurrences of `row["register"] if row["register"] else "operational"` replaced with `UoWRegister(row["register"] or "operational")`. `ClaimSucceeded.register` type updated from `str` to `UoWRegister`.
- `steward.py`: local `UoWStatus` removed. The local version was missing `EXECUTING` and `CANCELLED` (confirmed in registry.py lines 57, 66). The removal closes a dual-source orientation risk — the local version was both incomplete and diverged from the canonical source.

### Patterns applied from learnings.md / golden-patterns.md

**StrEnum constants in all enum-backed SQL and routing logic** (golden-patterns.md, 2026-04-24): This PR applies Rule 1 (StrEnum constants at read boundary) and Rule 3 (StrEnum for fields with a finite domain). The citation is behavioral: the golden pattern is what I used to confirm Stage 1 alignment was genuine rather than the adversarial prior being masked by implementation quality. Without this pattern in the vocabulary, I would have evaluated "three new StrEnums" as a routine type annotation PR. With it, I identified this as a named completion of an established design arc.

**Mirror-constant pattern** (learnings.md, multiple PRs): Removing the duplicate `UoWStatus` from steward.py is the correct fix for exactly this pattern. The local definition was not only incomplete — it was a divergence risk. The steward had been missing `EXECUTING` and `CANCELLED` from its local copy, which means any steward code testing `isinstance(status, UoWStatus)` or using the local enum's `is_terminal()` or `is_in_flight()` methods would have produced incorrect results for those statuses. The citation is behavioral: the mirror-constant pattern caused me to look specifically at whether the removed local enum was genuinely a duplicate or whether it had meaningful differences. ### Failure modes examined

**Backward compatibility:** `StrEnum` values compare equal to their string counterparts. Existing code that passes strings like `"operational"` to functions accepting `UoWRegister` continues to work. Existing frozenset membership checks, SQL WHERE clauses, and dict lookups are unaffected. The PR description documents this explicitly and it is verified by the test results (2035 passing, 0 new failures).

**Unknown DB values:** The `StrEnum(value or default)` pattern means a `None` or empty string falls to the default. A genuinely unknown value (e.g., a future DB migration adds a new type not yet in the enum) raises `ValueError` at `_row_to_uow`. This is fail-fast and correct — it is the same pattern used by `UoWStatus` and the same behavior the engineer's briefing describes. There is no silent propagation path for invalid values after this PR.

**Missing coercions:** No call paths were found where `type`, `posture`, or `register` are read from the DB and not coerced. The read path funnels through `_row_to_uow` for Registry queries and through `_claim`'s direct `row["register"]` accesses in executor.py — both are covered.

**`_build_trace` register parameter:** `_build_trace` still accepts `register: str`, not `register: UoWRegister`. Since `UoWRegister` is a `StrEnum`, it is a `str` subclass — passing a `UoWRegister` value satisfies the `str` type annotation. This is not a defect; it is the designed backward-compatible migration path.

### What this makes easier / harder

**Easier:** Any future code that reads `uow.type`, `uow.posture`, or `uow.register` can use enum comparisons (`uow.register == UoWRegister.PHILOSOPHICAL`) instead of string literals. Static analysis can now detect invalid assignments at write time. Adding a new posture or register value requires touching one named definition, and the `ValueError` at `_row_to_uow` makes stale DB values detectable.

**No new hardness introduced:** The migration is backward-compatible by design. No caller changes are required.

### Test coverage

2035 tests passing with 0 new failures is appropriate for a type-annotation-plus-boundary-coercion PR. The pre-existing 23 failures are confirmed pre-existing on main. The PR correctly excludes `test_registry_cli.py` which has a pre-existing `SyntaxError` in the test file itself.

One minor observation, not a blocker: there are no new tests specifically exercising the `ValueError` raise path for the three new enums (passing an invalid value like `"bogus"` to `UoWType(...)`). However, this behavior is identical to `UoWStatus` which already has the same pattern and the same test coverage posture. The test gap is pre-existing and symmetric.
