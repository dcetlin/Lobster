## PR #1007 — fix(test): import constants from wos_queue_monitor instead of re-declaring locally

### Stage 1 — Right Problem?

The test file was maintaining local hard-coded copies of `STARVATION_CONSECUTIVE_READINGS = 12`, `TOXICITY_DEPTH_THRESHOLD = 10`, and `TOXICITY_CONSECUTIVE_READINGS = 3`. These were copy-pasted from the production module — a silent divergence hazard. If the production values change, the tests would still pass against the old values, masking regressions.

The oracle advisory from the S3-D review of PR #700 (same pattern as PR #696) correctly identified this. The fix is: import the constants from the already-loaded `qm` module. This ensures tests always validate against actual production values.

The old comment "Constants — must match the spec, not derived from implementation" is a reasonable testing philosophy in isolation, but it requires active maintenance to stay in sync. Since the production module is already loaded at test setup time (`qm = _load_module()`), importing from it has zero cost and eliminates the divergence risk entirely.
