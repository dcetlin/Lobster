## Stage 2 — Quality Review

### Cold-reading findings (before engineer's briefing)

Reading the diff first, the issues I flagged:

1. `except Exception: pass` in `generate_drilldown_urls` — catches and discards all exceptions including unexpected ones (not just ValueError/upload errors). Zero observability on failures.
2. `_status_badge_class` uses raw string literals — the StrEnum golden pattern from golden-patterns.md (2026-04-24) applies.
3. `"closed"` in the status mapping has no corresponding `UoWStatus` enum member — either phantom value or legacy string.
4. `--with-drilldowns` is silently ignored for non-HTML formats — potential operator confusion.
5. No HTML escaping on `uow_id` or `url` values in `render_html` — low severity for an internal tool.
6. Test `test_with_drilldowns_calls_generate_drilldown_urls` only verifies `called_once()`, not the argument content.

### Comparison against engineer's briefing

The engineer flagged items 1 and 4, confirming both. Items 2, 3, 5, and 6 were not flagged. Analysis below.

### Issue-by-issue findings

**Finding 1: Silent exception swallow with contradictory comment (not blocking)**

The exception handler reads:
```python
except Exception:
    # Log failure silently — one bad UoW should not abort the whole batch.
    pass
```

"Log failure silently" is a contradiction — no logging occurs. The docstring correctly says "silently omitted," but the in-line comment creates a false expectation that something is being logged. The failure contract (partial results are acceptable) is reasonable for a display function, and the test verifies a failing UoW is excluded while others succeed. The comment should read: "Silently omit — one bad UoW should not abort the whole batch." This is a comment quality issue, not a logic defect. Not blocking.

**Finding 2: `_status_badge_class` uses raw string literals — StrEnum golden pattern**

The StrEnum golden pattern (golden-patterns.md, 2026-04-24) states: "Any function that returns one of N fixed string choices should define a StrEnum." This function does the inverse — it *receives* a status string and maps it to a CSS class. However, the mapping keys (status values) are raw string literals rather than `UoWStatus` constants.

The exception: `"closed"` has no `UoWStatus` enum member. Checking registry.py line 1017, `"closed"` is a legacy DB value that appears alongside enum-backed values in a terminal-status list. Including it in the display mapping is intentional — it provides graceful display for legacy records. This means using `UoWStatus` constants exclusively would drop coverage. The design choice (raw string literals that cover both enum-backed and legacy values) is defensible for a display fallback function. The StrEnum pattern's intended target is routing and SQL logic — using it for a CSS display mapper that explicitly handles non-enum strings would require a separate mapping approach.

Assessment: the spirit of the pattern applies but the specific form (CSS display function covering legacy strings) is a legitimate exception. Not blocking.

**Finding 3: `"closed"` is a legacy value, not phantom**

registry.py line 1017 includes `"closed"` in a list alongside enum members. This confirms it's a live legacy value — not phantom (contrary to my initial suspicion from the learnings.md entry about phantom values in PR #976). The mapping is correct.

**Finding 4: No HTML escaping — low severity**

`uow_id` values follow the `uow_YYYYMMDD_slug` naming convention enforced at the DB layer. URLs from bisque are internally generated. The risk of HTML injection from either source is negligible in this internal tool context. Not a practical concern.

**Finding 5: Test gap on argument content**

`test_with_drilldowns_calls_generate_drilldown_urls` verifies `called_once()` but does not verify the UoW IDs passed to `generate_drilldown_urls`. With `_patch_all` returning empty registry data, `uow_ids` will be `[]` in the test. This verifies the call is made but not that it receives the correct data from the active+stalled UoW lists. A stronger test would provide mock UoW data and assert the specific IDs are collected. Mild gap, not blocking.

**Finding 6: Lazy import + mock path — verified correct**

`generate_drilldown_urls` does `from src.orchestration import wos_uow_detail_gen` lazily inside the function body, then calls `wos_uow_detail_gen.generate_and_upload(...)`. The tests mock `src.orchestration.wos_uow_detail_gen.generate_and_upload`, which patches the attribute on the module object in `sys.modules`. When the lazy import runs, it retrieves the same module object — the mock is in effect. The mock target is correct.

**Finding 7: Encoded Orientation check**

This PR adds a new CLI output mode to a utility script. It does not change behavioral defaults, system constraints, agent identity, or decision-making rules. Encoded Orientation constraint-3 does not apply.

### Design quality

The compositional structure is clean: `build_dashboard_data` is untouched, `generate_drilldown_urls` isolates all side effects, `render_html` and `_uow_id_cell` are pure functions. This directly extends the "pure function composition" pattern already present in the module. The `seam-first abstraction` golden pattern applies: the `drilldown_urls: dict[str, str]` parameter to `render_html` is a clean seam — future callers can provide URLs from any source without touching the renderer.

The `--with-drilldowns` gate (off by default) correctly addresses the slow-for-large-queues concern. The `ledger_path` override is plumbed through correctly. No existing functions are modified.

### Summary

No blocking issues. Three minor observations (comment wording, StrEnum applicability question, test argument verification gap) are noted but do not warrant NEEDS_CHANGES. The fundamental design is sound, the test coverage is adequate for the new code, and the mock strategy is correct.
