## Stage 1: Vision Alignment

**Finding (formed before reading implementation):**

The PR addresses a circular bootstrap failure in Migration 46: the guard `[ -d "/var/spool/cron/crontabs" ]` prevented adding the lobster user to the crontab group until the crontabs directory existed, but the crontabs directory is created by the first successful crontab write — which itself requires group membership. The theory of change is that removing this guard unblocks crontab group membership on fresh installs, enabling cron-scheduled tasks to work as designed.

Adversarial prior check: could this guard exist for a good reason (protecting against systems where cron isn't installed)? On a system without cron, neither the directory nor the `crontab` group exists — `usermod` would fail gracefully with a warning rather than silently skip. The cost of the guard (permanent silent failure on fresh installs) is higher than the cost of removing it (a `usermod` failure warn on systems without cron). The guard was checking the wrong invariant: directory existence as a proxy for "cron is ready," when group membership is the actual prerequisite.

Could the fix be narrower — e.g., checking for the group's existence (`getent group crontab`) rather than removing the guard entirely? No — checking for the group doesn't fix the problem; the group may exist while the user isn't in it, which is exactly the population Migration 46 needs to fix.

Alignment verdict: **Confirmed**. This aligns with `principle-1` (structural prevention preferred over reactive recovery) — the guard created a permanently unrecoverable failure on fresh installs. The fix is a correctness restoration, not a new feature. It does not foreclose any future direction and imposes no opportunity cost relative to the vision's stated priorities.

Encoded Orientation check: This is an infrastructure bootstrap fix (shell scripts, group membership). It is not an agent behavioral default change and does not require a logged prior decision under constraint-3.

## Stage 2: Quality Review

**Two files changed: scripts/upgrade.sh (+22/-5) and scripts/cron-manage.sh (+35/-2). No feature bundling — both files are explicitly named in the PR description.**

### upgrade.sh — Migration 46

The removed guard was `if [ -d "$CRONTAB_DIR" ] && ! id -nG "$USER" | grep -qw "crontab"`. The replacement is `if ! id -nG "$USER" 2>/dev/null | grep -qw "crontab"` — checking the actual invariant (group membership) rather than the proxy (directory existence). The `2>/dev/null` on `id` is appropriate defensive handling. The `CRONTAB_DIR` variable is removed cleanly since nothing references it after the guard removal.

The extended comment in the migration is load-bearing: it names the original failure mode and explains why the guard was removed. Per the canonical-placement principle, this prevents future reintroduction of the guard "for safety."

The `migrated` counter increment is preserved and correctly gated on `usermod` success — not on the attempt.

### upgrade.sh — preflight_checks()

New advisory check fires before migrations, warns if the user is not in the crontab group, and points to Migration 46. The warning text accurately states "if sudo is available" as the condition. This is correct — on systems where passwordless sudo is unavailable, the migration warns but cannot fix it. The preflight check is advisory only (does not block or exit), which is correct.

### cron-manage.sh — crontab_write()

The helper function:
- Creates a temp file for stderr capture
- Runs `crontab -` with stderr redirected to the temp file
- On failure: reads the temp file, checks for permission-related error patterns, and emits a structured actionable error with the exact `usermod` command
- On non-permission failure: re-emits the original stderr without modification
- On success: cleans up the temp file

Temp file cleanup is correct in both paths: the failure branch calls `rm -f "$tmp_err"` before returning 1; the success path calls `rm -f "$tmp_err"` at the end. No temp file leak.

The `set -euo pipefail` interaction is handled correctly: `if ! crontab - 2>"$tmp_err"` disarms `set -e` for the command, allowing controlled error handling.

The pattern match `grep -qi "permission denied\|mkstemp\|cannot create"` covers the known failure messages. Minor caveat: this is locale-dependent — on a non-English system the error text may not match. This is an edge case unlikely to affect the target deployment, and the non-match path still re-emits the original stderr (better than the old behavior of silent failure).

Both `add` and `remove` subcommands now use `crontab_write` instead of bare `crontab -`. The header comment documents the prerequisite (crontab group membership) and points to the fix.

### Failure modes

1. Locale dependency in error matching: on non-English systems, the permission-denied detection may not fire; the raw stderr is still emitted (not silently suppressed). Severity: low.

2. Passwordless sudo requirement: the preflight warning and migration fix both require passwordless sudo. On systems without it, the user sees the correct guidance ("run manually: sudo usermod -aG crontab $USER") but the automation cannot apply the fix. This is the correct behavior and is explicitly documented.

3. No automated test coverage: appropriate for shell script infrastructure. `bash -n` syntax checks plus functional testing of the add/remove path are the right verification posture for this change scope.

### What this makes easier

Any future crontab write operation goes through `crontab_write()`, meaning permission failures are automatically surfaced with actionable guidance. Fresh installs now succeed where they previously silently failed. The extended comment in Migration 46 prevents the guard from being reintroduced.

### Patterns

The `crontab_write()` helper applies the principle of making errors actionable at the point of failure rather than requiring the caller to diagnose them. This is consistent with the system's infrastructure pattern of emitting the exact command needed to fix an error rather than a generic error message.

No new patterns to append to learnings.md or golden-patterns.md — this is a narrow correctness fix with no novel structural decisions.
