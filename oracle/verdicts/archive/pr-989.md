### Import chain verification

The implementer flagged two concerns:

**Concern 1 — `WOSRegistry.__init__` does `from src.orchestration.paths import REGISTRY_DB`.**
When `inbox_server.py` runs as a systemd service with `WorkingDirectory=/home/lobster/lobster`, Python's `sys.path[0]` is `''` (empty string), which resolves to the working directory. This makes the `src` package findable even though the handler only explicitly adds `/home/lobster/lobster/src` to sys.path (not the repo root). The import works correctly in the service context. Risk is low, but the dependency on `sys.path[0] == ''` is implicit and would break if the service ever changes to `WorkingDirectory=/home/lobster/lobster/src`. This is advisory, not a blocking defect.

**Concern 2 — `handle_write_wos_heartbeat` uses `from orchestration.registry import WOSRegistry` (not `from src.orchestration.registry`).**
This is the lazy `sys.path` patch approach. The handler adds `/home/lobster/lobster/src` to sys.path and then imports `orchestration.registry` (not `src.orchestration.registry`). This creates a dual-module-identity risk: if any code elsewhere in the MCP session has already imported `src.orchestration.registry`, there will be two module objects for the same file under different sys.modules keys. In practice this is unlikely to cause problems here (WOSRegistry has no class-equality checks), but it is an established failure surface in this codebase. The test mock patches `sys.modules["orchestration.registry"]` to match the actual import path — this is correct and matches the production import.

**The implementer's stated risk about mock silently passing if the import path changes:** Confirmed risk. If the import in `handle_write_wos_heartbeat` is ever changed to `from src.orchestration.registry import WOSRegistry`, the test mock patches the wrong key and the test will hit the real DB (or fail on missing DB) rather than using the mock. This is the mirror-constant pattern (learnings.md 2026-04-22 and multiple recurrences) applied to mock keys: the mock key `"orchestration.registry"` is a string literal in the test that mirrors the import path in the production function. However, the risk is lower than typical mirror-constant cases because: (a) the import path is visible in the production code directly adjacent to where it would be changed, (b) the test failure mode is loud (real DB not found in CI) rather than silent (test passes with stale value). This does not block approval but is worth a targeted learnings.md entry.

### Failure modes

**rowcount=0 stop signal with outcome=failed:** The implementer asked whether this is the right response. The semantics are: the Steward re-queued the UoW because it appeared stalled; the agent receives rowcount=0 and must stop. Calling `write_result(outcome=failed)` tells the Steward the task was not completed successfully — which is accurate from the Steward's perspective (the agent didn't complete its intended work). The alternative (write_result with a neutral "stopped" outcome) would require a new outcome value in the result contract. `outcome=failed` is semantically imprecise (the agent wasn't defective; it was interrupted) but operationally correct: the Steward's retry logic will re-queue the UoW, which is the desired behavior. This is a pre-existing design choice from PR #851 and is not changed by this PR.

**DB locked exception in `handle_write_wos_heartbeat`:** Correctly handled — exceptions are caught, logged as warnings, and returned as `{"error": "..."}` without raising. The agent's execution continues. The heartbeat sidecar provides backup coverage. This is the correct fail-safe pattern for a fire-and-forget liveness signal.

**Empty/missing `uow_id`:** Handled before the try block — returns `{"error": "uow_id is required"}` immediately. Correct.
