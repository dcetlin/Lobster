The architectural-proposal-20260426.md §Category 4 documents this explicitly from a live incident and names the consolidation as a targeted fix at the notification routing seam — not a core mechanics change.

The consolidation operates only at the notification layer. The upstream infrastructure fixes named in the architectural proposal (retry accounting seam, trace.json) are explicitly deferred to separate PRs. The consolidation does not suppress the underlying signal; it batches the surface. The adversarial scenario is not confirmed.

**Cheaper test question:** Could inbox-level deduplication serve the same purpose? The inbox layer operates at message type but not at semantic batch identity — it has no knowledge that 19 retry_cap notifications share a root cause. The per-cycle accumulator in `run_steward_cycle` is the correct site because it has access to the shared context (same heartbeat execution) that makes consolidation meaningful. The cheaper test was not skipped.

**What this forecloses:** Nothing significant. ## Stage 2: Quality Review

### Area 1: `not dry_run` gate in `_process_uow`

**Before this PR:** `_send_escalation_notification(uow)` fired unconditionally in the retry-cap block — even in `dry_run=True` mode. This was a latent bug: dry_run is intended to suppress all side effects. The notification write to `~/messages/inbox/` is a side effect.

**After this PR:** The block is:
```python
if not dry_run:
    _effective_notifier = escalation_notifier or _send_escalation_notification
    _effective_notifier(uow)
```

This is correct and consistent with every other side-effecting call in `_process_uow`. The `not dry_run` gate belongs here. The fix is well-scoped: it gates only the notification call, leaving the `registry.transition` and `_append_cycle_trace` calls unaffected (they already have their own dry_run guards or are appropriate in dry_run).

**Interaction with injected notifier:** `escalation_notifier=None if dry_run else _collect_escalation` is passed from `run_steward_cycle`. This means in dry_run mode: (a) `_process_uow` receives `escalation_notifier=None`, (b) the `if not dry_run` gate fires False, so the `_effective_notifier` line is never reached. The double gate (outer `if not dry_run` in `_process_uow` + `None if dry_run` in the call site) is redundant but not harmful — either guard alone would be sufficient. The redundancy is defensive and does not introduce inconsistency.

**Test coverage:** `test_dry_run_does_not_send_any_notification` verifies both `mock_individual.call_count == 0` and `mock_consolidated.call_count == 0` in dry_run mode. Correct.

### Area 2: `_collect_escalation` closure — second DB round-trip via `_fetch_audit_entries`

The closure:
```python
def _collect_escalation(uow: UoW) -> None:
    _return_reason = _most_recent_return_reason(_fetch_audit_entries(registry, uow.id))
    _pending_escalations.append(EscalationRecord(uow=uow, return_reason=_return_reason))
```

This performs a DB read (`_fetch_audit_entries`) per escalating UoW. The PR description acknowledges this as a design choice and gives the rationale: passing `return_reason` through `_process_uow`'s return value would require a new `Surfaced` variant on `StewardOutcome`.

The design choice is acceptable. The second DB read is bounded to escalating UoWs only — not every UoW in the cycle. In the 19-UoW session-kill scenario (the motivating incident), this is 19 additional DB reads per cycle. `_fetch_audit_entries` is a simple SELECT on the `audit_log` table indexed by `uow_id` — not a join or aggregate. The cost is proportional to the exact scenario the consolidation is meant to address, and is low per-read. Acceptable.

**Alternative not taken:** If `_process_uow` returned the `return_reason` as part of its result, the closure could be simpler. The PR author correctly identified that this would require a new `StewardOutcome` variant — a non-trivial interface change. The current approach prefers a bounded DB read over an interface extension. This is consistent with the seam-first pattern (golden-patterns.md): the interface seam is preserved; the DB read is the implementation behind it.

**Potential null handling:** `_most_recent_return_reason` is called on `_fetch_audit_entries` output. If the audit log has no entries (unusual for a UoW at retry cap), `_return_reason` will be `None`. `EscalationRecord.return_reason` is typed `str | None`. `_build_consolidated_escalation_text` handles `None` via `r.return_reason or "unknown"`. The null case is handled correctly at all layers.

### Area 3: Per-cycle consolidation window (not time-based)

The consolidation window is the heartbeat cycle boundary — same `run_steward_cycle` invocation. The PR description correctly notes this is simpler than a time-window approach and avoids a DB query.

**Validity for the session-kill failure mode:** A session-end event kills all in-flight subagents simultaneously. The Steward's next heartbeat cycle processes all orphaned UoWs in one `run_steward_cycle` call. All escalations from that event appear in the same cycle. The cycle boundary is the correct consolidation window for this failure mode.

**Adjacent-cycle case:** If 19 UoWs from the same session kill happen to span two heartbeat cycles (e.g., the session ends between heartbeat runs, some UoWs are processed in cycle N and some in cycle N+1), the consolidation would not fire. This is a known limitation documented in the PR description. It is acceptable: the edge case is unlikely given that `run_steward_cycle` processes all ready-for-steward UoWs in one invocation, and the threshold (3) is low enough to catch even partial-cycle scenarios.

**Multi-fire check:** `_pending_escalations` is defined at the start of `run_steward_cycle` as an empty list. It is not persisted across cycles (no module-level state). No multi-fire. Consistent with the learnings.md PR #704 pattern requirement.

### Implementation correctness

**`_build_consolidated_escalation_text` purity:** Confirmed pure function. No I/O, no DB reads, no global state mutation. The `test_pure_function_no_side_effects` test verifies idempotence. The function truncates `uow.summary` at 80 characters — reasonable defensive measure for long summaries.

**`_send_consolidated_escalation_notification` structure:** Follows the existing `_send_escalation_notification` pattern exactly: `chat_id` from env var with fallback, UUID message ID, JSON to `~/messages/inbox/`, `OSError` caught and logged. The metadata includes `condition: "retry_cap_consolidated"`, `escalation_count`, `causes`, and `uow_ids` — all appropriate for machine-readable downstream handling (e.g., future dispatcher filtering by condition).

**`ESCALATION_CONSOLIDATION_THRESHOLD` placement:** Module-scope constant, typed as `int`, commented with rationale and spec reference. Consistent with the golden-patterns.md PR #938 rule (module-scope importability for tests). Tests import it directly. Correct.

**`EscalationRecord` dataclass:** `frozen=True` is correct — the record is immutable once created. Fields are `uow: UoW` and `return_reason: str | None`. The choice to carry the full `UoW` object rather than just `uow_id` allows `_build_consolidated_escalation_text` to include the summary without an additional DB lookup. Correct design.

### Test quality

603 lines of new tests for 149 lines of production code. The coverage matrix is complete:
- `_build_consolidated_escalation_text` — 6 tests covering count, cause, UoW ID presence, purity, threshold boundary, heterogeneous causes.
- `_send_consolidated_escalation_notification` — 2 tests covering file count and metadata correctness.
- `run_steward_cycle` integration — 5 tests: below-threshold (individual), at-threshold (consolidated), above-threshold (still one), state-transition independence (all UoWs enter needs-human-review regardless), dry_run (no notifications).

**One test quality note (non-blocking):** `TestSendConsolidatedEscalationNotification.test_writes_exactly_one_inbox_file` patches `src.orchestration.steward.Path` globally via `side_effect`. This is the same mock fragility pattern noted in PR #894 Round 2 ("Path interception via side_effect"). The mock intercepts all `Path(...)` calls and routes only those containing `"messages/inbox"` to the fake directory, falling through to the real `Path` for all others. The fragility: if `_send_consolidated_escalation_notification` adds a new `Path(...)` call that doesn't contain `"messages/inbox"`, the mock will route it to `real_path(...)` transparently, which may or may not be correct. For the current function body this is acceptable; noted for awareness if the function grows.

**State-transition independence test** (`test_all_uows_transition_to_needs_human_review_regardless_of_consolidation`) is particularly well-designed — it mocks both notification paths and directly queries the DB to verify status, confirming that notification consolidation does not accidentally suppress registry transitions. This is the critical invariant.

### What would break without the key decisions made here

- Without the `not dry_run` gate: dry_run mode would write inbox files, contaminating test environments and misrepresenting the dry_run contract.
- Without the injected `escalation_notifier` parameter: the cycle-level accumulator would have no injection point; consolidation could only be implemented by patching module-level globals, which is less correct and less testable.
- Without the per-cycle boundary (vs. time-window): the implementation would require a DB query to find recent escalations for the same return_reason, adding DB load and time-window ambiguity to every UoW processing path.

### Failure modes

**Below-threshold behavior:** If the session kill orphans exactly 1 or 2 UoWs (below threshold=3), individual notifications fire — unchanged from pre-PR behavior. Correct.

**Threshold at 3:** This is a parameter value, not a structural constraint. If Dan receives a consolidated message for 3 UoWs and prefers individual messages, the threshold can be raised. If 3 turns out to be too low (some 3-UoW coincidences are genuinely independent), this is tunable. The constant is named and importable.

**Mixed-cause waves:** If 19 UoWs escalate with 3 different return_reasons (not all `executor_orphan`), the consolidated message lists all causes via `cause_str = causes[0] if len(causes) == 1 else ", ".join(causes)`. The message correctly states "Shared cause: executor_orphan, session_error" rather than implying a single shared cause. The `test_heterogeneous_return_reasons_listed` test covers this. Correct.

**No known failure modes introduced.**

### Patterns introduced

The `escalation_notifier` injectable parameter on `_process_uow` establishes a pattern: per-UoW side-effecting calls in the Steward loop can be injected for test isolation and for cycle-level accumulation. This is a well-placed seam (golden-patterns.md seam-first pattern): the injection point sits at the boundary between per-UoW processing and cycle-level aggregation, which is precisely where the evolutionary tension is located.

### What this forecloses

Nothing. Individual `needs-human-review` transitions are preserved. The notification accumulator is purely additive. No upstream infrastructure path is constrained. The threshold constant is tunable. The follow-on button-handler PR remains unblocked.
