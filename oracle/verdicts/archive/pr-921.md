## Stage 1: Vision alignment

**Prior entering review:** This implementation is solving the wrong problem, or solving the right problem in a direction that forecloses better paths.

**Theory of change in vision.yaml:** The WOS execution pipeline exists to reduce friction and screen dependency (constraint-4) while enabling routing decisions traceable to vision.yaml (constraint-2). The pipeline is only valuable if it executes reliably — orphaned UoWs that silently consume a dispatch event without spawning work are a direct structural failure. The confirmed root cause (three orphan incidents for uow_20260424_4fc5c8 in 36 minutes) is not ambiguous: the gate lived in prose instructions, prose does not survive context compaction, Python imports do.

**What would have to be true for this work to be the right path?** The dispatcher calling `route_wos_message` on compaction-surviving Python import is already in place. The missing piece was the function's own internal guard — if the dispatcher calls `route_wos_message` correctly but the function returns an action other than `spawn_subagent`, the dispatcher's behavior depends on whether it reads the action correctly. The guard makes `route_wos_message` fail-loudly rather than silently returning an action that causes `mark_processed` without spawn.

**Is there a cheaper test of the underlying assumption?** The assumption is confirmed, not hypothetical. Three operational incidents with identical fingerprint (3–14 seconds between mark_processing and mark_processed, no session_start, no Task call) leave no ambiguity about the root cause. A cheaper test would have been monitoring — but the monitoring reveals the failure mode, not an alternative fix path.

**What does this work foreclose?** Nothing. The guard is additive. The normal dispatch path is unchanged. The only new behavior is the error path (exception → send_reply, unexpected action → send_reply). Both of these were previously silent failures.

**Is the guard in the right place?** Yes. The correct fix layer is `route_wos_message` itself because: (a) the dispatcher calls it as a Python import that survives compaction, (b) the function is the single entry point for all WOS message routing, and (c) placing the guard here means it fires regardless of which handler is called. Placing the guard in the dispatcher's prose would re-create the exact vulnerability being fixed.

**Encoded Orientation check (constraint-3):** This PR does not change behavioral defaults, system constraints, agent identity, or decision-making rules in a durable way. It hardens an existing routing function's internal invariant. The wos_execute dispatch was already oracle-approved behavior. The `chat_id: 0` sentinel addition is an advisory fix for a previously approved decision (PR #914 oracle r2 named it explicitly). No new Encoded Orientation decision is introduced.

**Alignment verdict:** Confirmed. The work directly addresses a confirmed operational failure mode on the critical execution path. It is traceable to vision.yaml principle-1 (proactive resilience over reactive recovery) and principle-3 (determinism over judgment for conditionals — the guard is a code assertion, not an LLM instruction). The opportunity cost is negligible: 78 lines added to one function, 205 lines of tests.

## Stage 2: Quality review

### Does it do what it claims?

Yes. The diff wraps the dispatch block in a `try/except Exception` and adds a post-dispatch assertion `result.get("action") != "spawn_subagent"`. The two failure paths both return `action="send_reply"` with diagnostic text. The normal path is unchanged.

The PR also adds `chat_id: 0` and `source: system` to the `handle_steward_trigger` prompt frontmatter, completing the advisory fix from PR #914 oracle r2.

### ValueError handling — the engineer-flagged concern

The engineer asked: the `else: raise ValueError(...)` inside the `try:` block — is it acceptable for the outer `except` to catch it?

Looking at the diff:

1. The unrecognized-type `ValueError` fires at lines 568-572, BEFORE the `try:` block. This guard remains outside the `try/except` and re-raises unconditionally. This is correct: a message type not in `WOS_MESSAGE_TYPE_DISPATCH` should not be swallowed — the dispatcher's pre-check (`if msg_type in WOS_MESSAGE_TYPE_DISPATCH`) prevents this path in normal operation, and if something bypasses that, raising is the right behavior.

2. The `else: raise ValueError(...)` at the bottom of the `try:` block (the exhaustiveness guard) is inside the `try/except`. If somehow an unrecognized type passed the first guard and entered the dispatch block, it would hit this `ValueError`, which would then be caught by the `except Exception as exc` and returned as a `send_reply` alert. This is acceptable behavior: the alert message contains the exception text including the type name, so Dan would see which message type caused the failure. The alternative (re-raising) would propagate the exception to the dispatcher, which may or may not handle it gracefully. Returning `send_reply` is the safer choice.

The only scenario where this matters: a type is added to `WOS_MESSAGE_TYPE_DISPATCH` but no handler branch is added in `route_wos_message`. In that case, the first guard passes (type is registered), the dispatch block hits the `else: raise ValueError`, the `except` catches it, and Dan gets an alert. This is strictly better than a propagated exception with no user notification.

**Assessment:** The ValueError handling is acceptable. The engineer's concern is valid as a structural observation but does not require a change — the behavior is correct at both failure surfaces.

### try/except scope

The `except Exception` catches all exceptions from the dispatch block. This is deliberate and appropriate here: the function's contract is to return a routing result, never to raise. The caller (dispatcher) should not need exception handling around `route_wos_message`. The exception is logged at ERROR level with type and message before the alert is returned, so the failure is visible in logs.

One observation: `import logging as _logging` is placed inside the `except` block (twice — once in the exception handler and once in the spawn-gate failure path). This is a minor style issue, not a correctness issue. The import is idempotent in Python — `logging` is already in `sys.modules`. No functional impact.

### Test coverage

26 new tests across two test classes: `TestRouteWosStewardTrigger` (9 tests) and `TestRouteWosMessageSpawnGate` (5 tests).

**What the tests cover:**
- `steward_trigger` registered in `WOS_MESSAGE_TYPE_DISPATCH`
- `route_wos_message` returns `spawn_subagent` for both `wos_execute` and `steward_trigger`
- `handle_steward_trigger` prompt contains `chat_id: 0`
- Spawn-gate fires when handler returns non-`spawn_subagent` action (monkeypatched)
- Spawn-gate fires when handler raises (both `wos_execute` and `steward_trigger`)
- Negative assertion: neither message type returns `mark_processed`

**What the tests do not cover:**
- The unrecognized-type `ValueError` path (pre-try block) — but this is existing behavior, not new
- The `else: raise ValueError(...)` exhaustiveness path — also pre-existing, newly wrapped in try/except
- The exact alert text content beyond checking for "alert" or "error" substring — this is adequate

The monkeypatch approach for the spawn-gate test is correctly scoped: `handle_steward_trigger` is the right vector because it returns a full action dict, while `handle_wos_execute` is a prompt-builder that returns a string. The test correctly simulates a regression to `{"action": "noop"}`, which was the pre-PR #914 behavior.

**Pattern citation — advisory-to-hook promotion (golden-patterns.md 2026-04-22):** This PR does not promote the spawn-gate to a hook. The existing `wos-execute-gate.py` PostToolUse hook is acknowledged as a belt-and-suspenders layer. The in-function guard and the hook serve different failure modes: the hook catches dispatcher-level failures (calling `mark_processed` before spawn), the in-function guard catches handler-level failures (spawn command constructed incorrectly). Both layers are appropriate. The pattern would not apply here as a required action — the hook layer is already present.

**Pattern citation — learnings.md PR #914 (synchronous subprocess in routing function):** This PR does not introduce any synchronous work inside `route_wos_message`. The guard is a pure assertion on the return value; no I/O, no subprocess, no external calls. The failure mode named in PR #914 does not apply.

### What patterns does this introduce that will propagate?

The `route_wos_message` function now enforces its own invariant rather than relying on callers to validate its output. This is the correct pattern for functions that serve as compaction-resilient entry points: the function, not its callers, is responsible for maintaining the spawn-subagent contract. Any future handler added to the dispatch table benefits from this guard automatically — it does not need to be remembered, because it is structural.

### What does this make harder?

Nothing of consequence. A future handler that legitimately needs to return a non-`spawn_subagent` action (if such a need arises) would need to remove or bypass the guard. But the function's design contract is explicit: all WOS message types produce a subagent spawn. If a future type needs different behavior, it belongs in a different routing function, not `route_wos_message`.

### Encoded Orientation check

This PR does not constitute an Encoded Orientation decision. It hardens an existing behavioral contract that was already oracle-approved. No new defaults, constraints, or decision rules are introduced.

## Summary

The PR is correctly motivated, correctly placed, and well-executed. The spawn-gate guard is in the right layer (the function itself), uses the right failure mode (fail-loudly with user alert rather than silently), and does not disturb the normal path. The ValueError handling is acceptable. The steward_trigger advisory fix closes the PR #914 oracle r2 gap. Test coverage is adequate for the new behavior.

The only non-blocking observations:
1. `import logging as _logging` placed inside exception handlers rather than at module scope — cosmetic, functionally inert.
2. The exhaustiveness-guard `ValueError` is now catchable by the outer `except` — behavior is correct (alert returned), and this is the better outcome compared to propagating the exception.

Neither observation requires a change.
