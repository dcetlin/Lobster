### Stage 1: Vision Alignment

**Prior entering review:** This implementation is solving the wrong problem, or solving the right problem in a direction that forecloses better paths.

The implicit theory of change in vision.yaml: WOS executor is live (execution_enabled=True, 2026-05-18). The active constraint is the WOS bidirectional interface (#1118). Work is in flight. The system needs agents to execute UoWs reliably. When a UoW returns `outcome=blocked` because the dispatch template adds a boundary that directly contradicts the UoW's prescription, that is an execution loop failure.

The adversarial question: is the real problem that WOS source file modification is being sanctioned through the executor at all? Should executor.py and registry.py be modifiable by WOS-dispatched subagents?

Evidence check: The failing UoW (`uow_20260519_5f17b8`) was dispatched to implement the visibility-timeout model — a WOS-internal implementation task that requires modifying executor.py and registry.py. The prescription told the subagent to modify those files; the template told it not to. The subagent correctly identified the contradiction and returned `outcome=blocked`. This is not a case of the subagent being overcautious — it is a case of the template being wrong for WOS-internal tasks.

**What this work forecloses:** Removing the global boundary removes structural enforcement for non-WOS-internal UoWs that genuinely should not touch WOS source files. Protection for those UoWs now rests entirely on per-prescription boundary quality. The prescription audit (2026-05-03) showed only 66% of prescriptions include a `Boundary` line — this is pre-existing and is a Steward quality issue, not introduced by this PR.

**Alignment verdict: Confirmed.** The blanket template boundary was in the wrong register (template vs. prescription). Removing it and delegating to per-UoW prescriptions is the discriminator-improvement pattern from learnings.md (2026-04-22): "when a behavioral rule isn't followed, improve the discriminator — do not add more rules." The vision.yaml principle-5 ("Discriminator improvement over rule addition") is the anchor.

**Patterns from learnings.md and golden-patterns.md that shaped this Stage 1:**

- learnings.md "instruction-accumulation pattern" (2026-04-22): the hardcoded boundary was an instance of this anti-pattern — a stronger rule added at the template level that conflicted with discriminator-appropriate per-UoW prescription. Recognizing this pattern caused me to confirm rather than question the removal.
- golden-patterns.md "adversarial prior seeding" (2026-03-27): I actively sought the scenario where removing this line is a mistake. The scenario would be: the Steward reliably prescribes bad boundaries (or none) for WOS-internal tasks. The evidence from the prescription audit partially supports concern, but the correct fix is Steward prescription quality, not a blanket template constraint that blocks valid execution.

### Stage 2: Quality Review

**The change:** One line deleted from `src/orchestration/dispatcher_handlers.py` line 696:
```python
f"Boundary: do not modify executor.py, registry.py, or any WOS source files.\n"
```

**Does it do what it claims to do?** Yes. The removal is surgical. The line was in the final f-string of `handle_wos_execute` and was appended to every WOS dispatch prompt unconditionally.

**Does it correctly identify the location?** Confirmed. The line at position 696 in the post-PR file (context: after `Minimum viable output:` line, before the closing paren) is exactly the identified boundary string.

**Encoded Orientation check (constraint-3):** The removal constitutes a change to durable behavioral defaults — all WOS-dispatched subagents now receive no blanket source file boundary from the template. However, this is a removal of incorrect Encoded Orientation (the blanket constraint was never logged as a deliberate design decision and directly contradicted legitimate UoW prescriptions). The operational evidence of harm (uow_20260519_5f17b8 outcome=blocked) is the logged prior. vision.yaml anchors: `principle-5` ("Discriminator improvement over rule addition"), `active_project.current_phase` (WOS executor live, bidirectional interface in progress). Constraint-3 is satisfied: the removal is traceable to a vision.yaml anchor and the operational failure is the logged evidence of harm.

**What failure modes exist?**

1. **(Non-blocking, pre-existing)** ~34% of WOS prescriptions lack an explicit `Boundary` line (prescription audit, 2026-05-03). After this PR, those UoWs have no explicit boundary in either the template or the prescription. This was already true before — the template boundary was wrong for WOS-internal tasks and absent for non-WOS tasks that didn't need it. The gap is a Steward prescription quality issue that predates this PR.

2. **(Non-blocking)** Previously-generated dispatch prompt files (e.g., `wos-uow-0b408e/dispatch-prompt.md` line 119) already contain the removed boundary line baked in. These are historical artifacts from prior dispatches and are not re-used for new dispatch; they do not require cleanup.

3. **(Not a failure)** The `uow_20260517_0b408e` dispatch-prompt.md shows the removed line was being written into stored dispatch artifacts. After this PR, newly-dispatched UoWs will not receive this line. Existing stored artifacts are unaffected.

**What patterns does this introduce?** The per-UoW prescription becomes the sole boundary-setting mechanism. This is correct. It aligns with the CLAUDE.md dispatch template gate requirement ("Every subagent Task call must include `Boundary: do not produce [X]` in its prompt") which places the boundary obligation on the prescription author, not the template.

**Feature-bundling check:** 0 lines added, 1 line deleted, 1 file changed. No bundling.

**Mirror-constant check:** No constants introduced or changed. Not applicable.

**Finding worth noting:** The prescription audit's 34% missing-Boundary finding (filed 2026-05-03) becomes more load-bearing after this PR. The dispatch template gate in CLAUDE.md requires `Boundary` in every subagent prompt — but the Steward is the prescription author for WOS UoWs, not the dispatcher. There is no validation gate on the Steward's prescription output to enforce `Boundary` presence. The prescription audit's own recommendation was: "Add automated validation: when the Steward writes a prescription artifact, check that the instructions string contains 'Minimum viable output:' and 'Boundary: do not'." This is a follow-on concern for a separate PR, not a blocker for this one.

**Overall assessment:** This is a minimal, correctly targeted one-line removal that unblocks a live failing UoW and removes an incorrectly-placed blanket constraint. The quality is high, the scope is precise, and the alignment is confirmed.
