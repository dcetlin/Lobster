**Formed before reading implementation.**

The theory of change: the executor-heartbeat sidecar writes `heartbeat_at` for all in-flight UoWs every 3 minutes, regardless of whether the agent is alive. This means a dead agent looks live to the observation loop. The fix is a discriminating signal that only an alive agent can write — `token_usage IS NOT NULL` in `uow_heartbeat_log`. §2 of the multiposture spec (oracle-approved in #1237) defines this approach.

This is a known tension the spec acknowledges, and Change 1c (sidecar demotion) is the structural completion. The discriminator choice is correctly anchored to the approved spec decision.

**Does this foreclose any direction?** Change 1c (deferred) will demote the sidecar, which means the current discriminator is a transitional mechanism, not the final state. This PR's design is consistent with that trajectory.

### Gap 2: Hardcoded `'executing'` as `from_status` in audit log INSERT — incorrect for `'active'` UoWs

In `record_sidecar_masked_stall`:

```python
INSERT INTO audit_log
    (ts, uow_id, event, from_status, to_status, agent, note)
VALUES (?, ?, 'stall_detected', 'executing', 'ready-for-steward',
        'observation_loop', ?)
```

The preceding UPDATE transitions from `status IN ('active', 'executing')`. If the UoW is in `'active'` status at transition time, the audit trail records `from_status='executing'` — factually incorrect. The parallel function `record_heartbeat_stall` uses `'active'` as its hardcoded from_status (line 3125), which raises the same category of concern (neither derives from actual state), but the sidecar-masked version gets the value wrong for the `'active'` case. Several existing functions (e.g., `write_heartbeat` at line 2093, `close_uow` at line 2307) derive `from_status` from the DB row before transitioning — this is the correct pattern.

### Gap 3: SQL hardcodes `+300 seconds` grace window in NOT EXISTS subquery, mismatched with `min_age_seconds` parameter

The outer query age check uses the `min_age_seconds` parameter:

```sql
AND CAST((julianday(?) - julianday(r.updated_at)) * 86400 AS INTEGER) > ?
```

But the NOT EXISTS subquery hardcodes `+300 seconds`:

```sql
AND l.recorded_at > datetime(r.updated_at, '+300 seconds')
```

The two 300-second values are semantically intended to be the same threshold (the startup grace window from the spec). When called with a non-default `min_age_seconds` (e.g., in tests or if the threshold is tuned), the outer and inner windows become desynchronized. A UoW 400s old with a heartbeat at second 350 passes the NOT EXISTS check (heartbeat is after the hardcoded 300s mark) even when the outer check says it's past the startup window. The constant `SIDECAR_MASK_MIN_AGE_SECONDS = 300` is defined but not used in the SQL — it should be passed as a parameter to the `datetime()` modifier rather than hardcoded. This is also an instance of the learnings.md pattern "any time-window comparison in an orphan classifier must use a named constant, not an inline literal" (2026-04-26, PR #968).
