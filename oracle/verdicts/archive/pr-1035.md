## Stage 2: Quality Review

### Does it do what it claims to do?

Yes. Both write paths (`_backpropagate_result_to_output_file` and `_enrich_result_file`) now accept `token_usage: int | None = None` and thread the value to the on-disk result.json. `maybe_complete_wos_uow` passes the parameter to both call sites. The 4 tests cover the meaningful behavioral cases.

### Implementer-flagged concern 1: early-exit logic

**Original:** `if not result_text: return`
**New:** `if not result_text and token_usage is None: return`

This is logically correct. The function only skips when there is truly nothing to write. When `result_text=None` and `token_usage` is provided, the code correctly:
- Passes the early-return guard (token_usage is not None)
- Enters the try block
- Executes `else: refs = {}` (because `result_text` is falsy)
- Writes only `token_usage` to the payload
- Sets `enriched = True`
- Performs the atomic rewrite

The logic handles all four cases: (None, None) → early return; (None, int) → write token_usage only; (str, None) → write summary/refs only; (str, int) → write all. Each case is handled correctly.

### Implementer-flagged concern 2: `enriched` flag

The flag prevents unnecessary atomic rewrites when no fields change. Evaluated across all cases:

- If nothing changed (summary present, refs present, token_usage present) and `enriched` remains False: function returns without rewriting. **Correct** — no mutation needed.
- If only token_usage changes (not present → written): `enriched = True`, atomic rewrite fires. **Correct.**
- If summary or refs change and token_usage is already present: `enriched = True` from the summary/refs block, atomic rewrite fires. **Correct.**
- If `result_text=None` and `token_usage` is new: `enriched = True` from the token_usage block, rewrite fires. **Correct.**

The `enriched` flag does not skip any write that should happen. The concern is resolved.

### External gate check (learnings.md PR #891 pattern)

PR #891 learning: "When a back-propagation function writes a synthetic record with unconditional field values, document the external gate that guarantees the unconditional values are correct." This PR's `_backpropagate_result_to_output_file` adds `token_usage` conditionally (`if token_usage is not None`), not unconditionally. The existing unconditional fields (`success: True`, `outcome: "complete"`) are unchanged and remain gated by `WRITE_RESULT_SUCCESS_STATUS` at the outer function level. No new unconditional field is introduced. This learning does not produce a gap.

### Mirror-constant check (learnings.md recurring pattern)

The test file declares `TOKEN_USAGE_FIELD = "token_usage"`. This is a test-local string constant. The production code uses `payload["token_usage"]` as a literal, with no corresponding named constant in production. This is not a mirror-constant risk — the mirror-constant pattern applies when a production constant exists and is re-declared locally rather than imported. Here no production constant exists to mirror. The test constant is a legitimate readability aid.

### `refs` variable initialization

When `result_text` is None, the `else` branch sets `refs = {}` before the debug log references it. No NameError risk.

### Test completeness

Four tests, four behavioral cases:
1. `test_token_usage_written_to_backpropagated_result_json`: backprop path (no existing result.json). Token_usage appears in the created file. Checks exact value.
2. `test_token_usage_written_when_result_json_already_exists`: enrich path (result.json pre-exists). Token_usage added without clobbering existing fields.
3. `test_token_usage_absent_when_not_provided`: token_usage=None path. Correct that the file is created (result_text provided triggers backprop) and assertion runs. Field must be absent.
4. `test_existing_token_usage_in_result_json_is_not_overwritten`: pre-existing token_usage=5000 must survive when token_usage=9999 is passed. No-overwrite invariant verified.

All four tests assert exact values, not substring or softened assertions. No `>=` softening or loose-matching anti-patterns.

### Feature-bundling check

2 files changed: `src/orchestration/wos_completion.py` and `tests/unit/test_orchestration/test_wos_completion.py`. Both are described in the PR. No bundling.

### Encoded Orientation check (constraint-3)

This PR adds a new field to result.json. It does not change behavioral defaults, agent identity, system constraints, or decision-making rules. Not an Encoded Orientation decision. Constraint-3 does not apply.

### What failure modes exist?

1. **`_enrich_result_file` called when result.json does not exist, with only `token_usage` provided.** The file-not-found check returns early (`result_path.exists()` → False → logs debug, returns). Token_usage is silently not written. This is acceptable: the `_backpropagate` function runs first and creates the file with token_usage already included. The sequence in `maybe_complete_wos_uow` is: backprop first → then enrich. If backprop creates the file with token_usage, enrich's subsequent call sees token_usage already present and `enriched` stays False. No atomic rewrite — correct behavior.

2. **Double-call race:** If `_enrich_result_file` is called twice with different `token_usage` values, the second call sees `"token_usage" in payload` is True and skips. First-write wins. This is documented intent ("never overwrite an existing value").

### What patterns does this introduce that will propagate?

The conditional-parameter extension pattern (`param: int | None = None`) for pure function extension is clean and does not require caller changes. This is a safe pattern for adding optional enrichment fields to result.json in the future.

### What does this make easier/harder?

**Easier:** Any tooling that reads result.json files can now surface per-UoW token cost without a SQLite query. The oracle, agent-status.sh, and post-hoc analysis scripts all benefit.

**Harder:** Renaming the field in result.json later requires updating both the DB schema and this enrichment path. Minor coupling.

### Verdict

The implementation is correct. Both flagged concerns are sound. The 4 tests provide adequate behavioral coverage. No patterns from learnings.md produce a gap against this PR.
