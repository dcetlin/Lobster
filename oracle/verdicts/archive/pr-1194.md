## Stage 1: Vision Alignment

**Prior entering review:** This implementation is solving the wrong problem, or solving the right problem in a direction that forecloses better paths.

**Theory of change check:** The issue is a real operational hazard — upgrade.sh kills the active dispatcher mid-upgrade, before migrations run. The fix removes `lobster-claude` from `restart_services()`. For this to be the right fix, it would have to be true that: (a) the dispatcher does not need to restart to pick up upgrade changes, and (b) removing it from automatic restart does not foreclose a needed upgrade path.

**What would have to be wrong:** If `lobster-claude` needs to restart after some class of upgrades (new hooks, new bootup files, CLAUDE.md changes that the dispatcher must pick up fresh), then removing it from `restart_services()` leaves operators with no upgrade-time restart path and no guidance on when to trigger one manually. This is the main Stage 1 risk.

**What this forecloses:** Automatic dispatcher restarts during upgrades that require them. The PR description partially addresses this with "The dispatcher restarts itself on the next message cycle" — but this statement is slightly inaccurate (see Stage 2). The fix is correct directionally; the gap is whether the documentation of when NOT to use it is sufficient.

**Encoded Orientation check (constraint-3):** This PR changes a behavioral default in upgrade.sh (what gets restarted during upgrades). Is this an Encoded Orientation decision? The change affects system behavior during a well-defined operational procedure, not agent routing or decision-making. It does not change behavioral defaults, agent identity, or decision-making rules in a durable way for the dispatcher itself. It changes infrastructure lifecycle behavior in a script. Constraint-3 does not apply. No logged prior required.

**Vision.yaml alignment:** Vision.yaml principle-1 ("Proactive resilience over reactive recovery") and constraint-4 ("minimize metabolic cost of cybernetic engagement") both favor this fix. A dispatcher killed mid-upgrade is a pure availability loss that increases operator metabolic cost to diagnose and recover. The fix is aligned.

**Alignment verdict: Confirmed.** The problem being solved (dispatcher termination mid-upgrade) is the right problem. The fix direction (remove from restart_services) is correct.

## Stage 2: Quality Review

### What the PR does

The diff is minimal and precise:
1. Updates the dry-run message to reflect the new behavior.
2. Removes `lobster-claude` from the `local services=(...)` array in `restart_services()`.
3. Adds an explanatory comment at the removal site (4 lines) naming the root cause and referencing issue #1107.
4. Adds an `info` line at the end of `restart_services()` confirming the skip was intentional.

### Does it do what it claims to do?

Yes. `lobster-router` and `lobster-slack-router` still restart. `lobster-claude` is excluded with a comment that gives operators the reasoning. The dry-run message is updated consistently with the behavioral change.

**Step ordering verified:** The main() call sequence in upgrade.sh shows:
- `restart_services` is called at step 7
- `run_migrations` is called at step 9

The PR's stated root cause is correct — the dispatcher was being killed two steps before migrations ran.

### Is "restarts itself on the next message cycle" accurate?

Partially inaccurate, but benign. Looking at `claude-persistent.sh` and the service definition: `RemainAfterExit=yes` + `Restart=on-failure`. When `systemctl restart lobster-claude` is called, it kills the tmux session (ExecStop) and relaunches (ExecStart), restarting `claude-persistent.sh` and its inner loop. By *not* calling `systemctl restart`, the existing process is simply never killed. The dispatcher continues running uninterrupted — it does not "restart itself." The PR description would more accurately say "The dispatcher continues running without interruption — no restart is triggered or needed." This is a documentation inaccuracy, not a behavioral defect. The actual code change is correct regardless of how the PR describes the mechanism.

### Correct services still restart?

Yes. `lobster-router` remains in the services array. `lobster-slack-router` is handled in a separate conditional block that is unmodified. No regression here.

### Failure modes?

**The one real gap this PR introduces:** There is no guidance for operators on when a manual dispatcher restart IS needed after an upgrade. Some classes of changes — new hooks installed by upgrade.sh migrations, changes to `settings.json`, new MCP server configuration — may require the dispatcher to restart to take effect. The PR removes the automatic restart without providing a decision rule for when manual restart is appropriate.

The comment says "The dispatcher restarts itself on the next message cycle after upgrade.sh completes" — which is both inaccurate (it doesn't restart; it continues) and potentially misleading for operators who have made a change that genuinely needs a restart. An operator relying on this comment as guidance might wait for an automatic event that will never happen.

However: this gap existed before the PR — the old behavior was to always restart, which was wrong (it killed in-flight work). The PR trades one wrong behavior (always restart, killing work) for a different incomplete behavior (never restart, no operator guidance). The trade is clearly correct for the acute incident, and the gap (no operator decision rule) is a follow-on concern, not a blocker.

**The gap does not warrant NEEDS_CHANGES** because: (a) the old behavior was actively harmful, (b) the system self-documents the restart mechanism for operators who need it (`systemctl restart lobster-claude`), and (c) the comment is honest about what it does do — skip the automatic restart — even if the "restarts itself" phrasing is imprecise.

### Patterns introduced?

This PR introduces the pattern of service exclusion with an explanatory comment at the exclusion site. This is correct form. If future services need to be excluded from `restart_services()`, the pattern is clear.

### What this makes easier/harder?

Easier: upgrades that run while the dispatcher is active will no longer kill in-flight work. Future operators can see exactly why `lobster-claude` is excluded and what the trade-off is.

Harder: operators who need a deliberate dispatcher restart after an upgrade must know to run `systemctl restart lobster-claude` manually. No upgrade.sh guidance exists for this case.

### Learnings / golden patterns?

No new pattern warrants addition to oracle files. This is a narrow, correct incident fix that cleanly applies vision.yaml principle-1 at the infrastructure level. The operator-guidance gap is a follow-on issue, not a pattern to name.

The fix is correct, minimal, well-commented, and aligned with the vision. The one documentation inaccuracy ("restarts itself") does not affect the code's correctness. The operator-guidance gap is real but is a follow-on concern, not a blocker for this incident fix.
