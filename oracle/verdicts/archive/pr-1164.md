### Functional correctness

The hook's decision logic is correct:

- `contains_gh_pr_create` regex (`\bgh\s+pr\s+create\b`) uses a word boundary after "create" — correctly matches the canonical form without false positives.
- `extract_repo` and `extract_head_branch` regex patterns are sound for the command forms agents actually use. Branch names cannot contain spaces, so `\S+` is correct.
- `_detect_repo_from_remote` handles both HTTPS and SSH remote URL forms.
- `find_existing_pr` returns the first matching PR or None — correct, includes allow-on-error on all failure paths.
- `should_block` is a pure function that constructs a human-readable block message with branch, repo, PR number, and title.

### Allow-on-error policy

The hook never blocks when it cannot determine context. The allow-on-error path is correctly implemented: undetectable repo, undetectable branch, and `gh` subprocess failures all exit 0. This is the correct tradeoff — a false negative (allowing a duplicate) is preferable to a false positive (blocking a legitimate PR creation).

### CWD limitation in `_detect_repo_from_remote`

`_detect_repo_from_remote` runs `git remote get-url origin` with `cwd=None`, meaning the hook's own process working directory, not the directory of the intercepted command. If an agent runs `cd ~/lobster-workspace/projects/other-repo && gh pr create` without `--repo`, the hook would check for duplicates in the lobster repo, not other-repo.

This is a real edge case. Mitigations: (a) allow-on-error means the result is a permissive pass-through in ambiguous cases, not a blocking false positive; (b) WOS executor commands include explicit `--repo` flags; (c) the fallback fires only when `--repo` is absent from the command string. Not blocking.

### Migration 112

The condition `[ "${has_pr_idempotency_hook:-0}" = "0" ] || [ "${has_pr_idempotency_hook:-0}" = "" ]` handles both jq-failure (echo "0") and jq-empty-string edge cases.

The function is dead code that would also be broken if called. Not blocking — it cannot introduce incorrect behavior — but it is confusing dead code.

**`_load_module` helper:** The `try/except SystemExit` wrapping `pytest.raises(SystemExit)` has an unreachable except clause (pytest.raises already catches the exit), but this is harmless. The module loading pattern correctly returns `mod` with all functions accessible after the SystemExit fires at the non-Bash guard.

**Coverage of allow-on-error paths:** `TestExitCodeContract.test_gh_subprocess_failure_allows_create` correctly exercises the `find_existing_pr` failure path. There is no end-to-end test for the "no --repo in command AND git remote detection fails" → exit 0 path, but this is not blocking given the allow-on-error default.

**Tests exercise actual before-state:** `subprocess.run` is mocked to return empty list (no PR) or list with one PR — both cases produce verified pre-states, not just asserted in comments.
