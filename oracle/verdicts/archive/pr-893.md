## Stage 2: Quality Review

### Does it do what it claims?

Yes. The filter gate in `handle_memory_store` correctly intercepts before any embedding computation or DB insertion. `_should_skip_memory_write` is extracted as a named function, making it testable in isolation without importing the full MCP server (the PR description confirms inbox_server.py cannot be imported directly in tests due to namespace conflicts).

### Encoded Orientation check (constraint-3)

This PR does NOT constitute an Encoded Orientation decision. It changes how data is ingested into memory — not behavioral defaults, system constraints, agent identity, or decision-making rules. No prior logged decision is required. Constraint-3 does not fire.

### Key findings

- **Filter is correctly positioned.** The gate fires before `from memory.provider import VALENCE_VALUES` — before any embedding, DB write, or provider interaction. Zero resource cost for excluded events.

- **Test gap: `_should_skip_memory_write` has no direct test.** The 48 passing tests in `test_memory.py` exercise the memory provider layer, not the filter gate. A grep for `health_check`, `health_probe`, `cron_heartbeat`, `_should_skip`, and `EXCLUDED` in `test_memory.py` returns no matches. The PR's test plan is in manual verification steps ("confirm Skipped: message is returned"), not in the CI test suite. This is a coverage gap: the function is simple (frozenset membership check) but untested. The `Skipped:` return path is not exercised by any automated test. Per learnings.md PR #839 (assertion softening) and the general test design patterns, a filter gate with this operational consequence (380 events/day) should have at least one unit test asserting the excluded types are blocked and a non-excluded type is not.

- **`health_probe` as forward-compat alias is speculative.** The comment says "forward-compat alias in case producers adopt this name." The PR verification confirms `health-check-v3.sh` writes `type="health_check"` with `metadata={"tags": ["health_probe"]}` — the metadata tag is not checked by this filter. `health_probe` as a type is an anticipated future name with no current producer. This is low-risk but documents a divergence: the comment implies health_probe is a variant in use; it is not.

- **`cron_heartbeat` scope is potentially too narrow.** The PR description says executor/steward heartbeat noise is excluded via `cron_heartbeat`. It is worth verifying whether heartbeat scripts actually write with `type="cron_heartbeat"` or a different type. If the actual type differs, the exclusion is a no-op. The PR does not show verification of this claim equivalent to the health-check verification ("Confirmed actual event schema").

- **`Skipped:` response is caller-visible, not silent.** This is intentional and correct — callers can detect the gate fired. The `log.debug` line provides observability at the server level. No operational concern here.

- **Single changed file, no bundling.** The PR is clean: 36 additions, 0 deletions, one file. The feature-bundling check (learnings.md recurring pattern) passes cleanly.

### What would break without the key decisions made here?

Without the `frozenset` at module scope (if the sets were inline `in {}` checks per call), the sets would be reconstructed on every invocation at 380+/day. Using module-level `frozenset` constants is the correct performance choice at the filter frequency this gate operates at.

### Failure modes

1. A new noise producer writes with a type not in `MEMORY_EXCLUDED_TYPES` — it passes through unfiltered. The filter is a closed allowlist by inversion; new noise types require explicit additions. This is correct behavior — it should be explicit, not implicit.

2. A legitimate use of `type="health_check"` (e.g., a health system self-check that should be remembered for trend analysis) is silently dropped. The `Skipped:` response makes this detectable by the caller, but the decision to exclude the type is binary and unrecoverable at the call site.

### What this makes easier

Future noise type additions follow a clear pattern: add a string to `MEMORY_EXCLUDED_TYPES` or `MEMORY_EXCLUDED_SUBTYPES`. The module-level constants are the single point of truth.

### What this makes harder

Nothing is foreclosed by this change. The filter can be removed or made conditional without architectural rework.

## Verdict

VERDICT: APPROVED with advisory note.

The filter gate is correctly positioned, correctly scoped to the known noise types, and does not constitute an Encoded Orientation decision. The single-file change is clean.

**Advisory (not blocking):** `_should_skip_memory_write` should gain a unit test in a follow-on PR. The 48 passing tests do not cover the filter gate path. The gap is low-risk (the function is two frozenset membership checks) but non-zero: a future edit to the excluded types or the OR logic has no automated regression guard.

**Advisory (not blocking):** The `cron_heartbeat` exclusion should be verified against the actual event schema written by heartbeat scripts, in the same manner the health_check type was verified. If heartbeat scripts do not write `type="cron_heartbeat"`, the exclusion entry is dead code.
