## Stage 1 — Vision Alignment

**Adversarial prior:** Is this solving the wrong problem?

The proximate fix is correct: `handle_wos_start` was reading only `execution_enabled` and returning early without checking whether all wos_core jobs were actually enabled. The structural bug is that `execution_enabled=True` was treated as a sufficient proxy for "the pipeline is running" when it is actually only a necessary condition. The partial-enable state (executor-heartbeat enabled, steward-heartbeat disabled) was invisible to the idempotency check.

The more upstream question: should the system have prevented the partial state from being written in the first place? A subagent manually set `executor-heartbeat.enabled=true` and `execution_enabled=true` without going through `toggle_wos_core_jobs`. The PR does not address this root cause — it repairs the detection path, not the write path. However, the cheaper and more conservative fix at this stage is detection-and-repair at the user-invoked `wos start` command, not a structural write gate on jobs.json. Adding a write gate would require changes to more call sites and carries more risk. The PR is correctly scoped.

**What this forecloses:** An alternative was not to fix `handle_wos_start` but to add a periodic health-check cron that auto-repairs partial state. That would be an Encoded Orientation decision under constraint-3 and would require a logged prior. The PR's approach (repair only when user explicitly invokes `wos start`) correctly operates at the Constrained register, not the Encoded Orientation register. The conservative path is the right choice given vision principle-1 (proactive resilience) and constraint-3 (Encoded Orientation gate).

**Encoded Orientation check (constraint-3):** The PR does NOT constitute an Encoded Orientation decision. It modifies behavior that fires only when a user explicitly invokes `wos start` — not a background automated action. No prior decision is required.

**Alignment verdict: Confirmed**

**Patterns cited (behavioral effect, not labels):**
- `principle-1` (proactive resilience over reactive recovery) confirms this approach: making `wos start` detect-and-repair is structural prevention of a class of silent stall. Without this pattern in context, I might have flagged the repair-on-demand approach as insufficient compared to a periodic health check. Because principle-1 weights structural prevention, this scoped repair at a user-invoked boundary is the correct form.
- `mirror-constant pattern` (learnings.md, multiple PRs): Caused me to check `get_disabled_wos_core_jobs()` for any duplicated constants. No mirror risk found — the function reads from `_read_jobs_json()` at call time, not from a module-scope constant.

## Stage 2 — Quality Review

### What the PR does

Adds `get_disabled_wos_core_jobs()` — a pure reader function over `jobs.json` — and modifies `handle_wos_start` to invoke it when `execution_enabled=True`. If the reader returns any disabled wos_core jobs, the function calls `toggle_wos_core_jobs(enabled=True)` and reports the recovered names rather than declaring "already running."

### Correctness check

**The recovery path logic is correct:**

1. `get_disabled_wos_core_jobs()` reads `_read_jobs_json()` and returns only jobs where `wos_core` is truthy AND `enabled` is falsy. The filter is correct.

2. `toggle_wos_core_jobs(enabled=True)` is called on the partial-recovery path. This function also writes `execution_enabled=True` to `wos-config.json` — a redundant write since that flag is already `True`. This is harmless.

3. `toggle_wos_core_jobs` reads `_read_jobs_json()` again internally (second read). This is consistent with the existing pattern and does not create a TOCTOU issue in practice (jobs.json changes between reads would require a concurrent writer, which does not exist in this system).

4. The `log_control_event` call carries `partial_recovery: True` in the payload — this is correct audit trail enrichment. The registry instantiation pattern (`registry if registry is not None else _Registry()`) matches the existing pattern in `handle_wos_stop`.

**No functional gaps found.**

### Test review

Three new tests added in `TestHandleWosStart`:

1. `test_idempotent_when_already_started_and_all_core_jobs_enabled` — replaces the previous `test_idempotent_when_already_started`. The replacement correctly adds `_read_jobs_json` patch (required because the production code now calls `get_disabled_wos_core_jobs()` in the idempotent path). The old test without this patch would have hit the real filesystem — the replacement is correct.

2. `test_partial_recovery_when_execution_enabled_but_core_jobs_disabled` — exercises the recovery path. Correctly patches `_read_jobs_json`, `_write_jobs_json`, and `_write_wos_config`. Asserts the correct job state was written.

   One test isolation gap: the test does NOT mock `_Registry` or `log_control_event`. The production code instantiates `_Registry()` (using the live DB or the `REGISTRY_DB_PATH` env var). Since `log_control_event` is documented as non-fatal (swallows all errors), this does not cause the test to fail — but the test is touching a real DB component without verification. The registry call's behavior is untested. Advisory, not blocking.

3. `test_partial_recovery_does_not_fire_when_all_core_jobs_enabled` — verifies `toggle_wos_core_jobs` is not called when no jobs are disabled. Clean mock pattern.

### What this makes easier / harder

Easier: any future partial-enable state caused by manual jobs.json edits or a subagent bypassing `toggle_wos_core_jobs` will self-heal on the next `wos start`. The `get_disabled_wos_core_jobs()` function is independently testable and re-usable for a future health-check cron if one is added.

Harder: nothing foreclosed. The partial-state write path (the root cause) is still unguarded. Future PRs should consider whether `toggle_wos_core_jobs` should be the only permitted writer to `enabled` fields for wos_core jobs.

### Advisory note (not blocking)

The `test_partial_recovery_when_execution_enabled_but_core_jobs_disabled` test does not mock the registry's `log_control_event` call. The test passes because `log_control_event` is non-fatal, but the control event's audit trail is not verified. A complete test would pass a mock Registry to `handle_wos_start(registry=mock_registry)` and assert that `log_control_event` was called with `partial_recovery=True`. Not a blocker for this PR given the scope of the fix, but worth noting for future test hygiene.
