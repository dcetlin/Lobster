## Stage 1: Vision Alignment

**Prior entering this review:** This implementation is solving the wrong problem, or solving the right problem in a direction that forecloses better paths.

**Adversarial prior check (learnings.md):** The instruction-accumulation pattern (2026-04-22) does not apply — this PR adds instructions to a prompt block, not a behavioral rule document. The precondition-check-that-fails-in-the-opposite-direction pattern (2026-05-18) is relevant: the token tracking instructions are optional (pass None if counts unavailable) so they cannot fail by demanding what Claude Code cannot provide. No matching failure pattern constrains the Stage 1 analysis.

**Theory of change:** Token usage tracking at the `write_result()` Python helper layer and in the `_build_wos_context_block()` prompt instructions closes a gap in the WOS execution pipeline telemetry chain. The registry already has a `token_usage` column (confirmed present). The MCP `write_result` handler already passes through token_usage to `complete_uow()` (confirmed in inbox_server.py line 7951). The `write_wos_heartbeat` MCP handler already accepts token_usage per-heartbeat (confirmed line 7997). What was missing: (1) the Python-layer `write_result()` helper did not accept or write `token_usage`, and (2) dispatched subagents received no instruction to track and report token counts. This PR closes both gaps.

**What would have to be true for this to be the wrong path:**
- Claude Code sessions would need to reliably not expose token counts, making the instruction noise with no signal. The PR explicitly handles this: "If your Claude Code session does not expose token counts directly, omit token_usage" — the instruction degrades gracefully.
- The registry column would need to be absent. Confirmed present.
- The MCP handler would need to not pass through token_usage. Confirmed it does.
- The result.json format on disk would need to be the wrong layer to add this field. PR #1035 already approved adding token_usage to result.json at the `wos_completion.py` enrichment path. This PR adds it at the upstream source (the Python helper that subagents call directly), which is the more direct path.

**Assessment:** This is not the wrong problem. It is the final wiring step for a telemetry chain that was partially built in PR #1035. That prior verdict (APPROVED) established that token_usage at the WOS layer serves vision constraint-4 (minimize metabolic cost) by making cost visible per UoW. This PR extends that work to the client-library layer and the instruction layer.

**What does this work foreclose:** Nothing structural. The `token_usage` field name is already in use in the registry and in result.json from PR #1035. Adds no new field names. Does not constrain future token accounting approaches.

**Opportunity cost:** Minimal. Two small changes (one parameter addition, one prompt block extension) plus 3 tests. Consistent with vision principle-4 (wire what exists).

**Golden patterns:** Conditional-parameter extension pattern (`param: int | None = None`) from PR #1035 — applied correctly here. No-overwrite and optional-field omission patterns — applied correctly.

**Alignment verdict: Confirmed.** Directly serves `active_project.phase_intent` (WOS observability) and vision constraint-4 (metabolic cost minimization).

## Stage 2: Quality Review

### Does it do what it claims to do?

Yes. Three changes:
1. `_build_wos_context_block()` in `executor.py` — appends token tracking instructions to the WOS context block injected into every subagent prompt at dispatch time.
2. `write_result()` in `result_writer.py` — adds `token_usage: int | None = None` parameter; conditionally includes it in the result.json payload.
3. Tests — 1 test for context block inclusion, 2 tests for result_writer token_usage present/absent behavior.

### Implementation review: executor.py `_build_wos_context_block()`

The token tracking block appended to the prompt is correctly structured:
- Instructs subagents to pass `token_usage=<N>` to `mcp__lobster-inbox__write_result` — matches the actual MCP tool parameter name.
- Instructs subagents to pass `token_usage=<N>` to `mcp__lobster-inbox__write_wos_heartbeat` — matches the actual MCP tool parameter name (confirmed line 7981).
- Explicitly handles the unavailable-token-count case: "omit token_usage (pass None / leave it out)" — correct graceful degradation.

No issue with the instruction content. It is accurate, matches the actual MCP schema, and does not create a precondition that fails in the opposite direction.

### Implementation review: result_writer.py `write_result()`

The diff adds:
```python
token_usage: int | None = None,
```
to the signature and:
```python
if token_usage is not None:
    payload["token_usage"] = token_usage
```
to the payload construction.

This is the minimal correct implementation. The pattern matches the existing `artifacts` handling. The docstring is updated with accurate description. The schema comment is updated. No issues.

The `_build_wos_context_block()` change adds instructions to a prompt block, not to a behavioral default, system constraint, agent identity, or decision-making rule. ### Failure modes

1. **Claude Code session does not expose token counts.** Handled: instruction says to omit. The MCP handler and registry both handle `None` gracefully.
2. **Subagent passes token_usage to `write_result()` Python helper but not to MCP write_result.** These are two different call paths. WOS subagents that use the Python helper (`from orchestration.result_writer import write_result`) write to a result.json file on disk. WOS subagents that call the MCP tool directly bypass the Python helper. The PR correctly adds instructions to the prompt block (which applies to MCP-tool-calling subagents) AND adds the parameter to the Python helper (for direct-write subagents). Both paths are covered.
3. **Double-counting.** Not a concern — the `write_result` call is a single terminal call per UoW execution.

### What patterns does this introduce that will propagate?

The prompt instruction pattern for optional telemetry fields is clean. Future telemetry fields (e.g., wall_clock_seconds) can follow the same structure: one sentence per field in the prompt block, one conditional addition in the payload, two tests (present/absent).

### What does this make easier/harder?

**Easier:** Per-UoW cost attribution is now wired end-to-end. The WOS dashboard (`registry_cli.py` already handles `total_token_usage`), the steward log, and any post-hoc analysis can now surface actual measured costs, not just null values.

**Harder:** Nothing. Fully additive change.
