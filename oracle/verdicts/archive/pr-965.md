## Round 1 — 2026-04-26

### Stage 1: Vision Alignment

**Prior entering this review (held before reading the implementation):** This implementation is solving the wrong problem, or solving the right problem in a direction that forecloses better paths.

**The specific hypothesis I held:** The addition of `execution_attempts` could be a third partial counter layered on top of two already-partial counters (`retry_count` and `steward_cycles`). The learnings.md entry from the Sprint 3 audit names this exact failure mode: `steward_cycles` is per-attempt (resets on `decide_retry`), creating a pattern where cumulative failure history is masked. If `execution_attempts` also resets on `decide_retry`, it would reproduce the same limitation while adding the appearance of a fix.

**Pattern priors applied:**

- **learnings.md: "state field reset on manual retry masks cumulative failure history" (Sprint 3 audit)** — This pattern caused me to specifically look for whether `execution_attempts` resets on `decide_retry`. The diff does NOT include any changes to `decide_retry` handling, meaning `execution_attempts` persists across retries. This is the correct design and disconfirms the per-attempt-counter hypothesis. The pattern constrained what I looked for in Stage 2: I went directly to the `decide_retry` path rather than reading the diff linearly.

- **learnings.md: `>= on cumulative counter fires notification on every post-threshold cycle` (2026-04-08, PR #704)** — The gate change from `retry_count >= MAX_RETRIES` to `new_execution_attempts > MAX_RETRIES` directly activates this pattern as a prior concern. The author flagged this in Areas to Review (item 1). This caused me to trace the semantics carefully: with `MAX_RETRIES = 3` and `> MAX_RETRIES` (not `>=`), escalation fires when `new_execution_attempts = 4`, meaning the UoW gets exactly 3 confirmed attempts before escalating. This is intentional, documented, and tested — the multi-fire concern does not apply here because `execution_attempts` monotonically increases and the gate is `>`, not `>=`, preventing multi-fire at the boundary.

**Vision alignment assessment:** The work directly serves `active_project.current_phase` (WOS Phase 1 reliability). The 2026-04-26 failure cohort (19 UoWs with `lifetime_cycles=0` escalated to needs-human-review after zero actual execution attempts) is an acute reliability failure. Fixing retry accounting is WOS substrate maintenance — it serves vision `principle-1` (proactive resilience over reactive recovery) and keeps the execution pipeline functioning correctly. The implicit theory of change is sound: the WOS substrate cannot serve as a reliable semantic routing layer if its retry protection mechanism misidentifies infrastructure failures as execution failures.

What this PR does NOT foreclose: the `decide_retry` path, the hard cap, the needs-human-review escalation path. The change is additive (new column, new pure function, updated gate predicate) and does not narrow any existing capability.

**Alignment verdict: Confirmed**

### Stage 2: Quality Review

**Does it do what it claims to do?**

Yes. The implementation correctly achieves its stated goal: three orphan kill events no longer exhaust the retry budget. The mechanism is sound:

1. `ORPHAN_REASONS` frozenset enumerates the three infrastructure-event return reasons.
2. `_is_infrastructure_event()` is a pure function, trivially testable, with no side effects.
3. The retry-cap block computes `new_execution_attempts = uow.execution_attempts + (0 if is_infra_event else 1)` and gates escalation on `new_execution_attempts > MAX_RETRIES`.
4. `retry_count` continues to increment on all re-entries for diagnostic visibility.

**Gate condition semantics (author's area 1):**

`new_execution_attempts > MAX_RETRIES` with `MAX_RETRIES = 3` means escalation fires when `new_execution_attempts = 4`. The UoW receives exactly MAX_RETRIES (3) confirmed execution attempts before the cap fires. This is correct and intentional. The `> MAX_RETRIES` form (not `>=`) means the cap fires *after* the third attempt, not *at* the third attempt — the UoW gets the full budget. The test `test_three_genuine_failures_exhaust_retry_cap` sets `execution_attempts=MAX_RETRIES` (3) and validates escalation on the next re-entry (producing `new_execution_attempts=4 > 3`), which correctly matches the semantics.

The multi-fire concern from learnings.md PR #704 does not apply: `execution_attempts` is monotonically increasing and writing `execution_attempts=uow.execution_attempts` (not `new_execution_attempts`) at the cap path means the field does not increment at cap, consistent with the comment. The escalation fires exactly once at the crossing event.

**`executing_orphan` short-circuit (author's area 2):**

The diff shows the `4b-orphan` section transitions `executing_orphan` directly to `failed` before reaching `4c-retry-cap`. The `execution_attempts` counter is never touched in this path. The test `test_executing_orphan_does_not_increment_execution_attempts` verifies this by asserting `execution_attempts == 0` after the call. The interaction is understood and tested.

**Migration 0014 DEFAULT 0 (author's area 3):**

The backward compatibility analysis is correct. Post-deploy, existing UoWs with `retry_count=3, execution_attempts=0` will not be immediately escalated — they will receive up to MAX_RETRIES genuine execution attempts. This is conservative, not aggressive: it effectively grants all existing UoWs a fresh execution budget. The tradeoff is that a UoW that had 3 genuine failures before the deploy will be allowed 3 more. This is acceptable given the alternative (immediate escalation of all existing UoWs with any retry history) would be worse.

**`execution_attempts` counter lifecycle (Stage 1 hypothesis resolved):**

The diff contains no changes to `decide_retry`. `execution_attempts` is therefore a lifetime counter — it does not reset on manual retry. This is the correct design: the field tracks total confirmed executions across the UoW's entire lifetime, not per-attempt cycles. The Sprint 3 audit failure mode (per-attempt counter masking cumulative history) is not present here.

**`StrEnum` pattern check (golden-patterns.md 2026-04-24):**

`ORPHAN_REASONS` is defined as `frozenset[str]` rather than a StrEnum. The golden pattern's three rules target: (1) SQL enum-backed column filters, (2) new modules importing enum-backed status/type values, (3) functions returning one of N fixed string choices. None of these apply here: `ORPHAN_REASONS` is a membership set used for classification, not a return type or SQL filter. The frozenset is the correct data structure for O(1) membership testing and immutability.

If a new orphan classification is added in the future (e.g., a new recovery path introducing `dispatching_orphan`), it must be manually added to `ORPHAN_REASONS` or it will consume execution budget. This is a maintenance coupling, not a bug. The frozenset membership is explicit and the comment maps each entry to `_RETURN_REASON_CLASSIFICATIONS` — this coupling is documented.

The schema comment and UoW dataclass comment are both updated to document this change. No callers that gate on `retry_count` are present in the diff, and the field's new role is correctly documented.
