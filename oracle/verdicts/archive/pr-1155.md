## Stage 2: Quality Review

### What the PR does

**Bug 1 (self-trigger loop):** vault-processor.py now appends `LOBSTER_SYNC_SENTINEL = "[lobster-sync]"` to its commit subject lines. vault-watcher.py adds `_get_commit_message()` (reads subject via `git log -1 --format=%s <sha>`) and `_commit_is_lobster_sync()` (substring check: `LOBSTER_SYNC_SENTINEL in message`). In the main polling loop, when a new HEAD is detected, the sentinel check fires first. If the sentinel is present, `last_known_head`, `last_processed_head`, and `first_push_at_in_burst` are all updated and the function returns without firing the processor.

**Bug 2 (ACTIVE TODOS.md oscillation):** Pass 1 of `apply_status_delta()` now tracks `_in_additions` state. Lines between `<!-- lobster-additions -->` and `<!-- /lobster-additions -->` are excluded from `file_dedup_keys`. Pass 3 queries the DB for items whose dedup_key is NOT in `file_dedup_keys` — which now means "not in the main file body." Additions-block items pass through this filter and are included in the rebuilt block, eliminating the oscillation.

### Specific review areas (per engineer's brief)

**1. `_commit_is_lobster_sync()` and `--format=%s`:** `%s` is git's subject-line format (not body). The sentinel is appended to the subject line in vault-processor.py. The check is `LOBSTER_SYNC_SENTINEL in message` — substring match, not exact. On git failure (returncode != 0), returns `False` — the safe direction: fires processor unnecessarily rather than silently skipping a real user commit. All three requirements stated in the brief are met.

**2. Pass 1 `_in_additions` tracking:** The flag transitions correctly. `_LOBSTER_ADDITIONS_MARKER` sets `_in_additions = True` then `continue`s (skips the marker line itself from dedup). `_LOBSTER_ADDITIONS_END` sets `_in_additions = False` then `continue`s. All intermediate lines `continue` when `_in_additions` is True. The double `.strip()` call (`stripped_line.rstrip("\n")` then `.strip()` for the comparison) means the marker is correctly matched even with leading whitespace. The logic is clean and handles multi-line additions blocks correctly.

**3. State update when sentinel detected:** All three fields updated: `last_known_head = remote_head`, `last_processed_head = remote_head`, `first_push_at_in_burst = None`. State written atomically via `_write_json_atomic` before `return`. Complete.

### Findings

**Mirror-constant (learnings.md PR #696/#800/#967/#970/#973/#974):** `LOBSTER_SYNC_SENTINEL = "[lobster-sync]"` is declared independently in both `vault-processor.py` and `vault-watcher.py`. These are peer scripts — neither imports the other in production (hyphen in filename; importlib would be required). Both scripts already import from `obsidian_sync_core.py`, which is the correct single-source home for this shared protocol constant. If the two constants diverge (e.g., a future change to the sentinel format is made in vault-processor.py but not vault-watcher.py), the sentinel detection silently breaks: watcher no longer recognizes processor commits, the loop resumes at ~90s, with no runtime error or CI failure to surface it.

The learnings.md pattern constrained my analysis here specifically: I checked whether a single-source alternative existed (obsidian_sync_core.py) before assessing severity, and found it does. This is a non-blocking finding because: (a) both constants currently carry the same value, (b) the test file correctly imports the constant from each production module rather than re-declaring it locally, meaning `test_processor_commit_message_includes_lobster_sync_sentinel` would catch divergence in the producer direction. The recommended follow-on: move `LOBSTER_SYNC_SENTINEL` to `obsidian_sync_core.py` and import it in both scripts. This is a one-PR fix.

**Webhook mode gap (acknowledged):** The sentinel guard is inside the `else` branch of `if config.get("webhook_mode", False)`. Webhook mode does not call `_get_remote_head()` or `_commit_is_lobster_sync()` — it uses `last_push_at` and `last_known_head != last_processed_head` from the state written by the webhook receiver. The self-trigger loop in webhook mode remains open. The engineer acknowledges this explicitly. For the current polling-mode deployment, the fix is complete.

**Encoded Orientation check (constraint-3):** The behavioral changes in this PR restore intended behavior of existing components — they are bug fixes, not new autonomous decision authorities. The sentinel skip condition and the deduplication scoping change are both corrections to existing logic that was producing incorrect behavior. No logged prior in oracle/verdicts/ is required; the vision anchor is principle-1 and the vault pipeline's role as infrastructure for cognitive extension. Constraint-3 is not triggered.

**Test structure:** The new tests are correctly structured. `test_watcher_fires_processor_for_user_commit` patches `_load_state` directly (not `STATE_PATH`) to work around the module-level constant binding issue — consistent with learnings.md PR #839. Test constants are imported from production modules (`_vault_processor.LOBSTER_SYNC_SENTINEL`, `_vault_watcher.LOBSTER_SYNC_SENTINEL`), not re-declared locally. The additions-block stability tests directly exercise the oscillation scenario. Three pre-existing test failures (2 in test_subtasks.py, 1 in test_vault_watcher.py) are unchanged by this PR and pre-confirmed on main.

### What this makes easier / harder

Easier: vault-pipeline churn stops; Dan can observe silence as a health signal. Additions-block items are stable across sync cycles. The `_commit_is_lobster_sync()` function is a pure, side-effect-free predicate — it can be tested independently and extended (e.g., for additional sentinel types).

Harder: the mirror-constant makes future sentinel format changes require a two-file edit with no tooling enforcement. Webhook mode still has the loop condition open — any future switch to webhook would need to revisit this.
