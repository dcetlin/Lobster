## Stage 2 — Quality review

**Is this solving the right problem?**

The Stage 1 concern (rationalization of a defect) is resolved by examining what the test now asserts. The new test requires not only that done items are absent from the open body, but also that the done item IS present in the trailing done section (`assert "Done task" in done_section`). This second assertion is the key: if `render_active_todos` were discarding done items silently (a defect), the second assertion would fail. The test can only pass if the implementation is actively placing the done item into a trailing section. This is behavioral verification, not a loosened assertion.

The revised test is therefore a regression guard against two failure modes simultaneously: (1) done items appearing in the open section (the original concern), and (2) done items being silently dropped entirely (a defect the original test could not catch). The PR strengthens the test coverage, not weakens it.

**Is it well made?**

- **Rename** (`test_render_excludes_done_items` → `test_render_excludes_done_items_from_open_sections`): accurate and specific. The original name implied full-output exclusion; the new name scopes correctly to the open sections.

- **Docstring**: explains the intentional behavior, names the critical invariant, and frames what the test verifies. No ambiguity about what "passes" means.

- **Partition logic** (`output.partition("\n---\n")`): technically sound. `str.partition` always returns a 3-tuple, making the destructuring assignment safe regardless of whether the separator is present. If the separator were absent, `done_section` would be an empty string and `"Done task" in done_section` would fail loudly — the correct failure mode.

- **Assertion pair**: "Done task" not in open_body AND "Done task" in done_section is a complementary pair. Both directions are checked. This is assertion strengthening, not softening (learnings.md PR #839 pattern checked: does not apply here).

**What patterns does this introduce?**

The complementary assertion pair (not-in-open-body + in-done-section) is a well-structured pattern for testing content that has designated sections. Worth propagating: when a function partitions output into named sections, test both that an item is absent from the wrong section and present in the right section.

**What does this make harder?**
