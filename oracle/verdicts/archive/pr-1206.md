### Stage 2: Quality Review

**Does it do what it claims to do?**

The PR claims three things:
1. Add `token_usage` param to `write_result()` in `result_writer.py` — yes, done correctly.
2. Add token tracking instructions to `_build_wos_context_block()` in `executor.py` — yes, done.
3. All tests pass — plausible; the test additions are correct.

**Implementation correctness:**

`result_writer.py` changes are minimal and correct. The `token_usage: int | None = None` parameter is optional, the docstring is clear, and the write-only-if-provided pattern (`if token_usage is not None: payload["token_usage"] = token_usage`) is consistent with how `artifacts` is handled in the same function. The JSON schema comment is updated. No existing callers are broken — the parameter is optional.

`executor.py` context block addition is correct. The instruction is advisory ("if your session does not expose token counts, omit it"), uses the correct MCP tool names (`mcp__lobster-inbox__write_result`, `mcp__lobster-inbox__write_wos_heartbeat`), and is consistent with the framing in `dispatcher_handlers.py`. It adds to the existing artifact-stamping block cleanly.

**What would break without these changes?**

Without the `result_writer.py` change: Python callers using `write_result()` directly (the executor at line 723 for the pre-dispatch failure path) could not pass `token_usage`. The MCP path through `inbox_server.py` was already independent and working. This closes a gap in the Python helper that is less-trafficked but real.

Without the `executor.py` context block change: Subagents dispatched via the executor would receive artifact-stamping instructions but no token tracking instructions. The `dispatcher_handlers.py` context block (used in a different dispatch path) already included them. This creates path asymmetry: subagents dispatched via `route_wos_message` in `dispatcher_handlers.py` would see token instructions; subagents dispatched via `executor.py`'s `_dispatch_via_claude_p` path would not. The PR closes this asymmetry.

**Pattern check — mirror-constant (learnings.md, six recurrences):** No new constants are introduced. No values are mirrored between modules. Not applicable.

**Pattern check — bare python3 (learnings.md PR #804, #821, #830):** No shell invocations in this PR. Not applicable.

**Pattern check — feature bundling (learnings.md, multiple PRs):** Two related changes (result_writer.py and executor.py context block) plus tests for each. Both changes serve the same WOS token tracking UoW (uow_20260504_089962). The PR description names both changes explicitly. No undescribed file changes. No bundling concern.

**Test adequacy:**

`test_result_writer.py` additions: two tests, `test_token_usage_included_when_provided` and `test_token_usage_omitted_when_none`. These directly cover the two behavioral states of the optional parameter. Correct and sufficient for the change made.

`test_executor.py` addition: `TestWOSContextBlock.test_wos_context_block_includes_token_tracking`. The test checks for presence of "token_usage", "write_wos_heartbeat", and "mcp__lobster-inbox__write_result" in the returned string. This is a presence check, not a structure check. It is weaker than verifying the full instruction text, but for a pure-function string builder this level of coverage is consistent with the existing test style and sufficient to detect regression. No concerns.

**Failure modes:**

1. The `_write_subagent_result` call at executor.py line 723 (pre-dispatch failure path) does not pass `token_usage` — but this is correct: if the executor fails before dispatch, there is no token usage to report. No gap.

2. Subagents that do not track token counts will pass `None` or omit the field. The `if token_usage is not None` guard handles this correctly throughout the chain.

3. The context block instruction asks subagents to self-report token counts. Claude Code sessions may not expose this directly to the agent. The instruction correctly handles this with the opt-out clause. This is a known limitation acknowledged in the implementation.

**What this makes easier:** Closing the token tracking loop end-to-end. When a WOS subagent dispatched via `executor.py` now reports `token_usage` in its `write_result()` call, the value flows through `inbox_server.py → wos_completion.maybe_complete_wos_uow → registry.complete_uow → uow_registry.token_usage column → wos-metabolic-digest.py → summary reports`. The chain was already there; the instruction to subagents was missing from the executor path.

**What this makes harder:** Nothing. The changes are purely additive and opt-in.

**No structural concerns found.** The PR is a clean, minimal gap-close in an approved infrastructure track.
