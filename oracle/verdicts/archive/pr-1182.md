## Stage 1: Vision Alignment (written before seeing implementation)

**Prior entering review:** This implementation is solving the wrong problem, or solving the right problem in a direction that forecloses better paths.

**The problem stated:** ~88% of `audit_log` volume consists of `dispatch_eligibility_skip` events — non-decisions written every steward cycle for every ineligible UoW. Real forensic events (state transitions, oracle decisions, escalations) are buried.

**What would have to be true for this to be wrong?**

Option A: `dispatch_eligibility_skip` events are forensically valuable and are being mislabeled as noise. Disconfirmed: these events record "UoW X was not dispatched this cycle because of eligibility condition Y" — a non-event from an audit trail perspective. The condition that caused skipping (throttle, pause, etc.) is already captured at the UoW level via `write_gate_fired()`. The skip record duplicates diagnostic signal that exists elsewhere.

Option B: A cheaper path exists — a query filter, a view, or log rotation — that doesn't require schema change. Considered: a `WHERE event != 'dispatch_eligibility_skip'` filter on audit queries would restore signal-to-noise for query readers, but it does not reduce write volume or storage growth. At 88% of write volume, the structural cost is real and ongoing. Routing to a separate table is the correct intervention.

**Vision alignment assessment:** This work is directly in scope with vision.yaml `current_focus.this_week.primary` ("WOS observability and reliability"). The separation restores the forensic signal-to-noise ratio that makes post-incident diagnosis reliable — which is a direct dependency of `active_project.phase_intent` ("Build the substrate that lets every agent make intent-anchored decisions"). Agents that cannot read a clean audit trail cannot diagnose UoW failures, which degrades the entire orchestration loop's reliability.

This applies vision.yaml principle-1 (proactive resilience over reactive recovery): fixing the write routing prevents the accumulation problem from growing, rather than adding per-query filters that would need to be maintained everywhere audit_log is consumed.

**Pattern applied from golden-patterns.md — seam-first abstraction (2026-03-30):** The PR places the routing boundary precisely at the seam where `dispatch_eligibility_skip` diverges from "audit-worthy events." The `write_dispatch_skip()` method is the named interface at that seam. I weighted this positively — the seam is identifiable from the design grain (two systems with different evidentiary value meeting at the write boundary), and the abstraction is minimal. This caused me to check whether the seam is placed correctly (at the write method level, not at a query filter level), and I confirmed the write-level placement is correct.

**Alignment verdict: Confirmed.** The work is solving the right problem via the right path. No cheaper test of the underlying assumption is available — the 88% figure is an observed fact about live write volume, not a hypothesis.

**Encoded Orientation check (constraint-3):** This PR does not change behavioral defaults, decision-making rules, or agent identity. It changes where a DB write goes. Not an Encoded Orientation decision.

**What this forecloses:** Queries against `audit_log` for skip events will return empty (for post-migration data). Any diagnostic query that was filtering `audit_log` for `dispatch_eligibility_skip` events needs to be redirected to `dispatch_skip_log`. The PR description explicitly notes this is out of scope for this PR and calls it correct: `audit_queries.py` was not changed because no existing query was filtering for skip events. This is verified in Stage 2.

## Stage 2: Quality Review

### Migration number (migration collision check — learnings.md 2026-04-27 PR #1000)

Migration 0024 is sequential: migrations 0001–0023 exist; 0024 is new and has no collision. No concurrent PRs with the same number are visible in the open PR list. The `IF NOT EXISTS` guard in the DDL makes the migration idempotent regardless.

The engineer applied the migration directly to the live `wos.db` before running `run_migrations()`. This is safe: the `IF NOT EXISTS` guard means the DDL is a no-op on re-execution. When `Registry()` is next instantiated (which happens on every steward cycle), `run_migrations()` will apply the migration file, the DDL will succeed silently, and version 24 will be recorded in `_migrations`. Confirmed in live DB inspection: `dispatch_skip_log` table is present; `_migrations` currently records versions 1–23; version 24 will be auto-recorded on next Registry init. This is a clean state.

No `upgrade.sh` migration is needed for this change: DB schema migrations in this codebase are intentionally self-applying via `Registry._run_migrations()` on instantiation, consistent with all prior migrations.

### Two call sites in steward.py

Both call sites are correctly updated:
- Line ~5411: throttle burst exceeded branch
- Line ~5432: non-dispatch non-throttle patterns branch

The `write_gate_fired()` calls immediately after each `write_dispatch_skip()` are left intact — correct, since gate firing is a distinct event class from the skip record.

The entry dicts passed to `write_dispatch_skip()` in the PR branch no longer contain an `"event"` key. They contain: `actor`, `uow_id`, `steward_cycles`, `eligibility`, `timestamp`. These are all serialized into the `note` column via `json.dumps(entry)` in `write_dispatch_skip()`. The `eligibility` key is also promoted to a first-class column — correct.

### write_dispatch_skip() implementation

The method follows the same `BEGIN IMMEDIATE` / `commit` / `rollback` / `conn.close()` pattern as `append_audit_log()`. The `note` column receives `json.dumps(entry)` — the full entry dict — which preserves all fields for future diagnostic queries, including any that `write_dispatch_skip` does not explicitly promote to a column.

One observation: the `entry.get("eligibility", "unknown")` default for the `eligibility` column means if a caller omits the `eligibility` key, the row is recorded as `"unknown"` rather than raising. Both current call sites supply `eligibility` explicitly, so this is not a production gap, but it is an implicit softening. Not blocking.

### audit_log consumer check

Full-repo search for `dispatch_eligibility_skip` in the PR branch returns zero results — confirmed. No consumer of `audit_log` was querying for skip events. The `audit_queries.py` module is not changed and does not need to be. This is correct: the PR's scope is precisely the write side; the read side has no existing consumers of skip events to update.

### Patterns introduced

The PR establishes `dispatch_skip_log` as the write destination for non-decision skip events. This creates a precedent: future high-volume non-decision events should route to their own tables rather than `audit_log`. This is a positive structural pattern — consistent with the `seam-first abstraction` principle.

The `write_dispatch_skip()` method signature (`uow_id: str, entry: dict[str, Any]`) mirrors `append_audit_log()`. This consistency is correct. If a future PR adds more high-volume event classes that need similar treatment, the pattern is clear.

### What this makes easier / harder

**Easier:** Post-incident audit queries are now unambiguous — `audit_log` contains only actions and transitions. A query for "what happened to uow_X" no longer needs to filter noise. Future diagnostic tools that read `audit_log` inherit the clean table without any per-query filtering.

**Harder:** A future diagnostic that needs to correlate skip events with audit events must join `dispatch_skip_log` and `audit_log`. No such join exists today; if one is needed, the `ts` and `uow_id` columns in both tables support it.

### Summary

This is a clean, focused, correctly scoped PR. The migration is safe (idempotent, sequential, self-applying). Both call sites are updated. No consumers of the old write path are broken. The test coverage is appropriate. The implementation follows established patterns. No learning-pattern failures are present.
