## Stage 2 — Quality Review

### Issues found

**1. Assertion softening on token_unknown tests (learnings.md PR #839 pattern)**

Two tests — `TestBuildShortForm.test_token_usage_none_shows_unknown` and `TestBuildRichForm.test_token_usage_none_shows_unknown` — assert:
```python
assert "unknown" in text.lower() or "?" in text
```
The production code unambiguously produces the string `"unknown"`. The `or "?" in text` alternative would pass for any text containing a question mark. Per learnings.md PR #839, assertion softening cannot distinguish correct from failed-with-fallback. The correct assertion is `assert "unknown" in text` without the disjunction. Advisory — not a blocking defect, production code is correct.

**2. wos_done in WOS_MESSAGE_TYPE_DISPATCH and fast-path — spawn-gate else branch structural fragility**

`wos_done` is in the dispatch table AND has a fast-path before the spawn-gate — consistent with `wos_escalate`, `wos_surface`, `wos_pr_sweep_result`. The structure is correct. Advisory: if the fast-path is removed in a future refactor without removing `wos_done` from the dispatch table, it would reach the spawn-gate's `else: raise ValueError` branch, producing a misleading error. Not a current defect.

**3. notify_uow_failed scoped to executing_orphan only — intentional per spec**

The stuck path returns `Surfaced` via `wos_surface`, not `wos_done`. The retry_cap path goes to `wos_escalate`. This is correct per spec intent. Documented for future maintainers.

**4. chat_id type inconsistency (str vs int) between fast-path handlers**

`handle_wos_pr_sweep_result` returns `chat_id: int`; `handle_wos_done` returns `chat_id: str`. Both are accepted by `send_reply`. Advisory inconsistency.

**5. Mirror-constant check: PASSES**

`SHORT_FORM_TRIGGER_OUTCOME`, `SHORT_FORM_MAX_EXECUTION_ATTEMPTS`, `WOS_DONE_MESSAGE_TYPE` are imported from the production module and locally aliased — not re-declared. Values flow from production module. No divergence risk.

**6. Absence of end-to-end test for notify_uow_done / notify_uow_failed public API**

Inner functions are tested; the outer orchestrators have no direct test. Per learnings.md PR #893 ("extracted for testability but no test added for the extracted function"). Advisory — not blocking because each inner function is independently tested.

**7. Encoded Orientation check (constraint-3)**

New inbox message type and dispatcher handler constitute an Encoded Orientation decision. Prior: issue #1092 (Dan-filed) + pre-approved spec. Per learnings.md PR #914, Dan-filed issue provides stronger substantive authorization. The missing artifact is a formal oracle/decisions.md entry. Given this is a one-way observability notification layer (no automated action), this does not block APPROVED but should be logged in oracle/decisions.md in a follow-on.

### Verdict rationale

The core implementation is clean: pure builders at the core, side effects isolated, non-fatal wrapping correctly layered at every level, fast-path dispatch pattern consistent with existing handlers. The two advisory items (assertion softening, missing orchestrator-level test) are improvements, not defects. The test suite correctly imports production constants rather than re-declaring them.
