### Encoded Orientation Decision Record — ADR-003

*This section is the canonical location for the Encoded Orientation prior required by constraint-3. The same content was written to the deprecated `oracle/decisions.md` by the fix agent; this verdict file is the correct canonical location per current convention (oracle/decisions.md is deprecated per PR #875).*

**Decision:** Prescription Throttle Gate — Encoded Orientation Behavioral Default
**Date:** 2026-04-24
**Status:** Accepted
**WOS Reference:** uow_20260421_f91285

**What changed:** `promote_to_wos()` in `cultivator.py` previously promoted all eligible GitHub issues on every sweep. After this PR, that behavior is suppressed — returning `([], 0)` — when ALL of the following are true:
- `consumption_rate < 0.6` (fewer than 60% of UoWs in the 7-day window are closed)
- `backlog_depth >= 5` (at least 5 open UoWs in the 7-day window)

This is a durable behavioral default change, not a one-time operational action.

**Why this is Encoded Orientation:** The system acts without Dan's real-time input when throttle conditions are met.

**Vision anchor (primary):** `core.operating_principles.principle-1` — "Proactive resilience over reactive recovery. Structural prevention is preferred over better correction mechanisms." Suppressing new prescriptions when the queue is demonstrably not draining is a structural prevention measure.

**Vision anchor (secondary):** `active_project.phase_intent` — Phase 1 is complete when the Registry is live and populated with UoWs carrying vision_ref fields. A registry with unbounded growth and a sub-threshold consumption rate defeats the intent of "live and populated in a useful sense."

**Threshold values as durable defaults:**
- `threshold = 0.6`: Suppression fires when fewer than 60% of UoWs created in the last 7 days are closed.
- `min_depth = 5`: Dual condition prevents false positives on low-volume periods.

Both parameters are configurable at instantiation time.

**Status breakdown at decision time (live data, 2026-04-24):**
Consumption rate at oracle review: 0.727 (above threshold). Open backlog: 246 UoWs, dominated by proposed (190) and ready-for-steward (52). needs-human-review: 1, blocked: 3. Executor-dysfunction hypothesis disconfirmed. Queue is producer-dominated.

**Interaction with od-3 override (ADR-002):** GardenCaretaker's promotion path via `requalify_proposed()` is not gated by this throttle. During sustained throttling, cultivator cannot promote new GitHub issues while caretaker can advance existing proposed UoWs. This is coherent: drain the existing backlog before adding new work.

### Stage 2: Quality Review

#### What it claims to do and whether it does it

The PR claims to suppress new UoW prescriptions when backlog is critical (depth >= 5 AND rate < 0.6). The implementation does this correctly. The fail-open design is correctly implemented: `_fetch_counts()` returns `{}` on any error, `get_rate()` returns 1.0 on total=0, so suppression never fires when the monitor cannot read state.

#### Rate formula correctness

`consumption_rate = closed / (closed + open)` — this is a fraction of total UoWs in the rolling window that have reached terminal status. At rate=0.43, 43% of UoWs created in the last 7 days are closed. This is a ratio over a cohort (7-day window), not a completion velocity. It is reasonable but it has a known distortion: if many old UoWs were created outside the 7-day window and remain open, the depth signal (`backlog_depth()` — also window-scoped) undercounts true open depth. The design doc acknowledges this explicitly. At observed depth=184 in a 7-day window, the total open backlog is certainly higher.

#### Double-read problem

`should_suppress_prescription()` calls `is_backlog_critical()` (which calls `_fetch_counts()`) and `backlog_depth()` (which calls `_fetch_counts()` again). That is two DB reads per gate check. Then if the caller logs via `gate_status()`, another two reads occur. The data can be stale between reads — not a correctness issue given the 7-day window, but the snapshot is not guaranteed to be internally consistent.

**The correct design:** compute the snapshot once (`_fetch_counts()`) and cache it for the lifetime of the check call, or structure `should_suppress_prescription()` to return the status dict directly so the caller does not need to call `gate_status()` separately.

This is a named defect: the PR #906 learnings entry ("dead constant" anti-pattern) applies in spirit — the double-read issue is the structural equivalent: the two calls appear independent but are measuring the same underlying state at two slightly different moments.

#### Encoded Orientation check (constraint-3)

**This is the primary NEEDS_CHANGES finding.**

This PR changes the behavioral default of `promote_to_wos()` in `cultivator.py`. Before this PR, every cultivator sweep promotes eligible GitHub issues. After this PR, that behavior is suppressed silently (with only a WARNING log, no user notification) when rate < 0.6 and depth >= 5. This is an Encoded Orientation decision: the system acts without Dan's explicit input in real time, based on a rule encoded into the codebase.

Constraint-3 requires: (a) a prior logged decision of the same class exists, and (b) the change is traceable to a vision.yaml anchor.

Checking (a): `oracle/decisions.md` has no entry for prescription throttling or backpressure. The authorizing artifact is `uow_20260421_f91285.md` (a WOS work order, not an oracle/vision decision record). A WOS UoW is a dispatch artifact, not a decision record — it records what was built, not the authorization for the behavioral change the thing being built introduces. The learnings.md PR #894 entry is directly applicable: "A GitHub issue describing a design decision is not equivalent to a logged oracle/vision decision." The same logic applies to a WOS UoW artifact.

Checking (b): No vision.yaml field references backpressure, throttling, or prescription suppression. The closest anchor is `principle-1` ("proactive resilience over reactive recovery") or `principle-4` ("wire what exists before building more"), but neither constitutes a traceable anchor for this specific behavioral change.

**Result:** Encoded Orientation decision lacks logged prior in oracle/decisions.md and lacks a traceable vision.yaml anchor. Constraint-3 is not satisfied.

**Resolution contract:** Add an ADR entry to `oracle/decisions.md` that: (a) identifies this as an Encoded Orientation decision in the prescription pipeline, (b) states the authorizing rationale (observed rate and depth triggering suppression is a proactive stability measure aligned with principle-1), (c) cites the specific vision.yaml anchor by field path (e.g., `principle-1`, or `active_project.phase_intent` if the framing is that queue stability is prerequisite to Phase 1 completion), and (d) states the threshold values as durable defaults rather than implementation quirks.

#### Reactivation gap

The design doc notes GardenCaretaker can reactivate archived UoWs to `proposed` without being gated. With od-3 override authorizing age-based promotion of 236 stuck UoWs (authorized same day), the effective behavior is: the throttle blocks cultivator from promoting new GitHub issues, while caretaker and the od-3 age-promotion path continue to advance UoWs through proposed. This is coherent — drain the stuck backlog, don't add more — but it is not stated in the design doc, and the interaction is not obvious from reading either PR in isolation.

#### Missing tests

The PR has no tests. The boundary note in the UoW explicitly states "do not create tests (out of scope for this UoW)." The oracle records this as a known gap: the `ConsumptionRateMonitor` and `PrescriptionThrottleGate` logic (the dual-condition gate, the fail-open path, the rolling window cutoff) has no CI coverage. This is consistent with the pattern that a prescription-monitor function extracted for testability has no test (learnings.md PR #893). Not a blocking gap for NEEDS_CHANGES on its own, but named explicitly.

#### What this makes harder

If the consumption rate remains below 0.6 because of executor-side issues (unhandled callbacks, stuck needs-human-review) rather than overproduction, the throttle will suppress cultivator indefinitely. There is no timeout, no decay function, no automatic re-enable. The only exit from sustained suppression is the rate organically improving above 0.6 — which requires closing UoWs, which requires the very executor-side infrastructure that has open gaps. The throttle can become a permanent steady-state with no visible signal to the user.

This is the Stage 1 concern confirmed by the implementation: the gate is correct for the overproduction case and incorrect (or at minimum, insufficient) for the executor-dysfunction case. The two cases are not distinguishable from the gate's perspective.

### Revision Contract (Round 1)

**Gap 1: Encoded Orientation without logged prior (constraint-3)**
Add an ADR entry to `oracle/decisions.md` for the prescription throttle behavioral change. The entry must: (a) cite a specific vision.yaml field path as the anchor (not just a paraphrase), (b) state the threshold values and their rationale as durable configuration defaults, and (c) explicitly acknowledge that this is an Encoded Orientation decision that suppresses system behavior without Dan's real-time input. A WOS UoW reference alone does not satisfy this requirement.

**Gap 2: Double DB read in gate check**
Refactor `should_suppress_prescription()` and `gate_status()` to share a single `_fetch_counts()` call per invocation. The simplest form: add a `_cached_counts` pattern (compute once on first call, store for the object lifetime or as a method-local variable), or restructure so `should_suppress_prescription()` returns the status dict directly. This is not a correctness defect at the current read frequency, but it is a structural inconsistency that can produce internally-inconsistent logs (rate and depth computed at two different moments).

**Gap 3: No user-visible signal when suppression is active**
The throttle logs a WARNING but does not notify the user. At sustained suppression (which is the current state: rate=0.43, will remain below 0.6 until executor gaps close), there is no visibility path for Dan. A one-time Telegram notification when suppression first activates, and when it deactivates, would make the throttle's state visible without constituting multi-fire. This is advisory rather than blocking — but given the Stage 1 concern (suppression may mask executor dysfunction), the absence of user-visible signal is a risk worth naming.

Gap 1 is blocking. Gaps 2 and 3 are advisory — Gap 2 should be fixed before merge; Gap 3 may be deferred with a stated reason.
