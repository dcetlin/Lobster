## Stage 1 — Vision Alignment

**Prior entering review:** This implementation is solving the wrong problem, or solving the right problem in a direction that forecloses better paths.

The PR addresses dispatcher context compaction by capping the active-sessions block prepended to every `wait_for_messages` response at 5 entries with an "...and N more" overflow line.

**Theory of change under examination:** Per-WFM context overhead from the active-sessions block is a material compaction driver. Capping display at 5 entries makes that cost O(1) with respect to agent count. At 96 WFM calls/hour, uncapped sessions at 5+ concurrent agents consume up to 29K tokens/hour from sessions prefix alone — stated in the PR, plausible given the math.

**What would have to be true for this to be the right path:** (a) active-sessions is genuinely a top-3 compaction cause (stated by PR author, not independently verified here — acceptable for this scope); (b) the cap does not make the dispatcher blind to agent fan-out in ways that produce routing errors; (c) the fix is not papering over a more structural accumulation problem that will simply manifest elsewhere.

**Key concern tested:** If the dispatcher needs to know that 12 agents are running to route safely, capping at 5 hides that. Examining this carefully: the header line preserves the true total count (e.g., `[7 agents running, 3 system]`). The dispatcher retains the structural fact without paying per-entry token cost. This is the design decision that resolves the concern. The implementation correctly keeps the header computation unaffected by the cap — `header` is built from `user_sessions` and `system_sessions` counts before any truncation. The dispatcher can read `12 agents running` in the header and act accordingly; the 7 hidden entries are detail, not structure.

**Forecloses nothing.** The cap is a named module-level constant (`_ACTIVE_SESSIONS_DISPLAY_CAP = 5`), trivially tunable. No direction becomes harder to pursue.

**Pattern citation — compression as architectural response to accumulation critique (golden-patterns.md, 2026-03-23):** This pattern states that when an oracle review identifies "adding text to address text-length problems" as a structural contradiction, the correct fix is to compress the encoding — not remove the feature. This PR applies that logic to a display layer: the solution is not to remove session context but to compress its encoding. The citation constrained my review: I was looking for whether the compression preserved behavioral specification (total count in header) or silently discarded it. The header preservation is the key behavioral criterion this pattern makes load-bearing.

**Alignment verdict: Confirmed.** Serves constraint-4 (minimize metabolic cost of cybernetic engagement) and principle-1 (proactive resilience over reactive recovery). Consistent with WOS substrate stability focus. Not in what-not-to-touch scope.

## Stage 2 — Quality Review

**Does it do what it claims?** Yes. The implementation correctly: (1) builds `all_sessions = user_sessions + system_sessions`, (2) slices to `_ACTIVE_SESSIONS_DISPLAY_CAP` entries, (3) appends `...and N more` only when `hidden > 0`, (4) leaves the header computation untouched so true total is preserved.

**What would break without the key decisions made here:**
- If `displayed = all_sessions[:cap]` had been applied before computing the header, the header would show a false lower total. The implementation correctly computes the header first.
- If user/system ordering were reversed, user-facing agent sessions would be the ones truncated first. The implementation correctly puts user sessions first — the dispatcher sees the most operationally relevant sessions within the cap.

**Failure modes:**
- None identified. `hidden = total - len(displayed)` is always non-negative because `len(displayed) <= total`. The `if hidden > 0` guard prevents an empty overflow line.

**Encoded Orientation check (constraint-3):** This PR changes a display formatting function. It does not change behavioral defaults, system constraints, agent identity, or decision-making rules in a durable way. The cap affects what the dispatcher sees in session context, but the header preserves the total count — the dispatcher's routing information is not degraded, only its per-session detail is bounded. Constraint-3 does not fire.

**Feature-bundling check (learnings.md, recurring pattern):** 2 files changed. PR description accounts for both: `session_store.py` formatting change and test additions. No undescribed bundling.

**Test quality:**
- `test_format_active_sessions_block_cap` imports `_ACTIVE_SESSIONS_DISPLAY_CAP` from the module under test — correct, follows the "import production constants from the module under test" pattern (learnings.md PR #800). No locally-declared mirror constant.
- The assertion `len(lines) == cap + 2` (header + cap entries + 1 overflow line) is precise, not softened.
- The boundary test `test_format_active_sessions_block_at_cap` correctly verifies the `hidden > 0` guard: no overflow line at exactly cap entries.
- No stale assertions. No bare python3 invocations. No assertion softening.

**Patterns introduced:** Module-level named cap constant as display throttle. This is a clean, low-footprint pattern applicable anywhere a list is prepended to a high-frequency context payload. The pattern: cap at a named constant, preserve the aggregate count in the header, append an overflow hint for visibility. Reusable for any list formatted into dispatcher context.

**What this makes easier:** Future tuning of the cap (one constant), future tests that need to verify cap behavior (constant is directly importable). What this makes harder: nothing.
