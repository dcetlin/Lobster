## Round 1 — 2026-05-09

### Stage 1: Vision Alignment

The adversarial prior entering this review: this implementation is solving the wrong problem, or solving the right problem in a direction that forecloses better paths.

The WOS pipeline was paused at `execution_enabled=false` due to CalledProcessError exit 1/143 failures that killed in-flight agents during daemon restart. The fix is to change how the router spawns subprocesses. For this to be the wrong problem, one of the following would have to be true: (a) the SIGTERM propagation is a symptom of a deeper architectural misalignment — that the daemon should not be restarting mid-flight agents at all; (b) the real fix is to make the daemon more resilient at the process-manager level (systemd restart policy, WantedBy ordering); or (c) the 163 orphaned UoWs are better addressed by a recovery path than by a prevention mechanism.

None of these holds. Option (a) would require a fundamentally different architecture where in-flight agents are tracked and drained before daemon restart — a much larger and higher-risk change. Option (b) is incomplete: systemd can be told to delay restart, but the underlying process-group coupling still exists and will fire whenever any SIGTERM reaches the daemon. Option (c) is the current state (TTL/orphan recovery exists) and it is demonstrably not sufficient — the pipeline was paused because the recovery loop was insufficient, not because recovery was absent.

The work is solving the correct problem. `start_new_session=True` is the standard POSIX mechanism for process-group isolation, and this is its canonical use case. The approach aligns with principle-1 (proactive resilience over reactive recovery) and principle-2 (ergonomics over shortcuts — the correction path remains open; no TTL recovery is burned on infrastructure kills). No directions are foreclosed.

**Alignment verdict: Confirmed**

### Stage 2: Quality Review

**Encoded Orientation check (constraint-3):** This PR changes the dispatch mechanism in the router — a behavioral change — but it does not change decision-making rules, agent identity, system constraints, or behavioral defaults at the orientation layer. The dispatch mechanism is an implementation detail, not an Encoded Orientation decision. Constraint-3 does not apply.

**Patterns checked from learnings.md:**

- Mirror-constant pattern (PRs #696, #800, #967, #970, #973, #974, #980): `_MAX_TURNS: int = 40` is declared in the new module. The same value appears in `executor.py` as `"--max-turns", "40"`. This is a mild mirror risk — if the max-turns value changes in executor.py, the router's `_MAX_TURNS` constant will silently diverge. The engineer flagged this implicitly by noting the synchronous path is retained in executor.py. Given `_MAX_TURNS` is a module-scope named constant (not an inline literal in an f-string) and the value is the same in both, this is lower severity than the learnings.md instances, but it is the same pattern class. Not a blocker but worth flagging.

- Dead code before exec_module injection: The PR correctly removes the pre-`exec_module` attribute injection block (`mod.route_wos_message = MagicMock()` etc.) from `_get_router_module()`. This was dead code — `exec_module` overwrites those attributes via the `from orchestration.dispatcher_handlers import ...` statement, which resolves through the mocked `sys.modules`. The PR's comment explains this correctly. Removal is correct.

- `_get_router_module()` in `test_uses_popen_not_run`: The first line calls `_get_router_module()` and assigns the result to `mod`, which is never used. The test then independently loads `real_mod` to access the real `_dispatch_via_popen`. The `mod = _get_router_module()` call is dead code in that test. This is harmless but cosmetically messy. Not a blocker.

**What the implementation actually does:**

`_dispatch_via_popen` is a straightforward Popen wrapper with correct flags. `start_new_session=True` creates a new Unix session (new process group leader), which is the correct POSIX mechanism — not merely a hint. `close_fds=True` (Python default, explicitly stated) prevents daemon file descriptors from leaking into the child. `stdin/stdout/stderr=DEVNULL` prevents the child from inheriting the daemon's stdio. The `--` end-of-options sentinel in the command list is correct defensive practice for prompts that might start with `-`.

**Engineer-flagged areas, evaluated:**

1. `start_new_session=True` vs `os.setpgrp`: `start_new_session=True` is correct and complete. It creates a new session (which implies a new process group). `os.setpgrp` would require a `preexec_fn` argument and changes only the process group, not the session. `start_new_session=True` is the idiomatic Python way and covers the failure mode fully.

2. `env` fallback to `None`: The `try/except Exception: env = None` pattern has a silent failure mode the engineer correctly identified. If `_build_claude_env()` raises (e.g., credentials.json absent or unreadable), the child inherits the daemon's environment. In a stripped systemd environment, `CLAUDE_CODE_OAUTH_TOKEN` may be absent, causing the child to fail at auth — but this would produce a child-process auth failure, not a daemon crash. Since the child is fire-and-forget, this failure mode is silent except in the agent's own logs. The TTL/orphan recovery catches it. Acceptable as-is given the alternative (raising and marking the UoW failed) is worse.

3. Fire-and-forget with no post-spawn failure detection: Correct characterization. The only detectable errors are at Popen creation time (FileNotFoundError, OSError) — these are caught and cause the UoW to move to failed/. Runtime failures of the child process are invisible to the daemon and must be caught by TTL recovery. This is explicitly documented and is the right trade-off.

4. `stdout/stderr=DEVNULL`: Early `claude -p` crashes are silent. Covered by TTL. Acceptable.

**Test structure:**

The 4 new `TestDispatchViaPopen` tests each load a fresh module copy via `spec_from_file_location` with unique spec names (`wos_execute_router_real`, `wos_execute_router_sigterm`, etc.) — this avoids `sys.modules` collisions. `patch("subprocess.Popen", mock_popen)` is entered before `exec_module`, which is the correct ordering: at call time, `subprocess.Popen` is looked up on the `subprocess` module object, which has been patched. Assertions on `mock_popen.called` outside the context manager are correct — the mock object retains call history after the context exits.

The `test_start_new_session_is_true` assertion uses `kwargs.get("start_new_session") is True` — this tests only the keyword argument form. If `start_new_session` were passed positionally (it cannot be — it's keyword-only in CPython's Popen), this would silently pass. Not a real risk given Popen's API, but the comment "can be passed as positional or keyword" is inaccurate — `start_new_session` is keyword-only.

The env fallback path (`_build_claude_env()` raises → `env=None`) is not tested by any of the 4 new tests. The steward mock in sys.modules returns a MagicMock attribute when `_build_claude_env` is accessed, so the happy path runs in all 4 tests. This is a minor coverage gap but not a blocker.

**Test count verification:**

`uv run pytest tests/test_wos_execute_router.py` — 30 passed. `uv run pytest tests/test_wos_execute_router.py tests/unit/test_orchestration/test_executor_heartbeat.py` — 52 passed. The engineer's claimed counts are accurate.

**What this makes easier / harder:**

Easier: future daemon restarts are safe; WOS pipeline can be re-enabled; the dispatch path is now explicitly fire-and-forget, making the contract clear. Harder: post-dispatch observability is now fully delegated to TTL recovery — any failure between spawn and write_result is invisible to the router. This was already partially true, but the fire-and-forget contract makes it complete.

**Verdict: APPROVED**

No blocking issues. Two minor items noted (not blockers): (1) `_MAX_TURNS = 40` is a mirror of the value in executor.py — same pattern class as learnings.md PR #980, lower severity because it is a named constant, not an inline literal; (2) `mod = _get_router_module()` in `test_uses_popen_not_run` is dead code. Neither warrants a NEEDS_CHANGES round.
