## Stage 1 — Vision alignment

**Prior entering review:** This implementation is solving the wrong problem, or solving the right problem in a direction that forecloses better paths.

**Theory of change:** The WOS drilldown page is a human observability tool that lets Dan inspect UoW state. The steward writes markdown to `success_criteria`, `summary`, and `close_reason` fields. Those fields were displayed raw — the user saw hyphens, asterisks, and `##` as plain text. The fix renders them server-side before embedding in the HTML page.

**What would have to be true for this to be the right path?** The steward writes structured markdown to these fields (it does — prescriptions contain markdown checklists and bold status markers). The drilldown page is used to inspect UoWs (it is — that is its sole purpose). The cheapest fix is server-side rendering at generation time (it is — client-side would require a CDN dependency or bundled JS).

**Cheaper test of assumption not run?** None applicable. The symptom is visible, the cause is direct (no markdown processor at interpolation point), and the fix is surgical.

**What does this foreclose?** Nothing material. The server-side approach is strictly better than client-side: no CDN, no runtime dependency, pure HTML delivered to browser. If a richer markdown dialect is needed later, swap the Python `markdown` library or its extensions — the seam (`_render_markdown`) is a named function with no API surface beyond `str → str`.

**Opportunity cost:** This is a targeted rendering fix on a single internal page. The vision constraint `principle-4` ("integration rate before new feature rate") applies: this is not a new feature — it makes existing data legible. Wiring markdown rendering is integration of an existing capability (markdown in DB fields) to an existing display surface (drilldown page). Alignment: Confirmed.

**Encoded Orientation check:** Not applicable. This PR makes no changes to agent definitions, bootup files, gate tables, vision.yaml, CLAUDE.md, or behavioral defaults. It is a pure rendering fix.

## Stage 2 — Quality review

**Does it do what it claims to do?** Yes. `_render_markdown` converts markdown to HTML via Python's `markdown` library with `fenced_code`, `tables`, and `nl2br` extensions. `generate_html` maps it over `summary`, `success_criteria`, and `close_reason`, adding `*_html` keys to the JSON payload. The JS template switches to read from these pre-rendered keys, with fallback to the raw field for `summary` (`D.summary_html || u.summary || '(no summary)'`).

**What would break without the key decisions made here?**
- Server-side rendering (vs. client-side): eliminates CDN dependency and rendering flicker. The correct call.
- Separate `*_html` payload keys (vs. mutating the original keys): preserves the raw field values for any future code that reads the payload structurally. The correct call.
- `_render_markdown("")` returns `""` which JS evaluates as falsy: prevents empty Success Criteria sections from appearing. Correct — matches the prior behavior check in the PR description.

**Failure modes:**
1. **Docstring claim mismatch:** The `_render_markdown` docstring says "Convert a markdown string to sanitized HTML." The function does not sanitize — it converts markdown to HTML with no bleach or similar sanitization step. This is not a functional defect: the data source is the steward (internal system agent), not end users. XSS is not a realistic attack surface here. However, the docstring is misleading for future maintainers who might rely on that claim before injecting content from a different source. Severity: advisory. The word "sanitized" should be changed to "HTML" or the docstring should clarify that no user-supplied content sanitization is applied.

2. **`nl2br` extension and existing content:** The `nl2br` extension converts single newlines to `<br>` tags. This is correct for content written as prose, but for checklist-style success criteria (`- item\n- item`), `nl2br` may produce redundant line breaks between list items depending on the markdown library's rendering order. In practice, Python's `markdown` library renders `ul/li` blocks as block elements with their own newlines, so `nl2br` firing between them is benign. But if the steward writes a checklist with blank lines between items (which triggers `<p>` wrapping), the rendering may differ from intent. This is a display-quality concern, not a correctness concern. Not a blocker.

3. **`success_criteria_html` empty check is JS-side only:** The JS template gates the Success Criteria section on `D.success_criteria_html` being truthy. The server always writes `success_criteria_html` to the payload (as `""`). This is correct — empty string is falsy in JS. Consistent with the test that verifies `payload["success_criteria_html"] == ""` for empty input.

**Patterns introduced:**
- `_render_markdown` as a named seam for all markdown-to-HTML conversion in this module. Future fields can be added to the payload mapping in `generate_html` without touching the conversion logic. Consistent with the seam-first golden pattern.
- Pre-rendered `*_html` keys alongside original keys in the JSON payload. This is a clean extension pattern — the original keys are preserved, the rendered versions are additive. No downstream reader that expects the original structure breaks.

**What this makes easier:** Adding more text fields to the drilldown page with markdown rendering requires one line in `generate_html` (add the mapping) and one line in the JS template (switch to the `*_html` key). The seam is correctly placed.

**What this makes harder:** Nothing material. The `markdown` library is a well-maintained PyPI package with no unusual coupling.

**Test quality:** 9 new tests across two classes. `TestRenderMarkdown` covers the pure function directly (empty, whitespace, plain text, bullets, bold, heading). `TestGenerateHtml` additions cover the end-to-end path through `generate_html` (success criteria list, bold close reason, empty success criteria payload field). The payload extraction test (`test_empty_success_criteria_produces_empty_html_field`) correctly uses regex to extract the embedded JSON and asserts `payload["success_criteria_html"] == ""` — exact equality, not substring. Test quality is good.

**Pre-existing mirror constants:** `SONNET_4_6_INPUT_PER_MTK`, `SONNET_4_6_OUTPUT_PER_MTK`, and `SONNET_4_6_CACHE_READ_PER_MTK` are re-declared in the test file at lines 34-36 rather than imported from production. This is a pre-existing issue (flagged in learnings.md 2026-05-10 PR #1116 entry). This PR neither introduces nor fixes it. No action required in this PR.

**Summary:** The implementation is correct, minimal, and well-tested. The one advisory finding (docstring says "sanitized" when no sanitization is performed) is non-material given the internal data source. No structural concerns, no Encoded Orientation issues, no mirror-constant extension, no feature bundling. APPROVED.
