## Stage 1: Vision Alignment

**Finding (written before reading implementation):**

The implicit theory of change in vision.yaml for infrastructure work like this is principle-1: "proactive resilience over reactive recovery." The WOS execution pipeline's startup sweep was killing executing UoWs based purely on elapsed TTL, without checking whether the agent was actively alive. This produces spurious kills of legitimate long-running agents — infrastructure churn, wasted quota, and potential duplicate-side-effect risk on re-dispatch.

The precondition question for Stage 1: is this solving the wrong problem, or foreclosing a better path?

Evidence against that prior: PR #989 added `write_wos_heartbeat` so agents can now emit liveness signals. Without a corresponding reader in the sweep, that capability is structurally incomplete — the heartbeat exists but nothing uses it to prevent spurious kills. PR #993 closes that loop. The theory of change is: agents emit heartbeats → sweep reads them → live agents are spared. This is not a workaround; it is the completion of a liveness protocol that PR #989 opened.

Cheaper test of the assumption: one could audit whether Population 4 was actually killing live agents before implementing the gate. But the gate is cheap (single timestamp comparison), the failure mode is well-understood, and the 120-second threshold is well-calibrated (2x the ~60s heartbeat interval). The investment is proportionate.

What this forecloses: nothing of value. The gate has a named threshold constant and a `heartbeat_skip_threshold_seconds=0` override that restores original behavior. The door stays open.

**Alignment verdict: Confirmed.** The work serves principle-1 directly, closes an open capability loop, and does not displace higher-priority WOS substrate work.

## Stage 2: Quality Review

### Implementer concerns — verified

**Concern 1: `heartbeat_at` binding and shadowing.**
In the post-PR code, `heartbeat_at` is bound once via `getattr(uow, "heartbeat_at", None)` in the new gate block, before the `_is_heartbeat_recent` call. The old binding at the same location has been removed. The same `heartbeat_at` variable flows into `_classify_executing_orphan_kill_type` when the kill does fire. No shadowing exists. Verified clean.

**Concern 2: PR #968 classification preserved.**
The diff shows the gate fires between the TTL check and the kill classification block. When `_is_heartbeat_recent` returns True, the loop `continue`s before `_classify_executing_orphan_kill_type` is called — correct, no classification needed for a spared agent. When it returns False (stale or NULL), the code falls through to classification exactly as before PR #992. The `kill_classification` field appears in the audit note unchanged. Verified clean via test `test_kill_classification_present_in_audit_note_for_stale_heartbeat`.

**Concern 3: `StartupSweepResult` with `slots=True, frozen=True` and new defaulted field.**
`skipped_heartbeat_alive: int = 0` is added as the last field. Python's dataclass rules require fields with defaults to follow fields without defaults — this ordering is correct. All existing callers access result fields by attribute name (`.active_swept`, `.executing_swept`, etc.), never by positional destructuring. Verified across: `steward-heartbeat.py`, `test_startup_sweep.py`, `test_wos_registry_bugs.py`, `test_wos_ttl_recovery.py`. No positional unpacking found anywhere.

**Concern 4: `test_backward_compat_no_dispatch_audit_defaults_to_kill_before_start` updated.**
The change from `heartbeat_at=_iso_ago(60)` to `heartbeat_at=_iso_ago(300)` is correct. At 60s, the new gate would spare the UoW (it's within the 120s threshold), which was not the test's intent. At 300s, the heartbeat is stale and the kill fires, exercising the PR #968 classification path as intended. The comment in the PR explains this accurately.

### Independent findings

**Schema and migration correctness in tests.**
`test_heartbeat_gate.py`'s `_SCHEMA` DDL does not include `heartbeat_at` in its `CREATE TABLE IF NOT EXISTS uow_registry`. This initially appears to be a defect — the `_make_executing_uow` INSERT explicitly lists `heartbeat_at`. However, the `registry` fixture calls `Registry(tmp_db)` which runs `_run_migrations()`. Migration 0009 (`ALTER TABLE uow_registry ADD COLUMN heartbeat_at TEXT`) adds the column before any INSERT runs. The pattern is: raw DDL creates the base table → Registry migration runner adds all derived columns → INSERT succeeds. Verified that `test_kill_classification.py` uses the same pattern and that 85 tests pass. Clean, though the `_SCHEMA` comment "# Schema — must include heartbeat_at column" is mildly misleading (the schema itself doesn't include it; the migrations do).

**Mirror-constant check (first-pass, per learnings.md 2026-04-26 Document Review entry).**
The new constant `HEARTBEAT_SKIP_THRESHOLD_SECONDS = 120` is defined only in `startup_sweep.py`. The test imports it from the production module via `_sweep_mod.HEARTBEAT_SKIP_THRESHOLD_SECONDS` — correct golden-pattern application. No re-declaration in test files. No mirror-constant risk.

**`_is_heartbeat_recent` design.**
Pure function, no side effects, no registry access. Safe defaults: None → False, unparseable → False, threshold=0 → always False. The boundary is exclusive (`age < threshold`), which is the correct conservative choice — a heartbeat at exactly the boundary is treated as stale. The docstring documents this explicitly. The `timezone.utc` fallback for naive datetimes is correct. Exception handling catches ValueError, TypeError, and AttributeError covering all realistic bad input forms.

**Feature bundling check.**
3 changed files: `startup_sweep.py`, `test_heartbeat_gate.py` (new), `test_kill_classification.py` (1-line fix). All 3 are named in the PR description. No undescribed file changes. Clean.

**Encoded Orientation check (OODA constraint-3).**
This PR makes the startup sweep MORE conservative — it prevents kills when an agent is verifiably alive. This is a correctness fix, not a new autonomous decision pattern. The sweep was already killing executing UoWs autonomously; the PR gates that kill behind a liveness check. The behavioral default that changes is: "kill any executing UoW over TTL" → "kill any executing UoW over TTL unless its heartbeat is recent." This is a reduction in autonomous action, not an expansion. Constraint-3 does not apply.

**`age_seconds` in scope at gate log message.**
`age_seconds` is computed at line ~597 (before the gate check), and the gate fires only when `age_seconds > executing_orphan_threshold_seconds` (line ~605). The log message that references `age_seconds` in the new gate block is always within the scope where `age_seconds` is defined and verified to exceed the threshold. No UnboundLocalError risk.

**`steward-heartbeat.py` caller not updated.**
The log message in `steward-heartbeat.py` at line ~692 does not include `skipped_heartbeat_alive` in its output. This is not a defect — the field is logged by `startup_sweep.py`'s own summary log when `total > 0`. The steward heartbeat's log is a high-level summary; the detail appears in the sweep's own log. The `skipped_heartbeat_alive` counter is correctly included in the `total` calculation and in the `StartupSweepResult` return value. No operational gap.

### What this makes easier / harder

**Easier:** Diagnosing spurious kill incidents — the `skipped_heartbeat_alive` counter in logs provides a direct signal of how often live agents were being spared. The gate parameter `heartbeat_skip_threshold_seconds` makes the threshold tunable without code changes. Disabling the gate (`=0`) is explicit and documented.

**Harder:** Nothing material. The gate adds one pure-function call per executing UoW per sweep cycle — negligible overhead.

### Patterns introduced

The `_is_heartbeat_recent(heartbeat_at, now, threshold)` pure predicate pattern continues the established codebase convention of extracting timing comparisons into named, independently-testable functions with conservative safe defaults. This is consistent with `_classify_executing_orphan_kill_type`, `_classify_active_uow`, and the epoch-normalization golden pattern. The pattern is correctly applied.
