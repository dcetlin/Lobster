## Stage 1 — Vision Alignment (formed before reading implementation)

**Task served:** Fix a `KeyError` crash on `msg["instructions"]` in `route_wos_message` that caused every `wos_execute` inbox message to fail, since inbox messages only carry `type/uow_id/chat_id`.

**Vision alignment:** The WOS execution pipeline is the explicit center of `current_focus.this_week.primary`. A crash that fails every `wos_execute` message is a complete pipeline blockage. The fix directly serves `principle-4` (wire what exists before building more) and the active phase's success criterion sc-1 (unblocked execution pipeline). Alignment is strong.

**Adversarial prior applied:** The theory under which this work is wrong: the real root cause is not a missing key access but a deeper architectural gap in how instructions travel from producer to executor — and a defensive `msg.get()` plus artifact loader masks the underlying design problem rather than closing it. Specifically, if instructions were always *supposed* to travel inside the inbox message, then fixing the consumer to load from a secondary artifact normalizes a broken hand-off rather than restoring the intended one.

**Resolution of the prior:** The PR description explicitly states: "wos_execute messages are written to the inbox with only type, uow_id, and chat_id. The instructions live in the WorkflowArtifact file on disk." This is the documented architectural intent — lean inbox messages, full instructions in the artifact file. The `msg["instructions"]` bare access was always wrong for production messages; it only worked in test/manual invocations where the inline field was populated by the test. The artifact loader does not mask an architectural gap; it implements the correct architecture. Stage 1 adversarial prior is disconfirmed.

**Stage 1 verdict:** Confirmed alignment. Proceeding to Stage 2.

**Encoded Orientation check (constraint-3):** `route_wos_message` is a routing function. This PR restores intended routing behavior (previously broken by a KeyError). It does not encode a new behavioral default, change agent identity, or alter decision-making rules. Constraint-3 does not fire.

## Stage 2 — Quality Review

### What the PR does

- Adds `_load_instructions_from_artifact(uow_id)` in `dispatcher_handlers.py` that loads the `WorkflowArtifact` from disk using `artifact_path(uow_id)` and returns `artifact["instructions"]`.
- Changes `msg["instructions"]` to `msg.get("instructions") or _load_instructions_from_artifact(uow_id)` in the `wos_execute` branch.
- Adds 3 unit tests: inline instructions path, artifact fallback path, missing-artifact spawn-gate path.

### Does it do what it claims?

Yes. The change is minimal and targeted. The `or` fallback correctly handles both the absent-key case (production messages) and the None-value case (hypothetical). An empty-string `instructions` value would also trigger the artifact load — the correct behavior, since an empty instructions string is functionally identical to a missing one.

### Failure modes examined

**Is `artifact["instructions"]` safe after `from_frontmatter`?** Yes. `WorkflowArtifact` is a `TypedDict` with `instructions: str` as a required field. `from_frontmatter` always assigns `instructions` in the returned dict (line 288: `instructions=instructions`). The key access `artifact["instructions"]` cannot raise `KeyError` given a successful `from_frontmatter` return.

**Is `from_frontmatter`'s failure mode visible?** Yes. `from_frontmatter` raises `ValueError` for missing opener, malformed JSON, missing `uow_id`, and missing `executor_type`. All `ValueError` exceptions propagate up through `_load_instructions_from_artifact` and are caught by the existing spawn-gate `except Exception` block, which returns `action="send_reply"` with the error text. The failure is surfaced to Dan rather than silently dropped.

**Is the missing-artifact path visible?** Yes. The helper explicitly checks `path.exists()` and raises a descriptive `ValueError` that includes the file path and `uow_id`. This is more informative than the raw `KeyError` it replaces.

**Pattern from learnings.md — "library with no wiring" (PR #714):** The helper is invoked at the correct call site (`route_wos_message`), not left as a dead module. Wiring is complete.

**Pattern from learnings.md — "consumer-vs-producer fix at wrong layer" (PR #891):** The pre-PR learnings entry distinguishes per-producer non-conformance (fix the producer) from universal infrastructure gap (fix the protocol layer). Here the issue is universal — every production `wos_execute` message lacks `instructions`. The fix is at the router layer, which is the correct site: it is the consumer of inbox messages and the producer of dispatch instructions for the executor. Fixing the router is the right layer; fixing message producers (the Executor's inbox write) to embed instructions would bloat every inbox message with the full instructions prose, which the lean-message architecture correctly avoids.

**Docstring accuracy issue (minor):** `_load_instructions_from_artifact` states "Called by route_wos_message when the wos_execute inbox message does not embed an 'instructions' field (test/manual invocations)." This is backwards — the artifact fallback is the normal *production* path; the inline `instructions` field is the test/manual path. The docstring misleadingly characterizes the production case as exceptional. This is a documentation inaccuracy that does not affect behavior.

**Test coverage:** Three tests are added:
1. `test_inline_instructions_used_when_present` — verifies the `msg.get("instructions")` short-circuit path. Correct.
2. `test_artifact_fallback_when_instructions_absent` — monkeypatches `wa.artifact_path` to return a temp file with valid artifact content. Verifies `action="spawn_subagent"` and `"Do the thing." in result["prompt"]`. Correct. The monkeypatch pattern correctly avoids the module-scope-constant anti-pattern (learnings.md PR #839).
3. `test_missing_artifact_returns_send_reply_with_alert` — verifies that a missing artifact produces `action="send_reply"` with `"spawn-gate alert"` and `"instructions"` in the text. The assertion `"spawn-gate alert" in result["text"]` matches the actual spawn-gate alert text format ("WOS spawn-gate alert: handler..."). Correct.

**Pre-existing gap noted (not introduced by this PR):** The spawn-gate `send_reply` returns do not include `chat_id`, which means the dispatcher cannot route the alert to the user without knowing the chat from context. This is a pre-existing gap in the spawn-gate implementation not in scope here.

**Feature bundling check:** 2 files changed. PR description accounts for both (`dispatcher_handlers.py` + test file). No undescribed scope expansion.

### Patterns introduced

The `_load_instructions_from_artifact` helper establishes the pattern: when a routing function needs a field that is not in the inbox message, it loads it from the artifact file rather than requiring callers to embed the full field. This is the correct direction — inbox messages stay lean, artifacts carry the content. The pattern is reusable for any future field that follows the same lean-message/artifact-content split.

### What this makes easier

Future routing logic that needs artifact content can use the same pattern without duplicating the file-existence check and error handling. The spawn-gate already handles `ValueError` from any source, so new helpers using `ValueError` for missing/malformed content integrate naturally.

### What this makes harder

Nothing significant. The docstring inaccuracy (characterizing the production path as exceptional) could mislead a future engineer who reads the docstring without the PR context, but this is advisory and does not affect the correctness or robustness of the implementation.
