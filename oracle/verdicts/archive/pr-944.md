## Stage 2: Quality Review

**Diff: 7 additions, 2 deletions. Single-file change to `src/orchestration/registry.py`.**

### Does it do what it claims to do?

Yes. The change replaces:
```sql
WHERE status IN ('active', 'executing', 'ready-for-executor')
```
with parameterized placeholders bound to `UoWStatus.ACTIVE.value`, `UoWStatus.EXECUTING.value`, `UoWStatus.READY_FOR_EXECUTOR.value`. The enum values are verified to match the strings they replace. The SQL semantics are unchanged; the binding mechanism is now type-anchored.

### Implementation correctness

The PR uses `.value` on each enum member. This is the technically correct form for SQLite parameterized queries, which bind Python string values — not StrEnum objects. `UoWStatus.ACTIVE.value` evaluates to `"active"` (a plain str), which is what sqlite3 expects in the parameter tuple. This is subtly different from using `UoWStatus.ACTIVE` directly, which would also work because StrEnum instances compare equal to their string values and sqlite3 would serialize them correctly — but `.value` is the more explicit and less surprising form. No issue.
