## Stage 1 — Vision Alignment (locked before seeing diff)

The vision's current_focus names "WOS execution health" as primary — resolving the 44% pipeline failure rate, draining blocked UoWs, restoring negentropic-sweep cron. This PR adds bidirectional GitHub issue tracking: stamping issues when UoWs execute, complete, or fail. The adversarial prior: does adding GitHub API calls to an already-failing pipeline worsen the failure rate? The timing is the risk, not the feature itself.

What would have to be true for this to be the right path: the stamp calls must be genuinely non-blocking, adding no failure surface to the execution hot path. The idempotency guard solves a real problem named in the PR (duplicate UoW creation when an issue already has `wos:executing`). Cross-system provenance is a genuine WOS substrate value (golden-patterns.md: bidirectional provenance design). The question is whether the execution pipeline is healthy enough to absorb new external dependencies at lifecycle transitions.

**Alignment verdict: Questioned** — the feature is correct for the vision's WOS substrate goals, but arrives against a stated "pipeline must flow" constraint. The non-blocking guarantee (confirmed in Stage 2) resolves the timing concern structurally.

## Stage 2 — Quality Review

**Non-blocking guarantee is structurally enforced, not advisory:**
All four public stamp functions wrap their entire body in `try/except Exception → log + return False`. The executor wiring (`_stamp_executing_if_github`, `_stamp_failed_if_github`) adds an additional outer try/except. No stamp failure can propagate to the UoW processing path. The guarantee survives importlib failure, gh CLI absence, and rate-limit errors. This resolves the Stage 1 timing concern: the 44% failure rate cannot be made worse by stamp failures.

**Idempotency guard return value is unused — documentation gap:**
`stamp_issue_executing` returns `False` when the issue already has `wos:executing`, with the module docstring stating "callers use this as a creation guard to skip duplicate UoW creation." No caller inspects the return value. `_stamp_executing_if_github` in executor.py calls and discards it. The idempotency signal exists structurally (GitHub label prevents re-stamping) but the documented creation-guard contract is not implemented at any call site. This is a documentation defect, not a runtime defect — the label itself provides the idempotency, independent of the return value.

**Test softening is appropriate in this case:**
`test_github_source_posts_comment` changed from `assert_called_once()` to `assert called at least once + content filter`. Learnings.md PR #839 names this as a risk ("assertion softening to 'not-legacy-value' cannot distinguish wired-correctly from silently-failed-with-fallback"). However, the exception applies: this feature correctly adds more gh calls, so `==1` was over-constraining. The content filter (checking for `"comment"` and `"42"` in at least one call) preserves behavioral coverage. The softening is the correct response to a feature that adds calls, not an anti-pattern.

**"heat" → close issue is a stated design decision, not an oversight:**
The PR description maps `pearl/heat` → `stamp_issue_complete` (closes issue). The engineer notes this question. "Heat" typically means partial completion with caveats. Closing a GitHub issue on "heat" may be premature if the issue needs re-triage. This is a semantic question, not a structural defect. The implementation matches the stated design; if the semantics are wrong, a follow-on PR can reroute `heat` to `stamp_issue_unverifiable`. Not blocking.

**Patterns introduced:**
Non-blocking GitHub state sync wired at pipeline lifecycle transitions. Consistent with the "fire-and-forget CLI wrapper" pattern (golden-patterns.md, PR #865) extended to lifecycle events. The lazy import pattern in wos_completion.py is a variant not previously named — acceptable but the inline try/except per call (as in executor.py) would be cleaner for test isolation.

**What this forecloses:**
Issues closed by this feature on `pearl` completion cannot be re-opened by the sweeper without a human or tooling intervention. If a UoW is classified `pearl` incorrectly (artifact trail present but not verified), the issue closes and may fall out of the sweeper's view. The `stamp_issue_unverifiable` path mitigates this for the `seed` case.

**Opportunity cost note:**
Work is additive and non-blocking. The execution health crisis (44% failure rate, binary path + timeout causes) remains the primary named constraint and is not addressed by this PR.
