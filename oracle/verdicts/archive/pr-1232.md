**The prior entering Stage 1:** This implementation is solving the wrong problem, or solving the right problem in a direction that forecloses better paths.

**What would have to be true for awaiting_owner to be the right path:**
1. Subagents will regularly encounter decisions requiring human judgment that they cannot resolve.
2. The current fail-and-re-prescribe loop is an unacceptable experience compared to a structured hold.
3. Dan's attention — explicitly named as the resource to be freed — is well-spent on in-band Telegram decisions about individual UoW execution paths.

**Where this may be solving the wrong problem:** The vision states Dan's attention should be "freed for things that require genuine judgment." The `awaiting_owner` pattern routes individual UoW execution decisions to Dan's primary communication thread. This is adding a category of judgment-required interruptions, not reducing them. If subagents frequently hit `owner_decision_required`, the flow produces a new class of in-flight interruptions arriving in Telegram. The existing fail-and-re-prescribe loop at least batches these into deliberate re-prescription acts. This pattern fragments judgment into mid-execution interrupts.

**The cheaper test not run:** The actual incidence of "genuine human judgment required" in current WOS execution has not been measured. If subagents hit this case rarely, the infrastructure serves edge cases that could be handled with better executor contracts. If they hit it frequently, that is a signal of under-specified contracts, not a lifecycle management problem.

**What this forecloses:** Once `awaiting-owner` exists as a lifecycle state, it becomes a natural escape hatch for subagents encountering ambiguity. Subagent authors will reach for `write_owner_decision_required()` because it is the cleaner path than failing. This creates a latent accumulation risk: `awaiting-owner` becomes a holding tank that grows silently. The path back from awaiting-owner is manual, there is no timeout, no re-evaluation trigger, and no aging mechanism.

**Opportunity cost:** The active constraint is #1118 (WOS bidirectional interface). `awaiting_owner` adds complexity to the steward/registry/dispatcher triangle while the bidirectional interface is the stated priority. However, the PR description implies this is directly motivated by Dan's stated request for this pattern.

**Alignment verdict (Stage 1): Questioned.** The work serves a real need (structured holds instead of silent failures), but the framing as a lifecycle state with a manual-only exit creates a holding tank that is difficult to reconcile with vision constraint-4 (minimize metabolic cost) and principle-1 (proactive resilience over reactive recovery). The Stage 1 concern is specifically the absence of any expiry or escalation mechanism — a UoW that enters `awaiting-owner` when Dan is unavailable, distracted, or when the Telegram message is buried will be silently stuck with no forcing function. This is the same structural tension named in vision risk-1: "Vision Object becomes another prose document agents read as context rather than query structurally." Here: `awaiting-owner` becomes another state agents enter rather than a structurally-enforced hold with an exit condition.

This Stage 1 finding does not change after seeing the implementation. The implementation confirms the structure is technically correct. It does not resolve the concern about the absence of an expiry mechanism.

#### Gap 1: No code path from `awaiting-owner` back to `ready-for-steward` (CRITICAL)

**Status: Open.** This is an original finding confirmed by independent review.

The bootup doc at `.claude/sys.dispatcher.bootup.md` states:

> Dan's reply in the primary thread provides the decision. To re-queue the UoW after receiving the decision, set it back to `ready-for-steward` with the decision as a note.

This implies the dispatcher will handle Dan's Telegram reply and re-queue the UoW. It does not. No such handler exists in this PR.

`registry.decide_retry()` only accepts `RETRYABLE_STATUSES = frozenset({"blocked", "ready-for-steward"})`. `awaiting-owner` is not in this set. When Dan sends a decision reply in Telegram, there is no code path that: (a) recognizes the reply as a UoW decision, (b) reads the `awaiting-owner` UoW from the registry, (c) records the decision text as a note, and (d) transitions the UoW to `ready-for-steward`. The UoW will be stuck in `awaiting-owner` indefinitely until manual intervention via `set_status_direct`.

The wos-registry-reference.md state diagram correctly shows the intended flow (`Dan re-queues → ready-for-steward with decision note`) but the code does not implement it.

**What must be addressed:** Either (a) add `awaiting-owner` to `RETRYABLE_STATUSES` so `decide_retry` can re-queue it with decision text as a note, or (b) implement a new `decide_owner` dispatcher handler that accepts `awaiting-owner`, records the decision text, and transitions to `ready-for-steward` — and document the command in the bootup doc. The notification text "reply with the decision and I'll update the UoW" implies (b) is the intended design.

**What is addressed vs not addressed:** The UoW correctly enters `awaiting-owner` and Dan is correctly notified. The return path does not exist in code.

#### Gap 2: `set_awaiting_owner` is not idempotent (REQUIRED)

**Status: Open.** This is an original finding confirmed by independent review.

`set_awaiting_owner` does not check whether the UoW is already in `awaiting-owner` status before writing. If called twice for the same UoW (e.g., due to a steward cycle retry if the heartbeat crashes between the registry transition and the `return Surfaced(...)` call), it will:
- Append a duplicate `steward_log` entry with the same `event: "awaiting_owner"`
- Write a confusing audit entry showing `from_status=awaiting-owner, to_status=awaiting-owner`
- Potentially write a second `wos_owner_required` inbox message if the steward re-processes the result file

The function reads `old_status = row["status"]` but never compares it to `UoWStatus.AWAITING_OWNER` before proceeding. Compare to how `decide_retry` guards against double-firing: it gates on `status IN ({placeholders})` so the UPDATE is a no-op if the status does not match.

**What must be addressed:** Add an idempotency guard — if `old_status == UoWStatus.AWAITING_OWNER`, return early without writing duplicate entries. The `_write_owner_required_message` call in `_process_uow` also fires unconditionally after the `try/except` block; if `set_awaiting_owner` raises and the fallback transitions to `blocked`, the inbox message still fires with text saying "awaiting your decision" while the UoW is actually in `blocked` status — a state/message inconsistency.
