## Stage 2: Quality Review

### What the PR does

Single file change to `scheduled-tasks/executor-heartbeat.py`:
1. Adds module-level constant `_REGISTRY_WARN_SENTINEL = "/tmp/lobster-wos-registry-warned"` with a clear docstring explaining the boot-session lifecycle.
2. Wraps the `log.warning(...)` call in `if not os.path.exists(_REGISTRY_WARN_SENTINEL):`.
3. Touches the sentinel after the warning fires, with an `OSError` guard.
4. Adds an `else` branch emitting `log.debug` when the sentinel exists.

The PR description also notes that `wos-registry.db` was removed directly as part of WOS UoW `uow_20260517_d77ab9` (Migration 86 applied manually — the DB had 0 UoWs). Migration 86 in `upgrade.sh` handles future installs where the file may still be present.

### Does it do what it claims?

Yes. The sentinel mechanism is correct. Each cron invocation creates a fresh Python process, so module-level boolean flags have zero cross-invocation effect. The `/tmp/` sentinel file is the correct mechanism for cross-process suppression within a boot session. `/tmp/` is cleared on reboot, so the warning resurfaces once per boot — sufficient signal without flood.

### Key design decisions

**`/tmp/` sentinel vs. module-level boolean:** PR #1152 (also open, same issue #1071) uses a module-level `_LEGACY_REGISTRY_WARNED: bool = False`. This is structurally wrong for cron-invoked scripts: each 3-minute invocation starts a new Python process, resetting all module-level state. PR #1189's `/tmp/` sentinel is the correct mechanism because it persists across process boundaries within a boot session.

**`OSError` guard on sentinel write:** If `/tmp/` is unwritable, the guard prevents the heartbeat from crashing. The comment documents the fallback: warning will re-fire next cycle. This is the correct non-fatal handling for a logging-infrastructure concern.

**0-UoW branch:** The existing `log.debug` for the 0-UoW case (already present pre-PR, introduced by PR #1157 content that was merged separately) correctly handles the common case where Migration 86 was not yet run but the DB is empty. The sentinel only activates on the non-zero UoW branch — where the file genuinely warrants attention.

### Failure modes

1. **`open()` resource handling:** `open(_REGISTRY_WARN_SENTINEL, "w").close()` works correctly. The more idiomatic `pathlib.Path(_REGISTRY_WARN_SENTINEL).touch()` would be cleaner, but this is not a defect — the file is opened, written (empty), and closed synchronously. Advisory only; not a blocker.

2. **Sentinel name collision:** `/tmp/lobster-wos-registry-warned` is a fixed path. On a multi-user system this could theoretically be set by another process. Lobster runs as a single-user daemon; this is not a real risk in practice.

### Migration 86 check

Migration 86 in `upgrade.sh` (lines 2943-2973) is correctly placed in sequence between Migration 85 and Migration 87. It handles three cases: empty DB (remove), unreadable DB (skip conservatively), non-empty DB (rename to `.obsolete`, inspect manually). No migration number collision was found in any open PR. The migration's idempotency is correct: absent file → skips cleanly.

### Concurrent PR analysis

PR #1152 (module-level boolean sentinel — wrong mechanism) and PR #1157 (0-UoW INFO→DEBUG demotion — already present pre-PR in the target branch) are both open against the same issue #1071. PR #1189 supersedes both: it uses the correct cross-process sentinel mechanism and the 0-UoW demotion is already present. The concurrent PRs are rendered obsolete but do not create a conflict defect in PR #1189.

### Encoded Orientation check

This PR does NOT constitute an Encoded Orientation decision. It modifies a scheduled task's logging verbosity without changing behavioral defaults, agent identity, decision-making rules, or system constraints in any durable way. The underlying signal (warn about legacy registry when non-empty UoWs are present) is preserved; only the repetition rate per boot session is controlled. Constraint-3 does not apply.

### Patterns

**Pattern introduced:** Sentinel-file-in-/tmp/ for cross-process advisory warning suppression within a boot session. Clean, reusable, documented at the constant declaration. This pattern should be applied to other cron scripts that have multi-fire advisory warnings where per-boot frequency is the correct signal rate.

**What this makes easier:** Extending this pattern to other scheduled tasks with repetitive advisory conditions. **What this forecloses:** Nothing.
