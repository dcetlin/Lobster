### Starvation suppression logic (the stated subject)

**Implementation correctness:** The condition is correct. `default=True` in `.get("execution_enabled", True)` means: if the config is unreadable, the expression evaluates to `not True AND ...` = `False`, so the alert fires normally. Fail-safe direction is correct.

**Lifecycle correctness:**
- `handle_wos_stop` calls `toggle_wos_core_jobs(enabled=False, pause_reason="user_command")` — writes the field. Correct.
- `handle_wos_start` calls `toggle_wos_core_jobs(enabled=True)` — pops `pause_reason`. Correct.
- `handle_wos_stop` idempotent path (already paused) returns early without calling `toggle_wos_core_jobs` — does not overwrite an existing non-user-command reason. Correct.

**Latent risk (not a blocker):** The string `"user_command"` appears as a raw literal in both `dispatcher_handlers.py` (write site) and `steward-heartbeat.py` (read site). Neither is a named constant. If someone changes one without updating the other, suppression silently stops working. The correct form is a named constant (e.g., `WOS_PAUSE_REASON_USER_COMMAND = "user_command"`) in `dispatcher_handlers.py` imported by the heartbeat. This is the raw-string-literal fragility pattern (StrEnum golden pattern, 2026-04-24) applied to a config field value rather than a DB column. Worth a follow-on issue.

### Gap 1: Stale test — `test_stop_calls_toggle_with_enabled_false` [BLOCKER]

`handle_wos_stop` now calls `toggle_wos_core_jobs(enabled=False, pause_reason="user_command")`.

The test at `tests/unit/test_orchestration/test_toggle_wos_core_jobs.py` line 352 asserts:
```python
mock_toggle.assert_called_once_with(enabled=False)
```

`assert_called_once_with` checks exact argument matching. The actual call includes `pause_reason="user_command"` which is absent from the expected args. This test **will fail**. Verified by running the mock pattern directly:

```
Expected: mock(enabled=False)
  Actual: mock(enabled=False, pause_reason='user_command')
```

This is the PR #827 stale-assertion pattern (learnings.md 2026-04-22): a PR changes a behavior and does not update the test asserting the old calling convention. The fix is to update the assertion to:
```python
mock_toggle.assert_called_once_with(enabled=False, pause_reason="user_command")
```
