### Stage 2: Quality review

**Encoded Orientation check (constraint-3):** This PR does not constitute an Encoded Orientation decision. It introduces no behavioral defaults, no agent identity changes, no decision-making rules, and no durable system constraints. It is a test fixture that prevents production path writes during pytest runs. Constraint-3 does not apply.

**Does it do what it claims to do?**

Yes. `isolate_lobster_inbox_dir` is session-scoped, autouse=True, creates the four standard subdirectories (`inbox/`, `processing/`, `processed/`, `failed/`), sets `LOBSTER_INBOX_DIR` to the temp path via `os.environ`, and restores the original value (or unsets it) on teardown.

The fixture correctly uses `tmp_path_factory.mktemp` instead of `tmp_path` because session-scoped fixtures cannot use `tmp_path` (which is function-scoped in pytest). This is the correct pytest API choice.

**Teardown logic:** The restore logic handles both the set and unset cases:
- If `LOBSTER_INBOX_DIR` was not set before the session: `os.environ.pop("LOBSTER_INBOX_DIR", None)` — correct; the `None` default prevents KeyError if something else already removed it.
- If `LOBSTER_INBOX_DIR` was set before the session: restores the original value — correct.

No teardown gap exists.

**Subdirectory creation:** The four subdirectories created (`inbox`, `processing`, `processed`, `failed`) are the standard subdirectory layout used by `_write_steward_trigger()` and by the dispatcher's inbox processing path. The set covers the actual production code's expected layout. This is sufficient for the stated purpose.

**Autouse scope interaction with existing fixtures:** The existing `isolate_inbox_server_paths` fixture is per-test (`autouse=True` with `tmp_path`). The new `isolate_lobster_inbox_dir` is session-scoped. These two operate at different layers and do not conflict:
- `isolate_inbox_server_paths` patches module-level globals in `src.mcp.inbox_server` via `patch.multiple` — it patches the Python object graph.
- `isolate_lobster_inbox_dir` sets the OS environment variable — it patches the environment layer that `_write_steward_trigger()` reads.

The PR description correctly states this is the complementary mechanism for env-var-at-call-time code. The distinction is accurate; the two fixtures together provide full coverage for both isolation vectors.

**Scope choice:** Session-scoped is a defensible choice for this fixture. The learnings.md PR #839 entry notes the relevant failure mode for this class of fixture, and the positive detection (PR #805) confirms env-var-at-call-time patching is the right mechanism. The session-scoped approach means all tests in the session share one temp inbox directory. The PR description addresses this: "a fresh directory per session is sufficient to prevent production writes, and session scope avoids creating hundreds of throwaway directories." This is correct — the fixture's purpose is purely to prevent production writes, not to isolate tests from each other's inbox state. Tests that need per-test inbox isolation already have it via `isolate_inbox_server_paths`.

**Patterns this introduces:** The fixture extends the existing conftest isolation vocabulary by adding `LOBSTER_INBOX_DIR` to the set of automatically redirected environment variables. This is the same pattern as `isolate_executor_outputs` (WOS_OUTPUTS_DIR). Future fixtures for other call-time env vars should follow this same form: session-scoped when the env var is read at call time and tests do not need per-test isolation, per-test when inter-test state contamination is a concern.

**What would break without this fixture:** Any test that triggers the `executing → ready-for-steward` transition (i.e., calls `maybe_complete_wos_uow()` with success status on an executing UoW) without explicitly patching `LOBSTER_INBOX_DIR` would write to the live inbox. The PR description documents this as confirmed observed behavior: 82 ghost messages in 5 runs.

**What failure modes exist:** None I can identify in the fixture itself. The only potential future issue is if a production code path reads `LOBSTER_INBOX_DIR` at module import time rather than at call time — such code would not be protected by this fixture (same limitation as the existing `isolate_executor_outputs` fixture). The PR description explicitly acknowledges this known limitation and states it is handled by the existing `isolate_inbox_server_paths` fixture for module-level globals. This is accurate.

**What this makes easier for future work:** Any future code that reads `LOBSTER_INBOX_DIR` from the environment at call time is automatically isolated without requiring any per-test patching. The pattern is established and discoverable in conftest.py.

**What this makes harder:** Nothing. The fixture is additive and non-invasive.

**PR description accuracy:** The description is accurate and notably complete — it states the problem class, confirmed impact, implementation decision rationale, coverage gap note, and the relationship to PR #913. The definition of done is correctly marked N/A for integration and user-visible outcomes.

**Verdict:** APPROVED. The implementation is structurally correct, correctly scoped, uses the right pytest API, handles both teardown cases, and matches the established positive pattern (PR #805) for this isolation class. No changes needed.
