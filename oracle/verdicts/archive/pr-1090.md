#### Gap 1: Schema's `GateFired` literal type does not match `_check_dispatch_eligibility()` actual return values

**What is wrong:** The document defines:

```python
GateFired = Literal["spiral", "dead_end", "burst", "none"]
```

But `_check_dispatch_eligibility()` in `steward.py` actually returns `"escalate"`, `"pause"`, `"throttle"`, or `"dispatch"` — not the pattern-name values the schema proposes.

The document's description says `gate_fired` is "derived from `_check_dispatch_eligibility()` return value" and that `'none' means all cycles returned 'dispatch'." This is inconsistent: the function returns `"dispatch"` for the clean case, `"escalate"` for spiral (not `"spiral"`), `"pause"` for dead-end (not `"dead_end"`), and `"throttle"` for burst (not `"burst"`).

**Why this matters:** The plumbing section says the schema value can be written directly from the function's return value. That is false — a translation layer mapping `{"escalate": "spiral", "pause": "dead_end", "throttle": "burst", "dispatch": "none"}` is needed but not specified. Any implementation that takes the document literally will either (a) write the wrong values or (b) silently write the function's action-verb returns (`"escalate"`, `"throttle"`) into the `gate_fired` column rather than the pattern-name values the schema requires.

The two vocabularies should be reconciled. Either (a) rename `GateFired` literals to match the function's actual return values and update the schema comments, or (b) explicitly name the translation mapping in the Plumbing Additions section and note that `gate_fired` stores the semantic pattern name, not the action verb.

**Decidable test for resolution:** The verdict file and the schema document agree on which vocabulary `gate_fired` uses, and the Plumbing Additions section explicitly names the mapping between `_check_dispatch_eligibility()` return values and `gate_fired` storage values.
