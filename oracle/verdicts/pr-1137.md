### Finding 1 (BLOCKER): JOB_NAME mirror-constant breaks the repo's mirror-constant lint gate

The test file declares `JOB_NAME = "todo-obsidian-sync"` as a module-level string literal. The production module `todo_obsidian_sync.py` already exports `JOB_NAME = "todo-obsidian-sync"` at module scope, and the test already imports the production module as `_sync_mod`. The test has the fix available in one line: `JOB_NAME = _sync_mod.JOB_NAME`.

This is not a style issue — it breaks `tests/test_no_mirror_constants.py`, a lint test specifically written to catch this failure mode. The lint test is failing. The engineer did not catch this because the test suite has pre-existing mirror-constant violations that were already causing that lint test to fail on the base branch. The PR adds one new violation to the list, inheriting the failure. The pre-existing failures do not excuse the new one — the rule is import from the production module, and the mechanism to do so is already in place.

The mirror-constant pattern is named in learnings.md across PRs #696, #800, #967, #970, #973, #974, #980. It was elevated from a "catch during review" to a "check before review" posture. This PR introduces it at a test file that imports the production module and could easily use `_sync_mod.JOB_NAME`. Fix: replace `JOB_NAME = "todo-obsidian-sync"` with `JOB_NAME = _sync_mod.JOB_NAME`.

### Finding 2 (NEEDS-CHANGE): TODOS_FILENAME is a mirror-constant the lint test misses

The test declares `TODOS_FILENAME = "✅ ACTIVE TODOS.md"` as a module-level string literal. The production constant is `ACTIVE_TODOS_FILENAME = "✅ ACTIVE TODOS.md"` in `obsidian_sync_core` (re-exported through `_sync_mod`). The names differ (`TODOS_FILENAME` vs `ACTIVE_TODOS_FILENAME`), so the lint test does not catch this instance — but the divergence risk is identical. If `ACTIVE_TODOS_FILENAME` ever changes its value (e.g., to add a different emoji prefix), this test would silently assert against the old value while production behavior changed.

The test already has `_sync_mod` in scope. Fix: `TODOS_FILENAME = _sync_mod.ACTIVE_TODOS_FILENAME` (or import from `obsidian_sync_core` directly).

### Finding 3 (INFORMATIONAL — engineer acknowledged): Test 5 doesn't assert sync was skipped

Test 5 (missing vault) asserts `mock_pull.assert_not_called()` and `mock_commit.assert_not_called()`. It does not assert that `sync_obsidian_to_db` was not called. In the actual execution path with a missing vault directory, `sync_obsidian_to_db` is correctly skipped because `content == ""` — but the test doesn't verify this. The engineer noted this gap explicitly.

Additionally, the test's docstring states the spec as "clean exit, no crash" with pull and commit skipped — but the actual production behavior for a missing vault path includes creating the directory via `todos_path.parent.mkdir(parents=True, exist_ok=True)` and writing the rendered file to it. The test does not assert whether the file was written or the directory was created. This is a behavior the spec comment implies should not happen ("clean exit") but the code does. Whether this is correct behavior in production is a design question, but the test is not exercising the before-state or asserting the after-state of the filesystem.

This is informational, not a blocker — the engineer named it, and the test is not wrong about what it does assert. A follow-on test that checks the filesystem state would close this gap.

### Finding 4 (INFORMATIONAL): Test comment in Test 5 gives the wrong reason for pull being skipped

The assertion comment reads: "git_pull must not be called because the vault dir doesn't exist (no .git)." The production check on line 185 is: `if vault_path.exists() and (vault_path / ".git").exists()`. Pull is skipped because `vault_path.exists()` returns False — the `.git` check is never reached. The comment implies `.git` absence is the operative condition, but it's the parent directory that's absent. This is a documentation issue, not a defect.
