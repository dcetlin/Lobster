### Issues requiring changes

**Issue 1: Migration 118 cron detection pipe is broken (BLOCKER)**

The first condition in Migration 118 is:

```bash
if crontab -l 2>/dev/null | grep -qF "0 9 * * *" | grep -q "wos-metabolic-digest" 2>/dev/null; then
```

`grep -qF` suppresses output (the `-q` flag means "quiet — write nothing to standard output"). The second `grep -q` in the pipe receives empty input and will always return exit 1 (no match). This means the "update existing 09:00 entry" branch is dead code — it will never fire. The migration falls through to the `elif` (add fresh entry) or the `else` (already present at correct schedule), depending on crontab state.

In practice: on a system where the entry exists at 09:00 UTC, the `elif` condition (`! crontab -l | grep -qF "$WOS_DIGEST_MARKER_118"`) is false (the marker IS present), so it falls to the `else` block, which then checks `grep -F "$WOS_DIGEST_MARKER_118" | grep -q "^0 9 "`. This second-level check works correctly. So the 09:00→10:00 rescheduling does get handled, but via the else path rather than the if path, and the `migrated` counter is not incremented in the first branch (which is dead).

The net consequence: on a system with the 09:00 cron already present, the migration produces the correct cron change but the first conditional branch never fires, making the code misleading and the branch coverage analysis incorrect.

**Fix required:** Replace the broken pipe with:

```bash
if crontab -l 2>/dev/null | grep -F "0 9 * * *" | grep -q "wos-metabolic-digest"; then
```

(Remove the `-q` from the first grep so output passes through to the second grep.)

**Issue 3: `format_digest` signature receives `db_path` but only uses it for seeds fallback — naming gap**

The `format_digest` docstring says `db_path: Registry DB path (used only for failed-UoW gate parsing)`. This is incorrect — `db_path` is used for `query_seeds_from_artifacts` via `count_seeds_surfaced`, which is called in `run_digest` (not in `format_digest`). Looking at the actual code: `format_digest` receives `seeds_total: int` already computed, and `db_path: Path` as a parameter — but `db_path` is not referenced anywhere in the body of `format_digest`. The docstring says it is used "only for failed-UoW gate parsing" but no code in `format_digest` calls `db_path`. This is a dead parameter with a misleading docstring.

**Fix required:** Remove `db_path` from `format_digest` signature (and all call sites), since `seeds_total` is already pre-computed and passed in. The docstring note about db_path is incorrect — it is not used.

**Issue 4: `_avg_int` return type annotation is imprecise**

```python
def _avg_int(values: list[int | None]) -> float | None:
    nums = [v for v in values if v is not None]
    return round(sum(nums) / len(nums), 1) if nums else None
```

`round(..., 1)` returns a `float` in Python, which matches the annotation. But the function is named `_avg_int` (average of integers) while it returns a rounded float. The function is used for both token counts and cycle counts. Token averages are shown as `int(avg_tokens)` in the formatter (which discards the `.1` precision), meaning the rounding in `_avg_int` is immediately discarded for token usage. Minor naming imprecision — not a blocker, but worth noting.

**Issue 5: `count_seeds_surfaced` JSON parsing of `seeds_surfaced` column is speculative**

```python
parsed = json.loads(sv) if isinstance(sv, str) else sv
count = len(parsed) if isinstance(parsed, list) else int(parsed)
```

The schema for `seeds_surfaced` is undocumented in this PR. It handles three cases: string JSON array, list, and integer. But `seeds_surfaced` is noted as "migration 0020 approximation" and the column is selected defensively as NULL when absent. There is no schema definition for what `seeds_surfaced` will contain when it does exist. The code may never hit the non-NULL path until migration 0020 lands. Low severity since the fallback to artifacts table is correct, but the forward-compatibility parsing is untested (no test exercises a non-NULL seeds_surfaced value).
