## Stage 1 — Vision Alignment

**Prior:** This implementation is solving the right problem (duplicate PRs from re-dispatched UoWs) in a direction that forecloses the better path (structural enforcement).

**Theory of change:** WOS executor re-dispatches UoWs, agents create duplicate PRs because nothing checks for existing ones. The fix embeds bash-snippet idempotency checks into agent instruction markdown files.

**What would have to be true:** (1) Agents reliably execute multi-step bash boilerplate from markdown instructions. (2) Every agent definition that calls `gh pr create` is updated. (3) Advisory instruction enforcement is sufficient for an idempotency invariant.

**Stage 1 finding — advisory enforcement for a structural invariant violates vision.yaml principle-3:**

Vision.yaml operating principle-3 states: "If-then logic and field checks are code, not LLM instructions. Use LLMs where genuine interpretation is required." An idempotency check ("does a PR with this branch/title already exist? if yes, skip creation") is definitionally if-then logic with no interpretive component. Embedding it as a bash code block in a markdown instruction document and labeling it "MANDATORY" is advisory enforcement of a structural invariant.

The learnings.md pattern "instruction-accumulation as self-defeating behavioral rule remediation" (2026-04-22) applies directly: when a behavioral rule is not followed, the system's response is to add more instruction text. The idempotency check is a behavioral rule ("always check before creating") added as instruction text to agent definitions. The failure mode is identical: an LLM agent under context pressure may skip the check, producing the exact duplicate PR the instruction was meant to prevent.

The golden-patterns.md entry "advisory-to-hook promotion" (2026-04-22) names the structural alternative: if the enforcement condition is a string pattern (`gh pr create` in a bash command), it can be a PreToolUse hook. A hook that detects `gh pr create` and verifies no existing PR matches the branch/title would fire regardless of context state, agent definition, or instruction absorption. This is the correct enforcement layer.

**What this forecloses:** Shipping this as advisory enforcement establishes the pattern that idempotency checks belong in agent instructions. Future duplicate-PR incidents will be addressed by adding the check to more agent definitions (the accumulation spiral), rather than by implementing the structural hook that would have prevented all of them.

**Opportunity cost:** The bash snippet in functional-engineer.md is ~15 lines. A PreToolUse hook implementing the same logic would be ~30 lines of Python. The structural solution is marginally more work and categorically more reliable.

**Feature bundling:** The PR bundles four unrelated commits: (1) oracle verdict archive for PR #1146, (2) analytics feature `outcome_cost_correlation` for UoW uow_20260510_963e4d, (3) the idempotency check itself, (4) desloppify behavior update. The PR description is "Test PR on fork" — a placeholder. This is the documentation-review learnings.md pattern "feature-bundling" (2026-04-26): count changed files, compare against PR title and description, flag any file not mentioned.

### Idempotency check: advisory, not structural (blocking)

The check is embedded as a bash code block in markdown instruction files. This is advisory enforcement. The correct implementation is a PreToolUse hook that intercepts `Bash` tool calls containing `gh pr create` and programmatically verifies no matching PR exists before allowing the call to proceed. This would provide structural enforcement that cannot be bypassed by context pressure, instruction absorption, or agent definition omission.

### Idempotency check: incomplete coverage

The check is added to `functional-engineer.md` and `lobster-shop/desloppify/behavior/system.md`. Any other agent definition or ad-hoc subagent that calls `gh pr create` is not covered. A hook covers all agents by construction.

### Idempotency check: title-prefix matching is fragile

The title-prefix check (`--limit 50 ... startswith($t)`) uses the first 60 characters. If two different UoWs produce PRs with similar titles (e.g., "fix: update steward..."), a false positive will prevent a legitimate PR from being created. The branch-name check is the reliable discriminator; the title-prefix check adds false-positive risk without clear benefit.

### Analytics feature: quality observations (non-blocking, for when it is split to its own PR)

The `outcome_cost_correlation()` implementation in `analytics.py` follows the established pattern in the module well: pure function composition, `_connect_ro`, graceful `data_gap` handling, consistent return shape. The `_BAD_OUTCOME_CATEGORIES` set uses raw string literals rather than importing from a metabolic-category enum (if one exists) — the mirror-constant pattern (learnings.md, multiple PRs). The wiring into `wos_dashboard.py` uses a function-body lazy import, which is consistent with how that module handles cross-module imports. The `wos_metrics_report.py` text rendering is well-structured with a canonical display order.

### Encoded Orientation check

The changes to `functional-engineer.md` modify behavioral defaults for agent PR creation — this is an Encoded Orientation decision per constraint-3. No prior logged decision exists in `oracle/verdicts/` or `meta/premise-review.md` authorizing "idempotency checks should be advisory instructions in agent definitions." The vision.yaml anchor (principle-3) actually argues against this approach, not for it. This is flagged as an Encoded Orientation decision that lacks a vision.yaml-aligned prior.
