## Round 1 — 2026-05-14

### Stage 1: Vision Alignment

**Prior entering this review:** This implementation is solving the wrong problem, or solving the right problem in a direction that forecloses better paths.

**Theory of change check.** The PR closes a real gap: sweep detects structural smells at 2 AM, files GitHub issues, but those issues sit unqueued for up to 4+ hours until the cultivator runs the next morning. The theory is that immediacy of WOS queueing matters and that sweep-identified items are categorically appropriate work items. Both claims deserve scrutiny.

On the first claim (immediacy matters): the cultivator delay was not purely incidental — it provided a natural human-review window. Morning briefing, Dan's first look at the queue, the cultivator's own filtering logic — these all operated in the gap between issue filing and WOS queuing. This PR eliminates that window for sweep-originated UoWs. The question is whether the 4-hour delay was a bug or a feature. The PR treats it as a bug.

On the second claim (sweep items are categorically appropriate WOS work): the sweep's escalation quality varies. A given sweep night may surface structural smells of wildly different severity, relevance, and WOS-readiness. The cultivator applies its own signal evaluation before queuing; this PR bypasses that filter for sweep-originated items by routing directly to `registry.upsert()`. The items enter as `proposed` — which is the same state the cultivator places them in — so they are not auto-executed. But they exist in the queue without cultivator-level validation.

**Is there a cheaper test of the underlying assumption?** The assumption to test is: sweep-identified smells, when queued immediately as proposed UoWs, don't generate noise in the WOS pipeline or inflate the queue with low-signal items. The cheaper test would be to run one sweep night with the new wiring enabled, observe the proposed items, and verify the quality before committing this as a permanent behavioral default.

**What does this work foreclose?** It forecloses a "sweep items route through cultivator before queuing" path. Once sweep-originated UoWs are in the registry with `source="sweep"`, the cultivator's dedup check will skip them on its next run. If the sweep creates a low-signal UoW and the cultivator would have filtered it, that UoW is now in the pipeline. The downstream pressure falls on the Steward.

**Vision alignment verdict (locked before Stage 2):** This is the right direction — principle-4 ("wire what exists before building more") and principle-4's "missing arrows between existing systems are the velocity multiplier" both point here. The concern is not whether to wire the sweep to WOS but whether this PR constitutes an Encoded Orientation decision without a logged prior. The PR description claims "Decision from 2026-05-14 deep work digest: Lobster has delegated authority to promote sweep observations into the WOS queue." This citation needs to be verified under constraint-3 before the verdict can be APPROVED. I searched `oracle/verdicts/`, `oracle/decisions.md`, `vision.yaml open_decisions`, and recent sweep files — no logged decision with that substance was found.

**Alignment verdict: Questioned** — the implementation direction is sound but the Encoded Orientation authorization is unverified.

### Stage 2: Quality Review

**Does it do what it claims?** Yes. The three-file structure is clean: `sweep_uow_promoter.py` as the pure function with injected dependencies, `sweep-uow-promote.py` as the CLI wrapper, and `negentropic-sweep.md` as the instruction document. The PromoteResult enum pattern correctly applies the learnings.md pattern from PR #807 (named result type avoids bool/None ambiguity). The job-enabled gate is correctly co-located with the sweep job name.

**source_ref/source column question (author-flagged).** The author flagged: `source_ref=SWEEP_SOURCE` passes "sweep" as source_ref, which the registry uses directly as the `source` column. Verifying in `registry.py` lines 654-655: `source = source_ref if source_ref else f"github:issue/{issue_number}"`. So `SWEEP_SOURCE = "sweep"` becomes the `source` column value. This is correct — it mirrors how the cultivator uses `"github:issue/N"` as the source. The `source_ref` column is also written as "sweep" (lines 709). Both columns get the same value, which is consistent with the cultivator's behavior where `source_ref` for cultivator UoWs carries the canonical GitHub ref. This is an acceptable design choice with one caveat: the `source` column semantics for sweep UoWs differs from cultivator UoWs (cultivator uses `"github:issue/N"` as a queryable ref, sweep uses `"sweep"` which is less specific). If any downstream query filters by source to trace back to a specific issue, `source="sweep"` loses that traceability — only `source_issue_number` carries the issue identity. This is an acknowledged design choice, not a bug, but worth naming.

**step-1-matched boundary (author-flagged).** The instruction says NOT to call the script for items that matched an existing issue in step 1. This is correct. Those items may already have a UoW (the cultivator might have queued them on a prior run). The registry's dedup would catch it, but avoiding the call is cleaner. The boundary is the right one.

**Default success_criteria (author-flagged).** `_DEFAULT_SUCCESS_CRITERIA` is a fixed generic string. The PR notes "The Steward refines this at germination if the issue body contains a richer acceptance criteria section" — but the Steward's germination logic operates on the UoW record, not the issue body directly. Unless the Steward fetches the issue body, the default criteria will persist. This is advisory: the default is non-empty (satisfying `registry.upsert()`'s validation) and meaningful enough for routing, but it is not issue-specific.

**Encoded Orientation check (constraint-3, OODA constraint).** This PR changes a durable behavioral default: sweep-filed issues now automatically enter the WOS pipeline as proposed UoWs. Previously no such automatic promotion existed. This is an Encoded Orientation decision. Checking the two required conditions:

1. A prior logged decision exists: The PR cites "Decision from 2026-05-14 deep work digest." This language does not correspond to any entry in `oracle/verdicts/`, `oracle/decisions.md` (deprecated, checked), or `vision.yaml open_decisions`. The learnings.md PR #894 pattern is directly applicable: "A GitHub issue describing a design decision is not equivalent to a logged oracle/vision decision." A "deep work digest" is equally not an oracle/vision logged decision. The claimed authorization does not satisfy constraint-3. This is **Gap 1**.

2. Traceable vision.yaml anchor: There is a plausible anchor — vision.yaml `principle-4` ("wire what exists before building more") and `active_project.phase_intent` (Registry populated by the issue sweeper, steward picks up). These anchors exist. The gap is in item 1 (no logged prior), not in item 2.

**Mirror-constant detection (first-pass check per learnings.md PR #974).** The test asserts `row["source"] == "sweep"` using a raw string literal. `SWEEP_SOURCE = "sweep"` is defined as a named constant in the production module. The test should import `SWEEP_SOURCE` and assert `row["source"] == SWEEP_SOURCE`. This is the mirror-constant pattern, sixth-or-later recurrence. **Gap 2.**

**uv convention.** The negentropic-sweep.md change correctly uses `uv run ~/lobster/scheduled-tasks/sweep-uow-promote.py`. The PR description tests include `python3 -m py_compile ...` (bare python3) — but that is in the PR description, not in a committed file. No committed file uses bare python3. Pass.

**Feature-bundling check.** Four files changed, all described in the PR: `sweep_uow_promoter.py` (new), `sweep-uow-promote.py` (new), `negentropic-sweep.md` (modified), `test_sweep_uow_promoter.py` (new). No undescribed files. Pass.

**Multi-fire / notification flood check.** The promoter creates one UoW per issue and returns SKIPPED_DEDUP on re-run. The job-disabled gate returns SKIPPED_JOB_DISABLED. No inbox notifications are emitted — the script only prints to stdout and logs to stderr. The sweep agent records the one-line output in its sweep file. No notification flood risk.

**Dead code check.** No dead constants or dead branches visible.

**Path conventions.** The CLI script uses `Path(__file__).parent.parent` as repo root. The jobs_json default resolves to `Path.home() / "lobster-workspace" / "scheduled-jobs" / "jobs.json"`. This is the correct canonical path for `jobs.json` consistent with `LOBSTER_WORKSPACE`. The `LOBSTER_WORKSPACE` env var is not consulted — the path is hardcoded. This means if someone has a non-standard LOBSTER_WORKSPACE, the jobs.json path would resolve incorrectly. However, the hardcoded `Path.home() / "lobster-workspace"` matches the pattern used elsewhere in this codebase (e.g., `sweeper.py`). Advisory, not a blocker.

### Patterns introduced

The PromoteResult enum as the return type of `promote_sweep_issue()` is a clean application of the learnings.md classification-result-as-typed-return principle. The pure-function-with-injected-dependencies design is correct. The job-name as the gate key (co-locating promotion enablement with sweep enablement) is a pattern worth preserving.

### What this forecloses

Once sweep-originated UoWs enter the registry, the cultivator's pre-queue filtering cannot retroactively apply. Future attempts to add cultivator-level quality filtering to sweep items would require either a new pipeline stage before `registry.upsert()` or a post-upsert validation pass. The current design assumes sweep-filed issues are WOS-ready; that assumption is now structural.
