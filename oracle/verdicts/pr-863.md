## PR #863 — ScalingGovernor: cap WOS executor dispatch based on Attunement load

### Stage 1: Vision Alignment

**Stage 1 finding (locked before seeing implementation):**

The adversarial prior: this implementation is solving the wrong problem, or solving the right problem in a direction that forecloses better paths. The PR addresses a failure mode named explicitly in vision.yaml `current_focus.current_constraint`: "WOS executor is dispatching UoWs but pipeline has starvation symptoms — RALPH Cycle 7 all 4 UoWs failed." The PR description confirms the precipitating event: a 50-run failure with 252 injected UoWs and 250 failures (0.8% success rate). The causal mechanism ("success triggers collapse — a clean small-scale run unlocked maximum injection with no historical Attunement evidence at that scale") is specific and named.

The adversarial case to hold: is a dispatch cap the right intervention, or is it treating a symptom of a different root cause? Two alternatives are credible. First: the 50-run failure may reflect a problem with the UoW queue composition (wrong UoWs in ready-for-executor) rather than dispatch volume. A cap would not fix queue composition; it would just fail fewer UoWs per cycle. Second: the cap heuristic (N/5, power-of-2 attunement windows) may introduce a new pathology — correct attunement is reset whenever the workload mix changes structurally, causing the governor to cap legitimate throughput after a period of failure that is not representative of current queue quality.

Neither alternative defeats the premise, but they constrain how to evaluate the implementation: the governor must be auditable (can we see why it capped?), overridable (can we bypass it when the cap is wrong?), and fail-safe (does it default to conservative behavior under ambiguity?). These are the load-bearing design requirements. If the implementation satisfies them, the intervention is correctly structured even if the root cause is partially elsewhere.

The learnings.md pattern "new module with 733 additions and 0 deletions with no import in any existing file is a library with no wiring" (PR #714) is directly relevant. I will check whether the governor is correctly wired into the production dispatch path, not just introduced as a library.

**Alignment verdict:** Confirmed

### Stage 2: Quality Review

**Encoded Orientation check (constraint-3):** The ScalingGovernor changes behavioral defaults — it caps UoW dispatch based on a computed attunement signal, a durable change to the system's decision-making loop. This is an Encoded Orientation decision. Check: (a) prior logged decision — the PR description references "WOS UoW uow_20260423_c87689" as the design reference, and vision.yaml `current_focus.what_not_to_touch` explicitly includes "WOS dispatch loop pattern gating (spiral/dead-end/burst) is in scope as part of WOS execution health." That entry is the prior decision anchor. (b) vision.yaml anchor — `current_focus.current_constraint` names executor dispatch starvation as the live constraint; `current_focus.this_week.primary` names "fix executor dispatch starvation" as primary work. Both conditions are met. Constraint-3 is satisfied.

**Quality findings:**

- **Wiring is correct and complete.** The governor is imported inside `run_executor_cycle()` (deferred import, not module-level), called before the dispatch loop, and its decision slices `eligible_uows` in the production path. The learnings.md PR #714 failure mode ("library with no wiring") does not apply — the production call site is present and the slice is live.

- **Pure function separation is clean.** `_compute_decision()` takes `proposed_n` and `attunement_scale` as arguments and has no I/O. All DB access is isolated in `_query_attunement_scale()`. All config access is isolated in `_read_override_flag()`. This directly applies the golden pattern "classification result as typed frozen dataclass with observability fields" (PR #602): `GovernorDecision` carries `proposed_n`, `allowed_n`, `capped`, `cap_reason`, and `attunement_scale` — all four observability fields the golden pattern requires. The decision is auditable at log time without re-querying.

- **Failure modes are safe but one is notable.** The `check()` method catches all exceptions and applies `_compute_decision(proposed_n, 0)` as fallback — which always caps. This is the correct conservative default. One specific failure mode worth naming: `_query_attunement_scale()` uses `WHERE status IN ('done', 'failed') ORDER BY updated_at DESC LIMIT 500`. If `updated_at` is not indexed, this query will do a full table scan on every heartbeat cycle. On a registry with thousands of rows, this is a latency risk. The `sqlite3.connect(..., timeout=5.0)` guard handles the DB-unavailable case but not the slow-query case. This is not blocking — the query will eventually complete, and the governor itself is not on the critical path for the user — but it is a known debt surface.

- **Power-of-2 attunement window has a correctness subtlety.** The algorithm checks windows of size 1, 2, 4, 8, ... up to `proposed_n`. The test `TestEightUoWsSevenDone.test_cap_fires_when_proposed_is_4` documents this: with rows ordered DESC (failed first), w=1 gets 0% success, w=2 gets 50%, w=4 gets 75% — all below 80%. So `attunement_scale=0` for `proposed_n=4` even though 7/8 total UoWs succeeded. This is the intended behavior (the most recent UoW failed, so evidence at scale 1 is 0%) and the test explicitly documents it. The behavior is correct per spec; a reader not seeing the test might find it surprising. The test comment is sufficient documentation.

- **Test coverage is thorough and correctly scoped.** 22 tests cover: empty DB, 8-row clean window, 8-row mixed window with large proposed_n, override flag, DB failure (nonexistent and corrupt), proposed_n=1, and the pure function directly. The `_patch_inbox` extension in `test_executor_heartbeat.py` correctly isolates governor logic from dispatch-eligibility logic using a `_NoCapGovernor` stub — this prevents the governor from interfering with tests that are not testing the governor. The concern from learnings.md (PR #720: "test files may pass because they mock the function") is mitigated here because the production wiring test (the slice in `run_executor_cycle`) is exercised by the integration path, not by the unit test mock.

- **Feature-bundling check:** 4 files changed — `executor-heartbeat.py`, `scaling_governor.py`, `test_scaling_governor.py`, `test_executor_heartbeat.py`. All four are named in the PR description. No undocumented scope expansion.

- **`_DEFAULT_WOS_CONFIG_PATH` module-level variable:** This is set to `None` and resolved lazily. The learnings.md PR #839 pattern ("module-level constant computed from env var at import time is not overridable by monkeypatch.setenv") does not apply here because `_DEFAULT_WOS_CONFIG_PATH` is not computed at import time — it defaults to `None` and `_get_wos_config_path()` resolves it at call time. The `wos_config_path` constructor parameter correctly overrides at the instance level. Tests use this parameter directly; no monkeypatching of module-level state is required.

**Patterns introduced:**

- Governor-as-gate: a pure-function decision layer inserted between eligibility filtering and dispatch, with a frozen dataclass result carrying all observability fields. This is a clean application of the "classification result as typed frozen dataclass" golden pattern at the dispatch layer, not just the germination layer (PR #602). It is reusable: any future dispatch constraint (burst guard, per-type cap, rate limit) can follow the same structure.

- Override flag as escape hatch: `scaling_governor_override: true` in `wos-config.json` provides a documented, logging bypass path. The flag location is consistent with existing `execution_enabled` pattern in the same file.

**What this forecloses:**

The power-of-2 window heuristic is now the canonical attunement signal. A future PR that wants to use a different window shape (rolling average, exponential decay, per-type success rate) would need to either change `_query_attunement_scale()` or add a second governor class. The pure function separation makes this straightforward — the computation is behind a named seam. This is a mild foreclosure, not a significant one.

**Opportunity cost note:**

The vision.yaml `current_focus` names "drain 106 proposed UoWs through confirmed pipeline" as this week's primary work. The governor directly enables this by preventing the failure mode that produced 250 failures in 252 dispatches. This is correctly prioritized work.
