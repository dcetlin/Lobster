## Stage 1: Vision Alignment

**Prior entering this review:** This implementation is solving the wrong problem, or solving the right problem in a direction that forecloses better paths.

**The question this work is meant to serve:** Remove a dead code path (direct Anthropic SDK call in `extractor.py`) that was never executed in production, because the LOS scanner is a Type A job where the subagent performs extraction natively.

**Theory of change:** Dead code that imports the Anthropic SDK forces callers and tests to mock it, confuses the module's stated purpose, and carries ongoing cognitive overhead. Removing it clarifies the responsibility boundary: the subagent extracts; `extractor.py` only parses and persists.

**What would have to be true for this to be the right path:** The `_call_llm` path must genuinely be unreachable in production — and the new interface (subagent calls `persist_extracted_items()` with pre-extracted JSON) must be wired at every call site.

**What does this foreclose:** The primary concern is whether removing `_call_llm` forecloses a future where a Type B (cron-direct) runner calls `extract_action_items` synchronously without going through a subagent. The PR author correctly argued that LLM reasoning cannot be done in a Type B job anyway — so this path was architecturally foreclosed before this PR was written. The removal does not introduce new foreclosure.

**Alignment verdict: Confirmed** — removing dead code that confused module responsibility is consistent with vision.yaml principle-3 ("Determinism over judgment for conditionals — use LLMs where genuine interpretation is required"), which implies extraction belongs in the LLM-capable subagent layer, not in a synchronously-called module. It also aligns with principle-2 ("Ergonomics over shortcuts — prefer forms that keep the correction path open") by making the module's actual responsibility legible from its signature.

The oracle vocabulary check from learnings.md is clean on this task: no mirror-constant pattern (constants removed rather than mirrored), no feature-bundling (3 files, all named in the PR description), no dead-code-removal without grep verification (the PR is removing, not filing for removal of an unverified target). No Encoded Orientation issue — this is a code path removal, not a behavioral default change.

**Stage 1 finding (locked):** The removal is the right move at the right time. The risk is not in the decision to remove; it is in whether every call site was updated.

## Stage 2: Quality Review

### What changed

Three files: `extractor.py` (removes `_call_llm`, `anthropic` import, `EXTRACTION_MODEL`, `MAX_TOKENS`, `_SYSTEM_PROMPT`; changes `extract_action_items` signature from `text: str` to `items: list[dict]`), `los-action-scanner.py` (replaces `_scan_and_extract` with `persist_extracted_items`; `main()` now emits candidate JSON to stdout), `tests/unit/los/test_extractor.py` (rewrites tests to drop all anthropic mocks).

### Critical gap: call site not updated

**The task definition file at `scheduled-tasks/tasks/los-action-scanner.md` instructs the subagent to call `extract_action_items` with `text=msg["text"]`** — the old interface. This file is the operative instruction the Type A subagent reads at runtime. After this PR merges, every subagent invocation of the los-action-scanner job will call `extract_action_items(conn=conn, text=msg["text"], ...)` and receive a `TypeError` because `text` is no longer a valid parameter.

This is the PR #720 pattern from learnings.md: "Interface migration without updating all callers." The PR changed the public function's signature but left one production caller using the old signature. The learnings.md rule states: "when a PR changes a public function's return type... enumerate all production callers (not just test files) and verify each uses attribute access, not subscript access." The analogous rule here: when a PR changes a function signature, enumerate all callers including task definition markdown files. The test suite passes because it calls `extract_action_items` directly with the new interface — the task definition file is not part of the test surface.

The gap is concrete: `los-action-scanner.md` Step 3 contains Python code that calls `extract_action_items(conn=conn, text=msg["text"], ...)`. This code is executed by every subagent invocation of this job. After the PR merges, this code raises `TypeError: extract_action_items() got an unexpected keyword argument 'text'`.

### Secondary gap: `persist_extracted_items` is in the wrong module

The task context prompt (PR body) flags this as a known gap: "`persist_extracted_items` in the scanner is now unused by `main()` — the scanner emits candidate JSON to stdout for the subagent to consume."

This is correct and worth elevating. `persist_extracted_items` is defined in `los-action-scanner.py` — a script — but it is meant to be called by a subagent. A subagent cannot import from a scheduled-task script cleanly. The import path would be fragile (sys.path manipulation to a non-package script). The natural home for a function that parses JSON from a subagent and persists items is `src/los/extractor.py` — the module that already owns parse and persist logic. The function belongs there, not in the scanner script. As-is, the subagent has no clean way to call `persist_extracted_items` unless the task definition instructs it to call `extract_action_items` and `parse_llm_response` directly from `src.los.extractor` — which is the correct pattern but requires the task definition to describe it, which brings us back to Gap 1.

The canonical placement principle (golden-patterns.md, 2026-04-25) applies: "The right home for any file, module, or definition must be unambiguous without interpretation." `persist_extracted_items` as a function in a script file has no clear import path for the subagent that is supposed to call it. If the intended pattern is "subagent calls `parse_llm_response` + `extract_action_items` from `src.los.extractor` directly," then `persist_extracted_items` is dead code (main() doesn't call it; the subagent can't import it cleanly). If `persist_extracted_items` is meant to be the callable, it should move to `src/los/extractor.py`.

### What does work

- The removal of `_call_llm`, the Anthropic import, and the associated constants is clean and complete within `extractor.py`.
- Tests correctly drop all anthropic mocks and assert against imported constants (`PRIORITY_MIN`, `PRIORITY_MAX`, `PRIORITY_DEFAULT`) rather than local re-declarations — this avoids the mirror-constant pattern from learnings.md.
- `compute_dedup_key` is now correctly imported from `src.los.db` rather than re-exported through `extractor.py` (it never lived there; the old test imported it from the wrong place).
- `parse_llm_response` is retained as a pure function — correct, subagents still produce JSON that needs parsing.
- The scanner's `main()` emitting candidates to stdout is internally consistent with the Type A job model: the script does collection and output; the subagent does reasoning. The design is sound.
- Dry-run mode removal from the scanner is a simplification consistent with the new design (the scanner no longer does extraction; dry-run had meaning only for the extraction path).

### Encoded Orientation check

This PR does not change behavioral defaults, system constraints, agent identity, or decision-making rules in a durable way. It removes a code path and updates an interface. No constraint-3 issue.

## Revision contract

**Gap 1 (blocker): `los-action-scanner.md` task definition must be updated to use the new interface.** The Step 3 code block must be rewritten to describe the correct subagent flow: call `parse_llm_response` + `extract_action_items` from `src.los.extractor` directly (or call `persist_extracted_items` if that function is moved to the extractor module). "Addressed" means the task definition no longer contains a call to `extract_action_items(conn=conn, text=..., ...)`.

**Gap 2 (advisory, not blocking): Clarify the intended home and call path for `persist_extracted_items`.** Options: (a) move it to `src/los/extractor.py` so the subagent can import it cleanly, and update `los-action-scanner.md` to use it; (b) keep it in the scanner script but document that subagents must call `parse_llm_response` + `extract_action_items` separately from `src.los.extractor`, making `persist_extracted_items` either dead code to be removed or a local convenience for non-subagent callers. The current state — defined in a script, not called by `main()`, not importable cleanly by subagents — is the "library with no wiring" pattern from learnings.md (PR #714): the function exists but has no reachable caller. Addressed means the function is either (a) in a module the subagent can import and the task definition uses it, or (b) removed from the script with a comment explaining the subagent calls the lower-level functions directly.

Gap 1 must be resolved before merge. Gap 2 should be resolved in this PR or deferred with an explicit statement in the PR description naming it as a follow-on.
