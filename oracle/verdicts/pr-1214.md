## Round 1 — 2026-05-18

### Stage 1: Vision Alignment

**Prior entering review:** This implementation is solving the wrong problem, or solving the right problem in a direction that forecloses better paths.

**Finding (locked before reading implementation):**

The stated gap — `wos start`/`wos stop` doesn't manage systemd timers, leaving timer-backed jobs running after stop and not restored after start — is a real operational failure. The vision.yaml `current_focus.this_week.primary` explicitly states "WOS executor now live (execution_enabled=True, enabled 2026-05-18)" and the `current_constraint` is the bidirectional interface, not upstream feed health. An upstream feed that silently stopped producing UoWs on May 7 is a precondition failure that would invalidate all downstream WOS work. This is the right problem.

The theory of change: the WOS control surface (`wos start`/`wos stop`) has an implied semantic contract — start means "the pipeline runs," stop means "the pipeline is paused." A partial implementation that writes `jobs.json` and `wos-config.json` but leaves systemd timers as a separate unmanaged layer breaks that contract. Operators who issue `wos start` after `wos stop` will not observe the pipeline resuming, with no visible failure signal. The correct fix is to make the control surface manage all layers of the dispatch stack.

The cheaper alternative would be to detect disabled timers in a health check and surface a warning. But this is reactive (principle-1 violation) and still leaves the gap open for the next `wos stop` + `wos start` cycle. The PR's approach is preventive and closes the gap structurally.

**What this forecloses:** Nothing significant. The implementation adds a well-isolated function (`_toggle_systemd_timers`) with a strict safety guard. It does not change the jobs.json or wos-config.json management layer.

**Alignment verdict: Confirmed.** This is an infrastructure correctness fix directly serving the active WOS execution constraint named in `current_focus`.

### Stage 2: Quality Review

**Encoded Orientation check (constraint-3):** PR #1079 established the `wos start`/`wos stop` command pattern and explicitly concluded constraint-3 is not triggered for correctness fixes that close gaps between command intent and implementation. PR #1214 extends the same operator-initiated command to manage the systemd timer layer — same command, same semantic intent, same class of fix. The prior logged verdict (oracle/verdicts/pr-1079.md) is the authorizing record. Constraint-3 is not triggered.

**Does it do what it claims?**

Yes, with one minor accounting defect noted below.

**`_toggle_systemd_timers` function:**

The LOBSTER-MANAGED marker guard is correct: it reads the unit file content and checks for the string before calling `systemctl`. Verified against the live timer `/etc/systemd/system/lobster-issue-sweeper.timer` which carries `# LOBSTER-MANAGED` on line 3. All 7 jobs in `_WOS_CORE_TIMER_JOBS` have live timer files in `/etc/systemd/system/`. The frozenset is a proper subset of `_WOS_CORE_JOBS` with no phantom entries.

The fail-soft pattern is correct: `subprocess.run` errors and non-zero exit codes are caught and logged without raising. The function returns only names of successfully toggled timers (returncode == 0 path), not attempted-but-failed names. This is the right contract: the caller uses the return value to report success, so it must reflect actual success.

**Migration 119 safety:**

The `execution_enabled` conditional correctly gates the timer re-enable on the active pipeline state: if someone runs upgrade with `execution_enabled=false`, the migration correctly leaves timers disabled. The idempotency guard (`systemctl is-enabled --quiet`) prevents re-enabling timers that are already enabled.

**Minor defect — `_m119_enabled_count` inflates on systemctl failure:**

In the migration loop:
```bash
sudo systemctl enable --now "$_m119_timer" 2>/dev/null \
    && substep "..." \
    || warn "..."
_m119_enabled_count=$((_m119_enabled_count + 1))
```

The `_m119_enabled_count` increments unconditionally — including when `systemctl enable` fails (taking the `|| warn` branch). The subsequent `migrated=$(( migrated + _m119_enabled_count ))` then inflates the migration counter with failed attempts. In the common success case this is invisible; on a system where `sudo` is restricted, all 7 attempts fail but `migrated` reports 7 migrations ran. This is a misleading but harmless accounting error that affects only the human-readable summary line, not any state transition.

Not a blocker. The actual enablement is idempotent and verified by `systemctl is-enabled --quiet` before attempting the enable.

**Migration number:**

Main branch's highest migration is 117. The PR uses 119, skipping 118. No open PR claims 118. The skip is a minor numbering gap (not a collision) — the migration will run correctly. This is an anomaly worth noting but not a blocker.

**Test coverage:**

14 new tests covering:
- `_toggle_systemd_timers` enable/disable paths
- LOBSTER-MANAGED guard (skip when missing)
- Unit file absent (skip)
- systemctl non-zero returncode (continue, return partial list)
- OSError/SubprocessError (continue, return empty list)
- `handle_wos_start` calls `_toggle_systemd_timers(True)` on both normal and partial-recovery paths
- `handle_wos_stop` calls `_toggle_systemd_timers(False)`
- Idempotency: no timer call when already stopped

The mock pattern for `_SYSTEMD_UNIT_DIR` uses `mock_dir.__truediv__ = lambda self, name: mock_unit_file` — this correctly intercepts the `_SYSTEMD_UNIT_DIR / timer_name` path construction in the implementation. The mock returns a single `mock_unit_file` regardless of which timer name is requested. This means all 7 timers in `_WOS_CORE_TIMER_JOBS` will report as having the same mock unit file — all 7 are enabled/disabled in the test, and the `issue-sweeper` assertion checks that at least one was processed. This is correct test coverage for the function's behavior; the LOBSTER-MANAGED check is tested separately with `timer_is_managed=False`.

**Patterns this introduces:**

The two-frozenset pattern (`_WOS_CORE_JOBS` for jobs.json management, `_WOS_CORE_TIMER_JOBS` for systemd management) creates a maintenance synchronization requirement: if a new timer-backed WOS-core job is added, it must be added to both frozensets independently. The learnings.md "mirror-constant" pattern applies at the structural level, though these are distinct sets at different abstraction layers (jobs.json vs. systemd), not value-for-value mirrors. The divergence risk is low but real — a new timer-backed job added to `_WOS_CORE_JOBS` but omitted from `_WOS_CORE_TIMER_JOBS` would be silently unmanaged. This is documented behavior in the docstring but has no runtime check. Not a blocker for this PR.

**What failure modes exist?**

1. `daemon-reload` is called once per timer in the migration (before each `enable --now`), then again after the loop. This is redundant but harmless — the first per-timer daemon-reload is unnecessary since no unit files are changed, just enabled/disabled. Minor efficiency concern, not a correctness issue.

2. If `_toggle_systemd_timers` is called during a `wos start` that itself fails partway through (e.g., the config write fails after timer enable), the timers will be enabled but `wos-config.json` may report `execution_enabled=false`. The two layers would be out of sync. This is the same class of transactional risk that exists in the current code (jobs.json and wos-config.json are written separately). The fail-soft approach accepts this risk in exchange for resilience; this PR does not worsen the situation.

3. The test for `test_stop_does_not_call_toggle_timers_when_already_stopped` correctly verifies the early-return idempotent path does not call `_toggle_systemd_timers`. This is the right guard against double-disable on repeated `wos stop` calls.

**Overall assessment:** The implementation is correct and well-scoped. The LOBSTER-MANAGED guard is the critical safety control and it is correctly implemented and tested. The fail-soft pattern is appropriate for an infrastructure function called from an operator command. Migration 119 correctly handles the existing-install case with proper gating on `execution_enabled`.

The two noted items (migration counter inflation on failure, migration number skip) are minor enough that they do not warrant a NEEDS_CHANGES round. The core fix — WOS-core systemd timers are now managed by `wos start`/`wos stop` — is implemented correctly.
