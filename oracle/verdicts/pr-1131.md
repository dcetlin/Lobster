### Stage 2: Quality Review

#### Key design constraints verified

**1. vault-watcher.py does detection/debounce ONLY — no mutation, no execution**

Confirmed. `vault-watcher.py` contains no sync, render, git-pull, annotation dispatch, or DB logic. Its sole work is git fetch, state file updates, debounce calculation, and subprocess invocation of vault-processor.py. The boundary is clean.

**2. vault-processor.py is the sole writer — single writer invariant holds**

Confirmed for vault-processor.py itself. `todo_obsidian_sync.py` retains write capability during the 48-hour validation period (by design, per Section 10). The process mutex (`fcntl.flock`) gates concurrent runs. Design Section 11 Q1 is implemented: `todo_obsidian_sync.py` acquires the same lock, extending the mutex to cover both callers during validation.

**3. fcntl.flock mutex prevents concurrent processing runs**

Confirmed. Both scripts use `acquire_lock_or_skip()` from `obsidian_sync_core.py`. The lock is acquired non-blocking; if held, the caller logs and exits 0. The `finally` block in vault-watcher.py's main() ensures release.

**Note:** vault-watcher.py acquires the lock and then invokes vault-processor.py as a subprocess. The subprocess will attempt `acquire_lock_or_skip(LOCK_PATH)` again in its `main()` (for direct invocation path). When invoked from vault-watcher.py, the lock is already held by vault-watcher.py's fd, so vault-processor.py's `acquire_lock_or_skip()` call in `main()` will... return `None` and exit immediately. This is a bug in the direct-invocation guard but NOT a bug in the watcher-invoked path, because `vault-processor.py` is invoked via `run_processor()` from the subprocess, not `main()`. Examining the subprocess invocation in `_invoke_processor()`: it calls `[sys.executable, str(processor_script), "--config", config_path]`, which goes to `main()` — which does call `acquire_lock_or_skip(LOCK_PATH)`. Since vault-watcher.py holds the lock as an open fd, and the lock is held process-wide (fcntl.flock is per-process-per-fd), the subprocess is a different process and will fail to acquire the lock. This means vault-processor.py **always fails to acquire the lock when invoked by vault-watcher.py and exits immediately without processing**. This is a critical bug.

Wait — re-examining. `fcntl.flock` on Linux is per-process, not per-fd. A subprocess is a separate process. So vault-watcher.py holds the lock in its process; vault-processor.py is a *new subprocess*, which is a different process. The new process calls `acquire_lock_or_skip()`, which tries `fcntl.LOCK_EX | fcntl.LOCK_NB` on the same file. Since vault-watcher.py's lock fd is open *in the parent process*, and the child is a *different process*, `flock` in the child will see the lock as held by the parent and return `BlockingIOError`. The processor exits with "skipping: processor already running". **Processing never runs.**

This is the critical failure mode: the watcher holds the lock, invokes the processor subprocess, the processor tries to acquire the same lock and finds it held by the parent, exits immediately. Vault content is never processed. The lock mechanism as implemented makes the watcher-invoked path structurally inoperable.

The design doc (Section 3.2) acknowledges this concern: "vault-processor.py holds /tmp/vault-processor.lock (acquired by vault-watcher.py before invocation; vault-processor.py inherits the open fd when invoked as a subprocess, or re-acquires it if invoked directly)." The intended contract is that the processor *inherits* the fd when invoked as a subprocess. However, `subprocess.run()` does not pass file descriptors by default on Python 3 (`close_fds=True` is the default). The lock fd is **not inherited** unless `close_fds=False` is passed or the fd is explicitly listed in `pass_fds`.

The fix is one of: (a) pass the lock fd to the subprocess via `pass_fds` parameter so the child inherits it and does not re-acquire; (b) have vault-watcher.py release the lock before invoking vault-processor.py (and re-acquire after); (c) have vault-processor.py skip the lock acquisition when invoked by vault-watcher.py (e.g., via a flag). Option (a) is cleanest given the design intent.

Alternatively, vault-watcher.py should release the lock, invoke the processor, then re-acquire. But the design intent is that the lock is held during the full processor run to gate concurrent watcher ticks. The correct fix is `pass_fds=(lock_fd.fileno(),)` in the `subprocess.run()` call, and the processor checks for the inherited fd (or simply always re-acquires, which will succeed because the child inherits the lock context).

Actually — let me reconsider. When `pass_fds` passes the fd to the child, the child calls `acquire_lock_or_skip()` which opens the lock file *again* (via `lock_path.open("w")`), creating a new fd, and tries to flock that new fd. Even with the parent fd inherited, the child's new fd is separate and `flock` will see the parent still holds the lock. The correct design is: vault-watcher.py should invoke vault-processor.py **without holding the lock during the subprocess invocation**, or vault-processor.py needs a `--skip-lock` flag for when invoked by the watcher.

**This is NEEDS_CHANGES. The processor is structurally inoperable when invoked by vault-watcher.py.**

**4. DISABLE PROCESSING flag has exactly 3 states**

Confirmed in the design. The implementation in `check_disable_processing_guard()` correctly handles all three states. However, there is a subtle bug in the implementation: the function first checks the exact regex for checked/unchecked, then falls back to a loose `DISABLE_PROCESSING_TEXT in line.upper()` check with regex detection of `[xX]`. This fallback is not spec-compliant: the spec requires exact format (`- [ ] 🔒 DISABLE PROCESSING` or `- [x] 🔒 DISABLE PROCESSING`). A line like `## DISABLE PROCESSING POLICY` would trigger the fallback and be (incorrectly) treated as a guard. This is a minor correctness issue but does not block merging — the primary spec-compliant path is correct, and the fallback only fires when the primary regexes don't match but the text appears.

### Named gaps requiring resolution

**Gap 1: Lock inheritance failure makes watcher-invoked processor inoperable (critical)**

`vault-watcher.py` acquires `fcntl.flock` on `/tmp/vault-processor.lock`, then invokes `vault-processor.py` as a subprocess. The subprocess is a separate process. `subprocess.run()` defaults to `close_fds=True` — the lock fd is not inherited. The processor calls `acquire_lock_or_skip(LOCK_PATH)`, finds the lock held by the parent process, and exits with "skipping: processor already running". No vault processing occurs on any watcher-triggered invocation.

Required fix: one of (a) release the lock in vault-watcher.py before invoking the processor, then re-acquire after; (b) add a `--already-locked` / `--skip-lock` flag to vault-processor.py for watcher-invoked calls; (c) restructure so vault-watcher.py calls `run_processor()` directly in-process rather than as a subprocess (most straightforward — the subprocess boundary was intended for isolation but creates the lock problem). The tests do not catch this because they test `run_processor()` directly, not via `_invoke_processor()`.

**Gap 2: render_active_todos() behavior change in todo_obsidian_sync.py is undocumented**

The new shared `render_active_todos()` always inserts `"- [ ] 🔒 DISABLE PROCESSING"` in its output. The original `render_active_todos()` in `todo_obsidian_sync.py` did not. This means `todo_obsidian_sync.py` now writes the guard line on every run — a behavior change from the "refactor only, behavior unchanged" claim. This may be the correct and necessary migration behavior (it sets up the invariant vault-processor.py requires). If intentional, it must be documented explicitly (PR description, commit message, or code comment) rather than hidden under "behavior unchanged." If unintentional, the guard line should be conditioned on `last_synced is not None` (vault-processor.py path) to preserve true zero-change behavior for `todo_obsidian_sync.py`.

Resolution options:
(a) Document as intentional migration behavior — add a code comment in `render_active_todos()` explaining that the guard line is always rendered because vault-processor.py requires it, and that `todo_obsidian_sync.py` uses this to set up the invariant on first run.
(b) Make the guard line conditional: render it only when `last_synced is not None` (vault-processor.py path). Add a separate migration step that inserts the guard on first deploy.
(c) Author states why the gap does not apply.
