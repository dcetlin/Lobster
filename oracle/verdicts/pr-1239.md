### Stage 2 — Quality Review

#### Gap 1 (blocking): Four commands in `PRE_HANDLER_COMMANDS` are silently unreachable as slash commands

`pre_handler.py` defines handlers and registers `/subagents`, `/jobs`, `/wos`, `/restart` in `PRE_HANDLER_COMMANDS`. It also imports and exports `handle_help_command`. However, `lobster_bot.py` only registers three `CommandHandler` entries:

```python
bot_app.add_handler(CommandHandler("todos", handle_todos_command))
bot_app.add_handler(CommandHandler("quota", handle_quota_command))
bot_app.add_handler(CommandHandler("status", handle_status_command))
```

The `try_handle()` path is called from `handle_message()`, which is registered under `MessageHandler(filters.TEXT & ~filters.COMMAND, ...)`. The `~filters.COMMAND` filter means Telegram slash commands never reach `handle_message()` — and therefore never reach `try_handle()`.

**Result:** `/subagents`, `/jobs`, `/wos`, `/restart`, and `/help` are in `PRE_HANDLER_COMMANDS` with implementations, but are silently dropped by the bot layer when sent as slash commands. The only path that would have worked — `try_handle()` — is never called for command-type messages.

**Resolution contract:** Add `CommandHandler` registrations for each of the remaining commands alongside the existing three. For `/restart`, also add a handler function reference (it is handled via `try_handle` arguments, so either add a thin `handle_restart_command` wrapper or route through `try_handle` from a `CommandHandler`). The cleanest fix is to add `CommandHandler` entries for all five missing commands in the same block in `lobster_bot.py`.

#### Gap 2 (minor): Private function import from `wos_dashboard`

`_handle_wos()` imports `_get_bisque_relay_base_url` from `orchestration.wos_dashboard`. The leading underscore marks this as a module-private function — it is not part of the public API. The function is used in two places: once in `wos_dashboard.py` itself, and now in `pre_handler.py`.

This creates a coupling risk: `_get_bisque_relay_base_url` can be refactored or removed without the pre-handler receiving any static type warning. The function is already wrapped in a `try/except` with a graceful fallback, which limits the runtime damage — but the import is still structurally brittle.

**Resolution contract:** One of: (a) expose the function as a public API in `wos_dashboard.py` by removing the underscore, or (b) inline the URL construction in `_handle_wos()` directly (the function is likely a config read), or (c) leave as-is with a comment acknowledging the private import and explaining why the try/except makes it safe. Option (c) is acceptable if the function is unlikely to change.

#### Observations

- The authorization pattern (reading `TELEGRAM_ALLOWED_USERS` at module load into `_ALLOWED_USERS`) is correct and avoids the circular-import risk from the earlier draft that imported `ALLOWED_USERS` directly from `lobster_bot`. This is a genuine improvement.
- The `try/except` wrapping around each command in `try_handle()` is appropriate — command failures in the pre-handler should not crash the message loop.
- The callback intercept pattern (checking `cb_data.startswith("todo-")` before the inbox write in `handle_callback_query`) is correct and clean.
- The `/restart mcp` and `/restart all` handlers correctly refuse in-process execution with an informative message pointing to the shell script. This follows the documented MCP restart safety protocol.
- All 5 smoke tests pass. The existing `test_dispatcher_commands.py` suite (28 tests) passes unchanged.
