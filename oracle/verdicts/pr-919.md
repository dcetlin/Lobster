## Round 1 — 2026-04-24

### Stage 1: Vision alignment (formed before reading the implementation)

**Adversarial prior:** This implementation is solving the wrong problem — or solving the right problem in a direction that forecloses better paths.

**What the task claims to serve:** PR #919 adds a session-scoped autouse `isolate_lobster_inbox_dir` fixture at `tests/unit/test_orchestration/conftest.py`. It claims to fix Issue #915: `steward_trigger` messages leaking to the live inbox during pytest runs.

**What would have to be true for this to be the right path:** PR #918 (already merged) added an identically-named, identically-scoped fixture at the top-level `tests/conftest.py`. For PR #919 to be non-redundant, one of these must be true: (a) the top-level fixture does not actually cover `tests/unit/test_orchestration/` tests, (b) the subdirectory-level fixture adds behavior the top-level does not have, or (c) the two fixtures are meant to compose in a way that provides belt-and-suspenders isolation beyond what either provides alone.

**Cheaper test of assumption:** Read the top-level conftest to verify the fixture name, scope, and coverage. If the fixture is session-scoped and autouse=True at the top level, pytest applies it to all tests in the session including tests in `tests/unit/test_orchestration/`. The subdirectory conftest is redundant unless it adds something the top-level does not.

**What this forecloses:** A duplicate session-scoped autouse fixture with the same name produces a fixture conflict in pytest. When pytest collects tests in `tests/unit/test_orchestration/`, both the top-level fixture and the subdirectory fixture are in scope. **Pytest fixture resolution: when two fixtures share the same name, the closer (more local) fixture wins — the subdirectory fixture overrides the top-level fixture.** This is not composition; this is substitution. The subdirectory fixture strips the subdirectory layout creation (`inbox/`, `processing/`, `processed/`, `failed/`) and the `yield`-path restore logic that the top-level version provides, replacing them with a simpler `pytest.MonkeyPatch.context()` approach.

**Who is this optimizing for:** Tests that run only `tests/unit/test_orchestration/` in isolation (i.e., without the full test suite). In that configuration, the top-level conftest is still loaded and the fixture is still applied — pytest always loads conftest.py files for all parent directories. The subdirectory conftest does not add protection for the partial-run case.

**Alignment verdict:** Questioned — the foundational assumption that PR #918's top-level fixture does not cover orchestration tests is incorrect. PR #919 is not additive; it is a naming conflict that substitutes the simpler subdirectory fixture for the richer top-level fixture when tests in `tests/unit/test_orchestration/` are collected.

### Stage 2: Quality review

**Encoded Orientation check:** This is a test-only change. It does not modify behavioral defaults, system constraints, agent identity, or decision-making rules in any durable way. Constraint-3 does not apply.

**Does it do what it claims to do?**

The PR description states: "Per-test `patch.dict` calls in `TestWriteStewardTrigger` take precedence over the session fixture — verified correct." This is accurate — per-test `patch.dict` calls override session fixtures because they run later. However, the PR's framing obscures the critical issue:

**Fixture naming conflict with PR #918:**

Both `tests/conftest.py` (PR #918, merged) and `tests/unit/test_orchestration/conftest.py` (PR #919, proposed) define a fixture named `isolate_lobster_inbox_dir` with `scope="session"` and `autouse=True`.

In pytest, when a fixture name appears in both a parent conftest and a child conftest, the child conftest's fixture overrides (shadows) the parent's. For all tests in `tests/unit/test_orchestration/`, the PR #919 fixture would be the one that runs — not the PR #918 fixture.

The behavioral difference:

- PR #918 top-level: creates subdirectories (`inbox/`, `processing/`, `processed/`, `failed/`), uses manual `os.environ` manipulation with explicit `yield` and restore logic, yields the path.
- PR #919 subdirectory: does not create subdirectories, uses `pytest.MonkeyPatch.context()`, yields nothing (plain `yield`).

The subdirectory fixture is simpler. The top-level fixture is richer. For the orchestration tests, PR #919 would substitute the simpler fixture. This is a regression relative to PR #918 for any orchestration test that depends on the subdirectory structure being present.

**What failure modes exist?**

1. **Fixture shadowing, not composition.** The PR description says this is a "companion fixture," implying both run. They do not both run. The subdirectory fixture shadows the top-level fixture for all tests in `test_orchestration/`. The PR's stated test result (78 tests pass, no leakage) is consistent with this — the subdirectory fixture does prevent leakage — but it does so by replacing the top-level fixture, not adding to it.

2. **Missing subdirectory structure.** `_write_steward_trigger()` calls `inbox_dir.mkdir(parents=True, exist_ok=True)` internally, so no subdirectory layout is needed for the function itself. However, the top-level fixture's explicit subdirectory creation (`inbox/`, `processing/`, `processed/`, `failed/`) documents the expected production layout and provides structure that other tests using `LOBSTER_INBOX_DIR` might depend on. The subdirectory fixture provides only the root temp directory, not the layout.

3. **PR description misrepresents the relationship.** The PR says PR #918 "added a session-scoped autouse fixture in the top-level tests/conftest.py" and this PR "adds a companion fixture at the orchestration subdirectory level" — implying the two coexist. They do not coexist; the subdirectory fixture shadows the top-level fixture for the affected tests.

**What does this make harder?**

- Future maintainers will see two fixtures with the same name and same scope and reasonably assume either (a) there is intentional duplication they do not understand, or (b) one is stale. Neither assumption is accurate. This is a maintenance debt pattern.
- The learnings.md PR #839 entry ("Assertion softening to 'not-legacy-value' cannot distinguish wired-correctly from silently-failed-with-fallback") does not directly apply, but the spirit of the pattern does: the PR's passing test suite does not distinguish "both fixtures active" from "subdirectory fixture substituting for top-level fixture." The tests pass either way.

**What patterns does this introduce?**

The fixture-shadowing pattern, used intentionally or not, introduces a conftest layering approach where subdirectory-level fixtures silently override parent fixtures. If this lands, it establishes a precedent for other packages to define their own `isolate_*` fixtures, each with slightly different behavior, where the override semantics are not visible without reading both files.

**Golden pattern check:** The learnings.md PR #839 entry on "module-level constant computed from env var at import time is not overridable by monkeypatch.setenv" is directly relevant here. PR #919's `_write_steward_trigger` reads `LOBSTER_INBOX_DIR` at call time (not import time), so the monkeypatch approach works correctly — this is not the bug. But the fixture layering creates a different subtle problem: maintainers may read the test isolation story as "two layers of protection" when it is actually "subdirectory fixture displaces top-level fixture."

### Verdict findings

**The core issue:** PR #919 does not add to PR #918 — it shadows it for the orchestration test package. The PR description presents this as complementary layering when it is actually fixture substitution via pytest's name-resolution semantics.

**Revision contract:**

The PR must be revised in one of three ways:

1. **Remove PR #919 entirely** — if PR #918's top-level fixture is sufficient (which it is: pytest applies session-scoped autouse fixtures from parent conftests to all child tests), then PR #919 serves no purpose and adds confusion.

2. **Rename the subdirectory fixture** — if there is a genuine reason to have orchestration-specific inbox isolation behavior that differs from the top-level behavior, give it a distinct name (e.g., `isolate_orchestration_inbox_dir`) and document explicitly why the two coexist and what each protects.

3. **Document the override intentionally** — if the intent is to substitute the simpler fixture for the orchestration tests specifically (e.g., because the subdirectory layout is unneeded there), document this substitution explicitly in the fixture docstring and PR description. The current docstring says nothing about overriding the top-level fixture.

Option 1 is the preferred resolution: PR #918 already provides correct coverage. Principle-1 from vision.yaml (structural prevention over reactive recovery) and principle-2 (keep the correction path open over forms that produce faster output) both argue for removing the shadowing layer rather than carrying the ambiguity forward.

**Named gaps:**

- **Gap 1: Fixture shadowing misrepresented as composition.** The PR description and docstring present PR #919 as a "companion fixture" to PR #918. In pytest semantics, a same-named fixture in a child conftest overrides, not accompanies, the parent fixture. The PR is addressed when either (a) the fixture is removed as redundant, (b) it is given a distinct name with explicit rationale for coexistence, or (c) the docstring explicitly states "this fixture overrides `isolate_lobster_inbox_dir` from tests/conftest.py for orchestration tests" with the reason for the behavioral difference.

- **Gap 2: Subdirectory fixture drops structural layout from top-level.** PR #918's top-level fixture creates `inbox/`, `processing/`, `processed/`, `failed/` subdirectories. PR #919's fixture provides only the root temp directory. If any orchestration test (current or future) reads from `LOBSTER_INBOX_DIR/inbox/` (the production message path, not just `LOBSTER_INBOX_DIR/`), the subdirectory fixture is insufficient. The PR is addressed when either the subdirectory fixture is removed (Gap 1 resolution) or the subdirectory fixture creates the same layout as the top-level fixture.
