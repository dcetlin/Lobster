## Stage 1: Vision alignment (formed before reading implementation)

The adversarial prior entering this review: WOS parallelism may be the wrong investment vector at this moment. The system currently suffers from observability and reliability gaps (vision.yaml current_focus names PRs #891-#898 closing in the prior week). Lifting the serial execution ceiling assumes execution throughput is the current bottleneck — but the active constraint in vision.yaml is "oracle/verdict completeness" and "callback handlers not yet dispatched." Adding parallelism before those structural gaps close could generate harder-to-diagnose interference between concurrent UoWs in an already-fragile pipeline.

The disconfirming evidence I was looking for: is the serial constraint itself causing visible system failures, or is it a preemptive optimization? And does the PR carry Encoded Orientation risk (constraint-3) since it changes dispatch behavioral defaults?

**What had to be true for this to be the right path:** The serial constraint must be producing real starvation (one long-running UoW blocking ready work), the file_scope annotation infrastructure must already exist or be cheap to populate, and the implementation must preserve serial behavior as the default so unnannotated UoWs see no behavior change. If any of these were false, the work is premature optimization at the wrong moment.

**Encoded Orientation check:** `max_parallel: 2` in `_DEFAULT_WOS_CONFIG` and the four-gate dispatch decision are Encoded Orientation decisions — they change the default behavioral rules for dispatch. The question is whether a prior logged decision exists and whether there is a vision.yaml anchor. Dan explicitly approved this work per the task prompt. The vision.yaml active_project.current_phase is "WOS Phase 1 + Vision Object substrate," with throughput improvements as a natural progression from #891-#898's reliability work. The behavioral change is conservative by design (null scope = exclusive, default = serial). This is sufficient for Encoded Orientation under constraint-3, though the logged decision is verbal approval rather than a formal decisions.md entry.

**Alignment verdict:** Confirmed — with the observation that the file_scope annotation infrastructure is not yet populated by any automated producer (cultivator or steward do not yet write file_scope), making parallelism opt-in by annotation only. This is the correct design: default-serial behavior is preserved, and the feature activates only when a human or automated producer explicitly annotates a UoW. The PR correctly identifies this as the right scope.

## Stage 2: Quality review

### Encoding Orientation check

PR modifies `dispatcher_handlers.py` `_DEFAULT_WOS_CONFIG` (adds `max_parallel: 2`) and `steward.py` `run_steward_cycle` (adds dispatch gate). These are behavioral default changes. The gate is conservative: existing UoWs with null file_scope preserve serial behavior exactly. Dan's explicit authorization per the task prompt and the vision.yaml WOS Phase 1 anchor together satisfy constraint-3. No prior decisions.md entry exists for this specific decision, which is a gap in traceability but not a blocking issue given explicit owner authorization.

### Does it do what it claims?

Yes. The four-gate logic is correctly implemented as pure functions in `shard_dispatch.py`. Gate priority (first failure wins) is correctly encoded and verified by tests. The within-cycle accumulation (`_executing_uows = list(_executing_uows) + [uow]`) correctly prevents over-dispatch within a single heartbeat. Backward compatibility for unnannotated UoWs (null file_scope = exclusive, serial behavior preserved) is correctly implemented.

### Specific concerns from the implementer

**1. Gate fires before _process_uow — Done/Surfaced deferred by one cycle when max_parallel full**

The comment in steward.py notes this correctly: at most one extra heartbeat delay (3 minutes) for completion acknowledgment when the shard gate fires before success-criteria evaluation. This is acceptable. The Done/Surfaced paths are idempotent — the extra cycle costs latency, not correctness. The PR comment is accurate about the tradeoff.

**2. `_scopes_overlap()` prefix matching semantics**

The implementation appends "/" before the `startswith` check: `path_b.startswith(path_a + "/")`. This means "src/orches" does NOT overlap "src/orchestration/steward.py" — only full path segment boundaries are matched. The test at line 728-734 explicitly verifies this. The semantics are correct for filesystem paths. One edge case not tested: what if `file_scope` contains an empty string? `"".startswith("/" )` is False, and `_normalize("")` returns `""`. A UoW with `file_scope: [""]` would overlap nothing via prefix but would match another `""`. This is a degenerate input that validation at the cultivator level should prevent, but the function has no guard.

**3. `list_executing()` includes ready-for-executor — conservative, is it correct?**

Including `ready-for-executor` in the in-flight set is correct and conservative. A UoW in `ready-for-executor` has been prescribed and is waiting for the executor to claim it — its file_scope resources are logically locked even before execution begins. Excluding it would create a window where two UoWs with overlapping scopes could both reach `ready-for-executor` before either transitions to `active`. The conservative choice is correct here.

### What would break without the key decisions?

- Without the within-cycle `_executing_uows` accumulation: a single steward heartbeat with max_parallel=2 and 5 annotated non-overlapping UoWs could prescribe all 5, exceeding the cap. The accumulation is the correct fix.
- Without the `ready-for-executor` inclusion in `list_executing()`: overlap conflicts would slip through the window between prescription and executor claim.
- Without null-scope exclusivity: annotated UoWs could run concurrently with unannotated UoWs that might touch any file.

### Failure modes

1. **Migration comment number mismatch (advisory):** The file is correctly named `0012_add_shard_stream_fields.sql` but the internal comment at line 152 says `-- Migration 0011`. This is a documentation error, not a schema error — the migration runs correctly regardless. Advisory only, not blocking.

2. **`src/ooda/fast_thorough_selector.py` bundled without description:** The PR description acknowledges this as a cherry-pick to make the build importable. The file is a clean, standalone module with no side effects at import time. It is not wired to any caller in this PR. This is the "library with no wiring" pattern from learnings.md PR #714, but here it is explicitly acknowledged as a dependency-resolution cherry-pick, not a feature claim. The module is structurally sound. Advisory only.

3. **`read_max_parallel()` swallows all exceptions silently:** `except Exception: pass` with return DEFAULT_MAX_PARALLEL means any misconfiguration (including a wos-config.json with `max_parallel: -1`) falls back to the default silently. The positive guard `isinstance(value, int) and value > 0` correctly rejects invalid values, so the fallback is safe. The silent exception path does not produce a visible warning. Low severity — the fallback is the correct behavior — but a `log.warning` on unexpected exceptions would aid debugging.

4. **`file_scope` annotation population gap:** No current producer (cultivator, steward, germinator) writes `file_scope` at UoW creation time. The parallelism feature is complete but produces no benefit until annotation infrastructure is added. This is the correct scoping decision (PR description acknowledges it), but it means the production impact of this PR is zero until a follow-on PR wires annotation.

### Tests

34 new tests for `shard_dispatch.py` — comprehensive coverage of all four gates, gate priority ordering, and within-cycle accumulation. Test design is correct: tests use named constants from production code (not magic literals), use the `_StubUoW` minimal duck-typed stub, and verify both allow and block cases for each gate. The `test_max_parallel_zero_always_blocks` edge case test is a valuable addition. No assertion softening observed.

The `test_shard_dispatch.py` file does NOT import `_scopes_overlap` via the public API — it imports the private function directly for unit testing. This is acceptable for a pure function that has no reason to be private. If the function is ever refactored, the test import will break loudly (not silently), which is the correct failure mode.

### Encoded Orientation check (OODA constraint-3)

The `max_parallel: 2` default in `_DEFAULT_WOS_CONFIG` changes the system's behavioral default for dispatch. Dan explicitly authorized this work. The vision.yaml anchor is `active_project.current_phase: "WOS Phase 1 + Vision Object substrate"` and `principle-4` (wire what exists before building more — this wires the dispatch constraint that was previously implicit). The prior decision is Dan's explicit approval per task prompt. This satisfies constraint-3 at the substance level. A formal `vision.yaml open_decisions` entry for the parallelism design would strengthen traceability but is not blocking given direct owner authorization.

### Patterns introduced

- Pure-function gate module with typed `DispatchAllowed | DispatchBlocked` result type — extends the classification-result-as-typed-dataclass golden pattern to dispatch decisions. High reuse value.
- Within-cycle accumulation pattern for in-flight tracking — correct solution to the over-dispatch-in-one-cycle problem.
- `list_executing()` as a named method with explicit semantic documentation — naming intent at the call site without requiring the reader to reconstruct status-set semantics.

### What this forecloses

Nothing material. The null-scope exclusivity default preserves backward compatibility completely. The `max_parallel` cap is runtime-configurable via wos-config.json without a code change.

### What this makes harder

Annotating `file_scope` on the full UoW backlog (77+ proposed UoWs) will require a sweep or manual annotation pass. The PR creates the obligation without the tooling to fulfill it. This is acceptable scoping but should be tracked as a follow-on.
