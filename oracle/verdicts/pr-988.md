### Does it do what it claims?

Yes. `extract_friction_trace()` correctly extracts the friction-trace body from the markdown format described in `OUTPUT_FORMAT.md`. The regex handles multiline content, embedded `**bold**` markers, and embedded `*italic*` markers within the trace body without truncation. ### Regex correctness

The pattern `r'\*friction-trace:\s*((?:[^*]|\*(?!\s*(?:\n|\Z)))*)\*'` with `re.DOTALL`:

- Opening `\*friction-trace:\s*` is unambiguous.
- Body `(?:[^*]|\*(?!\s*(?:\n|\Z)))*` allows any non-asterisk or any asterisk not immediately before optional whitespace + newline/end. This correctly permits embedded `*exact form*` (the closing `*` is followed by `.`, not whitespace + newline) and `**bold**` markers.
- Closing `\*` matches the first `*` that IS before optional whitespace + newline/end.

Verified against `SAMPLE_WITH_FRICTION_TRACE`: the closing `*` of the friction-trace block is followed by `\n\n`, so the body terminates there correctly without consuming the subsequent `*orientation quality...*` block.

### Minor observations (non-blocking)

1. **`orientation_quality` field is never populated by the extractor.** The field is declared on `FrictionTrace` with `None` default, and `SAMPLE_WITH_FRICTION_TRACE` includes an `*orientation quality:...* ` block in the fixture text ŌĆö but `extract_friction_trace` only extracts `text`. The field is dead on the extraction path. It has a safe default, gates no behavior, and three dataclass tests exercise it directly. This is acceptable as a future-reserved field. A docstring comment noting "not currently populated by extract_friction_trace; reserved for future multi-section parsing" would prevent confusion for the next engineer who sees the fixture and wonders why it is not extracted. Advisory only.

2. **`navigation_record` is a new unregistered type string.** The existing `MemoryObservation` type field accepts arbitrary strings with no enum constraint. This is consistent with current system behavior ŌĆö this PR does not introduce a new vulnerability, it is operating within an existing permissive type system. Advisory.

3. **`friction_traces_stored` counter is always 0 or 1 per harvest call** given the single-match design. The plural name is a minor misnomer but harmless given the existing architecture.
