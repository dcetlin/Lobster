## Stage 1: Vision alignment

**Prior from learnings.md:** No directly matching failure patterns. The shared-utility extraction pattern was established as golden in PR #805 — this PR extends it. The "right-place, right-word, right-register" design principle from CLAUDE.md directly governs this refactoring.

**Theory of change:** Two heartbeat scripts contain identical 50-line `_write_crash_alert` implementations. The module for shared inbox-write utilities (`src/utils/inbox_write`) already exists and already owns this behavioral territory. The PR moves the function to where it belongs.

**Cheaper test of the assumption:** None needed. The assumption (duplicated code should be deduplicated) is structurally sound without validation.

**What this forecloses:** Nothing. Future heartbeat scripts can now import `write_crash_alert` rather than copy-pasting it.

**Opportunity cost:** Minimal. This is a small, targeted refactoring that completes a gap explicitly named in the PR #1222 oracle verdict.

**Alignment verdict:** Confirmed. This is precisely the refactoring the shared-utility pattern calls for.

## Stage 2: Quality review

**Does it do what it claims?** Yes. The diff shows: (a) 58–60 lines removed from each heartbeat script (the local `_write_crash_alert` definition and `_ADMIN_CHAT_ID` constant), (b) `from src.utils.inbox_write import write_crash_alert` added, (c) call site updated from `_write_crash_alert(...)` to `write_crash_alert(...)`, (d) 45 lines added to `inbox_write.py` implementing the public function.

**Implementation quality:**
- The centralized implementation is strictly better than both local copies:
  - Uses `_inbox_dir()` helper (consistent with rest of module) rather than the inline path construction in the originals.
  - Uses module-level `log = logging.getLogger(__name__)` rather than the delayed-import `from datetime import datetime as _datetime` pattern in `executor-heartbeat.py`.
  - `admin_chat_id` is read from env at call time (correct — env can change without restarting the module).
- Module docstring is updated to document the new public function.
- Signature is identical to both originals: `(job_name: str, exc: BaseException, extra: str = "") -> None`.

**Test coverage observation:** `tests/unit/test_inbox_write.py` does not contain direct tests for `write_crash_alert`. The existing tests cover `write_inbox_message`, `_inbox_dir`, and `_task_outputs_dir`. The PR description cites 33 tests passing in `test_executor_heartbeat.py` — these exercise the call site but not the function itself. This is a minor gap. The function's behavior is simple (build a dict, write a JSON file), the logic is identical to what was already deployed in production without unit tests, and the module-level function is structurally simpler than the originals. This gap is acceptable given the refactoring context but noted for future improvement.

**OODA constraint-3 check:** This PR does not constitute an Encoded Orientation decision. It makes no behavioral change — it reorganizes existing behavior. No prior logged decision or vision.yaml anchor is required.

**Patterns introduced:** The shared-utility import pattern (`from src.utils.inbox_write import ...`) now covers crash-alert delivery, not just `write_inbox_message`. This is the correct direction.

**What this makes easier:** Any future heartbeat script can import `write_crash_alert` directly. The crash-alert behavior has a single definition to maintain.

**What this makes harder:** Nothing.

**Failure modes:** None identified beyond the pre-existing best-effort nature of the function (write failures are logged, not raised).
