## Stage 2: Quality Review

### What the PR does

Three changes:
1. `dispatcher_handlers.py`: adds `uow_id: {uow_id}` to WOS executor prompt frontmatter — one line, clean.
2. `token-ledger-collect.sh`: inline Python in a tmpfile extracts `uow_id` from frontmatter YAML, writes it to the ledger entry conditionally.
3. `wos_uow_detail_gen.py`: `_fetch_token_data` prefers `entry.get("uow_id")` exact match; falls back to `_UOW_PATTERN` regex for historical entries.

### Gap 1: Mirror-constant divergence risk in test file (NEEDS_CHANGES)

The test file declares at module scope:
```python
SONNET_4_6_INPUT_PER_MTK = 3.0
SONNET_4_6_OUTPUT_PER_MTK = 15.0
SONNET_4_6_CACHE_READ_PER_MTK = 0.30
```

The production file `wos_uow_detail_gen.py` declares the identical constants at module scope (lines 48-50). The test assertions in `TestEstimateCost` use the test-file-local constants. When Anthropic adjusts pricing and the production constants change, the test silently validates against stale values — the test continues to pass while the production cost estimates are wrong.

This is the **exact pattern flagged in learnings.md PR #1116**: "Mirror-constant in same-PR new module: when a new module and its test file are both introduced in the same PR, the test file can re-declare the production module's named constants locally rather than importing them, creating a mirror-constant divergence risk from day one."

PR #1116 named `wos_uow_detail_gen.py` explicitly. The constants were apparently not fixed in that PR's scope — but the learnings entry applies here as a first-pass check.

**Required fix:** Replace the test-file constant declarations with imports from the production module:
```python
from src.orchestration.wos_uow_detail_gen import (
    SONNET_4_6_INPUT_PER_MTK,
    SONNET_4_6_OUTPUT_PER_MTK,
    SONNET_4_6_CACHE_READ_PER_MTK,
)
```

### Gap 2: New exact-match code path has zero test coverage (NEEDS_CHANGES)

The primary purpose of this PR is to improve join accuracy by using the explicit `uow_id` field. But `_build_ledger_entries` in the test file only creates entries with `task_id` fields — no `uow_id` field in any entry. All tests therefore exercise the regex fallback path. The new `entry.get("uow_id")` branch is untested.

The claim "321 token tests pass" is true, but none of those tests exercise the new code path the PR is adding. A test must exist that:
- Creates a ledger entry with an explicit `uow_id` field (no `task_id` or with a mismatching `task_id`)
- Calls `_fetch_token_data` with the matching uow_id
- Asserts the entry is included (exact match path fires)

And a counterpart:
- Creates an entry with `uow_id` set to a different UoW's ID
- Asserts the entry is excluded (exact match path correctly rejects)

Without these, the exact-match branch could be deleted or broken and no test would fail.

**Required fix:** Add tests for the explicit `uow_id` field path to `TestFetchTokenData`.

### Gap 3: Bare `python3` in new shell code (advisory, not blocking)

Line 256 of `token-ledger-collect.sh` uses bare `python3 "${TMPPY3}"`. The CLAUDE.md convention is "Always use `uv`." The existing line 227 uses the same bare `python3` pattern (pre-existing), and line 156 uses `uv run python3`. The new code follows the wrong existing pattern.

This is the pattern flagged in learnings.md PRs #804, #821, and #830 ("bare `python3` in bash scripts" — third occurrence labeled a pattern). The failure mode is silent: if `python3` is not in PATH because uv manages the Python installation, the tmpfile produces empty output, `UOW_ID_VAL` is empty, and all ledger entries silently fall back to the regex path — the feature is inert with no error signal.

However, since line 227 (pre-existing, same pattern) has the same vulnerability, and the script does run in an environment where python3 is available (the existing feature works), this is advisory rather than blocking. But it should be fixed alongside the pre-existing violation.

**Advisory fix:** Change `python3 "${TMPPY3}"` to `uv run python3 "${TMPPY3}"` consistent with line 156. The pre-existing line 227 should be fixed in the same commit.

### What works well

The implementation logic in `wos_uow_detail_gen.py` is correct: explicit `uow_id` takes priority, fallback to regex for historical entries, structurally additive. The `dispatcher_handlers.py` change is minimal and clean. The jq conditional object merge `+ (if $uow_id != "" then {"uow_id": $uow_id} else {} end)` correctly omits the field for non-WOS entries rather than writing an empty string.

The backward-compat design (fallback path retained, new field additive-only) is the right approach.

The behavioral outcome (tokens attributed to UoWs) already existed; this improves accuracy of an existing mechanism. Not classified as Encoded Orientation. No constraint-3 gap.
