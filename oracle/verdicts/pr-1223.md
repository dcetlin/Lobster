### Stage 1 — Vision alignment (formed before seeing implementation)

The theory of change in vision.yaml is reducing metabolic cost of engagement (constraint-4) while keeping decisions traceable to structural anchors (constraint-3). The WOS PR coordinator addresses a real problem: 6+ sequential dispatcher round-trips per PR during WOS drain bursts causes context compaction and fragments session state. Internalizing the oracle→fix→merge loop into a coordinator is a legitimate architectural move — one that applies principle-4 ("wire what exists before building more") by composing existing agents behind a new coordination seam.

What would have to be true for this to be the wrong path: if the dispatcher compaction problem during WOS drains is not caused by round-trip count but by session token volume, a coordinator that makes the same oracle/fix/merge calls does not reduce compaction — it moves the compaction from the dispatcher session to the coordinator session. This would be solving the wrong problem (wrong level of abstraction). The PR description does not address this: it asserts "36–48 dispatcher reads in a single session" but does not show that round-trip count, not token volume, is the binding constraint.

This is a real question but not sufficient to override the architectural clarity of the move. The coordinator seam is well-placed regardless of whether it fully solves the compaction issue — it is the correct abstraction at the correct boundary.

What this forecloses: future per-PR routing decisions that require dispatcher context are now harder to add because the dispatcher hands off immediately. This is acceptable — the coordinator prompt receives `task_context` from the result text.

**Alignment verdict: Confirmed.** The work is solving the right problem. The coordinator pattern is structurally correct for the stated goal.

### Stage 2 — Quality review

#### The blocking defect: coordinator is never selected

The decision record authorizes routing WOS PR results to the `wos-pr-coordinator` agent. The implementation does not route to the coordinator. `route_wos_pr_result()` returns:

```python
return {
    "action": "spawn_subagent",
    "task_id": coordinator_task_id,
    "prompt": prompt,
    "agent_type": "lobster-generalist",
}
```

The dispatcher then does:

```python
Task(
    subagent_type=wos_routing["agent_type"],
    ...
)
```

`subagent_type="lobster-generalist"` loads `.claude/agents/lobster-generalist.md`. The dedicated coordinator definition at `.claude/agents/wos-pr-coordinator.md` (`name: wos-pr-coordinator`) is never selected.

All other agent definitions in `.claude/agents/` use their `name:` field as the `subagent_type` value. `session-note-polish` → `subagent_type="session-note-polish"`. `nightly-consolidation` → `subagent_type="nightly-consolidation"`. `review` → `subagent_type="review"`. The coordinator breaks this pattern by using `"lobster-generalist"` instead of `"wos-pr-coordinator"`.

The unit tests do not test for `agent_type == "wos-pr-coordinator"` — they test for `agent_type == "lobster-generalist"` (`assert result["agent_type"] == "lobster-generalist"`), which locks in the incorrect behavior.

The `lobster-generalist` agent spawned in place of the coordinator will receive a prompt saying "Follow the wos-pr-coordinator agent definition at .claude/agents/wos-pr-coordinator.md exactly." It may follow those instructions, but:

- The coordinator's YAML frontmatter model selection (`model: claude-sonnet-4-6`) is not applied — the generalist's model selection governs.
- Session store logs the agent as `lobster-generalist`, not `wos-pr-coordinator` — sessions are not discoverable by coordinator identity.
- The pattern named in learnings.md (PR #714): "any production routing path that requires a new caller to be added is not effective until the caller is added." The coordinator definition is orphaned.

**Fix required:** Change `"agent_type": "lobster-generalist"` to `"agent_type": "wos-pr-coordinator"` in `route_wos_pr_result()`. Update `test_agent_type_is_lobster_generalist` to assert `result["agent_type"] == "wos-pr-coordinator"`.
