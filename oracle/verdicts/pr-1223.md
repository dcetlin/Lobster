### Stage 1: Vision Alignment (locked before implementation review)

**Adversarial prior:** This implementation is solving the wrong problem, or solving the right problem in a direction that forecloses better paths.

**Theory of change in vision.yaml:** Build a substrate where agents make intent-anchored decisions and the system routes work without requiring Dan's attention on every step. Active phase: WOS executor live, active constraints are the WOS bidirectional interface (#1118) and completion notification layer. `principle-4` ("Wire what exists before building more") and `principle-3` ("Determinism over judgment for conditionals — if-then logic is code, not LLM instructions") are directly relevant.

**What would have to be true for this to be the right path?** The 36–48 dispatcher reads per WOS burst is a real and measured problem. Context compaction from those reads causes routing errors, not just latency. The cost of a new coordinator agent (complexity, new failure mode surface) is less than the cost of the compaction problem.

**Adversarial test:** Is the compaction-from-dispatch-volume problem measured or hypothesized? The PR description says "During WOS drain bursts (6+ UoWs), the current per-stage dispatcher round-trip pattern generates 36–48 dispatcher reads in a single session, causing repeated context compaction." The causal claim (reads cause compaction which causes errors) is asserted, not evidenced. The opportunity cost of the active named priorities (bidirectional interface #1118, completion notification layer) is not assessed.

**What does this foreclose?** The coordinator internalizes oracle→fix→merge behind a new agent boundary. This makes the oracle verdict trail less observable to the dispatcher: the coordinator runs the oracle loop internally and the dispatcher only sees the final write_result. It also adds a new long-lived background agent as a structural dependency for WOS PRs. If the coordinator stalls or fails, the PR is stuck with no user-visible signal until the coordinator's escalation path fires (round 4+).

**Is there a cheaper test?** Yes: measure whether compaction actually causes routing errors in the WOS PR path before introducing a new agent tier. The OODA Orient step here is underdeveloped — the PR jumps to a structural solution without first verifying the symptom is the root cause.

**However:** The implementation is directionally sound. The `route_wos_pr_result` routing pattern is consistent with `route_wos_message` (the established compaction-resilient dispatch pattern). The coordinator eliminates N round-trips in the right way — by delegating to a background agent rather than blocking the dispatcher. The direction is not wrong; the Stage 1 concern is about whether the prior for making this a default routing behavior exists.

**Alignment verdict:** Questioned — not Misaligned. The implementation direction is sound and aligns with `principle-3` (encode in code, not prose) and the existing `route_wos_message` golden pattern. The question is whether this Encoded Orientation change has a logged prior (constraint-3). That is the gate, not the direction.

### Stage 2: Quality Review

**Encoded Orientation check (OODA constraint-3):**

This PR makes an Encoded Orientation decision: it changes the default routing behavior for all WOS-originated PR results from "spawn review agent" to "spawn coordinator agent which owns oracle→fix→merge internally." This is a durable behavioral default change — once merged, every WOS subagent that returns a PR URL will be routed to the coordinator rather than the review agent.

Constraint-3 requires: (a) a prior logged decision exists in `oracle/verdicts/` or `oracle/decisions.md`, and (b) the change is traceable to a `vision.yaml` anchor.

**Check result:** No prior logged decision for this behavioral default was found in `oracle/verdicts/`, `oracle/decisions.md`, or `oracle/decisions/`. The PR description cites `WOS-UoW: uow_20260516_71b777` as authorization. Per learnings.md (2026-05-01, PR #1032; 2026-04-24, PR #913): "WOS UoW artifacts are dispatch records, not decision records." This is the third or fourth recurrence of this exact pattern — WOS UoW cited as authorization for an Encoded Orientation decision. The gap fires on the same class of evidence that has been rejected in previous reviews.

**Gap 1 (BLOCKING — Encoded Orientation without logged prior):** `route_wos_pr_result` changes the durable routing default for all WOS PR results. No logged decision exists in `oracle/decisions.md` or `oracle/verdicts/` authorizing this behavioral change. The UoW artifact `uow_20260516_71b777` is dispatch provenance, not authorization. Constraint-3 is not satisfied.

**Resolution criteria:** File a decision record in `oracle/decisions.md` (or `oracle/decisions/decision-wos-pr-coordinator.md`) that: (a) names the behavioral default being changed, (b) cites `vision.yaml` by field path (suggested anchor: `core.operating_principles.principle-3`, `core.operating_principles.principle-4`, and `active_project.phase_intent`), (c) states the specific routing change as a durable default, and (d) is not itself a WOS UoW artifact. The oracle verdict for this PR serves as the structural record once APPROVED; the decision file is the explicit logged prior that constraint-3 requires.

**Gap 2 (ADVISORY — no unit tests for `route_wos_pr_result`):**

The PR description states "route_wos_pr_result() manual test: WOS task_id routes to coordinator, non-WOS falls through, None falls through — all pass." There are zero unit tests in `tests/` for `route_wos_pr_result`. The function is a pure function with no I/O, making it trivially testable. The PR touches `tests/unit/test_orchestration/test_dispatcher_commands.py` (130 tests) but adds no test class for the new function. Per learnings.md PR #714: "When a PR's scope is a new routing module, verify that at least one production call site (not test file) imports from it before treating the routing capability as live." More directly: a pure routing function with no tests in a codebase that has test infrastructure for this module is a regression surface with no CI guard.

**Resolution criteria:** Add a `TestRouteWosPrResult` test class in `tests/unit/test_orchestration/test_dispatcher_commands.py` covering: (a) WOS task_id → spawn_subagent action, (b) non-WOS task_id → fallthrough, (c) None task_id → fallthrough, (d) malformed PR URL → fallthrough with warning. All four cases are stated in the PR description as manually tested; encoding them as assertions closes the gap and prevents regression.

**Other findings (non-blocking):**

**Coordinator oracle spawning**: The coordinator spawns oracle via `subagent_type=lobster-oracle`, which is a valid agent definition in `.claude/agents/lobster-oracle.md`. The merge agent is spawned via `subagent_type="lobster-generalist"`, which is also valid. Both subagent types exist. The hook `hooks/pr-merge-gate.py` will fire on the merge agent's `gh pr merge` call and gate on the verdict file — this is correct and means the PR merge gate hook remains the structural enforcement mechanism even inside the coordinator loop.

**Verdict file archival timing**: The coordinator archives `oracle/verdicts/pr-{N}.md` to `oracle/verdicts/archive/pr-{N}.md` after merge. This is structurally correct. The `archive/` directory already exists (confirmed in PR description). The PR merge gate hook only needs the verdict file to exist before `gh pr merge` is called, not after — the archival ordering is sound.

**task_id deduplication**: The coordinator uses `coordinator_task_id = f"wos-pr-coord-{pr_number}"` as the task_id. Two concurrent WOS subagents that both open PRs with the same number would produce a coordinator_task_id collision. At current scales this is unlikely, but it is worth noting that the deduplication check uses the coordinator_task_id rather than the PR URL itself. Not a blocking issue at current scales.

**Fallthrough correctness**: The `route_wos_pr_result` function returns `{"action": "fallthrough"}` for non-WOS task_ids. The dispatcher correctly falls through to the existing review agent path without any state mutation. The existing dedup check (`reviewer_task_id = f"review-{msg.get('task_id', 'unknown')}"`) correctly handles non-WOS PRs. The fallthrough path is structurally unchanged.

**Import placement**: `from src.orchestration.dispatcher_handlers import route_wos_pr_result` is placed inside the `if pr_url_match:` block in the bootup doc pseudocode. This is a lazy import inside a conditional — consistent with the dispatcher's established pattern for cross-module imports to avoid circular dependencies. Sound.
