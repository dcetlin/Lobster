### Finding: Mirror-constant pattern in test file

**Pattern citation (behavioral effect):** learnings.md entries for PRs #696, #800, #967, #970, #973, #974, #980 name the mirror-constant pattern as a first-pass oracle check. Because of this, the constant below was the first thing checked in the test file before reading the assertions. Without the named pattern, this would have been read as an informational constant.

In `tests/unit/test_orchestration/test_dispatcher_commands.py`, lines under the "Constants matching the spec requirements" comment:

```python
# Stale threshold in hours: data older than this is treated as unavailable.
# Matches the poller's 30-minute schedule — 2 hours gives headroom for gaps.
QUOTA_STALE_THRESHOLD_HOURS = 2
```

The production module `dispatcher_handlers.py` defines `QUOTA_STALE_THRESHOLD_HOURS: int = 2` at module scope. The test re-declares this constant locally rather than importing it.

**Severity assessment:** The constant is never used in any assertion or fixture branching. The `stale_state` fixture hardcodes `timedelta(hours=3)` directly. This is the "dead test constant" sub-case from learnings.md PR #970: the constant is not currently diverging from production because it's not used, but its presence signals to future authors that it can be used for test branching — and doing so would create a live divergence risk with no tooling enforcement.

Per learnings.md PR #970: "Dead constants in test files should be removed; if documentation of related values is needed, use a code comment that names the enum and its mapping function."

**Required fix:** Remove the local `QUOTA_STALE_THRESHOLD_HOURS = 2` declaration from the test file's constants section. If the stale boundary needs to be named in the test, replace it with an import:

```python
from src.orchestration.dispatcher_handlers import QUOTA_STALE_THRESHOLD_HOURS
```
