**Depth-cap enforcement in `handle_todo_add`:** Checks `parent_item.parent_id is not None` to prevent 3-level nesting. Correct.

**Regex for `--parent` flag:** `r"^/todo\s+add\s+(.+?)(?:\s+--parent\s+(\d+))?\s*$"` with non-greedy `(.+?)` ensures the text group captures up to (but not including) the `--parent` suffix. Correct.

**`render_active_todos` SQL:** `WHERE parent_id IS NULL AND (status = 'open' OR (status = 'snoozed' AND snoozed_until < datetime('now')))` is correct and NOT the tautological OR pattern from PR #1128 learnings — `snoozed` is not in the `status = 'open'` branch, so the snoozed branch is not subsumed.

**Structural fallback for new parents:** When a parent item has no `<!-- id:N -->` comment (first-time Obsidian sync for a brand-new item), `current_parent_item_id` is `None`. Subtasks typed under it via structural indent would be inserted as top-level items on the first sync. On the next render cycle, the parent gets `<!-- id:N -->` and subsequent syncs work correctly. This is an accepted first-cycle limitation, consistent with the PR's design: "2-space indent is the visual signal in Obsidian; parent:N is the machine signal."

**Workstream annotation removal (unlisted behavioral change):** The new `_append_item_with_subtasks` helper renders lines as `- [ ] text <!-- id:N -->` with no workstream annotation. The annotation is still parsed (stripped) by `_ANNOTATION_RE` in `_parse_item_text`, so round-trips from files still containing old annotations work. This change is intentional (annotations are replaced by HTML id comments as the machine signal), but it is not described in the PR. This is a minor unlisted scope item, not a blocker.

### Gap — Assertion-softening via OR in `test_parse_done_subtask_has_parent_id`

**Gap 1: Tautological OR branch accepts item's own id as parent_id**

In `tests/unit/los/test_subtasks.py`, `test_parse_done_subtask_has_parent_id` asserts:

```python
assert done_sub.parent_id == 44 or done_sub.parent_id == 42  # parent:42 from comment
```

The fixture line is `- [x] Old subtask done <!-- id:44 parent:42 -->`. The `_parse_item_text` function extracts `parent_id` from the `parent:` group in the HTML comment — which is `42`. The item's own id is `44`. The correct, specific assertion is `assert done_sub.parent_id == 42`.

The `or done_sub.parent_id == 44` arm accepts 44, which would mean "the item's own id was incorrectly returned as its parent." This is the assertion-softening via OR anti-pattern from learnings.md PR #1126: an OR branch that is satisfied by both correct AND incorrect outputs makes the assertion unfalsifiable for the incorrect case. The comment on the same line (`# parent:42 from comment`) names the correct expected value — the `or 44` branch is inconsistent with the comment and with the semantics of the HTML comment format.

**Fix required:** Change the assertion to `assert done_sub.parent_id == 42`.
