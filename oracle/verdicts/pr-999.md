## Stage 2: Quality Review

### What works well

**Constant placement:** All six CONFIDENCE_* constants and both threshold constants are module-scope, correctly named, and imported directly in the test file (not re-declared locally). This is exactly the StrEnum/module-constant golden pattern from 2026-04-24. No mirror-constant risk.

**Decision table ordering:** The implementer flagged the high-cycles/high-attempts ordering risk. The implementation correctly places `steward_cycles >= CONFIDENCE_HIGH_CYCLES_THRESHOLD` as the first guard, before `execution_attempts >= CONFIDENCE_HIGH_ATTEMPTS_THRESHOLD`. The test `test_high_cycles_takes_priority_over_high_attempts` explicitly verifies this ordering with both thresholds at their boundary values. The PR #968 ordering-bug pattern (return_reason early-returns must precede general conditions) is not present here.

**NULL guard in _write_steward_fields:** The `if prescription_confidence is not None:` guard correctly preserves NULL-as-not-written semantics. A caller that omits the field leaves the column unchanged, not cleared. The test `test_confidence_not_overwritten_when_not_passed` verifies this round-trip.

**executor_uow_view exclusion:** Verified structurally. The view's SELECT clause enumerates columns explicitly (`id, status, output_ref, ...`) without wildcards. `prescription_confidence` is not in the list. Steward-private isolation is schema-enforced, not advisory.

**Test constant imports:** All assertions use imported production constants (`CONFIDENCE_FIRST_EXECUTION`, etc.) rather than local re-declarations. Mirror-constant pattern (PRs #696, #800, #967, #970, #974) is not present.

**Pure function structure:** `_compute_prescription_confidence` has no side effects and no I/O. All five branches are tested individually with exact equality assertions (not relational), satisfying the pattern from learnings.md PR #800 (assertion softening to "not-legacy-value" cannot distinguish wired-correctly from silently-failed-with-fallback).

### Gap 1: Audit log coverage is declared but not tested

**The gap:** The test module docstring lists "Audit: confidence included in steward_prescription audit log entry" as a coverage target. The `_get_audit_entries` helper is defined in the test file (lines 304–309) but is never called by any of the 16 tests. No test verifies that `confidence` appears in the `steward_prescription` audit log entry. The PR diff confirms the production write (`"prescription_confidence": confidence` added to `registry.append_audit_log` in `_capturing_prescriber`) but this path is not exercised in CI.

**Why it matters:** The audit log is the stated purpose of this PR alongside the registry row — "included in the steward_prescription audit log entry" is in the PR description, the SQL migration comment, and the module docstring. A dead coverage claim is a learnings.md PR #906 pattern ("dead constant with no enforcement") applied to test scope: the coverage is declared, not verified. The helper exists but its non-use means future regressions in audit log wiring would not be caught by CI.

**What closure looks like:** A test that calls `_capturing_prescriber` (or a sufficiently deep integration path) and asserts `prescription_confidence` appears in the `steward_prescription` audit entry. If `_capturing_prescriber` is too difficult to exercise directly, the coverage claim should be narrowed to "registry row" only, and the audit log coverage should be deferred explicitly with a follow-on issue noted in the test file.

### Gap 2: Steward_cycles off-by-one in audit record (advisory — not blocking, but must be documented)

**The gap:** The confidence is computed with `steward_cycles=cycles` (the pre-prescription per-attempt cycle count, e.g., 3 on the 4th steward cycle). The same `_write_steward_fields` call writes `steward_cycles=new_cycles` (cycles+1, e.g., 4). The audit log entry also uses `new_cycles`. This creates an internal inconsistency: on the UoW's 4th steward cycle, the DB row and audit entry show `steward_cycles=4` while `prescription_confidence=0.6` (LOW_ATTEMPTS), because `cycles=3 < CONFIDENCE_HIGH_CYCLES_THRESHOLD=4` at computation time. The threshold named `CONFIDENCE_HIGH_CYCLES_THRESHOLD=4` means "4 or more steward cycles → low confidence," but the low confidence value (0.2) does not appear in the audit record until `steward_cycles=5`.

**Why it matters:** This is a correctness matter for the audit corpus the PR is building. If this field is later used to calibrate the thresholds (the stated future use case), the off-by-one means every UoW that crossed the 4-cycle threshold will appear to have crossed it one cycle later than it did. The inconsistency is deterministic and reversible in analysis (subtract 1 from steward_cycles to find the cycles value at confidence computation time), but it is not documented anywhere in the code.

**What closure looks like:** Either (a) change the call to use `steward_cycles=new_cycles` for consistency with the rest of the audit record (and update CONFIDENCE_HIGH_CYCLES_THRESHOLD to 5 if the intent is "after 4 complete cycles"), or (b) add a docstring to `_compute_prescription_confidence` explicitly stating "steward_cycles is the pre-prescription per-attempt count (not post-increment)" and a comment at the call site explaining why `cycles` rather than `new_cycles` is the correct input. The behavior is not clearly wrong — using the pre-increment value reflects "what did I know when I made this prescription decision" — but it must be documented because the audit record's steward_cycles field will show new_cycles, creating a gap between the two columns for threshold-boundary UoWs.

**Mirror-constant pattern (PRs #696, #800, #967, #970, #974):** I checked every constant in the test file against the production module. The gap 1 finding was shaped by this pattern: "for every named threshold constant in a scoring or gating module, verify the constant appears in the body of the function that is supposed to use it" — extended here to "for every coverage claim in a module docstring, verify a test actually exercises it."

**Decision table ordering (PR #968):** The learnings.md entry on "return_reason-based early returns must execute before the `not audit_entries → first_execution` guard" made me audit the `_compute_prescription_confidence` ordering carefully before seeing the test. **Assertion softening (PR #800, PR #838):** All computation tests use exact equality (`assert result == CONFIDENCE_HIGH_CYCLES`), not relational assertions. No softening present.

**Golden pattern — classification result with observability fields (2026-04-04):** The PR follows the spirit of this pattern (materializing a classification signal with named constants) but as a scalar float rather than a typed dataclass. This is appropriate for a single-signal case without branching observability needs. The pattern did not require NEEDS_CHANGES — it confirmed the design direction is correct.
