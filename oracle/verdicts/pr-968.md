## Round 1 — 2026-04-26

### Stage 1: Vision Alignment

**Epistemic prior:** This implementation is solving the wrong problem, or solving the right problem in a direction that forecloses better paths.

**What would have to be true for this to be the right path?** The heartbeat signal must be a reliable differentiator between kill-before-start and kill-during-execution. The test file itself surfaces the key risk: `set_heartbeat_ttl` writes an initial `heartbeat_at` at claim time — before the executor dispatches. If this initial write's timestamp is within the 120-second dispatch window, the classifier correctly identifies it as not-yet-started. The correctness of the full classification depends on whether claim time reliably precedes dispatch time by less than the window. The PR's conservative default (kill-before-start on all ambiguous cases) means false positives in the kill-during-execution direction are structurally prevented; false negatives in that direction are the only residual risk, and they degrade to the prior behavior (treating all cases as kill-before-start).

**Does this serve the theory of change in vision.yaml?** Yes directly. `principle-3` (determinism over judgment for conditionals) is satisfied by the pure function with explicit branching and named constants. `constraint-3`'s OODA framing requires Orient before Decide — this PR enriches the audit log at startup_sweep time so the Steward has structured evidence when it reaches the Decide layer. `principle-4` (integration rate before new feature rate) is served: the PR wires the existing `heartbeat_at` field and the existing audit log into a new classification, rather than adding new infrastructure.

**What does this foreclose?** The `orphan_kill_before_start` and `orphan_kill_during_execution` string taxonomy is now committed to two concurrent implementation layers: this PR (startup_sweep classification) and PR #967 (steward diagnosis classification). Both write to the same JSON keys. The concurrent-PR situation means both strings acquire backward-compatibility weight simultaneously across two code paths. A future refinement of the taxonomy needs to be string-compatible across both. This is manageable — the two PRs are layered signal, not duplication — but the taxonomy is more locked after both land than either PR description acknowledges.

**Cheaper test available?** No. The classification is low-cost, defers consequential action, and the signal itself is correct directionally. The scope is appropriate.

**Alignment verdict: Confirmed.** The work is correctly scoped to the Orient layer, defers retry accounting and differential handling to follow-on PRs, and applies the named oracle vocabulary from `principle-3` and `constraint-3`.

### Stage 2: Quality Review

#### Does it do what it claims to do?

Yes. The classification function `_classify_executing_orphan_kill_type` is a pure function that compares two ISO timestamps with explicit timezone normalization and a 120-second window constant. The startup sweep loop calls `registry.get_executor_dispatch_timestamp()`, passes the result to the classifier, and passes the classification forward to `record_startup_sweep_executing()`. The registry method reads the most recent `executor_dispatch` audit entry, preferring the `timestamp` field in the note JSON over the `ts` column. The steward's `_determine_reentry_posture` is updated to route the new values through before the existing `not audit_entries → first_execution` guard, which is the correct ordering fix.

#### Test quality — learnings.md pattern check (named patterns constrained this analysis)

**Pattern applied — "Test file locally re-declares production string constants" (learnings.md PR #696, #800, #967):** I checked specifically for local constant re-declarations in `test_kill_classification.py`. The test file does NOT re-declare the string constants locally. Instead, it imports `DISPATCH_WINDOW_SECONDS` from the production module (`_sweep_mod.DISPATCH_WINDOW_SECONDS`) and uses the imported constant directly in the boundary test cases (e.g., `_iso_ago(1800 - DISPATCH_WINDOW_SECONDS - 120)`). The string values `"orphan_kill_before_start"` and `"orphan_kill_during_execution"` are used as literals in assertions, which is a minor remaining concern (see Observation 1 below), but they are short stable constants whose values match the production module's return values, and the test assertions would fail with a clear mismatch if the production values changed. This is meaningfully better than the pattern PR #967 was cited for: there the local mirror had a *different name* from the production constant, hiding the divergence. Here the string values are used directly, so any value change in the production module causes immediate test failure.

**Pattern applied — "Gate frozenset defined inside a function body" (learnings.md PR #967):** I scanned for frozenset/set constructions inside function bodies. The new code in startup_sweep.py does not introduce any function-local frozensets. The steward.py additions are `match/case` statements and dict lookups — no function-local frozensets. The `_RETURN_REASON_CLASSIFICATIONS` dict addition is at module scope. No violation.

**Pattern applied — "StrEnum constants in all enum-backed SQL and routing logic" (golden-patterns.md 2026-04-24):** The SQL query in `get_executor_dispatch_timestamp` filters on `event = 'executor_dispatch'` — a raw string literal for an audit event type. However, audit event type constants are string literals throughout the registry (the surrounding code uses `'executor_dispatch'` in multiple other places as the established pattern). The PR is consistent with the existing codebase convention. This is not a new vulnerability introduced by this PR.

#### Key correctness decision: ordering fix in `_determine_reentry_posture`

The PR moves the explicit `return_reason`-based early returns (for `executor_orphan`, `executing_orphan`, and the two new values) to occur before the `not audit_entries → first_execution` guard. The PR description explains this correctly: a UoW with `return_reason=orphan_kill_before_start` but empty audit entries would have previously returned `first_execution`, which is wrong — the return_reason signal is more specific and should take precedence. The fix is correct. The test `test_return_reason_orphan_kill_before_start_maps_to_correct_posture` specifically exercises the empty-audit-entries case with a populated return_reason, which is the exact edge case the ordering fix addresses.

#### What breaks without the key decisions made here?

Without the `DISPATCH_WINDOW_SECONDS` constant: threshold tuning is implicit and undiscoverable. Without the safe-default-on-any-error behavior: a single bad audit row causes a hard exception in the startup sweep, blocking all executing-orphan recovery. Without the `record_startup_sweep_executing()` backward-compatible default (`kill_classification: str = "orphan_kill_before_start"`): callers that don't pass the new argument would cause a TypeError. All three decisions are correctly made.

#### Encoded Orientation check (OODA constraint-3)

This PR enriches the audit note written by `record_startup_sweep_executing()` with a `kill_classification` field and updates how `_determine_reentry_posture` reads that field. The classification changes what the Steward sees as the `stuck_condition` value when surfacing to Dan — but does not change the action taken (all three orphan types still result in the same re-prescription path). This is observability enrichment to the Orient layer, not a new behavioral default. The Steward's action on orphan cases (re-prescribe) is unchanged. Constraint-3 does not apply.

#### What failure modes exist?

1. **Clock skew between dispatch time and claim time.** If the executor machine's clock is ahead of the heartbeat writer's clock, a heartbeat written post-dispatch could appear to predate dispatch, causing a false kill-before-start classification. The conservative default means this is a false negative (misses a kill-during-execution), not a false positive. Acceptable.

2. **Multiple executor_dispatch entries for the same UoW.** `get_executor_dispatch_timestamp` returns `ORDER BY id DESC LIMIT 1` — the most recent dispatch. If a UoW was dispatched, killed before start, re-dispatched, and then killed during execution, the most recent dispatch is the correct reference. This is correct behavior.

3. **`set_heartbeat_ttl` initial timestamp written before first real heartbeat.** This is the central assumption of the classification. The 120-second window absorbs this. The test case `test_heartbeat_before_dispatch_is_kill_before_start` explicitly exercises this scenario. Covered.

#### What patterns does this introduce that will propagate?

The two new classification strings (`orphan_kill_before_start`, `orphan_kill_during_execution`) join the startup_sweep classification vocabulary. They are already present in the concurrent PR #967 (steward diagnosis layer). Both PRs use the same string values, which ensures consistency across the two code paths. The concurrent implementation means the taxonomy is now established at two layers — startup_sweep and steward diagnosis — and will be treated as durable by any future work reading `kill_classification` from the audit log.

#### What does this make easier / harder for future work?

**Easier:** Future retry accounting PRs can gate on `kill_classification` to apply different retry strategies for kill-before-start vs. kill-during-execution. The audit note field is already present and tested. The steward's prescription context already includes distinct messages for each type.

**Harder:** Refining the kill taxonomy (adding sub-types) requires maintaining string compatibility across both the startup_sweep path (#968) and the steward diagnosis path (#967). The taxonomy's dual-layer commitment is not acknowledged in this PR's description, but it is the natural consequence of concurrent implementation.

### Observations (not blocking)

**Observation 1: String literals in test assertions vs. production constant names.** The test file uses `"orphan_kill_before_start"` and `"orphan_kill_during_execution"` as literals in assertions (e.g., `assert result == "orphan_kill_before_start"`). The production function returns these same strings directly (not from a named constant — they are returned as literal strings in the function body). So the test-literal and production-literal are the same string, and any change to either would cause immediate test failure. This is slightly better than the PR #967 failure mode (different names, same values) but slightly worse than the ideal (import a named constant). Since the function body itself uses string literals (not named constants), there is nothing to import. The gap is that the production function uses literal strings rather than named constants — which would require a separate refactoring PR. Not introduced by this PR; consistent with the existing codebase pattern for similar functions.

**Observation 2: `from datetime import timedelta` inside the function body.** Line 91 of the diff: `from datetime import timedelta` appears inside the `_classify_executing_orphan_kill_type` function body, after `datetime` and `timezone` are already imported at module scope (they are used at lines 74, 80). This is not incorrect — Python caches module imports — but it is inconsistent with the surrounding code and would be slightly cleaner as a module-level import. Not blocking.

**Observation 3: `except (AttributeError, Exception)` in startup_sweep.py.** The except clause catches both `AttributeError` (anticipated — registry doesn't have the method) and bare `Exception` (catch-all). The bare `Exception` catch swallows any genuine error from `get_executor_dispatch_timestamp`. This is consistent with the safe-default design intent and the existing pattern in the codebase. The debug log line captures the exception text. Acceptable for this use case.
