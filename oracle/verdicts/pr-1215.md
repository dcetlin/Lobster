## Stage 2 — Quality Review

### The critical finding: root cause diagnosis is incorrect

The PR description states: "The context pressure check reads active sessions from the session store — 5 executing UoWs blocked all new dispatch even with 0-1 real sessions running."

This is technically inaccurate. The code path is:

```python
from src.agents.session_store import get_active_sessions
active_session_count = len(get_active_sessions())
```

`get_active_sessions()` queries `agent_sessions WHERE status IN ('running', 'starting') AND agent_type NOT IN ('dispatcher', 'hook')`. This is the **Claude session registry** — it tracks Claude Code subagent processes, not WOS UoW statuses.

Dead WOS subagents with `executing` UoWs in the registry would only appear in `get_active_sessions()` if their `agent_sessions` row was never transitioned from `running` to `completed`/`dead`. The PR description conflates two separate tracking systems:
- `uow_registry.status = 'executing'` — WOS Registry, counts dispatched work orders
- `agent_sessions.status = 'running'` — Session Store, counts live Claude processes

These are different counts measuring different things. The PR claims 5 executing UoWs blocked dispatch, but what actually blocked dispatch was 5 orphaned `agent_sessions` rows that were never cleaned up when those subagents died. The UoW status is not what the context pressure check reads.

**Consequence:** The heartbeat sidecar explanation in the PR description ("heartbeat sidecar refreshes heartbeat_at every 3 minutes for all active/executing UoWs — including those with no live subagent") describes a real bug — the sidecar DOES keep stale executing UoWs fresh, preventing TTL recovery. But this bug manifests in the WOS registry, not in the context pressure check. The two bugs are:

1. **Bug A (heartbeat sidecar freshens dead slots in WOS registry):** Real bug, described correctly. Causes `executing` UoWs to stay in `executing` long past the 24h TTL because `heartbeat_at` is kept fresh. BUT — TTL recovery uses `started_at`, not `heartbeat_at`, as the cutoff.

TTL recovery uses `started_at`, not `heartbeat_at`. So the heartbeat sidecar refreshing `heartbeat_at` does NOT prevent TTL recovery from firing. The PR description's point 3 ("The 24h hard TTL couldn't fire because `started_at` was just under the cutoff for 4 of the 5") is the actual reason — the UoWs were 22–24h old and 4 were just under the 24h cutoff. The heartbeat sidecar is not the cause here; it is a red herring.

2. **Bug B (orphaned agent_sessions rows):** The actual cause of the context pressure threshold being saturated. When subagents die without writing a result, their `agent_sessions` rows stay in `running` status. The reconciler (`prune_dead_sessions`) should clean these up, but if it's not running or has a gap, orphaned sessions accumulate.

**Therefore:** The fix (raising 5→20) treats Bug B's symptom (threshold saturation from orphaned sessions) without fixing Bug B (orphaned session accumulation). Raising the threshold means more orphaned sessions can accumulate before the next blockage — it does not prevent the accumulation.

### Are the real throttles present and functional?

Yes. Both throttles are present and correct in the code:

1. **CC quota gate:** `_read_cc_quota()` returns the CC usage percentage; dispatch is skipped when `cc_pct >= CC_QUOTA_SKIP_THRESHOLD (90.0%)`. Fail-open: absent or stale data (>60 min) does not block dispatch. Structurally sound.

2. **GitHub rate limit gate:** `check_github_rate_limit()` checks remaining API calls against `GITHUB_RATE_LIMIT_DISPATCH_THRESHOLD (100)`. The dry-run guard is correct — this check only runs in non-dry-run mode. Structurally sound.

Both gates are real throttles with independently sufficient gating logic.

### Could raising 5→20 cause runaway dispatch in a degraded state?

The CC quota gate is the primary protection against over-dispatch. If the CC quota gate is bypassed (stale quota data > 60 minutes), dispatch is fail-open by design. With threshold at 20 instead of 5, the degraded scenario (quota gate stale, GH rate limit gate ok, 5-15 orphaned sessions) would now dispatch freely up to 20 concurrent sessions. Previously 5 orphaned sessions would have blocked this.

This is an acceptable trade-off IF the queue throughput concern is genuine and the CC quota gate is reliable — both appear to be true. The fail-open design of the quota gate was a deliberate choice, not a gap.

### Is the heartbeat-refreshing-dead-subagents bug fixed?

No. The PR does not fix the heartbeat sidecar bug. The sidecar still refreshes `heartbeat_at` for all `active` and `executing` UoWs, including dead ones. As noted above, this does not actually prevent TTL recovery (which uses `started_at`), but it does mean the observation loop's stall detection continues to be suppressed for dead UoWs. The structural follow-up (visibility-timeout model) is the correct fix.
