**Does it do what it claims?**

The diff shows exactly two hunks, one per call site:

- `_dispatch_via_claude_p` (~line 1072): `stdout=error.stdout` → `output=error.stdout`
- `_dispatch_via_stub` (~line 1200): `stdout=error.stdout` → `output=error.stdout`

Both are `CalledProcessError` constructor calls. Python's `subprocess.CalledProcessError.__init__(returncode, cmd, output=None, stderr=None)` accepts `output=` and `stderr=` as the keyword parameters for captured output. `stdout=` is not a valid kwarg and raises `TypeError` at construction time. The fix is correct.

The PR description notes that `.stdout` attribute access at the catch site already works correctly after the fix — `output=` populates the `.stdout` attribute (Python's CalledProcessError aliases `.stdout` to `.output`). The catch site's access pattern is therefore unaffected.

**What would break without the key decisions made here?**

Without this fix, any code path that triggers the `raise subprocess.CalledProcessError(...)` in either function receives a `TypeError` instead of the intended `CalledProcessError`. The TypeError propagates up the call stack, replacing the subprocess failure with a Python API error. The actual subprocess output (which `error.stdout` carries) is lost. The router daemon crash with `TypeError: CalledProcessError.__init__() got an unexpected keyword argument 'stdout'` observed for `uow_20260425_639b91` confirms this is not hypothetical.

None identified. The change is a verbatim substitution of one valid kwarg name for another. The positional arguments (`returncode`, `cmd`) and the `stderr=` kwarg are unchanged. The only behavioral change is that `CalledProcessError` is now constructable, making the original subprocess error visible instead of masked.
