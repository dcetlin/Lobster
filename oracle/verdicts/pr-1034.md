### Gap 1: task-outputs path divergence (blocking)

`write_output()` resolves the output directory as:

```python
workspace = Path(os.environ.get("LOBSTER_WORKSPACE", Path.home() / "lobster-workspace"))
output_dir = workspace / "messages" / "task-outputs"
```

This produces `~/lobster-workspace/messages/task-outputs`.

The canonical pattern — used by `steward-heartbeat.py`, `wos-queue-monitor.py`, and the shared `src/utils/inbox_write.py` — is:

```python
messages_base = Path(os.environ.get("LOBSTER_MESSAGES", Path.home() / "messages"))
task_outputs = messages_base / "task-outputs"
```

This produces `~/messages/task-outputs`.

The MCP `check_task_outputs` tool reads from the canonical location. Outputs written to `~/lobster-workspace/messages/task-outputs` are not surfaced by `check_task_outputs`. The sweeper's outputs will be invisible to the dispatcher's standard tooling. The PR description says the dispatcher reviews sweep results — but it cannot, because the outputs land in the wrong directory.

**What this forecloses:** Any dispatcher workflow that calls `check_task_outputs` to read sweeper results will silently find nothing. The operational intent of the sweeper (surface stale PRs for dispatcher action) fails at the output-delivery layer.

**Fix:** Replace the `write_output()` path construction with the canonical pattern using `LOBSTER_MESSAGES` env var.

### Gap 2: Multi-fire inbox notifications (blocking)

`emit_inbox_notification()` fires whenever `summary["stale_open"] or summary["merged_pending_close"]` is non-empty — which is every 6-hour run as long as any stale PR exists. There is no state file tracking which stale PRs have already triggered a notification. A PR open for 30 days would generate approximately 120 inbox notifications before it is resolved.

This is the PR #896 pattern ("Static YAML field used as escalation threshold produces multi-fire escalations") in a structurally identical form. The PR #896 learnings entry was specific: "when a threshold-gated notification reads its threshold input from a static config file and the job does not update that file, verify whether the notification is bounded to a single firing or repeats every cycle above threshold. If it repeats, require either a state file tracking last-escalated date, or a write-back mechanism."

The sweeper has no state file and no write-back mechanism. It fires every cycle where the condition holds.

The correct fix for an advisory-only sweeper is a state file recording `last_notified_at` per stale PR, with a minimum re-notification interval (e.g., 24 hours). This prevents inbox flood while preserving the signal for genuinely new stale items.

**What this forecloses:** Without deduplication, the inbox floods with repeated notifications. The dispatcher will either learn to ignore `wos_pr_sweep_result` messages (defeating the sweeper's purpose) or be overwhelmed by them.

### Gap 3: Inbox notification bypasses canonical shared utility (advisory)

`emit_inbox_notification()` writes directly to `Path.home() / "messages" / "inbox"` using hardcoded path construction and a direct `open().write()` without atomic tmp-then-rename:

```python
inbox_dir = Path.home() / "messages" / "inbox"
```

The canonical pattern (established by PR #805, named in golden-patterns.md as "caller-declared identity in shared utilities") is `src/utils/inbox_write.py`'s `write_inbox_message()`, which: (a) uses `LOBSTER_MESSAGES` env var for path resolution, (b) uses atomic tmp-then-rename for write safety, and (c) gives the message a job-name-prefixed ID for origin traceability.

The sweeper's inbox write has a hardcoded path (ignores `LOBSTER_MESSAGES`), no atomic write, and uses a timestamp-based ID without the job name prefix. If `LOBSTER_MESSAGES` is set to a non-default location (e.g., in tests or alternate deployments), the inbox write goes to the wrong directory.

The golden-patterns pattern says: "Apply whenever extracting a utility function that produces identifiable outputs (IDs, log entries, file names). The identity should flow in as a parameter, not be inferred inside the function." The existing utility satisfies this; the sweeper reinvents it incorrectly.

This is advisory (the functional impact in the standard deployment is limited) but should be fixed to use `write_inbox_message()` for consistency and correctness under non-default `LOBSTER_MESSAGES`.

### Gap 4: `#!/usr/bin/env python3` shebang (advisory)

The script shebang is `#!/usr/bin/env python3`. The cron entry correctly uses `uv run scheduled-tasks/wos-pr-sweeper.py`. However, the shebang means that running the script directly (not via `uv run`) would invoke bare `python3` — the pattern flagged in learnings.md for PRs #804, #821, and #830 as the "bare python3" anti-pattern.

The shebang is not exercised when invoked via `uv run` (which the cron and the test plan use), so this is not a runtime failure. However, it is a consistency gap and misleads future maintainers who read the shebang and assume direct invocation is safe. The shebang should be `#!/usr/bin/env -S uv run` (if that form is supported in the environment) or the docstring "Run standalone" instruction should be the only invocation path documented.

This is advisory — the production path uses `uv run` correctly.

---

### Gap 5: Missing dispatcher handler for `wos_pr_sweep_result` (advisory)

`emit_inbox_notification()` writes messages with `type: "wos_pr_sweep_result"`. The PR description says "action on sweep results is deferred to the dispatcher." There is no dispatcher handler for this message type in the current codebase. Messages of this type will land in the inbox and sit unprocessed — the dispatcher has no routing entry for `wos_pr_sweep_result`.

The PR is explicitly scoped as read-only and advisory; the missing handler is a known gap. However, unhandled messages in the inbox accumulate as noise. The correct path is either: (a) note explicitly in the PR that a follow-on PR will add dispatcher handling, or (b) use the existing `subagent_result` type (which the dispatcher does handle) with an appropriate `category` field to distinguish sweep results.

This is advisory because the sweeper is correctly scoped as read-only; the missing handler is a consequential but anticipated gap.
