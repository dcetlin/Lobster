## Stage 2 — Quality Review

### Gap 1: Encoded Orientation decision missing — constraint-3 is not satisfied

The PR cites WOS UoW `uow_20260427_7ffcb3` as its authorization. Per learnings.md (PR #913, PR #1032): "A WOS UoW artifact is a dispatch record, not a decision record." This is the third distinct form of WOS-artifact-as-authorization in the learnings index.

### Gap 2: Existing tests broken — 4 failures in test_inbox_write.py

Verified by running the tests at 18:11 PT (outside morning window):

```
FAILED tests/unit/test_inbox_write.py::TestWriteInboxMessageSchema::test_writes_valid_json_with_correct_fields
FAILED tests/unit/test_inbox_write.py::TestWriteInboxMessageSchema::test_source_uses_lobster_default_source_env
FAILED tests/unit/test_inbox_write.py::TestWriteInboxMessageSchema::test_source_defaults_to_telegram_when_env_absent
FAILED tests/unit/test_inbox_write.py::TestAtomicWrite::test_output_file_named_after_msg_id
```

The root cause: `write_inbox_message()` is the function under test in `test_inbox_write.py`. The tests set `LOBSTER_MESSAGES` via `monkeypatch.setenv` and assert that the message appears in the inbox. The new circadian gate — `is_non_urgent(msg) and not is_morning_window()` — fires outside the morning window, diverts the message to `pending-deliveries.jsonl`, and returns early. The message never reaches the inbox. The test assertions (`assert len(written) == 1`, `exists()`) fail.

This is not a test environment artifact. It is a behavioral change to the function under test that the existing test suite does not account for. The `try/except Exception: pass` fallback means CI will not produce a loud import error — it will produce the wrong behavior silently in tests run during morning hours and loud failures in tests run outside morning hours. The test suite's correctness is now time-of-day dependent.

**Required to close this gap:** The existing tests in `test_inbox_write.py` must be updated to either (a) mock `is_morning_window()` to return True so the circadian gate is disabled for the existing tests, or (b) set `LOBSTER_WORKSPACE` to a tmp_path so the circadian queue goes to a test location and the assertions are updated to account for the gate. The correct approach is (a): patch `is_morning_window` to return True in the existing tests so they verify the direct-delivery path, and add new tests that patch `is_morning_window` to return False to verify the circadian deferral path through `write_inbox_message`.

### Gap 3: is_morning_window() has no test coverage

The 16 tests in `test_circadian_delivery.py` cover `is_non_urgent`, `queue_message`, and `flush_morning_queue`. They do not test `is_morning_window()`. This function reads the real clock and `ZoneInfo("America/Los_Angeles")`. The learnings.md PR #839 pattern applies: a function that reads real clock state at call time without a test is not covered. The correct pattern: provide a `_now_fn` injection point (callable, defaulting to `datetime.now`) and test with a mock that returns specific Pacific times.

As a secondary concern: the window 06:00–10:00 Pacific is a 4-hour range. The cron fires at 14:00 UTC = 07:00 PDT or 06:00 PST. If the flush runs early in the window, messages queued after 10:00 AM yesterday will have waited up to 20 hours before delivery. This is the intended behavior — it should be stated in the module docstring as an explicit design choice rather than an implicit consequence of the window definition.

**Required to close this gap:** At minimum, a test that patches the clock dependency and verifies the window boundary (06 ≤ hour < 10 → True; hour = 10 → False; hour = 5 → False). This also enables CI to test the `write_inbox_message` integration path correctly — see Gap 2.
