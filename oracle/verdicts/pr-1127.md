#### Gap 1: Routing ambiguity — `/todos` matches the `/todo` routing trigger

The bootup doc instructs the dispatcher: `# msg.text starts with "/todo"`. The command `/todos` starts with `/todo`, so this routing condition will match both `/todos` and `/todo add|done|snooze` messages.

`route_todo_command` receives `/todos` as input, finds no regex match, and returns the `_USAGE` help text. The user who typed `/todos` expecting their item list gets a usage string instead.

The dispatcher is LLM-driven and will likely handle this contextually from the full document — but the instruction is ambiguous, the failure mode is real, and no test verifies that `/todos` is NOT processed by `route_todo_command`.

**Fix required:**
- In the bootup doc, change the routing comment to: `# msg.text starts with "/todo " (space after /todo) or msg.text == "/todo"` — making the `/todos` exclusion explicit.
- Add a test: `route_todo_command({"text": "/todos", ...})` should NOT be called at all, or if called, verify it returns the help text (documenting the safe fallback behavior). The test should assert that the dispatcher doc's routing condition correctly discriminates.

The learnings.md PR #1125 entry ("consumer PR written against producer schema that changed in a sibling PR") pattern applies structurally: the dispatcher routes `/todos` and `/todo add` through adjacent code paths that share a prefix; an ambiguous routing condition produces a silent behavioral failure.

#### Gap 2: Snooze regex silently misparses item text ending in numbers — no test, no doc warning

`_SNOOZE_RE = re.compile(r"^/todo\s+snooze\s+(.+?)(?:\s+(\d+))?\s*$", re.IGNORECASE)` uses a lazy quantifier that makes the trailing `(\d+)` group greedy from the right. For input `/todo snooze call 911`, the regex parses `query='call'` and `days='911'`.

If a user has an item called "call 911" and types `/todo snooze call 911`, they get: item "call" is searched (not found or wrong match), not item "call 911" snoozed for the default 3 days.

This is the learnings.md PR #980 "parse_diagnose_command returns full remainder instead of first token" pattern in reverse — here the parse is too aggressive at extracting the days argument. The ID-based form exists as the workaround (`/todo snooze 42`), but there is no warning to the user, no test covering this case, and the failure is non-obvious.

**Fix required:**
- Add a comment in `_SNOOZE_RE` naming the ambiguity: "A trailing integer is always parsed as days — use `/todo snooze <id>` for items whose text ends in a number."
- Add the disambiguation note to `_USAGE` (e.g., "Tip: for items whose text ends in a number, use the item ID.").
- Add a test that demonstrates the behavior: `route_todo_command({"text": "/todo snooze call 911", ...})` with an item named "call 911" in DB — verify the user sees an informative result (either the item is found by partial match "call" or a disambiguation/not-found message explains what happened).

#### Gap 3: Natural language routing instruction adds Encoded Orientation without a logged prior (constraint-3)

The bootup doc adds:
```
Natural language triggers the dispatcher should also route via route_todo_command:
- "add X to my list" / "remind me to X" → treat as /todo add X
- "mark X done" / "X is done" → treat as /todo done X

For natural-language routing, construct a synthetic /todo command from the extracted intent and pass it as msg["text"] to route_todo_command.
```

This encodes a new durable behavioral default: when the dispatcher sees these natural-language patterns, it should (a) classify intent, (b) construct a synthetic `/todo` command, and (c) route to `route_todo_command` — all without a subagent and without Dan's per-message input. This is an Encoded Orientation decision under OODA constraint-3.

The learnings.md PR #894 entry establishes the pattern: "A GitHub issue describing a design decision is not equivalent to a logged oracle/vision decision." There is no entry in `oracle/decisions.md`, no open_decision in `vision.yaml`, and no oracle verdict that authorizes natural language → synthetic command construction as a durable dispatch default.

The natural language routing also introduces a failure mode the `/todo` command path cannot handle: the dispatcher extracts intent from free text, constructs a synthetic `msg["text"]`, and calls `route_todo_command`. If the extraction is imprecise (e.g., "remind me to call Sarah tomorrow" → `/todo add call Sarah`), `route_todo_command` receives the synthetic text with no awareness that it came from NL extraction. The structured command path assumes precise input; the NL extraction path produces approximate input. There is no error surface when the extraction is wrong — the item is silently added with extracted text.

**Fix required:**
- Either remove the natural language routing section from the bootup doc for now and defer it to a follow-on PR with a logged decision, OR
- Add an entry to `oracle/decisions.md` with: (a) the specific vision.yaml anchor (principle-3 "Determinism over judgment for conditionals" is directly relevant — NL intent extraction is judgment, not deterministic), (b) explicit statement that this is authorized as a durable dispatch default, and (c) the risk acknowledgment that extraction errors will produce silently incorrect add operations.

The `/todo add|done|snooze` routing itself (items 1 and 2) does NOT require a decision record — it is deterministic routing of explicit commands. Only the NL extraction path falls under constraint-3.
