## Stage 1: Vision Alignment

**Prior entering review:** This implementation is solving the wrong problem, or solving the right problem in a direction that forecloses better paths.

**Theory of change check.** The vision's fundamental intent is a system that knows what Dan is building toward and routes work accordingly — specifically with structural access to intent, not just prose reconstruction. The current WOS substrate is in Phase 1: registry live, UoW lifecycle instrumented, oracle/steward loop wired. The question a join key between Telegram messages and UoWs serves is: "which conversations led to pearls?" — i.e., which user-initiated inputs produced high-value WOS outcomes.

**What would have to be true for this to be the right path?** The registry must already produce enough data to make the join useful. `activity_timeline` is only valuable if (a) the audit_log is being written with meaningful data and (b) the control_events table from PR #1105 is populated. Both are true: audit_log fires on every status transition and control_events just landed. The timing is correct.

**What does this foreclose?** Almost nothing. A nullable column with no FK constraint and a `CREATE VIEW IF NOT EXISTS` are both additive-only changes. The view could be dropped and replaced; the column can never be removed (SQLite), but NULL-by-default with no existing callers means no existing behavior is affected. There is no path foreclosure here.

**What is the opportunity cost?** The PR is a schema preparation step that enables future attribution queries. It does not produce user-visible output itself. The cost is the migration slots 0022 and 0023, which are narrow. The PR is explicitly scoped to infrastructure; it does not bundle execution logic changes. This is the correct register for this work: `principle-4` (wire what exists before building more) — `trigger_message_id` is the missing arrow between an existing system (Telegram inbox) and another existing system (WOS registry). The join key is the wiring; the dispatcher caller that passes it is the second half.

**Is there a cheaper test of the underlying assumption?** The assumption is: "the join between a Telegram message and its resulting UoW is useful enough to encode." A cheaper test would be a manual log-scraping exercise. But that has apparently already been done (the PR description describes it as the pre-existing state). Adding the column has near-zero risk and is reversible in behavior if not in schema. Stage 1 prior is **not confirmed** — this is not solving the wrong problem.

**Alignment verdict: Confirmed.** The work extends `principle-4` directly and serves the WOS observability substrate required by `active_project.phase_intent`.

### Agent-flagged area 3: Write point — no live caller yet

Confirmed: no existing caller passes `trigger_message_id`. The cultivator/garden_caretaker path (GitHub sweep) has no Telegram message_id to pass. The dispatcher_handlers.py has no path that creates UoWs from user messages currently — all UoW creation flows through the cultivator sweep. This is acceptable. It follows the seam-first abstraction golden pattern: the schema boundary and read path (`get_uow_trigger`) are established at the correct seam; the dispatcher caller that will populate the field is the next incremental step. The question is whether there is a current dispatcher path that creates UoWs from user messages and is *now* silently failing to capture attribution. Based on inspection of dispatcher_handlers.py, there is no such path — UoW creation is cultivator-only at this time. The column's null population is intentional, not a missed wiring.

### What would break without the key decisions made here

The join key design is correct: TEXT column with no FK (inbox messages live in a separate DB), NULL for cultivator paths (not all UoWs have a Telegram origin), TEXT format matching the inbox message_id convention `"{ts}_{telegram_msg_id}"`. An integer FK would be wrong (cross-DB constraint not enforceable in SQLite). A NOT NULL constraint would break all existing cultivator callers. All design choices are justified.

### What patterns does this introduce that will propagate

The `trigger_message_id` join key establishes a convention: inbox message IDs are the primary trace between user conversations and system artifacts. This is correct — the inbox message_id is already the canonical unique identifier in the Lobster inbox system. Future attribution work (which conversations led to PRs, which conversations led to completed UoWs, conversation-to-outcome pipelines) will use this same join key.

The `activity_timeline` view establishes the pattern of UNION ALL across audit_log and control_events as a unified event stream. If more event sources are added in the future, this view is the natural extension point. The view ordering by `ts DESC` is a reasonable default for "what happened recently?" queries.

### What this makes easier / harder

**Easier:** Attribution queries. "Which Telegram messages spawned which UoWs?" — single WHERE clause. Conversation-to-outcome analysis. Dashboard views over WOS activity.

**Harder:** Nothing is foreclosed. The schema is additive.
