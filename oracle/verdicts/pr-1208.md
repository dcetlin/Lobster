## Stage 2: Quality Review

### Critical Blocker: Migration Number Collision

PR #1208 adds **Migration 112** to `scripts/upgrade.sh`. The current main branch already has a Migration 112:

```
# Migration 112: Register pre-tool-use-pr-idempotency.py PreToolUse hook.
```

(line 4034 in the current `scripts/upgrade.sh` on main)

This is the exact failure mode documented in learnings.md 2026-04-27 PR #1000:

> "Migration number collision between simultaneous PRs produces silent feature deletion. [...] When the second PR merges, its migration file is silently skipped — the feature's DB table is never created."

The collision here is even harder to detect than the PR #1000 case: the `upgrade.sh` migration runner uses a substep-based structure without an explicit visited-version guard in the same pattern as the DB migrations. Both migration 112 blocks will execute, but both use the same label in comments — on an existing install where the upgrade was already run for migration 112, the second block (this PR's task-file copy) will execute because its guard condition is `[ -f "$_wos_uow_src" ] && [ ! -f "$_wos_uow_dst" ]`, which is a file-existence check rather than a migration-version deduplication check. On a fresh install they will both run. On an existing install that ran the old migration 112, the new one will also run because its guard is independent.

The actual behavior depends on whether the task file destination already exists. But the labeling collision is a maintenance defect: future migrations numbered 113 and beyond from this PR's perspective will collide with the actual current 113, 114, 115, 116 on main.

The PR was authored against a branch state that predates the current main. The branch cut point (single commit, authored 2026-05-18T20:42:41Z) is from a snapshot where migrations only went to 111. Main has since added migrations 112–116. **This PR's migration must be renumbered to 117** before it can be merged without collision.

This is a structural blocker per learnings.md 2026-04-27, PR #1000: "When two PRs are in concurrent oracle review, the oracle should flag migration number conflicts as a NEEDS_CHANGES blocker regardless of which merges first, since merge order cannot be guaranteed."

The fix is: renumber the PR's migration block from 112 to 117.

### Docstring Misrepresentation in handle_wos_uow

The `handle_wos_uow` docstring says:

> "return a task file path for dispatcher subagent dispatch"
> "Returns: A string describing the outcome — either the task file path (success) or an inline not-found message."

The function returns `"found"` (a sentinel string), not a task file path. The bootup doc shows the dispatcher checking `if result != "found":` and then opening the task file from a hardcoded path. The docstring is describing an interface that was changed during implementation but the docstring was not updated. The comment mismatch is the learnings.md 2026-04-14, PR #607 pattern: "Comment/code mismatch at state-machine transition causes silent state divergence."

Impact: Low for correctness (the code is right), but misleading for any future maintainer adding a similar handler who reads the docstring as the contract. The docstring should say the function returns `"found"` sentinel string on success, not "the task file path."

### Duplicate Suffix-Match Logic

The suffix LIKE query appears in two places: `handle_wos_uow` in `dispatcher_handlers.py` (the gate function) and Steps 1-2 in `dispatcher-wos-uow-detail.md` (the subagent task file). When `handle_wos_uow` returns `"found"` after a suffix match, the subagent re-does the same suffix LIKE query in Step 1 to get the actual resolved UoW. This is correct behavior (the gate confirms existence; the subagent resolves the full object). But the subagent also has its own ambiguity/not-found branches that are now unreachable via the normal dispatch path — the dispatcher's call to `handle_wos_uow` would have already handled those cases and sent inline errors. The task file's ambiguity/not-found branches serve as redundant dead code under normal operation.

This is not a blocker — the redundancy is safe and provides a defensive fallback if the task file is ever invoked directly. But the task file should note this explicitly to prevent future confusion about why the branches exist.

### fmt_ts Timezone: Displays UTC, Not Dan's Timezone

The `fmt_ts` helper formats timestamps as `"%Y-%m-%d %H:%M UTC"` — hardcoded UTC display. Vision.yaml constraint-4 and CLAUDE.md behavioral guideline 5 ("Always display times in the user's local timezone") both point toward converting to Dan's timezone (America/New_York from the owner.toml migration). This is a secondary concern — the UoW timestamps are DB-internal, and UTC display is defensible for system records — but it is inconsistent with the declared user-facing timezone rule.

### No Tests Included

The PR has no automated test coverage. The `handle_wos_uow` gate function has three branches (exact match, suffix match → 0/1/many results) that are eminently unit-testable. Given the mirror-constant pattern history in this codebase, the absence of tests is not catastrophic here since there are no constants to mirror — but the gate logic's `registry._connect()` call (using a private method directly rather than a public API) is an abstraction violation that a test would surface. The `_connect` method is private (underscore prefix). The gate function should use a public registry method or a separate SQL helper rather than reaching into `_connect`.

### registry._connect() as Public Usage

`handle_wos_uow` calls `registry._connect()` directly. This is a private method by Python convention. It works, but it couples the handler to the internal implementation of `Registry`. If `Registry._connect()` changes (e.g., adds a logging layer, or changes the connection factory), `handle_wos_uow` would not participate in those changes. The preferred pattern is to add a public method to `Registry` (e.g., `get_by_suffix(suffix: str) -> list[str]`) or use an existing pattern. This is a quality note, not a blocker.
