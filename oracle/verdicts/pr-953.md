## Stage 1: Vision Alignment

Formed before reading the implementation.

**Theory of change:** The PR addresses a documented incident (433 restart iterations in ~3 hours) triggered by quota exhaustion. The implicit theory is that restart instability is caused by identifiable, quantifiable conditions — message storms, unthrottled WOS dispatch, and a dirty-inbox state at quota-recovery time — and that threshold-based suppression with pre-flight checks is the correct intervention.

**Vision anchor:** Principle-1 ("Proactive resilience over reactive recovery") directly authorizes this class of work. Constraint-4 (minimize metabolic cost of cybernetic engagement) is a second anchor: a restart storm that runs 433 times over 3 hours without user-visible signal and requires manual diagnosis is the highest-cost form of cybernetic failure. The work is load-bearing for the WOS execution pipeline — a system that restarts under its own load cannot deliver reliable WOS throughput.

**Adversarial prior applied:** The scenario in which this work is wasted effort: the root cause is not the feedback loop itself but something deeper (e.g., the health checker's staleness-detection logic is too sensitive, or the quota sleep logic has a timing error that causes premature restarts). In this scenario, the three fixes add suppression on top of a misconfigured trigger, and the next failure mode will be different but also unexpected. I tested this scenario: the PR description gives a specific count (433 restarts, 54 stale messages, 3 hours), and the feedback loop is structurally real — stale messages recovered from processing/ → inbox/ is a documented behavior of the restart path. The adversarial scenario is not confirmed.

**Learnings.md active priors checked:**
- PR #913 pattern (suppression mechanism with no timeout and no user notification can remain in sustained suppression indefinitely) applies to Fix 1 (storm cap) and Fix 2 (context pressure throttle). I checked both: Fix 1 has a Telegram alert and re-evaluates on every health check cycle — it is not indefinite suppression. Fix 2 is a per-heartbeat skip, not a latch — it re-evaluates at each 15-minute cron invocation. Neither creates a permanent suppression state. The PR #913 failure mode (no escape hatch) is not present.
- PR #896 multi-fire pattern (threshold check fires continuously rather than at crossing only): Fix 1 and Fix 2 are designed to fire continuously when the condition persists — which is correct behavior here. The storm cap is not a "notify Dan once" pattern; it is a "skip restart each time condition is true" pattern. This is the right design for a storm suppressor.

**Alignment verdict: Confirmed.** The work is correctly scoped, anchored to principle-1 and constraint-4, and the adversarial prior was tested and not confirmed.

## Stage 2: Quality Review

### What the PR does

Three additive guard clauses — no deletions, no behavioral changes to existing paths:

1. **Fix 1 (health-check-v3.sh `do_restart()`):** Before executing a restart, count inbox files via `ls "$INBOX_DIR" | wc -l`. If > 20, skip the restart, log a warning, and send a deduplicated Telegram alert. Return 0 (not an error — re-evaluate next cycle).

2. **Fix 2 (executor-heartbeat.py `main()`):** After the `execution_enabled` gate but before `run_executor_cycle()`, query `get_active_sessions()`. If >= 5 sessions, skip new dispatch for this cycle. TTL recovery and heartbeat sidecar continue unconditionally. Fail-open on exception (active_session_count defaults to 0).

3. **Fix 3 (claude-persistent.sh `handle_exit()`):** After the quota-wait midnight sleep, poll inbox file count every 30 seconds. If count < 5 or timeout (600s) elapses, proceed with restart. The timeout ensures restart is not blocked indefinitely.

### Does it do what it claims?

Yes. The before/after flow in the PR description is accurate and matches the code. The feedback loop mechanism (inbox recovered from processing/ → health check fires → restart → recovery → repeat) is correctly targeted by Fix 1. Fix 3 buys time for Fix 1 to hold during the early post-quota-recovery period when the inbox is at its highest count.

### Defects found

**Defect 1 (minor, non-blocking): Inconsistent inbox file counting method.**

The existing codebase uses two different counting strategies:
- `claude-persistent.sh`'s wake-signal loop: `find "$inbox_dir" -maxdepth 1 -name "*.json" 2>/dev/null | wc -l`
- `health-check-v3.sh`'s main inbox drain check: `find "$INBOX_DIR" -maxdepth 1 -name "*.json" -print0 2>/dev/null`

The new code in both Fix 1 and Fix 3 uses `ls "$INBOX_DIR" | wc -l` and `ls "$MESSAGES_DIR/inbox/" | wc -l` respectively, which counts all directory entries including non-JSON files (temp files, lockfiles, directories). The effect is conservative — the storm cap and drain threshold will fire at lower actual-message-counts than the threshold implies — but this is an inconsistency with the established pattern in the same codebase. In a clean inbox, non-JSON files are uncommon enough that this is unlikely to cause false suppression. Not a correctness defect; flagged as a style inconsistency.

**Defect 2 (minor, non-blocking): CONTEXT_PRESSURE_AGENT_THRESHOLD defined inside `main()` rather than at module scope.**

The golden-patterns.md entry [2026-04-25] "named display cap with aggregate-preserving header" (PR #938) establishes the rule: cap constants belong at module scope so they are importable by tests. `CONTEXT_PRESSURE_AGENT_THRESHOLD: int = 5` is defined inside `main()`, making it unreachable for test imports without re-running `main()`. This is not a functional defect in this PR since no test file currently imports this constant, but it deviates from the established pattern.

**No defects found in:**
- The Encoded Orientation check (constraint-3): These are guard clauses, not behavioral default changes. They suppress existing behaviors (restart, dispatch) under new conditions but do not add new automated behaviors. No logged prior is required because the trigger is new (the feedback loop was unmitigated before) and the action is suppression of an existing action, not a new automated action.
- Feature bundling: Three files, three named fixes, all described in the PR body. No undescribed modifications.
- The fail-open behavior of Fix 2: Correct. An unreadable session store is not a reason to hold off WOS dispatch.
- The 10-minute drain timeout in Fix 3: The PR correctly notes that Claude is stopped during the quota wait, so the inbox won't drain during the wait — the timeout is the expected path. The timeout prevents Fix 3 from blocking the restart indefinitely when the inbox genuinely cannot drain.
- The `send_telegram_alert_deduped "storm-cap"` call: The function exists and is used elsewhere. The deduplication key "storm-cap" is unique within the existing calls (confirmed by grep). The alert fires once per storm event, not once per suppressed restart (this is the deduped design — correct).

### What would break without the key decisions made here?

Without Fix 1: the 433-restart scenario repeats on the next quota exhaustion event.
Without Fix 3: Claude restarts immediately into a flood inbox, and Fix 1 must hold every check cycle from the start rather than having the drain window.
Without Fix 2: WOS runs at full dispatch speed during context pressure, which the PR author identifies as a contributing cause of faster compaction cycles and increased restart frequency.

### What failure modes exist?

**Fix 1:** If the inbox count is consistently above 20 for reasons unrelated to a restart storm (e.g., genuine user message backlog from offline period), restarts are suppressed even when they would be beneficial. The health check re-evaluates every 4 minutes; if the inbox never drains, restarts are suppressed indefinitely. However: (a) the Telegram alert fires, giving Dan visibility; (b) this scenario requires a genuine backlog of 20+ user messages, which is a legitimate signal to investigate rather than restart through. This failure mode is acceptable.

**Fix 2:** If the session store accumulates stale "running" session records (TTL not clearing them), the active session count may remain >= 5 permanently, throttling WOS dispatch indefinitely with no user notification. The TTL recovery path (Phase 1 in the heartbeat) is supposed to clear stale sessions, but that recovery works on the registry DB (UoW status), not the session store. Dan would need to observe the throttle from logs. This is the closest analog to the PR #913 indefinite-suppression pattern, though at reduced severity since the heartbeat is 15-minute cycles and the session store has its own TTL mechanism (the session_end calls from completed subagents). No Telegram alert fires for sustained context pressure throttle.

**Fix 3:** Not a failure mode risk — it is bounded by a 600-second timeout and always proceeds.

### What does this make easier/harder for future work?

Easier: diagnosing future restart storms — the storm cap's Telegram alert and log line give a concrete starting point.
Harder: nothing of significance. The guard clauses are additive and do not change existing behavior paths.

### Patterns introduced

No new structural patterns. The three fixes apply existing patterns (threshold guard with Telegram alert, fail-open exception handling, polling with timeout) to new trigger conditions.

## Summary

Two minor non-blocking issues found:
1. `ls | wc -l` counting method inconsistent with `find -name "*.json" | wc -l` used elsewhere — conservative overcounting, not a correctness defect.
2. `CONTEXT_PRESSURE_AGENT_THRESHOLD` defined inside `main()` rather than at module scope — deviates from the golden-patterns.md PR #938 constant placement rule, but no test currently requires module-scope access.

One failure mode noted: sustained stale sessions in the session store could cause Fix 2 to throttle WOS dispatch indefinitely without a Telegram alert. This is low-probability given existing TTL recovery, and the log is observable. It does not block approval but is noted for follow-on consideration.

No Encoded Orientation check required. No feature bundling. Root cause of the incident is correctly targeted. All three fixes are additive and correctly bounded.
