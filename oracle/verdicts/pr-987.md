## Stage 2 — Quality Review

### What the PR does

Adds `check_github_rate_limit()` (pure function, subprocess-based) and `RateLimitStatus` (frozen dataclass) to `executor-heartbeat.py`. Calls the check once per dispatch cycle in `main()`, after the dry_run guard and context pressure throttle, before `run_executor_cycle()`. If `remaining < 100`, skips the cycle and returns 0. Fail-open on all error paths.

### What works well

The implementation is technically clean. The golden pattern "classification result as typed frozen dataclass with observability fields" (golden-patterns.md 2026-04-04) is correctly applied: `RateLimitStatus` carries `ok`, `remaining`, and `reset_at` — exactly the observability fields that allow the caller to log without re-querying. The fail-open design is correct: a tooling failure is not a reason to suppress user work. The threshold is a named module-scope constant imported by tests rather than re-declared locally (avoiding the mirror-constant pattern). The test suite is well-structured: boundary case at exact threshold, fail-open paths for non-zero exit code, FileNotFoundError, and unparseable JSON, plus integration tests that exercise `main()` directly.

### Issues found

**Issue 1 — Encoded Orientation without logged prior (constraint-3, blocks merge).**

`GITHUB_RATE_LIMIT_DISPATCH_THRESHOLD = 100` encodes a new behavioral default: the system autonomously suppresses dispatch cycles without Dan's explicit input when remaining API quota is below 100. This meets the definition of Encoded Orientation under constraint-3. No prior logged decision exists in `oracle/verdicts/`, `vision.yaml open_decisions`, or a decision document. PR #894 (needs-human-review) and PR #896 (smell-pattern automation) were both returned with NEEDS_CHANGES under this exact requirement. The same constraint applies here. Resolution: create a decision document (e.g., `oracle/verdicts/decision-github-rate-limit-gate.md`) citing the vision.yaml anchor (principle-1: "structural prevention over reactive recovery" is the relevant anchor) and logging the threshold value as the authorized default.

**Issue 2 — Module docstring in test file references stale `--jq` expression.**

The test module's top docstring (line 6) says the function "shells out to `gh api rate_limit --jq .rate.remaining`". The actual implementation uses `--jq '.rate | {remaining, reset}'` (a JSON object with both fields). The docstring is a stale copy from issue #984, which used the single-field form. The `_rate_json` helper's inline comment on line 79 correctly reflects the actual expression. The discrepancy is documentation-only and does not affect test correctness, but it will mislead future engineers reading the test file to understand the shell command. Fix: update the module docstring to match the actual `--jq` expression.

**Issue 3 — Dead code in `_run_main_with_clean_argv` test helper.**

The helper builds a `context_managers` list via a list comprehension on the first line (before `with patch("sys.argv", ...)`), then immediately builds a second `all_patches` list that supersedes it. The `context_managers` variable is never used. This is the "dead code immediately before an unconditional override" failure mode (learnings.md PR #848). The dead list comprehension also has a type mismatch: it mixes `patch.object(...)` and `patch.object(..., **kwargs)` based on whether `kwargs` is a dict — but `all_patches` always uses `patch.object(hb_module, name, new=val)`, so the dead comprehension's branching is never evaluated. Fix: remove `context_managers` entirely. Note: `_run_main_with_clean_argv` is never actually called by any test — it is defined but unused. The integration tests call `_hb_fresh.main()` directly via the `with (...)` block. Consider removing the helper or noting its status.

**Issue 4 — Pre-existing failures claim: verified as consistent with claim.**

The PR claims 3 pre-existing failures in `test_executor.py::TestOptimisticLock`. These are described as expecting `RuntimeError` not to propagate. The failures are in `test_executor.py`, not in the new test file, and the diff shows no changes to `src/orchestration/executor.py` or `test_executor.py`. The claim is consistent with the diff scope. No new failures are introduced by this PR. The pre-existing failures are not introduced here.

### Failure modes

- If the GitHub API schema for `gh api rate_limit` changes (`.rate` key removed or restructured), the `data["remaining"]` access raises `KeyError` — caught by the outer `except Exception`, returning fail-open. This is correct defensive behavior.
- The check is per-cycle, not per-UoW, so a rate limit drop during a cycle (after the check passes) will still cause subagent failures. This is acceptable and documented in the PR description.
- The 10-second `subprocess.run` timeout adds latency to each dispatch cycle when the gh CLI is slow. At a 3-minute cycle cadence, this is negligible.

### Patterns introduced

The `RateLimitStatus` frozen dataclass with `ok` + observability fields (`remaining`, `reset_at`) is a clean application of the classification-result-as-typed-dataclass pattern. It will propagate well if other rate limit checks are needed in the future.
