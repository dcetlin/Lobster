## Stage 2 — Quality Review

### What the PR does

Creates `src/utils/jobs.py` with a canonical `is_job_enabled(job_name: str, default: bool = True) -> bool`. Removes local `_is_job_enabled` definitions from 14 of the 17 scripts that had them, replacing with `from src.utils.jobs import is_job_enabled`.

### What the canonical implementation looks like

```python
def is_job_enabled(job_name: str, default: bool = True) -> bool:
    workspace = Path(os.environ.get("LOBSTER_WORKSPACE", Path.home() / "lobster-workspace"))
    jobs_file = workspace / "scheduled-jobs" / "jobs.json"
    try:
        data = json.loads(jobs_file.read_text())
        return bool(data.get("jobs", {}).get(job_name, {}).get("enabled", default))
    except Exception:
        return default
```

The implementation is clean and correct. The `default` parameter is a meaningful addition over the local definitions (most of which hardcoded `True`). The exception handling correctly returns `default` on any failure path. The function reads `LOBSTER_WORKSPACE` at call time, not at module import time — avoiding the PR #839 anti-pattern.

The pre-PR state had 17 scripts with local `_is_job_enabled` definitions (verified via `git grep`). The PR updated 14. Three remain with local definitions. A new contributor reading `granola_ingest.py` will see a local `_is_job_enabled` and have no signal that the canonical version lives in `src/utils/jobs.py`. **Revision contract:** Update the 3 missed scripts to `from src.utils.jobs import is_job_enabled` and remove their local definitions, matching the pattern applied in the 14 updated scripts.

### Gap 2: Test files mock the removed local function by old name

The following test files still mock `_is_job_enabled` (the old local function name) after the PR removes or renames the function in the production scripts:

- `tests/los/test_todo_obsidian_sync.py` — multiple `patch(f"{_MOD}._is_job_enabled", ...)` calls targeting `todo_obsidian_sync._is_job_enabled`, which no longer exists
- `tests/los/test_e2e_todo_pipeline.py` — `patch(f"{_MOD}._is_job_enabled", return_value=True)`
- `tests/unit/los/test_vault_watcher.py` — `patch.object(_vault_watcher, "_is_job_enabled", return_value=True)`
- `tests/unit/test_orchestration/test_prescriptions_per_cycle.py` — `patch.object(steward_heartbeat, "_is_job_enabled", return_value=True)` (multiple tests)
- `tests/unit/test_orchestration/test_github_rate_limit_check.py` — `patch.object(_hb_fresh, "_is_job_enabled", return_value=True)` (multiple tests)

After the PR, the production modules that were updated no longer have a `_is_job_enabled` attribute — they have `is_job_enabled` (imported from `src.utils.jobs`). `patch.object(module, "_is_job_enabled")` will raise `AttributeError` at test time because the attribute no longer exists. Tests that gate on the enabled-check returning True will either raise or silently fail to intercept the real function call.

The correct fix for updated scripts is to patch the canonical location: `patch("src.utils.jobs.is_job_enabled", return_value=True)`. This is the standard approach when a function is extracted to a utility module — patch the module where it lives, not the consuming module.

**Revision contract:** For each test file that mocks `_is_job_enabled` on a script that was updated in this PR, change the mock target from `module._is_job_enabled` to `src.utils.jobs.is_job_enabled`. The 3 scripts not yet updated (Gap 1) retain their local function and can continue to be patched at `module._is_job_enabled` until they are updated.
