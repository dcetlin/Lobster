## Round 1 — 2026-05-01

### Stage 1: Vision Alignment

**Prior entering review:** This implementation is solving the wrong problem, or solving the right problem in a direction that forecloses better paths.

**Implicit theory of change:** Adding `subject` and `signal_type_hint` columns to the memory events table will improve classification quality by letting writers pre-classify events at write time, allowing the slow-reclassifier to skip expensive content inference. This is a write-layer enrichment pattern — betting that producers have more context than content inference can recover.

**What would have to be true:** The bottleneck in memory quality must be at the classification layer (slow content inference producing wrong signal types), not at the retrieval layer, the consumer layer, or the Orient routing layer. It would also need to be true that producers reliably know their own signal type at write time — which is true for structured producers (harvesters, scheduled jobs) but uncertain for the dispatcher writing ad-hoc observations.

**What this forecloses:** Adding a classification vocabulary at write time couples producers to the vocabulary. When the vocabulary changes (new signal types, renamed types), producers must be updated. The `VALID_SIGNAL_TYPES` frozenset becomes a contract surface — adding or removing values requires coordinating across every producer that has hardcoded a hint. This is a low-cost coupling at current scale but accumulates as a friction surface.

**Tension with vision.yaml:** `current_focus.what_not_to_touch` states: "New detection or classification rules — improve Orient routing before adding more detection." Adding `signal_type_hint` and the fast-path fast-classifying logic is classification infrastructure. This is the most direct tension with the stated constraint.

**Resolving the tension:** The PR is described as "Part of WOS UoW uow_20260501_8de2bc" — a dispatch record, not a logged decision. The `what_not_to_touch` constraint reads as operational guidance ("don't add more detection until Orient routing is stable"), and there is an argument that enriching existing signals (rather than adding new detection) falls outside the constraint's intent. However, the fast-path in `slow_reclassifier.py` that bypasses pattern detection entirely for hinted events is a durable behavioral change to the classification decision-making path — this is an Encoded Orientation decision under constraint-3.

**Alignment verdict:** Questioned — the work has coherent technical logic, but the Encoded Orientation decision embedded in the fast-path classification bypass lacks a logged prior. The `what_not_to_touch` tension is borderline; the constraint-3 gap is structural.

### Stage 2: Quality Review

**The implementation does what it claims to do.** The schema columns are added correctly, with backward-compatible migration in both `vector_memory.py` (in-process ALTER TABLE with exception-ignore) and `upgrade.sh` (Migration 91, idempotency-guarded). The `MemoryEvent` dataclass gains `subject` and `signal_type_hint` as optional fields with correct `to_dict`/`from_dict` round-trip. The `memory_store` MCP tool accepts and validates the new fields. The slow-reclassifier fast path correctly gates on `signal_type_hint` presence, and the `AND slow.entry_id IS NULL` guard in `read_recent_events` means the fast path is idempotent — events tagged by the fast path on one run are not re-processed on subsequent runs.

The OR logic in the four pattern detectors (`detect_design_session`, `detect_brainstorm_mode`, `detect_meta_thread`, `detect_philosophy_thread`) correctly implements "prefer hint over quick_tags inference" with proper fallback semantics.

**Named gaps:**

---

**Gap 1: Mirror-constant — schema enum in `inbox_server.py` is a raw string list, not derived from `VALID_SIGNAL_TYPES`**

The `memory_store` tool schema in `inbox_server.py` (line ~3362) declares:
```python
"enum": ["task_request", "design_question", "voice_note", "status_check", "system_observation", "meta_reflection", "philosophy", "casual"]
```
This is a raw string list, not derived from `VALID_SIGNAL_TYPES` in `provider.py`. These two representations are currently in sync, but this is the mirror-constant pattern named in learnings.md (PRs #696, #800, #967, #970, #974). When a new signal type is added to `VALID_SIGNAL_TYPES`, the schema enum won't be updated automatically, and callers using the MCP tool schema for enumeration (clients that inspect tool definitions) will see a stale vocabulary. The validation in `handle_memory_store` correctly imports `VALID_SIGNAL_TYPES` at runtime, so the actual validation is single-sourced — only the tool schema description diverges. The schema enum is the mirror. Fix: derive the schema enum from `VALID_SIGNAL_TYPES` at server startup, or document the synchronization requirement explicitly in a comment adjacent to both definitions.

---

**Gap 2: No tests for the fast path or new MemoryEvent fields**

The fast-path logic in `run_pass` (the `hinted_event_ids` block) has no unit test coverage. Specifically:
- No test verifies that an event with `signal_type_hint` set gets tagged with the correct `signal_type` in the written `ClassificationTag`.
- No test verifies that a hinted event is excluded from cluster-based pattern detection (the `events_for_pattern_detection` filter).
- No test verifies that the `MemoryEvent.to_dict()` / `from_dict()` round-trip correctly handles `subject` and `signal_type_hint` (the existing `test_memory.py` tests do not include these fields and would pass with `None` values, providing no coverage of the new serialization paths).
- No test verifies that `handle_memory_store` rejects an invalid `signal_type_hint` with the expected error text.

The precedent from learnings.md PR #893: "when a function is extracted to enable testability, verify that a test was actually added for the extracted function — extraction without a new test achieves structural clarity but does not close the test gap." The same applies here: the fast path is a new code path with behavioral consequences, not just a structural refactor.

---

**Gap 3: Encoded Orientation decision without logged prior (constraint-3)**

The fast-path classification bypass in `slow_reclassifier.run_pass` is an Encoded Orientation decision under vision.yaml constraint-3. It changes a durable behavioral default: events with `signal_type_hint` now bypass all pattern detection and receive a `confidence="high"` slow-v1 tag using the caller's self-declared type. This is not a reactive or constrained decision — it is encoded as a structural rule in the classification pipeline.

The PR description cites "WOS UoW uow_20260501_8de2bc" as the authorizing reference. Per learnings.md PR #913: "A WOS UoW artifact is a dispatch record, not a decision record. Citing the WOS UoW as the authorizing prior does not satisfy constraint-3." The correct prior is a logged decision in `oracle/decisions.md` with a traceable `vision.yaml` anchor by field path.

This gap is the same structural failure as PR #913 and PR #894. The behavioral change is real and technically sound; what's missing is the audit trail that makes it traceable under constraint-3.

**Required resolution:** Add an entry to `oracle/decisions.md` that: (a) names the behavioral default being encoded (caller-provided `signal_type_hint` bypasses pattern detection in slow-reclassifier and receives `confidence=high`), (b) cites a specific vision.yaml field path as the anchor (e.g., `active_project.phase_intent` or `principle-4`), (c) states who authorized it and why. The WOS UoW may be referenced as context but cannot serve as the authorization record.

---

**What patterns this introduces:**

- A two-tier classification contract: structured producers (harvesters, scheduled jobs) self-classify at write time; ad-hoc writers go through content inference. This is a reasonable architectural distinction that could propagate positively — but it should be named explicitly as a design pattern, not left implicit in the implementation.
- The subject field as a noun-phrase label has no constraints (length, vocabulary). This is appropriate now but may accumulate inconsistency across producers. Future retrieval queries that group by subject will need to normalize.

**What this makes harder:**

- Auditing classification quality: hinted events never go through pattern detection, so their classification is entirely dependent on caller accuracy. A miscategorized `signal_type_hint` written by a producer with bad context will be tagged `confidence=high` with no correction path. The slow-reclassifier's pass/revised counters conflate hinted events with pattern-detected events (`total_revised = tags_revised + len(hinted_event_ids)`), making it harder to track true reclassification rates.

**Encoded Orientation check:** Yes, this PR constitutes an Encoded Orientation decision (the fast-path classification bypass changes a durable behavioral default in the classification pipeline). Verification: (a) no prior logged decision exists in `oracle/verdicts/` or `oracle/decisions.md` — Gap 3 above; (b) the `vision.yaml` anchor is not cited by field path in the PR or its referenced UoW. Both conditions for a constraint-3 NEEDS_CHANGES flag are present.

Deferred only if the reason is a CI import constraint (as in PR #893), with explicit documentation.
- **Gap 3 (Encoded Orientation without prior):** Address by adding a logged decision to `oracle/decisions.md`. Cannot be disputed (the behavioral change is real and meets constraint-3's definition). Deferred only if the behavioral change is reverted in this PR.
