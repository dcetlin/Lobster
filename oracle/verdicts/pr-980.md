## Stage 2: Quality Review

### Flagged area 1: wos_diagnose branch placement in route_wos_message

The `wos_diagnose` branch is placed **inside** the spawn-gate block (alongside `wos_execute` and `steward_trigger`), not in the fast-path that precedes it (where `wos_escalate` lives).

The block comment on `handle_wos_diagnose` says:

> Unlike wos_execute/steward_trigger, this handler is exempt from the spawn-gate because it legitimately returns action="spawn_subagent" only (no send_reply branches). route_wos_message handles this analogously to the wos_escalate fast-path.

This comment is factually wrong on both claims:
1. The handler is NOT exempt from the spawn-gate — it runs through it.
2. It is NOT analogous to the wos_escalate fast-path — wos_escalate is dispatched before the spawn-gate block; wos_diagnose is dispatched inside it.

The placement is structurally correct: because `handle_wos_diagnose` always returns `action="spawn_subagent"`, it passes the spawn-gate enforcement check with no issue. Being inside the gate is actually preferable — the gate provides error handling and action validation as a safety net. The comment should be corrected to say the handler is **compliant with** the spawn-gate (not exempt from it), and placed inside the spawn-gate block because its always-spawn semantics make the gate redundant but not harmful.

The dispatch table comment also says it is "exempt from the spawn-gate" — this should be corrected to match the actual code structure.

This is the PR #607 learnings.md pattern: comment/code mismatch at a routing branch. The consequence is that a future maintainer adding a new always-spawn message type will be confused about whether to use the fast-path (wos_escalate model) or the spawn-gate block (wos_diagnose model), because the comment implies wos_diagnose is in the fast-path when it isn't.

**Gap 1 — comment/code mismatch in handle_wos_diagnose:** The block comment on `handle_wos_diagnose` (lines 97-101 of the PR diff) and the dispatch table comment say it is "exempt from the spawn-gate" and "analogous to the wos_escalate fast-path." The actual placement is inside the spawn-gate block. The comment must be corrected. Addressed = the comment accurately describes the placement (inside spawn-gate block, not fast-path).

The new section contains **two inconsistent code examples**:

Block 1 does not call `mark_processing` before routing and includes a `send_reply` acknowledgment to Dan. Block 2 calls `mark_processing` first (consistent with the standard WOS message processing pattern) but omits the `send_reply`. These two blocks give contradictory integration guidance.

3. `Task(prompt=..., task_id=..., run_in_background=True)` — spawn
4. `send_reply(chat_id, f"Diagnosing UoW {uow_id} — report incoming.")` — acknowledge (for manual triggers only)
5. `mark_processed(message_id)` — close

### Flagged area 5: parse_diagnose_command returns full remainder

`parse_diagnose_command("diagnose uow_001 extra ignored")` returns `"uow_001 extra ignored"` — the full remainder after "diagnose ", not just the first token. The test at TestParseDiagnoseCommand.test_uow_id_token_is_first_word_after_diagnose documents this behavior:

```python
# The function returns the full remainder after 'diagnose ' with leading/trailing
# whitespace stripped — so "uow_001 extra ignored" is the raw uow_id token.
```

The test name says "first_word" but the comment says "full remainder." The test assertion (`assert "uow_001" in result`) uses substring matching rather than equality — which passes for both "uow_001" (first-word behavior) and "uow_001 extra ignored" (full-remainder behavior).

The consequence: if Dan types `diagnose uow_20260426_abc123 please` (trailing context, which is natural language usage), the subagent receives `"uow_20260426_abc123 please"` as the UoW ID. The prompt embeds this in:

```
1. Run: uv run {registry_cli_path} trace --id uow_20260426_abc123 please
```

This passes `uow_20260426_abc123` to `--id` and `please` as a positional argument. `argparse` will reject this with an error (unrecognized argument). The subagent command fails silently with an error in the trace output.

The design spec prompt template (line 313) shows `uv run .../registry_cli.py trace --id {uow_id}` — implying a single clean token. The function should strip the token to the first whitespace-delimited word: `uow_id_token = stripped[len("diagnose "):].split()[0]` (with a check for empty after split).

**Gap 3 — parse_diagnose_command returns full remainder:** The function should return only the first whitespace-delimited token after "diagnose " — not the full remainder. Full-remainder behavior breaks `registry_cli trace --id` when Dan adds any trailing text. The test assertion `assert "uow_001" in result` is too loose — it passes for both correct and broken behavior. Addressed = (a) function strips to first token; (b) test asserts `result == "uow_001"` (equality, not substring) for the multi-word input case.

### Mirror-constant in prompt string: MAX_RETRIES and HARD_CAP

The prompt embeds:
```python
f"   MAX_RETRIES = 3\n"
f"   HARD_CAP = 9\n\n"
```

The design spec (self-diagnosing-subagent-design.md §3) explicitly says:

> **Constants (from steward.py and registry.py — do not hardcode)**
> `from src.orchestration.steward import _HARD_CAP_CYCLES  # currently 9`
> `MAX_RETRIES = 3  # current execution_attempts cap; confirm from steward constants`

Production values: `MAX_RETRIES: int = 3` (steward.py line 678), `_HARD_CAP_CYCLES = 9` (steward.py line 669). Both match today. But the design spec's comment "do not hardcode" is prescriptive: these values should be interpolated from production constants. If `_HARD_CAP_CYCLES` changes to 10, the subagent silently uses 9 until this file is also updated.

The fix: import `MAX_RETRIES` and `_HARD_CAP_CYCLES` from their production modules at the top of `handle_wos_diagnose` (or at module scope if acceptable given circular import constraints), then interpolate into the f-string:

```python
from src.orchestration.steward import MAX_RETRIES as _STEWARD_MAX_RETRIES, _HARD_CAP_CYCLES
# ...
f"   MAX_RETRIES = {_STEWARD_MAX_RETRIES}\n"
f"   HARD_CAP = {_HARD_CAP_CYCLES}\n\n"
```

The test assertions `assert str(MAX_RETRIES) in result["prompt"]` use a locally-declared test constant (3) — not the imported production value. If the production constants change, the test assertion also passes against the wrong hardcoded value. The test should import the production constants and assert against them.

**Gap 4 — hardcoded MAX_RETRIES and HARD_CAP in prompt:** These values must be imported from production modules (`steward.MAX_RETRIES` and `steward._HARD_CAP_CYCLES`) and interpolated via f-string, not hardcoded as literals. The design spec is explicit on this point. Test assertions for these values must also use imported production constants. Addressed = prompt uses `{imported_constant}` interpolation; test assertions use imported constant values.

### Encoded Orientation check (constraint-3)

This PR embeds autonomous `registry_cli decide-retry` capability in the subagent prompt. When posture is `reset`, the subagent calls `decide-retry` without a second confirmation from Dan. This is triggered by Dan's explicit "diagnose X" command, making it Constrained register rather than pure Encoded Orientation — the autonomy is bounded within a user-initiated workflow.

The design spec (§6 Safety Constraints) documents the explicit safety bounds: all return_reasons must be orphan types, execution_attempts must be 0 or kill_type is orphan_kill_before_start, execution must be enabled, and lifetime_cycles must be below HARD_CAP. These constraints are encoded in the prompt.

The design spec at `workstreams/wos/design/self-diagnosing-subagent-design.md` is the authorization document. This is weaker than an oracle/verdicts/ logged decision but the spec is detailed, explicit about safety constraints, and in the workstream design directory. Advisory: a logged decision in oracle/verdicts/ or oracle/decisions.md would provide a stronger audit trail for this autonomous action capability.
