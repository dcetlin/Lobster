### Stage 1: Vision Alignment

**Prior entering review:** This implementation is solving the wrong problem, or solving the right problem in a direction that forecloses better paths.

The PR inserts a programmatic triage layer between "retry cap exceeded" and "notify Dan." The right-problem question: is the symptom "Dan gets too many notifications" or "the system cannot distinguish infrastructure turbulence from genuine execution failure"? The design doc and issue #969 frame it as the latter — the 2026-04-26 cohort produced 19 notifications for a single infrastructure event precisely because the failure classification was absent. The triage layer is the structural fix for a classification gap that previous PRs (#965-968) established the vocabulary for.

What would have to be true for this to be the wrong path? If the correct fix were upstream — preventing orphan kills rather than routing them after the fact — this PR would be technically sound but at the wrong layer. That upstream fix is not feasible at the current stage (session kills are infrastructure events outside the WOS pipeline's control). The triage layer is at the correct seam.

What does this foreclose? The early-return structure and pure function design deliberately leave the upstream wiring (Steward write path) to a follow-on PR. This does not foreclose that path — it establishes the receiver before wiring the sender, which is the correct build order for testability. The pure function is testable in isolation; wiring it first would require full Steward integration in the same PR.

**Alignment verdict: Confirmed.** The triage layer is consistent with vision.yaml principle-1 (proactive resilience over reactive recovery) and principle-4 (integration rate before new feature rate — wiring an existing gap). Dan authored issue #969 and the design doc, making the direction explicitly authorized.

### Stage 2: Quality Review

#### What the PR does

Registers `wos_escalate` as a new WOS message type and implements `handle_wos_escalate(msg)` — a 4-branch decision tree. Branch 4 (human-judgment register) fires first; Branch 3 (execution_attempts >= 3) fires second; Branch 1 (attempts == 0, orphan) auto-retries; Branch 2 (attempts > 0, orphan) retries with resume context; default surfaces to Dan. The handler returns a dict describing what the dispatcher should do — the dispatcher executes the action. `route_wos_message` dispatches `wos_escalate` via an early-return before the spawn-gate, correctly exempting it from the "must always spawn" invariant.

#### Confirmed correct decisions

**Branch priority (Branch 4 before Branch 3):** Correct. A `human-judgment` UoW with 3+ execution_attempts should surface as "wrong tool for this work," not as "execution budget exhausted." The register classification is semantically more informative for the human receiver. The implementer's analysis is correct.

Relaxing the gate assertion to tolerate `send_reply` from any handler would weaken it for the execution types that must always spawn. Early-exit is the correct mechanism to exempt a specific type while preserving the gate for others. The dispatch table comment is accurate and sufficient.

#### Required changes (blocking)

**Gap 1: `_ESCALATE_ORPHAN_CLASSIFICATIONS` uses raw string literal instead of `ReturnReasonClassification.ORPHAN`**

`_ESCALATE_ORPHAN_CLASSIFICATIONS = frozenset({"orphan"})` — the string `"orphan"` is the value of `ReturnReasonClassification.ORPHAN` from `src/orchestration/steward.py`. The StrEnum golden pattern (golden-patterns.md, 2026-04-24) requires: "any frozenset of status/posture/return-reason strings defined inside a function body [or at module scope] should be module-scope constants with enum-backed values." The pattern applies here: if `ReturnReasonClassification.ORPHAN` is renamed, the frozenset does not fail at compile time — it fails only at runtime when no orphan UoW auto-retries. The detection rule from learnings.md PR #967: "Gate frozenset defined [with] raw string literals for enum-backed values... a posture string renamed in the enum would not fail at the frozenset site, only at runtime."

Required fix: import `ReturnReasonClassification` from `src.orchestration.steward` and use `frozenset({ReturnReasonClassification.ORPHAN})`. If the import creates a circular dependency, define the orphan classification constant in a shared location both modules can import from.

**Gap 2: Test file re-declares production constants instead of importing them**

The test file declares:
- `SURFACE_TO_DAN_EXECUTION_THRESHOLD = 3` (mirrors `_ESCALATE_SURFACE_EXECUTION_THRESHOLD = 3` in production)
- `HUMAN_JUDGMENT_REGISTERS = ("human-judgment", "philosophical")` (mirrors `_ESCALATE_HUMAN_JUDGMENT_REGISTERS`)
- `WOS_ESCALATE_MESSAGE_TYPE = "wos_escalate"` (mirrors the string value registered in `WOS_MESSAGE_TYPE_DISPATCH`)

These are locally-re-declared mirrors of production constants. This is exactly the pattern named in learnings.md PR #696 (first instance), PR #800 (recurrence), and PR #967 (third recurrence): "locally-re-declared mirrors of production constants create a silent divergence: when the production constant changes, the test constant does not, causing test assertions to fail silently." The behavioral consequence here is clear — if `_ESCALATE_SURFACE_EXECUTION_THRESHOLD` changes from 3 to 5, the test asserts against 3 and silently validates the wrong boundary.

Required fix: import `_ESCALATE_SURFACE_EXECUTION_THRESHOLD`, `_ESCALATE_HUMAN_JUDGMENT_REGISTERS`, and the message type string from `src.orchestration.dispatcher_handlers` rather than redeclaring them locally. If the constants are currently private (prefixed with `_`), either promote them to module-level public constants or accept that the test imports them via their private names (standard practice for testing private constants by reference).

**Gap 3: `ORPHAN_RETURN_REASONS` test constant is dead code with a semantic mismatch**

The test file declares `ORPHAN_RETURN_REASONS = ("executor_orphan", "executing_orphan", "diagnosing_orphan", "orphan_kill_before_start", "orphan_kill_during_execution")` but this constant is never used in any assertion. It lists raw `return_reason` strings (the pre-mapping values from steward.py's dispatch table), not `return_reason_classification` values (the mapped `"orphan"` string that `_ESCALATE_ORPHAN_CLASSIFICATIONS` actually checks). A future test author who encounters this constant and uses it in assertions against `return_reason_classification` would write tests that pass for the wrong reason — the fixture defaults to `return_reason_classification="orphan"` regardless of which `ORPHAN_RETURN_REASONS` value is used. The semantic mismatch plus dead code status together create a trap.

Required fix: remove `ORPHAN_RETURN_REASONS` from the test file, or replace it with an import of the actual production constant `_ESCALATE_ORPHAN_CLASSIFICATIONS` and name it accurately. If the intent was to document which raw return_reason strings map to the orphan classification, add a code comment referencing `steward.py`'s dispatch table rather than a constant that implies it can be used in assertions.

#### Advisory (non-blocking)

**Purity claim in docstring is imprecise:** The docstring states "Pure function — no side effects, no I/O." The function calls `os.environ.get("LOBSTER_ADMIN_CHAT_ID", "0")` and constructs filesystem paths via `os.path.abspath(os.path.join(os.path.dirname(__file__), ...))`. These are reads, not writes, and the function does not execute any I/O — it returns a description of work for the dispatcher to execute. The function is effectively pure in the sense that matters for testing (no state mutation, no I/O execution). The docstring should be corrected to say "No I/O executed — returns a descriptor for the dispatcher to execute" to avoid misleading readers about `os.environ` reads.

**`test_result_always_contains_message_type` does not cover Branch 4:** The test iterates over `execution_attempts in (0, SURFACE_TO_DAN_EXECUTION_THRESHOLD)` but does not test a register=human-judgment message. Branch 4 is covered by `TestHumanJudgmentRegisterSurfaceToDan`, but the "all branches" intent of the purity test is not fully realized. Non-blocking; add a `register="human-judgment"` case to the loop for completeness.
