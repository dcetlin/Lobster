## Stage 1 — Vision Alignment (before seeing implementation)

**Prior entering this review:** This implementation is solving the wrong problem, or solving the right problem in a direction that forecloses better paths.

**What the PR's theory of change is:**
Subagents have been reimplementing ad-hoc SQL queries incorrectly → provide three new CLI subcommands + a canonical doc guide → future subagents will use the canonical path instead of raw sqlite3.

**The question the adversarial prior asks:**
What would have to be true for CLI + docs to be the right abstraction level? And what would make it the wrong one?

**What would have to be true:**
1. Subagents choosing to use raw sqlite3 rather than the CLI is a **choice made from ignorance** — they didn't know there was a canonical path.
2. A document they can discover at the right moment will change that choice.
3. The CLI itself is structurally sound — its path resolution, DB connection, and output format are stable enough to be canonical.

**The disconfirming case:**
The PR description says the problem was "raw sqlite3 calls with relative paths, direct Python imports via sys.path.insert, and ad-hoc GROUP BY queries that fail mid-execution." These are not ignorance errors — they are the behaviors of an agent that lacks a reliable mechanical path to the right abstraction. A doc doesn't change this structurally: the next subagent launched in a new context will have the same gap unless the doc is guaranteed to be in-context at the moment the subagent decides how to access WOS data.

**The foreclosure question:**
Does this foreclose a better path? The decision NOT to add MCP tools is stated explicitly in the PR and is the load-bearing architectural choice. The stated rationale: "~100 lines of boilerplate for 3 read operations that work fine via subprocess." This is reasonable at face value — but it assumes that subagent invocation of a subprocess CLI is the correct ergonomic pattern going forward. If subagent context includes the doc (WOS-DATA-ACCESS.md), this works. If the doc is not guaranteed in-context, the CLI path requires the subagent to know to run it — which is the same problem as knowing to use the Registry class directly.

**Vision alignment check:**
- `vision.yaml principle-3` ("Determinism over judgment for conditionals"): CLI access is subprocess-based and produces JSON — this is in the spirit of principle-3.
- `vision.yaml principle-1` ("Proactive resilience over reactive recovery"): A doc is reactive documentation; it prevents future bugs only if subagents read it. A structural check would be an enforced access path (MCP tool) or a linting step that catches raw sqlite3 imports. The PR explicitly declines MCP tools. This is a mild tension with principle-1 but not a disqualifying one — the CLI *is* a structural boundary; the question is whether it's reliably invoked.
- `active_project.phase_intent`: The Phase 1 goal is "the Registry is live and populated" and "agents can answer is this aligned with Dan's intent." This PR supports that by making Registry queries reliable — it's clearly within phase scope.

**Stage 1 verdict:** Alignment is **Confirmed** for the problem definition and the CLI approach. The adversarial prior does not land a decisive hit. The risk is not that this solves the wrong problem — it does address a real recurring failure mode. The risk is that docs-only prevention doesn't close the loop unless the docs are structurally guaranteed to be in-context at subagent instantiation time. That's a real gap worth naming but not a disqualifying misalignment.

### cmd_status_breakdown: direct sqlite3 usage

**Finding — Gap 2 (substantive issue):**

`cmd_status_breakdown` opens a raw sqlite3 connection via `registry.db_path` instead of going through the Registry class. The PR description acknowledges this: "cmd_status_breakdown uses registry.db_path (public attribute) for a direct aggregation query."

**Gap 1: Contradiction between approved import pattern and anti-pattern (NEEDS_CHANGES)**

The anti-patterns section identifies "absolute but brittle" as the prohibited pattern and gives `/home/lobster/lobster` as the example of what not to do.
