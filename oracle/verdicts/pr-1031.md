### Test 1: `test_uow_id_interpolated_in_heartbeat_call`

**What the production code actually produces (verified against `dispatcher_handlers.py`):**

Line 374:
```
mcp__lobster-inbox__write_wos_heartbeat(uow_id='test-uow-001', token_usage=<cumulative_tokens>)
```
→ matches `"uow_id='test-uow-001'"` (branch 1 of OR)

Line 383:
```
WOSRegistry().write_heartbeat('test-uow-001', token_usage=<cumulative_tokens>)
```
→ matches `"write_heartbeat('test-uow-001'"` (branch 2 of OR)

Both branches of the OR are genuine evidence of UoW ID interpolation. Neither branch over-accepts: any prompt containing `uow_id='test-uow-001'` or `write_heartbeat('test-uow-001'` contains the actual UoW ID (`test-uow-001`), substituted from the fixture parameter. A prompt with an unsubstituted placeholder (e.g., `{uow_id}` or `<uow_id>`) would match neither. The OR is a union of valid evidence forms, not assertion softening.

The learnings.md PR #839 pattern ("assertion softening to 'not-legacy-value' cannot distinguish wired-correctly from silently-failed-with-fallback") does not apply here. The original assertion was not a weaker version of the new one — it was asserting a string form (`write_heartbeat('test-uow-001')`, closed paren, no additional args) that is simply absent from the production prompt. The MCP path has never produced that form. The original test was always wrong about what the MCP path produces. The fix is not a softening; it is a correction that extends to cover both production paths.

**Mirror-constant check:** The string `'test-uow-001'` in the assertion is not an independently declared constant — it flows from the fixture call to `handle_wos_execute(uow_id="test-uow-001", ...)`. No divergence risk.

### Test 2: `test_60_90_second_interval_documented`

**What the production code actually produces (verified against `dispatcher_handlers.py` line 365):**

```python
f"You must write a liveness heartbeat every 60\u201390 seconds throughout execution.\n"
```

U+2013 (en-dash) is used in the production string. The original assertion checked for ASCII hyphen (U+002D), which is not present between "60" and "90" anywhere in the production prompt. The original test was asserting a character that never appears in the production string — meaning the test was always failing (as stated in the PR description). The fix substitutes the correct Unicode literal.

This is not a case where the production prompt has a bug that the test should catch. En-dash between numerals in a range is correct typographic convention. The test was wrong, not the code.

### Pattern note for learnings.md

The OR condition for multi-path assertions is a clean pattern where the production function has a preferred path (MCP tool) and a fallback path (direct registry call), and both embed the required value. The correct test posture is: verify the value appears in at least one of the valid forms, not that it appears in the form the test author first expected. This does not conflict with learnings.md PR #839 (assertion softening) because softening accepts forms that could be wrong; this OR accepts forms that are both provably correct.
