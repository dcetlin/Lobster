## Stage 1: Vision alignment (written before seeing implementation)

**Theory of change:** The PR fixes a correctness failure in WOS intake — duplicate UoW proposals produced by a race between scan() and tend() in GardenCaretaker. This is directly in service of WOS reliability, which is the active phase (`active_project.current_phase: "WOS Phase 1"`). A duplicated-proposal bug corrupts the Registry's single source of truth about what work exists, which undermines every downstream routing and prioritization decision that reads Registry state. Fixing this is necessary before orientation and routing improvements are meaningful.

**What would have to be false for this to be the wrong path?** The race is real (667 duplicates, 317 affected issues in production). The fix site (Registry + _reactivate_uow) is correctly identified. The architecture does not offer a cheaper path: serializing scan/tend at the call site would require more invasive changes to GardenCaretaker's public surface. A read-check-write guard at the Registry level is the minimal intervention that closes the gap for the identified failure mode.

**What does this foreclose?** Nothing significant. The fix is additive — a new guard function plus a status addition. It does not change the broader lifecycle logic or foreclose refactoring options.

**Secondary fix concern (pre-Stage-1):** The PR bundles two logically distinct fixes: (1) the _reactivate_uow race guard; (2) adding 'closed' to the terminal-status exclusion list in _upsert_typed. The 'closed' fix corrects a data-quality gap affecting 768 legacy rows. Bundling these is not feature-bundling in the egregious sense (they share a test class and a conceptual space around "idempotent intake"), but the 'closed' fix is an Encoded Orientation decision about what constitutes a terminal status — the kind of change that requires a logged prior under constraint-3.

**Alignment verdict: Confirmed** — the work addresses the right problem. The implementation direction is correct. Stage 2 surfaces specific structural issues.

## Stage 2: Quality review

### Does it do what it claims to do?

Yes. The `has_active_uow_for_issue` check in `_reactivate_uow` correctly prevents the double-proposal scenario. The test `test_reactivation_skipped_when_new_proposed_uow_exists` exercises the exact race scenario described in the PR. The normal reactivation path is preserved. The 'closed' addition to `_upsert_typed` unblocks the 768 legacy rows.

### Critical findings

**Finding 1 (NEEDS_CHANGES): `_TERMINAL_STATUSES_FOR_ISSUE_CHECK` frozenset is declared but not used.**

The PR adds a module-level constant `_TERMINAL_STATUSES_FOR_ISSUE_CHECK` with the docstring "Mirrors the exclusion list in `_upsert_typed` exactly." However, the `has_active_uow_for_issue` SQL query does not reference this frozenset — the terminal values are hardcoded as raw string literals directly in the SQL:

```python
AND status NOT IN ('done', 'failed', 'expired', 'cancelled', 'closed')
```

This is the dead-constant pattern from learnings.md (PR #906: "A declared, exported, and tested threshold constant that is never referenced in the function it is meant to gate is a dead constant"). The frozenset is not used anywhere; the function's correctness depends entirely on the raw SQL strings. The comment "Mirrors the exclusion list in `_upsert_typed` exactly" is therefore a documentation claim the code does not enforce.

More concretely: this is the raw-SQL-strings failure mode the StrEnum golden pattern (2026-04-24) exists to catch. The pattern states: "Raw string literals for enum-backed DB columns are forbidden." The `has_active_uow_for_issue` SQL contains raw terminal-status strings rather than using `UoWStatus` values or the declared frozenset.

The frozenset should either be: (a) actually used — parameterized into the SQL via a placeholder-per-element pattern so the list is the single source; or (b) removed entirely if the approach of SQL parameterization is deferred, with the mirror-claim comment removed. As written, the frozenset creates a false sense of synchronization while the SQL strings are free to diverge silently.

**Finding 2 (NEEDS_CHANGES): Three-way terminal-status divergence introduced.**

Before this PR, terminal-status was represented in two places:
- `UoWStatus.is_terminal()` — DONE, FAILED, EXPIRED, CANCELLED
- `_upsert_typed` SQL — `('done', 'failed', 'expired', 'cancelled')`

After this PR, there are three representations:
- `UoWStatus.is_terminal()` — DONE, FAILED, EXPIRED, CANCELLED (unchanged — 'closed' NOT added)
- `_upsert_typed` SQL — now includes 'closed'
- `_TERMINAL_STATUSES_FOR_ISSUE_CHECK` frozenset — now includes 'closed'
- `has_active_uow_for_issue` SQL — hardcoded with 'closed' (independent from frozenset)

`UoWStatus.is_terminal()` is the canonical programmatic check for terminal status in Python call sites. The PR correctly treats 'closed' as terminal in SQL but does not update `is_terminal()` to reflect this. A caller that uses `is_terminal()` to determine whether re-proposal is allowed (rather than a DB query) would incorrectly treat 'closed' as non-terminal. The docstring of `is_terminal()` reads: "True for statuses that allow re-proposal (done, failed, expired, cancelled)" — 'closed' is not mentioned, and it does allow re-proposal after this PR.

The correct fix is one of: (a) add a note to `is_terminal()` that 'closed' is a legacy DB status treated as terminal only at the SQL layer, explicitly excluded from the enum method because it is not a valid enum value; or (b) handle 'closed' in the enum by making it a CANCELLED alias or a named legacy value. The current state is silent divergence between the enum method and the SQL-level behavior.

**Finding 3 (NEEDS_CHANGES): Encoded Orientation decision for 'closed' status lacks logged prior.**

Adding 'closed' to the terminal-status exclusion list changes the behavioral default for 768 production rows: issues that were permanently blocked from re-proposal will now be re-proposable. This is a durable, behavioral-default change — Encoded Orientation under vision.yaml constraint-3.

The PR description documents the technical rationale but does not cite a logged decision in `oracle/verdicts/` or `oracle/decisions.md`. The learnings.md PR #894 entry is the applicable pattern: "A GitHub issue describing a design decision is not equivalent to a logged oracle/vision decision." The PR description itself is not a logged decision.

The _reactivate_uow race guard is a correctness fix (not Encoded Orientation — it enforces a pre-existing intended invariant). The 'closed' terminal-status reclassification is an Encoded Orientation decision: it changes what the system will do for a class of issues that previously received a specific (blocking) treatment. A one-paragraph ADR in `oracle/decisions.md` citing the 768-row production evidence and a vision.yaml anchor (e.g., `principle-1: proactive resilience`) satisfies constraint-3 for this change.

**Finding 4 (advisory): Race window acknowledged but not documented.**

The `has_active_uow_for_issue` check is outside the write transaction in `set_status_direct`. Between the SELECT (in `has_active_uow_for_issue`) and the eventual `set_status_direct` write, a concurrent scan() could INSERT a new proposed row — the very race the guard is meant to prevent. In practice this race window is very narrow (GardenCaretaker is single-threaded in the current deployment), but it is not documented. A docstring comment noting "this check is best-effort in the absence of a distributed lock; concurrent scan() invocations are not expected in current deployment" would prevent a future parallel deployment from silently reintroducing the bug.

This is advisory (not NEEDS_CHANGES) because the current deployment model makes the race window effectively impossible.

**Finding 5 (advisory): `issue_number is None` path silently bypasses the dedup guard.**

In `_reactivate_uow`:
```python
issue_number = self._extract_issue_number(uow.source or "")
if issue_number is not None and self.registry.has_active_uow_for_issue(issue_number):
```

If `_extract_issue_number` returns None (malformed or non-GitHub source_ref), the guard is skipped entirely and reactivation proceeds unconditionally. This is the correct failure mode (fail open rather than fail closed for unknown source_refs), but it is undocumented. A comment explaining why `None` falls through to reactivation (and what types of source_refs would produce `None`) would prevent future engineers from removing the guard as "redundant."

### Encoded Orientation check (constraint-3)

The `_reactivate_uow` guard: not an Encoded Orientation decision — it enforces an intended invariant that was not previously encoded.

The 'closed' terminal-status addition: IS an Encoded Orientation decision. It durably changes behavioral defaults for a class of rows. No logged prior exists in `oracle/verdicts/` or `oracle/decisions.md`. **Flag: NEEDS_CHANGES (Finding 3 above).**

### What patterns does this introduce?

Positive: The `has_active_uow_for_issue` pattern as a "is this issue already claimed?" check is clean and reusable. If future subsystems need to gate on existing UoW presence, this is the right abstraction.

Concerning: The frozenset-declared-but-not-used pattern (Finding 1) introduces a false synchronization signal. If left in, future readers will assume the frozenset is the authoritative list when the SQL is the operative list. This produces the same failure mode as the learnings.md mirror-constant pattern — a reader updating the frozenset would not update the SQL, and the behavior would silently diverge.

### 6 pre-existing test failures

The PR claims 6 pre-existing failures in `tests/test_garden_caretaker.py`. The diff shows `tests/test_garden_caretaker.py` was modified (96 additions, 0 deletions). The test run reports "25 passed, 6 pre-existing failures (unchanged)." The PR does not identify which 6 tests fail or confirm they exist in the base branch at the same count. This is an information gap, not a blocking finding — the claim is plausible given the "(surface-on-close and pending tests)" parenthetical, but "pre-existing" should be verifiable from the base branch test run.

## Revision contract

**Gap 1 (NEEDS_CHANGES): Remove or wire `_TERMINAL_STATUSES_FOR_ISSUE_CHECK`.**
Resolution options: (a) Use the frozenset to build the SQL IN clause dynamically so it is the single source of truth; (b) Remove the frozenset and the mirror-claim comment — the SQL is the operative list. Option (a) is preferred: it satisfies the StrEnum golden pattern and makes the synchronization real rather than documentary. The "mirrors _upsert_typed exactly" claim in the docstring must match the implementation.

**Gap 2 (NEEDS_CHANGES): Document the `UoWStatus.is_terminal()` / 'closed' divergence.**
Add a comment to `is_terminal()` explicitly noting that 'closed' is a legacy DB string not representable as a `UoWStatus` value, treated as terminal only at the SQL layer. This makes the three-list divergence intentional-and-documented rather than a silent bug surface.

**Gap 3 (NEEDS_CHANGES): Log a decision for the 'closed' Encoded Orientation change.**
Add a short ADR to `oracle/decisions.md` or a decision record that: (a) names the change (adding 'closed' to terminal-status exclusion lists), (b) states the production evidence (768 rows, legacy pre-enum status), (c) states the authorization (technical necessity — these rows represent genuinely closed issues), (d) cites the vision.yaml anchor (e.g., `principle-1: proactive resilience over reactive recovery`). The ADR need not be long; the constraint is that it exists and is traceable.

Each gap must be resolved by addressing the named issue. Removal of an artifact without addressing the underlying concern is not resolution.
