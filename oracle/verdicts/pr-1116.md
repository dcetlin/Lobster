### Stage 2: Quality review

**Does the fix address the root cause?**

Yes. The prior implementation called `.replace("{uow_id}", uow_id).replace("{uow_id_display}", uow_id).replace("{D_JSON}", d_json)` on a template written with `str.format()` double-brace escaping. Python's `str.replace()` performs literal string substitution only — it does not process `{{ }}` escape sequences. The result was that all 106 `{{ }}` pairs in the template were passed through unchanged, making every JS brace literal (object literals, template literals, function bodies) syntactically broken. The rendered page was a white screen.

The fix is minimal and structurally correct: one `.format()` call at line 618, three named keyword arguments matching the three substitution points in the template. The `{{ }}` pairs are now processed as `str.format()` escape sequences and resolve to single `{ }` in the output.

**Reviewer's question: is the single `}}` at the end of the `@media` line a valid CSS closing pair, not a JS brace bug?**

Confirmed valid. The `@media` line reads:
```
@media(prefers-color-scheme:dark){{:root{{...}}}}
```

Tracing the `str.format()` escaping:
- `@media(prefers-color-scheme:dark){{` → `@media(prefers-color-scheme:dark){` (opening @media brace)
- `:root{{` → `:root{` (opening :root brace)
- `...properties...`
- `}}}}` → `}}` (two closing braces: one for `:root{...}`, one for `@media{...}`)

This is correct CSS. The `}}}}` at the template level resolves to `}}` at output, which is the syntactically required two-brace close for a nested `@media { :root { } }` block. Not a bug.

**Other structural issues from Round 1:**

Round 1 reviewer notes identified token regex, pricing, and substitution order as verified correct. These remain correct in the fix commit — the token regex (`_UOW_PATTERN`), pricing constants, and the format call order are all unchanged from the Round 1 passing verifications.

**Mirror-constant risk in tests (advisory, not blocking):**

The test file declares `SONNET_4_6_INPUT_PER_MTK = 3.0`, `SONNET_4_6_OUTPUT_PER_MTK = 15.0`, `SONNET_4_6_CACHE_READ_PER_MTK = 0.30` locally, and the production module declares the same constants at module scope under the same names. The test's `TestEstimateCost` assertions compare against the test-file-local constants (`assert abs(cost - SONNET_4_6_INPUT_PER_MTK) < 0.0001`) rather than importing from `wos_uow_detail_gen`. This is the learnings.md PR #696/PR #800 mirror-constant pattern — when the production pricing constants change, the test assertions will silently validate against the stale local values.

This is advisory rather than blocking for the following reason: this is a new module with no pre-existing production pricing constants to import from an older location. The constants are defined in the same PR as the tests. The drift risk is real but the immediate correctness is verified. A follow-on PR that adds a model pricing table or reprices Sonnet 4.6 should update the test imports at that time. This finding is logged to learnings.md.

**What failure modes exist?**

The fix closes the primary failure mode (JS invalid, white screen). Remaining failure modes visible in the implementation:

1. `_bisque_base_url()` has a `subprocess.run` fallback to `ifconfig.me` with a 6-second timeout. If the bisque relay is not running and the network call is slow, a CLI invocation blocks for up to 6 seconds before returning a potentially wrong IP. This is pre-existing design and not introduced by the fix.

2. The `generate_and_upload()` function does not verify that `d_json` contains no `{` or `}` characters that could interfere with `str.format()`. In practice, `json.dumps()` does not produce bare `{` or `}` in the JSON value for the `D_JSON` substitution point — JSON uses `{` and `}` as delimiters but they appear inside the string value, which by the time `str.format()` processes the template has already been substituted via `D_JSON=d_json`. The ordering is correct: `d_json` is a complete JSON string, and `.format(D_JSON=d_json)` substitutes it as a literal string value into the template. The template's `{D_JSON}` is a single substitution point, not a format specifier. This is structurally safe.

3. `token_usage` column in `uow_registry` is fetched as an integer in `_fetch_uow_data` but the UoW page displays it only in stat cards. The token breakdown uses the ledger-aggregated `token_data` dict, not the registry column. These two sources may diverge (registry stores current heartbeat, ledger stores per-call aggregation). This is a pre-existing architectural characteristic, not introduced by the fix.

This is a golden pattern application, not a new risk.
