## Stage 2: Quality Review

### Item 1 ŌĆö Local import of `_ESCALATE_SURFACE_EXECUTION_THRESHOLD` inside `_compute_suggested_action`

The PR uses a function-body local import (`from .dispatcher_handlers import _ESCALATE_SURFACE_EXECUTION_THRESHOLD`) inside `_compute_suggested_action`. This is the existing pattern: steward.py already has two prior local imports from dispatcher_handlers (lines 106 and 4883), both placed inside function bodies to avoid the circular dependency (dispatcher_handlers imports `ReturnReasonClassification` from steward at module scope; a top-level import in the other direction would deadlock at module load time).

The function-body local import is the correct call given the constraint. Moving the threshold to a shared constants module (e.g., `wos_constants.py`) would be the architecturally cleaner alternative, but that is a refactor outside this PR's scope and the local import pattern is already established. No action required on this point.

### Item 2 ŌĆö EscalationRecord positional construction in existing tests

No positional construction found. All existing test constructors use keyword arguments (`EscalationRecord(uow=uow, return_reason="executor_orphan")`). The new default fields (`execution_attempts=0`, `reentry_posture=""`, `audit_entries=()`) are backward-compatible. Clear.

### Item 3 ŌĆö Mirror-constant anti-pattern (sixth recurrence)

**Production-level mirror: `_HUMAN_JUDGMENT_REGISTERS` in steward.py**

This is the blocking issue. The PR introduces in steward.py:

```python
_HUMAN_JUDGMENT_REGISTERS: frozenset[str] = frozenset({
    "human-judgment",
    "philosophical",
})
```

The production constant in `dispatcher_handlers.py` is:

```python
_ESCALATE_HUMAN_JUDGMENT_REGISTERS: frozenset[str] = frozenset({
    "human-judgment",
    "philosophical",
})
```

These are the same values under different names in different modules. This is a true mirror-constant: if `"philosophical"` is dropped from `_ESCALATE_HUMAN_JUDGMENT_REGISTERS` in a future PR (or a new register is added), steward.py's local copy diverges silently. The PR comment acknowledges this explicitly: "Duplicated here to keep suggested_action a pure local computation without a cross-module import."

The intended functional impact of divergence is limited: `handle_wos_escalate` routes from first principles using `_ESCALATE_HUMAN_JUDGMENT_REGISTERS` ŌĆö the dispatcher does not read `suggested_action` from the message for routing. So a diverged `_HUMAN_JUDGMENT_REGISTERS` would produce a misleading hint in the wos_escalate message body without changing actual dispatch behavior. However, the `suggested_action` field is presumably used by callers of the message or in diagnostic contexts, and the misleading hint is a silent defect that CI cannot detect.

The inconsistency compounds the problem: `_ESCALATE_SURFACE_EXECUTION_THRESHOLD` is handled via local import inside the function body (correct), while `_HUMAN_JUDGMENT_REGISTERS` is mirrored (the pattern the local import is meant to avoid). The two constants sit side-by-side in the same function and are treated inconsistently with no stated reason for the difference.

The correct fix: remove `_HUMAN_JUDGMENT_REGISTERS` from steward.py and instead import `_ESCALATE_HUMAN_JUDGMENT_REGISTERS` via local import inside `_compute_suggested_action`, consistent with how `_ESCALATE_SURFACE_EXECUTION_THRESHOLD` is handled.

**Test-level constants in `test_steward_wos_escalate_write_path.py`**

The new test file declares:
- `WOS_ESCALATE_TYPE = "wos_escalate"` (line ~383)
- `ORPHAN_KILL_BEFORE_START_POSTURE = "orphan_kill_before_start"` (line ~386)
- `ORPHAN_KILL_DURING_EXECUTION_POSTURE = "orphan_kill_during_execution"` (line ~387)
- `EXECUTOR_ORPHAN_POSTURE = "executor_orphan"` (line ~388)

For `WOS_ESCALATE_TYPE`: this is the same situation as `WOS_ESCALATE_MESSAGE_TYPE = "wos_escalate"` in `test_wos_escalate_handler.py` ŌĆö the PR #970 oracle verdict ruled this acceptable because no named production constant for "wos_escalate" exists (it is a dict key literal in `WOS_MESSAGE_TYPE_DISPATCH`). However, there are now two test files with different local names for the same string. The PR adds a second local alias rather than unifying them. The absence of a named production constant is the root problem; this is advisory and does not block.

For `ORPHAN_KILL_BEFORE_START_POSTURE` etc.: these posture strings (`"orphan_kill_before_start"`, `"orphan_kill_during_execution"`) are not exported as named module-level constants ŌĆö they appear as string literals throughout steward.py. The test-local declarations are not importable because there is nothing to import. This is distinct from the mirror-constant pattern: there is no production constant to mirror. However, this creates an opportunity: promoting these strings to named module-level constants in steward.py would close both the test-local-literal issue and the `_ORPHAN_POSTURE_TO_KILL_TYPE` raw-literal issue simultaneously. This is advisory ŌĆö a follow-on cleanup opportunity, not a blocking gap.

### Item 4 ŌĆö Consolidated path asymmetry (intentional)

The consolidated escalation path (`>= ESCALATION_CONSOLIDATION_THRESHOLD`) continues to call `_send_consolidated_escalation_notification` (the old wos_surface path), while the individual path now calls `_write_wos_escalate_message`. The PR description explicitly documents this asymmetry as intentional: "a separate future PR." The `_send_consolidated_escalation_notification` function sends a single Telegram message listing all affected UoWs ŌĆö the wos_escalate design (one message per UoW carrying structured failure_history) does not compose naturally with consolidation. The asymmetry is architecturally coherent given the constraint. No action required.

### Item 5 ŌĆö `_build_wos_escalate_failure_history` assembly from audit_log entries

Resolution (c) is acceptable if the follow-on is filed as a tracked issue.

## Findings appendix

**Mirror-constant pattern recurrence (sixth):** learnings.md (PR #696, #800, #967, #970, #973) all named test-level mirrors. This PR adds a production-level mirror ŌĆö the pattern has escalated from test files into production code. The detection surface in learnings.md should be broadened: the mirror-constant check must apply to production module-scope constants as well as test-file constants.

**Suggested action field as hint-only:** `suggested_action` is computed pre-dispatch and embedded in the wos_escalate message. The dispatcher's `handle_wos_escalate` routes from first principles without reading `suggested_action`. This means the field is documentation only ŌĆö it helps observers reading inbox messages understand what routing is expected, but a wrong value does not cause a wrong dispatch. The field's hint-only status should be documented in `_write_wos_escalate_message`'s docstring so future readers understand it is not load-bearing.
