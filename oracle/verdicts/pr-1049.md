## Stage 1: Vision Alignment (formed before reading implementation)

**The adversarial prior:** This implementation is solving the wrong problem, or solving the right problem in a direction that forecloses better paths.

**What would have to be true for this work to be right:** The steward's prescriptions are currently suboptimal because they lack access to Dan's developmental orientation. Injecting a prose excerpt from user.base.context.md would close that gap in a durable way.

**The vision's actual position:**

vision.yaml `fundamental_intent` says: "The system should be able to answer 'is this aligned with Dan's intent?' without asking me — because it has structural access to my intent, not just a prose reconstruction of it." `constraint-2` is explicit: "Every agent decision that touches Dan's priorities must be traceable to a specific field in the Vision Object or a logged open decision — not inferred from conversational texture." `success_criteria[0]` says agents must point to a specific field in vision.yaml — not a paraphrase.

The "Lobster Developmental Map" section of user.base.context.md is a prose narrative describing Dan's developmental arc. It is not a machine-queryable field. It is not cited by field path. It is precisely "a prose reconstruction" of the intent the vision says the system should access structurally.

**What this work forecloses — partially:**

This PR does not prevent future structural access to vision.yaml. But it introduces a second orientation source with no stated authority relationship to vision.yaml. When user.base.context.md and vision.yaml drift — and they have different update disciplines — the steward has no mechanism to prefer one over the other, and no way to detect the divergence. The structural anchor (vision.yaml) and the prose anchor (developmental map) can silently conflict.

The vision's `what_not_to_touch` field flags "New detection or classification rules — improve Orient routing before adding more detection." This PR is not detection but it is analogous in structure: adding input richness before the structural orient substrate is built, which is what issue #733 is named for.

**Cheaper test not run:**

The PR description does not include any evidence that prescription quality is currently inadequate due to missing developmental context. The improvement is asserted, not measured. The 7 tests verify injection mechanics, not prescription quality.

**Pattern match from learnings.md:**

This PR exhibits the "design-spec-as-prerequisite bundled into implementation PR without completing the prerequisite design step" pattern (PR #906). The vision's own `horizon` field says "Orient implementation (#733) — the guardrail names the next structural investment." This PR implements part of that investment as a prose-injection approach rather than a structural seam, before the design step for Orient has produced a design artifact.

**Alignment verdict: Questioned**

The PR may produce real short-term improvement. But it implements the intent of #733 as a string-slice shortcut using a non-canonical proxy source rather than the structural seam vision.yaml anticipates. The risk is not that it makes things worse; it is that it fills the orient gap with a proxy approach, reducing urgency for the structural work while introducing a dual-source orientation problem.

## Stage 2: Quality Review

**The implementation is technically clean and minimally invasive.** The function is pure, gracefully degrades, and the injection is isolated to the input side of `_llm_prescribe` without touching output parsing. These are genuine virtues. The Stage 2 findings are not about quality; they are about specific gaps that must be resolved.

**Gap 1: Encoded Orientation without logged prior (constraint-3 blocker)**

`_llm_prescribe` is a behavioral default for how the steward generates every prescription. This PR changes that default by injecting developmental context into every prescription call — a durable change to agent behavior that fires without Dan's explicit input on each prescription cycle. This is Encoded Orientation under constraint-3.

The constraint is explicit: "Encoded Orientation decisions require a prior logged decision of the same class and a traceable vision.yaml anchor. When neither exists, escalate to the Deliberative register."

There is no entry in `oracle/verdicts/` or `oracle/decisions.md` authorizing this behavioral default. The PR closes issue #301 (Steward MVP umbrella from 2026-03), but that issue predates the current vision.yaml and the oracle/decisions.md constraint-3 mechanism. Closing an old umbrella issue is not authorization for an Encoded Orientation decision under current governance.

Per learnings.md PR #1032: "Fast-path classification bypass is an Encoded Orientation decision. The PR cited a WOS UoW artifact as authorization. Per the established PR #913 pattern, UoW artifacts are dispatch records, not decision records." The same applies here.

**Required to close Gap 1:** A logged decision in `oracle/decisions.md` that: (a) names this as an Encoded Orientation change, (b) cites a specific vision.yaml field path as the anchor, (c) states the resulting behavioral default, and (d) addresses the dual-source drift risk (user.base.context.md vs. vision.yaml as concurrent orientation sources for the steward).

**Gap 2: Systematic truncation at the stated cap (implementer-flagged, structural)**

The production file's developmental map section is ~6,888 chars. The cap is 1,500 chars. The excerpt will always truncate at roughly 22% of the section, cutting through the "Capability ceiling and Embodiment distinction" subsection (added 2026-04-22) and never reaching "Attentional Budget Constraint" or "Capability Coupling Structure" — both of which are the more structurally relevant orientation signals for prescription quality.

The implementer flagged this. But the PR ships anyway with a cap that systematically truncates the most load-bearing content. This is not advisory — it means the feature will always inject an incomplete, potentially mid-sentence fragment of the orientation document in production.

The tests do not detect this because they use synthetic short content. A test that loads `user.base.context.md` from the actual file path (when present) and asserts that at least the "Capability ceiling" subsection is included would catch this immediately.

**Required to close Gap 2:** Either (a) raise the cap to a value that includes the complete developmental map on production (8,000 chars would safely cover it), or (b) implement section-aware truncation that stops at the last complete markdown section boundary rather than at a character count. The current implementation always sends a truncated mid-section excerpt in production.

**Gap 3: Anchor string robustness (minor, advisory)**

The production code anchors on `"## Lobster Developmental Map"`. The actual file heading is `"## Lobster Developmental Map (Theory of Learning, 2026-03-26)"`. This works correctly via prefix match in `str.find()`. However, if the section heading is ever edited (e.g., the date is removed, or the section is renamed), `find()` returns `-1` and the injection silently disappears. There is no test that uses the exact production heading string — `test_returns_excerpt_starting_at_anchor` uses `"## Lobster Developmental Map (Theory of Learning)\n\n"` which is different from both the anchor and the actual heading. This is advisory: the anchor works today, but the test does not validate it against the actual file's heading.

**Gap 3 resolution:** Either add a test that uses the exact heading string from the production file, or document in the function docstring the exact heading the anchor must match — making heading changes visible to future editors.

**Encoded Orientation check (constraint-3):**

Yes, this PR constitutes an Encoded Orientation decision — it changes behavioral defaults in `_llm_prescribe`, a durable production function that runs on every steward prescription cycle. No logged prior exists in `oracle/verdicts/` or `oracle/decisions.md`. No vision.yaml field path is cited as the anchor. This fails constraint-3. Gap 1 is the blocker.

**Patterns introduced:**

This PR introduces the pattern of prose narrative from user.base.context.md as a prompt-engineering input to production agent functions. Future implementations may extend this to other agents without recognizing that user.base.context.md is a proxy, not the canonical structural source. If this ships without a logged decision, the pattern will propagate (following the mirror-constant pattern history in learnings.md — things that are done once become templates).

**What this makes easier:**

Steward prescriptions will have access to Dan's developmental orientation immediately. For short-term ergonomics this is a real gain.

**What this makes harder:**

The structural seam for Orient (#733) becomes less urgent once prescriptions are "good enough" via prose injection. The dual-source problem becomes harder to articulate once the prose approach is operational. Removing the prose injection in favor of a vision.yaml-native approach later requires a change to an already-running production system.
