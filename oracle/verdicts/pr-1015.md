## PR #1015 — fix(wos): repair intake dedup to prevent duplicate UoW rows

### Stage 1 — Right Problem?

Two genuine gaps are identified and closed:

**Gap 1 — Hardcoded terminal status list in `_upsert_typed`**: The cross-sweep-date dedup query had hardcoded strings `('done', 'failed', 'expired', 'cancelled', 'closed')` while `has_active_uow_for_issue()` used `_TERMINAL_STATUSES_FOR_ISSUE_CHECK`. Both currently contain identical values, but the duplication is a maintenance hazard — any future addition to the frozenset would silently diverge from the upsert dedup check. Using the frozenset as a single source of truth is the correct fix.

**Gap 2 — TOCTOU race in `GardenCaretaker._reactivate_uow`**: The old code performed `has_active_uow_for_issue()` on connection A (closed) then `set_status_direct()` on connection B. A concurrent intake process between the check and write could produce two proposed UoWs for the same issue. `BEGIN IMMEDIATE` on a single connection eliminates this window.

Both gaps are real, the analysis is correct, and the fixes address the root cause.

### Stage 2 — Well Made?

**`_upsert_typed`**: Replaces hardcoded strings with `_TERMINAL_STATUSES_FOR_ISSUE_CHECK` via f-string parameterized placeholders. Pattern is identical to `has_active_uow_for_issue()`. The existing `BEGIN IMMEDIATE` transaction is preserved. Correct.

**`Registry.reactivate_if_no_active(uow_id, issue_number) -> bool`**: New method. Uses `BEGIN IMMEDIATE` from the start (write lock, no TOCTOU). Checks for conflict first (returns `False` + commit if found). Checks UoW exists (returns `False` + commit if missing). Writes structured audit via `_write_audit`, then UPDATE, then commit. On exception: rollback + re-raise. Returns `True` on success. Pattern is consistent with existing registry transaction patterns.

**`GardenCaretaker._reactivate_uow`**: Four paths:
1. `issue_number is None` — fallback to `set_status_direct` (rare: malformed source_ref). Preserved and documented.
2. Exception from `reactivate_if_no_active` — logged as warning, returns 0. Safe.
3. `reactivated = False` — dedup guard fired, logs and appends audit. Returns 0. Correct.
4. `reactivated = True` — appends `reactivated_by_caretaker` audit, returns 1. Correct.

Dual audit entries on success (one `status_change` inside `reactivate_if_no_active`, one `reactivated_by_caretaker` in caretaker) are intentional — different semantic levels. Not a duplication error.

**Tests**: `TestAtomicReactivation` covers 3 cases (success when no conflict, skipped when conflict exists, False for missing UoW). Two new registry unit tests cover dedup on active/terminal status. Test assertions are direct and correct.

No concerns. The fix is precise, well-bounded (additive method, no signatures changed), and tested.
