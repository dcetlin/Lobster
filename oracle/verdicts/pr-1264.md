### Stage 1: Vision Alignment (formed before reading diff)

**Prior entering review:** This implementation is solving the wrong problem, or solving the right problem in a direction that forecloses better paths.

**Evidence-gathering questions:**

*What would have to be true for this work to be the right path?*
The deduplication gate must be solving a problem that is actually causing harm. The PR itself concedes: "No code queries `events` for `type='pattern_observation'` to consume it downstream." If there is no downstream consumer, the 528 accumulated health-check-v3 entries the PR cites as the triggering problem are noise in a column no one reads. The gate prevents future noise accumulation — but the existing accumulation remains untouched and the thing being protected (downstream consumption) does not yet exist.

*What does this foreclose?*
The 12-hour dedup window makes a permanent classification decision: a pattern_observation firing within 12 hours of an identical one is noise, not signal. This is an architectural decision about what "signal" means in the pattern observation system, encoded in production code without a logged design decision.

*Is there a cheaper test of the underlying assumption?*
Yes: a one-time SQL DELETE of historic health-check-v3 pattern_observation entries addresses the accumulation problem without any code change. The dedup gate is prevention for a future that doesn't yet exist (downstream consumption). The PR did not evaluate this alternative.

*What is the opportunity cost?*
vision.yaml `what_not_to_touch` explicitly states: "New detection or classification rules — improve Orient routing before adding more detection." The dedup gate is a new classification rule: it classifies whether a pattern observation should be recorded. This lands squarely within the stated exclusion zone.

*Feature-bundling assessment (mechanical check, first-pass):*
The PR contains two commits from two distinct WOS UoWs:
- `uow_20260522_c2b5a0`: HYPOTHESIS section bootup protocol — modifies CLAUDE.md, `.claude/sys.dispatcher.bootup.md`, `.claude/sys.subagent.bootup.md`, `.claude/agents/lobster-hygiene.md`
- `uow_20260522_121905`: dedup gate in `slow_reclassifier.py` — modifies `src/classifiers/slow_reclassifier.py`, adds `tests/unit/test_slow_reclassifier_dedup.py`

The PR title and description cover only the dedup gate (uow_20260522_121905). Four bootup/agent files changed by uow_20260522_c2b5a0 are not mentioned in the PR description at all. This is a feature-bundling violation per learnings.md 2026-05-14 (#1148): "Each workstream should be a separate PR so that review verdicts do not cross-block unrelated changes." The learnings.md threshold is explicit: "distinct workstreams = NEEDS_CHANGES."

**Stage 1 Alignment Verdict: Questioned**

The dedup gate may be technically sound but is constrained by `what_not_to_touch` ("New detection or classification rules"). The feature-bundling violation is a structural block independent of substantive quality. The Stage 1 prior ("solving the wrong problem") is partially confirmed for the bundling dimension (the PR is solving two problems, one of which it doesn't describe) and conditionally confirmed for the dedup gate dimension (the gate protects a downstream consumer that does not yet exist, and the accumulation problem has a cheaper fix).

### Stage 2: Quality Review

Reading the diff with Stage 1 findings locked.

**Dedup gate (src/classifiers/slow_reclassifier.py)**

The implementation is technically clean. `_recent_duplicate_exists()` uses a parameterized SQL query on `source + pattern_type` with a `timestamp >= cutoff` bound. It returns early with a single `LIMIT 1` row. `json_extract(metadata, '$.pattern_type')` is the correct accessor for a JSON metadata column. The check is placed before `write_pattern_event()` and the tag revision loop correctly remains outside the guard — contributing events still get revised tags even when the event write is skipped. This is a load-bearing design choice and it is correct.

The `hours` parameter defaults to 12 and is hardcoded at the call site: `_recent_duplicate_exists(conn, pattern)`. This means the 12-hour window is a magic number at the call site — the function accepts `hours` as a parameter (correct seam), but the call site uses the default without naming it. Per learnings.md 2026-04-26 (#968): "any time-window comparison in an orphan classifier must use a named constant." By analogy: a 12-hour dedup window is a policy threshold that belongs as a named module-level constant (`_DEDUP_WINDOW_HOURS = 12` or similar), not an implicit default. If the policy changes, the call site provides no search anchor — only the function signature does.

**Test file (tests/unit/test_slow_reclassifier_dedup.py)**

Four tests, all correctly scoped: recent duplicate → True; no entry → False; old entry >12h → False; different pattern_type → False. In-memory SQLite schema matches the production table shape (id, timestamp, type, source, content, metadata). The tests use actual datetime arithmetic rather than mocked time — this is correct for a test that checks timestamp comparison logic. `_make_obs` constructs a `PatternObservation` with a `signal_type` parameter (`"design_session"`) that mirrors `pattern_type` — this tests the actual detection correctly.

One gap: `test_dedup_gate_allows_when_entry_is_old` uses `hours_ago=13` against a 12-hour window, which gives a 1-hour margin. This is acceptable but borderline — a test at exactly `hours=12.0` (the boundary) would be more precise. Not a blocker.

No mirror constants in the test file — the function is imported directly from production. This is correct per learnings.md.

**HYPOTHESIS bootup protocol (CLAUDE.md + bootup files)**

The HYPOTHESIS section protocol is a well-formed structural improvement. The poiesis→poiema graduation framing is coherent with the vision's vocabulary. The `# HYPOTHESIS` placeholder sections added to `sys.dispatcher.bootup.md` and `sys.subagent.bootup.md` are empty at ship time (correct — they are structural scaffolding, not content). The hygiene agent Step 2b is correctly scoped to surface only, not graduate — graduation remains a Dan decision.

This means the HYPOTHESIS commit may be describing something that already exists in production. If the HYPOTHESIS protocol was already in CLAUDE.md before this PR, then this commit is a re-encoding of something already live — which changes the Encoded Orientation analysis from "new behavioral default without prior" to "documentation of existing configuration."

Let me assess what is actually new:

The `check_routing_preferences` function exists in `dispatcher_handlers.py` but there is no oracle decision for the routing preferences feature. - The "Routing Preferences" section added to `sys.dispatcher.bootup.md` — this encodes a new behavioral instruction: "before routing a user message, check routing preferences." This is also a new Encoded Orientation decision.

The routing preferences additions within the HYPOTHESIS commit are an Encoded Orientation decision without a logged prior or explicit vision.yaml anchor. The `check_routing_preferences` function exists in production code but has no oracle lineage.

**What patterns does this introduce that will propagate?**

The PR commits a routing preferences instruction section to `sys.dispatcher.bootup.md` without a corresponding oracle decision file. If this ships, future PRs will treat routing preferences as an established, oracle-approved feature when it has never been reviewed as a standalone decision.

**What failure modes exist?**

If a real signal fires twice within 12 hours (a genuine recurring design session, rapid-fire user mode changes), the second observation is silently suppressed with only a debug log. No observability beyond the debug log — no counter, no metric, no surfacing to Dan.

### Revision Contract

**Gap 1: Feature-bundling — two distinct WOS UoWs in one PR (NEEDS_CHANGES blocker)**

The PR bundles `uow_20260522_c2b5a0` (HYPOTHESIS bootup protocol) and `uow_20260522_121905` (slow_reclassifier dedup gate) in a single PR. The PR description covers only the dedup gate. Resolution: split into two PRs, one per UoW, so each can be independently reviewed and merged.

If splitting is impractical (commits are already merged to the branch), resolution can be: (a) update the PR description to explicitly cover both workstreams with their own summary sections, or (b) acknowledge the bundling as a deliberate exception with a stated reason. Either resolves this gap.

**Gap 2: Encoded Orientation without logged prior — Routing Preferences section (NEEDS_CHANGES blocker)**

The `sys.dispatcher.bootup.md` additions include a "Routing Preferences" section and callback dispatch wiring for `routing_pref_confirm:`/`routing_pref_reject:` that change dispatcher behavioral defaults. No prior logged oracle decision exists for the routing preferences feature. Resolution: (a) create a decision file in `oracle/verdicts/` for the routing preferences feature with a vision.yaml anchor, or (b) move the routing preferences section to a HYPOTHESIS entry (which is appropriate given this PR introduces the HYPOTHESIS pattern), or (c) demonstrate a prior decision exists by citing it.

**Gap 3: Named constant for 12-hour dedup window (advisory — does not block approval alone)**

The `_recent_duplicate_exists(conn, pattern)` call site uses the 12-hour default implicitly. A module-level constant (`_DEDUP_WINDOW_HOURS = 12`) with a comment naming the policy intent is the correct encoding per learnings.md (#968). This is advisory — it does not block approval if Gaps 1 and 2 are resolved, but the fix is a one-line addition.

**Gap 4: `is_explicit_command` pseudocode reference in dispatcher bootup (advisory)**

The routing preferences section includes `if agent_hint and not is_explicit_command(msg["text"]):` as pseudocode. This function does not exist in the codebase. If a dispatcher follows this as a code example, it will produce a NameError. The section should use a comment-style conditional or reference the actual pattern the dispatcher should use (e.g., check whether `msg["text"].startswith("/")`). Advisory — does not block if the routing preferences section is moved to HYPOTHESIS or backed by a decision (Gap 2 resolution).
