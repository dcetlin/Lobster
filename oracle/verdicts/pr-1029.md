### StrEnum golden pattern check

The fetch_for_latency SQL uses `WHERE started_at >= ?` and `AND status = ?` with user-supplied string values bound via parameterized query. No raw string enum literals are embedded in the SQL itself — the status value is externally supplied and parameterized. This is structurally consistent with how `cmd_list` works (registry.list accepts raw string status). The `--status` argparse argument does not include `choices=` constraint, while the `list` command does. This is an inconsistency flagged by the implementer and worth noting, but it is consistent with the stated design choice ("consistent with how the list command works" in the PR body) and not a new pattern — the inconsistency is preexisting in different form. Not a blocker; advisory.

### `_percentile` function review

The implementer specifically requested close review. Walkthrough:

- **n=1**: `return sorted_values[0]` — correct. Single-value list returns that value for any percentile.
- **General case**: `rank = (p / 100.0) * (n - 1)`. For p=100, n=5: rank=4.0, lower=4, upper=5. `upper >= n` (5 >= 5) → returns `sorted_values[-1]` = sorted_values[4]. Correct.
- **Linear interpolation**: `sorted_values[lower] + fraction * (sorted_values[upper] - sorted_values[lower])`. This is exactly numpy's default 'linear' interpolation. Correct.
- **Edge: p=0**: rank=0, lower=0, upper=1, fraction=0.0 → returns sorted_values[0]. Correct.
- **The upper >= n guard**: rank can be at most (n-1) since p<=100, so lower can be at most n-1, and upper can be at most n. The guard is correct and necessary.

The `_percentile` implementation is correct. The n=1 path is covered by `test_rows_without_completed_at_excluded_from_execution_time` (single row, p50==300 asserted). The upper-bound clamping is sound.

### `fetch_for_latency` filter semantics

Filters `WHERE started_at >= since_iso`. UoWs never started (proposed, pending) have NULL started_at, and SQLite NULL comparisons return false — these rows are excluded. This is intentional and correctly documented. The filter window is the `started_at` timestamp, not `created_at`, meaning a UoW created before the window but started within it will be included. This is correct for the stated purpose: measuring latency for UoWs that were actually dispatched in the window.

The status filter is a raw string pass-through with no enum validation. Invalid status values return 0 rows silently — the implementer flagged this explicitly. Since `list` also passes raw strings (though with argparse choices as a guard), the absence of choices= on `queue-latency --status` is a minor inconsistency. It does not affect correctness; it is an ergonomics gap.

### Test review

**Test quality:** 25 tests across three classes (TestComputeLatencyStats, TestFormatLatencyReport, TestQueueLatencyCommand). Coverage is substantive for a pure-function CLI command.

**Dead code in test_percentiles_correct_for_known_dataset:**

This is the "dead code immediately before unconditional override" pattern from learnings.md PR #848. The first `rows` list comprehension (lines 325-334 in the diff) is constructed with `if False else None` for started_at — an artifact of the drafting process. It is immediately overwritten by `rows = []` and the subsequent loop. `base_created = "2026-01-01T10:00:00+00:00"` is also declared and never used. This dead code introduces noise and the `if False else None` expression is misleading. However, neither affects test correctness: the operative rows are built in the explicit loop that follows, and the test passes correctly against the real computation.

The dead construction is present in a test, not production code. It does not introduce a synchronization risk or a semantic trap. Per the behavioral posture established in learnings.md PR #848 ("dead code immediately before an unconditional override produces misleading intent signals"), this is worth naming but is not a blocker for a test method.

**Missing test for negative delta case:** The docstring on `_parse_seconds_between` documents that negative values are returned for reversed timestamps and "callers decide how to handle." The caller (`_compute_latency_stats`) guards with `if qw is not None and qw >= 0`. No test exercises a row where `started_at < created_at` (reversed anomaly). The guard is correct, but without a test, a future refactor that removes the `>= 0` guard would not be caught. Advisory, not a blocker.

**Assertion in test_queue_latency_uows_without_started_at_excluded:** The final assertion is `assert output` — it verifies the command ran, not that the UoW was excluded from count. The test name promises exclusion verification but the assertion only verifies non-empty output. This is a softened assertion per learnings.md PR #839 ("assertion softening to 'not-legacy-value'..."). However, the exclusion behavior is mechanically guaranteed by the SQL WHERE clause (`started_at >= since_iso` with NULL started_at always excluded), and the unit tests directly verify count==0 for missing started_at. The integration-test assertion is weaker than named, but the behavior is covered at the unit level.

### What patterns does this introduce that will propagate?

The `_stats` inner function pattern (local closure that computes stats dict from a list) is clean and reusable within this module. The plain-text alignment-column rendering approach (f-strings with fixed labels) is consistent with the rest of `registry_cli.py`'s output conventions.

The `--status` without `choices=` inconsistency, if not corrected before similar subcommands are added, will normalize the pattern of silent-zero-rows for invalid status values. This is the pattern-propagation risk to watch.

### What this makes easier / harder

Easier: diagnosing queue throughput problems — a question that is currently unanswerable without raw DB queries. The decomposition into queue_wait vs. execution_time is the correct cut; conflating them in a single wall-clock number hides which half is the bottleneck.

Harder: nothing significant. The subcommand is additive.
