### Gap 1: SQL defect in render_active_todos — snoozed items always shown regardless of snooze_until

**Severity: NEEDS_CHANGES — functional defect**

`render_active_todos` contains a SQL query with a tautological OR clause:

```sql
WHERE status IN ('open', 'snoozed')
   OR (status = 'snoozed' AND snoozed_until < datetime('now'))
```

The second OR branch (`status = 'snoozed' AND snoozed_until < datetime('now'))`) is entirely subsumed by the first (`status IN ('open', 'snoozed')`), since any row matching the second branch already matches the first. The net effect: all snoozed items are included unconditionally, regardless of whether their `snoozed_until` is in the future.

The intent — confirmed by the function docstring ("Renders items with status IN ('open', 'snoozed')") and by the parallel `find_open_items` function in db.py (line 249) which implements the correct pattern — was to include snoozed items only when their snooze period has expired.

If Dan snoozes a todo item until next week via Telegram (`mark_snoozed(conn, id, "2026-05-17")`), the item will continue to appear in every regenerated `ACTIVE TODOS.md` file through the sync cycle, defeating the snooze.

**Correct query** (matches the `find_open_items` pattern in db.py):
```sql
WHERE status = 'open'
   OR (status = 'snoozed' AND snoozed_until < datetime('now'))
```

No test catches this because no test asserts that a snoozed-with-future-date item is absent from the render output. The revision contract requires:
1. Fix the SQL predicate
2. Add a test: insert an item, call `mark_snoozed(conn, id, future_date)`, call `render_active_todos(conn)`, assert the item is NOT in the output

### Advisory findings (not requiring NEEDS_CHANGES)

**_apply_migrations comment inaccuracy:** The comment above `_SCHEMA_MIGRATIONS` says "SQLite silently no-ops duplicate ADD COLUMN via the IF NOT EXISTS guard we implement in _apply_migrations." There is no `IF NOT EXISTS` clause — the guard is a Python-level PRAGMA table_info check. The implementation is correct; the comment misattributes the mechanism to SQLite rather than the Python wrapper. Should be corrected to: "skipped via PRAGMA table_info pre-check."

**_apply_migrations column extraction (engineer-acknowledged):** `sql.split("ADD COLUMN")[1].strip().split()[0]` is brittle if a migration statement contains `ADD COLUMN` twice. This is safe for the current 6 statements, which are all simple `ALTER TABLE ... ADD COLUMN <name> <type>`. The risk is forward-looking: a future migration with compound syntax would silently extract the wrong column name and skip the column check. Adding a docstring note explicitly banning compound ADD COLUMN statements in `_SCHEMA_MIGRATIONS` would prevent the failure class without changing the implementation.

**Module-level env var computed at import time (PR #839 pattern):** `_WORKSPACE = Path(os.environ.get("LOBSTER_WORKSPACE", ...))` is evaluated at module import time. Tests that call the script's functions directly (as this test file does) cannot override `_WORKSPACE` via `monkeypatch.setenv` because the constant is already frozen. This affects only `_is_job_enabled` and argparse defaults — neither is exercised by the test suite, which calls `sync_obsidian_to_db` and `render_active_todos` directly. Advisory because the affected code paths are not tested and the pattern does not cause silent test passing.

**Vault git remote not yet configured (engineer-acknowledged):** The primary integration path (Mac edits → vault pull on VPS → sync to DB) is dormant until the vault remote is wired. The script handles this gracefully (no-op). Not a code defect, but the feature is delivering no production value until the remote is configured. The `_git_pull` and `_git_commit_and_push` no-op paths are tested implicitly only by their absence of side effects.
