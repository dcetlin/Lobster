- **Learnings PR #827 (stale assertion after behavior removal):** Reading this pattern caused me to search for existing tests that assert the old `_uow_id_cell` behavior (pre-generated URLs in the href). With the pattern, I checked whether the tests assert the *removed* behavior. This produced Gap 2.

### Gap 1 — Dead fallback branch in `_uow_id_cell` (BLOCKING)

In `src/orchestration/wos_dashboard.py` lines 643–647:

```python
url = f"/wos/uow/{uow_id}"
legacy_url = drilldown_urls.get(uow_id)
if legacy_url and not url:
    url = legacy_url
```

`url` is always truthy — it is assigned a non-empty f-string on line 644. The condition `not url` is always `False`. The `url = legacy_url` branch is permanently unreachable dead code.

The docstring says "Falls back to a legacy drilldown_urls URL if present." This claim is false — the fallback never executes. Any caller passing a legacy URL will observe the dynamic route regardless. This is the learnings.md PR #848 pattern: dead code before an unconditional override produces misleading intent signals. The comment that precedes the block compounds the confusion: "Prefer the stable dynamic route; fall back to pre-generated static URL if provided" — the code cannot implement the fallback it describes.

**Resolution options (required, not optional):**
- (a) Remove the dead branch and update the docstring to state that `drilldown_urls` is now ignored: "Always links to the stable /wos/uow/<uow_id> route. The drilldown_urls parameter is accepted for backward compatibility but not used."
- (b) Invert the logic so the legacy URL takes precedence when provided: `url = drilldown_urls.get(uow_id) or f"/wos/uow/{uow_id}"`. This preserves the documented fallback semantics.

Option (a) matches the stated intent of the PR description ("always links to the stable route"). Option (b) is only correct if the caller ever passes meaningful legacy URLs, which the PR description suggests is no longer the intended use.

### Gap 2 — Stale test assertions after behavior change (BLOCKING)

The PR changes `_uow_id_cell` to always produce a link. Two existing tests in `tests/unit/test_orchestration/test_wos_dashboard.py` assert the old behavior and will fail:

**`test_drilldown_link_rendered_when_url_provided` (line 617):**
```python
urls = {"uow_20260101_abc": "http://test:9101/files/abc.html"}
html = render_html(self._base_data(), drilldown_urls=urls)
assert 'href="http://test:9101/files/abc.html"' in html
```
Under the new code, the cell href is `/wos/uow/uow_20260101_abc`, not `http://test:9101/files/abc.html`. This assertion will fail.

**`test_drilldown_links_for_stalled_uow` (line 655):**
```python
urls = {"uow_stalled_x": "http://test:9101/files/stalled.html"}
html = render_html(data, drilldown_urls=urls)
assert 'href="http://test:9101/files/stalled.html"' in html
```
Same failure: the href is now `/wos/uow/uow_stalled_x`.

These are stale assertions per learnings.md 2026-04-22 (PR #827): when a PR removes a behavior, delete or invert any test that asserts the removed behavior before merge.

A third test passes accidentally:
**`test_no_link_when_no_drilldown_url` (line 625):**
```python
html = render_html(self._base_data(), drilldown_urls={})
assert "uow_20260101_abc" in html
assert "http://test:9101/files/" not in html
```
This passes because it checks the absence of `http://test:9101/files/` (an absolute URL) while the new code produces a relative URL `/wos/uow/...`. The test docstring is now incorrect ("UoW ID appears as plain text — no href link") — but the assertion itself happens to pass. This test should be updated to document the actual new behavior: the cell always has a link, it is always the dynamic route `/wos/uow/<id>`.

**Required:** Update all three tests to assert the new behavior before the PR can be approved.

### Gap 3 — Private API coupling (advisory, non-blocking)

`relay_server.py` invokes 9 private functions of `wos_uow_detail_gen` (prefixed `_`). These are module-private by convention. The seam-first golden pattern (2026-03-30) would suggest placing a named public composition function in `wos_uow_detail_gen` (e.g., `generate_html_for_uow_id(uow_id, db_path, ledger_path) -> str | None`) that encapsulates the fetch-and-render pipeline. This would give `relay_server.py` a stable public seam to call and would make `wos_uow_detail_gen`'s internal refactors invisible to the caller.

This gap is advisory. The codebase is a single-owner system with no external consumers of these APIs. Not blocking.

### Gap 4 — Lazy sys.path insertion at request time (advisory, non-blocking)

The handler inserts the src path into `sys.path` on each request (guarded by a presence check). This pattern exists elsewhere in scheduled tasks but is atypical for a web request handler. The module-level import style used in the rest of `relay_server.py` (`from bisque.auth import ...`) is the established pattern. A module-level import of `wos_uow_detail_gen` (or a factory constructed once at server startup) would be cleaner.

The guard prevents duplicate insertion, so there is no functional defect. Advisory.

Two blocking gaps must be resolved: the permanently-unreachable fallback branch (Gap 1) and the stale test assertions (Gap 2).
