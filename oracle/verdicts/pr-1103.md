## Stage 1: Vision Alignment (before reading implementation)

**Prior entering this review:** This implementation is solving the wrong problem, or solving the right problem in a direction that forecloses better paths.

**Theory of change test:** The task this PR addresses — "abort a running WOS subprocess" — is an operational safety valve. The vision's operating principle-1 names "proactive resilience over reactive recovery." A kill command is a reactive recovery tool. The question Stage 1 must hold: is adding a reactive abort command the right problem, or is the right problem reducing the conditions that make abort necessary (runaway subagents, infinite loops, lack of TTL precision)?

**What would have to be true for this to be the right path:**
- TTL and heartbeat recovery are working but take too long in some cases, and a targeted fast-path kill genuinely reduces operational burden
- The kill path can't be eliminated by improving TTL precision or adding a timeout parameter to `Popen`
- Dan actually needs to abort in-flight UoWs on demand, rather than just wanting faster TTL

**What this forecloses:** Once a manual abort command exists, there's reduced pressure to improve the automatic recovery system. The "wait up to 24 hours for TTL recovery" framing that justifies this PR points to a TTL duration problem, not a missing kill mechanism. If TTL were 2 hours instead of 24, would this PR be necessary?

**Cheaper test not run:** Reduce TTL for executing UoWs, or add a configurable `abort_after_seconds` timeout. Both would be proactive-resilience solutions rather than reactive-abort solutions.

**Alignment verdict:** Questioned (not Misaligned — the abort command is genuinely useful even if TTL were improved; it gives Dan direct control over the pipeline, which serves the vision's "cognitive extension" theme. The constraint is that the PR's stated justification — "no way to stop it short of 24-hour TTL" — points at a TTL configuration gap that this PR does not address and arguably reduces pressure to address.)

Despite the Questioned alignment, the PR's stated scope is reasonable. A manual abort command is a legitimate operational tool, and the vision explicitly values Dan's ability to issue directives to the system. Proceeding to Stage 2.

**Pattern match from learnings.md:** The "library with no wiring" pattern (PR #714 learning) is the primary pattern active here — it names the situation where a new capability is added to the library layer but no production call site is connected. This constrained my Stage 2 analysis significantly: I looked specifically for whether `set_executor_pid` is called anywhere in the production dispatch path, because the PR description explicitly flags this as the known gap.

## Stage 2: Quality Review

### What this PR does

Adds four Registry methods (`set_executor_pid`, `get_executor_pid`, `clear_executor_pid`, `kill_executor`), one dispatcher handler (`handle_wos_abort`), one parser (`parse_wos_abort_command`), a SQL migration (0020), an upgrade.sh migration (103), and 19 unit tests.

### Encoded Orientation check (constraint-3)

Does this PR change behavioral defaults, system constraints, agent identity, or decision-making rules durably? The `wos abort` command is a new operator capability added to the command routing table. It does not encode an automated behavior — it requires explicit Dan input to execute. This is Reactive register (Dan initiates → system responds). No prior logged decision or vision.yaml anchor is required for a new user-command handler. Constraint-3 does not fire.

### The blocker: set_executor_pid is never called

The PR description states explicitly: "The PID write at dispatch time (in executor.py after Popen) is NOT included." This is the central finding. The infrastructure is complete and correct — schema, registry methods, handler, tests. But `set_executor_pid` is never called at dispatch time. Every `wos abort <uow_id>` command will return "no running process found (executor_pid is not set)" for every UoW, because no producer populates the field.

This is not a documentation gap or a future-enhancement note. It is a structural incompleteness: the command exists, users can invoke it, and it will consistently give the wrong answer. The handler docstring says "the killed subprocess will fail to write its result file, and TTL/heartbeat recovery will detect it" — but that path is never reached because `kill_executor` always returns False (no PID stored).

The PR description frames this as "infrastructure exists, PID write is follow-on to minimize risk surface." This framing is self-consistent but produces a shipped command that never works as described. The "After" behavior in the PR description:

```
→ (if PID stored) Aborted: SIGTERM sent to process group of PID 99999.
```

...cannot occur in any production deployment until the follow-on PR ships. The only reachable outcome on production is the "if no PID" branch.

**This is a blocker.** The command exists in the module docstring as a callable user command. The dispatcher will parse and route it. Dan will use it. It will silently fail (reporting "no PID" when a PID exists but wasn't written). This is worse than no command — it gives false confidence that abort was attempted.

**Required resolution:** Either (a) include the `set_executor_pid` call in `executor.py` at dispatch time in this PR, or (b) make the `wos abort` command explicitly non-functional until the follow-on PR is merged (add a clear message: "wos abort: PID tracking not yet active — executor.py not updated. Command will be available in PR N."), or (c) remove `parse_wos_abort_command` and `handle_wos_abort` from the dispatcher routing table and module docstring until the follow-on ships.

### PermissionError: should it be caught?

The PR asks: "kill_executor catches only ProcessLookupError — should PermissionError also be handled?"

Answer: Yes, but with different semantics than ProcessLookupError. `PermissionError` from `os.killpg` means the process exists but is owned by a different user. In the Lobster deployment model (single-user systemd service), this should not occur — all subprocesses are spawned by the same user account. However, if it does occur, the current code will raise an unhandled exception that propagates out of `kill_executor` and into `handle_wos_abort`, producing an internal error rather than a user-readable reply.

The correct handling: catch `PermissionError` separately, clear the PID (since we can't kill it), and return a specific error message: "Process PID {pid} exists but cannot be killed (PermissionError). This is unexpected — subprocesses should be owned by the same user." Do NOT clear the PID on PermissionError — the process is still running.

Actually, on reflection: clearing the PID on PermissionError would be incorrect (the process is still running). The stale PID should be left in place so the user knows there is an active process they can't kill. The correct behavior is: log the error, return a message describing the situation, do NOT clear the PID. This is distinct from ProcessLookupError (process gone, always clear the PID).

This is a required change, not advisory. An unhandled `PermissionError` from `os.killpg` will surface as an unhandled exception to the dispatcher. The handler must be defensive.

This creates a false-negative experience that is harder to debug than a clear error message.

## Gaps requiring resolution before APPROVED

**Gap 1: set_executor_pid never called (blocker)**
The `wos abort` command is unreachable in its success path because no code calls `registry.set_executor_pid()` at dispatch time. The command will always report "no running process found" for any UoW dispatched via executor.py. Resolution options: (a) add the `set_executor_pid` call to executor.py in this PR; (b) mark the command as non-functional pending follow-on and communicate this explicitly in the user-visible reply; (c) remove the command from the module docstring routing table until the follow-on ships.

**Gap 2: PermissionError unhandled in kill_executor (required)**
`os.killpg` raises `PermissionError` when the target process group is owned by a different user. This exception is not caught and will propagate out of `handle_wos_abort`, producing an internal error instead of a user-readable reply. The fix: catch `PermissionError` in `kill_executor`, log it, do NOT clear the PID (the process is still running), and return False with the error captured for the caller to surface. The handler should then produce a "cannot kill: permission denied" reply. A corresponding test must be added.
