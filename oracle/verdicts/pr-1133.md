**Label-only form and ATTRIBUTION_LINE_RE — fix agent's question resolved**

The fix agent flagged that the label-only form `[telegram msg]` (no middot) does not match `ATTRIBUTION_LINE_RE`, and asked whether this is acceptable. It is.

The regex comment documents the exclusion explicitly:
> "the label-only form '[telegram msg]' (no middot, produced when created_at_iso is absent) is not matched here but is idempotent: re-rendering produces the same string."

The idempotency claim is correct in the production context. `apply_status_delta` rebuilds the entire `_LOBSTER_ADDITIONS_` block wholesale from DB state on each pass — it does not do line-by-line idempotency detection using the regex. The label-only form will simply be re-rendered in the same form on each render pass. The regex is exported for use by callers and future phases, not for internal idempotency detection in this PR. The comment is accurate. The design decision is acceptable and documented.

One informational note (no change required): `test_render_attribution_line_matches_attribution_re` validates that full-form attribution lines match the regex, but does not add an explicit negative assertion that the label-only form does not match. Adding that assertion would close the documentation loop for the intentional exclusion. This is not a NEEDS_CHANGES item — the comment in the code makes the intent explicit — but it would strengthen the test as documentation.
