### Stage 2: Quality review

**What the reviewer confirmed:** 54 pass / 1 pre-existing failure on both main and PR branch. Parser safety confirmed (attribution lines do not match checkbox patterns). No DB writes. No schema changes. No regression.

**Issue 1 (NEEDS_CHANGES): Docstring-implementation mismatch — false API contract on a public pure function**

The `format_attribution` docstring states:
```
- created_at_iso absent: "[<label> · Mon May 11]" (date only)
```

The implementation does the opposite: it returns `None` when `created_at_iso` is absent for telegram/voice/direct sources. The test `test_attribution_missing_created_at_returns_none_for_non_obsidian` acknowledges the discrepancy in a comment ("The spec says ... However...") but then asserts `None`, validating the implementation against itself rather than the spec.

This is a false API contract on an exported, independently-testable pure function. `format_attribution` will be called in future phases. A caller reading the docstring will either:
- Expect a string when `None` is returned (causing an AttributeError or incorrect conditional), or
- Intentionally pass no timestamp expecting a date-only render, then silently receive no attribution

The learnings.md PR #839 entry applies here: "Assertion softening... cannot distinguish wired-correctly from silently-failed-with-fallback." Here the test comment names the gap explicitly but then resolves it in favor of the implementation without correcting the contract.

**Required resolution (one of two):**
- (a) Update the docstring to state "if created_at_iso absent: return None" — removing the date-only promise that the implementation does not deliver. This is the honest description of what the function does.
- (b) Implement the date-only fallback path — fetch date from the ISO timestamp up to the 'T', display `[telegram msg · Mon May 11]` — and test it. This delivers what the docstring promised.

Option (a) is the minimal fix. Option (b) is the spec-correct fix. Either closes the gap.

**Issue 2 (advisory, fix in same pass): Dead regex alternative**

`ATTRIBUTION_LINE_RE` includes `obsidian` in the bracket-dot alternatives:
```python
ATTRIBUTION_LINE_RE = re.compile(r"^  - (?:\[\[.+\]\]|\[(?:telegram msg|voicenote|direct|obsidian) ·)")
```

`format_attribution` never produces `[obsidian · ...]` — obsidian sources always produce `[[...]]` wiki-link form, which is handled by the first alternative. The `obsidian` entry in the bracket-dot branch is unreachable. The risk: a future engineer reads the regex, concludes `[obsidian · ...]` is a valid attribution format, and introduces a code path that produces it. One-word removal (`obsidian|` dropped from the alternatives) closes this.

The reviewer correctly flagged this as low severity. The oracle agrees. It should be fixed in the same pass as Issue 1 to avoid a Round 2 just for a one-word deletion.

**Issue 3 (informational, no change required): ATTRIBUTION_LINE_RE comment overpromises**

The regex comment says it is used "so idempotent renders can skip re-rendering" attribution lines. Neither render path in this PR calls the regex internally — `render_active_todos` rebuilds from scratch; `apply_status_delta` strips and rewrites the additions block. The regex is exported for future use. The comment should say "exported for use by callers and future phases" rather than implying internal idempotency logic exists. Acceptable as a follow-on note rather than a NEEDS_CHANGES blocker.

**Encoded Orientation check (OODA constraint-3):** This PR adds a display pass to a render function. It does not change behavioral defaults, system constraints, agent identity, or decision-making rules in a durable way. Constraint-3 does not fire.

**Row-index fragility:** The `row[3]`, `row[4]`... pattern with `len(row) > N` guards in both render paths is fragile if column order in the SELECT changes. The index comment at line 855-862 documents the mapping. This is reviewable but not a bug today. The existing codebase uses tuple row access throughout SQLite result processing; no learnings.md pattern flags this as a canonical defect in this context. Not a NEEDS_CHANGES item.
