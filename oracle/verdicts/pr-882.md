## Stage 1: Vision Alignment

The PR title and body describe a documentation-only change: updating `docs/wos-pipeline-architecture-20260422.md` to represent the metabolic primitives as typed stock with outcome_refs as the causal carrier, and updating the WOS workstream README with a note. The stated design anchor is Issue #880 (outcome_refs schema spec — not yet implemented in code).

Stage 1 question: is this the right problem? The metabolic vocabulary was already present in the architecture document, described in the Outcome Categories table (Section 6) and the lifecycle section. The missing piece was a Mermaid subgraph showing how outcomes flow *after* a UoW result is written — i.e., where pearls, seeds, heat, and shit go structurally. Issue #880 specifies the outcome_refs schema that makes this flow machine-traversable. The diagram update extends the existing design to show this layer, referenced to Issue #880 as the formal anchor. This is forward specification ahead of implementation — a pattern consistent with how WOS architecture documents have been used in this repo (the pipeline diagram has always led code). This is not a one-way door: the diagram describes a design, not a commitment.

Vision.yaml current_focus names "WOS execution health: resolve 44% pipeline failure rate" as primary. Documentation clarification of the metabolic layer is not in the critical path, but it is not in tension with it either. The architecture document is a shared coordination artifact; having the metabolic output layer missing from the flowchart while Issue #880 is active is a coherence gap. Vision principle-4 ("wire what exists before building more") does not prohibit documentation of design — it prohibits feature addition ahead of integration. This is documentation.

**Alignment verdict: Confirmed** — the architecture diagram update is aligned with the vision. The problem being solved is real and the approach is correct.

However, Stage 1 surfaces a scope discrepancy that becomes the central Stage 2 finding: the PR description says "documentation-only PR — no code changes" and lists two changed files (architecture diagram + workstream README). The actual diff contains four changed files with 401 additions, of which 360 belong to two files not mentioned in the PR description.

## Stage 2: Quality Review

### Encoded Orientation check

This PR modifies documentation only — no behavioral defaults, system constraints, agent identity, or decision-making rules changed. The outcome_refs schema in the diagram is a forward specification anchored to Issue #880. Constraint-3 does not fire.

### Feature-bundling — primary finding

The PR diff contains:

1. `docs/wos-pipeline-architecture-20260422.md` — 41 lines changed. This is what the PR title describes.
2. `docs/assessments/2026-04-23-system-retrospective.md` — 285 lines added. Not mentioned in the PR title or body.
3. `data/smell-patterns.yaml` — 75 lines added. Not mentioned in the PR title or body.
4. `data/.gitkeep` — empty sentinel file for the new `data/` directory.

The PR description explicitly says "documentation-only PR" but adds a new `data/` directory with a YAML registry file that is described in the retrospective as a "Machine-readable registry of named smell patterns with detection heuristics, severity, threshold, and issue templates" intended for automated smell detection. This is not documentation — it is a data file intended to drive automated behavior in a future `system-retrospective` scheduled job.

This is the feature-bundling pattern named in learnings.md for PRs #717, #712, #714, and #498: "a PR described as [narrow scope] modified [N] files including [undescribed content]." The test from learnings.md is mechanical: count the changed files, compare against the PR title and description, flag any file not mentioned. Applied here: 4 changed files; PR description mentions 2. The undescribed additions are 87% of the diff by line count.

The smell-patterns.yaml file in particular crosses a design threshold: it names smells with `github_label`, `issue_template`, and `status` fields, and the retrospective document describes a feedback loop where the scheduled job reads this file to file GitHub issues automatically. This is Encoded Orientation in a data file — it encodes behavioral defaults for a not-yet-built system. It was added without a PR that describes it, without oracle review of the schema design, and without a logged decision about the automated issue-filing behavior the data file implies.

### Architecture diagram assessment

The Mermaid subgraph itself is well constructed:

- Syntax is valid. The `subgraph METABOLIC [...]` block uses the correct Mermaid flowchart TD subgraph syntax. Node labels use `\n` correctly for multiline. Arrow labels with `|category=pearl|` etc. are syntactically correct.
- Internal consistency: `RESULT --> OREFS` correctly places the metabolic layer after result write. The four outcome types (pearl/seed/heat/shit) are correctly differentiated as stock-with-address vs. energy-with-log. `SEED_UOW -.->|generative loop| GH` correctly represents the seed→issue→UoW generative cycle with a dashed line (indicating logical rather than direct flow). The heat and shit paths correctly terminate at log entries with no downstream stock.
- The `OREFS` node correctly names outcome_refs as the provenance carrier. The subgraph label references Issue #880 as the formal anchor.
- The updated Outcome Categories table correctly adds the `Stock type`, `outcome_refs required?`, and `Downstream flow` columns. The "Key principle" note at the bottom ("seeds and pearls are stock with addresses; heat and shit are energy with logs") is the design principle in one sentence.

One internal inconsistency: the `ORACLE` node still references `oracle/decisions.md` ("verdict written to oracle/decisions.md") but PR #875 — merged the day before this PR — deprecated decisions.md and made oracle/verdicts/ the sole source of truth. The architecture document carries a now-stale path reference.

### Retrospective document

The 2026-04-23 system retrospective is a substantial document (285 lines) covering metabolic accounting, smell patterns, waste analysis, and an implementation plan. As a retrospective, it reads accurately — the metabolic ratios, smell descriptions, and waste estimates are internally consistent. However:

- It was added to `docs/assessments/` without being described in the PR.
- It contains a "Feedback Loop Proposal" section that proposes a new scheduled job (`system-retrospective`) that reads `smell-patterns.yaml` and files GitHub issues automatically. This is a DESIGN_OPEN proposal, not a documentation artifact. Bundling it into a diagram-update PR removes it from the design gate it would otherwise need to pass.

### Summary of findings

- The Mermaid subgraph and table update are correct, well-formed, and internally consistent. This portion of the PR is APPROVED on its merits.
- The PR is not what it describes. It bundles two undescribed artifacts (retrospective + smell registry) into a diagram-update PR. The feature-bundling pattern from learnings.md applies directly.
- The smell-patterns.yaml file encodes behavioral intent for a not-yet-built automated system without a logged design decision or oracle review of the schema.
- The architecture diagram carries a stale `oracle/decisions.md` path reference that should be updated to `oracle/verdicts/` following PR #875.

Addressed when: either (a) a logged decision exists authorizing the automated smell-detection feedback loop and is cited in the PR body, or (b) the smell-patterns.yaml file is removed from this PR and moved to its own PR after the scheduled job design is oracle-reviewed.
