This PR adds a new durable behavioral write in `dispatcher_handlers.py` — every `wos start`, `wos stop`, and `handle_decide_close` now writes to the registry. This is an Encoded Orientation change in that it adds new side effects to existing handler code paths.

#### Gap 2: Event type strings are raw literals with no named production constants — NEEDS_CHANGES (minor)

The event type strings `"wos_start"`, `"wos_stop"`, `"wos_abort"` appear as inline string literals in `dispatcher_handlers.py` (at the `log_control_event` call sites). No named constants are defined at module scope in production code.

The test file defines local constants (`CONTROL_EVENT_TABLE`, `_EVENT_TYPE_WOS_START`, etc.) but cannot import them from production because they don't exist there. The PR's own docstring says `CONTROL_EVENT_TYPES contains at least: wos_start, wos_stop, wos_abort` — implying these should be a defined set, but this set is test-only.

This is the StrEnum golden pattern applied imperfectly: per golden-patterns.md (2026-04-24), "any function that returns one of N fixed string choices (path selection, strategy names, mode strings) should define a StrEnum." The event_type column is N fixed string choices. The fix: define a small set of named constants (or StrEnum) at module scope in `registry.py` or `dispatcher_handlers.py` and import them at all call sites, including in the test file.

This is minor — the strings are simple and unlikely to drift — but the test file's locally-declared event type constants (`_EVENT_TYPE_WOS_START = "wos_start"`) are mirror constants that will diverge silently if the production string values ever change. The mirror-constant pattern (first-pass oracle check, learnings.md 2026-04-26) is present here in its lowest-severity form: test file re-declares string values that the production code also has as inline literals. No production named constant exists to import — the test is doing the right thing given what the production code provides, but production should provide the named constant.

#### Known concerns evaluation

**Concern 1: handle_decide_close → wos_abort mapping**

The implementing agent is correct that `handle_decide_close` is the appropriate write point given the current codebase. The semantic mapping is sound: `decide_close` transitions a UoW from `blocked → failed(user_closed)` — this is the operational equivalent of an abort on a stuck UoW. The docstring now makes the "wos abort" alias explicit.

The concern about PR #1103 is valid but not a blocker for this PR: when PR #1103 merges and adds `handle_wos_abort`, that handler should write `"wos_abort"` via `log_control_event`. At that point, two paths will write `"wos_abort"`:
- `handle_decide_close` (Dan selects "Close" on a stuck UoW via inline button)
- `handle_wos_abort` (Dan sends `wos abort <uow_id>` via Telegram command)

These are semantically distinct commands with a shared event type, which is correct — the control log records the fact that an abort occurred, and the payload's `uow_id` distinguishes the affected UoW. No double-write risk exists because `handle_decide_close` closes an already-blocked UoW (blocked → failed) while `handle_wos_abort` kills a running process (no status transition). They operate on different UoW states. This concern is resolved.
