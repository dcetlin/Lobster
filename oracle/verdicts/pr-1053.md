## Round 1 — 2026-05-03

### Stage 1: Vision Alignment

**Prior entering review:** This implementation is solving the wrong problem, or solving the right problem in a direction that forecloses better paths.

**What would have to be true for this to be the right path:**
The WOS prescription pipeline needs a formal spec to ensure determinism and auditability. The spec should be authoritative over the format that the Steward produces and the Executor consumes. The issue (dcetlin/Lobster #573) was created specifically to produce this spec document. The work is a doc-only PR adding spec + schema + examples.

**Alignment verdict:** Questioned

**Stage 1 finding:** The theory of change is that formalizing the prescription format will anchor Executor behavior and enable auditability. This is sound in isolation. But the PR presents the 7-section format with `"schema_version": "1.0.0"` and `"Effective: 2026-05-03"` as if it describes the current production system. It does not. The production Steward writes `WorkflowArtifact` — a flat 5-field front-matter + prose `.md` format (`uow_id`, `executor_type`, `constraints`, `prescribed_skills`, `instructions`). The Executor reads this format via `from_frontmatter()` in `workflow_artifact.py`. The 7-section JSON structure in this spec — with `diagnosis`, `prescription`, `workflow`, `constraints`, `success_criteria`, `dan_context`, `audit_metadata` — does not correspond to any format the current Steward produces or the current Executor consumes.

Issues #574 (implement v2 prescription generation in steward.py), #576 (migration framework), #577 (evaluation pass), and #578 (observability) all remain open. This spec is a design document for a *future* format. The PR title and the spec's own "Effective: 2026-05-03" framing present it as a current-production specification. That framing forecloses a clear path: it makes the spec's authority relationship to the live production format ambiguous and potentially misleading to any agent or engineer consulting the spec.

The opportunity cost question: Is the spec wrong, or is it design-first? Design-first is fine. But the framing must say so. A spec that says "this is what we are building toward" is structurally sound. A spec that says "schema version 1.0.0, effective today" implies the system already implements it. If an Executor agent reads `docs/prescription-format-spec.md` tomorrow as orientation, it will form an incorrect model of what `WorkflowArtifact` looks like — and produce or parse files in the 7-section format that the current steward.py will not write and executor.py will not read.

**What this forecloses:** Any agent that reads this spec as authoritative will be misconfigured. The "effective date" and semver create an implicit claim of production readiness that will not be resolved until issues #574-#578 land. The spec will be consulted as orientation before those issues are implemented, which is exactly when the misleading framing is most dangerous.

### Stage 2: Quality Review

**Encoded Orientation check:** This PR does not change behavioral defaults, agent identity, system constraints, or decision-making rules in a durable way. It adds a docs-only artifact. Constraint-3 does not apply.

**Feature-bundling check (from learnings.md — Document Review pre-scan):** 5 files changed. PR description accounts for all 5: the spec, the schema, and three examples. No unmentioned files. Feature bundling is not present.

**Path reference canonicality:** The spec references `executor-contract.md` and `wos-v3-steward-executor-spec.md` — both exist in `docs/`. No stale runtime paths embedded. The `$id` in the schema points to `https://github.com/SiderealPress/lobster/docs/...` (the upstream repo) while the file lives in `dcetlin/Lobster`. This is a URI mismatch: if anyone resolves the `$id`, it will 404 or reach the wrong repo. This is a schema metadata defect, not a blocking issue by itself.

**The primary quality finding — spec/implementation divergence:**

The spec describes a 7-section JSON prescription object. The live production format is:

```
---json
{"uow_id": "...", "executor_type": "...", "constraints": [], "prescribed_skills": []}
---
<instructions prose>
```

Key divergences:
1. The production format has 5 fields flat; the spec has 7 top-level sections with 25+ total fields.
2. The production format stores `instructions` as prose after the front-matter delimiter; the spec stores it in `prescription.instructions`.
3. The production format has `executor_type` (which maps to workflow type); the spec has `workflow.agent_type`.
4. The production format has no `diagnosis` section (the Steward does not write its diagnostic reasoning into the artifact — it is stored in `steward_log` in the registry). The spec makes `diagnosis` a required section.
5. The production format has no `success_criteria` section in the artifact (success_criteria lives in the UoW registry row, not in the artifact written to disk).
6. The production format has no `dan_context` section.
7. The production format has no `audit_metadata` section (the Steward writes audit entries to the registry `audit_log`, not the artifact).

**The "validated" claim:** The PR says "all three examples validate cleanly against the schema." This is true — the examples are self-consistent with the schema. But the examples are not validated against what steward.py actually produces. A JSON Schema validates JSON against itself; it says nothing about whether the live system will ever emit JSON in this shape.

**Schema correctness against its own spec:**
- The `uow_id` pattern `^uow_\d{8}_[a-f0-9]{6}$` correctly represents the ID format used by the registry.
- The `executor_posture` enum `["first_execution", "continuation", "remediation"]` is a 3-value collapsed form of the 9-value `reentry_posture` enum. The mapping table in the spec (section 7) is internally consistent.
- The `$schema` URI uses Draft 2020-12 correctly.
- The `oneOf` for `executor_outcome` (string enum or null) and `fan_out` (array or null) are correctly structured.
- `dan_context` uses `"required": []` — which is valid but unusual; all fields in `dan_context` are truly optional. The section itself is required in the root schema. This is internally consistent but could be written more clearly as `"required"` omitted entirely (empty required array is a JSON Schema noop for Draft 2020-12 as well).
- The `$id` points to `https://github.com/SiderealPress/lobster/docs/prescription-format.schema.json`. The file lives in `dcetlin/Lobster`. This is incorrect.

**Example consistency with spec:**
- All three examples include all 7 required sections.
- `design-review.json`: `audit_metadata.cycle = 1` and `diagnosis.prior_cycle_count = 1` — consistent.
- `execution-pass.json`: `audit_metadata.prescribed_skills = ["functional-engineer"]` — this is a skill identifier, but the spec says prescribed_skills are "skill identifiers to activate." The value "functional-engineer" matches an agent type, not a skill. This is either imprecise or shows that the two concepts (agent type and skill activation) are being conflated. The spec doesn't define what constitutes a valid skill identifier.
- The `design-review.json` workflow uses `lobster-generalist` for the top-level `workflow.agent_type` (the synthesis step) while the fan-out branches also use `lobster-generalist`. The spec's invariant says "workflow.steps describes the merge/synthesis step when fan_out is present" — the example is consistent with this.
- `investigation-pass.json` has `no_deploy: true` for a read-only audit task. This is correct.

**The `execution_failed` → `remediation` mapping is underspecified:**
The mapping table says `execution_failed with corrective intent → remediation` but the spec provides no machine-readable criterion for "corrective intent." This is a human-judgment call that is not expressed in the schema. The schema permits `executor_posture = "remediation"` without requiring any specific pattern in `diagnosis`. This is a gap between the spec's stated invariant and the schema's enforcement.

**What patterns this introduces:** The spec establishes "7-section format" as the vocabulary for WOS prescriptions going forward. Future issues #574-#578 will reference this spec. If the spec is merged as-is (with "Effective: 2026-05-03" framing), every downstream issue will treat it as production-authoritative. This is not wrong if the intent is "this is the target format," but it must be stated as such.

### Named Gaps

**Gap 1: Spec-production divergence without explicit future-state framing** — The spec presents a 7-section format that does not match the production `WorkflowArtifact`. The document must either (a) state clearly that this is a design specification for the v2 format not yet implemented, with a link to issue #574 (the implementation issue), or (b) accurately describe the current production format. The current framing ("schema_version: 1.0.0, Effective: 2026-05-03") implies production readiness. This must be corrected before any agent or engineer uses this spec as orientation.

**Resolution contract for Gap 1:** The revision must add a prominently placed "Implementation Status" section at the top of the spec (below the Introduction) that states: (a) this is the specification for the v2 prescription format, (b) the current production format is `WorkflowArtifact` as defined in `src/orchestration/workflow_artifact.py`, (c) the v2 format will be implemented in issues #574, #576, #577, #578. Alternatively, the "Effective" date should be removed and replaced with a "Status: design specification" label. The `schema_version: 1.0.0` should be `0.1.0` or the semver should be accompanied by a stability disclaimer. Addressed vs. not-addressed is decidable: the spec either contains an explicit statement of its relationship to the current production format, or it does not.

**Gap 2: `$id` URI points to wrong repository** — `"$id": "https://github.com/SiderealPress/lobster/docs/prescription-format.schema.json"` references the upstream repository, not `dcetlin/Lobster` where this file lives. A JSON Schema `$id` is used for schema resolution and referencing. This will resolve to the wrong repository (or 404 if the upstream repo does not have this file).

**Resolution contract for Gap 2:** Change `$id` to `"https://github.com/dcetlin/Lobster/blob/main/docs/prescription-format.schema.json"` or remove the `$id` entirely if cross-repository schema resolution is not required. Addressed vs. not-addressed is decidable by inspection of the `$id` value.

**Gap 3: `execution_failed → remediation` mapping is not machine-enforceable** — The spec states this mapping requires "corrective intent" but does not define what constitutes corrective intent or how the schema enforces the distinction between `execution_failed → continuation` and `execution_failed → remediation`. The schema permits any combination. This is a specification gap, not a schema bug — the spec's prose constraint cannot be enforced by the schema.

**Resolution contract for Gap 3:** Either (a) define "corrective intent" with a machine-readable criterion (e.g., a `diagnosis.corrective_intent: boolean` field added to the schema), or (b) explicitly document this as a human-judgment call in the spec prose that the schema cannot enforce, with a note that the Steward LLM prescriber must use judgment. The current text implies enforcement that does not exist. Addressed vs. not-addressed is decidable: the spec either adds a machine-readable criterion or explicitly disclaims enforcement of the distinction.

### Patterns introduced

The spec introduces a "7-section vocabulary" for prescriptions. This vocabulary will propagate into issue descriptions, LLM prescriptions, and agent orientation. If adopted without Gap 1 resolved, agents oriented by this spec will form incorrect expectations about the current `WorkflowArtifact` format.

### Learnings

The "spec-as-current-production" framing when the spec describes a future state is a new pattern instance. It is distinct from the prior "documentation PR with stale oracle path" entries in learnings.md (those described existing stale references; this describes a spec presented as current when it describes a future state). Adding to learnings.md.

### VERDICT: NEEDS_CHANGES

**Revision contract:**
- Gap 1 (spec-production divergence framing): Required. The spec must state its relationship to the current production format explicitly.
- Gap 2 (`$id` URI): Required. The `$id` must reference the correct repository or be removed.
- Gap 3 (remediation mapping): Required. The corrective-intent criterion must be defined or explicitly disclaimed as human-judgment.

All three gaps are in the spec document and schema. The examples themselves are internally consistent and well-constructed; they do not need to change unless Gap 1 is resolved by aligning the spec with the current production format (in which case the examples would need to show the `WorkflowArtifact` front-matter format instead).
