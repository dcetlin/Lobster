# Oracle Verdict: PR #873 — fix: WOS registry DB path mismatch

**VERDICT: APPROVED**

## Stage 1 — Vision alignment (written before seeing implementation)

This PR fixes a DB path mismatch for WOS registry access in two callers. The adversarial prior entering Stage 1: this implementation is solving the wrong problem, or solving the right problem in a direction that forecloses better paths.

The vision's `current_focus.secondary` explicitly calls out "file wos_completion.py REGISTRY_DB fragility issue" — this PR addresses exactly that class of problem. The two-path divergence could theoretically be a symptom of a missing service layer (each caller independently accessing the registry without going through a shared accessor), but at WOS Phase 1 scale the canonical path constant is the seam-first abstraction the design calls for. Centralizing on `paths.REGISTRY_DB` does not foreclose a future service layer — it makes the path to one cleaner by ensuring all callers already import from the same anchor module.

**Alignment verdict: Confirmed**

## Stage 2 — Quality findings

### What the diff does

- `cultivator.py`: removes `_build_db_path()` (a local env-var resolver using `LOBSTER_WORKSPACE`) and `os` import. Inside `promote_to_wos()`, replaces `db_path = _build_db_path()` with a deferred import of `REGISTRY_DB` from `src.orchestration.paths`, then uses it directly. The deferred import is inside the function body — this was the engineer's circular-import concern.
- `wos_report.py`: removes the module-level `REGISTRY_DB` constant that read `WOS_REGISTRY_DB` env var with a fallback, replaces it with `from src.orchestration.paths import REGISTRY_DB`. The `os` import is retained (still needed for `OUTBOX_DIR`).

### Correctness

**Deferred import in cultivator.py** — the import is placed inside `promote_to_wos()`, not at module scope. This is the standard Python pattern for breaking circular imports. It fires once on first call and is thereafter cached by Python's import system. No risk of double-execution or performance concern. The engineer's concern is correctly addressed by this placement.

**`REGISTRY_DB` in paths.py** — the constant is module-level: `Path(os.environ["REGISTRY_DB_PATH"]) if os.environ.get("REGISTRY_DB_PATH") else LOBSTER_WORKSPACE / "orchestration" / "registry.db"`. This is the learnings.md PR #839 pattern: module-level constant computed from env var at import time is frozen at import time, making test monkeypatching via `setenv` ineffective. However, this is not a new defect introduced by this PR — `paths.py` already worked this way for all its other callers. This PR does not make the testability profile worse; it aligns two callers with the same testability profile as all other `paths.REGISTRY_DB` consumers. Flagging as advisory, not blocking.

**`WOS_REGISTRY_DB` env var now silently ignored** — the engineer correctly notes this. Grep confirms no other caller uses this env var. The env var's semantics (`WOS_REGISTRY_DB` → absolute override path) are replaced by `REGISTRY_DB_PATH` in `paths.py`. Any operator who had `WOS_REGISTRY_DB` set in the environment will find it silently ignored after this PR. This is an acceptable breaking change at Phase 1 scale — there is no evidence of production use of `WOS_REGISTRY_DB` outside `wos_report.py`, and the env var was only used as an override path fallback, not a signal. A comment in the PR noting the deprecation is sufficient (the comment in the diff serves this: `# REGISTRY_DB is imported from src.orchestration.paths — resolved via REGISTRY_DB_PATH`).

**os import removal from cultivator.py** — correct. `os` was only used in `_build_db_path()`. With that function removed, `os` is unused. Confirmed by engineer's grep.

**4 pre-existing integration test failures** — `src.ooda` missing module, not introduced by this PR. No new test failures expected.

### Pattern analysis

**Seam-first abstraction (golden-patterns.md)** — this PR is an instance of the seam-first pattern applied correctly: the boundary between "where the DB lives" and "who accesses it" is now exclusively owned by `src.orchestration.paths`. Two callers that had independently derived the same path are now consuming the canonical seam. This is the direction the pattern calls for.

**Module-level constant vs. env-var-at-call-time (learnings.md PR #839)** — `paths.REGISTRY_DB` is a module-level constant, which means it cannot be overridden via `monkeypatch.setenv` in tests without a module reload. This is pre-existing behavior in `paths.py` and is not made worse by this PR. Advisory only: any future test of `cultivator.promote_to_wos` that needs a non-default DB path must pass it explicitly via `REGISTRY_DB_PATH` before import, or restructure `promote_to_wos` to accept `db_path` as a parameter. That restructuring is deferred work, not this PR's scope.

### What this does not address

The deferred import inside `promote_to_wos()` is correct for circularity prevention, but it means `REGISTRY_DB` is resolved at call time, not at import time. In practice this matches production behavior (env vars are stable across a session), but it is a subtle asymmetry from `wos_report.py`'s module-level import. Not a defect — worth noting for future callers considering which pattern to follow.

## Summary

Two callers independently resolving the registry DB path now import the canonical constant. The fix is minimal, correct, and directly traceable to `current_focus.secondary`. The `WOS_REGISTRY_DB` env var deprecation is handled cleanly. No circular import risk. No new test defects.

**VERDICT: APPROVED**
