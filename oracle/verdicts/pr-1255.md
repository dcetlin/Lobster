## Stage 2: Quality Review

**Does it do what it claims?**

Yes. The two-part fix is correct:
1. Adding `"failed"` to `RETRYABLE_STATUSES` means `decide_retry` now processes failed UoWs (was returning 0 before).
2. `steward_log_clause = ", steward_log = NULL" if from_status == "failed" else ""` clears stale failure context only on the failed path, preserving prior prescription context for blocked and awaiting-owner retries.

**SQL construction safety:** `updated_at = ?{steward_log_clause}` is safe. `steward_log_clause` is derived deterministically from `from_status == "failed"`, where `from_status` is a DB-read value already constrained by `WHERE status IN ({placeholders})`. No injection risk.

**Encoded Orientation check (OODA constraint-3):** This PR adds `'failed'` to a set it was always supposed to be in — it fixes a bug, not a behavioral direction change. The PR does not change agent identity, decision-making rules, or system constraints in a durable new way. The `RETRYABLE_STATUSES` set was always logically supposed to include `'failed'`; its absence was a bug. Constraint-3 does not block.

**Failure modes:**
- The `from_status` read happens inside `BEGIN IMMEDIATE`, so there is no race between reading status and applying the update. Correct.
- If `from_status` is somehow not `"failed"` despite the `WHERE` clause constraining to `RETRYABLE_STATUSES`, the `steward_log_clause` is `""` — steward_log is preserved. This is the safe default.

**Patterns introduced:** `steward_log_cleared` in the audit note JSON — good observability pattern.

- 7 new tests in `TestDecideRetryFromFailed` — covers the full transition path
- `test_blocked_uow_retry_preserves_steward_log` is a good regression guard
- All 87 existing registry tests pass
