### Stage 2: Quality Review

#### Is the classification applied at the right point?

Yes. `mark_processing()` is the correct injection point: it is called once per message after claiming, before routing, and before any dispatcher logic runs. This matches the existing pattern for `_processing_started_at`. The comment in the diff explicitly grounds this in issue #1023. No concern.

#### Mirror-constant pattern — the central defect

**`_SYSTEM_TYPES` in `lobster_meta.py` is a mirror of `INBOX_SYSTEM_TYPES` in `message_types.py`.**

This is the most significant structural issue in this PR. The learnings.md mirror-constant pattern has recurred at least seven times in this codebase (PRs #696, #800, #967, #970, #973, #974, #976, #980). Here it appears again in a new module.

The `_SYSTEM_TYPES` frozenset in `lobster_meta.py` contains exactly the same 22 values as `INBOX_SYSTEM_TYPES` in `message_types.py`. The PR description notes that `lobster_meta.py` "mirrors `message_types.py` in design" — but the intent was module structure (dependency-free), not value mirroring. The two constants are now two independent definitions of the same fact.

When a new system type is added to `INBOX_SYSTEM_TYPES` (which happens as the system evolves), `_SYSTEM_TYPES` in `lobster_meta.py` will not be updated automatically. A new type will be misclassified as `intent_class: "operational"` with `is_user_facing: True` when the message source is `telegram` and `chat_id != 0` — even though it is a system message. The misclassification will be silent: no crash, no test failure, just wrong hints flowing to the dispatcher.

The stated rationale for making `lobster_meta.py` dependency-free is test isolation — the module can be imported and tested without the full inbox_server stack. That is a legitimate concern. But it does not require mirroring `_SYSTEM_TYPES` as a local constant. The correct pattern (applied elsewhere in this codebase: function-body local import, or restructuring `message_types.py` to be importable without pulling the full stack) would preserve both dependency-free testability and single-source-of-truth for the system type set.

**`_USER_FACING_SOURCES` in `lobster_meta.py` is a partial mirror of `INBOX_MESSAGE_SOURCES`.**

`INBOX_MESSAGE_SOURCES` contains 9 sources including `bot-talk` and `gmail`. `_USER_FACING_SOURCES` contains 6 sources, excluding `system`, `bot-talk`, and `gmail`. Excluding `system` is intentional and correct (system source is handled separately). But `bot-talk` and `gmail` are silently absent.

`bot-talk` messages are cross-Lobster inbound messages (real bot-to-bot exchanges, source="bot-talk"). These would have `is_user_facing: False` under the current `_USER_FACING_SOURCES` definition. Whether this is correct depends on whether cross-Lobster messages should be considered "user-facing" — the upstream comment says they are "cross-Lobster bot-to-bot messages" which suggests not user-facing, so this exclusion is defensible. But it is undocumented — a comment explaining why `bot-talk` is excluded from `_USER_FACING_SOURCES` is absent.

`gmail` messages are from the email poller. These are potentially user-initiated (Dan sends email → system receives it). Under `_USER_FACING_SOURCES` as defined, gmail messages will always have `is_user_facing: False`. Whether this is correct is unclear from the PR. The `_HUMAN_SOURCES` definition in `inbox_server.py` (line 762) also excludes `gmail` and `bot-talk`, suggesting this is a known convention — but the absence of documentation in `lobster_meta.py` makes the omission look like an oversight rather than a decision.

#### Are the `intent_class` values well-defined and exhaustive enough?

The six values (`operational`, `emotional`, `code`, `question`, `reaction`, `system`) cover the stated scope. The priority ordering — code → question → emotional → operational — and the documentation in the bootup doc are clear. The fallback to `operational` is appropriate.

One issue: the docstring says `urgency` is "implemented here" and other fields should be "add more fields in follow-ups." The issue spec (#1023) explicitly says "Start with `is_user_facing` and `intent_class`; add `urgency` in a follow-up." But the PR implements `urgency` in the same PR. This is not a defect, but the docstring comment that `urgency` was added here while `other` fields go to follow-ups reads as if it was originally written expecting `urgency` to be a follow-up — and then `urgency` was added without updating that framing. Minor inconsistency.

#### Clean separation from inbox_server.py?

Mostly yes. The import in `inbox_server.py` is a single line: `from lobster_meta import build_lobster_meta`. The call site is a one-liner. The module has no dependencies on inbox_server state. The dependency-free architecture is correct.

One concern: the import line includes `# noqa: E402 — same path-setup above`, which means it depends on sys.path manipulation that occurs earlier in `inbox_server.py`. This is consistent with how `message_types.py` is imported (same pattern), but it is a fragile dependency — if `lobster_meta.py` needs to be imported from outside inbox_server.py (e.g., in a new consumer), the sys.path manipulation would need to be replicated. This is a known limitation of the `src/mcp/` module layout and is not introduced by this PR, so it does not rise to NEEDS_CHANGES.

#### No LLM calls confirmed?

Confirmed. The module uses only `re`, `datetime`, and `timezone` from the standard library. No LLM calls, no network calls, no I/O.

#### Do the 29 tests cover meaningful edge cases or just happy paths?

The tests are reasonably thorough for the stated scope:

### Verdict Rationale (Round 1)

NEEDS_CHANGES. One structural defect blocks approval:

**Gap 1: _SYSTEM_TYPES is a mirror constant (blocking)**

`lobster_meta.py` defines `_SYSTEM_TYPES` as a local frozenset with 22 values that match `INBOX_SYSTEM_TYPES` in `message_types.py` exactly. This introduces a silent divergence risk: when a new system type is added to `INBOX_SYSTEM_TYPES`, `_SYSTEM_TYPES` will not be updated, causing misclassification. This is the mirror-constant pattern named in `oracle/learnings.md` (eight prior occurrences).

The correct fix is to import `INBOX_SYSTEM_TYPES` from `message_types.py` directly, or — if the dependency-free constraint is real — to explicitly document why mirroring is intentional, enumerate the divergence risk, and add a test that validates the two sets are equal at import time (catching future divergence).

**Gap 2: `_USER_FACING_SOURCES` missing `bot-talk` and `gmail` — undocumented omission**

`bot-talk` and `gmail` exist in `INBOX_MESSAGE_SOURCES` but are absent from `_USER_FACING_SOURCES`. The exclusion may be intentional (both are non-human-direct sources), but there is no documentation explaining the decision. This is not a crash-level defect, but the undocumented omission will silently produce `is_user_facing: False` for future gmail messages that Dan originates. A one-line comment naming the intent would close this gap.
