## Stage 2: Quality Review

### Encoded Orientation check (constraint-3)

### Key findings from the five specific items

**Item 1 — `_suggest_diagnosis()` orphan check with empty list**

The implementation:

```python
all_orphan = (
    bool(return_reasons)
    and all(r["return_reason"] in _ORPHAN_RETURN_REASONS for r in return_reasons)
)

if all_orphan and len(return_reasons) >= 2:
    ...
```

`bool(return_reasons)` is `False` when the list is empty, so `all_orphan` evaluates to `False` for an empty list. The `and` short-circuits. This is correct — an empty list does NOT trigger the kill-wave pattern. This item passes.

The `>= 2` guard is an additional check that a single orphan entry is not treated as a "wave." This is a reasonable design choice, but it is undocumented. A UoW with exactly one orphan return reason falls through to the `kill_classification`-based branches or the default. This is probably the intended behavior but the lack of a comment creates a minor readability gap.

**Item 2 — `_read_trace_json()` path derivation**

The implementation in registry_cli.py:

```python
trace_file = p.with_suffix(".trace.json") if p.suffix else Path(str(output_ref) + ".trace.json")
if not trace_file.exists():
    alt = Path(str(output_ref) + ".trace.json")
    if alt != trace_file and alt.exists():
        trace_file = alt
    else:
        return None
```

The steward.py reference:

```python
trace_file = Path(output_ref).with_suffix(".trace.json")
if not trace_file.exists():
    trace_file_alt = Path(str(output_ref) + ".trace.json")
    if trace_file_alt.exists():
        trace_file = trace_file_alt
    else:
        return None
```

There is a semantic divergence. Steward.py unconditionally calls `Path(output_ref).with_suffix(".trace.json")` as the primary path. registry_cli.py introduces a branch: if `p.suffix` is falsy (the path has no extension), it uses string-append directly. This creates a case where the two implementations compute different primary paths when `output_ref` has no suffix. For `output_ref = "/tmp/uow_20260101_ABCDEF"` (no extension), steward.py produces `/tmp/uow_20260101_ABCDEF.trace.json` via `with_suffix`. registry_cli.py produces the same result via string-append. For `output_ref = "/tmp/uow_20260101_ABCDEF.json"`, steward.py produces `/tmp/uow_20260101_ABCDEF.trace.json` (correct) and registry_cli.py also produces `/tmp/uow_20260101_ABCDEF.trace.json` (correct). The two paths diverge only on the logic flow, not the outputs — in both cases the fallback covers the other convention. However, there is a subtle issue: if `output_ref` has a suffix of `.json`, the `p.suffix` branch in registry_cli.py takes the `with_suffix` path (because `p.suffix` is truthy), producing `.trace.json` correctly. If `output_ref` has some other unexpected suffix, registry_cli.py also applies `with_suffix`, which replaces that suffix with `.trace.json`. Both produce the same output for the canonical output_ref convention.

This is essentially equivalent in practice but the conditional branch is not present in steward.py. The learnings.md PR #967 entry ("Dual-path result.json convention with no test for the alt-path and no docstring explaining which producer writes which form") named this exact surface — and this PR does not have a test for the alt-path (the fallback string-append path). The test `test_trace_reads_trace_json_when_output_ref_set` only exercises the `with_suffix` primary path. The alt-path has no test coverage and no comment naming which producer writes files without an extension.

**This is a gap.** The learnings.md PR #967 pattern directly applies: "both paths must be tested with a comment naming the producer, or one path must be removed with an explanation of which convention is canonical."

**Item 3 — `_ORPHAN_RETURN_REASONS` as local copy**

The implementer flagged this as intentional: importing from steward.py would create a circular dependency or cross-module coupling. The stated reasoning is defensible — steward.py is a complex module with non-trivial transitive imports, and registry_cli.py is a CLI tool that should remain lightweight.

However, the mirror-constant anti-pattern from learnings.md has now escalated from test files to a production module. PR #974 learnings entry is explicit: "A production-module mirror is higher-severity: no test will catch divergence between the two constants." The recommended alternative in that entry is a function-body local import — but steward.py uses that pattern specifically for constants with circular dependency risk.

The correct resolution in this codebase is to place shared constants in a module that both `steward.py` and `registry_cli.py` can import without circular dependency risk. The `ReturnReasonClassification` enum and `_RETURN_REASON_CLASSIFICATIONS` dict already live in `steward.py`. `_ORPHAN_RETURN_REASONS` in `registry_cli.py` does not mirror any named constant in `steward.py` — it mirrors the set of keys in `_RETURN_REASON_CLASSIFICATIONS` whose value is `_CLASSIFICATION_ORPHAN`. This set is implicit in steward.py, not named. The only orphan-classified `return_reason` values in steward.py's `_RETURN_REASON_CLASSIFICATIONS` are `"executor_orphan"` and `"diagnosing_orphan"`.

`_ORPHAN_RETURN_REASONS` in registry_cli.py contains five values: `"executor_orphan"`, `"executing_orphan"`, `"diagnosing_orphan"`, `"orphan_kill_before_start"`, `"orphan_kill_during_execution"`. Only two of these appear in steward.py's `_RETURN_REASON_CLASSIFICATIONS`. The other three (`"executing_orphan"`, `"orphan_kill_before_start"`, `"orphan_kill_during_execution"`) do not appear anywhere in the current codebase.

**This is a gap.** The five values in `_ORPHAN_RETURN_REASONS` include three strings not present in any current production module. There is no migration path or comment explaining what module will produce these values, whether they are future-oriented additions, or whether they were intended but missed. The comment on the constant says "Must match ORPHAN_REASONS in steward.py" — but `ORPHAN_REASONS` does not exist in steward.py. There is no `ORPHAN_REASONS` symbol in the codebase. The comment names a non-existent production constant, which is the same failure mode identified in learnings.md PR #894 ("Decision-file vision anchor citation can name a non-existent field path while still satisfying constraint-3 at the substance level but not at the traceability level"). Here it does not "still satisfy" the intent — the named anchor simply does not exist.

**Item 4 — Mirror-constant anti-pattern in test files**

Searching the test diff: the test file uses inline string literals `"executor_orphan"`, `"diagnosing_orphan"` in `_write_audit_entry` calls. These are not imported from `registry_cli._ORPHAN_RETURN_REASONS` or from any production module. The test `test_trace_diagnosis_hints_infrastructure_kill_wave` uses `["executor_orphan", "executor_orphan", "executor_orphan"]` as literals. This is the mirror-constant anti-pattern (learnings.md PR #967, #970, #974). The values should be imported from `_ORPHAN_RETURN_REASONS` rather than re-declared as literals in the test body.

However — given that `_ORPHAN_RETURN_REASONS` itself has questionable values (three strings not present in steward.py), fixing the test imports is contingent on resolving Item 3 first. Still, this is the sixth recurrence of the pattern.

**Item 5 — `Registry.fetch_corrective_traces()` vs. `get_corrective_trace_history()`**

The PR description correctly describes the distinction: `get_corrective_trace_history()` returns only `prescription_delta` strings (up to `limit`); `fetch_corrective_traces()` returns all columns of all rows. The docstring on `fetch_corrective_traces()` says "Use this for the forensics trace view; use get_corrective_trace_history() when only prescription_delta values are needed." This distinction is clear and non-conflicting. The SQL query in `fetch_corrective_traces()` explicitly enumerates columns (good — no `SELECT *`). The bare `except Exception: return []` guard is appropriate for a forensics/diagnostic read path that should not raise on pre-migration DBs. This item passes.

### Additional findings

**`kill_type` string mismatch in kill-before-start branch.** The `_suggest_diagnosis` function checks:

```python
if kt == "orphan_kill_before_start" or (not trace_json and kt):
```

But the test `test_trace_includes_kill_classification_when_present` writes `"kill_type": "kill_during_execution"` (not `"orphan_kill_during_execution"`) and the matching branch in `_suggest_diagnosis` checks `kt == "orphan_kill_during_execution"`. There is an inconsistency between the test's string value and the pattern match string. The test will not fire the `kill-during-execution` branch in `_suggest_diagnosis` because `"kill_during_execution" != "orphan_kill_during_execution"`. The test only checks that `kill_type` and `heartbeats_before_kill` are surfaced in the output — it does not verify the diagnosis branch. This passes as a structural test but misses verifying the diagnostic logic. Not a blocking issue on its own, but worth noting alongside Item 3's undocumented values.

**Diagnosis for UoW with single orphan return reason:** A UoW with exactly one orphan return reason and no `kill_classification` falls through all branches to the default. The `kill-before-start` catch `(not trace_json and kt)` requires a `kill_classification` to be present. The `retry-cap-from-orphans` branch requires `status == "needs-human-review"`. A UoW that has returned once as `executor_orphan` and is back in `ready-for-steward` would produce the generic "no known failure pattern matched" output. This is arguably correct behavior but not discussed anywhere in the docstring.

## Summary of gaps requiring resolution

**Gap 1: `_ORPHAN_RETURN_REASONS` references a non-existent constant and contains three values not present in any production module**

The constant's comment says "Must match ORPHAN_REASONS in steward.py" but `ORPHAN_REASONS` does not exist in steward.py. The three values `"executing_orphan"`, `"orphan_kill_before_start"`, `"orphan_kill_during_execution"` are not produced by any current code in the repository. The constant needs either: (a) a corrected comment naming the actual steward.py construct it mirrors (the orphan-classified keys in `_RETURN_REASON_CLASSIFICATIONS`) and removing or explaining the three unproduced values, or (b) a docstring explaining that these three values are anticipatory for PRs #965-#968 extensions, with a reference to where they will be introduced.

Resolution contract: the comment must name a real symbol or explain the three absent values. "Must match ORPHAN_REASONS in steward.py" pointing at a non-existent symbol is not acceptable — it will mislead future maintainers who search steward.py for that name.

**Gap 2: `_read_trace_json()` alt-path has no test and no comment naming the producer**

The fallback path `Path(str(output_ref) + ".trace.json")` is exercised when the primary `with_suffix` path is absent. No test covers this case. learnings.md PR #967 named this exact surface. Resolution contract: add a test for the fallback path, or remove the fallback path with a comment explaining that only the `with_suffix` convention is in use, citing which producer writes which form.

**Gap 3: Test file uses raw string literals that should be imported from the production constant**

`"executor_orphan"`, `"diagnosing_orphan"` in test assertions are re-declared as literals rather than imported from `registry_cli._ORPHAN_RETURN_REASONS`. This is the sixth recurrence of the mirror-constant anti-pattern (learnings.md PRs #696, #800, #967, #970, #974). The literals should be imported from the production module. This is contingent on Gap 1 being resolved — if `_ORPHAN_RETURN_REASONS` is corrected first, the test imports become straightforward.
