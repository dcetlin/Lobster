## Round 1 — 2026-05-19

### Stage 1: Vision Alignment (written before reading implementation)

**Adversarial prior:** This implementation is solving the wrong problem, or solving the right problem in a direction that forecloses better paths.

**What would have to be true for this work to be the right path:**

The philosophy session workflow generates `action_seeds` YAML blocks. Those seeds contain bootup candidates and memory observations that currently accumulate without being routed into attention flow. For this PR to be on the right path, the right intervention point would have to be at the file-scan-and-route layer — a script that reads session files and pushes outputs downstream.

**The foundational assumption under examination:**

The seeds have been accumulating "without being routed into the system's attention flow." This assumes the bottleneck is routing, not production. It also assumes the `action_seeds` blocks in existing philosophy sessions are high-quality enough to mass-deliver via inbox notifications and reflective surface queue entries without human curation.

**What this forecloses:**

Harvesting 61 bootup candidates and 75 surface queue observations from 69 session files in a single run is a volume event, not a signal event. The reflective surface queue and Telegram inbox are attention resources with metabolic cost (constraint-4). Mass-importing philosophy session artifacts into the attention system assumes the session outputs were curated for delivery at the time they were written — not that they represented exploratory residue worth post-processing. The script cannot distinguish between a bootup candidate that was genuinely worth encoding and one that was speculative note-taking during a session.

**Alignment verdict: Questioned**

The task is motivated by a real gap (accumulated seeds not flowing into the system) but the architectural response may substitute volume for signal. The vision's principle-4 (wire what exists before building more) and constraint-4 (minimize metabolic cost of cybernetic engagement) both point against a mechanism that generates large batch imports into attention queues. The one-time harvest of 61 bootup notifications and 75 surface queue items is a specific scenario where this risk materializes. The design does not include any signal-quality gate before routing to attention resources.

**Cheap test not yet run:** Was there a manual review of the existing 69 session files' `action_seeds` blocks to verify their routing quality? If yes, the volume concern is addressed at the source. If no, the script is automating delivery of unvetted material into attention channels.

This Stage 1 finding does not rise to "misaligned" because the architecture is correct in class: philosophy session artifacts should flow somewhere, and the routing destination (inbox for bootup candidates, surface queue for observations) is appropriate. The concern is signal-quality gate, not architectural direction. Stage 1 is confirmed before reading the diff.

### Stage 2: Quality Review

The implementation is a single file: `scheduled-tasks/philosophy-harvest.py` (314 lines). No tests are included in this PR (an existing `test_philosophy_harvester.py` tests a different module `src.harvest.philosophy_harvester`, not this script). No upgrade.sh migration is included. No jobs.json addition is in the diff.

**What the script does:** Scans `~/lobster/philosophy/**/*.md` for YAML fenced blocks containing `action_seeds`. Routes `bootup_candidates` to Telegram inbox via `write_inbox_message`. Routes `memory_observations` of type `design_gap`, `tension`, or `principle` to the reflective surface queue JSON file. Tracks harvested files in a JSON state file for idempotence.

**Gaps identified:**

**Gap 1: Missing `is_job_enabled()` gate (BLOCKING)**

Every Type C cron-direct job in this codebase calls `is_job_enabled(JOB_NAME)` at the start of `main()`. This is explicitly required by CLAUDE.md: "add `_is_job_enabled(job_name)` at the top of `main()` before any DB work." Examples across the codebase: `wos-health-check.py`, `garden-caretaker.py`, `morning-delivery-flush.py`, `todo_obsidian_sync.py`, `transcription-monitor.py`, `vault-watcher.py`, `wos-pr-sweeper.py`, `los-action-scanner.py`, `system-retrospective.py`.

The philosophy-harvest script's `run()` function never checks `is_job_enabled`. The `jobs.json` entry for this job has `"enabled": true` — if a future operator sets `enabled: false` to pause the job, the script will continue running on cron. The CLAUDE.md convention is structurally enforced by the presence of the gate, not by the `enabled` field alone.

Applied pattern citation: This is the CLAUDE.md cron-direct job pattern ("**Required: jobs.json gate.** Add `_is_job_enabled(job_name)` at the top of `main()` before any DB work"). Absence of this gate is a convention violation that impacts operability, not just style.

**Gap 2: No tests for this script in this PR (BLOCKING)**

The PR description claims: "Unit tests pass with proper mocking (no production hits in automated tests) -- no external service calls in this script." This appears to describe a test run, but there are no test files in the PR diff. The existing `test_philosophy_harvester.py` tests `src.harvest.philosophy_harvester` — a different module from `scheduled-tasks/philosophy-harvest.py`. The pure functions in the script (`extract_action_seeds`, `format_bootup_notification`, `make_surface_item`, `load_state`) are all directly testable. Their absence means there is no automated verification that the extraction logic handles edge cases: malformed YAML, missing keys, plain-string candidates vs. dict candidates, or the `SURFACEABLE_TYPES` filter.

The PR description explicitly marks "[x] Unit tests pass" — this appears to be a false checkbox given that no test files are in the diff. The PR claims tests were run manually (`uv run scheduled-tasks/philosophy-harvest.py --dry-run`) but that is an integration-level smoke test, not a unit test.

**Gap 3: No upgrade.sh migration for jobs.json registration**

The `jobs.json` in the runtime workspace already contains a `philosophy-harvest` entry (added separately at some point). However, CLAUDE.md requires: "For changes that affect existing installs (new cron entries, new directories, config renames, new service files), add a numbered migration to `scripts/upgrade.sh`." A new cron-direct script that is registered in `jobs.json` on any instance requires an upgrade.sh migration to register it on other instances. The PR contains no upgrade.sh modification. This is not a safety regression (the script exits early if `philosophy_dir` does not exist), but it is a convention violation.

**Gap 4: Signal-quality gate absent (advisory — not blocking alone)**

The `SURFACEABLE_TYPES` frozenset gates observations by type but applies no content quality check. Any `memory_observation` with `type` in `{"design_gap", "tension", "principle"}` is appended to the reflective surface queue regardless of text length, specificity, or relevance. This is a design choice that may be intentional — the surface queue delivery agent is responsible for curation downstream. If that is the design, a comment stating it explicitly would close this gap. As written, there is no documentation of what the downstream surface queue consumer does with low-quality entries. This is advisory in the context of Stage 1's metabolic cost concern.

**What would break without key decisions made here:**

The atomic tmp-file write pattern for both state persistence and queue append is correct. The `extract_action_seeds` function returns on the first matching block, which is correct for the single-block-per-session convention. The dry-run support is correctly isolated to the side-effectful paths. The pure/impure function split is clean. The `repo_root` parameter to `harvest_session` is accepted but never used — this is dead code in the function signature (the parameter was included for import of `src.*` modules but the actual imports happen at module scope). This is a minor issue, not a gap.

**Encoded Orientation check (constraint-3):**

This script routes automatically from philosophy session files to Telegram inbox (bootup candidates) and reflective surface queue (design_gap/tension/principle observations). This is an Encoded Orientation decision: the system acts without Dan's explicit input on each session's seeds. The CLAUDE.md requirement for a prior logged decision applies. The PR description references WOS UoW `uow_20260427_6064ab` — which provides a WOS traceability anchor. Vision.yaml principle-4 (wire what exists before building more) provides a vision anchor for the general direction. The specific Encoded Orientation decision (automatic routing of philosophy session seeds into attention queues) would benefit from a logged decision, but the UoW reference provides sufficient traceability for this class of automation. This is noted, not a hard block.
