## Round 1 — 2026-04-27

### Stage 1: Vision alignment (written before reading implementation)

The theory of change: the WOS executor runs agents that may stall — alive (heartbeats arriving) but not progressing (zero token delta). Without a signal distinguishing "alive and working slowly" from "alive but stuck," the steward has no basis to intervene. This PR adds per-heartbeat token snapshots and a consecutive-zero-delta detector as that signal.

What would have to be true for this to be the right path: (1) stuck agents produce meaningfully different token-usage trajectories than slow-but-progressing agents; (2) 60-90s heartbeat sampling captures useful deltas — burst-producing agents might false-positive; (3) N=2 consecutive zero-delta threshold gives sufficient sensitivity without false positives.

Opportunity cost and foreclosure: none concerning — detection-only with no kill action. This cannot break the pipeline.

Pattern priors active at Stage 1:
- learnings.md PR #653 (off-by-one in reverse-scan non-improving counter): `_consecutive_zero_delta_tail` is exactly this class of function. I will check the termination expression specifically.
- learnings.md PR #704 (`>=` on cumulative counter fires on every post-threshold cycle): the detection loop runs on every steward cycle; multi-fire is the expected behavior for a stuck UoW.

**Alignment verdict: Confirmed.** Stuck-agent detection is direct WOS observability and reliability work, squarely within the active phase intent. The migration-numbering conflict flagged in the task prompt was identified before reading the implementation as the primary structural risk to evaluate.

### Stage 2: Quality review

#### BLOCKER — Gap 1: Migration 0016 collision with PR #999

PR #999 (`0016_add_prescription_confidence.sql`) and PR #1000 (`0016_heartbeat_token_log.sql`) both claim migration version 16. The migration runner in `src/orchestration/migrate.py` uses the 4-digit numeric prefix as the unique version key — `_migrations.version` is defined `UNIQUE NOT NULL`. The runner skips any version already in `_migrations`.

Failure scenario when PR #999 merges first:
1. PR #999 migration applies, records `version=16` in `_migrations`.
2. PR #1000 is merged. The runner sees version 16 already applied and skips `0016_heartbeat_token_log.sql`.
3. The `uow_heartbeat_log` table is never created.
4. `write_heartbeat` calls `INSERT INTO uow_heartbeat_log` → `sqlite3.OperationalError: no such table`. The inner `try/except` catches this and logs at DEBUG level with `log.debug("write_heartbeat: could not write heartbeat log ...")`. No user-visible signal, no crash.
5. All stuck-agent detection is permanently disabled. `get_heartbeat_log` returns `[]` for every UoW (same table-absent exception path, returns `[]`). `detect_stuck_heartbeat_uows` finds zero candidates and returns `StuckHeartbeatResult(checked=0, flagged=0)` on every cycle.

This is a silent total failure of the feature. The backwards-compatibility try/except — which is correctly designed for pre-0016 deploys — also masks the collision failure on any deploy where PR #999 merged first. The two cases (pre-migration deploy, collision deploy) produce identical behavior: no heartbeat log rows, no stuck detection.

**Required fix:** Rename `0016_heartbeat_token_log.sql` to `0017_heartbeat_token_log.sql`. The PR description and all references to "migration 0016" in comments and docstrings must be updated to `0017`. This applies to:
- `src/orchestration/migrations/0016_heartbeat_token_log.sql` → `0017_heartbeat_token_log.sql`
- The comment in `src/orchestration/registry.py` `write_heartbeat` docstring: "The uow_heartbeat_log table is created by migration 0016."
- The `# Phase 2c: Stuck-agent detection — progress governor (migration 0016, issue #994).` comment in `steward-heartbeat.py` main()
- The SQL file header comment "migration 0016"

**If PR #999 is not yet merged:** the renumber is still required — whichever merges second will collide. The resolution is deterministic regardless of merge order: this PR must use `0017`.

#### Advisory finding — Gap 2: Raw string literals in new SQL (StrEnum golden pattern)

`get_executing_uows_with_heartbeats` contains:
```sql
WHERE u.status IN ('active', 'executing')
```

The surrounding `registry.py` consistently uses `UoWStatus.ACTIVE` and `UoWStatus.EXECUTING` in SQL queries (see lines 1177-1178 in the current main). The golden-patterns.md StrEnum rule (2026-04-24, "StrEnum constants in all enum-backed SQL and routing logic") names this exact failure. The learnings.md PR #904 entry records the prior recurrence in `list_executing()` using the same file.

Since `UoWStatus` is a `StrEnum`, values compare equal to their string counterparts. This will not cause incorrect behavior. However, the pattern exists in the codebase precisely to prevent silent enum-value drift — if `UoWStatus.ACTIVE` or `UoWStatus.EXECUTING` were renamed, the SQL literal would not fail at the grep site.

Addressable in the same fix round as the renumber: change `'active'` to `UoWStatus.ACTIVE.value` and `'executing'` to `UoWStatus.EXECUTING.value` (or use the enum directly with SQLite's text comparison, which works for StrEnum). Not a blocking gap on its own, but given that the migration renumber already requires a fix round, this should be resolved in the same pass.

#### Findings that do NOT require changes

**`_consecutive_zero_delta_tail` is correctly implemented.** The learnings.md PR #653 off-by-one concern (reverse-scan plateau counter uses `return count - 1`) was the primary pattern to check. This implementation uses `return count` directly — no off-by-one. The break fires on the first positive delta without decrementing. Doctests and test assertions use exact equality (`== 2`, `== 1`, `== 0`). The critical early-return path is covered by `test_interior_zero_then_nonzero` with `[100, 100, 200]` → 0. Clean.

**Backwards compatibility is genuine.** The `try/except` around the heartbeat_log INSERT with DEBUG-level logging correctly handles pre-0016 deploys without crashing. `rowcount` is captured before the inner try, preserving heartbeat semantics. The `get_heartbeat_log` and `get_executing_uows_with_heartbeats` methods return `[]` on any exception — safe fallback.

**No Encoded Orientation concern (constraint-3).** Detection-only: no automated kill, no re-queue, no status transition. No durable behavioral default is changed. Constraint-3 does not fire.

**Multi-fire is acceptable at this stage.** A stuck UoW that persists across multiple steward cycles will generate repeated `log.warning` and `_append_observation` calls — the learnings.md PR #704 pattern in detection form. This is a known tradeoff for detection-only logic with no first-crossing guard. Acceptable now; the kill gate (future PR) will require a crossing guard or per-UoW deduplication.

**Test coverage is solid.** The 24 tests in `test_stuck_heartbeat_detection.py` cover the pure function (9 cases including early-return path with exact equality), the detection loop (8 cases including failure paths and dry_run), and the registry integration (7 cases with in-memory SQLite). Constants are imported from the production module, not mirrored locally. The 8 tests in `test_write_wos_heartbeat_token.py` cover the MCP handler's validation logic.
