### Stage 2: Quality Review

#### Gap 1: Test file locally re-declares production string constants (mirrors, not imports)

The test file defines:

```python
KILL_BEFORE_START = "kill_before_start"
KILL_DURING_EXECUTION = "kill_during_execution"
COMPLETED_WITHOUT_OUTPUT = "completed_without_output"
```

The production module defines `ORPHAN_KILL_BEFORE_START`, `ORPHAN_KILL_DURING_EXECUTION`, `ORPHAN_COMPLETED_WITHOUT_OUTPUT` — same string values, different names. The tests assert against the locally-declared mirrors, not the production constants. If the production constant values change, the tests silently validate against the stale string values. This is the named failure pattern from learnings.md PR #696 and PR #800 ("Locally-declared mirror of a production constant creates a divergence risk"). The test should import and assert against the production module constants directly, not redeclare them. The `_classify_orphan_from_trace` and `_enrich_orphan_completion_rationale` functions return the production constants; the tests should assert equality against those exact module attributes.

**Behavioral change from pattern:** This pattern caused me to look for the test constant declarations immediately after reading the production constant names, and to check whether the test assertions use the imported or the locally-declared values. Without the named pattern, I would likely have skimmed the assertion lines as correct.

**Required fix:** Replace `KILL_BEFORE_START = "kill_before_start"` etc. in the test file with imports: `from orchestration.steward import ORPHAN_KILL_BEFORE_START, ORPHAN_KILL_DURING_EXECUTION, ORPHAN_COMPLETED_WITHOUT_OUTPUT`. Update all assertions to use the imported names.

#### Gap 2: `_ORPHAN_POSTURES` frozenset is function-local, re-created on every `_diagnose_uow` call, and uses raw string literals

`_ORPHAN_POSTURES` is defined inside the body of `_diagnose_uow` as:

```python
_ORPHAN_POSTURES = frozenset({"executor_orphan", "executing_orphan", "diagnosing_orphan"})
```

This has two problems:

(a) **Allocation on every call.** `_diagnose_uow` is called in every steward cycle for every UoW in an orphan posture. Defining the frozenset inside the function body re-creates it on every invocation. The correct placement is module scope (or at least outside the function).

(b) **Raw string literals for posture values.** The StrEnum golden pattern (2026-04-24) requires that enum-backed values use their enum constant, not raw string literals. If `executor_orphan` is a posture value backed by an enum or constant elsewhere in the codebase, using the raw string here creates an undiscovered mismatch risk and violates the golden pattern's Rule 1.

**Behavioral change from pattern:** The StrEnum golden pattern caused me to check whether `executor_orphan`, `executing_orphan`, and `diagnosing_orphan` appear elsewhere in the codebase as constants or enum values, and to ask whether the frozenset should use those instead of raw strings.

**Required fix:** Move `_ORPHAN_POSTURES` to module scope. If the posture values are backed by an enum or constant anywhere in the codebase, use those constants in the frozenset. If they are not, at minimum elevate the frozenset to module scope with a comment linking it to the posture constants.

#### Gap 3: `_classify_orphan_from_trace` alt-path for result.json has no test coverage and the naming convention is undocumented

The function checks two naming conventions for result.json:

```python
result_path = Path(output_ref).with_suffix(".result.json")  # foo/bar.result.json
if not result_path.exists():
    result_path_alt = Path(str(output_ref) + ".result.json")  # foo/bar.json.result.json
    if result_path_alt.exists():
        result_path = result_path_alt
```

`with_suffix(".result.json")` replaces the existing suffix (`.json`) with `.result.json`, producing `bar.result.json`. The alt-path appends `.result.json` to the full filename, producing `bar.json.result.json`. These are different conventions. No test exercises the alt-path — all test cases for `completed_without_output` use `with_suffix`. The docstring does not explain which convention is canonical or why both are checked.

Two concerns:
- The alt-path is untested dead code from a coverage standpoint. If the system never writes `output.json.result.json`, the alt-path check never fires and the code is misleading.
- If both conventions are in active use (legacy vs. new), this should be documented with a comment explaining which scripts write which form.

The requested "close review" item explicitly flagged this. The `completed_without_output` path is the most consequential classification — it means the agent finished its work but the steward never received confirmation. False negatives here (missed result.json files) mean these UoWs are misclassified as `kill_before_start` instead of `completed_without_output`, causing them to be retried unnecessarily.

**Required fix:** Either (a) add a test that exercises the alt-path by writing a `foo.json.result.json` file, with a comment in the test explaining which producer writes this form; or (b) remove the alt-path if no producer uses it and document that `with_suffix` is the canonical form; or (c) add a docstring clarifying which naming conventions are in use and why both must be checked.

#### Observations (not blocking but noted)

**`_classify_orphan_from_trace` docstring claims "Pure function" but performs filesystem I/O.** The docstring states: "Pure function: only reads files (result.json presence check)." Reading files is a side effect — the output depends on filesystem state, not only on arguments. The function is more precisely described as "deterministic given stable filesystem state" or "no DB writes, no mutation." This is a documentation precision issue, not a correctness issue, but it will mislead future readers who refactor under the assumption that the function can be called freely without filesystem concerns.

**`_enrich_orphan_completion_rationale` returns base_rationale unchanged when `output_ref is None` but the guard condition `if trace_data is None and kill_class == ORPHAN_KILL_BEFORE_START` does not cover the case where `output_ref is None` AND `trace_data is not None`.** If output_ref is None but a trace file exists (perhaps via a different path resolution), `_read_trace_json(None, ...)` returns None (correct — it cannot resolve a path from None). So the guard works in practice. But the guard's logic is expressed as a consequence of the classification outcome rather than the input condition, which makes the intent less obvious than `if output_ref is None and trace_data is None: return base_rationale`.
