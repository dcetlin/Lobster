## Stage 1 — Vision Alignment (formed before seeing implementation)

**Theory of change:** When Dan approves an option in conversation, the choice evaporates into the thread. The PR makes `decision:` a first-class footer type and routes it to a durable ledger. The implicit claim: structural persistence of decisions serves the system's ability to answer "is this aligned with Dan's intent?" without asking him again.

**Alignment with vision.yaml:** This directly addresses `fundamental_intent` — "structural access to my intent, not just a prose reconstruction of it." Decisions made during conversation are exactly the kind of intent-layer content the system needs structural access to. Persisting them to a queryable file is a necessary (though not sufficient) step toward making past intent retrievable.

**What would have to be true for this to be wrong path:** If the decisions-ledger.md never gets queried — if it becomes an append-only sink with no consumer — the persistence infrastructure delivers no structural benefit. A side-channel observation: the PR creates the ledger but does not wire it to any retrieval path (no hook, no agent, no scheduled task reads from it). This is not a blocker — persistence before retrieval is the correct sequencing — but it is a risk worth naming. The ledger is currently write-only infrastructure.

**Foreclosure check:** Adding `decision:` as a valid footer type does not foreclose other approaches to decision provenance. The ledger format (append-only markdown) is not a one-way door — a future consumer can parse it, index it, or migrate it. The hook machinery is the same PostToolUse infrastructure used by all other hooks.

**Constraint-3 check (Encoded Orientation):** Installing a PostToolUse hook is an Encoded Orientation decision — it changes behavioral defaults durably. The prior logged decision question: does a logged prior exist? The hook arises from a design need that is directly traceable to `fundamental_intent` in vision.yaml (structural decision provenance). The vision.yaml anchor is present. No separate logged prior exists in oracle/verdicts/ or meta/premise-review.md, but the vision.yaml anchor is sufficient under constraint-3 ("Encoded Orientation decisions require a prior logged decision of the same class and a traceable vision.yaml anchor" — the anchor is the vision.yaml `fundamental_intent` field, which explicitly names structural access to intent as the goal). This passes constraint-3.

**Alignment verdict: Confirmed.** The work serves the theory of change. The right first step is persistence infrastructure. The missing consumer is a known gap, not a design flaw.

## Stage 2 — Quality Review

### Flagged area 1: `has_valid_footer` change in signal-footer-check.py

The original Check 2 logic:
```python
if not SIDE_EFFECTS_BLOCK_RE.search(text):
    wrong_label = detect_wrong_label(text)
    if wrong_label is not None:
        ...block...
```

The new logic:
```python
has_valid_footer = SIDE_EFFECTS_BLOCK_RE.search(text) or DECISION_BLOCK_RE.search(text)
if not has_valid_footer:
    wrong_label = detect_wrong_label(text)
    if wrong_label is not None:
        ...block...
```

**Side-effects: behavior unchanged?** Yes. The logic is semantically equivalent for messages with `side-effects:` blocks: the OR short-circuits on `SIDE_EFFECTS_BLOCK_RE.search(text)` being truthy, setting `has_valid_footer` to truthy, so the inner block (wrong-label check) is skipped. No regression on existing side-effects: behavior.

**`decision` in WRONG_LABEL_PATTERNS?** Verified directly. The five WRONG_LABEL_PATTERNS match: `signals:`, `effects:`, `side_effects:`, `side-effects` (without colon), and any block label containing "signal", "effect", or "side". The word "decision" contains none of these substrings. Empirically confirmed: a `decision:` block produces zero matches against all five patterns. Correct.

**Logic gap worth noting:** When `has_valid_footer` is False (no side-effects: or decision: block), the code only blocks if a *wrong-label* block is present. A message with `decision:` and a wrong-label block simultaneously would set `has_valid_footer` to True (from the decision: match) and skip the wrong-label check entirely. This means a message like:

```
Some text.
```decision:
Choice B
```
```effects:
whatever
```
```

...would pass validation. This is an edge case — both a valid decision: block and a wrong effects: block in the same message. The original code had the same behavior for side-effects: (a side-effects: block plus a wrong label would pass). This is a pre-existing corner case, not introduced by this PR. Not a blocker.

### Flagged area 2: decision-router.py exits 0 on all non-match paths

The hook exits 0 on: JSON parse failure, non-send_reply tool name, empty text, no `decision:` match, empty decision text. For a PostToolUse hook, exit 0 on non-match is correct. PostToolUse hooks are observational — they cannot block the tool call (it has already executed). Silent non-blocking exit is the appropriate posture for all non-match paths. The hook should not exit 2 for any PostToolUse case; doing so would be incorrect (exit 2 on PostToolUse would signal an error but could not reverse the already-completed send_reply). The implementation is correct.

The hook reads from `data.get("tool_input", {}).get("text", "")` — it reads the *input* to send_reply, not the output. For a PostToolUse hook, this is correct: the hook receives the full event including both `tool_input` and `tool_output`. Reading from `tool_input.text` is appropriate since that is where the message content lives. This matches how `secret-scanner.py` and `dispatch-template-check.py` access tool input data.

### Patterns introduced

The `decision:` footer type introduces a second consumer class for PostToolUse hook infrastructure. The pattern of using code block labels as routing signals (not just validation signals) is new — `side-effects:` was validation-only; `decision:` is both validated (via signal-footer-check.py) and routed (via decision-router.py). This is a clean extension of the existing pattern.

### What this makes easier / harder

Easier: future retrieval of past decisions, accountability for choices made in conversation, any future agent that needs to cite a decision's history.

Harder: nothing structurally foreclosed. The ledger is append-only markdown — not a schema that constrains future query patterns. The missing consumer (nothing currently reads the ledger) remains a known open gap.

### One open observation (not a blocker)

The decisions ledger is write-only infrastructure. The PR description frames this as "decisions persist to the ledger for future retrieval" but no retrieval path exists. This is correct sequencing (persist first, query later), but the gap between persistence and retrieval is where the value actually lives. Flagging as an observation for the next iteration: consider wiring the ledger into morning briefing or memory consolidation so the infrastructure delivers behavioral value rather than remaining latent.
