**The scenario where this is wasted effort:** cron-based executor dispatch is architecturally brittle (PATH stripping being symptomatic), and the right fix is to move executor dispatch off cron and onto a mechanism that inherits a proper user environment. ## Stage 2 — Quality Review

### Fix 1: PATH augmentation in `_build_claude_env()`

The implementation is correct. Key observations:

**Placement is correct:** The PATH augmentation happens at the top of `_build_claude_env()`, before the fast-path return (`if env.get("CLAUDE_CODE_OAUTH_TOKEN"): return env`). This ensures both the fast path (token already in env — interactive session) and the slow path (credentials.json read — cron) get the augmented PATH. This was verified by reading the PR diff: `_local_bin` and the PATH mutation happen before the early return. Test `test_fast_path_also_includes_local_bin` covers this.

**Idempotency is correctly implemented:** The guard `if _local_bin not in path_entries` prevents duplicate insertion. The duplicate-suppression test (`test_local_bin_not_duplicated_when_already_present`) covers this.

**Prepend vs. append:** `~/.local/bin` is prepended, not appended. This is correct — the user's `claude` binary should shadow any system-installed one. `test_local_bin_prepended_not_appended` verifies index <= 1, which is the right check (index 0 is the correct position after prepend; index 1 is also acceptable if a leading empty entry appears after splitting).

**One possible gap (advisory, not blocking):** The test `test_local_bin_prepended_not_appended` asserts `idx <= 1`. This is a slightly soft assertion — it would pass if `_local_bin` were at index 1 due to an empty string at index 0 from splitting a PATH that started with ":". In practice this is not a reachable case given the guard logic, but the assertion is technically weaker than `assert idx == 0`. This is minor and does not rise to NEEDS_CHANGES.

### Fix 2: CalledProcessError stderr capture in `_run_step_sequence()`

The implementation is correct. Key observations:

**Type discrimination is clean:** `isinstance(exc, subprocess.CalledProcessError)` is the right pattern. The PR description explicitly notes this is "pure conditional on isinstance" — not string matching. This is consistent with the vision.yaml `principle-3` ("Determinism over judgment for conditionals").

**Both result surfaces are updated consistently:** The `failure_detail` variable is used in:
1. `_write_subagent_result(output_ref, status="failed", summary=...)` — the result.json summary
2. `execution_summary=f"Executor crashed: {failure_detail}"` — the trace.json execution_summary
3. `surprises=[failure_detail]` — the trace.json surprises array

The prior code used `reason` for all three and `str(exc)` for surprises specifically. The PR correctly propagates `failure_detail` to all three sites where diagnostic information appears. Consistency across all three write surfaces is correct — a future reader should not need to cross-reference result.json and trace.json to get the full error.

**One consistency note (advisory):** The `reason` variable is still used in `_write_output_ref_content(output_ref, f"execution failed: {reason}")` (line 703 in the diff) and in `self.registry.fail_uow(uow_id, reason)` (line 722 in the diff). The output_ref content and the registry fail_reason still use the generic `reason` (which for CalledProcessError is `"CalledProcessError: <exit code only>"`), while result.json and trace.json use the richer `failure_detail`. This is a minor asymmetry — the registry and output_ref get less diagnostic context than result.json. This is not a correctness failure and it is unlikely to matter in practice (result.json is the primary diagnostic artifact), but it is worth noting as a potential follow-on improvement.

**Truncation at 500 chars:** `[:500]` on both stderr and stdout excerpts is a reasonable cap. The PR description notes the actual stderr messages are short (e.g., "Claude: authentication failed: token expired"). No concern here.

**The `failure_detail` variable is not initialized before the conditional.** If a future engineer adds a code path inside the `try` block that could reach the `_write_subagent_result` call without passing through the `except`, `failure_detail` would be an unbound name. However, this is in an `except` block — `failure_detail` is always set before use within the except handler. No actual risk, but the code reads as if `reason` covers the base case and `failure_detail` might be unset — this is not the case because `failure_detail = reason` is set in the `else` branch. The code is correct.
