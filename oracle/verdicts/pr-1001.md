## Stage 1: Vision Alignment

**Prior:** This implementation is solving the wrong problem, or solving the right problem in a direction that forecloses better paths.

**Assessment:** The task — persisting `outcome_category` (metabolic taxonomy: heat/shit/seed/pearl) to the `uow_registry` DB column — directly serves the vision's WOS Phase 1 substrate. The metabolic taxonomy was introduced in PR #759 and has been live as a JSONL-only signal since then. Moving it into the registry makes it DB-queryable for pipeline visibility, routing decisions, and the `registry_cli report` command (PR #997, already merged). This is a "wire what exists before building more" move (principle-4) that closes a real observability gap without introducing new behavioral logic.

**What would have to be true for this to be the wrong path:** The metabolic taxonomy itself would have to be wrong as a classification scheme, or the DB column would have to foreclose a better observability pattern (e.g., external time-series store). Neither is present: the taxonomy is already validated and written at `write_result` time; the DB column is additive and doesn't constrain alternative approaches.

**What this forecloses:** Nothing significant. The `TEXT NULL` column with `COALESCE` write semantics leaves all existing behavior unchanged for callers that don't pass `outcome_category`.

**Opportunity cost check:** The work is small (one migration, parameter threading through four layers, report integration). The alternative cost of leaving the taxonomy JSONL-only is real: `registry_cli report` can't surface it, and any correlation query requires joining an external flat file. The investment is proportionate.

**Encoded Orientation check:** No new behavioral defaults are introduced. The migration adds a column; `complete_uow` writes it when provided. No automated routing decisions read this field. Constraint-3 does not apply.

**Alignment verdict:** Confirmed

### Non-Blocking Findings

**Finding 1: `_EXECUTING_STATUSES` is defined but never referenced in `_classify_status` (learnings.md PR #997 pattern)**

`_EXECUTING_STATUSES = frozenset({"active", "executing", "ready-for-steward", "ready-for-executor", "diagnosing"})` is declared at module scope in `registry_cli.py`. `_classify_status` uses an else-branch (return "executing") rather than checking membership in `_EXECUTING_STATUSES`. The frozenset is dead code — it is not referenced by `_classify_status` or anywhere else in the diff.

This is the PR #997 learnings.md pattern exactly: "A module-scope frozenset that names a catch-all bucket but is not referenced in the classification function is dead code following the PR #906 anti-pattern." At current DB population where `proposed` is the dominant bucket, the "executing" label in the summary misleads operators. The dead frozenset also creates a future trap: a maintainer who wants to add explicit bucket logic will reach for `_EXECUTING_STATUSES` not knowing it was never wired.

This is not a merge blocker on its own (the implementation is functionally correct; the else-branch covers the catch-all case), but it should be cleaned up in the same fix pass that resolves the blocking gaps: either remove `_EXECUTING_STATUSES` (since the else-branch is the canonical implementation) or replace the else-branch with an explicit `if status in _EXECUTING_STATUSES: return "executing"; raise ValueError(...)` pattern. The latter is safer at scale because it surfaces unknown statuses rather than silently lumping them.

**Finding 2: `_VALID_OUTCOME_CATEGORIES` is a mirror constant (learnings.md mirror-constant pattern)**

`_VALID_OUTCOME_CATEGORIES = frozenset({"heat", "shit", "seed", "pearl"})` in `registry_cli.py` mirrors `VALID_OUTCOME_CATEGORIES` in `inbox_server.py`. The comment correctly names the source: "mirrors VALID_OUTCOME_CATEGORIES in inbox_server." The stated reason for the local declaration — "so registry_cli can classify without importing the MCP server" — is valid (inbox_server imports trigger namespace conflicts in CI, established pattern for this codebase per learnings.md PR #848).

This is the same pattern accepted previously (e.g., `_should_skip_memory_write` extracted to avoid importing inbox_server). The risk is acknowledged; the mitigation needed is that a future change to `VALID_OUTCOME_CATEGORIES` in `inbox_server.py` must update this constant too. The comment names the source but does not create enforcement. This is acceptable given the established pattern and the small, stable vocabulary (four values), but should be noted for the maintainer.

**Finding 3: `fetch_in_window` test coverage gap**

`registry.fetch_in_window` is a new public method added in this PR. The test suite exercises it only through `registry_cli report` command tests (which call it indirectly via the CLI). There is no direct unit test for `fetch_in_window` — specifically, no test for the `wall_clock_seconds` computed column (the `CASE` expression in the SQL), no test for the OR condition (`started_at >= ? OR completed_at >= ?`), and no test for the ordering behavior. This is a coverage gap worth noting but not a blocker: the indirect tests provide reasonable coverage of the happy path, and the SQL is straightforward.
