## Stage 1 — Vision Alignment

Formed before reading the implementation.

**Epistemic prior:** This implementation is solving the wrong problem, or solving the right problem in a direction that forecloses better paths.

**Theory of change:** The WOS executor is now live (execution_enabled=True as of 2026-05-18 per vision.yaml). When a user deliberately stops WOS via `wos stop`, steward-heartbeat continues running and generates `backlog_starvation` observations. These observations are false-positive noise when the stop is intentional. Suppressing them reduces cybernetic cost (vision constraint-4) and eliminates unnecessary escalations to Dan during deliberate pauses.

**What would have to be true for this to be right:** (1) The starvation alert genuinely fires during deliberate pauses and generates disruptive noise. (2) The steward-heartbeat is the correct suppression site rather than the routing layer. (3) The `pause_reason` discriminator correctly distinguishes user-commanded stops from infrastructure-triggered disables.

**What would make this wasted effort:** If the correct design would quiesce alerting infrastructure entirely when deliberately paused rather than having the alerting code run-and-suppress. However, steward-heartbeat performs other health checks beyond starvation detection, and quiescing the entire heartbeat during pauses would suppress those. The narrowest correct fix — conditional suppression at the starvation alert site — is architecturally sound. The alternative (routing-layer suppression) would require filtering after the observation is written, which is messier.

**Does this foreclose anything:** No. The `pause_reason` field is additive. `wos start` clears it. Infrastructure disables that don't set `pause_reason` continue to generate alerts normally. No direction foreclosed.

**Encoded Orientation check (constraint-3):** The PR adds a new field to `wos-config.json` and a suppression gate in steward-heartbeat. The behavioral change is: a class of observations that previously fired will now be suppressed. This is an Encoded Orientation decision — it changes behavioral defaults in a durable way (the config file persists). The change is traceable to vision.yaml constraint-4 ("minimize metabolic cost of cybernetic engagement") and the active focus on WOS executor health. This is a logged change with clear vision anchor; constraint-3 does not block.

**Alignment verdict: Confirmed.** The work is correctly scoped to an active operating problem (false-positive alerting during deliberate WOS pauses) and aligned with constraint-4.

**Pattern citations from learnings.md / golden-patterns.md (behavioral effect):**

- **Learnings 2026-05-18, PR #1190 ("New function with zero test coverage when PR states N passed"):** Reading this pattern caused me to enumerate every new code path introduced by this PR and check for corresponding test cases. Without this pattern I would have accepted "24 existing tests passed" as coverage evidence. With the pattern, I looked for test cases specifically asserting the suppression behavior. None exist. This produced Gap 1.

- **Learnings 2026-05-18, PR #1190 ("Mirror-constant extends to f-string prompt templates"):** Reading this caused me to check whether `pause_reason="user_command"` is a raw string literal in the call site that could diverge from any constant. The string appears twice: once in `handle_wos_stop` and once in the suppression guard in `check_backlog_alerts`. There is no shared constant. This is a latent mirror risk — if the reason string is ever changed, both sites must be updated manually. Raised as Gap 2 (advisory, not blocking).

## Stage 2 — Quality Review

### What the PR actually does

The diff contains changes to five files. The `relay_server.py`, `wos_dashboard.py`, and `tests/unit/test_orchestration/test_wos_dashboard.py` changes were already merged to main via PR #1191 (merged 2026-05-18T01:32:34Z; PR #1198 opened 2026-05-18T01:37:13Z from the same branch). The net new changes are in `steward-heartbeat.py` and `dispatcher_handlers.py` only.

**Not a feature-bundling concern** — the repeat files are a diff-base artifact, not independently added scope.

### steward-heartbeat.py

**Module-level import of `read_wos_config` (line ~61):** Correct. The new code needs `read_wos_config` at call time inside `check_backlog_alerts`. The module-level import is consistent with how the file handles other `src.orchestration` imports. The `noqa: PLC0415` comment on the function-body `is_execution_enabled` import is accurate and correct.

**Suppression guard in `check_backlog_alerts`:**

The guard fires only after `starvation_alert` is already True — so `read_wos_config()` is called only when the starvation condition has been met. This is good: no performance cost on normal cycles.

The `try/except Exception` wrapping `read_wos_config()` with fallback to `{}` is the correct defensive posture. On fallback, `execution_enabled` defaults to `True` (the guard condition: `not _starvation_wos_config.get("execution_enabled", True)` evaluates to `not True = False`), so the alert fires normally. The fail-safe direction is correct.

Guard condition: `not execution_enabled AND pause_reason == "user_command"`. The AND ensures that only deliberate user-commanded pauses are suppressed. An infrastructure disable (absent `pause_reason`) still fires. A different pause reason still fires. Both correct.

**The log message on suppression is info-level only** — it does not write to observations. The distinction between info log (developer visibility) and observation (user-facing alert) is maintained correctly.

### dispatcher_handlers.py

**`toggle_wos_core_jobs(enabled, pause_reason=None)` signature extension:**

The new `pause_reason` parameter is optional with default `None`, maintaining backward compatibility. The behavioral matrix:
- `enabled=True`: clears `pause_reason` from config. Correct.
- `enabled=False, pause_reason="user_command"`: writes reason. Correct.
- `enabled=False, pause_reason=None`: leaves existing `pause_reason` unchanged.

**Potential stale-reason risk:** The "leave unchanged" behavior on `enabled=False, pause_reason=None` means: if sequence is (1) user runs `wos stop` → writes `pause_reason="user_command"`, (2) something calls `toggle_wos_core_jobs(False)` without clearing pause_reason → old reason persists, starvation alert would be suppressed when it should not be. However, checking the codebase, `toggle_wos_core_jobs` is called from two places: `handle_wos_stop` (passes `pause_reason="user_command"`) and `handle_wos_start` (clears it). There is no other production caller that passes `enabled=False`. So the "leave unchanged" path is only reachable via a direct call that bypasses `handle_wos_stop` — which is not a current production path. The risk is latent but acceptable given the current call surface.

The docstring clearly documents the "leave unchanged" behavior, which is the right future-proofing.

### Encoded Orientation check

The `pause_reason` field addition changes the behavioral semantics of `wos-config.json` — previously a two-field file (`execution_enabled`, `prescription_model`), now three fields. The change is traceable to constraint-4 and the current operational focus. No logged prior is required beyond the PR itself as the decision record.

## Gap 1 — Missing test coverage for suppression behavior (BLOCKING)

**What is missing:** The PR introduces a new conditional branch in `check_backlog_alerts` that suppresses the starvation observation when `execution_enabled=False` and `pause_reason="user_command"`. This branch has zero test coverage. The PR introduces new behavior in `toggle_wos_core_jobs` (writing `pause_reason` on stop, clearing on start) with zero test coverage.

**Why it matters:** The suppression is the PR's stated purpose. If the condition is wrong (e.g., the default for `execution_enabled` in the guard uses `True` vs. `False`), the suppression either always fires (all alerts suppressed) or never fires (alert fires even when WOS is paused). Both failure modes are silent at merge time — existing tests pass regardless.

**What the revision must show:** At minimum, three tests in `test_backlog_alerts.py` or `test_toggle_wos_core_jobs.py`:

1. Starvation suppressed: `check_backlog_alerts` with starvation condition met, mocked `read_wos_config` returning `{"execution_enabled": False, "pause_reason": "user_command"}` → `_append_observation` NOT called, info log fired.

2. Alert fires when execution disabled but no pause_reason: mocked config `{"execution_enabled": False}` (no `pause_reason`) → `_append_observation` IS called.

3. `toggle_wos_core_jobs(enabled=False, pause_reason="user_command")` → `pause_reason` field written to config; `toggle_wos_core_jobs(enabled=True)` → `pause_reason` absent from config.

## Gap 2 — Raw string literal duplication for "user_command" (Advisory, not blocking)

**What is present:** The string `"user_command"` appears as a raw literal in two places: (1) `handle_wos_stop` call site in `dispatcher_handlers.py`; (2) the suppression guard comparison in `steward-heartbeat.py`. No shared constant.

**Why it matters:** If the value of this discriminator changes (e.g., renamed to `"manual"` or `"user_initiated"`), both sites must be updated manually. There is no static tooling to catch the divergence.

**Disposition:** Advisory — not blocking for this round. The current PR is the introduction of this string; both sites are touched in the same diff. The divergence risk is low while the string is stable. However, if the value ever changes, this will manifest as the suppression not firing (the starvation alert would fire incorrectly). A module-level constant `_PAUSE_REASON_USER_COMMAND = "user_command"` in `dispatcher_handlers.py` imported by both sites would close this permanently. Recommend but not requiring.
