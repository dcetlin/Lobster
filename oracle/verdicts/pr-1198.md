## Stage 1: Vision Alignment

**Prior entering review:** This implementation is solving the wrong problem, or solving the right problem in a direction that forecloses better paths.

**Finding:** The problem is real and correctly identified. When Dan runs `wos stop`, the starvation alert fires on every subsequent steward cycle because an empty ready-for-executor queue is anomalous under normal operation but expected under intentional pause. The alert is a monitoring signal, and a monitoring signal that fires continuously during expected conditions erodes trust in the signal when it fires during genuinely anomalous conditions. Suppressing it when `execution_enabled=false AND pause_reason="user_command"` is deterministic if-then logic (vision.yaml principle-3), and the suppression is appropriately scoped — infrastructure-triggered disables still alert normally.

The alternative path (stopping the heartbeat entirely when WOS is off) would be a larger change with broader side effects; the heartbeat performs useful Phase 1/2 work even when execution is off. Targeted suppression of the starvation alert is the right granularity.

**What this forecloses:** Nothing material. The `pause_reason` field in `wos-config.json` is additive and extensible.

**Alignment verdict:** Confirmed

**Encoded Orientation check (constraint-3):** The suppression behavior is a conditional logging decision (log.info vs. log.warning + observation). This is not an Encoded Orientation decision in the constraint-3 sense — it does not change agent identity, behavioral defaults in the OODA sense, or routing decisions. It gates a monitoring alert emission. No prior decision record required.

### Starvation suppression logic (the stated subject)

**Implementation correctness:** The condition is correct. `default=True` in `.get("execution_enabled", True)` means: if the config is unreadable, the expression evaluates to `not True AND ...` = `False`, so the alert fires normally. Fail-safe direction is correct.

**Lifecycle correctness:**
- `handle_wos_stop` calls `toggle_wos_core_jobs(enabled=False, pause_reason="user_command")` — writes the field. Correct.
- `handle_wos_start` calls `toggle_wos_core_jobs(enabled=True)` — pops `pause_reason`. Correct.
- `handle_wos_stop` idempotent path (already paused) returns early without calling `toggle_wos_core_jobs` — does not overwrite an existing non-user-command reason. Correct.

**Latent risk (not a blocker):** The string `"user_command"` appears as a raw literal in both `dispatcher_handlers.py` (write site) and `steward-heartbeat.py` (read site). Neither is a named constant. If someone changes one without updating the other, suppression silently stops working. The correct form is a named constant (e.g., `WOS_PAUSE_REASON_USER_COMMAND = "user_command"`) in `dispatcher_handlers.py` imported by the heartbeat. This is the raw-string-literal fragility pattern (StrEnum golden pattern, 2026-04-24) applied to a config field value rather than a DB column. Worth a follow-on issue.

### Gap 1: Stale test — `test_stop_calls_toggle_with_enabled_false` [BLOCKER]

`handle_wos_stop` now calls `toggle_wos_core_jobs(enabled=False, pause_reason="user_command")`.

The test at `tests/unit/test_orchestration/test_toggle_wos_core_jobs.py` line 352 asserts:
```python
mock_toggle.assert_called_once_with(enabled=False)
```

`assert_called_once_with` checks exact argument matching. The actual call includes `pause_reason="user_command"` which is absent from the expected args. This test **will fail**. Verified by running the mock pattern directly:

```
Expected: mock(enabled=False)
  Actual: mock(enabled=False, pause_reason='user_command')
```

This is the PR #827 stale-assertion pattern (learnings.md 2026-04-22): a PR changes a behavior and does not update the test asserting the old calling convention. The fix is to update the assertion to:
```python
mock_toggle.assert_called_once_with(enabled=False, pause_reason="user_command")
```

### Gap 2: Missing tests for both new behaviors [NEEDS_CHANGES]

This PR adds two new behaviors with no new tests:

**a) `toggle_wos_core_jobs` `pause_reason` lifecycle** — `test_toggle_wos_core_jobs.py` has no test verifying:
- `pause_reason="user_command"` written to config when `toggle_wos_core_jobs(enabled=False, pause_reason="user_command")` is called
- `pause_reason` absent from written config when `toggle_wos_core_jobs(enabled=True)` is called
- `pause_reason` unchanged in config when `toggle_wos_core_jobs(enabled=False)` is called without `pause_reason`

**b) `check_backlog_alerts` suppression gate** — `test_backlog_alerts.py` has no test verifying:
- Alert is suppressed (info log only, `_append_observation` not called) when `execution_enabled=False` and `pause_reason="user_command"`
- Alert still fires when `execution_enabled=False` and `pause_reason` is absent (infrastructure-triggered disable)
- Alert still fires normally when `execution_enabled=True`

The existing `_run` helper in `TestCheckBacklogAlerts` patches `_load_backlog_state` and `_append_observation` but does not mock `read_wos_config`. The new code path calls `read_wos_config()` at runtime — new tests must mock this to control the suppression condition. The new module-level import of `read_wos_config` in `steward-heartbeat.py` means the patch target is `src.orchestration.dispatcher_handlers.read_wos_config` (as used elsewhere in the test suite) or patching the module attribute on `_MODULE`.

This matches the 2026-05-18 #1190 pattern: "New function with zero test coverage when PR states 'N passed' from the unchanged test file."
