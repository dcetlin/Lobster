## Stage 1 — Vision Alignment (written before reading the diff)

**Theory of change:** Adding an exhaustiveness test for `_RETURN_REASON_CLASSIFICATIONS` prevents silent misclassification of return reasons. Without the test, a new return_reason omitted from the dict falls through to `_CLASSIFICATION_ERROR`, silently misrouting UoWs.

**Adversarial prior:** The test might be solving the wrong problem in the following specific way — if `ALL_KNOWN_RETURN_REASONS` in the test is a hardcoded local list rather than a programmatic derivation from the production enum, the test shifts the slip surface rather than eliminating it. A new return_reason added to the production system and not to the test's hardcoded list would produce a false pass: the test asserts coverage against a stale snapshot. This is the mirror-constant failure pattern from learnings.md (PRs #696, #800, #967, #970).

**Vision alignment:** Directly serves `principle-1` (proactive resilience over reactive recovery) and the WOS reliability orientation active this week. The work makes a previously-silent failure loud. No foreclosure concerns — adding tests does not constrain future directions.

**Alignment verdict: Confirmed.** The adversarial prior is the right one to hold; Stage 2 resolves it.

## Stage 2 — Quality Review

### What the implementation does

The test file imports `_RETURN_REASON_CLASSIFICATIONS`, `ReentryPosture`, and `ReturnReasonClassification` directly from `steward.py`. It builds `ALL_KNOWN_RETURN_REASONS` as the union of two sets:

1. `_REENTRY_POSTURE_RETURN_REASONS`: values from the `ReentryPosture` StrEnum — using `ReentryPosture.X` (enum member references), not string literals.
2. `_ADDITIONAL_RETURN_REASONS`: six locally-declared string constants for return_reasons that do not yet appear in `ReentryPosture`.

The exhaustiveness check is `ALL_KNOWN_RETURN_REASONS - _RETURN_REASON_CLASSIFICATIONS.keys()` asserted empty. Nine tests cover the full mapping.

### Resolving the Stage 1 adversarial prior

**For `_REENTRY_POSTURE_RETURN_REASONS`:** The enum values are referenced as `ReentryPosture.STALL_DETECTED`, `ReentryPosture.EXECUTOR_ORPHAN`, etc. These are imported enum members, not local string copies. If any of these enum values are renamed in production, the test breaks at import/attribute-access time rather than silently passing against a stale string. The mirror-constant failure mode does not apply to this set. This is the correct design.

**For `_ADDITIONAL_RETURN_REASONS`:** The six values (`_ORPHAN_KILL_BEFORE_START`, `_ORPHAN_KILL_DURING_EXECUTION`, `_OBSERVATION_COMPLETE`, `_NEEDS_STEWARD_REVIEW`, `_BLOCKED`, `_TIMEOUT`) are locally-declared string constants in the test file. These values do not appear in the `ReentryPosture` enum; they are written as raw strings by registry.py, startup_sweep.py, and the heartbeat kill classifier. Local constants are unavoidable for these values given the current production architecture.

This is a residual exposure, not a test design flaw. The test correctly documents this in a comment: "Additional return_reason string literals produced by the system that are not (yet) in the ReentryPosture enum." The comment acknowledges the architecture gap and signals the intended path (eventual enum promotion). The exposure is the same class as the mirror-constant pattern, but the mitigating factor is that there is no production enum to import from — the production code itself uses raw strings at these sites.

**Assessment:** The Stage 1 adversarial prior is partially confirmed in the narrow sense (the additional constants are local mirrors), but the risk is correctly bounded: these constants are local mirrors of string values that have no enum counterpart in production. The enum-backed portion of `ALL_KNOWN_RETURN_REASONS` is correctly structured.

### Exclusions from `_REENTRY_POSTURE_RETURN_REASONS` — correctness check

The PR excludes `FIRST_EXECUTION`, `EXECUTION_COMPLETE`, and `STARTUP_SWEEP_POSSIBLY_COMPLETE` from the exhaustiveness check, claiming these are "internal diagnosis postures, not written as return_reasons to the log."

Verified against the production code:

- `first_execution`: Used as a reentry_posture value returned by `_determine_reentry_posture()` and used in guards throughout the steward cycle (`if reentry_posture == "first_execution": return False`). It is a diagnostic output of `_determine_reentry_posture`, not a return_reason written to the audit log. The audit log `return_reason` field comes from `_most_recent_return_reason(audit_entries)`, which reads from the audit trail entries written by the executor — not from the posture computed by the steward. `first_execution` is never written as an audit log return_reason. Exclusion confirmed correct.

- `execution_complete`: Similarly used as a posture value in `_determine_reentry_posture()` and as a guard condition. The function `_determine_reentry_posture` returns `"execution_complete"` when the last execution event was `execution_complete` — this is a diagnostic assessment, not a return_reason the executor writes. Exclusion confirmed correct.

- `startup_sweep_possibly_complete`: Returned by `_determine_reentry_posture()` in crash-recovery paths where the output exists but came from a startup sweep. Same reasoning — it is a diagnostic classification, not an executor-written return_reason. Exclusion confirmed correct.

All three excluded values are internally-computed diagnostic postures. The reentry_posture field and the return_reason field are distinct: posture is computed by the steward from audit trail analysis; return_reason is what the executor wrote (or what the heartbeat classifier wrote for kill events). The three excluded values exist only on the posture side.

### The hardcoded list question

The implementer flagged: "ALL_KNOWN_RETURN_REASONS is hardcoded in the test — not derived programmatically. Is this acceptable or should it be dynamic?"

This is the right question. The answer is: acceptable under the current production architecture, with one condition. The condition is that the test file's comment accurately states why enumeration is unavoidable and what would constitute a fix (promote the additional reasons to the enum). The comment does this. The test is as programmatic as the production code permits: enum values are imported; non-enum values use named local constants with a stated rationale for each. The hardcoded portion is not arbitrary — it exactly mirrors the set of production return_reasons that lack enum backing.

The alternative (a fully dynamic derivation) would require the production code to maintain a canonical registry of all return_reason strings, which does not exist. Creating such a registry to enable dynamic test derivation would be the right long-term architecture — but that is a production change, not a test change. This PR correctly defers that work while providing the maximum coverage achievable without it.

### What would break without this

Without the test: a new ReentryPosture enum value added for a new execution failure mode (e.g., future `TIMED_OUT_HARD` or `PRECONDITION_FAILED`) would be omitted from `_RETURN_REASON_CLASSIFICATIONS` silently. The fallback `_CLASSIFICATION_ERROR` would fire on every UoW with that return_reason, routing them as errors when they might be abnormal or blocked. Historical precedent: this happened twice (`diagnosing_orphan` missing in #559; `orphan_kill_before_start`/`orphan_kill_during_execution` added in #963 without a guard).

### What patterns this introduces

The test introduces an explicit exhaustiveness pattern for StrEnum-backed dispatch tables. The structure is clean: enum members are referenced by attribute (not value), non-enum string values are documented as such with a comment explaining why they lack enum backing. This pattern should propagate to any similar dispatch table in the codebase (any dict keyed on enum-or-nearly-enum string values where coverage completeness matters).

### What this makes harder

Nothing foreclosed. The test does not constrain the production enum's evolution — adding a new enum member will cause test #1 to fail loud, prompting the implementer to add the corresponding classification. This is exactly the intended behavior.

### Residual observation (not blocking)

The six non-enum string constants in `_ADDITIONAL_RETURN_REASONS` are the same pattern class as the mirror-constant failures in learnings.md PRs #696, #800, #967, #970 — now in their fifth occurrence. The difference here is that there is no production constant to import. The correct long-term fix is to promote these strings to the `ReentryPosture` enum (or a companion enum), making them importable. The PR's comment acknowledges this with "not (yet)." A follow-on issue tracking this promotion would make the intent machine-traceable, but this is not a blocking gap — the current test is correctly structured for the current production architecture.
