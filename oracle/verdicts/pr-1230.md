## Round 1 — 2026-05-19

### Stage 1: Vision Alignment

**Adversarial prior:** This implementation is solving the wrong problem, or solving the right problem in a direction that forecloses better paths.

**Theory of change under scrutiny:** The PR assumes the forensic gap for silent WOS UoW failures is "no subprocess output was captured." The remedy is to redirect `claude -p` subprocess stdout+stderr to per-execution log files. The implicit assumption is that stdout/stderr is where the diagnostic signal lives when a subagent dies silently.

**Challenge to the assumption:** Claude Code agents (`claude -p`) write their primary result artifacts via MCP calls (`write_result`) and `result.json` — not to stdout. When a subagent dies silently (OOM, context length exceeded, binary crash), the forensic signal is likely in: the process exit code (not captured), the Claude API error response (not accessible from the parent process), the Claude Code binary's own session log (`~/.claude/logs/`), or the absence of `result.json` (already detected by orphan recovery). stdout from `claude -p` is the binary's operational chatter, not the subagent's reasoning trace. A cheaper test of the underlying assumption would have been: examine one failed UoW's post-mortem state and determine what information was actually missing from the diagnosis.

**However:** The PR's direction is not wrong — it is incomplete. Capturing subprocess output is additive, reversible, and zero-risk to existing behavior. It is a legitimate improvement even if stdout is not the primary crash signal. The work does not foreclose any better forensic path. The improvement serves `principle-1` ("Proactive resilience over reactive recovery") from vision.yaml — the change makes crash diagnosis structurally possible even if incomplete.

**Vision alignment:** Confirmed. This is WOS executor hardening, directly serving `current_focus.primary` ("WOS executor now live... Active work: WOS..."). The alignment is genuine — removing this change would not leave a structural hole in the executor's correctness, but it would leave a forensic blind spot that principle-1 argues against accepting.

**What this forecloses:** Nothing. The log file approach is additive.

**Opportunity cost:** Low. The change is small and the forensic value, while incomplete, is real.

**Pattern citations that constrained Stage 1:**
- The `multi-fire / accumulation` failure pattern (PRs #704, #896, #1034) caused me to hold log accumulation as a structural concern to verify in Stage 2, not dismiss as an implementation detail. Prior history shows this codebase has hit accumulation failures from scheduled tasks without rotation mechanisms — the same risk class applies here.
- The `canonical placement` golden pattern (2026-04-25) caused me to flag the log directory choice as a pre-read concern rather than accept it as obvious. Post-read, CONVENTIONS.md establishes `orchestration/` as the canonical parent — this concern was partially resolved by reading the conventions document.

### Stage 2: Quality Review

**Does it do what it claims?** Yes. `_dispatch_via_popen` now redirects stdout+stderr to a per-execution log file named `<run_id>.log` under `_SUBPROCESS_LOG_DIR`. The context manager wraps the Popen call. The log path is included in the dispatch log line.

**File handle / context manager correctness (advisory):** The `with open(log_path, "w") as log_fh:` block wraps `Popen(...)`. Since `start_new_session=True` and the subprocess is non-blocking, the file handle is closed by Python immediately after `Popen` returns — the subprocess is still running. On Linux, `Popen` duplicates the file descriptor into the child process at fork time before the parent closes it. OS-level reference counting keeps the underlying file open until the child exits. This is correct behavior on Linux. It is not documented in a comment, which means a future reader may incorrectly diagnose "parent closes fd before child writes" as a bug and "fix" it by holding the handle longer. A comment explaining the Linux fd inheritance is warranted. Advisory only.

**Log accumulation with no rotation (Gap 1 — disputed vs prior oracle):** `_SUBPROCESS_LOG_DIR` accumulates one `.log` file per dispatch indefinitely. No TTL, no cap, no cleanup mechanism. The prior oracle noted this as a real concern but treated it as a follow-on non-blocker, citing the `failure-traces/` cleanup precedent in steward-heartbeat. CONVENTIONS.md does list `failure-traces/` as "capped at 500 files / 30-day age by steward-heartbeat cleanup" — meaning the capping is an existing convention. The `logs/` directory introduced here has no equivalent cap. This is a real gap: a new subdirectory of `orchestration/` is added without the capping convention that CONVENTIONS.md implies for similar directories. The prior oracle's dismissal is defensible (immediate operational need outweighs accumulation risk), but the gap is real. This oracle holds it as a NEEDS_CHANGES item; the prior oracle held it as advisory.

**Log directory placement (advisory — resolved by CONVENTIONS.md):** `_SUBPROCESS_LOG_DIR = Path.home() / "lobster-workspace" / "orchestration" / "logs"`. CONVENTIONS.md establishes `orchestration/` as "Orchestration agent artifacts: intermediate outputs, observation queues, and multi-step pipeline artifacts produced by orchestration agents." The `logs/` subdirectory is analogous to `failure-traces/` (listed in CONVENTIONS.md). The placement is defensible per existing conventions. The prior oracle correctly cited CONVENTIONS.md. Advisory only — not a NEEDS_CHANGES item.

**Test assertions verify presence, not contract (Gap 2 — disputed vs prior oracle):** The test `test_stdin_devnull_stdout_stderr_log_file` asserts:
- `stdin == subprocess.DEVNULL` — correct, specific
- `stdout != subprocess.DEVNULL` — negative; does not verify the log file contract
- `stderr != subprocess.DEVNULL` — same
- `hasattr(stdout, "write")` — satisfied by any writable object (StringIO, sys.stdout, BytesIO mock)
- `hasattr(stderr, "write")` — same
- `stdout is stderr` — verifies shared handle; strongest assertion

The test does NOT verify:
- That the file path is under `_SUBPROCESS_LOG_DIR`
- That the filename embeds the `run_id` (the stated correlation mechanism)
- That `_SUBPROCESS_LOG_DIR.mkdir(parents=True, exist_ok=True)` was called

The PR's stated key feature is that "the log filename embeds `run_id` (`<uow_id>-<timestamp>`), so logs are directly correlated to registry entries." This correlation mechanism is the forensic value being added. The test does not verify it. The prior oracle called the assertions "substantive" and considered `stdout is stderr` load-bearing. This oracle holds that the correlation naming convention is equally load-bearing and equally untested — a future refactor that breaks the `run_id` naming would pass this test. The test-assertion-softening pattern from learnings.md (PRs #839, #1126, #1130) applies: `!= DEVNULL` plus `hasattr(..., "write")` cannot distinguish "correctly named log file at the right path" from "any writable object that isn't DEVNULL."

**Encoded Orientation check (constraint-3):** This PR changes subprocess I/O plumbing only. No behavioral defaults, decision-making rules, or agent identity are affected. Constraint-3 does not trigger.

**Feature bundling check:** Two files changed (source + its test file), both described in the PR. No bundling.

**Sub-second run_id collision (advisory):** If two dispatches for the same UoW occur within the same second (retry scenario), log filenames would collide. Edge case; not a NEEDS_CHANGES item.

### Revision Contract

**Gap 1 — Log accumulation with no capping convention:**
CONVENTIONS.md implies that `orchestration/` subdirectories have cleanup mechanisms (per `failure-traces/` precedent). `orchestration/logs/` lacks this. The revision must either: (a) add a cleanup step to steward-heartbeat for `orchestration/logs/` analogous to the `failure-traces/` cap, or (b) add a comment in the code and CONVENTIONS.md explicitly noting this directory does not have automated cleanup and stating where it will be added. Deferred-acceptable with an explicit tracked issue reference.

**Gap 2 — Test assertions verify presence, not contract:**
The revision must add assertions that verify: (1) the log file path is under `_SUBPROCESS_LOG_DIR` (import and use the constant in the test), and (2) the filename contains the `run_id` passed to `_dispatch_via_popen`. The test should verify the naming correlation that is the stated forensic value of this PR.
