## Stage 1: Vision Alignment

**Prior entering review:** This implementation is solving the wrong problem, or solving the right problem in a direction that forecloses better paths.

The problem being addressed (138h staleness in CC quota data) is real. The `statusLine` hook only fires during active CC sessions, so idle weekends — including quota reset cycles — leave `state.json` with pre-reset numbers for hours or days. Any executor dispatch gate or usage display built on this file reads garbage after an idle period. This is a direct violation of vision constraint-4 ("Minimize metabolic cost of cybernetic engagement") — stale quota data causes surprise exhaustion during AngelList sessions, which are the primary income source per `vision.yaml`.

**What would have to be true for this to be the right path:** The undocumented `claude.ai/api/usage` endpoint must remain stable enough to be worth maintaining. The session cookie approach must not become a recurring maintenance friction item that costs more cognitive load than the staleness problem it solves. If Anthropic adds an official quota API, the correct path flips.

**What does this foreclose:** Once a poller is in production, the pressure to find a more durable solution (official API, Anthropic SDK quota endpoint) decreases. The cookie-dependency is a recurring maintenance item with no retirement path. However, the scope is small and the problem is genuine. Principle-4 ("wire what exists before building more") supports fixing the staleness path before layering executor dispatch gates on top of it.

**Alignment verdict:** Confirmed. The problem is real and the fix is bounded in scope.

## Stage 2: Quality Review

### Patterns applied from learnings.md

**Mirror-constant pattern (eighth recurrence — learnings.md PRs #696, #800, #967, #970, #973, #974, #1032, #1116):** The oracle vocabulary loaded before this review made me check all test-file constants against the production module before reading any behavioral logic. This caused me to flag the `ADMIN_CHAT_ID` and `SENTINEL_PREFIX` re-declarations in `test_cc_usage_poller_expiry_alert.py` before evaluating whether the test logic was correct. The comment on those constants ("changes here are intentional and signal a spec change, not just an implementation change") is an unusual defense — but it does not change the divergence risk. When `COOKIE_EXPIRY_SENTINEL_PREFIX` or `ADMIN_CHAT_ID` changes in production, the test uses stale values silently. The golden pattern (named display cap with importable module-scope constant) confirms: the constants should be imported, not re-declared.

**Canonical inbox write utility bypass (learnings.md PR #1034):** `_write_cookie_expiry_alert` re-implements inbox writing instead of using `src/utils/inbox_write.py`. The PR #1034 learning explicitly named this as a defect pattern. However, this case has a defensible rationale: `write_inbox_message` in the canonical utility now has a circadian gate that defers non-urgent messages outside the 06:00-10:00 Pacific window. For a cookie expiry alert, deferral to the morning window is undesirable — the alert should be immediate. The bypass is defensible but undocumented. The code comment in `_write_cookie_expiry_alert` does not explain why it bypasses the canonical utility rather than using it.

**Caller-declared identity (golden-patterns.md 2026-04-22):** The `_write_cookie_expiry_alert` function writes a UUID4 message ID without a job-name prefix. The canonical utility uses `{job_name}_{uuid4().hex}` so log entries are traceable back to origin. The custom implementation loses this identity signal.

### Specific findings

**Gap 1: ADMIN_CHAT_ID is hardcoded — deviates from established env-var pattern**

`ADMIN_CHAT_ID = 8075091586` is a module-scope integer constant with no env-var override path. Every other scheduled task in the codebase that uses a chat_id follows the pattern:

```python
ADMIN_CHAT_ID: int = int(os.environ.get("LOBSTER_ADMIN_CHAT_ID", "8075091586"))
```

This applies to `system-retrospective.py`, `proposals-digest.py`, `los-action-scanner.py`, `dead-letter-sweep.py`, and `wos-health-check.py`. The cc-usage-poller deviates silently. In test environments or alternate deployments where `LOBSTER_ADMIN_CHAT_ID` is set to a different value, the poller still hardwires alerts to 8075091586. This makes the constant untestable without patching the module attribute directly — a pattern the learnings.md PR #839 entry identifies as a test barrier.

Fix: `ADMIN_CHAT_ID: int = int(os.environ.get("LOBSTER_ADMIN_CHAT_ID", "8075091586"))`

**Gap 2: Cookie file is created without chmod 600**

Migration 107 creates `~/lobster-user-config/cc-usage-session-cookie` as a comment-only placeholder. The file will contain a live Claude.ai session cookie once the user pastes their `sessionKey`. No `chmod 600` is applied in the migration or in the poller itself. On systems with default umask 022, the file is created as 644 — readable by any user on the system. The task prompt flagged this as an area to check and it is unaddressed.

Fix: Add `chmod 600 "$_cookie_config"` after the `cat > "$_cookie_config"` block in the migration.

**Gap 3: Mirror-constant in expiry alert test file**

`tests/unit/test_cc_usage_poller_expiry_alert.py` re-declares `ADMIN_CHAT_ID = 8075091586` and `SENTINEL_PREFIX = "/tmp/cc-usage-cookie-expired-alert-"` as module-scope constants instead of importing them from the production module. This is the eighth instance of the mirror-constant pattern. The file's comment ("changes here are intentional and signal a spec change") does not address the divergence risk — it describes a desired review behavior, not a structural prevention.

Fix: Replace local declarations with:
```python
ADMIN_CHAT_ID = cp.ADMIN_CHAT_ID
SENTINEL_PREFIX = cp.COOKIE_EXPIRY_SENTINEL_PREFIX
```

**Gap 4: Test file canonical placement — split home for the same module's tests**

`test_cc_usage_poller_expiry_alert.py` is placed in `tests/unit/` while `test_cc_usage_poller.py` is placed in `tests/unit/test_scheduled_tasks/`. Both test files test `scheduled-tasks/cc-usage-poller.py`. The golden pattern for canonical placement (2026-04-25): "the right home for any file must be unambiguous without interpretation." A new contributor looking for cc-usage-poller tests would find one file, not both. The other scheduled-task test files (`test_decay_detector.py`, `test_gmail_poll.py`) are in `test_scheduled_tasks/`.

Fix: Move `test_cc_usage_poller_expiry_alert.py` to `tests/unit/test_scheduled_tasks/test_cc_usage_poller_expiry_alert.py`.

### What works well

**Atomic write pattern is correct.** `tempfile.mkstemp(dir=state_path.parent)` + `os.replace` is the correct pattern. The temp file is created in the same directory as the destination, ensuring the rename is atomic even on systems where `/tmp` is a different filesystem. The exception handler calls `os.unlink(tmp_path)` to clean up on failure. This is clean.

**`_is_job_enabled` follows the reference implementation.** The function matches the Type B reference pattern documented in CLAUDE.md (citing `executor-heartbeat.py` and `steward-heartbeat.py`). Defaults to `True` on absent/unreadable jobs.json.

**Response shape fallback is defensible.** `parse_usage_response` tries `data.get("rate_limits")` first, then `data.get("usage", {}).get("rate_limits")` as an alternate shape. The error message on failure includes the actual keys returned, which surfaces unexpected API changes in the log. For an unverified endpoint, this is appropriate defensive coverage.

**Message type for cookie expiry alert is correct.** `type: "message"` is the Telegram message type — the dispatcher routes it as a user-visible message. `type: "subagent_result"` would require `sent_reply_to_user: False` and the dispatcher would relay it via `send_reply`; `type: "message"` causes the dispatcher to treat it as an incoming user message. On review, both paths deliver to Dan, but `type: "message"` with `chat_id` set means the dispatcher will process it as a user message needing a response — which may be unexpected. The canonical pattern for internal alerts is `type: "subagent_result"`. This warrants clarification in the code comment explaining the intent.

**`_inbox_dir()` correctly uses LOBSTER_MESSAGES.** The custom write function does use the `LOBSTER_MESSAGES` env override, avoiding the PR #1034 path-divergence defect.

**Migration number is correct.** Migration 107 follows Migration 106 with no collision in any open PR.

**jobs.json fields for Type B.** The PR description confirms the cron entry and jobs.json update are applied directly to production (not in the repo). This is consistent with Type B dispatch: `"type": "B", "dispatch": "cron-direct", "task_file": null`.

**Encoded Orientation assessment.** The automated cron polling and alert behavior constitutes Encoded Orientation in that the system acts autonomously every 30 minutes and sends alerts without per-run user input. However, the problem is documented (issue #1101), the behavior is self-evidently correct given the 138h staleness observation, and the oracle verdict for this PR serves as the logged prior going forward. Constraint-3 does not require a separate ADR for every Type B cron addition of this scope.
