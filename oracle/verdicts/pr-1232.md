## Round 1 — 2026-05-19

### Stage 1 — Vision Alignment (written before reading implementation)

**Theory of change in vision.yaml:** The WOS executor is live. The active constraint is `current_focus.current_constraint` — the WOS bidirectional interface (#1118) needs to support user-initiated actions. The phase intent is "Wire what exists before building more" (principle-4). The active project focus includes "WOS bidirectional interface (#1118) and WOS completion notification layer."

**Is `awaiting_owner` solving the right problem?** Yes. The alternative — fail the UoW and require Dan to re-prescribe — discards the execution context accumulated by the subagent and forces a cold re-start. The `awaiting-owner` state preserves that context and delivers a structured escalation with the specific decision text. This is a different and better path than fail-and-represcribe for the class of situations where a subagent has done useful work but hit a judgment gate.

**What would have to be true for this to be the right path:** (1) Subagents actually call `write_owner_decision_required()` rather than failing — the function must be in the executor contract and reachable. (2) The Telegram notification contains enough context for Dan to make the decision without going to the dashboard. (3) Dan's reply creates a recoverable state — there must be a working path from `awaiting-owner` back to `ready-for-steward`. (4) The UoW won't silently rot if Dan never responds.

**What this forecloses:** If `awaiting-owner` becomes overused for anything ambiguous (rather than specifically binary judgment calls), the retry budget signal degrades as an execution boundary. The guidance in the PR mitigates this with "Use only for genuine owner-level decisions."

**Opportunity cost:** This is building a new lifecycle state during a phase where "wire what exists" is the stated priority. However, it is a direct component of the WOS bidirectional interface (#1118) — the active constraint — so the cost is aligned with the stated horizon.

**Alignment verdict: Confirmed.** `awaiting_owner` is solving the right problem and the direction is correct. The concern entering Stage 2 was whether the path back from `awaiting-owner` to execution is implemented or merely documented.

#### Gap 1: No code path from `awaiting-owner` back to `ready-for-steward` (CRITICAL)

The bootup doc at `.claude/sys.dispatcher.bootup.md` says:

> Dan's reply in the primary thread provides the decision. To re-queue the UoW after receiving the decision, set it back to `ready-for-steward` with the decision as a note.

This implies the dispatcher will handle Dan's Telegram reply and re-queue the UoW. It does not. No such handler exists.

`registry.decide_retry()` only accepts `RETRYABLE_STATUSES = frozenset({"blocked", "ready-for-steward"})`. `awaiting-owner` is not in this set. When Dan sends a decision reply, there is no code path that reads the `awaiting-owner` UoW, records the decision, and transitions it to `ready-for-steward`. The UoW will be stuck in `awaiting-owner` indefinitely without manual intervention via `set_status_direct` (a test/admin function).

The wos-registry-reference.md state diagram correctly shows the intended flow (`Dan re-queues → ready-for-steward with decision note`) but the code does not implement it.

**What must be addressed:** Either (a) add `awaiting-owner` to `RETRYABLE_STATUSES` so `decide_retry` can re-queue it, or (b) implement a new handler (`decide_owner`, `handle_decide_owner_input`) that accepts the decision text and transitions `awaiting-owner → ready-for-steward` — and document the command in the bootup doc. The notification text "reply with the decision and I'll update the UoW" implies (b) is the intended design.

**What is addressed vs not addressed:** The UoW correctly goes into `awaiting-owner` and Dan is correctly notified. The return path does not exist in code.

---

#### Gap 2: `set_awaiting_owner` is not idempotent

`set_awaiting_owner` does not check whether the UoW is already in `awaiting-owner` status before writing. If called twice for the same UoW (e.g., due to a steward retry on an already-transitioned UoW), it will:
- Append a duplicate `steward_log` entry with the same `event: "awaiting_owner"`
- Write a confusing audit entry showing `from_status=awaiting-owner, to_status=awaiting-owner`

The function reads `old_status = row["status"]` but never compares it to `UoWStatus.AWAITING_OWNER` before proceeding. Compare to how other transition methods guard: `decide_retry` gates on `status IN ({placeholders})` so the UPDATE is a no-op if the status does not match.

**What must be addressed:** Add an idempotency guard — if `old_status == UoWStatus.AWAITING_OWNER`, return early (or return a sentinel) without writing duplicate entries.

---

#### Gap 3: No circuit breaker for indefinitely-stuck UoWs in `awaiting-owner`

No TTL, expiry mechanism, or escalation path exists if Dan never responds. Unlike `needs-human-review` (which has a hard cap / retry budget as prior context), `awaiting-owner` has no counter that drives escalation. This is a lower-severity gap — the system won't fail, but operator visibility of stale `awaiting-owner` UoWs requires a manual `/wos status awaiting-owner` query.

**What must be addressed:** Document in the executor-contract.md `Key invariants` section that `awaiting-owner` has no built-in expiry. Consider adding `awaiting-owner` to the WOS dashboard's status display cap so stale entries surface without manual query. This is a documentation/observability gap, not a code defect — may be deferred with explicit acknowledgment.

#### Non-blocking observations

- **Notification text creates false expectation:** `_write_owner_required_message` produces "reply with the decision and I'll update the UoW." Until Gap 1 is resolved, this text is aspirational, not descriptive. After Gap 1 is resolved, the text will be accurate.

- **Duplicate audit log write:** In `_process_uow`, `registry.append_audit_log()` writes an `owner_decision_required` event, then `registry.set_awaiting_owner()` writes a `status_change` event — both in the same code path. The first is redundant since `set_awaiting_owner` already writes the audit entry. Not a defect (double-logging is not harmful) but it adds noise to the audit trail.

- **Pattern check: mirror-constant (learnings.md multi-PR pattern):** The `ExecutorOutcome.OWNER_DECISION_REQUIRED` value `"owner_decision_required"` matches the literal used in the steward's `_assess_completion` branch (`elif outcome == "owner_decision_required"`). The steward does not import `ExecutorOutcome` — it uses the raw string. This is a StrEnum golden-pattern violation (golden-patterns.md 2026-04-24): enum-backed routing logic should use the enum constant, not the raw string. However, the same pattern exists for existing outcomes in the same branch (`"partial"`, `"failed"`, `"blocked"`) — this PR followed the existing convention rather than introducing it. Not a blocker for this PR; the pre-existing pattern should be filed as a separate issue.
