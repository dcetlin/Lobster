### Findings

**Finding 1 — Migration SQL comment misrepresents the schema (advisory)**

The migration's comment block says `Add gate_fired TEXT NULL DEFAULT 'none' to uow_registry` but the actual SQL is `ALTER TABLE uow_registry ADD COLUMN gate_fired TEXT NULL;` — no DEFAULT clause. The PR description claims SQLite's ALTER TABLE doesn't support DEFAULT expressions; this is factually incorrect. SQLite does support DEFAULT values on ADD COLUMN. The functional handling is correct (NULL is treated as severity 0 via COALESCE), but the stated reason is wrong and the comment creates a false specification contract. A future reader consulting the migration comment will believe the column defaults to 'none' when it does not.

**Finding 2 — Test column schema differs from production schema (advisory)**

All three registry integration tests manually add the gate_fired column with `ADD COLUMN gate_fired TEXT NULL DEFAULT 'none'`. The production migration adds the column without DEFAULT, so existing rows are NULL (not 'none') until first written. The first-write test (`test_write_gate_fired_stores_value`) calls `write_gate_fired(uow_id, "spiral")` on a row that starts as 'none' (because of the test's own `DEFAULT 'none'`), not NULL as production would have. The NULL first-write code path — `COALESCE(severity_case, 0)` returning 0 — is correct by inspection but not exercised by any test. The tests verify upgrade-only semantics against a slightly different starting state than production will encounter on its first write cycle.

This is not a defect — the COALESCE logic is obviously correct. It is a test coverage gap: the before-state matters for upgrade semantics, and the before-state tested is not the before-state production will see.

**Finding 3 — Mirror-constant pattern in test file (advisory)**

The test file declares local string constants `_GATE_SPIRAL = "spiral"`, `_GATE_DEAD_END = "dead_end"`, `_GATE_BURST = "burst"`, `_GATE_NONE = "none"`, and eligibility verdict strings, rather than importing them from `gate_fired.py`. This is the mirror-constant pattern (learnings.md, PRs #696–#980). The risk is lower than prior instances because the test assertions use the production `_GATE_TRANSLATION` and `_GATE_SEVERITY` dicts directly, so a gate name change would surface in those assertions. But the local constants remain a divergence point in the `TestGateTranslation` and `TestTranslateEligibilityToGate` classes. The correct form is to import the string values from the production module.

**Finding 4 — No steward-level dry_run integration test (advisory)**

The unit tests cover registry behavior and pure function logic. There are no tests verifying that the steward's `if not dry_run:` gate suppresses `write_gate_fired` when dry_run=True. The suppression is structurally correct by code inspection, but the call site is untested. If the indentation were ever changed (the `if not dry_run:` block refactored), no test would catch the regression.

### Summary

All four findings are advisory. The core implementation — SQL upgrade semantics, dry_run gating, try/except isolation, migration correctness — is sound. The COALESCE handles NULL correctly for first writes even though the tests don't directly exercise that path. The 30 tests pass cleanly. The PR is safe to merge.

Findings 1 and 2 should be tracked as a follow-on: the migration comment should be corrected, and a test should be added that exercises the NULL starting state explicitly (by adding the column without DEFAULT and verifying the first write succeeds).
