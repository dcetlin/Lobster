## Stage 2 — Quality Review

### Gap 1: `scheduled_task_crash` is not a registered message type — the dispatcher cannot route it

**This is the critical gap.** The PR description claims: "The dispatcher's LLM loop reads the message and relays it to Dan as a Telegram notification (same pattern as `wos_pr_sweep_result` and `wos_done` fast-path messages)."

This claim is false. `wos_pr_sweep_result` and `wos_done` have explicit handlers in `src/orchestration/dispatcher_handlers.py` and named entries in `sys.dispatcher.bootup.md`. The dispatcher's LLM loop knows exactly what to do with those types.

`scheduled_task_crash` has neither:
1. It is absent from `INBOX_SYSTEM_TYPES` in `src/mcp/message_types.py`
2. It has no handler in `sys.dispatcher.bootup.md`
3. It has no handler in `dispatcher_handlers.py`

When this message lands in the inbox, `mark_processing` will log a warning: `"mark_processing: unknown message type 'scheduled_task_crash'"`. The dispatcher will then encounter a message it has no instruction to handle. The message has `source: "system"` and a non-zero `chat_id`. The dispatcher's Section 261 rule says "Do NOT call `send_reply` for system messages — there is no user to reply to." But crash alerts with a non-zero `chat_id` ARE meant to relay to the user.

The actual behavior depends entirely on the LLM dispatcher's in-context judgment at the moment the message arrives — which is exactly the "prose routing" failure mode that golden-patterns.md calls out as the anti-pattern to eliminate. The golden pattern `[2026-04-01] Golden pattern candidate: Python dispatch table as compaction-resilient routing seam` states that routing must be structurally encoded, not left to LLM prose interpretation.

The golden patterns file affected my weighting here: because the Python dispatch table pattern is named as the canonical solution for precisely this failure mode, the absence of a dispatch table entry is not a minor gap — it is the structural failure mode this pattern exists to prevent.

**Required fix:** Add `"scheduled_task_crash"` to `INBOX_SYSTEM_TYPES` in `message_types.py`. Add an explicit handler section to `sys.dispatcher.bootup.md` that instructs the dispatcher to call `send_reply(chat_id=msg["chat_id"], text=msg["text"], source=msg.get("source", "telegram"))` and then `mark_processed`. The routing is simple — the missing piece is the structural encoding.

### Gap 2: `_write_crash_alert` is copy-pasted between the two heartbeat files rather than using `inbox_write.py`

`src/utils/inbox_write.py` was created specifically to consolidate the copy-pasted `write_inbox_message` pattern that previously appeared in six separate scripts (PR #805, learnings entry `[2026-04-22] Pattern: caller-declared identity in shared utilities`). The `_write_crash_alert` function is an identical implementation in both heartbeat files — same logic, same filesystem write path (`LOBSTER_MESSAGES / "inbox"`), same atomic tmp-then-rename pattern.

The correct structural home for this function is `inbox_write.py` (or a crash-specific utility). Each caller passes `job_name` as an identity parameter — the exact pattern established in PR #805.

This is not a correctness gap (both copies are correct), but it is a structural debt gap. As soon as a third heartbeat script needs crash alerting, the pattern triples. The golden-patterns entry `canonical placement must be obvious from first encounter` applies: a new heartbeat author reading the codebase will see `inbox_write.py` but also the ad-hoc `_write_crash_alert` in the heartbeat files — the canonical home is not unambiguous.

**Required fix:** Extract `_write_crash_alert` into `src/utils/inbox_write.py` as `write_crash_alert(job_name: str, exc: BaseException, extra: str = "") -> None`. Both heartbeat files import and call it. The PR description's "Follow-up" note acknowledges other scheduled tasks need the same pattern — this makes the utility version even more important before that follow-up ships.

### What does not fail

- The outer try/except in `main()` correctly catches `Exception` (not `BaseException`). This is the right choice: `KeyboardInterrupt` and `SystemExit` should not be caught by crash alerting. The pattern is structurally sound.
- The `_write_crash_alert` function is correctly marked best-effort (try/except around the write, log but do not mask original exception). The original exception is not swallowed.
- The exit code change (return 1 on crash) is correct — cron records the failure.
- The ImportError handler change (warning→error + inbox alert) introduces no double-alerting risk. The ImportError fires in `run_ttl_recovery()`, which is called from `_main_inner()`. If the ImportError escapes `run_ttl_recovery()` uncaught, the outer crash handler in `main()` would also fire — but the ImportError IS caught inside `run_ttl_recovery()` and handled explicitly. Only one alert fires per cron cycle.
- The traceback truncation (last 2000 chars) is a sound design for Telegram's message length limits.
- The atomic tmp-then-rename write pattern matches the canonical inbox write pattern.
- The `_ADMIN_CHAT_ID` type (str) matches the pattern in `wos-health-check.py` and `wos-metabolic-digest.py`. No type mismatch risk.
- The `user-update.sh` path fix (lobster-workspace → lobster) is a correct bug fix for the oracle learnings weekly job.

### Encoded Orientation check

This PR does not change behavioral defaults, agent identity, or decision-making rules in a durable way. The crash-alert pattern is compensatory infrastructure, not a new orientation. Constraint-3 does not apply.

### What patterns this introduces

If Gap 1 is not fixed before merge, this PR introduces a new message type that the dispatcher will encounter but has no structural routing for — relying instead on LLM judgment under compaction pressure. Each subsequent heartbeat crash during production would produce an inbox message the dispatcher must improvise a response to. That improvised response may be correct sometimes and silent sometimes — the behavior is non-deterministic.
