## Stage 1 — Vision Alignment (written before reading implementation)

**Theory of change:** The CC quota gate prevents new UoW dispatch when the five-hour Claude Code token budget is critically low. This converts reactive failure (subagents failing mid-execution from token exhaustion) into structural prevention (dispatch never starts when headroom is insufficient).

**Vision anchor:** `constraint-4` ("Minimize metabolic cost of cybernetic engagement — reduce friction and screen time, not maximize output volume or feature density") is the primary anchor. When a dispatched subagent fails mid-flight from quota exhaustion, Dan must intervene to recover work and restart execution. That is increased screen dependency and metabolic cost. The gate eliminates the failure case that causes that intervention. `principle-1` ("Structural prevention is preferred over reactive recovery") is the implementation anchor.

**What would have to be true for this to be right:** The `cc-budget/state.json` file is a reliable, machine-readable signal written by the cc-usage-poller (PR #1122). The fail-open design (absent/stale/malformed → proceed) is correctly conservative. The gate is cheap per-cycle and non-invasive.

**What this forecloses:** Nothing significant. The gate is advisory-to-structural and fail-open. It does not alter state, block TTL recovery, or create locked conditions.

**Alignment verdict:** Confirmed. This PR serves `constraint-4` via `principle-1`. No foreclosure risk.

## Stage 2 — Quality Review

### Gap 1: Schema mismatch — gate is permanently dead in production

**This is the critical finding.** The production `~/.claude/cc-budget/state.json` written by the cc-usage-poller (PR #1122, merged 19:07 UTC the same day this PR was opened at 19:13 UTC) does NOT contain the keys `_read_cc_quota()` reads.

The PR reads:
- `data["five_hour_pct"]` — top-level key
- `data["fetched_at"]` — top-level key

The actual production schema (from the live file and from `cc-usage-poller.py`'s `merge_into_state` function) is:
- `data["rate_limits"]["five_hour"]["pct"]` — the percentage value
- `data["last_updated"]` — the ISO freshness timestamp (written by cc-usage-poller)

Both `data["five_hour_pct"]` and `data["fetched_at"]` cause `KeyError` at the JSON parse step, which is caught by `except (KeyError, ValueError, TypeError): return None`. The function therefore always returns `None` in production, the gate always passes fail-open, and the CC quota gate is structurally wired but semantically dead from day one.

The tests all pass because the test helper `_write_state()` writes a synthetic file containing `five_hour_pct` and `fetched_at` — which is the schema the PR assumes, not the schema that exists. The tests verify the function's internal logic correctly but against a phantom schema.

**What the fix requires:** Change `_read_cc_quota()` to read the actual production schema:
- `pct = float(data["rate_limits"]["five_hour"]["pct"])`
- `fetched_at_str = data["last_updated"]`

Update `_write_state()` in the test helper and all test assertions to use the actual schema. The `_CC_QUOTA_FRESHNESS_SECONDS` constant may also need to be re-evaluated: `last_updated` is written every 30 minutes (the poller's cron cadence), so a 60-minute freshness window is two missed cycles — correct as a staleness gate.

**Learnings.md anchor:** PR #1122 entry ("Endpoint change without test update leaves stale fixtures") is the exact pattern class. A schema change in the producer (PR #1122's `merge_into_state`) was not reflected in the consumer (PR #1125's `_read_cc_quota`). The PR #1122 commit message explicitly documented the shape change. Behavioral effect on this analysis: the schema mismatch was the first thing verified after confirming that both PRs touched the same file type. Without the learnings.md pattern as prior, the focus would have been on the gate logic and test coverage rather than the contract between poller output and reader input.

### Gap 2: Encoded Orientation — advisory finding, not a blocker

The CC quota gate is an Encoded Orientation decision under constraint-3: the system autonomously suppresses dispatch cycles without Dan's explicit per-cycle input. A prior logged decision exists in `oracle/verdicts/decision-github-rate-limit-gate.md` that explicitly names the structural class ("the system autonomously suppresses dispatch cycles without Dan's explicit input per invocation") and is anchored to `principle-1`.

This prior covers the structural class but not the specific mechanism. The GitHub rate limit gate decision references two sibling decisions (`decision-needs-human-review-escalation.md`, `decision-system-retrospective-automation.md`) as structural-class references, establishing the pattern that each new Encoded Orientation decision in the same class gets its own logged decision file.

**The CC quota gate does not have its own logged decision file.** Given the close structural identity with `decision-github-rate-limit-gate.md` (same structural class, same principal-1 anchor, same fail-open pattern, same per-cycle cadence), I treat this as an advisory finding rather than a NEEDS_CHANGES blocker. However, a `decision-cc-quota-gate.md` in `oracle/verdicts/` should accompany this PR to close the audit trail correctly. This is required before APPROVED.

**Revision contract for Gap 2:** Add `oracle/verdicts/decision-cc-quota-gate.md` with: (a) vision.yaml anchor (`constraint-4`, `principle-1`) by exact field path, (b) reference to the structural class established by `decision-github-rate-limit-gate.md`, (c) the authorized threshold value (`CC_QUOTA_SKIP_THRESHOLD = 90.0`), (d) the fail-open constraint, and (e) the scope constraint (TTL recovery unaffected).

### Passing checks

**Functional correctness (conditional on fix):** The gate logic in `main()` is correct — `cc_pct is not None and cc_pct >= CC_QUOTA_SKIP_THRESHOLD` is the right guard. The `return 0` after logging correctly exits the cycle without affecting TTL recovery (which runs earlier in `main()`). The debug log for below-threshold quota is correct operator tooling.

**Fail-open design:** Correct throughout. The double `except Exception: return None` in the file read, plus the `except (KeyError, ValueError, TypeError): return None` in the parse step, ensure all failure modes are fail-open.

**Env-var override:** `LOBSTER_CC_BUDGET_STATE` is the correct pattern. The test coverage for this env-var (`test_uses_env_var_path_over_default`) confirms the override works.

**Module-level constant imports:** The test correctly imports `CC_QUOTA_SKIP_THRESHOLD` from the production module rather than re-declaring it locally. This follows the golden pattern (StrEnum/named-constant import) and avoids the mirror-constant anti-pattern (learnings.md PRs #696, #800, et al.).

**Importlib re-export bridge pattern:** The test loading mechanism (`_load_executor_heartbeat()` using importlib + sys.modules pre-registration) correctly applies the golden pattern from golden-patterns.md ("importlib re-export bridge for backward-compatible file splits"), patching heavy imports before exec_module. The `sys.modules` pre-registration prevents double-execution.

**Feature bundling:** Two files changed (production script + test file), both described in the PR. No undisclosed additions.

**TTL recovery isolation:** The gate is positioned after the GitHub rate-limit check and before `run_executor_cycle`. TTL recovery (`_recover_stale_uows`) runs unconditionally before the gate. This is correct — the quota gate does not suppress recovery operations.
