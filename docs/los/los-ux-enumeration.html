State System
Every todo item in LOS has exactly one state. The state machine is source-agnostic — where an item came from (Obsidian vs. Telegram vs. Lobster) does not change how states work or transition. Source is provenance metadata only.
State Meaning How to enter Exits to
open Active, unfinished. Appears in Obsidian and surfaces in triage/nudges. Created (any source). Snooze expires. done, snoozed, dismissed, deleted
done Completed. Retained in DB for weekly review. DB is authoritative — Obsidian checkbox reverts if unchecked. Dan checks the box in Obsidian, or /todo done via Telegram. open (BUILD not yet implemented — reopen is planned)
snoozed Deferred until a date. Hidden from active view during snooze window; auto-resurfaces when snoozed_until passes. /todo snooze via Telegram, or inline button. open (automatic on date expiry)
dismissed Intentionally dropped — not done, not snoozed, just not worth doing. Retained in DB. Does not reappear. Telegram inline dismiss button. (terminal — no automatic exit)
deleted Line removed from Obsidian file without checking it. System infers intent and marks deleted. Does not reappear. Dan deletes the line in Obsidian (Obsidian-sourced items only — see §1.4). (terminal)
Valid transitions: open ↔ snoozed  |  open → done  |  open → dismissed  |  open → deleted (Obsidian-sourced)  |  BUILD reopen (done → open) ○ PLANNED.

CHECK A TODO ITEM (mark done)
✓ LIVE
Trigger: Dan toggles [ ] → [x] in Obsidian
Lifecycle
  1. Dan checks the box; Obsidian git plugin pushes to remote
  2. vault-watcher detects new remote HEAD (~30s polling interval)
  3. After debounce (~60s), vault-processor fires
  4. git pull --rebase --autostash merges latest
  5. sync_obsidian_to_db(): [x] line → mark_done() in DB
  6. apply_status_delta(): DB says done, file already shows [x] — no change
  7. git commit + push (if any other changes)
Failure: If Obsidian git plugin doesn't push, no sync fires and item stays open in DB. Recovery: /todo done <id> via Telegram.

§1.3
SNOOZE A TODO ITEM
✓ LIVE
Trigger: /todo snooze <query> [days] via Telegram, or inline button todo-snooze-{id}-{date}
Lifecycle
  1. mark_snoozed(conn, item_id, until_date) sets status='snoozed', snoozed_until=YYYY-MM-DD
  2. On next apply_status_delta(): snoozed items with future snoozed_until treated as open for file rendering
  3. After snoozed_until passes: get_open_items() treats item as open again
No snooze mechanism in Obsidian directly — Telegram-only today.
Design: Snooze Archetypes
Snooze is not a single concept. Before expanding it, we need to decide which archetypes we're building:
  • ① Temporal deferral — "I'll deal with this on date X." Item is hidden until that date, then resurfaces as open. This is what snooze does today. The item's state changes to snoozed and auto-transitions to open on expiry.
  • ② Notification trigger — "Remind me about this at time/event T." The item stays open; Lobster pings Dan at the specified moment. This is a different behavior: the item is not hidden, a notification is added. This does not exist yet. It requires a scheduled alert mechanism, not a state change.
  • ③ Visual declutter — "I know it's there; hide it from view until I ask." No date, no reminder — just suppressed from the active view. Closer to a soft-dismiss than a deferral. Partially overlaps ① today (a long snooze achieves this), but the intent is different: the user is not setting a return date, just reducing clutter.
DECISION Current snooze = ① + ③ (temporal deferral that also visually hides). Should we also build ②? Archetype ② is the higher-leverage bet for a personal assistant — the user shouldn't have to initiate recall; Lobster should push. These three could be separate commands or a unified "snooze with optional remind" model.

§1.4
DISMISS A TODO ITEM
✓ LIVE
Trigger: Telegram inline button todo-dismiss-{id}
Lifecycle
  1. mark_dismissed() sets status='dismissed', records dismissed_at
  2. Dismissed items retained in DB (reviewable in weekly review)
  3. On next vault sync: absent from apply_status_delta() output
Dismissed vs. Done — they are different terminal states:
  • Done (status='done') — the task was completed. Dan toggled [x] in Obsidian, or sent /todo done. The item carries a completion timestamp. It appears in weekly review as an accomplishment.
  • Dismissed (status='dismissed') — the task was intentionally dropped. Dan decided it isn't worth doing. No checkbox involved. Triggered only via Telegram inline button. Retained in DB for review but does not resurface.
The key distinction: done means finished; dismissed means abandoned. Both are retained in the DB. Neither resurfaces automatically.
Why does origin affect the deletion scan?
Today, the deletion scan (sync_obsidian_to_db()) only marks items as deleted if their source='obsidian:ACTIVE TODOS.md'. The rationale was: Obsidian items were explicitly added to the file by Dan; if they're gone from the file, Dan removed them, so deletion is the correct inference. Telegram-sourced items were written to the file by Lobster (in the additions block); their absence from the file could mean the block wasn't rendered yet, not that Dan deleted them — so deletion was unsafe to infer.

Position going forward: REMOVE This origin-branching in the state machine is a pragmatic holdover, not a design principle. Source should be provenance metadata only — it should describe where the item came from, not how the state machine behaves. The right long-term fix is smarter deletion inference (e.g., a deletion confirmation flow, or tracking render vs. user-removal separately). See §2.5–§2.7 for the broader Source Parity principle.

Design: Category Grouping Within Priority
Items under a priority section (e.g., Active) are further grouped by workstream subsection (### LOS, ### Site, etc.). This grouping currently serves two functions simultaneously: spatial organization (items cluster visually) and tagging (DB workstream field is set from the subsection header).

The coupling is convenient but creates questions: What if an item belongs to a workstream but shouldn't be grouped with others from that workstream? What if the best spatial grouping doesn't align with the workstream taxonomy? The current model treats "where it appears" and "what it belongs to" as the same signal. TRACK This is fine for now but may eventually need to be two separate fields — a display section and a workstream tag — especially as the todo file grows in density.

Design: New Item Surfacing (Lobster-Generated Items)
When Lobster adds items to the vault (via /todo add or autonomous capture), they land in the <!-- lobster-additions --> block at the bottom of the file — unplaced, unranked, awaiting Dan's triage. This is correct behavior but the surfacing is passive. Dan may not notice new items have been added until the next time he scrolls to the bottom.

The higher-leverage pattern: new Lobster-generated items should be visibly flagged as awaiting triage. Options under consideration:
  • (a) Insert at top of Someday — Lobster-added items default to Someday/P8 and appear at the top of that section. Dan sees them on next open. Requires Lobster to make a placement decision (which section to default to).
  • (b) Insert in most appropriate section + [NEW] flag — Lobster infers priority and places the item, but decorates it with a [NEW] inline comment. The flag disappears once Dan moves or touches the item. Requires inference + decoration tracking.
  • (c) Separate "Inbox" section — All Lobster-generated items land in a dedicated ## Inbox section at the top of the file. Dan triages from Inbox into priority sections. Clean separation, maximum visibility. Adds a triage step Dan currently doesn't have.
DECISION Option (c) is most structurally clean and is consistent with the "extension of mind" framing — the Inbox is the cognitive buffer. Option (a) is easiest to implement today. Option (b) requires tracking that doesn't exist yet.

§2.4
MOVE OR REORDER AN ITEM
✓ LIVE
Trigger: Dan cuts/pastes or drags an item to a different position or section
Lifecycle
  1. If item moves to a different section: sync_obsidian_to_db() detects priority band change → _update_priority()
  2. If item moves within the same section: no DB change; file order preserved
  3. apply_status_delta() preserves the new order (only checkbox prefixes touched)
Design: Priority–Section Coupling
The Obsidian file structure encodes priority as spatial position: items under ## Urgent are P1–P3, ## Active is P4–P6, ## Someday is P7–P9. This is elegant because the file is readable without any metadata — the structure is the priority system. Dan's intuition ("tool for my mind") aligns naturally with spatial layout.

The tension: priority (urgency/importance) and spatial section (grouping/organization) are currently the same concept. That means you cannot mark something as urgent-priority without moving it to the top section, and you cannot organize the Someday section into subsections without implying it's all P7–P9. As the system grows, these two concerns may want to diverge — for example, an item that is strategically important but genuinely long-horizon might benefit from a separate priority signal independent of section placement.

DEFER The markdown/hierarchical model is expressive enough for the current scale. The coupling is a limitation of the format, not a design mistake. If this ever becomes a pain point, the right move is an explicit priority: tag in the item text (e.g., - [ ] Fix login flow <!-- priority:urgent -->), but that introduces complexity that isn't justified today.

◕ SCAFFOLDED
Trigger: Dan edits the text of an existing checkbox line
Lifecycle
  1. sync_obsidian_to_db(): new text has no matching dedup_key → inserted as new item

Gap: If item was Telegram-sourced, the deletion scan skips it — DB has two open rows (old + new text). BUILD No automated dedup — ○ PLANNED.
Architectural Principle: Source Parity
The fact that this gap is source-dependent points to a deeper structural issue worth naming explicitly. The current system branches on source in multiple places: deletion scan, additions block rendering, file-presence inference. Each branch was added for a pragmatic reason, but together they make the state machine source-aware in ways that should not be necessary.

The Source Parity principle: Source is provenance metadata. It tells you where an item came from for audit, attribution, and triage purposes. It should not determine how the state machine behaves. All items — regardless of origin — should compete on the same footing, transition through the same states, and be subject to the same sync logic.

The current branching-on-origin is a pragmatic holdover from the incremental build. Each origin-specific branch is a candidate for removal once the underlying uncertainty it was handling (e.g., "we don't know if the user deleted this or if the block wasn't rendered") is resolved by better mechanism design. See §2.6 and §2.7 for related instances.

§2.6
DELETE AN ITEM (remove line)
✓ LIVE
Trigger: Dan removes the - [ ] <text> line entirely (without checking it)
Lifecycle
  1. Deletion scan in sync_obsidian_to_db(): DB open items with source='obsidian:ACTIVE TODOS.md' whose dedup_key was not seen → mark_deleted()
  2. Deleted items do not reappear in apply_status_delta()
Important: Only obsidian-sourced items get deletion-detected. Telegram-sourced items removed from the file will reappear in the additions block on next render.
Design note: Why shouldn't all sourced items get the same deletion treatment? The current asymmetry exists because the system can't distinguish "Dan deleted this Telegram-sourced line" from "Lobster just hasn't written it to the file yet." Obsidian-sourced items don't have this ambiguity — if they're in the DB and not in the file, Dan removed them.

The Source Parity goal would remove this distinction. BUILD One path: track whether a Lobster-added item has been successfully rendered to the file (a rendered_at timestamp or rendered=true flag). Once rendered, its absence from the file can be confidently attributed to Dan's intent, and the deletion scan can treat it the same as Obsidian-sourced items. The system should trust and defer to Dan regardless of origin — the current asymmetry is a workaround for a gap in render tracking, not an intended feature.

ADD AN @LOBSTER ANNOTATION
✓ LIVE
Trigger: Dan types @lobster <command text> anywhere in any vault .md file
Full Lifecycle
  1. vault-watcher detects push; vault-processor fires after debounce
  2. Step 5: _collect_files_to_scan() returns files per annotation_scope config (all = all vault .md files; watched_only = watched list only)

  6. Dispatcher picks up inbox JSON and processes as a normal user message
Gap — No acknowledgement ping: After dispatch, Dan receives no "received your annotation" Telegram message. The annotation is silently consumed. The dispatched_at: comment is written and immediately removed — no trace remains in the file. Dan must infer dispatch from the annotation line disappearing on next pull.

BUILD Planned mitigation (○ PLANNED, Tier 2a): Obsidian plugin will show inline status decorations (dispatched / open / done) on @lobster lines.
Deduplication: each annotation gets a SHA-256 content-hash message_id. If the same annotation fires twice (retry scenario), the dispatcher's dedup gate prevents double-processing.

§3.2 vs. §3.1 — what's the difference?
§3.1 covers the infrastructure of @lobster annotations: how they're detected, marked, dispatched to the inbox, and removed from the file. It describes the wiring regardless of what you write after @lobster.

§3.2 covers the content of @lobster annotations when they include a specific command or structured text — e.g., @lobster /todo add Fix the login flow. The distinction matters because §3.1 explains "this annotation was seen and dispatched" while §3.2 explains "once dispatched, what happens to the text." The answer for §3.2 is: the annotation text is handed to the Lobster dispatcher as if Dan had typed it in Telegram. All Telegram commands, natural-language routing, and NLP apply. There is no vault-specific command vocabulary today — writing @lobster /todo add X in the vault is functionally identical to sending /todo add X in Telegram.

EDIT SECTION HEADERS OR STRUCTURE
✓ LIVE (partial)

Lifecycle

Design note — structure, views, and flexibility
The current section structure is a good default but it should be treated as convention, not law. The system works best when it bends to how Dan's mind works, not the other way around. A few directions worth noting:

DEFER Views: The Obsidian file is one view — a flat, spatially-organized list. As the item count grows, other views become useful: by workstream, by staleness (items untouched for 7+ days), by snooze expiry, by a custom filter. These don't require changing the file structure; they're computed views over the DB. This is where Lobster can add leverage without touching the vault format.

BUILD Antifragility: The system should be robust to structural experiments. Dan reshuffling headers, adding new subsections, or trying different organizational schemes should produce graceful degradation (items default to Active priority) rather than silent data corruption. The current failure mode (silent priority misassignment) is suboptimal — the system should alert when it can't parse a header, not silently fall back.

TRACK Extension of mind: The most important structural property is that the file remains a direct cognitive interface. Every affordance Lobster adds — auto-placement, [NEW] flags, inline decorations — must earn its place by reducing cognitive friction, not adding to it. When in doubt, prefer invisible infrastructure over visible UI chrome.

§6 Vision and High-Leverage Bets
§6 — section comment

LOS is an extension of mind, not a productivity app. The goal is not to help Dan track more things — it is to reduce the cognitive load of the tracking itself to near zero, so his attention is free for the actual work. What follows is Lobster's first-principles read on what would make a 10x version of this system.

Bet 1 — Capture Fidelity: Zero Friction from Intent to Record
The highest-leverage primitive is making capture so fast and natural that the cost of not capturing an item exceeds the cost of capturing it. Today there are still too many capture paths with different friction levels: typing in Obsidian (high trust, medium friction), /todo add in Telegram (medium friction), @lobster in any .md file (low friction but vault-scoped). The 10x version has one pattern that works everywhere and costs under 5 seconds. The Obsidian Tier 2a plugin (hotkey capture, quick-add modal) is the right direction. The bet: every unrecorded item that gets lost costs more than any amount of infrastructure to prevent it.
Bet 2 — Triage Surfacing: Lobster-Added Items Are Never Invisible
When Lobster adds an item to the vault, Dan should know about it within the next time he opens Obsidian — not by scrolling to the bottom of the file. This is the highest-leverage UX fix in the current system. An Inbox section (see §2.3 option c) or a Telegram ping ("I added 2 items to your vault — want to place them?") would close this gap. The deeper principle: Lobster is a collaborator, not a silent scribe. When it makes a decision (adding an item, inferring a priority), it should surface that decision to Dan so he can accept, reject, or redirect it. Silent adds are a trust gap.
Bet 3 — Proactive Push: The System Initiates, Dan Doesn't Have to Remember
The gap between "I snoozed this" and "it resurfaced when I needed it" is where most productivity systems fail. The current snooze (archetype ①) resurfaces items on date expiry but doesn't push — Dan still has to open Obsidian. The 10x version closes this gap with archetype ②: Lobster pings Dan at the right moment, on the right channel. This is especially high-leverage for items that are context-dependent ("remind me when I'm near X", "resurface when PR #N merges") rather than date-dependent. The bet: proactive delivery of the right item at the right time is worth more than any organizational scheme.
Bet 4 — Staleness Detection: Items That Don't Move Should Prompt Action
An item that stays in Active for 14 days without any status change is a signal — either it's blocked, no longer relevant, or the friction to close it is too high. Today the system doesn't react to staleness. The 10x version tracks item age and uses it as a triage signal: items open beyond a threshold surface in a weekly nudge ("3 items have been open for 2+ weeks — snooze, dismiss, or commit?"). This doesn't require Dan to maintain the list; it requires Lobster to surface the list's own drift. The nudge is the leverage, not the tracking.
Bet 5 — Source Parity and Structural Simplicity
Every place the current codebase branches on source in the state machine is a candidate for removal. The architectural goal is a source-agnostic state machine where all items compete on the same footing: same states, same transitions, same sync logic. The payoff is not just cleaner code — it's that the system becomes predictable. Dan should be able to reason about what will happen to an item without knowing or caring where it came from. Source should be an audit field, not a routing field. This is structural debt with a clear payoff trajectory.
Bet 6 — Views Without File Complexity
As the item count grows, a single flat file organized by priority/workstream becomes harder to navigate. The high-leverage answer is not to make the file more complex — it's to provide computed views over the DB that can be rendered on demand: "show me everything tagged #lobster that's been open 7+ days," "show me today's Urgent items only," "show me what I snoozed this week that's coming due." These views require no changes to the Obsidian file format; they're just queries. A Telegram command (/todo view today) or a secondary Bisque dashboard would surface them without adding any structure to Dan's vault. The principle: the file stays simple; the views add richness.
