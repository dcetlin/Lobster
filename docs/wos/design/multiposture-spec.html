WOS Multi-Posture Execution Architecture
status: DRAFT date: 2026-05-19 supersedes: issue #1216 (visibility-timeout) converges: design discourse between Dan and Lobster
§1 Architecture Overview

The WOS pipeline currently has one execution lane: executor-heartbeat dispatches all UoWs via batch polling, rate-limited and autonomous, running every 3 minutes via cron. This spec adds two more lanes, creating a three-posture system where each posture addresses a distinct execution need without replacing the others.

Posture 1
Batch Dispatch

executor-heartbeat polls the registry every 3 minutes and dispatches ready-for-executor UoWs via inbox message. Rate-limited, autonomous, works even when the dispatcher session is down.

keep — no change
Posture 2
Dispatcher-Direct

Dispatcher spawns a specific UoW as a background Agent via the CC Agent tool, on user command or priority trigger. Full Claude Code session with sub-agents, tools, file access. Priority lane for things that need the full stack immediately.

new — this spec
Posture 3
Self-Initiated Upward

UoW agents surface to Dan when stuck, via outcome=owner_decision_required and the awaiting-owner lifecycle state. The bottom-up channel that prevents silent stalls.

merged — PR #1232
§1.1 Root Cause: Why Compaction Kills UoWs
Root Cause

UoW agents do not reliably write heartbeats to the registry while executing. executor-heartbeat's heartbeat sidecar (Phase 1b) compensates by writing heartbeat_at for every active/executing UoW every 3 minutes, which prevents the observation loop from detecting stalls. The result is a false sense of liveness: the pipeline believes agents are running when they are dead.

When context compaction kills a dispatcher session mid-execution, active UoWs accumulate in executing state with no live agent and no heartbeat gap, blocking new dispatch through the context-pressure threshold and the TTL-recovery window.

The incident on 2026-05-18/19 (referenced in issue #1216) demonstrated this failure mode: 5 executing UoWs with no live subagents occupied all dispatch slots for 22–24h, blocking 168 ready-for-steward UoWs. The fix at the time was to raise the context-pressure threshold from 5 to 20 (PR #1215). That is a guard rail, not a solution.

The structural fix is to make heartbeat writes agent-originated rather than sidecar-compensated. When agents write their own heartbeats, the sidecar can be demoted from a liveness mask to a backup, and the observation loop can detect genuine stalls in minutes rather than hours.

§1.2 Three-Change Structural Plan

Three targeted changes implement this architecture. They are independent and can be shipped separately, in the order listed below. No change removes an existing capability; each adds precision where bluntness currently exists.

ChangeWhat it doesPosture affectedPriority
Change 1 UoW agents write their own heartbeats. The sidecar becomes a backup, not the primary liveness signal. Orphan detection becomes reliable. Posture 1 Critical — fixes the root cause
Change 2 New dispatcher handler: /dispatch <uow_id> spawns a specific UoW as a background Agent immediately, bypassing the 3-minute poll cycle. Posture 2 High — enables priority lane
Change 3 Disposition of claimed_until / visibility-timeout (issue #1216) in light of the heartbeat fix. Heartbeats make TTL-recovery mostly redundant; this change clarifies what remains. Posture 1 Medium — cleanup after Change 1
§1.3 Coexistence Model

The three postures share the same registry and use the same status lifecycle. They do not conflict:

Posture 1 (Batch) Posture 2 (Direct) Posture 3 (Upward) ──────────────────── ──────────────────── ──────────────────── executor-heartbeat dispatcher receives agent writes outcome= polls registry every /dispatch <uow_id> owner_decision_required 3 min, claims and dispatches via dispatcher calls steward transitions UoW wos_execute inbox msg Agent tool directly to awaiting-owner agent writes heartbeat agent writes heartbeat Dan replies via Telegram every 60-90s every 60-90s dispatcher calls agent writes result agent writes result handle_owner_decide() to output_ref to output_ref UoW → ready-for-steward steward reads result dispatcher receives on next cycle write_result callback

Postures 1 and 2 converge on the same agent execution path: same prompt template, same heartbeat contract, same result file schema. The dispatcher-direct path (Posture 2) is simply a faster on-ramp that bypasses the 3-minute polling cycle and writes a Claude Code session directly rather than routing through an inbox message.

§2 Change 1: Agent-Side Heartbeat Writes

This change makes the heartbeat contract in the UoW agent prompt template mandatory and structurally enforced, and revises the sidecar's role from primary liveness writer to backup gap-filler. The observation loop then distinguishes live agents from truly dead ones based on whether the agent itself has written a heartbeat since dispatch.

§2.1 The Field: heartbeat_at in uow_registry

The heartbeat_at column already exists in uow_registry. No schema migration is required for this change. The field is an ISO-8601 UTC timestamp, updated by registry.write_heartbeat(uow_id, token_usage). The companion uow_heartbeat_log table (migration 0017) records per-heartbeat token snapshots for cost telemetry.

FieldTypeCurrent semantics
heartbeat_atTIMESTAMP NULLLast write by sidecar or agent; NULL until first write after dispatch
heartbeat_ttlINTEGERMax silence (seconds) before stall detection fires; default 300

After Change 1: heartbeat_at written by the agent itself is the primary liveness signal. The sidecar continues to write heartbeats as a backup but its writes no longer mask dead agents.

§2.2 What Agents Must Write, and When

The existing handle_wos_execute prompt template in dispatcher_handlers.py already documents a heartbeat contract. Change 1 upgrades this from advisory to structural: the observation loop will not tolerate an agent that fails to write its first heartbeat within 5 minutes of dispatch.

Required checkpoints (minimum):

CheckpointWhen to writeWhy
startupImmediately after receiving the prompt, before any other workProves the agent started. No startup heartbeat = agent never ran
post-readAfter reading the GitHub issue and understanding the taskDistinguishes "started but stuck reading" from "started"
post-implAfter completing the implementation, before PR creationConfirms work completed; last heartbeat before high-latency gh operations
pre-resultImmediately before writing the result fileProves the agent is about to commit its outcome

Interval: At most 90 seconds between heartbeat writes throughout execution. If no natural checkpoint falls within 90 seconds, write one unconditionally.

Preferred form (MCP tool — no Python imports)
mcp__lobster-inbox__write_wos_heartbeat( uow_id='<uow_id>', token_usage=<cumulative_input_plus_output_tokens> )
Fallback form (Bash via registry)
import sys; sys.path.insert(0, '/home/lobster/lobster') from src.orchestration.registry import Registry Registry().write_heartbeat('<uow_id>', token_usage=<cumulative_tokens>)

Return value semantics: write_heartbeat() returns rowcount 1 on success, 0 if the UoW status has changed (i.e. the steward has already recovered or re-queued it). An agent that receives rowcount 0 must stop execution immediately and call write_result with outcome=failed. This prevents zombie agents from writing stale results.

§2.3 How executor-heartbeat Uses Heartbeats After Change 1

The sidecar (write_heartbeats_for_active_uows in heartbeat_sidecar.py) continues to run but its classification role changes. The observation loop in steward-heartbeat needs to distinguish three states:

Agent stateheartbeat_at signatureCorrect response
Alive and writing heartbeat_at updated within last 90s, last write was agent-originated (token_usage present in heartbeat_log) Leave it alone
Dead — sidecar masking heartbeat_at fresh but no agent-originated heartbeats in heartbeat_log since dispatch (token_usage always NULL) Declare orphan, kill, re-queue
Dead — sidecar absent or failed heartbeat_at older than heartbeat_ttl + buffer Declare orphan, kill, re-queue (existing path)

The critical detection path is the second row: an agent that never wrote a heartbeat (all NULL token_usage in heartbeat_log) with a recently-dispatched UoW is dead. The sidecar's writes are the only signal in heartbeat_at and they are masking a corpse.

Implementation approach: The observation loop (steward-heartbeat, Phase 2b) adds a sub-check: for any UoW in executing status whose heartbeat_at appears fresh, query uow_heartbeat_log for any rows written after dispatch_ts + 300s. If none exist and the UoW is more than 5 minutes old, classify as orphan_kill_before_start regardless of the value of heartbeat_at.

Detection query (conceptual)
-- UoWs that are executing but have never had an agent-originated heartbeat -- (sidecar writes heartbeat_at but not uow_heartbeat_log rows) SELECT r.id FROM uow_registry r WHERE r.status IN ('active', 'executing') AND r.updated_at < datetime('now', '-5 minutes') AND NOT EXISTS ( SELECT 1 FROM uow_heartbeat_log l WHERE l.uow_id = r.id AND l.recorded_at > datetime(r.updated_at, '+300 seconds') )

Once Change 1 is fully deployed and all active UoW agents are writing heartbeats, the sidecar can be downgraded to a pure backup: it writes heartbeats only for UoWs that have not received an agent-originated heartbeat within the last 90 seconds, rather than writing for all active UoWs unconditionally. This change to the sidecar is a follow-on after Change 1 is validated.

§3 Change 2: Dispatcher-Direct Spawn Path

Posture 2 adds a new dispatcher command — /dispatch <uow_id> — that spawns a specific UoW as a background Claude Code Agent immediately, without waiting for the next executor-heartbeat poll cycle. This is the priority lane: for UoWs where the user wants immediate execution with the full tool stack available.

§3.1 New Handler: handle_wos_dispatch

A new function handle_wos_dispatch(uow_id, registry) is added to src/orchestration/dispatcher_handlers.py. The dispatcher calls this when it receives a message matching /dispatch <uow_id>.

The function is pure: it validates the UoW state and returns a prompt string. The dispatcher owns the actual Agent tool call, the session registration, and the status transition.

Handler signature (dispatcher_handlers.py)
def handle_wos_dispatch(uow_id: str, *, registry: "Registry") -> str | None: """ Build the Task prompt for a dispatcher-direct UoW dispatch. §3.2 Prompt Template for Dispatcher-Direct Agents

The dispatcher-direct agent prompt reuses handle_wos_execute's output as its core, but adds Posture 2 metadata and changes the chat_id to the user's actual chat ID so that write_result notifies Dan directly upon completion.

Prompt template (built by handle_wos_dispatch)
--- task_id: wos-{uow_id} uow_id: {uow_id} chat_id: {admin_chat_id} # real chat_id — dispatcher-direct agents notify Dan source: telegram posture: direct # marks this as Posture 2 for audit logging --- You are executing a Work Order System (WOS) unit of work via the DISPATCHER-DIRECT path (Posture 2). You were spawned by the dispatcher in response to a /dispatch command — not by the batch executor. All contracts below are identical to the standard wos_execute path. The only difference is that write_result will notify Dan directly upon completion (chat_id is set to the operator's Telegram ID). {...standard heartbeat contract from handle_wos_execute...} {...PR-close guard from handle_wos_execute...} ## Instructions {instructions} ## Result contract {...standard result contract from handle_wos_execute...}

The task_id is the same wos-{uow_id} slug used by the Posture 1 path. The posture: direct YAML field is metadata for audit logs — it does not change agent behavior.

When to use Posture 2: A UoW that is high-priority and cannot wait for the next executor-heartbeat cycle; a UoW that requires direct feedback to Dan on completion; a UoW that was stuck in ready-for-steward and the operator wants to skip steward re-diagnosis; a debugging run where the operator wants to observe execution in real time.

When not to use Posture 2: Normal steady-state throughput — that is Posture 1's job. Posture 2 should not become the default execution path. It adds a CC session that counts against the context-pressure threshold, bypasses the scaling governor, and routes completion notifications to Dan's Telegram directly.

§3.4 How the Dispatcher-Direct Agent Writes Its Result Back

The agent writes the same result file at output_ref and calls write_result in exactly the same way as a Posture 1 agent — with one difference: chat_id is the operator's real Telegram ID, so the dispatcher receives a subagent_notification message and relays the outcome to Dan.

The steward's next heartbeat cycle reads the result file and transitions the UoW from executing to ready-for-steward as normal. No special Posture 2 path is needed in steward-heartbeat. The posture distinction lives entirely in the dispatch side, not the result-processing side.

Posture 2 execution flow: Dan: /dispatch wos-abc123 │ Dispatcher: handle_wos_dispatch('abc123', registry) │ → validates UoW status (ready-for-executor or ready-for-steward) │ → claims UoW (status → executing) │ → builds prompt with chat_id = admin_chat_id │ Dispatcher: Agent(prompt=...) → background CC session spawned │ Agent: write_heartbeat (startup) ← first signal that agent is alive Agent: reads issue, implements, writes heartbeat at each checkpoint Agent: writes output_ref (result JSON) Agent: write_result(task_id="wos-abc123", chat_id=admin_chat_id, ...) │ Dispatcher: receives subagent_notification → send_reply to Dan Dan: sees "WOS UoW abc123: outcome=complete" │ Steward (next 3-min cycle): reads output_ref → transitions UoW to done
§4 Change 3: claimed_until / Visibility-Timeout Disposition

Issue #1216 proposed replacing the heartbeat model with a visibility-timeout (claimed_until) model: each UoW dispatch sets a timestamp N minutes in the future, and the registry automatically makes the UoW available again when that timestamp expires. This is the Amazon SQS visibility-timeout pattern.

The heartbeat fix (Change 1) resolves the underlying problem that motivated issue #1216. This section documents the disposition: what to keep, what to deprecate, and why.

§4.1 Current State of claimed_until

claimed_until already exists in uow_registry and is already in use. The executor sets it to dispatch_ts + 2 × estimated_runtime on dispatch (at minimum 3600s), and reset_expired_claims() in registry.py resets expired claims back to ready-for-executor. This is the Phase 1 stall recovery path in executor-heartbeat.

§4.2 Recommended Disposition: Make Optional
Recommended: Make Optional (keep as backstop)

Keep claimed_until in the schema and keep reset_expired_claims() as a fallback recovery path. But let the heartbeat mechanism be the primary detection path. When agents write their own heartbeats (Change 1), the observation loop detects dead agents in 90–300 seconds rather than waiting for claimed_until to expire (potentially hours). In steady state, claimed_until expiry becomes a rare event — only when both the agent fails AND the observation loop misses the stall.

Alternative: Deprecate and Remove

Remove claimed_until entirely once Change 1 is validated in production. Simpler schema, fewer recovery paths. Higher risk during the transition period if heartbeat compliance is imperfect. Not recommended until Change 1 has run for at least 2 weeks with no orphan incidents.

The "make optional" path means: after Change 1 ships, claimed_until is still set on dispatch and reset_expired_claims() still runs in executor-heartbeat Phase 1. But the TTL can be shortened significantly (e.g. from 2 × estimated_runtime to 30 minutes) because the observation loop is now the primary detection mechanism. A UoW that loses its agent will be detected via heartbeat gap within 5–10 minutes; the claimed_until expiry is a defense-in-depth fallback.

§4.3 Issue #1216 Disposition

Issue #1216 proposed adding claimed_until as a new field and building the entire recovery model around it. The field already exists and is already wired into the executor. The multi-posture architecture supersedes the issue's proposed design: rather than replacing the heartbeat model with visibility-timeout, we fix the heartbeat model and keep visibility-timeout as a backstop.

Recommended action: close issue #1216 as "superseded by multiposture-spec" and reference this document. The schema change proposed in #1216 (adding claimed_until) has already been implemented; the behavioral changes (making it the primary recovery mechanism) are deferred in favor of the heartbeat fix.

§5 Migration Path

The three changes are independent and deployable in sequence. No single deployment is a big bang. Each change is observable via the existing logging and audit infrastructure.

§5.1 Deployment Sequence
1

In handle_wos_execute() (dispatcher_handlers.py), strengthen the heartbeat section: (a) Add an explicit "startup heartbeat" instruction as the first action item before any other work. (b) Specify the maximum interval (90 seconds) as a hard requirement, not a suggestion. (c) Add: "if write_heartbeat returns 0 (rowcount=0), stop immediately — the steward has already recovered this UoW." No registry schema changes required. Close issue #1216 as superseded.

§6 Non-Goals

This spec explicitly does not do the following. These items are excluded by design, not oversight. If any of them are desirable in the future, they require a separate spec.

  • Replace executor-heartbeat. Posture 1 (batch dispatch) remains the steady-state throughput lane. Posture 2 is a second lane, not a replacement. executor-heartbeat.py is not modified by this spec.
  • Make the dispatcher an orchestrator-of-orchestrators. The dispatcher spawns one UoW at a time in Posture 2. It does not manage a pool of concurrent UoW sessions, does not re-dispatch on failure, and does not monitor running UoW agents. Those responsibilities stay with steward-heartbeat and executor-heartbeat.
  • Implement the full visibility-timeout model proposed in issue #1216. That design replaces heartbeats with a TTL-expiry model. This spec retains heartbeats as the primary mechanism and uses claimed_until as a backstop.
  • Change the UoW status lifecycle. No new statuses are added. The existing lifecycle (proposed → pending → ready-for-steward → ready-for-executor → executing → done/failed) is unchanged.
  • Add streaming or progress updates from UoW agents. Agents remain fire-and-forget. Heartbeats prove liveness but are not surfaced to Dan in real time during Posture 1 execution. (Posture 2 does notify Dan on completion; it does not stream intermediate state.)
  • Change how the steward prescribes work. The steward's prescription logic, retry arc, and hard-cap enforcement are out of scope. The steward continues to write wos_execute inbox messages for Posture 1 exactly as today.
  • Modify executor-heartbeat.py in any way. The only changes are to: dispatcher_handlers.py (Change 2), steward-heartbeat.py Phase 2b (Change 1b), heartbeat_sidecar.py (Change 1c, deferred), and the UoW agent prompt template in handle_wos_execute (Change 1a).
  • Build a UI or dashboard for multi-posture monitoring. Observability remains in the existing logs, audit table, and wos-health-check job.
