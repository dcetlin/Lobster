# WOS Sprint 3 Part 2 — High-Leverage Prerequisites

*April 2026*

---

Sprint 3 Part 1 (S3-A through S3-G) was scoped to install the governing structures that make the WOS pipeline self-regulating: the commitment gate cleanup arc, the corrective trace temporal gate, the early-warning defensive form, the asymmetric governor, the oracle-docs protocol, the cultivator restoration, and the pre-existing test failure resolution. During Part 1 scoping, eight additional issues were identified that were not blocking enough to delay Part 1 but are high-leverage enough to affect the durability of every Part 1 delivery. This document captures those eight issues as Sprint 3 Part 2.

The framing distinction is important. Part 2 is not a second sprint — it is a set of structural prerequisites that, if left unaddressed, will degrade Part 1's governing structures during the first real operation. Part 1 installs the correct behavioral commitments. Part 2 removes the structural brittleness that would cause Part 1's logic to operate on bad inputs, ambiguous authority relationships, or unobservable failure modes. Some Part 2 items are active correctness hazards (S3P2-A, S3P2-B, S3P2-C). Some are confirmed-prior-incident flood risks (S3P2-D). Some are structural debt that, left unaddressed, will silently accumulate until a sprint hits the boundary it creates (S3P2-E, S3P2-F, S3P2-G, S3P2-H).

The ordering below is by blast radius: how many steward prescription cycles, executor dispatches, or governing-structure outputs are affected if the issue is not resolved.

---

## Core Premises

**P1: The steward's typed migration is a precondition for reliable governing structure operation.**
Every Part 1 UoW that touches steward.py operates on raw `dict[str, Any]` rows — no StrEnum for status values, no frozen result types, no match/case dispatch, no typed function signatures. This is not a style preference: it means the type checker cannot verify exhaustiveness of state transitions, silent key-name drift is undetectable, and status comparisons can misfire on string literal variations. The commitment gate cleanup arc (S3-A), the temporal gate enforcement (S3-B), and the early-warning threshold (S3-C) all add new branches to the steward's dispatch logic. Those branches are guaranteed to be fragile if they are added to an untyped dispatch chain.

**P2: The prescription JSON-in-JSON format is an active correctness hazard for every prescription cycle.**
Sprint 001's execution track observed 8/9 prescription failures attributable to the LLM emitting preamble text before the JSON block, causing `JSONDecodeError`. The in-flight parser fix (PR #673) treated the symptom. Issue #613 identifies the root cause: the format requires the LLM to write prose encoded as a JSON string value — a structural guarantee-violation for any LLM that naturally emits preamble. Every prescription cycle in Part 1 runs against this failure mode. The format must change before Part 1 PRs that install governing structures can be tested against clean prescription data.

**P3: The startup-sweep minimum-age gap is an active duplicate-execution hazard.**
The startup sweep correctly applies a minimum-age threshold to Population 2 (`ready-for-executor` UoWs). It does not apply one to Population 1 (`active` UoWs). Any executor subprocess taking longer than the heartbeat interval (3 minutes) is classified `crashed_output_ref_missing` and transitions to `ready-for-steward` while still running. The pipeline then re-prescribes and launches a second subprocess. This was flagged in an earlier learnings entry. Part 1's governing structures cannot produce trustworthy audit data if the substrate can generate duplicate executions.

**P4: Structural debt in shared vocabulary, interface contracts, and observability gaps compounds with each Part 1 PR.**
Issues #571/#567 (posture vocabulary), #509 (vision_ref routing), and #449/#450 (GardenCaretaker interface drift) each affect a narrow surface today. But each Part 1 PR that extends the steward or the caretaker expands that surface. The cost of reconciling these issues increases monotonically with each sprint that defers them.

## Sprint 3 Part 2 UoWs

### S3P2-A: Steward Typed Migration — UoW Dataclass Adoption
**Issue:** https://github.com/dcetlin/Lobster/issues/423
**Why this sprint:** Every Part 1 UoW that modifies steward.py adds new dispatch branches to a codebase operating on `dict[str, Any]` throughout. PR #335 introduced the `UoW` dataclass and the dict bridge with the explicit intent that all engine code would use typed instances, with dict conversion only at persistence boundaries. PR #336 (steward engine) did not adopt it. The typing audit of PR #336 found 5 FAILs: no StrEnum for status/decision values, no frozen result types, no match/case dispatch, no named decision unions, and pervasive `dict[str, Any]` signatures throughout. Part 1's commitment gate (S3-A) adds a new cleanup arc branch. Part 1's temporal gate (S3-B) adds a `WaitForTrace` outcome type. Both of these new constructs will be added to untyped dispatch chains unless the migration happens first. The type checker cannot verify that `match uow.status` is exhaustive if `uow` is a `dict`. The oracle will flag this in every Part 1 steward PR.
**Success criterion:** All `dict[str, Any]` parameters and return types in `steward.py` are replaced with `UoW` (or typed sub-structures where appropriate). A `StrEnum` exists for steward decision/status values. `@dataclass(frozen=True)` result types are defined for steward outcomes. All `if/elif` status dispatch chains are replaced with `match`/`case` on `UoWStatus` or the new decision enum. Registry rows are deserialized to `UoW` at the steward's entry point; re-serialized only at write-back. No behavior change — every existing hard cap, escalation path, and cycle counter is preserved exactly. PR passes oracle review with the type checker showing no `dict[str, Any]` in steward function signatures.
**Dependencies:** PR #335 (`UoW` dataclass and dict bridge) — confirmed merged. No Part 1 UoW may dispatch to subagents touching `steward.py` until this PR merges, or its oracle will find the same typing failures.
**Risk:** The migration is a focused refactor with no behavior change required. The primary risk is a missed code path where a raw dict field is referenced by string key rather than typed attribute. A side-by-side read of the pre- and post-migration steward is sufficient to verify completeness. If any Part 1 PR is already in-flight against the untyped steward when this merges, a rebase conflict is likely — coordinate before dispatch.

### S3P2-B: Prescription Format — Split Routing Envelope from Prompt Body
**Issue:** https://github.com/dcetlin/Lobster/issues/613
**Why this sprint:** Sprint 001's execution track observed 8 out of 9 prescription failures due to `JSONDecodeError`. The root cause is that `_llm_prescribe` asks the LLM to produce a pure JSON document containing both machine-routing fields and a prose executor prompt encoded as a JSON string value. LLMs naturally emit preamble before the JSON block. The defensive parser fix (PR #673) reduces the failure rate but does not close the structural guarantee-violation: the format asks for something LLMs cannot reliably produce. Every Part 1 prescription cycle runs against this defect. Part 1's governing structures (commitment gate cleanup arc, temporal gate) depend on prescription-cycle data to evaluate their thresholds. Prescription failures during Part 1 testing will produce corrupt or absent trace records that trigger false temporal-gate firings and mislead the commitment gate. The format must change before Part 1 can be tested cleanly.

The redesign proposed in #613 splits the prescription into a hybrid front-matter + prose format: a small JSON routing envelope (executor_type, prescribed_skills, posture, estimated_runtime) followed by a `---` separator and free-form markdown as the executor prompt. A PostToolUse hook on Write for workflow artifact files validates the front-matter before the write commits. If `executor_type` is missing or invalid, the hook blocks the write and returns a specific error message — the LLM corrects and retries. The prose body is unrestricted markdown; no JSON encoding is required.
**Success criterion:** `_llm_prescribe` is refactored to use the Write tool plus file read instead of subprocess stdout capture. A `_parse_workflow_artifact(path)` function reads front-matter and body. A PostToolUse hook at `hooks/validate-workflow-artifact.py` validates the routing envelope before writes commit, with specific rejection messages for missing/invalid fields. The hook is registered in Claude Code settings as a PostToolUse hook (this registration step was identified as missing from the original #613 specification and must be included). Test coverage for front-matter parsing edge cases (colons in values, missing fields, extra whitespace) and hook validation is present. Zero `JSONDecodeError` in prescription logs for any new prescription after this PR merges. PR passes oracle review.
**Dependencies:** PR #673 (defensive colon-split parser fix) — confirmed merged. S3P2-A (steward typed migration) should merge before this PR if feasible: the `_llm_prescribe` refactor touches the steward dispatch path where typed signatures will conflict with in-progress `dict[str, Any]` signatures.
**Risk:** The PostToolUse hook registration is the riskiest step. If the hook is written but not registered in settings, the validator exists on disk but never fires — the prescription format appears to work in testing but is unguarded in production. The registration step must be verified by explicitly triggering a prescription write and confirming the hook fires. Additionally: the `posture` field in the routing envelope must be reconciled with the posture vocabulary defined in S3P2-F — the front-matter schema must not introduce a third independent posture vocabulary.

### S3P2-C: Startup Sweep Minimum Age for Active UoWs
**Issue:** https://github.com/dcetlin/Lobster/issues/572
**Why this sprint:** The startup sweep's Population 1 classification (`active` UoWs) has no minimum age threshold before classifying a UoW as `crashed_output_ref_missing`. `_EXECUTOR_ORPHAN_THRESHOLD_SECONDS` is correctly applied to Population 2 (`ready-for-executor`) but absent from Population 1. The consequence is a timing race: any executor subprocess that takes longer than the heartbeat interval (3 minutes) is classified as crashed and transitioned back to `ready-for-steward` while still running. The pipeline then re-prescribes and launches a second subprocess against the same UoW. This race was identified in an earlier learnings entry and confirmed as a live hazard pattern by PRs #667 and #668, which fixed the analogous Population 2 gap. Part 1's temporal gate (S3-B) makes this hazard worse: if a duplicate execution is launched, two `trace.json` writes race for the same output path, and the temporal gate may observe a corrupt or partial trace.
**Success criterion:** Population 1 classification in `startup_sweep.py` applies a minimum age threshold before classifying `active` UoWs as `crashed_output_ref_missing`. The threshold uses `started_at + estimated_runtime < now` (per-UoW accuracy) rather than a global constant. If `estimated_runtime` is null, a fallback global constant is used. The `_EXECUTOR_ORPHAN_THRESHOLD_SECONDS` pattern from Population 2 is the implementation reference. A test verifying that an `active` UoW younger than its estimated runtime is not reclassified — and one verifying that an `active` UoW older than its estimated runtime is reclassified — is present. PR passes oracle review.
**Dependencies:** None. This is a targeted fix to `startup_sweep.py` with no cross-file dependencies. It may be dispatched independently of other Part 2 UoWs.
**Risk:** Low. The fix mirrors an already-correct pattern applied to Population 2. The only non-trivial edge case is handling null `estimated_runtime` — the fallback constant must be documented explicitly. If the fallback constant is set too low, long-running legitimate executors will still be reclassified; too high, and crashed executors will be missed. The existing Population 2 threshold value is the starting reference for the fallback.

### S3P2-D: Reconciler Startup Flood — One Summary Per Restart
**Issue:** https://github.com/dcetlin/Lobster/issues/469 (root-cause fix); https://github.com/dcetlin/Lobster/issues/464 (batch-limit defense-in-depth)
**Why this sprint:** The reconciler's startup sweep produces one inbox notification per completed task. A session ending with N in-flight tasks generates N messages on the next restart — the flood is structural, not incidental. This was a confirmed prior incident: the system generated a 663+ message flood (issue #459). Issue #464 applied a batch-size cap as a symptom treatment. Issue #469 identifies the root-cause fix: the startup sweep should collect all unnotified sessions, group by chat_id, and produce one summary per user. Internal agents (chat_id=0) should be excluded entirely, routing to `task-outputs/` per the already-established convention. Once the root-cause fix lands, issue #464's batch cap is demoted to defense-in-depth, not the primary gate. Part 1's cultivator restoration (S3-F) and asymmetric governor (S3-D) both produce task outputs that will be routed through this reconciler path. Without the flood fix, a restart following a busy Part 1 session could generate dozens of redundant reconciler messages, drowning the actual governing-structure outputs.
**Success criterion:** The reconciler startup sweep collects all unnotified sessions, groups by chat_id, and produces one summary message per user per restart with the format: "N tasks completed while compacted: [list of task IDs and brief descriptions]." chat_id=0 sessions are excluded from inbox notification and routed to `task-outputs/` only. The batch-size cap from #464 remains as defense-in-depth after this fix, not as the primary flood gate. A test verifying that N completed tasks with the same chat_id produce exactly one summary message (not N messages) is present. PR passes oracle review.
**Dependencies:** None for the root-cause fix. The batch-cap from #464 should remain registered in the codebase as defense-in-depth — this PR should document the relationship explicitly rather than removing #464's fix.
**Risk:** The grouping logic must handle the case where tasks completed across multiple sessions have different `completed_at` timestamps but the same chat_id. The summary must not lose information — each completed task ID and description must appear in the summary even when collapsed. If the grouping silently drops tasks from the summary, the fix trades a flood failure mode for a silent-omission failure mode.

### S3P2-E: Dispatch Boundary Observability — Executor Inbox Coupling Failure Mode
**Issue:** https://github.com/dcetlin/Lobster/issues/589
**Why this sprint:** PR #584 (merged during Part 1 planning) refactored executor dispatch from subprocess to dispatcher inbox pattern. This was the correct architectural direction — it eliminates the polling hop and the executor's direct subprocess management. But it introduces a new failure mode: the executor now depends on dispatcher inbox health. The only current guard against inbox dispatch failure is a reactive 4-hour TTL recovery. There is no proactive observability at the dispatch boundary, no logging of dispatch attempts or retries, and no documented fallback if the inbox becomes unavailable. Part 1's governing structures are designed to make the commitment gate irreversible and the temporal gate enforced — both require that dispatch failures are visible and recoverable within a known time bound. A silent inbox failure that goes undetected for 4 hours defeats the temporal gate entirely: the temporal gate's one-heartbeat dwell becomes a four-hour dwell with no record of why. The governor gate (S3-D) cannot distinguish a genuine starvation signal from a starvation-due-to-inbox-failure without dispatch observability.
**Success criterion:** All executor dispatch attempts, retries, and failures at the dispatcher→executor boundary are logged with a structured record: `{uow_id, dispatch_attempt, timestamp, outcome: success|retry|failure, failure_reason?}`. *(Implementation note: the `retry` outcome and `dispatch_attempt>1` were not implemented — `outcome` is `success` or `failure` only; `retry` is reserved in the schema. See `docs/wos-dispatch-failure-modes.md`.)* The log is queryable from the audit log or a dedicated dispatch-boundary log file. A documented failure mode entry exists in `docs/architecture.md` (or equivalent) describing: what happens when the inbox is unavailable, what the recovery path is, and what the current recovery time bound is. If a fallback dispatch mechanism (file-based queue or secondary notification path) is implemented, it is gated behind a feature flag — the primary path remains inbox dispatch. PR passes oracle review.
**Dependencies:** PR #584 (inbox dispatch refactor) — the observability is a follow-on to this already-merged PR. No other Part 2 UoWs are dependencies.
**Risk:** The observability layer must not itself become a dispatch bottleneck. Log writes at the dispatch boundary must be non-blocking and must not gate the dispatch outcome. If the log write fails, the dispatch should proceed and the write failure should be noted separately. The architecture.md documentation task is the higher-risk item — incomplete documentation of the failure mode is worse than no documentation (it creates false confidence).

### S3P2-F: Posture Vocabulary Reconciliation
**Issue:** https://github.com/dcetlin/Lobster/issues/571 (implementation vocabulary); https://github.com/dcetlin/Lobster/issues/567 (design vocabulary)
**Why this sprint:** Two independent posture classification systems coexist without a rule for which is authoritative. `wos-v2-design.md` introduces cycle-trace postures (`orienting`, `clarifying`, `waiting_for_signal`, `scope_challenged`, `closing_with_discovery`). `steward.py` uses `reentry_posture` values (`solo`, `verify`, `explore`, `pending_evaluation`, `first_execution`, `execution_complete`). These are not the same concept — they potentially govern different layers (execution routing vs. trace annotation). But there is no documentation stating which layer each vocabulary governs, which is authoritative for dispatch decisions, and whether they can or should be unified. S3P2-B's prescription format redesign introduces a third touch point: the routing envelope includes a `posture` field with values from a third list (`solo`, `verify`, `explore`, `pivot`, `escalate`). If this tension is not resolved before Part 1 and Part 2 implementation work begins, future Steward PRs will extend one vocabulary in isolation, compounding the disambiguation problem each sprint. The learnings entry from 2026-04-01 flagged this explicitly: parallel posture vocabularies without a disambiguation layer create classification authority ambiguity.
**Success criterion:** A design decision is documented in a GitHub comment on #571 (or a new ADR-style document referenced from #571) that: (1) names which vocabulary governs dispatch decisions — what posture value causes a UoW to be routed differently; (2) names which vocabulary governs trace annotation — what posture value appears in `trace.json` and `cycle_trace`; (3) states whether the two are unified (one vocabulary for both purposes with different fields), layered (each vocabulary owns its layer with no overlap), or deprecated (one vocabulary is authoritative, the other is removed). The `reentry_posture` values in `steward.py` are updated to match the authoritative vocabulary. The `posture` field in S3P2-B's prescription front-matter schema references the authoritative dispatch vocabulary. PR passes oracle review.
**Dependencies:** S3P2-B (prescription front-matter schema) must not finalize the `posture` field values until this reconciliation is resolved — coordinate to avoid introducing a third independent vocabulary through the front-matter schema. S3P2-A (steward typed migration) should adopt the authoritative StrEnum once this reconciliation is complete.
**Risk:** The design decision may require reading more context than is visible in the two open issues — PR #564 (agenda-trace postures) and the oracle decision from 2026-04-01 are the source documents. If the reconciliation cannot reach a verdict without Dan's explicit input on the design intent (trace annotation vs. dispatch routing authority), the UoW should surface the unresolved question rather than picking one system arbitrarily.

### S3P2-G: Wire vision_ref to Classifier Routing Rules
**Issue:** https://github.com/dcetlin/Lobster/issues/509
**Why this sprint:** `vision_ref` is a TEXT JSON field in the WOS Registry schema (`{layer, field, statement, anchored_at}`). The issue-sweeper populates it when creating UoWs. The registry deserializes it when reading UoWs. No code reads it to influence any routing or dispatch decision. `route_reason` is set independently of `vision_ref` — the field is stored and dropped. `vision.yaml` explicitly names this as the next Phase 1 step: "connect vision layer to WOS classifier routing rules so route_reason can reference vision fields directly." The registry schema and population exist. The consumption pathway does not. Part 1's cultivator restoration (S3-F) and the governing structures it depends on are designed around a pipeline that routes UoWs by their vision-layer position. Without `vision_ref` feeding into routing, the pipeline routes by heuristic rather than by stated vision alignment — this is the Phase 1 completion criterion sc-4: "Removing vision.yaml would cause structurally different routing outcomes."
**Success criterion:** A function in `registry.py` or the WOS classifier reads `vision_ref` from a UoW and includes the referenced vision field in routing decisions. `route_reason` references `vision_ref.field` when `vision_ref` is non-null. `vision_ref` content is surfaced in the morning briefing staleness check (Phase 1 success criterion sc-3). The success criterion sc-4 is met: disabling `vision_ref` population produces measurably different routing outcomes in at least one test case. PR passes oracle review.
**Dependencies:** `vision_ref` schema and population in issue-sweeper — confirmed present. No Part 2 UoWs are hard prerequisites. S3P2-A (steward typed migration) is a soft dependency: the routing function should operate on typed `UoW` instances, not raw dict rows, so the migration should land first.
**Risk:** The routing logic must not silently fall through to heuristic routing when `vision_ref` is null — the fallback must be explicit and logged. The more significant risk is that the `vision_ref` structure (`layer`, `field`, `statement`) may not map cleanly to the classifier's existing routing vocabulary. If a gap exists, the PR should document it as a follow-on rather than silently papering over it with a default.

### S3P2-H: GardenCaretaker Interface Drift
**Issue:** https://github.com/dcetlin/Lobster/issues/449 (source_ref prefix mismatch); https://github.com/dcetlin/Lobster/issues/450 (IssueSource Protocol interface divergence)
**Why this sprint:** Two related issues were filed during the oracle review of PRs #440 and #444. Issue #449: `registry.upsert()` hardcodes `github:issue/{issue_number}` as the `source_ref` prefix unconditionally, regardless of which `IssueSource` substrate seeded the UoW. For the current single-substrate setup, this works. When multi-source support is added, UoWs seeded by a non-GitHub source will carry `github:`-prefixed source_refs, and any substrate-aware routing will silently misroute them. Issue #450: the `IssueSource` Protocol interface in `issue_source.py` implements `scan()` while the garden-caretaker design doc specifies `list_open_issues()` — with `source_name` absent, parameter names diverged, and return type changed from `Sequence` to `Iterator`. Today these differences are purely naming-level. But each Part 1 PR that extends the caretaker widens the gap between what the design says and what the code does, making future oracle reviewers flag the same drift repeatedly. Part 1's S3-A (commitment gate cleanup arc) and S3-B (temporal gate) both interact with the caretaker's in-flight detection. A caretaker whose interface drifts from its design doc is harder to reason about during oracle review of those PRs.

Issue #449's fix (pass `source_ref` from the `IssueSnapshot` directly rather than reconstructing from issue_number plus hardcoded prefix) is the higher-priority item — it prevents a schema migration requirement when multi-source is added. Issue #450's fix (update the design doc to match implemented interface names) is lower-risk but should be done in the same PR to close both tracking items cleanly.
**Success criterion:** `registry.upsert()` derives the `source_ref` from the `IssueSnapshot` directly, rather than constructing `github:issue/{issue_number}` independently. The `IssueSource` Protocol in `issue_source.py` is the authoritative interface definition. `assessments/garden-caretaker-design.md` is updated to match the implemented interface names (`scan()`, `source_ref`, `Iterator` return type). A note in the design doc documents why `scan()` was chosen over `list_open_issues()` (consumed once, not twice — Iterator is correct for GardenCaretaker's usage). PR passes oracle review with no remaining design doc / implementation mismatches on the caretaker interface.
**Dependencies:** None. This is a targeted fix to `registry.py` and a documentation update. It may be dispatched independently. S3P2-A (steward typed migration) has no dependency relationship with this UoW.
**Risk:** The `source_ref` field change requires verifying that all existing UoWs in the registry have consistent `source_ref` values — if the registry already contains `github:issue/42`-format records, the code change must not break reads of those existing records. The fix is forward-only (new upserts use the correct format; existing records are read correctly). A migration for existing records is out of scope for this UoW unless the oracle flags it as required.

## Ordering and Dependencies

The eight UoWs divide into three priority tiers based on blast radius and blocking relationships.

**Tier 1 — Active correctness hazards (must land before Part 1 execution begins):**
S3P2-A (steward typed migration), S3P2-B (prescription format), and S3P2-C (startup sweep minimum age) are active hazards. S3P2-A is a precondition for all Part 1 PRs that touch `steward.py`: until the steward operates on typed `UoW` instances, every oracle review of a Part 1 steward PR will flag the same typing failures. S3P2-B must land before Part 1 prescription cycles are tested — otherwise prescription failures will corrupt the trace data that Part 1's temporal gate and commitment gate depend on. S3P2-C must land before any Part 1 UoW is executed against the live system — otherwise long-running executors will be falsely classified as crashed, producing duplicate executions that generate corrupt trace records.

Within Tier 1, S3P2-A must merge before S3P2-B, because the `_llm_prescribe` refactor touches steward dispatch paths where typed and untyped signatures will conflict if both changes are in flight. S3P2-C is independent of both and may be dispatched in parallel with S3P2-A.

**Tier 2 — Confirmed incident risk and governor gap (should land before Part 1 operations begin):**
S3P2-D (reconciler flood) and S3P2-E (dispatch boundary observability) address failure modes with confirmed prior incidents or identified governor blindspots. S3P2-D's reconciler flood was a confirmed prior incident (663+ messages). Part 1's cultivator and governor produce task outputs through this reconciler path — a Part 1 restart with many in-flight tasks will reproduce the flood. S3P2-E's dispatch boundary observability gap means the asymmetric governor (S3-D, Part 1) cannot distinguish genuine starvation from starvation-due-to-dispatch-failure. Both are independent of each other and of Tier 1 items; they may be developed in parallel with Tier 1 once Tier 1 sequencing is set.

**Tier 2 — Confirmed incident risk and governor gap (should land before Part 1 operations begin):**
S3P2-G (vision_ref routing) is elevated to Tier 2 per Dan's decision. Vision_ref feeds into routing decisions that the cultivator and governor depend on — leaving the consumption pathway absent while Part 1 governing structures land makes the Phase 1 completion criterion sc-4 unverifiable during Part 1 testing. S3P2-G is independent of S3P2-D and S3P2-E and may be developed in parallel with them.

**Tier 3 — Structural debt accumulation (should land during Part 1, before any UoW extends the affected surfaces):**
S3P2-F (posture reconciliation) and S3P2-H (caretaker interface drift) affect surfaces that will be extended by Part 1 PRs. They are not active correctness hazards today, but each Part 1 PR that extends steward dispatch, routing decisions, or caretaker behavior without resolving these gaps makes future oracle review harder and increases the cost of eventual reconciliation. S3P2-F must be resolved before S3P2-B finalizes the `posture` field in the prescription front-matter schema — this is the only hard ordering constraint in Tier 3. S3P2-H is fully independent and may proceed in any order.

**Parallelism summary:**
The following pairs may be developed in parallel: (S3P2-A and S3P2-C), (S3P2-D, S3P2-E, and S3P2-G), (S3P2-H). S3P2-B must wait for S3P2-A to merge. S3P2-F must resolve before S3P2-B finalizes its posture schema. All other pairs are independent. S3P2-G is elevated to Tier 2 — treat it as a peer of S3P2-D and S3P2-E for sequencing purposes.

## Sprint 3 Part 2 COMPLETE Definition

Part 2 is complete when all four gates below are satisfied. Partial completion does not count. Each gate is independently verifiable.

**UoW gate.** All eight UoWs (S3P2-A through S3P2-H) have their corresponding PRs merged on main with an `oracle_status: approved` entry recorded in `oracle/decisions.md`. A UoW whose PR is open, blocked, or oracle-pending holds the gate. The merge-order constraints (S3P2-A before S3P2-B; S3P2-F resolved before S3P2-B finalizes posture schema) must be honored — the gate verifies merge order for those two constraints, not just merge count. Tier 1 UoWs must be merged before any Part 1 UoW dispatches an executor against the live system.

**Integration test gate.** The startup sweep minimum-age fix (S3P2-C) includes at least two integration-level tests: one verifying that an `active` UoW younger than its estimated runtime is not reclassified as `crashed_output_ref_missing`, and one verifying that an `active` UoW older than its estimated runtime is reclassified. These tests run against the real state machine transitions, not mocks of the classification function. The reconciler flood fix (S3P2-D) includes at least one integration-level test verifying that N completed tasks with the same chat_id produce exactly one summary message on restart. Both test sets must pass on main.

**Oracle gate.** This document carries `oracle_status: approved` in its frontmatter. Per the S3-E protocol (Part 1), the oracle review of this document must be recorded in `oracle/decisions.md` with the corresponding PR URL. An oracle review conducted but not machine-recorded does not satisfy this gate.

**Retro gate.** A brief Sprint 3 Part 2 retro section is appended to `docs/wos-sprint3-retro.md` (the Part 1 retro document) before Sprint 3 closes fully. The Part 2 retro section must answer: which Tier 1 items revealed unexpected complexity during implementation, and whether any Tier 3 items produced oracle findings that should be promoted to `oracle/learnings.md`. The Part 2 retro does not require a standalone document — it is a named section in the Part 1 retro. It is oracle-reviewed as part of that document's oracle review.
