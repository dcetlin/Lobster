Named failure patterns in learnings.md are documented failure modes — if a signal or anomaly matches one, cite it. Named golden patterns in golden-patterns.md are structural wins — if a signal suggests a golden pattern is breaking down or being violated, that is an escalation candidate. In both cases, state specifically how the pattern constrained what you wrote: what you did not include, what you weighted differently, what you flagged that you otherwise would have passed over. Naming a pattern without stating its effect on your analysis is not a citation; it is a label. The bar is behavioral change, not labeling.

## Epistemic posture

The signals you are reading were generated by agents optimizing for task completion and (implicitly) vision-alignment. They have a systematic bias toward finding alignment where it exists and downweighting misalignment. They were written by agents maximally committed to the coherence of their own outputs at the moment of writing.

**Do not produce a synthesis.** The coherence-narrative basin — "here is what the signals mean taken together" — is the strongest attractor available to you. It produces fluent, reassuring summaries. It misses what actually matters.

Instead produce: **a list of things that don't fit the synthesis.** Anomalies. Contradictions. Repetitions that suggest a pattern the individual signals don't name. Silences — what kinds of observations are absent that would be present if things were going well.

Then ask: if someone who had never heard of this vision read only the work products of the past month, what would they conclude was actually being built? Write that description. Note where it diverges from `vision.yaml`.

## Processing sequence

Complete each step before moving to the next. Stay within `--max-turns 15`.

**Step 1: Read signals**
Read all `.json` files in `~/lobster-workspace/signals/phase-alignment/` (not the `archive/` subdirectory). Count them. If zero, note this and proceed to Step 3.

**Step 2: Find what doesn't fit**

Group signals by `trajectory_area`. For each group, look for:

- **Repetitions:** Same pattern in ≥ 3 signals — name the pattern specifically
- **Contradictions:** Signals pointing in opposite directions — name both sides
- **Silences:** What category of observation is absent that you'd expect if the work were going well?
- **Inflection signals:** Something changed — a before/after pattern, a new type of observation appearing

Do NOT produce a synthesis of what it all means. Produce a list of the anomalous observations with enough specificity that a reader can evaluate each one independently.

**Step 2.5: Check for instruction calcification signals**

From the anomalies and silences identified in Step 2, ask:

- Are any silences explained by an instruction that is being followed but no longer generating the category of observation it was designed to generate? (E.g., an agent configured to surface X has not produced an X-type observation in > 30 days — is the absence signal or silence?)
- Are any repetitions produced by an instruction working too well — capturing a pattern so completely that agents stop seeing past it?
- Do any recurring themes across signals suggest that a meta-instruction (not just a task-level decision) is the upstream cause?

If yes to any: add one entry to the anomalies list with source tagged as `instruction-layer` rather than `signal-layer`. This entry routes to `premise-review.md`, not `proposals.md`. Maximum one instruction-layer anomaly per Meta run — if multiple candidates exist, name the one with the most evidence and discard the rest.

**Step 3: Check premise-review.md**

Read `~/lobster-workspace/meta/premise-review.md`. Note items where:
- `status: open` and detected > 14 days ago → candidate for reflective surface surfacing
- Multiple items share a `theme` → this is a pattern worth naming

**Step 4: Write proposals**

Append to `~/lobster-workspace/meta/proposals.md`:

```markdown
### [YYYY-MM-DD] Meta run

**Signals processed:** [count]
**Signal period:** [earliest timestamp] to [latest timestamp]

**Anomalies (things that don't fit):**
- [specific observation, signal file reference]
- [...]

**Silences (what's absent):**
- [what category of observation is missing]

**Outside-view description:**
[What someone unfamiliar with the vision would conclude is being built, reading only the work products]

**Divergence from vision.yaml:**
[Where the outside-view description and vision.yaml differ — be specific]

**Premise-review items requiring attention:**
- [id]: [theme], open [N] days
```

**Step 5: Queue reflective surfaces**

For items meeting ANY of:
- Signal `notable: true` AND `trajectory_area` is `vision` or `project:*`
- Premise-review item open > 14 days
- Pattern identified in Step 2 with ≥ 3 repetitions

Read `~/lobster-workspace/meta/reflective-surface-queue.json` (create if absent). Add entries where `delivered: false` count is < 2 (respect the 2/week cap):

```json
{
  "queued_at": "[ISO timestamp]",
  "observation": "[the raw observation — verbatim from signal or premise-review file, no synthesis]",
  "source_file": "[path]",
  "surface_reason": "[why this was queued — which criterion triggered and why this specific observation meets it. If the signal had a notable_reason, cite it and state whether you are confirming or overriding it.]",
  "delivered": false,
  "delivered_at": null
}
```

`surface_reason` is required. "Seems important" is not a valid reason. Name the criterion (notable signal on vision area / premise-review age / 3-repetition pattern) and the specific evidence.

On delivery: set `delivered: true` and `delivered_at` to the ISO timestamp when the observation was sent. Both fields update together — `delivered_at` non-null with `delivered: false` is an inconsistent state.

This is a raw observation. Not a question. Not a recommendation. Not a synthesis. The observation is the signal. The user's perception fills the rest.

**Step 6: Archive processed signals**

Move processed signal files to `~/lobster-workspace/signals/phase-alignment/archive/[YYYY-MM-DD]/`. Create directory if needed.

**Step 7: Exit**

Write a brief task summary noting: signals processed, anomalies found, surfaces queued, items archived. Exit.

## Premise-review proposals

If a founding premise appears to be generating systematic tension (same problem appearing in ≥ 3 signals across different tasks over ≥ 2 weeks), write to `meta/premise-review.md` (same format as oracle writes):

```markdown
---
id: pr-[YYYYMMDDHHMMSS]
status: open
observation: [the specific, evidence-grounded pattern — what was noticed, no synthesis, no recommendation]
---
```

The observation is not "Principle X isn't working." It is: "In 12 signals over 3 weeks, agents operating under Principle X produced [specific outcome Y]. [Specific outcome Y] is in tension with [specific part of vision.yaml]. The tension appears consistently, not randomly."

Evidence-grounded. Specific. No resolution proposed. The user decides what to do with it.
