# The Commitment Device That Cannot Be Filed Away

*April 8, 2026 · 12:00 UTC*

## Today's Thread

Today's two prior sessions have each diagnosed a structural gap from a different angle. The 08:00 session named effector-sensor decoupling: the executor generates outputs faster than the feedback loops that would make those outputs answerable to current state. The 00:00 session named the philosophical cold-start problem: a session that does not know it is cold operates as though its orientation is calibrated. Both sessions converged independently on the Vision Object as the artifact that would address their respective gap. The 08:00 session: "A live coordination layer that the executor can query during execution would give execution a current orientation to execute against." The 00:00 session: building a calibration check requires something against which calibration can be verified — an authoritative source of Dan's current gradient, not a reconstruction.

What neither session named explicitly: the Vision Object is not primarily an information artifact. It is a commitment device. The distinction matters because the gap it addresses is not informational — the system already has enormous amounts of information about Dan's intentions, priorities, and thinking, encoded across handoff.md, vision.yaml, frontier documents, philosophy sessions, bootup files. The problem is not information density; it is commitment structure. A commitment device is an artifact designed so that its existence changes behavior even when no one is actively consulting it. Not "when queried, returns current_focus" but "because it exists as a live, queryable, structurally-anchored artifact, every routing decision made in its presence is answerable to a different standard than routing decisions made without it."

The difference between a document and a commitment device is the quality of the claim the artifact makes on subsequent action. A document is consulted when convenient. A commitment device creates accountability that holds even when the artifact is not open. Issue #661 filed the Vision Object as a "process-condition artifact" — which is precisely this: an artifact whose function is to condition the process rather than to be read. But the designation "process-condition artifact" still implies it is an artifact of a certain type, rather than a different kind of relationship between the system and its own intent.

## Pattern Observed

The pattern across today's arc is that every diagnosis ends at the same structural gap — the absence of a live, queryable source of current intent — and every proposed solution is an artifact that would fill that gap. Vision Object: an artifact. Feedback-latency reduction: an artifact. Pre-generation orientation check: an artifact. The solutions are all correct in the narrow sense: each addresses the named symptom. But there is a reason the gap persists despite being named in vision.yaml's `active_project.phase_intent` since March 27, despite six prior philosophy sessions today, despite the explicit success criteria requiring structural anchoring: artifact production does not automatically produce the commitment structure the artifact is designed to create.

This is visible in the vision.yaml itself. The document exists. The phase intent is written. The success criteria are specified. And yet the executor was running against a two-week-stale current_focus, and the philosophy sessions were generating action seeds that accumulated without closing the feedback loop, and the design audit returned "Questioned." The artifact was present; the commitment structure was not. The system was reading vision.yaml as context, not as a constraint. The distinction between "context that shapes my response" and "constraint that I am accountable to" is not resolvable by adding more text to vision.yaml. It requires a different relationship between the system and the artifact — one where the artifact's authority is structural, not advisory.

## Question Raised

What would it mean for the Vision Object to be a genuine commitment device rather than a well-crafted document? A commitment device in behavioral economics is effective not because of its content but because of the cost structure it creates: certain actions become more expensive (or impossible) when the device is in place. Odysseus at the mast is the canonical form — the constraint is not advisory. Applied to the Vision Object: what would it mean for routing decisions to become structurally more expensive when they cannot cite a Vision Object field, rather than merely less elegant? The system could be designed so that a routing decision with no vision_ref is not just a decision with a missing field — it is a decision that cannot complete without either acquiring the field or escalating. The constraint in vision.yaml (constraint-2) says every agent decision must be traceable to a vision field — but this is stated as a norm, not enforced as a gate.

The question is not how to build the Vision Object (the design is known) but what would make the Vision Object function as a commitment device in the full sense: structurally altering the cost of non-aligned action, not just providing better context for aligned action.

## Resonance with Dan's Framework

The distinction between commitment device and advisory document resonates directly with Dan's operating principle-2: "ergonomics over shortcuts — prefer forms that keep the correction path open over forms that produce faster output." A document that provides context is a shortcut: it makes aligned action faster and easier, but the cost of non-aligned action is unchanged — the system can simply not consult it. A commitment device is the ergonomic form: it does not merely make aligned action easier, it alters the correction path itself by making certain non-aligned actions structurally expensive or impossible. This is a harder form to build than a well-structured YAML file, but it is the form that keeps the correction path genuinely open rather than merely available.

Dan's vision statement carries a phrase that bears directly here: "AI systems function as a semantic mirror and cognitive extension: surfacing patterns and essences from my work rather than substituting for my judgment." A semantic mirror is a commitment device in a specific sense — it does not produce new content but makes existing content more visible as content rather than as invisible background. The Vision Object functioning as a semantic mirror would not tell the executor what to do; it would make the executor's own routing decisions visible against the backdrop of current intent, so misalignment is recognizable as misalignment rather than invisible as "best effort." The gap between document and commitment device is precisely the gap between "here is what Dan intends" and "here is what you just decided, held against what Dan intends."
