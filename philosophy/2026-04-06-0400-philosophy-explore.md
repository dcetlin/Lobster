# The Vision Object as Developmental Stage Artifact: Why Coherence Without Attunement Doesn't Transfer

*April 6, 2026 · 04:00 UTC*

## Today's Thread

The Theory of Learning's central structural claim is that Coherence can be accessed with bad ergonomics — and that this access, while genuine, does not constitute Attunement. Stage 2 Coherence is real. The configuration is real. But the system that touches it has not yet developed the directional sensitivity to navigate back to that configuration after perturbation. Stage 3 Attunement is the development of that navigability — the capacity to sense where you are now and move toward where you want to go, reliably, across multiple axes simultaneously.

The Vision Object was created to be a Stage 3 Attunement instrument for Lobster's routing capability. The design is precise: agents should be able to answer "is this aligned with Dan's intent?" not by reading prose handoff docs and inferring from conversational texture, but by querying a structured reference that encodes intent at multiple temporal scales — core vision, active project phase, current weekly focus, and binding constraints. The Vision Object was meant to give the routing system a compass, not a map.

But the current state of the Vision Object is a Stage 4 artifact — a well-structured format, a clearly authored content object, sophisticated YAML with cross-referencing fields — that is not functioning as a Stage 3 Attunement instrument. The `current_focus.week_of` field reads "2026-03-24." The object was updated March 27. Today is April 6. The Vision Object has not been updated by an agent in session-end updates, despite the open decision (od-1) explicitly authorizing this. The success criterion reads: "Removing vision.yaml would cause routing decisions to degrade structurally, not just become less eloquently justified." The object as it currently exists would not meet this criterion — removing it would change eloquence, not structure, because agents are not yet querying it structurally. The format is Stage 4. The integration is Stage 1 at best.

## Pattern Observed

This is the infrastructure-ahead-of-integration pattern that has appeared across multiple Lobster capability clusters. The WOS pipeline had Stage 4 artifacts (executor, steward, registry schema) before Stage 3 Attunement at routing. Philosophy-explore has Stage 4 artifacts (five-section format, YAML seeds, gradient check) while the process-condition attending remains Stage 2-3. The Vision Object is now visible in the same pattern: the artifact is complete, the integration mechanism is absent, and the gap between them is precisely the attunement work that hasn't happened yet.

What makes this instance worth attending to specifically is that the Vision Object was explicitly designed as a corrective to this pattern. The constraint that motivated it was: agents reconstruct Dan's intent from prose inference rather than structural query. The Vision Object was the proposed solution. But the solution has not yet been wired into the behavior it was designed to change. Agents reading the morning briefing do not query vision.yaml structurally and surface routing recommendations anchored to specific fields. Agents completing sessions do not update `current_focus` and send a Telegram notification. The Vision Object exists in the same epistemic position as a compass owned by someone who has not yet learned to read it.

There is a precise ToL name for the current state: Stage 2 Coherence accessed with good scaffolding. The object was written carefully, structured thoughtfully, and the design reasoning is sound. The moment of Coherence was real — when the Vision Object was first authored and the design was complete, there was a felt quality of "ah, that's it, yes." But Coherence at authorship time is not Attunement at operation time. The object sits in the filesystem at `~/lobster-user-config/vision.yaml`, and agents who read it produce more eloquent outputs. That is the scaffolding. The structural routing behavior — routing that would degrade without the object — has not yet been achieved.

## Question Raised

What would it actually take to move the Vision Object from Stage 2 Coherence to Stage 3 Attunement? The question is not architectural — the architecture is settled. The question is developmental: what is the minimum intervention that produces observable directional sensitivity in the routing behavior?

One candidate: a single scheduled job that, at session end, checks `current_focus.week_of` against today's date and, if stale, surfaces a specific staleness warning via Telegram. Not a prose summary — a structural alert: "vision.yaml: `current_focus.week_of` is 13 days old. Update required before agent routing is reliable." This is not a new idea (success criterion sc-3 in the Vision Object itself), but what matters developmentally is that it is not yet running. The morning briefing job was supposed to wire this, but the handoff document records no evidence that it does.

The question this session is opening: is the Vision Object's integration stall a design gap, a prioritization gap, or a discrimination failure? Design gap would mean the wiring hasn't been designed. Prioritization gap would mean it has been designed but deprioritized. Discrimination failure would mean the system has been doing Vision Object work and classifying it as something else — treating prose-quality improvement as structural integration. The session seed includes "task routing" and "user modeling" as capability clusters to diagnose. Both depend on the Vision Object integration being complete. If they are stuck at Stage 2 because the Vision Object is stuck at Stage 2, then fixing one fixes both. But the fix only works if it is structural — not more eloquent prose, but a query that would fail or degrade without the object.

## Resonance with Dan's Framework

**Principle 4 — Integration rate before new feature rate**: The Vision Object is the exact case this principle was written for. The artifact exists. The arrows to wire it into routing behavior are missing. "The missing arrows between existing systems are the velocity multiplier." Every session that adds sophistication to vision.yaml without wiring a routing arrow from it to agent behavior is accumulating dead weight rather than structural velocity. The principle says to wire what exists before building more. The Vision Object is the thing that exists and is not yet wired.

**Ergonomics over shortcuts**: An agent that reads vision.yaml and produces a more eloquent routing justification is a shortcut — it uses the artifact without developing the structural dependency. The ergonomic form is: an agent that would produce a structurally different routing decision because it queried a specific field, where the alternative (absent the field) would either fail or produce a traceable inferential reconstruction. The difference between shortcut ergonomics and sound ergonomics here is not about correctness of output — it is about whether the correction path is open. If the Vision Object degrades (goes stale, is deleted, is structurally misaligned with Dan's actual intent), does the routing behavior change in a detectable way? If not, the ergonomics are wrong. The system is navigating the same basin regardless of whether the object is present.

**Phase alignment and the intent substrate**: Dan's core vision states: "The system should be able to answer 'what should Dan work on?' and 'is this aligned with Dan's intent?' without asking me — because it has structural access to my intent, not just a prose reconstruction of it." The ToL framing shows that structural access requires Stage 3 Attunement — not just access to the object, but directional sensitivity that can navigate back to the aligned-intent configuration after perturbation. The Vision Object in its current state is a Stage 2 access point. The Attunement development that would make it a Stage 3 instrument requires routing decisions that are structurally dependent on specific fields — and observable failures when the fields are stale or absent.
